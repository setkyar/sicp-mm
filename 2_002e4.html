2.4Abstract Data အတွက် ကိုယ်စားပြုပုံ အမျိုးမျိုး

ကျွန်ုပ်တို့သည် data abstraction ဟုခေါ်သော စနစ်များကို ဖွဲ့စည်းသည့် နည်းစနစ်တစ်ခုကို မိတ်ဆက်ပေးခဲ့ပြီး ဖြစ်ပါသည်။ ဤနည်းစနစ်သည် ပရိုဂရမ်တစ်ခုမှ ကိုင်တွယ်အသုံးပြုမည့် ဒေတာအရာဝတ္ထု (data objects) များကို ဖော်ဆောင်ရာတွင် အသုံးပြုသည့် နည်းလမ်းများပေါ်တွင် မမှီခိုဘဲ ပရိုဂရမ်၏ အစိတ်အပိုင်း အတော်များများကို သတ်မှတ်နိုင်ရန် လုပ်ဆောင်ပေးပါသည်။ ဥပမာအားဖြင့်၊ ရာရှင်နယ်ကိန်း (rational numbers) များကို အသုံးပြုသည့် ပရိုဂရမ်တစ်ခုကို ဒီဇိုင်းထုတ်သည့် လုပ်ငန်းနှင့် ရာရှင်နယ်ကိန်းများကို ကွန်ပျူတာဘာသာစကား၏ အခြေခံယန္တရားများဖြင့် ဖော်ဆောင်သည့် လုပ်ငန်းတို့ကို မည်သို့ခွဲခြားရမည်ကို 2.1.1 တွင် မြင်တွေ့ခဲ့ရပါသည်။ အဓိက အယူအဆမှာ abstraction barrier တစ်ခု တည်ဆောက်ရန် ဖြစ်သည်။ ဤနေရာတွင် ရာရှင်နယ်ကိန်းများအတွက် selectors နှင့် constructors များ (make-rat၊ numer၊ denom) သည် ရာရှင်နယ်ကိန်းများကို အသုံးပြုပုံနှင့် ၎င်းတို့ကို list structure ဖြင့် ဖော်ဆောင်ထားပုံတို့အကြား သီးခြားစီ ဖြစ်အောင် ခွဲခြားပေးထားပါသည်။ အလားတူ abstraction barrier တစ်ခုသည် ရာရှင်နယ်ကိန်း သင်္ချာလုပ်ဆောင်ချက်များ (add-rat၊ sub-rat၊ mul-rat နှင့် div-rat) ၏ အသေးစိတ်များကို ရာရှင်နယ်ကိန်းများကို အသုံးပြုသည့် “အဆင့်မြင့်” procedures များမှ ခွဲခြားပေးထားသည်။ ရရှိလာသော ပရိုဂရမ်၏ ဖွဲ့စည်းပုံကို Figure 2.1 တွင် ပြသထားသည်။

ဤ data-abstraction barrier များသည် ရှုပ်ထွေးမှုများကို ထိန်းချုပ်ရန်အတွက် အားကောင်းသော ကိရိယာများ ဖြစ်ကြသည်။ ဒေတာအရာဝတ္ထုများ၏ အောက်ခြေဖော်ဆောင်မှုများကို သီးခြားခွဲထုတ်လိုက်ခြင်းဖြင့်၊ ကြီးမားသော ပရိုဂရမ်တစ်ခုကို ဒီဇိုင်းထုတ်သည့် လုပ်ငန်းစဉ်ကို သီးခြားစီ လုပ်ဆောင်နိုင်သော သေးငယ်သည့် လုပ်ငန်းစဉ်များအဖြစ် ခွဲခြမ်းနိုင်ပါသည်။ သို့သော် ဤကဲ့သို့သော data abstraction မျိုးသည် လုံလောက်စွာ အားမကောင်းသေးပါ။ အကြောင်းမှာ ဒေတာအရာဝတ္ထုတစ်ခုအတွက် “အောက်ခြေဖော်ဆောင်မှု” (the underlying representation) တစ်ခုတည်းအကြောင်း ပြောဆိုရန် အမြဲတမ်း မဖြစ်နိုင်သောကြောင့် ဖြစ်သည်။

အကြောင်းတစ်ခုမှာ ဒေတာအရာဝတ္ထုတစ်ခုအတွက် အသုံးဝင်သော ကိုယ်စားပြုပုံ (representation) တစ်ခုထက်မက ရှိနိုင်ပြီး၊ ကိုယ်စားပြုပုံ အမျိုးမျိုး (multiple representations) ကို ကိုင်တွယ်နိုင်သော စနစ်များကို ကျွန်ုပ်တို့ ဒီဇိုင်းထုတ်လိုနိုင်သောကြောင့် ဖြစ်သည်။ ရိုးရှင်းသော ဥပမာတစ်ခုအနေဖြင့်၊ complex numbers (ကိန်းရှုပ်များ) ကို နည်းလမ်းနှစ်မျိုးဖြင့် ဖော်ပြနိုင်သည်- rectangular form (ကိန်းစစ်နှင့် စိတ်ကူးယဉ် အစိတ်အပိုင်းများ) နှင့် polar form (magnitude နှင့် angle) တို့ ဖြစ်ကြသည်။ အချို့သော အခြေအနေများတွင် rectangular form က ပို၍ သင့်လျော်ပြီး အချို့သော အခြေအနေများတွင် polar form က ပို၍ သင့်လျော်ပါသည်။ အမှန်စင်စစ်၊ complex numbers များကို နည်းလမ်းနှစ်မျိုးလုံးဖြင့် ဖော်ပြထားပြီး၊ ၎င်းတို့ကို ကိုင်တွယ်အသုံးပြုသည့် procedures များသည် မည်သည့်ဖော်ပြချက်မျိုးဖြင့်မဆို အလုပ်လုပ်နိုင်သော စနစ်တစ်ခုကို စိတ်ကူးကြည့်ရန်မှာ ဖြစ်နိုင်ခြေ အပြည့်အဝ ရှိပါသည်။

ပို၍အရေးကြီးသည်မှာ ပရိုဂရမ်ရေးဆွဲသည့် စနစ်များကို လူအများအပြားက အချိန်ကြာမြင့်စွာ ဒီဇိုင်းထုတ်ကြရပြီး အချိန်နှင့်အမျှ ပြောင်းလဲလာသော လိုအပ်ချက်များ ရှိတတ်ခြင်း ဖြစ်သည်။ ထိုသို့သော ပတ်ဝန်းကျင်တွင် လူတိုင်းက ဒေတာဖော်ဆောင်ပုံ (data representation) ကို ကြိုတင်၍ သဘောတူညီချက် ရယူထားရန်မှာ မဖြစ်နိုင်ပါ။ ထို့ကြောင့် ဖော်ဆောင်ပုံနှင့် အသုံးပြုပုံကို သီးခြားခွဲထုတ်ထားသော data-abstraction barrier များအပြင်၊ ကွဲပြားသော ဒီဇိုင်းရွေးချယ်မှုများကို တစ်ခုနှင့်တစ်ခု သီးခြားဖြစ်စေပြီး ပရိုဂရမ်တစ်ခုတည်းတွင် ကွဲပြားသော ရွေးချယ်မှုများကို အတူတကွ တည်ရှိခွင့်ပြုသည့် abstraction barrier များလည်း လိုအပ်ပါသည်။ ထို့အပြင်၊ ကြီးမားသော ပရိုဂရမ်များကို သီးခြားစီ ဒီဇိုင်းထုတ်ထားသော ရှိနှင့်ပြီးသား modules များကို ပေါင်းစပ်၍ ဖန်တီးလေ့ရှိသောကြောင့်၊ ပရိုဂရမ်မာများအနေဖြင့် ထို modules များကို ပြန်လည်ဒီဇိုင်းထုတ်ရန် သို့မဟုတ် ပြန်လည်ဖော်ဆောင်ရန် မလိုဘဲ ကြီးမားသော စနစ်များအတွင်းသို့ ဖြည့်စွက်ကာ (additively) ထည့်သွင်းနိုင်မည့် လုပ်ထုံးလုပ်နည်းများ လိုအပ်ပါသည်။

ဤကဏ္ဍတွင်၊ ပရိုဂရမ်၏ မတူညီသော အစိတ်အပိုင်းများမှ မတူညီသော နည်းလမ်းများဖြင့် ဖော်ပြထားသည့် ဒေတာများကို မည်သို့ကိုင်တွယ်ရမည်ကို လေ့လာကြမည် ဖြစ်သည်။ ဤသည်မှာ generic procedures—ဆိုလိုသည်မှာ နည်းလမ်းတစ်ခုထက်မက ဖော်ပြထားသော ဒေတာများအပေါ်တွင် အလုပ်လုပ်နိုင်သော procedures များ တည်ဆောက်ရန် လိုအပ်သည်။ Generic procedures များ တည်ဆောက်ရန် ကျွန်ုပ်တို့၏ အဓိက နည်းပညာမှာ type tags—ဆိုလိုသည်မှာ ၎င်းတို့ကို မည်သို့ ကိုင်တွယ်ရမည်ဆိုသည့် တိကျသော အချက်အလက်များ ပါဝင်သော ဒေတာအရာဝတ္ထုများကို အသုံးပြုခြင်း ဖြစ်သည်။ ကျွန်ုပ်တို့သည် data-directed programming ဟုခေါ်သော generic လုပ်ဆောင်ချက်များပါရှိသော စနစ်များကို ဖြည့်စွက်စုစည်းရန်အတွက် အားကောင်းပြီး အဆင်ပြေသော နည်းဗျူဟာတစ်ခုအကြောင်းကိုလည်း ဆွေးနွေးပါမည်။

ကျွန်ုပ်တို့သည် ရိုးရှင်းသော complex-number ဥပမာဖြင့် စတင်ပါမည်။ Type tags များနှင့် data-directed ပုံစံများက complex numbers များအတွက် abstract “complex-number” ဒေတာအရာဝတ္ထုဟူသော အယူအဆကို ထိန်းသိမ်းထားစဉ်မှာပင်၊ သီးခြားစီဖြစ်သော rectangular နှင့် polar ဖော်ဆောင်မှုများကို မည်သို့ ဒီဇိုင်းထုတ်နိုင်သည်ကို မြင်တွေ့ရမည် ဖြစ်သည်။ ကျွန်ုပ်တို့သည် ကိန်းရှုပ်များ (complex numbers) အတွက် သင်္ချာလုပ်ဆောင်ချက်များကို (add-complex၊ sub-complex၊ mul-complex နှင့် div-complex)၊ ကိန်းရှုပ်တစ်ခုကို မည်သို့ပင် ဖော်ပြထားစေကာမူ ၎င်း၏ အစိတ်အပိုင်းများကို ရယူနိုင်သော generic selectors များကို အသုံးပြု၍ သတ်မှတ်ခြင်းဖြင့် ဤအရာကို လုပ်ဆောင်ပါမည်။ ရရှိလာသော complex-number စနစ်တွင်၊ Figure 2.19 ၌ ပြထားသည့်အတိုင်း မတူညီသော abstraction barrier အမျိုးအစား နှစ်မျိုး ပါဝင်ပါသည်။ “ရေပြင်ညီ” abstraction barrier များသည် Figure 2.1 ရှိ အတားအဆီးများကဲ့သို့ပင် တူညီသော အခန်းကဏ္ဍမှ ပါဝင်ပါသည်။ ၎င်းတို့သည် “အဆင့်မြင့်” လုပ်ဆောင်ချက်များကို “အောက်ခြေ” ဖော်ဆောင်မှုများမှ သီးခြားခွဲထုတ်ပေးပါသည်။ ထို့အပြင်၊ အခြားသော ဖော်ဆောင်မှုများကို သီးခြားစီ ဒီဇိုင်းထုတ်ရန်နှင့် ထည့်သွင်းရန် အခွင့်အရေးပေးသည့် “ဒေါင်လိုက်” အတားအဆီးတစ်ခုလည်း ရှိပါသည်။

2.5 တွင်၊ generic arithmetic package တစ်ခုကို တည်ဆောက်ရန် type tags နှင့် data-directed ပုံစံကို မည်သို့ အသုံးပြုရမည်ကို ပြသပါမည်။ ဤသည်က “ကိန်းဂဏန်း” အမျိုးမျိုးကို ကိုင်တွယ်ရန် အသုံးပြုနိုင်ပြီး၊ ကိန်းဂဏန်း အမျိုးအစားသစ် လိုအပ်လာပါက လွယ်ကူစွာ တိုးချဲ့နိုင်သော procedures များ (add၊ mul စသည်ဖြင့်) ကို ပံ့ပိုးပေးပါသည်။ 2.5.3 တွင်၊ symbolic algebra လုပ်ဆောင်သော စနစ်တစ်ခုတွင် generic arithmetic ကို မည်သို့ အသုံးပြုရမည်ကို ပြသပါမည်။

2.4.2Tag လုပ်ထားသော အချက်အလက်များ

Data abstraction ကို ရှုမြင်နိုင်သည့် နည်းလမ်းတစ်ခုမှာ “principle of least commitment” (အနည်းဆုံး ကတိကဝတ်ပြုခြင်း နိယာမ) ကို အသုံးချခြင်းပင် ဖြစ်သည်။ 2.4.1 ရှိ complex-number စနစ်ကို အကောင်အထည်ဖော်ရာတွင် Ben ၏ rectangular representation သို့မဟုတ် Alyssa ၏ polar representation နှစ်သက်ရာကို အသုံးပြုနိုင်သည်။ selectors နှင့် constructors များဖြင့် ဖွဲ့စည်းထားသော abstraction barrier သည် ကျွန်ုပ်တို့၏ data object များအတွက် ခိုင်မာသော ကိုယ်စားပြုမှုပုံစံ (concrete representation) ကို ရွေးချယ်ခြင်းအား နောက်ဆုံး ဖြစ်နိုင်သမျှ အချိန်အထိ ဆိုင်းငံ့ထားနိုင်ရန် ခွင့်ပြုပေးသဖြင့် စနစ် ဒီဇိုင်းတွင် အမြင့်ဆုံး ပြောင်းလွယ်ပြင်လွယ်ရှိမှုကို ထိန်းသိမ်းထားနိုင်သည်။

principle of least commitment ကို ပိုမိုပြင်းထန်သော အခြေအနေများအထိ ဆက်လက် အသုံးပြုနိုင်သည်။ ကျွန်ုပ်တို့ ဆန္ဒရှိပါက selectors နှင့် constructors များကို ဒီဇိုင်းဆွဲပြီးသည့် နောက်တွင်ပင် ကိုယ်စားပြုမှုပုံစံ၏ မရေရာမှုကို ဆက်လက်ထားရှိနိုင်ပြီး Ben ၏ ကိုယ်စားပြုမှုပုံစံ ရော Alyssa ၏ ကိုယ်စားပြုမှုပုံစံကိုပါ အသုံးပြုရန် ရွေးချယ်နိုင်သည်။ သို့သော် ကိုယ်စားပြုမှုပုံစံ နှစ်မျိုးလုံးကို စနစ်တစ်ခုတည်းတွင် ထည့်သွင်းထားပါက polar ပုံစံရှိ ဒေတာနှင့် rectangular ပုံစံရှိ ဒေတာများကို ခွဲခြားရန် နည်းလမ်းတစ်ခု လိုအပ်မည်ဖြစ်သည်။ သို့မဟုတ်ပါက ဥပမာအားဖြင့် (3, 4) အတွဲ၏ magnitude ကို ရှာခိုင်းလျှင် အဖြေမှာ 5 (rectangular ပုံစံဟု ယူဆလျှင်) လား သို့မဟုတ် 3 (polar ပုံစံဟု ယူဆလျှင်) လား ဆိုသည်ကို ကျွန်ုပ်တို့ သိနိုင်မည် မဟုတ်ပေ။ ဤခွဲခြားမှုကို ပြုလုပ်နိုင်ရန် ရိုးရှင်းသော နည်းလမ်းတစ်ခုမှာ complex number တစ်ခုစီ၏ အစိတ်အပိုင်းတစ်ခုအဖြစ် type tag (အမျိုးအစား တံဆိပ်) — rectangular သို့မဟုတ် polar ဆိုသည့် သင်္ကေတ — ကို ထည့်သွင်းခြင်း ဖြစ်သည်။ ထိုအခါ complex number တစ်ခုကို ကိုင်တွယ်ဖြေရှင်းရန် လိုအပ်သည့်အခါ မည်သည့် selector ကို အသုံးပြုရမည်ကို ဆုံးဖြတ်ရန် ထို tag ကို အသုံးပြုနိုင်သည်။

tag လုပ်ထားသော အချက်အလက်များကို ကိုင်တွယ်ရန်အတွက်၊ data object တစ်ခုမှ tag နှင့် အမှန်တကယ် ပါဝင်သော အချက်အလက် (complex number တစ်ခုတွင်ဆိုလျှင် polar သို့မဟုတ် rectangular ကိုဩဒိနိတ်များ) ကို ထုတ်ယူပေးသည့် type-tag နှင့် contents ဆိုသည့် procedures များ ရှိသည်ဟု ယူဆပါမည်။ tag တစ်ခုနှင့် ပါဝင်သော အချက်အလက်များကို ယူပြီး tag လုပ်ထားသော data object တစ်ခုကို ထုတ်ပေးသည့် attach-tag ဆိုသည့် procedure တစ်ခု ရှိသည်ဟုလည်း ယူဆပါမည်။ ၎င်းကို အကောင်အထည်ဖော်ရန် ရိုးရှင်းသော နည်းလမ်းမှာ သာမန် list တည်ဆောက်ပုံကို အသုံးပြုခြင်း ဖြစ်သည်: 

(define (attach-tag type-tag contents)
  (cons type-tag contents))

(define (type-tag datum)
  (if (pair? datum)
      (car datum)
      (error "Bad tagged datum: 
              TYPE-TAG" datum)))

(define (contents datum)
  (if (pair? datum)
      (cdr datum)
      (error "Bad tagged datum: 
              CONTENTS" datum)))

ဤ procedures များကို အသုံးပြုပြီး rectangular နှင့် polar ကိန်းများကို အသီးသီး ခွဲခြားသိရှိနိုင်သော predicates များဖြစ်သည့် rectangular? နှင့် polar? တို့ကို သတ်မှတ်နိုင်သည်: 

(define (rectangular? z)
  (eq? (type-tag z) 'rectangular))

(define (polar? z)
  (eq? (type-tag z) 'polar))

type tags များကို အသုံးပြုခြင်းဖြင့် Ben နှင့် Alyssa တို့သည် ၎င်းတို့၏ မတူညီသော ကိုယ်စားပြုမှုပုံစံ နှစ်မျိုးစလုံး စနစ်တစ်ခုတည်းတွင် အတူတကွ တည်ရှိနိုင်စေရန် ၎င်းတို့၏ ကုဒ်များကို ပြင်ဆင်နိုင်သည်။ Ben က complex number တစ်ခုကို တည်ဆောက်သည့်အခါတိုင်း ၎င်းကို rectangular အဖြစ် tag လုပ်သည်။ Alyssa က complex number တစ်ခုကို တည်ဆောက်သည့်အခါတိုင်း ၎င်းကို polar အဖြစ် tag လုပ်သည်။ ထို့အပြင် Ben နှင့် Alyssa တို့သည် ၎င်းတို့၏ procedures အမည်များ တစ်ခုနှင့်တစ်ခု ထပ်မနေစေရန် (conflict မဖြစ်စေရန်) သေချာအောင် ပြုလုပ်ရမည်။ ဤသို့ ပြုလုပ်နိုင်သည့် နည်းလမ်းတစ်ခုမှာ Ben က သူ၏ ကိုယ်စားပြုမှု procedure တစ်ခုစီ၏ အမည်တွင် rectangular နောက်ဆက်တွဲကို ထည့်သွင်းရန်နှင့် Alyssa က သူမ၏ procedures အမည်များတွင် polar ကို ထည့်သွင်းရန် ဖြစ်သည်။ 2.4.1 မှ ပြန်လည်ပြင်ဆင်ထားသော Ben ၏ rectangular ကိုယ်စားပြုမှုပုံစံမှာ အောက်ပါအတိုင်း ဖြစ်သည်: 

(define (real-part-rectangular z) (car z))
(define (imag-part-rectangular z) (cdr z))

(define (magnitude-rectangular z)
  (sqrt (+ (square (real-part-rectangular z))
           (square (imag-part-rectangular z)))))

(define (angle-rectangular z)
  (atan (imag-part-rectangular z)
        (real-part-rectangular z)))

(define (make-from-real-imag-rectangular x y)
  (attach-tag 'rectangular (cons x y)))

(define (make-from-mag-ang-rectangular r a)
  (attach-tag 
   'rectangular
   (cons (* r (cos a)) (* r (sin a)))))

(define (real-part-polar z)
  (* (magnitude-polar z) 
     (cos (angle-polar z))))

(define (real-part z)

        (else (error "Unknown type: 
               ANGLE" z))))

complex-number သင်္ချာဆိုင်ရာ လုပ်ဆောင်ချက်များကို အကောင်အထည်ဖော်ရန်အတွက် 2.4.1 မှ add-complex, sub-complex, mul-complex, နှင့် div-complex procedures များကိုပင် အသုံးပြုနိုင်သည်။ အကြောင်းမှာ ၎င်းတို့ခေါ်ယူသော selectors များသည် generic ဖြစ်သောကြောင့် ကိုယ်စားပြုမှုပုံစံ နှစ်မျိုးလုံးနှင့် အလုပ်လုပ်မည် ဖြစ်သည်။ ဥပမာအားဖြင့် add-complex procedure မှာ ယခင်အတိုင်းပင် ဖြစ်သည်: 

(define (add-complex z1 z2)
  (make-from-real-imag 
   (+ (real-part z1) (real-part z2))
   (+ (imag-part z1) (imag-part z2))))

နောက်ဆုံးအနေနှင့် ကျွန်ုပ်တို့သည် complex numbers များကို Ben ၏ ကိုယ်စားပြုမှုပုံစံကို သုံး၍ တည်ဆောက်မည်လား သို့မဟုတ် Alyssa ၏ ကိုယ်စားပြုမှုပုံစံကို သုံးမည်လား ဆိုသည်ကို ရွေးချယ်ရမည်။ သင့်လျော်သော ရွေးချယ်မှုတစ်ခုမှာ ကျွန်ုပ်တို့တွင် real part နှင့် imaginary part များ ရှိသည့်အခါ rectangular ကိန်းများကို တည်ဆောက်ရန်နှင့် magnitude နှင့် angle များ ရှိသည့်အခါ polar ကိန်းများကို တည်ဆောက်ရန် ဖြစ်သည်: 

(define (make-from-real-imag x y)
  (make-from-real-imag-rectangular x y))

(define (make-from-mag-ang r a)
  (make-from-mag-ang-polar r a))

ထွက်ပေါ်လာသော complex-number စနစ်တွင် Figure 2.21 တွင် ပြထားသော တည်ဆောက်ပုံ ရှိသည်။ ဤစနစ်ကို အပြန်အလှန် မှီခိုမှု နည်းပါးသော အစိတ်အပိုင်း သုံးခုအဖြစ် ခွဲထုတ်ထားသည်: complex-number သင်္ချာဆိုင်ရာ လုပ်ဆောင်ချက်များ၊ Alyssa ၏ polar အကောင်အထည်ဖော်မှုနှင့် Ben ၏ rectangular အကောင်အထည်ဖော်မှုတို့ ဖြစ်သည်။ polar နှင့် rectangular အကောင်အထည်ဖော်မှုများကို Ben နှင့် Alyssa တို့က သီးခြားစီ ရေးသားနိုင်ပြီး၊ တတိယမြောက် ပရိုဂရမ်မာ တစ်ဦးက abstract constructor/selector interface ကို အသုံးပြု၍ complex-arithmetic procedures များကို အကောင်အထည်ဖော်ရာတွင် ယင်းတို့ နှစ်ခုလုံးကို အခြေခံ ကိုယ်စားပြုမှုပုံစံများအဖြစ် အသုံးပြုနိုင်သည်။

data object တစ်ခုစီကို ၎င်း၏ အမျိုးအစားဖြင့် tag လုပ်ထားသောကြောင့် selectors များသည် ဒေတာများအပေါ် ယေဘုယျကျသော နည်းလမ်း (generic manner) ဖြင့် လုပ်ဆောင်သည်။ ဆိုလိုသည်မှာ selector တစ်ခုစီသည် ၎င်းကို အသုံးပြုသည့် ဒေတာ အမျိုးအစားအပေါ် မူတည်၍ ကွဲပြားသော လုပ်ဆောင်ချက်များ ရှိရန် သတ်မှတ်ထားသည်။ သီးခြား ကိုယ်စားပြုမှုပုံစံများကို ချိတ်ဆက်ပေးသည့် ယေဘုယျ ယန္တရားကို သတိပြုပါ- ပေးထားသော ကိုယ်စားပြုမှု အကောင်အထည်ဖော်မှုတစ်ခုအတွင်း (ဥပမာ Alyssa ၏ polar package) တွင် complex number တစ်ခုသည် type မပါသော pair (magnitude, angle) တစ်ခု ဖြစ်သည်။ generic selector တစ်ခုသည် polar အမျိုးအစား ကိန်းတစ်ခုအပေါ် လုပ်ဆောင်သည့်အခါ ၎င်းသည် tag ကို ဖယ်ရှားပြီး ပါဝင်သော အချက်အလက် (contents) ကို Alyssa ၏ ကုဒ်သို့ လွှဲပြောင်းပေးသည်။ ဆန့်ကျင်ဘက်အားဖြင့် Alyssa သည် အများသုံးရန်အတွက် ကိန်းတစ်ခုကို တည်ဆောက်သောအခါ၊ ၎င်းကို အဆင့်မြင့် procedures များက သင့်လျော်သလို ခွဲခြားသိရှိနိုင်ရန် အမျိုးအစား tag တစ်ခု ကပ်ပေးသည်။ data object များသည် အဆင့်တစ်ခုမှ တစ်ခုသို့ ဖြတ်သန်းသွားသည့်အခါ tag များကို ဖယ်ရှားခြင်းနှင့် တွဲကပ်ခြင်းဆိုင်ရာ ဤစည်းကမ်းသည် အရေးကြီးသော ဖွဲ့စည်းပုံဆိုင်ရာ မဟာဗျူဟာ တစ်ခု ဖြစ်နိုင်ကြောင်း 2.5 တွင် ကျွန်ုပ်တို့ မြင်တွေ့ရမည်ဖြစ်သည်။

2.4.3Data-Directed Programming နှင့် ပေါင်းစပ်နိုင်စွမ်း (Additivity)

ဒေတာတစ်ခု၏ အမျိုးအစားကို စစ်ဆေးပြီး သင့်လျော်သော လုပ်ထုံးလုပ်နည်း (procedure) ကို ခေါ်ယူခြင်းဟူသည့် ယေဘုယျဗျူဟာကို အမျိုးအစားအပေါ်မူတည်၍ ခွဲဝေလုပ်ဆောင်ခြင်း (dispatching on type) ဟု ခေါ်သည်။ ၎င်းသည် စနစ်ဒီဇိုင်းတွင် မော်ဂျူးပုံစံ (modularity) ရရှိရန်အတွက် အစွမ်းထက်သော ဗျူဟာတစ်ခု ဖြစ်သည်။ အခြားတစ်ဖက်တွင်မူ၊ 2.4.2 တွင်ကဲ့သို့ ခွဲဝေလုပ်ဆောင်မှုကို အကောင်အထည်ဖော်ခြင်းတွင် သိသာထင်ရှားသော အားနည်းချက်နှစ်ခု ရှိသည်။ အားနည်းချက်တစ်ခုမှာ ယေဘုယျ အင်တာဖေ့စ် လုပ်ထုံးလုပ်နည်းများ (real-part၊ imag-part၊ magnitude နှင့် angle) သည် ကွဲပြားသော ကိုယ်စားပြုပုံဖော်မှု (representation) အားလုံးအကြောင်းကို သိရှိထားရခြင်း ဖြစ်သည်။ ဥပမာအားဖြင့်၊ ကျွန်ုပ်တို့၏ ကိန်းရှုပ်စနစ် (complex-number system) ထဲသို့ ကိန်းရှုပ်များအတွက် ကိုယ်စားပြုပုံဖော်မှုအသစ်တစ်ခု ထည့်သွင်းလိုသည်ဟု ဆိုပါစို့။ ကျွန်ုပ်တို့သည် ဤကိုယ်စားပြုပုံဖော်မှုအသစ်ကို အမျိုးအစား (type) တစ်ခုအဖြစ် သတ်မှတ်ပေးရန် လိုအပ်မည်ဖြစ်ပြီး၊ ထို့နောက် ယေဘုယျ အင်တာဖေ့စ် လုပ်ထုံးလုပ်နည်း တစ်ခုချင်းစီတွင် အမျိုးအစားအသစ်ကို စစ်ဆေးရန်နှင့် ထိုကိုယ်စားပြုပုံဖော်မှုအတွက် သင့်လျော်သော ရွေးချယ်ကိရိယာ (selector) ကို အသုံးပြုရန် clause တစ်ခု ထပ်ပေါင်းထည့်ရန် လိုအပ်မည်ဖြစ်သည်။

ဤနည်းပညာ၏ နောက်ထပ်အားနည်းချက်တစ်ခုမှာ တစ်ခုချင်းစီစီ၏ ကိုယ်စားပြုပုံဖော်မှုများကို သီးခြားစီ ဒီဇိုင်းဆွဲနိုင်သော်လည်း၊ စနစ်တစ်ခုလုံးတွင် မည်သည့်လုပ်ထုံးလုပ်နည်းနှစ်ခုမျှ အမည်တူမရှိစေရန် ကျွန်ုပ်တို့ အာမခံရမည် ဖြစ်သည်။ ထို့ကြောင့် Ben နှင့် Alyssa တို့သည် သူတို့၏ မူလလုပ်ထုံးလုပ်နည်းများ၏ အမည်များကို 2.4.1 မှ ပြောင်းလဲခဲ့ကြရခြင်း ဖြစ်သည်။

ဤအားနည်းချက်နှစ်ခုလုံး၏ နောက်ကွယ်မှ ပြဿနာမှာ ယေဘုယျ အင်တာဖေ့စ်များကို အကောင်အထည်ဖော်သည့် နည်းပညာသည် ပေါင်းစပ်နိုင်စွမ်း (additive) မရှိခြင်း ဖြစ်သည်။ ယေဘုယျ ရွေးချယ်ကိရိယာ လုပ်ထုံးလုပ်နည်းများကို အကောင်အထည်ဖော်သူသည် ကိုယ်စားပြုပုံဖော်မှုအသစ်တစ်ခုကို ထည့်သွင်းတိုင်း ထိုလုပ်ထုံးလုပ်နည်းများကို ပြင်ဆင်ရမည်ဖြစ်ပြီး၊ တစ်ခုချင်းစီစီ၏ ကိုယ်စားပြုပုံဖော်မှုများကို ချိတ်ဆက်ပေးသူများသည်လည်း အမည်တူပဋိပက္ခ (name conflict) များမဖြစ်စေရန် သူတို့၏ ကုဒ်များကို ပြင်ဆင်ရမည် ဖြစ်သည်။ ဤအခြေအနေတစ်ခုချင်းစီတွင်၊ ကုဒ်၌ ပြုလုပ်ရမည့် ပြောင်းလဲမှုများသည် ရိုးရှင်းသော်လည်း မလွဲမသွေ ပြုလုပ်ရမည်ဖြစ်ပြီး၊ ၎င်းသည် အဆင်မပြေမှုနှင့် အမှားအယွင်းများ ဖြစ်ပေါ်စေသည့် အရင်းအမြစ်တစ်ခု ဖြစ်သည်။ ဤအချက်သည် လက်ရှိရှိနေသည့် ကိန်းရှုပ်စနစ်အတွက် သိပ်ပြဿနာမဟုတ်သော်လည်း၊ ကိန်းရှုပ်များအတွက် ကိုယ်စားပြုပုံဖော်မှုပေါင်း နှစ်ခုမကဘဲ ရာနှင့်ချီရှိနေသည်ဟု ဆိုပါစို့။ ထို့ပြင် abstract-data interface တွင် ထိန်းသိမ်းရမည့် ယေဘုယျ ရွေးချယ်ကိရိယာများစွာ ရှိနေသည်ဟု ဆိုပါစို့။ အမှန်စင်စစ်၊ မည်သည့် ပရိုဂရမ်မာတစ်ဦးမျှ အင်တာဖေ့စ် လုပ်ထုံးလုပ်နည်းအားလုံး သို့မဟုတ် ကိုယ်စားပြုပုံဖော်မှုအားလုံးကို မသိရှိဟု ဆိုပါစို့။ ဤပြဿနာသည် အမှန်တကယ်ရှိနေပြီး အကြီးစား ဒေတာဘေ့စ်စီမံခန့်ခွဲမှုစနစ်များ (data-base-management systems) ကဲ့သို့သော ပရိုဂရမ်မျိုးတွင် ဖြေရှင်းရန် လိုအပ်ပါသည်။

ကျွန်ုပ်တို့လိုအပ်သည်မှာ စနစ်ဒီဇိုင်းကို ပိုမို၍ မော်ဂျူးပုံစံ ဖွဲ့စည်းနိုင်မည့် နည်းလမ်းတစ်ခု ဖြစ်သည်။ ၎င်းကို data-directed programming ဟုခေါ်သော ပရိုဂရမ်ရေးသားခြင်း နည်းပညာဖြင့် ရရှိနိုင်ပါသည်။ Data-directed programming မည်သို့အလုပ်လုပ်သည်ကို နားလည်ရန်၊ ကွဲပြားသော အမျိုးအစားအစုအဝေးတစ်ခုအတွက် တူညီသော ယေဘုယျ လုပ်ဆောင်ချက်များ (generic operations) အစုအဝေးတစ်ခုကို ကိုင်တွယ်သည့်အခါတိုင်း၊ အမှန်တကယ်တွင် ကျွန်ုပ်တို့သည် ဝင်ရိုးတစ်ခုပေါ်တွင် ဖြစ်နိုင်သော လုပ်ဆောင်ချက်များနှင့် အခြားဝင်ရိုးတစ်ခုပေါ်တွင် ဖြစ်နိုင်သော အမျိုးအစားများ ပါဝင်သည့် နှစ်ဘက်မြင်ဇယား (two-dimensional table) တစ်ခုကို ကိုင်တွယ်နေခြင်းဖြစ်သည်ဟူသော အချက်မှ စတင်ကြည့်ကြပါစို့။ ဇယားရှိ အချက်အလက်များသည် ပေးထားသော အမျိုးအစားတစ်ခုချင်းစီ၏ argument အတွက် လုပ်ဆောင်ချက်တစ်ခုချင်းစီကို အကောင်အထည်ဖော်သည့် လုပ်ထုံးလုပ်နည်းများ ဖြစ်သည်။ ယခင်ကဏ္ဍတွင် ဖော်ထုတ်ခဲ့သော ကိန်းရှုပ်စနစ်၌ လုပ်ဆောင်ချက်အမည်၊ ဒေတာအမျိုးအစားနှင့် အမှန်တကယ် လုပ်ဆောင်မည့် လုပ်ထုံးလုပ်နည်းတို့အကြား ဆက်သွယ်ချက်သည် ယေဘုယျ အင်တာဖေ့စ် လုပ်ထုံးလုပ်နည်းများရှိ အမျိုးမျိုးသော conditional clause များတွင် ပြန့်ကျဲနေခဲ့သည်။ သို့သော် အလားတူ အချက်အလက်များကို Figure 2.22 တွင် ပြထားသည့်အတိုင်း ဇယားတစ်ခုဖြင့် ဖွဲ့စည်းနိုင်သည်။

Data-directed programming ဆိုသည်မှာ ထိုကဲ့သို့သော ဇယားတစ်ခုနှင့် တိုက်ရိုက်အလုပ်လုပ်ရန် ပရိုဂရမ်များကို ဒီဇိုင်းဆွဲသည့် နည်းပညာ ဖြစ်သည်။ ယခင်က၊ ကျွန်ုပ်တို့သည် complex-arithmetic ကုဒ်ကို ကိုယ်စားပြုပုံဖော်မှု ပက်ကေ့ချ်နှစ်ခုနှင့် ချိတ်ဆက်ပေးသည့် ယန္တရားကို အမျိုးအစားအပေါ် တိုက်ရိုက်ခွဲဝေလုပ်ဆောင်ပေးသော လုပ်ထုံးလုပ်နည်း အစုအဝေးတစ်ခုအဖြစ် အကောင်အထည်ဖော်ခဲ့သည်။ ဤနေရာတွင် ကျွန်ုပ်တို့သည် အင်တာဖေ့စ်ကို လုပ်ဆောင်ချက်အမည်နှင့် argument အမျိုးအစား ပေါင်းစပ်မှုကို ဇယားတွင် ရှာဖွေပြီး အသုံးပြုရန် မှန်ကန်သော လုပ်ထုံးလုပ်နည်းကို ရှာဖွေကာ၊ ၎င်းကို argument ၏ အကြောင်းအရာများပေါ်တွင် အသုံးချပေးသည့် လုပ်ထုံးလုပ်နည်း တစ်ခုတည်းအဖြစ် အကောင်အထည်ဖော်မည် ဖြစ်သည်။ အကယ်၍ ကျွန်ုပ်တို့ ဤသို့ပြုလုပ်ပါက၊ စနစ်ထဲသို့ ကိုယ်စားပြုပုံဖော်မှု ပက်ကေ့ချ်အသစ်တစ်ခု ထည့်သွင်းရန်အတွက် လက်ရှိရှိပြီးသား လုပ်ထုံးလုပ်နည်းများကို ပြောင်းလဲရန် မလိုအပ်တော့ဘဲ၊ ဇယားထဲသို့ အချက်အလက်အသစ်များ ဖြည့်စွက်ရုံသာ လိုအပ်မည်ဖြစ်သည်။

ဤအစီအစဉ်ကို အကောင်အထည်ဖော်ရန်အတွက်၊ လုပ်ဆောင်ချက်နှင့် အမျိုးအစားဇယားကို စီမံခန့်ခွဲရန် put နှင့် get ဟူသော လုပ်ထုံးလုပ်နည်းနှစ်ခုရှိသည်ဟု ယူဆပါ-

  • (put ⟨op⟩ ⟨type⟩ ⟨item⟩) သည် ⟨item⟩ ကို ⟨op⟩ နှင့် ⟨type⟩ တို့ဖြင့် အညွှန်းပြု၍ ဇယားထဲတွင် ထည့်သွင်းပေးသည်။
  • (get ⟨op⟩ ⟨type⟩) သည် ဇယားထဲရှိ ⟨op⟩၊ ⟨type⟩ အချက်အလက်ကို ရှာဖွေပြီး တွေ့ရှိသည့် item ကို ပြန်ပေးသည်။ အကယ်၍ မတွေ့ရှိပါက get သည် false ကို ပြန်ပေးသည်။

လောလောဆယ်တွင် put နှင့် get တို့ကို ကျွန်ုပ်တို့၏ ဘာသာစကားတွင် ထည့်သွင်းထားသည်ဟု ယူဆနိုင်သည်။ Chapter 3 (3.3.3) တွင် ဤလုပ်ဆောင်ချက်များနှင့် ဇယားများကို စီမံခန့်ခွဲရန် အခြားသော လုပ်ဆောင်ချက်များကို မည်သို့အကောင်အထည်ဖော်ရမည်ကို တွေ့ရပါမည်။

ကိန်းရှုပ်စနစ်တွင် data-directed programming ကို မည်သို့အသုံးပြုနိုင်ပုံမှာ အောက်ပါအတိုင်း ဖြစ်သည်။ ထောင့်မှန်စတုဂံပုံစံ ကိုယ်စားပြုပုံဖော်မှု (rectangular representation) ကို ဖော်ထုတ်ခဲ့သူ Ben သည် သူ၏ကုဒ်ကို မူလအတိုင်းပင် အကောင်အထည်ဖော်သည်။ သူသည် လုပ်ထုံးလုပ်နည်း အစုအဝေးတစ်ခု သို့မဟုတ် ပက်ကေ့ချ် (package) တစ်ခုကို သတ်မှတ်ပြီး၊ ထောင့်မှန်စတုဂံပုံစံ ကိန်းများပေါ်တွင် စနစ်က မည်သို့လုပ်ဆောင်ရမည်ကို ပြောပြပေးသည့် အချက်အလက်များကို ဇယားတွင် ဖြည့်စွက်ခြင်းဖြင့် စနစ်၏ ကျန်အစိတ်အပိုင်းများနှင့် ချိတ်ဆက်ပေးသည်။ ၎င်းကို အောက်ပါလုပ်ထုံးလုပ်နည်းကို ခေါ်ဆိုခြင်းဖြင့် လုပ်ဆောင်နိုင်သည်- 

(define (install-rectangular-package)
  ;; internal procedures
  (define (real-part z) (car z))
  (define (imag-part z) (cdr z))
  (define (make-from-real-imag x y) 
    (cons x y))
  (define (magnitude z)
    (sqrt (+ (square (real-part z))
             (square (imag-part z)))))
  (define (angle z)
    (atan (imag-part z) (real-part z)))
  (define (make-from-mag-ang r a)
    (cons (* r (cos a)) (* r (sin a))))
  ;; interface to the rest of the system
  (define (tag x) 
    (attach-tag 'rectangular x))
  (put 'real-part '(rectangular) real-part)
  (put 'imag-part '(rectangular) imag-part)
  (put 'magnitude '(rectangular) magnitude)
  (put 'angle '(rectangular) angle)
  (put 'make-from-real-imag 'rectangular
       (lambda (x y) 
         (tag (make-from-real-imag x y))))
  (put 'make-from-mag-ang 'rectangular
       (lambda (r a) 
         (tag (make-from-mag-ang r a))))
  'done)

  (let ((type-tags (map type-tag args)))
    (let ((proc (get op type-tags)))
      (if proc
          (apply proc (map contents args))
          (error
            "No method for these types: 
             APPLY-GENERIC"
            (list op type-tags))))))

  1. သတ်မှတ်ထားသော personnel ဖိုင်တစ်ခုမှ သတ်မှတ်ထားသော ဝန်ထမ်းတစ်ဦး၏ မှတ်တမ်းကို ထုတ်ယူပေးသည့် get-record ပရိုစီဂျာတစ်ခုကို ဌာနချုပ်အတွက် ဖော်ဆောင်ပေးပါ။ ထိုပရိုစီဂျာသည် မည်သည့် division ၏ ဖိုင်ကိုမဆို အသုံးပြုနိုင်ရမည်။ တစ်ခုချင်းစီသော division ဖိုင်များကို မည်သို့တည်ဆောက်ထားသင့်သည်ကို ရှင်းပြပါ။ အထူးသဖြင့် မည်သည့် အမျိုးအစားဆိုင်ရာ အချက်အလက် (type information) ကို ပေးဆောင်ရမည်နည်း။
  2. မည်သည့် division ၏ personnel ဖိုင်မှမဆို ပေးထားသော ဝန်ထမ်းမှတ်တမ်းတစ်ခုမှ လစာအချက်အလက်ကို ပြန်ပေးမည့် get-salary ပရိုစီဂျာတစ်ခုကို ဌာနချုပ်အတွက် ဖော်ဆောင်ပေးပါ။ ဤလုပ်ဆောင်ချက် အလုပ်လုပ်နိုင်ရန် မှတ်တမ်းကို မည်သို့တည်ဆောက်ထားသင့်သနည်း။
  3. ဌာနချုပ်အတွက် find-employee-record ပရိုစီဂျာတစ်ခုကို ဖော်ဆောင်ပေးပါ။ ဤပရိုစီဂျာသည် ပေးထားသော ဝန်ထမ်းတစ်ဦး၏ မှတ်တမ်းအတွက် division အားလုံး၏ ဖိုင်များကို ရှာဖွေပြီး မှတ်တမ်းကို ပြန်ပေးရမည်။ ဤပရိုစီဂျာသည် argument များအဖြစ် ဝန်ထမ်းအမည်နှင့် division ဖိုင်အားလုံး၏ စာရင်း (list) ကို ယူသည်ဟု ယူဆပါ။
  4. Insatiable သည် ကုမ္ပဏီအသစ်တစ်ခုကို လက်ဝယ်ရရှိသည့်အခါ၊ အသစ်ရရှိလာသော ဝန်ထမ်းအချက်အလက်များကို ဗဟိုစနစ်ထဲသို့ ထည့်သွင်းရန် မည်သည့်ပြောင်းလဲမှုများ ပြုလုပ်ရမည်နည်း။
Message passing

Data-directed programming ၏ အဓိက စိတ်ကူးမှာ ပုံ ၂.၂၂ ရှိ ဇယားကဲ့သို့သော operation-and-type table များကို တိုက်ရိုက်ကိုင်တွယ်ခြင်းဖြင့် ပရိုဂရမ်များရှိ generic operation များကို ဖြေရှင်းရန်ဖြစ်သည်။ ၂.၄.၂ တွင် ကျွန်ုပ်တို့ အသုံးပြုခဲ့သော programming ပုံစံသည် operation တစ်ခုချင်းစီကို ၎င်း၏ ကိုယ်ပိုင် dispatching အတွက် တာဝန်ယူစေခြင်းဖြင့် လိုအပ်သော type ပေါ်မူတည်၍ dispatch လုပ်ခြင်းကို စီစဉ်ခဲ့သည်။ လက်တွေ့တွင်၊ ဤအရာသည် operation-and-type table ကို အတန်း (rows) များအဖြစ် ခွဲထုတ်လိုက်ခြင်းဖြစ်ပြီး generic operation ပရိုစီဂျာတစ်ခုစီသည် ဇယား၏ အတန်းတစ်ခုကို ကိုယ်စားပြုသည်။

အခြားရွေးချယ်စရာ ဖော်ဆောင်မှု နည်းဗျူဟာတစ်ခုမှာ ဇယားကို ကော်လံ (columns) များအဖြစ် ခွဲထုတ်ရန်ဖြစ်ပြီး၊ data type များပေါ်မူတည်၍ dispatch လုပ်သော “intelligent operations” များကို အသုံးပြုမည့်အစား operation အမည်များပေါ်မူတည်၍ dispatch လုပ်သော “intelligent data objects” များနှင့် အလုပ်လုပ်ရန်ဖြစ်သည်။ rectangular number ကဲ့သို့သော data object တစ်ခုကို လိုအပ်သော operation အမည်ကို input အဖြစ်ယူ၍ သတ်မှတ်ထားသော operation ကို လုပ်ဆောင်ပေးသည့် ပရိုစီဂျာတစ်ခုအဖြစ် ကိုယ်စားပြုစေခြင်းဖြင့် ဤအရာကို ပြုလုပ်နိုင်သည်။ ထိုကဲ့သို့သော နည်းလမ်းတွင် make-from-real-imag ကို အောက်ပါအတိုင်း ရေးသားနိုင်သည်- 

(define (make-from-real-imag x y)
  (define (dispatch op)
    (cond ((eq? op 'real-part) x)
          ((eq? op 'imag-part) y)
          ((eq? op 'magnitude)
           (sqrt (+ (square x) (square y))))
          ((eq? op 'angle) (atan y x))
          (else
           (error "Unknown op: 
            MAKE-FROM-REAL-IMAG" op))))
  dispatch)

argument တစ်ခုအပေါ် generic operation တစ်ခုကို သက်ရောက်စေသော သက်ဆိုင်ရာ apply-generic ပရိုစီဂျာသည် ယခုအခါ operation ၏ အမည်ကို data object ထံသို့ ပေးပို့ (feed) လိုက်ရုံသာဖြစ်ပြီး object ကိုသာ အလုပ်လုပ်စေသည်- 114 

(define (apply-generic op arg) (arg op))

make-from-real-imag မှ ပြန်ပေးလိုက်သော တန်ဖိုးမှာ ပရိုစီဂျာတစ်ခု (အတွင်းပိုင်းရှိ dispatch ပရိုစီဂျာ) ဖြစ်သည်ကို သတိပြုပါ။ ဤသည်မှာ apply-generic က operation တစ်ခု လုပ်ဆောင်ရန် တောင်းဆိုသောအခါ ခေါ်ယူ (invoke) ခံရသော ပရိုစီဂျာဖြစ်သည်။

ဤကဲ့သို့ ပရိုဂရမ်ရေးသားခြင်း ပုံစံကို message passing ဟု ခေါ်ဆိုပါသည်။ ဤအမည်သည် ဒေတာ အရာဝတ္ထု (data object) တစ်ခုကို တောင်းဆိုထားသော လုပ်ဆောင်ချက် အမည်အား “သတင်းစကား” (message) တစ်ခုအဖြစ် လက်ခံရရှိသည့် အရာတစ်ခု (entity) အဖြစ် မြင်ယောင်ခြင်းမှ ဆင်းသက်လာခြင်း ဖြစ်သည်။ ကျွန်ုပ်တို့သည် 2.1.3 တွင် message passing ၏ ဥပမာတစ်ခုကို တွေ့မြင်ခဲ့ပြီး ဖြစ်သည်။ ထိုနေရာတွင် cons၊ car၊ နှင့် cdr တို့ကို ဒေတာ အရာဝတ္ထုများ မပါဘဲ လုပ်ထုံးလုပ်နည်းများ (procedures) သက်သက်ဖြင့် မည်သို့ အဓိပ္ပာယ်ဖွင့်ဆိုနိုင်သည်ကို တွေ့ရှိခဲ့ရသည်။ ယခုအခါတွင် message passing သည် သင်္ချာဆိုင်ရာ လှည့်ကွက်တစ်ခု မဟုတ်ဘဲ ယေဘုယျ လုပ်ဆောင်ချက်များ (generic operations) ပါဝင်သော စနစ်များကို စနစ်တကျ ဖွဲ့စည်းရန်အတွက် အသုံးဝင်သော နည်းစနစ်တစ်ခု ဖြစ်ကြောင်း တွေ့မြင်ရပါသည်။ ဤအခန်း၏ ကျန်ရှိသော အပိုင်းများတွင် ယေဘုယျ ဂဏန်းသင်္ချာ လုပ်ဆောင်ချက်များ (generic arithmetic operations) အကြောင်း ဆွေးနွေးရန်အတွက် message passing အစား data-directed programming ကိုသာ ဆက်လက် အသုံးပြုသွားမည် ဖြစ်သည်။ Chapter 3 တွင် ကျွန်ုပ်တို့သည် message passing သို့ ပြန်လည် ရောက်ရှိလာမည်ဖြစ်ပြီး၊ ၎င်းသည် simulation ပရိုဂရမ်များကို ဖွဲ့စည်းတည်ဆောက်ရန်အတွက် အားကောင်းသော ကိရိယာတစ်ခု ဖြစ်နိုင်ကြောင်း တွေ့မြင်ရမည် ဖြစ်သည်။

Exercise 2.76: ယေဘုယျ လုပ်ဆောင်ချက်များ ပါဝင်သော ကြီးမားသော စနစ်တစ်ခု တိုးတက်ပြောင်းလဲလာသည်နှင့်အမျှ ဒေတာ အရာဝတ္ထု အမျိုးအစားသစ်များ သို့မဟုတ် လုပ်ဆောင်ချက် အသစ်များ လိုအပ်လာနိုင်ပါသည်။ မဟာဗျူဟာ သုံးမျိုးဖြစ်သည့်—generic operations with explicit dispatch၊ data-directed style နှင့် message-passing-style—တစ်ခုချင်းစီအတွက်၊ အမျိုးအစားသစ်များ သို့မဟုတ် လုပ်ဆောင်ချက်အသစ်များ ပေါင်းထည့်ရန် စနစ်တစ်ခုတွင် ပြုလုပ်ရမည့် အပြောင်းအလဲများကို ဖော်ပြပါ။ အမျိုးအစားသစ်များကို မကြာခဏ ပေါင်းထည့်ရန် လိုအပ်သော စနစ်တစ်ခုအတွက် မည်သည့် ဖွဲ့စည်းပုံက အသင့်တော်ဆုံး ဖြစ်မည်နည်း။ လုပ်ဆောင်ချက်အသစ်များကို မကြာခဏ ပေါင်းထည့်ရန် လိုအပ်သော စနစ်တစ်ခုအတွက် မည်သည့်အရာက အသင့်တော်ဆုံး ဖြစ်မည်နည်း။
