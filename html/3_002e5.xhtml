3.5စီးကြောင်းများ (Streams)

ကျွန်ုပ်တို့သည် ပုံစံတူဖန်တီးခြင်း (modeling) တွင် တန်ဖိုးသတ်မှတ်ခြင်း (assignment) ကို ကိရိယာတစ်ခုအဖြစ် အသုံးပြုခြင်းကို ကောင်းစွာ နားလည်ခဲ့ကြပြီး၊ တန်ဖိုးသတ်မှတ်ခြင်းကြောင့် ဖြစ်ပေါ်လာသည့် ရှုပ်ထွေးသော ပြဿနာများကိုလည်း သတိပြုမိခဲ့ကြပြီ ဖြစ်သည်။ ဤပြဿနာအချို့ကို ရှောင်ရှားနိုင်ရန်အတွက် ကျွန်ုပ်တို့သည် အခြားနည်းလမ်းတစ်ခုဖြင့် လုပ်ဆောင်နိုင်မလားဆိုသည်ကို မေးခွန်းထုတ်ရမည့် အချိန်ရောက်ပြီ ဖြစ်သည်။ ဤကဏ္ဍတွင် ကျွန်ုပ်တို့သည် စီးကြောင်းများ (streams) ဟုခေါ်သော ဒေတာတည်ဆောက်ပုံများအပေါ် အခြေခံ၍ အခြေအနေ (state) ကို ပုံစံတူဖန်တီးခြင်းဆိုင်ရာ အခြားရွေးချယ်စရာ နည်းလမ်းတစ်ခုကို လေ့လာပါမည်။ ကျွန်ုပ်တို့ မြင်တွေ့ရမည့်အတိုင်း စီးကြောင်းများသည် အခြေအနေကို ပုံစံတူဖန်တီးခြင်း၏ ရှုပ်ထွေးမှုအချို့ကို လျှော့ချပေးနိုင်ပါသည်။

ဤရှုပ်ထွေးမှုများသည် မည်သည့်နေရာမှ စတင်လာသည်ကို ပြန်လည်သုံးသပ်ကြည့်ကြပါစို့။ လက်တွေ့ကမ္ဘာမှ ဖြစ်စဉ်များကို ပုံစံတူဖန်တီးရန် ကြိုးပမ်းရာတွင် ကျွန်ုပ်တို့သည် ကျိုးကြောင်းဆီလျော်ပုံရသော ဆုံးဖြတ်ချက်အချို့ကို ချမှတ်ခဲ့ကြသည်- ကျွန်ုပ်တို့သည် လက်တွေ့ကမ္ဘာမှ ဒေသန္တရအခြေအနေ (local state) ရှိသော အရာဝတ္ထုများကို ဒေသန္တရကိန်းရှင်များ (local variables) ရှိသော တွက်ချက်မှုဆိုင်ရာ အရာဝတ္ထုများ (computational objects) ဖြင့် ပုံစံတူဖန်တီးခဲ့ကြသည်။ လက်တွေ့ကမ္ဘာရှိ အချိန်ပြောင်းလဲမှုကို ကွန်ပျူတာအတွင်းရှိ အချိန်ပြောင်းလဲမှုနှင့် ထပ်တူပြုခဲ့ကြသည်။ ကွန်ပျူတာအတွင်းရှိ မော်ဒယ်အရာဝတ္ထုများ၏ အခြေအနေများ အချိန်နှင့်အမျှ ပြောင်းလဲမှုကို ထိုအရာဝတ္ထုများ၏ ဒေသန္တရကိန်းရှင်များသို့ တန်ဖိုးသတ်မှတ်ခြင်းများ (assignments) ဖြင့် အကောင်အထည်ဖော်ခဲ့ကြသည်။

အခြားနည်းလမ်း ရှိပါသေးသလား? ကွန်ပျူတာအတွင်းရှိ အချိန်နှင့် ပုံစံတူဖန်တီးထားသော ကမ္ဘာရှိ အချိန်တို့ကို ထပ်တူပြုခြင်းကို ကျွန်ုပ်တို့ ရှောင်ရှားနိုင်ပါသလား? ပြောင်းလဲနေသော ကမ္ဘာတစ်ခုရှိ ဖြစ်စဉ်များကို ပုံစံတူဖန်တီးရန်အတွက် မော်ဒယ်ကို အချိန်နှင့်အမျှ ပြောင်းလဲရန် လိုအပ်ပါသလား? ဤကိစ္စကို သင်္ချာဆိုင်ရာ ဖန်ရှင်များ (mathematical functions) ၏ ရှုထောင့်မှ စဉ်းစားကြည့်ပါ။ ပမာဏ x ၏ အချိန်နှင့်အမျှ ပြောင်းလဲနေသော အပြုအမူကို အချိန်၏ ဖန်ရှင် x ( t ) အဖြစ် ဖော်ပြနိုင်သည်။ အကယ်၍ ကျွန်ုပ်တို့သည် x ကို တစ်ခဏချင်းစီ (instant by instant) အပေါ်တွင်သာ အာရုံစိုက်ပါက၊ ၎င်းကို ပြောင်းလဲနေသော ပမာဏတစ်ခုအဖြစ် မြင်ပါလိမ့်မည်။ သို့သော် အကယ်၍ ကျွန်ုပ်တို့သည် တန်ဖိုးများ၏ အချိန်မှတ်တမ်းတစ်ခုလုံး (entire time history) ကို အာရုံစိုက်ပါက၊ ပြောင်းလဲမှုကို အလေးမထားတော့ပေ—ဖန်ရှင်ကိုယ်တိုင်က ပြောင်းလဲခြင်း မရှိပါ။180

အကယ်၍ အချိန်ကို သီးခြားအဆင့်များ (discrete steps) ဖြင့် တိုင်းတာပါက၊ အချိန်ဖန်ရှင်တစ်ခုကို (အကန့်အသတ်မရှိ ဖြစ်နိုင်သော) အစီအစဉ် (sequence) တစ်ခုအဖြစ် ပုံစံတူဖန်တီးနိုင်သည်။ ဤကဏ္ဍတွင် ပုံစံတူဖန်တီးထားသော စနစ်များ၏ အချိန်မှတ်တမ်းများကို ကိုယ်စားပြုသည့် အစီအစဉ်များ၏ ရှုထောင့်မှ ပြောင်းလဲမှုကို မည်သို့ပုံစံတူဖန်တီးရမည်ကို မြင်တွေ့ရပါမည်။ ဤအချက်ကို အကောင်အထည်ဖော်ရန်အတွက် ကျွန်ုပ်တို့သည် စီးကြောင်းများ (streams) ဟုခေါ်သော ဒေတာတည်ဆောက်ပုံအသစ်များကို မိတ်ဆက်ပေးပါမည်။ စိတ္တဇရှုထောင့် (abstract point of view) မှကြည့်လျှင် စီးကြောင်းတစ်ခုသည် ရိုးရှင်းသော အစီအစဉ် (sequence) တစ်ခုသာ ဖြစ်သည်။ သို့သော် စီးကြောင်းများကို စာရင်းများ (lists) အဖြစ် တိုက်ရိုက်အကောင်အထည်ဖော်ခြင်းသည် (2.2.1 တွင်ကဲ့သို့) စီးကြောင်းစီမံဆောင်ရွက်မှု (stream processing) ၏ စွမ်းအားကို အပြည့်အဝ မဖော်ပြနိုင်သည်ကို ကျွန်ုပ်တို့ တွေ့ရှိရပါလိမ့်မည်။ အခြားရွေးချယ်စရာတစ်ခုအနေဖြင့်၊ ကျွန်ုပ်တို့သည် နှောင့်နှေးတွက်ချက်ခြင်း (delayed evaluation) နည်းပညာကို မိတ်ဆက်ပါမည်။ ၎င်းသည် ကျွန်ုပ်တို့အား အလွန်ကြီးမားသော (အကန့်အသတ်မရှိပင် ဖြစ်နိုင်သော) အစီအစဉ်များကို စီးကြောင်းများအဖြစ် ကိုယ်စားပြုနိုင်စေပါသည်။

စီးကြောင်းစီမံဆောင်ရွက်မှု (Stream processing) သည် တန်ဖိုးသတ်မှတ်ခြင်း (assignment) သို့မဟုတ် ပြောင်းလဲနိုင်သော ဒေတာများ (mutable data) ကို လုံးဝအသုံးမပြုဘဲ အခြေအနေ (state) ရှိသော စနစ်များကို ပုံစံတူဖန်တီးနိုင်စေသည်။ ၎င်းတွင် သီအိုရီအရရော လက်တွေ့အရပါ အရေးကြီးသော သက်ရောက်မှုများ ရှိသည်၊ အကြောင်းမှာ တန်ဖိုးသတ်မှတ်ခြင်းကို မိတ်ဆက်ခြင်းတွင် ပါရှိသော အားနည်းချက်များကို ရှောင်ရှားနိုင်သည့် မော်ဒယ်များကို ကျွန်ုပ်တို့ တည်ဆောက်နိုင်သောကြောင့် ဖြစ်သည်။ အခြားတစ်ဖက်တွင်မူ စီးကြောင်းမူဘောင် (stream framework) သည် ၎င်း၏ကိုယ်ပိုင် အခက်အခဲများကို ဖြစ်ပေါ်စေပြီး၊ မည်သည့်ပုံစံတူဖန်တီးခြင်း နည်းပညာက ပိုမိုကောင်းမွန်သော မော်ဂျူးဖြစ်ခြင်း (modular) နှင့် ထိန်းသိမ်းရ ပိုမိုလွယ်ကူသော စနစ်များဆီသို့ ဦးတည်စေသနည်း ဆိုသည့် မေးခွန်းမှာ ပွင့်လင်းနေဆဲ ဖြစ်သည်။

3.5.1Streams Are Delayed Lists

2.2.3 တွင် ကျွန်ုပ်တို့ မြင်တွေ့ခဲ့ရသည့်အတိုင်း၊ sequence များသည် ပရိုဂရမ် မော်ဂျူးများကို ပေါင်းစပ်ရန်အတွက် စံသတ်မှတ်ထားသော interface များအဖြစ် အသုံးဝင်ပါသည်။ ကျွန်ုပ်တို့သည် sequence များကို ကိုင်တွယ်ရန်အတွက် map၊ filter နှင့် accumulate ကဲ့သို့သော အစွမ်းထက်သည့် abstraction များကို ဖော်ထုတ်ခဲ့ပြီး၊ ၎င်းတို့သည် အမျိုးမျိုးသော လုပ်ဆောင်ချက်များကို တိုတိုတုတ်တုတ်နှင့် ပြေပြစ်ချောမွေ့သော ပုံစံဖြင့် ဖော်ပြနိုင်ပါသည်။

ကံမကောင်းစွာဖြင့်၊ ကျွန်ုပ်တို့သည် sequence များကို list များအဖြစ် ကိုယ်စားပြုပါက၊ ဤပြေပြစ်ချောမွေ့မှုအတွက် တွက်ချက်မှုဆိုင်ရာ အချိန်နှင့် နေရာလပ် (time and space) နှစ်ခုစလုံးတွင် ကြီးမားသော ထိရောက်မှုမရှိခြင်း (inefficiency) ကို ပေးဆပ်ရမည်ဖြစ်ပါသည်။ sequence များအပေါ် လုပ်ဆောင်ချက်များကို list များအဖြစ် အသွင်ပြောင်းခြင်း (transformation) ဖြင့် ကိုယ်စားပြုသောအခါ၊ ကျွန်ုပ်တို့၏ ပရိုဂရမ်များသည် အဆင့်တိုင်းတွင် (အလွန်ကြီးမားနိုင်သော) data structure များကို တည်ဆောက်ပြီး ကူးယူရမည်ဖြစ်ပါသည်။

ဒါက ဘာကြောင့် မှန်ကန်သလဲဆိုတာကို သိနိုင်ဖို့၊ ကြားကာလတစ်ခုအတွင်းမှာရှိတဲ့ အခြေခံကိန်း (prime number) အားလုံးရဲ့ စုစုပေါင်းပေါင်းလဒ်ကို တွက်ချက်တဲ့ ပရိုဂရမ်နှစ်ခုကို နှိုင်းယှဉ်ကြည့်ရအောင်။ ပထမပရိုဂရမ်ကို စံနှုန်းဖြစ်တဲ့ iterative ပုံစံနဲ့ ရေးသားထားပါတယ်:181 

(define (sum-primes a b)
  (define (iter count accum)
    (cond ((> count b) accum)
          ((prime? count)
           (iter (+ count 1)
                 (+ count accum)))
          (else (iter (+ count 1) accum))))
  (iter a 0))

ဒုတိယပရိုဂရမ်ကတော့ 2.2.3 မှာပါတဲ့ sequence လုပ်ဆောင်ချက်တွေကို အသုံးပြုပြီး အလားတူတွက်ချက်မှုကို လုပ်ဆောင်ပါတယ်- 

(define (sum-primes a b)
  (accumulate 
   +
   0
   (filter prime? (enumerate-interval a b))))

တွက်ချက်မှုကို လုပ်ဆောင်ရာတွင်၊ ပထမပရိုဂရမ်သည် စုဆောင်းထားသော ပေါင်းလဒ် (sum) ကိုသာ သိမ်းဆည်းထားရန် လိုအပ်ပါသည်။ ၎င်းနှင့် ဆန့်ကျင်ဘက်အနေဖြင့်၊ ဒုတိယပရိုဂရမ်ရှိ filter သည် enumerate-interval မှ ကြားကာလအတွင်းရှိ ကိန်းဂဏန်းများ၏ list အပြည့်အစုံကို တည်ဆောက်ပြီးသည့်တိုင်အောင် မည်သည့်စမ်းသပ်မှုကိုမျှ မလုပ်ဆောင်နိုင်ပါ။ filter သည် အခြား list တစ်ခုကို ထုတ်ပေးပြီး၊ ၎င်းကို ပေါင်းလဒ်အဖြစ် မပြောင်းလဲမီ accumulate သို့ ထပ်မံပေးပို့ပါသည်။ ထိုကဲ့သို့သော ကြီးမားသည့် ကြားဖြတ်သိုလှောင်မှု (intermediate storage) သည် ပထမပရိုဂရမ်အတွက် မလိုအပ်ပါ။ ပထမပရိုဂရမ်ကို ကြားကာလရှိ ကိန်းများကို တဖြည်းဖြည်းချင်း ထုတ်ယူပြီး (incrementally) အခြေခံကိန်း တစ်ခုစီထွက်လာတိုင်း ပေါင်းလဒ်ထဲသို့ ထည့်သွင်းပေးသည့် ပုံစံဟု ကျွန်ုပ်တို့ မှတ်ယူနိုင်ပါသည်။

(car (cdr

ဤ expression သည် ဒုတိယမြောက် အခြေခံကိန်းကို ရှာဖွေနိုင်သော်လည်း၊ တွက်ချက်မှုဆိုင်ရာ ကုန်ကျစရိတ် (computational overhead) မှာ အလွန်ကြီးမားပါသည်။ ကျွန်ုပ်တို့သည် ကိန်းဂဏန်းပေါင်း တစ်သန်းနီးပါးရှိသော list တစ်ခုကို တည်ဆောက်ရပြီး၊ ထို list ရှိ အစိတ်အပိုင်းတစ်ခုချင်းစီကို အခြေခံကိန်း ဟုတ်မဟုတ် စစ်ဆေးကာ filter လုပ်ရပြီးနောက်၊ ရရှိလာသော ရလဒ်အားလုံးနီးပါးကို လျစ်လျူရှုလိုက်ပါသည်။ ပိုမိုရိုးရာကျသော ပရိုဂရမ်ရေးနည်းပုံစံတွင်မူ ကျွန်ုပ်တို့သည် ကိန်းစဉ်ထုတ်ယူခြင်းနှင့် filter လုပ်ခြင်းတို့ကို တစ်လှည့်စီလုပ်ဆောင်ပြီး ဒုတိယမြောက် အခြေခံကိန်းကို ရရှိသည်နှင့် ရပ်တန့်သွားမည်ဖြစ်ပါသည်။

Stream များသည် sequence များကို list များအဖြစ် ကိုင်တွယ်ရာတွင် ကုန်ကျမည့် စရိတ်စကများ မရှိဘဲ sequence manipulations များကို အသုံးပြုနိုင်စေသည့် လိမ္မာပါးနပ်သော အကြံဥာဏ်တစ်ခု ဖြစ်ပါသည်။ stream များကို အသုံးပြုခြင်းဖြင့် ကျွန်ုပ်တို့သည် အကောင်းဆုံး ကမ္ဘာနှစ်ခုလုံးကို ရရှိနိုင်ပါသည်- ကျွန်ုပ်တို့သည် ပရိုဂရမ်များကို sequence manipulations များအဖြစ် ပြေပြစ်စွာ ဖော်ထုတ်နိုင်သလို၊ incremental computation ၏ ထိရောက်မှုကိုလည်း ရရှိနိုင်ပါသည်။ အခြေခံ အယူအဆမှာ stream တစ်ခုကို တစ်စိတ်တစ်ပိုင်းသာ တည်ဆောက်ရန် စီစဉ်ထားပြီး၊ ထို တစ်စိတ်တစ်ပိုင်း တည်ဆောက်မှုကို stream ကို အသုံးပြုမည့် (consume လုပ်မည့်) ပရိုဂရမ်သို့ ပေးပို့ရန် ဖြစ်ပါသည်။ အကယ်၍ အသုံးပြုသူ ပရိုဂရမ်သည် မတည်ဆောက်ရသေးသည့် stream ၏ အစိတ်အပိုင်းတစ်ခုကို ရယူရန် ကြိုးပမ်းပါက၊ stream သည် လိုအပ်သော အစိတ်အပိုင်းကို ထုတ်ပေးနိုင်ရန်အတွက် ၎င်း၏ နောက်ထပ် လိုအပ်သမျှကို အလိုအလျောက် ထပ်မံတည်ဆောက်ပေးမည်ဖြစ်ရာ၊ stream တစ်ခုလုံး ရှိနေသည်ဟူသော အထင်အမြင် (illusion) ကို ထိန်းသိမ်းပေးမည် ဖြစ်ပါသည်။ တစ်နည်းအားဖြင့် ဆိုရသော် ကျွန်ုပ်တို့သည် ပရိုဂရမ်များကို sequence အပြည့်အစုံကို ကိုင်တွယ်နေသကဲ့သို့ ရေးသားမည် ဖြစ်သော်လည်း၊ ကျွန်ုပ်တို့၏ stream implementation ကို stream တည်ဆောက်ခြင်းနှင့် ၎င်းကို အသုံးပြုခြင်းတို့အား အလိုအလျောက်နှင့် သိသာထင်ရှားမှုမရှိဘဲ (transparently) တစ်လှည့်စီ လုပ်ဆောင်နိုင်ရန် ဒီဇိုင်းထုတ်ထားပါသည်။

အပြင်ပန်းအားဖြင့် stream များသည် ၎င်းတို့ကို ကိုင်တွယ်သည့် procedure များအတွက် နာမည်အမျိုးမျိုးပေးထားသော list များသာ ဖြစ်ပါသည်။ ၎င်းတွင် constructor တစ်ခု ဖြစ်သည့် cons-stream နှင့် selector နှစ်ခု ဖြစ်သည့် stream-car နှင့် stream-cdr တို့ ပါဝင်ပြီး အောက်ပါ ကန့်သတ်ချက်များကို ပြည့်မီစေပါသည်- 

(stream-car (cons-stream x y)) = x
(stream-cdr (cons-stream x y)) = y

သီးခြားခွဲခြားနိုင်သော အရာဝတ္ထုဖြစ်သည့် the-empty-stream ရှိပြီး၊ ၎င်းသည် မည်သည့် cons-stream လုပ်ဆောင်ချက်၏ ရလဒ်မျှ မဖြစ်နိုင်သလို၊ ၎င်းကို stream-null? predicate ဖြင့် ခွဲခြားသတ်မှတ်နိုင်ပါသည်။182 ထို့ကြောင့် ကျွန်ုပ်တို့သည် sequence အဖြစ် စီစဉ်ထားသော aggregate data များကို ကိုယ်စားပြုရန်အတွက် list များကို ပြုလုပ်ပြီး အသုံးပြုသကဲ့သို့ပင် stream များကိုလည်း ပြုလုပ်ပြီး အသုံးပြုနိုင်ပါသည်။ အထူးသဖြင့်၊ ကျွန်ုပ်တို့သည် အခန်း ၂ မှ list လုပ်ဆောင်ချက်များ ဖြစ်သည့် list-ref၊ map၊ နှင့် for-each တို့၏ stream ဗားရှင်းများကို တည်ဆောက်နိုင်ပါသည်:183 

(define (stream-ref s n)
  (if (= n 0)
      (stream-car s)
      (stream-ref (stream-cdr s) (- n 1))))

(define (stream-map proc s)
  (if (stream-null? s)
      the-empty-stream
      (cons-stream 
       (proc (stream-car s))
       (stream-map proc (stream-cdr s)))))

(define (stream-for-each proc s)
  (if (stream-null? s)
      'done
      (begin 
        (proc (stream-car s))
        (stream-for-each proc 
                         (stream-cdr s)))))

(define (display-stream s)
  (stream-for-each display-line s))

stream implementation သည် stream တည်ဆောက်ခြင်းနှင့် ၎င်းကို အသုံးပြုခြင်းတို့ကို အလိုအလျောက်နှင့် သိသာထင်ရှားမှုမရှိဘဲ တစ်လှည့်စီ လုပ်ဆောင်နိုင်ရန်အတွက်၊ stream ၏ cdr ကို cons-stream ဖြင့် stream တည်ဆောက်ချိန်တွင် မဟုတ်ဘဲ stream-cdr procedure ဖြင့် ၎င်းကို ရယူအသုံးပြုချိန်မှသာ တွက်ချက်နိုင်ရန် ကျွန်ုပ်တို့ စီစဉ်ပါမည်။ ဤကဲ့သို့သော implementation ရွေးချယ်မှုသည် 2.1.2 ပါ rational number များအကြောင်း ဆွေးနွေးမှုကို သတိရစေပါသည်။ ထိုနေရာတွင် rational number များကို အပိုင်းဝေနှင့် အပိုင်းခြေအား အငယ်ဆုံးပုံစံသို့ လျှော့ချခြင်းအား တည်ဆောက်ချိန် (construction time) တွင် ဖြစ်စေ သို့မဟုတ် ရွေးချယ်ချိန် (selection time) တွင် ဖြစ်စေ လုပ်ဆောင်နိုင်ကြောင်း ကျွန်ုပ်တို့ တွေ့ရှိခဲ့ပါသည်။ rational-number implementation နှစ်ခုစလုံးသည် တူညီသော data abstraction ကို ထုတ်ပေးသော်လည်း၊ ရွေးချယ်မှုသည် ထိရောက်မှုအပေါ် သက်ရောက်မှု ရှိပါသည်။ stream များနှင့် သာမန် list များကြားတွင်လည်း အလားတူ ဆက်စပ်မှု ရှိပါသည်။ data abstraction တစ်ခုအနေဖြင့် stream များသည် list များနှင့် တူညီပါသည်။ ကွာခြားချက်မှာ အစိတ်အပိုင်းများကို တွက်ချက်သည့် အချိန် (time) ဖြစ်ပါသည်။ သာမန် list များတွင် car နှင့် cdr နှစ်ခုစလုံးကို တည်ဆောက်ချိန်တွင် တွက်ချက်ပါသည်။ stream များတွင်မူ cdr ကို ရွေးချယ်ချိန်မှသာ တွက်ချက်ပါသည်။

ကျွန်ုပ်တို့၏ stream များ၏ implementation ကို delay ဟုခေါ်သော special form အပေါ်တွင် အခြေခံပါမည်။ (delay ⟨exp⟩) ကို တွက်ချက်ခြင်းသည် expression ⟨exp⟩ ကို ချက်ချင်း တွက်ချက်ခြင်း မဟုတ်ဘဲ၊ delayed object ဟု ခေါ်သော အရာကို ပြန်ပေးပါသည်။ ၎င်းကို အနာဂတ် တစ်ချိန်ချိန်တွင် ⟨exp⟩ ကို တွက်ချက်ရန် "ကတိကဝတ်" (promise) တစ်ခုဟု ကျွန်ုပ်တို့ မှတ်ယူနိုင်ပါသည်။ delay နှင့် တွဲဖက်ရန်အတွက် force ဟုခေါ်သော procedure တစ်ခု ရှိပြီး၊ ၎င်းသည် delayed object ကို argument အဖြစ် ယူကာ တွက်ချက်မှုကို လုပ်ဆောင်ပါသည် — အကျိုးသက်ရောက်မှုအားဖြင့် delay ကို ၎င်း၏ ကတိအတိုင်း လုပ်ဆောင်ရန် တိုက်တွန်းခြင်း (forcing) ဖြစ်ပါသည်။ delay နှင့် force တို့ကို မည်သို့ implement လုပ်နိုင်သည်ကို အောက်တွင် မြင်တွေ့ရမည် ဖြစ်သော်လည်း၊ ဦးစွာ ၎င်းတို့ကို အသုံးပြု၍ stream များကို တည်ဆောက်ကြည့်ရအောင်။

Cons-stream သည် အောက်ပါအတိုင်း သတ်မှတ်ထားသော special form တစ်ခု ဖြစ်ပါသည်- 

(cons-stream ⟨a⟩ ⟨b⟩)

၎င်းသည် အောက်ပါအတိုင်း ညီမျှပါသည်- 

(cons ⟨a⟩ (delay ⟨b⟩))

ဒါက ဘာကို ဆိုလိုသလဲဆိုတော့ ကျွန်ုပ်တို့ဟာ pair တွေကို အသုံးပြုပြီး stream တွေကို တည်ဆောက်မှာ ဖြစ်ပါတယ်။ ဒါပေမယ့် ကျန်ရှိနေတဲ့ stream ရဲ့ တန်ဖိုးကို pair ရဲ့ cdr ထဲကို ထည့်မယ့်အစား၊ အကယ်၍ လိုအပ်ခဲ့ရင် ကျန်ရှိနေတဲ့ အစိတ်အပိုင်းကို တွက်ချက်ပေးမယ့် ကတိကဝတ် (promise) တစ်ခုကို ထိုနေရာမှာ ထည့်သွင်းမှာ ဖြစ်ပါတယ်။ Stream-car နှင့် stream-cdr တို့ကို အောက်ပါအတိုင်း procedure များအဖြစ် သတ်မှတ်နိုင်ပါသည်- 

(define (stream-car stream) 
  (car stream))

(define (stream-cdr stream) 
  (force (cdr stream)))

Stream-car သည် pair ၏ car ကို ရွေးချယ်ပြီး၊ stream-cdr သည် pair ၏ cdr ကို ရွေးချယ်ကာ ထိုနေရာတွင် တွေ့ရှိရသော delayed expression ကို တွက်ချက်ပြီး stream ၏ ကျန်ရှိသော အစိတ်အပိုင်းကို ရယူပါသည်။184

The stream implementation in action

ဤ implementation မည်သို့ အလုပ်လုပ်သည်ကို သိနိုင်ရန်၊ ကျွန်ုပ်တို့ အထက်တွင် မြင်တွေ့ခဲ့ရသည့် stream ပုံစံဖြင့် ပြန်လည်ရေးသားထားသော "ကုန်ကျစရိတ်ကြီးမားသည့်" အခြေခံကိန်း တွက်ချက်မှုကို ဆန်းစစ်ကြည့်ရအောင်- 

(stream-car 
 (stream-cdr
  (stream-filter 
   prime? (stream-enumerate-interval 
           10000 1000000))))

၎င်းသည် အမှန်တကယ်ပင် ထိရောက်စွာ အလုပ်လုပ်သည်ကို ကျွန်ုပ်တို့ တွေ့ရှိရပါမည်။

ကျွန်ုပ်တို့သည် ဦးစွာ stream-enumerate-interval ကို ၁၀,၀၀၀ နှင့် ၁,၀၀၀,၀၀၀ ဟူသော argument များဖြင့် ခေါ်ဆိုပါမည်။ Stream-enumerate-interval သည် enumerate-interval (2.2.3) ၏ stream ဗားရှင်း ဖြစ်ပါသည်- 

(define (stream-enumerate-interval low high)
  (if (> low high)
      the-empty-stream
      (cons-stream
       low
       (stream-enumerate-interval (+ low 1)
                                  high))))

ထို့ကြောင့် stream-enumerate-interval မှ ပြန်ပေးလိုက်သော cons-stream ဖြင့် ဖွဲ့စည်းထားသည့် ရလဒ်မှာ185 

(cons 10000
      (delay 
        (stream-enumerate-interval 
         10001 
         1000000)))

ဆိုလိုသည်မှာ stream-enumerate-interval သည် ၎င်း၏ car တွင် ၁၀,၀၀၀ ပါဝင်ပြီး ၎င်း၏ cdr တွင် လိုအပ်ပါက ကြားကာလ၏ နောက်ထပ် အစိတ်အပိုင်းများကို ထုတ်ပေးမည်ဟူသော ကတိကဝတ် တစ်ခု ပါဝင်သည့် pair တစ်ခုအဖြစ် stream တစ်ခုကို ပြန်ပေးပါသည်။ ယခု ဤ stream ကို filter procedure (2.2.3) ၏ stream ဗားရှင်းကို အသုံးပြု၍ အခြေခံကိန်းများအတွက် filter လုပ်ပါမည်- 

(define (stream-filter pred stream)
  (cond ((stream-null? stream) 
         the-empty-stream)
        ((pred (stream-car stream))
         (cons-stream 
          (stream-car stream)
          (stream-filter 
           pred
           (stream-cdr stream))))
        (else (stream-filter 
               pred 
               (stream-cdr stream)))))

Stream-filter သည် stream ၏ stream-car (pair ၏ car ဖြစ်သော ၁၀,၀၀၀) ကို စမ်းသပ်ပါသည်။ ၎င်းသည် အခြေခံကိန်း မဟုတ်သောကြောင့်၊ stream-filter သည် ၎င်း၏ input stream ၏ stream-cdr ကို စစ်ဆေးပါသည်။ stream-cdr ကို ခေါ်ဆိုခြင်းသည် delayed ဖြစ်နေသော stream-enumerate-interval ကို တွက်ချက်စေပြီး အောက်ပါအတိုင်း ပြန်ပေးပါသည်- 

(cons 10001
      (delay 
        (stream-enumerate-interval 
         10002 
         1000000)))

Stream-filter သည် ယခုအခါ ဤ stream ၏ stream-car ဖြစ်သော ၁၀,၀၀၁ ကို ကြည့်ရှုပြီး၊ ၎င်းသည်လည်း အခြေခံကိန်း မဟုတ်သည်ကို တွေ့ရသောကြောင့် နောက်ထပ် stream-cdr တစ်ခုကို တိုက်တွန်း (force) လုပ်ဆောင်စေပြီး၊ stream-enumerate-interval မှ အခြေခံကိန်း ၁၀,၀၀၇ ကို ထုတ်ပေးသည့်တိုင်အောင် ထိုသို့ ဆက်လက်လုပ်ဆောင်ပါသည်။ ထိုအခါ stream-filter သည် ၎င်း၏ အဓိပ္ပာယ်ဖွင့်ဆိုချက်အရ အောက်ပါအတိုင်း ပြန်ပေးပါသည်- 

(cons-stream 
 (stream-car stream)
 (stream-filter pred (stream-cdr stream)))

ဤကိစ္စတွင် ၎င်းမှာ- 

(cons 10007
      (delay
        (stream-filter
         prime?
         (cons 10008
               (delay
                 (stream-enumerate-interval 
                  10009 1000000))))))

ဤရလဒ်ကို ကျွန်ုပ်တို့၏ မူလ expression ရှိ stream-cdr သို့ ပေးပို့ပါသည်။ ၎င်းသည် delayed ဖြစ်နေသော stream-filter ကို တွက်ချက်စေပြီး၊ ၎င်းမှတစ်ဆင့် နောက်ထပ် အခြေခံကိန်းဖြစ်သော ၁၀,၀၀၉ ကို တွေ့ရှိသည်အထိ delayed ဖြစ်နေသော stream-enumerate-interval ကို ထပ်ခါတလဲလဲ တိုက်တွန်းတွက်ချက်စေပါသည်။ နောက်ဆုံးတွင် ကျွန်ုပ်တို့၏ မူလ expression ရှိ stream-car သို့ ပေးပို့သော ရလဒ်မှာ- 

(cons 10009
      (delay
        (stream-filter
         prime?
         (cons 10010
               (delay
                 (stream-enumerate-interval 
                  10011 1000000))))))

Stream-car သည် ၁၀,၀၀၉ ကို ပြန်ပေးပြီး တွက်ချက်မှု ပြီးဆုံးသွားပါသည်။ ဒုတိယမြောက် အခြေခံကိန်းကို ရှာဖွေရန် လိုအပ်သမျှသော ကိန်းဂဏန်းများကိုသာ အခြေခံကိန်း ဟုတ်မဟုတ် စမ်းသပ်ခဲ့ပြီး၊ prime filter သို့ ပေးပို့ရန် လိုအပ်သမျှ အကွာအဝေးအထိသာ ကြားကာလကို ထုတ်ယူခဲ့ပါသည်။

ယေဘုယျအားဖြင့်၊ ကျွန်ုပ်တို့သည် delayed evaluation ကို "လိုအပ်ချက်အရ မောင်းနှင်သော" (demand-driven) ပရိုဂရမ်ရေးနည်းဟု မှတ်ယူနိုင်ပြီး၊ ထိုနည်းဖြင့် stream လုပ်ငန်းစဉ်ရှိ အဆင့်တစ်ခုစီသည် နောက်အဆင့်တစ်ခုကို ကျေနပ်စေရန် လုံလောက်ရုံမျှသာ အလုပ်လုပ်ပါသည်။ ကျွန်ုပ်တို့ လုပ်ဆောင်ခဲ့သည်မှာ တွက်ချက်မှုတွင် အမှန်တကယ် ဖြစ်ပျက်သည့် အစဉ်အတိုင်းကို ကျွန်ုပ်တို့၏ procedure များ၏ အပြင်ပန်း တည်ဆောက်ပုံနှင့် သီးခြားစီ ခွဲထုတ်လိုက်ခြင်း ဖြစ်ပါသည်။ ကျွန်ုပ်တို့သည် stream များသည် "အားလုံး တစ်ပြိုင်နက်တည်း" ရှိနေသကဲ့သို့ procedure များကို ရေးသားသော်လည်း၊ အမှန်တကယ်တွင်မူ တွက်ချက်မှုကို ရိုးရာ ပရိုဂရမ်ရေးနည်း စတိုင်များကဲ့သို့ပင် တဖြည်းဖြည်းချင်း (incrementally) လုပ်ဆောင်ပါသည်။

Implementing delay and force

Delay နှင့် force တို့သည် လျှို့ဝှက်ဆန်းကြယ်သော လုပ်ဆောင်ချက်များဟု ထင်ရသော်လည်း၊ ၎င်းတို့၏ implementation သည် အမှန်တကယ်တွင် အလွန်ရိုးရှင်းပါသည်။ Delay သည် expression တစ်ခုကို နောက်မှ လိုအပ်သလို တွက်ချက်နိုင်ရန်အတွက် ထုပ်ပိုးပေးရမည်ဖြစ်ပြီး၊ ၎င်းကို expression အား procedure တစ်ခု၏ body အဖြစ် သဘောထားခြင်းဖြင့် လွယ်ကူစွာ လုပ်ဆောင်နိုင်ပါသည်။ Delay သည် အောက်ပါအတိုင်း special form တစ်ခု ဖြစ်နိုင်ပါသည်- 

(delay ⟨exp⟩)

၎င်းသည် အောက်ပါအတွက် syntactic sugar ဖြစ်ပါသည်- 

(lambda () ⟨exp⟩)

Force သည် delay မှ ထုတ်ပေးသော (argument မပါသော) procedure ကို ခေါ်ဆိုခြင်းသာ ဖြစ်သောကြောင့်၊ force ကို procedure တစ်ခုအဖြစ် အောက်ပါအတိုင်း implement လုပ်နိုင်ပါသည်- 

(define (force delayed-object)
  (delayed-object))

ဤ implementation သည် delay နှင့် force တို့အား ကြေညာထားသည့်အတိုင်း အလုပ်လုပ်စေရန် လုံလောက်သော်လည်း၊ ကျွန်ုပ်တို့ ထည့်သွင်းနိုင်သည့် အရေးကြီးသော optimization တစ်ခု ရှိပါသည်။ အသုံးချမှု အများအပြားတွင် ကျွန်ုပ်တို့သည် တူညီသော delayed object ကို အကြိမ်ကြိမ် force လုပ်မိတတ်ပါသည်။ ၎င်းသည် stream များ ပါဝင်သော recursive ပရိုဂရမ်များတွင် ဆိုးရွားသော ထိရောက်မှုမရှိခြင်းကို ဖြစ်ပေါ်စေနိုင်ပါသည်။ (လေ့ကျင့်ခန်း ၃.၅၇ ကို ကြည့်ပါ။) ဖြေရှင်းနည်းမှာ delayed object များကို ပထမဆုံးအကြိမ် force လုပ်သည့်အခါ တွက်ချက်လိုက်သော တန်ဖိုးကို သိမ်းဆည်းထားရန် တည်ဆောက်ခြင်း ဖြစ်ပါသည်။ နောက်ပိုင်းတွင် ထပ်မံ force လုပ်ပါက တွက်ချက်မှုကို ထပ်မံ မလုပ်ဆောင်တော့ဘဲ သိမ်းဆည်းထားသော တန်ဖိုးကိုသာ ပြန်ပေးမည် ဖြစ်ပါသည်။ တစ်နည်းအားဖြင့် ဆိုရသော် ကျွန်ုပ်တို့သည် delay ကို လေ့ကျင့်ခန်း ၃.၂၇ တွင် ဖော်ပြထားသော ပုံစံနှင့် ဆင်တူသည့် အထူးရည်ရွယ်ချက်သုံး memoized procedure တစ်ခုအဖြစ် implement လုပ်ခြင်း ဖြစ်ပါသည်။ ၎င်းကို လုပ်ဆောင်နိုင်သော နည်းလမ်းတစ်ခုမှာ အောက်ပါ procedure ကို အသုံးပြုရန် ဖြစ်ပြီး၊ ၎င်းသည် (argument မပါသော) procedure တစ်ခုကို argument အဖြစ် ယူကာ ထို procedure ၏ memoized version ကို ပြန်ပေးပါသည်။ memoized procedure ကို ပထမဆုံးအကြိမ် run သည့်အခါ တွက်ချက်ထားသော ရလဒ်ကို သိမ်းဆည်းထားပါသည်။ နောက်ပိုင်း တွက်ချက်မှုများတွင် ရလဒ်ကိုသာ ပြန်ပေးပါသည်။ 

(define (memo-proc proc)
  (let ((already-run? false) (result false))
    (lambda ()
      (if (not already-run?)
          (begin (set! result (proc))
                 (set! already-run? true)
                 result)
          result))))

(memo-proc (lambda () ⟨exp⟩))

                   s2car

3.5.3Stream Paradigm ကို အသုံးချခြင်း

Delayed evaluation ပါရှိသော Stream များသည် local state နှင့် assignment တို့၏ အကျိုးကျေးဇူးများစွာကို ပေးစွမ်းနိုင်သည့် အားကောင်းသော ပုံစံထုတ်လုပ်သည့် ကိရိယာ (modeling tool) တစ်ခု ဖြစ်နိုင်ပါသည်။ ထို့အပြင်၊ ၎င်းတို့သည် ပရိုဂရမ်မင်းဘာသာစကားတစ်ခုအတွင်းသို့ assignment ကို ထည့်သွင်းရာတွင် ပါရှိလာတတ်သည့် သီအိုရီဆိုင်ရာ ရှုပ်ထွေးမှုအချို့ကိုလည်း ရှောင်ရှားနိုင်ပါသည်။

Stream ချဉ်းကပ်မှုသည် အလင်းပြနိုင်စွမ်းရှိပါသည်၊ အကြောင်းမှာ ၎င်းသည် state variables များသို့ assignment ပြုလုပ်ခြင်းအပေါ် အခြေခံ၍ ဖွဲ့စည်းထားသော စနစ်များထက် မတူညီသော မော်ဂျူးနယ်နိမိတ် (module boundaries) များဖြင့် စနစ်များကို တည်ဆောက်ရန် ခွင့်ပြုသောကြောင့် ဖြစ်သည်။ ဥပမာအားဖြင့်၊ ကျွန်ုပ်တို့သည် တစ်ဦးချင်းစီ၏ အချိန်အခိုက်အတန့်များတွင် ရှိသော state variables များ၏ တန်ဖိုးများထက် အချိန်စီးရီး (time series) တစ်ခုလုံး (သို့မဟုတ် အချက်ပြလှိုင်း - signal) ကို စိတ်ဝင်စားဖွယ်အချက်အဖြစ် သတ်မှတ်နိုင်သည်။ ၎င်းသည် မတူညီသော အချိန်အခိုက်အတန့်များမှ state အစိတ်အပိုင်းများကို ပေါင်းစပ်ရန်နှင့် နှိုင်းယှဉ်ရန် အဆင်ပြေစေပါသည်။

ယခုအချိန်အထိ၊ stream of states ချဉ်းကပ်မှုကို အသုံးပြုခြင်းသည် state variables များကို update လုပ်ခြင်းနှင့် များစွာမကွာခြားသေးပါ။ သို့သော် stream များသည် စိတ်ဝင်စားဖွယ်ကောင်းသော လှည့်ကွက်အချို့ကို လုပ်ဆောင်ရန် အခွင့်အရေးပေးပါသည်။ ဥပမာအားဖြင့်၊ ကျွန်ုပ်တို့သည် နီးစပ်သည့်တန်ဖိုးများ၏ sequence တစ်ခုကို မူလတန်ဖိုးအတိုင်းပင် သို့သော် ပိုမိုမြန်ဆန်စွာ စုစည်း (converge) သော sequence အသစ်တစ်ခုသို့ ပြောင်းလဲပေးသည့် sequence accelerator တစ်ခုဖြင့် stream တစ်ခုကို အသွင်ပြောင်း (transform) နိုင်သည်။

ဆယ့်ရှစ်ရာစု ဆွစ်ဇာလန် သင်္ချာပညာရှင် Leonhard Euler ဖော်ထုတ်ခဲ့သော ထိုကဲ့သို့သော accelerator တစ်ခုသည် alternating series (လက္ခဏာများ အလှည့်ကျပြောင်းသော series များ) ၏ အစိတ်အပိုင်းပေါင်းလဒ်များ (partial sums) ဖြစ်သော sequences များအတွက် ကောင်းမွန်စွာ အလုပ်လုပ်ပါသည်။ Euler ၏ နည်းပညာတွင်၊ အကယ်၍ S n သည် မူလပေါင်းလဒ် sequence ၏ n th မြောက် term ဖြစ်လျှင်၊ အရှိန်မြှင့်ထားသော sequence တွင် ပါဝင်သော terms များမှာ- S n + 1 − ( S n + 1 − S n ) 2 S n − 1 − 2 S n + S n + 1 . ထို့ကြောင့်၊ အကယ်၍ မူလ sequence ကို တန်ဖိုးများ၏ stream တစ်ခုအဖြစ် ကိုယ်စားပြုပါက၊ အသွင်ပြောင်းထားသော sequence ကို အောက်ပါအတိုင်း ရရှိသည်- 

(define (euler-transform s)
  (let ((s0 (stream-ref s 0))     ; Sₙ₋₁
        (s1 (stream-ref s 1))     ; Sₙ
        (s2 (stream-ref s 2)))    ; Sₙ₊₁
    (cons-stream 
     - s2 (/ (square (- s2 s1))
              (+ s0 (* -2 s1) s2)))
     (euler-transform (stream-cdr s)))))

  (cons-stream 
   1.0
   (stream-map (lambda (guess)
                 (sqrt-improve guess x))
               (sqrt-stream x))))

အပိုင်း 2.2.3 မှာ sequence paradigm ဟာ ရိုးရာ nested loop တွေကို pair များ၏ sequence အပေါ်မှာ သတ်မှတ်ထားတဲ့ process တွေအဖြစ် ဘယ်လို ကိုင်တွယ်သလဲဆိုတာကို ကျွန်ုပ်တို့ တွေ့မြင်ခဲ့ရပါတယ်။ အကယ်၍ ကျွန်ုပ်တို့ဟာ ဒီနည်းစနစ်ကို infinite streams တွေအထိ ယေဘုယျပြု (generalize) လိုက်မယ်ဆိုရင်၊ loop အနေနဲ့ ကိုယ်စားပြုဖို့ မလွယ်ကူတဲ့ ပရိုဂရမ်တွေကို ရေးသားနိုင်မှာ ဖြစ်ပါတယ်၊ ဘာကြောင့်လဲဆိုတော့ “looping” ဟာ infinite set တစ်ခုအပေါ်မှာ ပျံ့နှံ့နေရမှာ ဖြစ်လို့ပါ။

ဥပမာအားဖြင့်၊ 2.2.3 မှ prime-sum-pairs procedure ကို i ≤ j ဖြစ်ပြီး i + j ဟာ prime number ဖြစ်တဲ့ integer pair ( i , j ) အားလုံးရဲ့ stream ကို ထုတ်ပေးနိုင်ဖို့ generalize လုပ်ချင်တယ် ဆိုပါစို့။ အကယ်၍ int-pairs ဟာ i ≤ j ဖြစ်တဲ့ integer pair ( i , j ) အားလုံးရဲ့ sequence ဖြစ်မယ်ဆိုရင်၊ ကျွန်ုပ်တို့ လိုအပ်တဲ့ stream ကတော့ ရိုးရိုးရှင်းရှင်းပဲ194

 (lambda (pair)
   (prime? (+ (car pair) (cadr pair))))

    (scale-stream integrand dt) int)))

ထိုသို့သော သွယ်ဝိုက်သောအဓိပ္ပာယ်ဖွင့်ဆိုချက်ကို အင်တာပရီတာ (interpreter) က ကိုင်တွယ်နိုင်စွမ်းမှာ cons-stream တွင် ပေါင်းစပ်ပါဝင်သော delay အပေါ် မူတည်ပါသည်။ ဤ delay မရှိပါက၊ အင်တာပရီတာသည် cons-stream သို့ ပေးပို့သော အာဂူးမန့် (arguments) နှစ်ခုလုံးကို အကဲမဖြတ်မီ int ကို တည်ဆောက်နိုင်မည် မဟုတ်ပါ။ ၎င်းသည် int ကို ကြိုတင်အဓိပ္ပာယ်ဖွင့်ဆိုထားရန် လိုအပ်ပါလိမ့်မည်။ ယေဘုယျအားဖြင့်၊ loop များပါဝင်သော အချက်ပြမှုလုပ်ငန်းစဉ်စနစ်များကို ပုံဖော်ရန် စထရင်းများကို အသုံးပြုရာတွင် delay သည် အလွန်အရေးကြီးပါသည်။ delay မပါရှိပါက၊ ကျွန်ုပ်တို့၏ မော်ဒယ်များသည် မည်သည့် အချက်ပြမှုလုပ်ငန်းစဉ်အစိတ်အပိုင်းသို့မဆို ပေးပို့သော input များကို ရလဒ် (output) မထုတ်လုပ်မီ အပြည့်အဝ အကဲဖြတ်ပြီး ဖြစ်နေစေရန် ဖော်ပြရပါလိမ့်မည်။ ၎င်းသည် loop များ အသုံးပြုခြင်းကို တားဆီးရာ ရောက်ပါလိမ့်မည်။

ကံမကောင်းစွာဖြင့်၊ loop များပါသော စနစ်များ၏ စထရင်းမော်ဒယ်များသည် cons-stream မှ ထောက်ပံ့ပေးထားသော "ဝှက်ထားသော" delay ထက်ပို၍ delay ကို အသုံးပြုရန် လိုအပ်နိုင်ပါသည်။ ဥပမာအားဖြင့်၊ ပုံ ၃.၃၄ တွင် f သည် ပေးထားသော ဖန်ရှင်တစ်ခုဖြစ်သည့် ဒစ်ဖရန်ရှယ်ညီမျှခြင်း d y / d t = f ( y ) ကို ဖြေရှင်းရန် အချက်ပြမှုလုပ်ငန်းစဉ်စနစ်တစ်ခုကို ပြသထားပါသည်။ ထိုပုံတွင် input signal ကို f ဖြင့် တွက်ချက်ပေးသော mapping component တစ်ခုကို အင်တီဂရေတာ (integrator) တစ်ခုနှင့် feedback loop အဖြစ် ချိတ်ဆက်ထားပုံကို ပြထားပါသည်။ ၎င်းသည် ထိုသို့သော ညီမျှခြင်းများကို ဖြေရှင်းရန် အမှန်တကယ်အသုံးပြုသော analog computer circuit များနှင့် အလွန်ဆင်တူပါသည်။

y အတွက် ကနဦးတန်ဖိုး y 0 ကို ပေးထားသည်ဟု ယူဆလျှင်၊ ဤစနစ်ကို အောက်ပါ လုပ်ထုံးလုပ်နည်းဖြင့် ပုံဖော်ရန် ကြိုးစားနိုင်ပါသည် - 

(define (solve f y0 dt)
  (define y (integral dy y0 dt))
  (define dy (stream-map f y))
  y)

ဤလုပ်ထုံးလုပ်နည်းသည် အလုပ်မလုပ်ပါ။ အကြောင်းမှာ solve ၏ ပထမစာကြောင်းတွင် integral ကို ခေါ်ဆိုရာ၌ input dy ကို အဓိပ္ပာယ်ဖွင့်ဆိုထားရန် လိုအပ်သော်လည်း၊ dy ကို solve ၏ ဒုတိယစာကြောင်း ရောက်မှသာ အဓိပ္ပာယ်ဖွင့်ဆိုထားခြင်းကြောင့် ဖြစ်သည်။

အခြားတစ်ဖက်တွင်မူ၊ ကျွန်ုပ်တို့၏ အဓိပ္ပာယ်ဖွင့်ဆိုချက်၏ ရည်ရွယ်ချက်သည် ကျိုးကြောင်းဆီလျော်မှု ရှိပါသည်။ အကြောင်းမှာ မူအားဖြင့် dy ကို မသိရှိဘဲ y စထရင်းကို စတင်ထုတ်လုပ်နိုင်သောကြောင့် ဖြစ်သည်။ အမှန်စင်စစ်၊ integral နှင့် အခြားသော စထရင်းလုပ်ဆောင်ချက်များစွာသည် အာဂူးမန့်များနှင့်ပတ်သက်သော အချက်အလက်အချို့သာ သိရသေးသော်လည်း အဖြေ၏ အစိတ်အပိုင်းတစ်ခုကို ထုတ်လုပ်နိုင်သည့် cons-stream ကဲ့သို့သော ဂုဏ်သတ္တိများ ရှိကြပါသည်။ integral အတွက်မူ၊ output စထရင်း၏ ပထမဆုံး အစိတ်အပိုင်းသည် သတ်မှတ်ထားသော initial-value ဖြစ်သည်။ ထို့ကြောင့် integrand dy ကို အကဲမဖြတ်ဘဲ output စထရင်း၏ ပထမဆုံးအစိတ်အပိုင်းကို ကျွန်ုပ်တို့ ထုတ်လုပ်နိုင်ပါသည်။ y ၏ ပထမဆုံးအစိတ်အပိုင်းကို သိရှိပြီးသည်နှင့် solve ၏ ဒုတိယစာကြောင်းရှိ stream-map သည် dy ၏ ပထမဆုံးအစိတ်အပိုင်းကို စတင်ထုတ်လုပ်နိုင်မည်ဖြစ်ပြီး၊ ၎င်းမှတစ်ဆင့် y ၏ နောက်ထပ်အစိတ်အပိုင်းကို ထုတ်လုပ်ပေးသွားမည် ဖြစ်ပါသည်။

ဤအချက်ကို အသုံးချနိုင်ရန်အတွက်၊ integrand စထရင်းကို နှောင့်နှေးအာဂူးမန့် (delayed argument) အဖြစ် လက်ခံရန် integral ကို ပြန်လည်အဓိပ္ပာယ်ဖွင့်ဆိုပါမည်။ Integral သည် output စထရင်း၏ ပထမဆုံးအစိတ်အပိုင်းထက် ပိုမိုထုတ်လုပ်ရန် လိုအပ်လာမှသာ integrand ကို force လုပ်၍ အကဲဖြတ်ပါလိမ့်မည် - 

(define (integral
         delayed-integrand initial-value dt)
  (define int
    (cons-stream 
     initial-value
     (let ((integrand 
            (force delayed-integrand)))
       (add-streams 
        (scale-stream integrand dt)
        int))))
  int)

ယခုအခါ y ၏ အဓိပ္ပာယ်ဖွင့်ဆိုချက်တွင် dy အကဲဖြတ်ခြင်းကို delay လုပ်ခြင်းဖြင့် ကျွန်ုပ်တို့၏ solve လုပ်ထုံးလုပ်နည်းကို အကောင်အထည်ဖော်နိုင်ပြီ ဖြစ်သည် -199 

(define (solve f y0 dt)
  (define y (integral (delay dy) y0 dt))
  (define dy (stream-map f y))
  y)

ယေဘုယျအားဖြင့်၊ integral ကို ခေါ်ဆိုသူတိုင်းသည် ယခုအခါ integrand အာဂူးမန့်ကို delay လုပ်ရမည် ဖြစ်သည်။ ကနဦးအခြေအနေ y ( 0 ) = 1 ရှိသော ဒစ်ဖရန်ရှယ်ညီမျှခြင်း d y / d t = y ၏ အဖြေကို y = 1 တန်ဖိုးတွင် တွက်ချက်ခြင်းဖြင့် e ≈ 2.718 ကို ခန့်မှန်းပြပြီး solve လုပ်ထုံးလုပ်နည်း အလုပ်လုပ်ပုံကို သက်သေပြနိုင်ပါသည် - 

(stream-ref 
 (solve (lambda (y) y) 1 0.001) 1000)
2.716924

လေ့ကျင့်ခန်း ၃.၇၇ - အထက်တွင် အသုံးပြုခဲ့သော integral လုပ်ထုံးလုပ်နည်းသည် အပိုင်း ၃.၅.၂ ရှိ ကိန်းပြည့်အနန္တစထရင်း (infinite stream of integers) ၏ "သွယ်ဝိုက်သော" အဓိပ္ပာယ်ဖွင့်ဆိုချက်နှင့် ဆင်တူပါသည်။ တစ်နည်းအားဖြင့်၊ (၃.၅.၂ တွင်ပင်ပါရှိသော) integers-starting-from နှင့် ပိုမိုဆင်တူသော integral အဓိပ္ပာယ်ဖွင့်ဆိုချက်ကိုလည်း အောက်ပါအတိုင်း ပေးနိုင်ပါသည် - 

(define (integral
         integrand initial-value dt)
  (cons-stream 
   initial-value
   (if (stream-null? integrand)
       the-empty-stream
       (integral 
        (stream-cdr integrand)
        (+ (* dt (stream-car integrand))
           initial-value)
        dt))))

loop များပါသော စနစ်များတွင် အသုံးပြုသောအခါ၊ ဤလုပ်ထုံးလုပ်နည်းသည် ကျွန်ုပ်တို့၏ မူလ integral ဗားရှင်းကဲ့သို့ပင် ပြဿနာရှိပါသည်။ integrand ကို delayed argument အဖြစ် လက်ခံရန်နှင့် အထက်ပါ solve လုပ်ထုံးလုပ်နည်းတွင် အသုံးပြုနိုင်ရန် ဤလုပ်ထုံးလုပ်နည်းကို ပြင်ဆင်ပါ။

လေ့ကျင့်ခန်း ၃.၇၈ - အောက်ပါ homogeneous second-order linear differential equation ကို လေ့လာရန် အချက်ပြမှုလုပ်ငန်းစဉ်စနစ်တစ်ခု ဒီဇိုင်းထုတ်သည့် ပြဿနာကို စဉ်းစားကြည့်ပါ - d 2 y d t 2 − a d y d t − b y = 0. y ကို ပုံဖော်ထားသော output စထရင်းကို loop တစ်ခုပါဝင်သည့် ကွန်ရက်တစ်ခုမှ ထုတ်လုပ်ပါသည်။ အကြောင်းမှာ d 2 y / d t 2 ၏ တန်ဖိုးသည် y နှင့် d y / d t တန်ဖိုးများအပေါ် မူတည်နေပြီး၊ ၎င်းတို့ နှစ်ခုလုံးကို d 2 y / d t 2 ကို အင်တီဂရေးရှင်း (integration) လုပ်ခြင်းဖြင့် ရရှိသောကြောင့် ဖြစ်သည်။ ကျွန်ုပ်တို့ ကုဒ်ရေးလိုသော ပုံချပ်ကို ပုံ ၃.၃၅ တွင် ပြထားပါသည်။ ကိန်းသေများ ဖြစ်ကြသော a ၊ b ၊ နှင့် d t နှင့် y နှင့် d y / d t အတွက် ကနဦးတန်ဖိုးများဖြစ်သော y 0 နှင့် d y 0 တို့ကို အာဂူးမန့်များအဖြစ် လက်ခံပြီး y ၏ ဆက်တိုက်တန်ဖိုးများပါဝင်သော စထရင်းကို ထုတ်လုပ်ပေးမည့် solve-2nd လုပ်ထုံးလုပ်နည်းကို ရေးသားပါ။

လေ့ကျင့်ခန်း ၃.၇၉ - ယေဘုယျ second-order differential equation များဖြစ်သော d 2 y / d t 2 = f ( d y / d t , y ) ကို ဖြေရှင်းနိုင်ရန် လေ့ကျင့်ခန်း ၃.၇၈ ပါ solve-2nd လုပ်ထုံးလုပ်နည်းကို ယေဘုယျပြုပြင်ပေးပါ။

လေ့ကျင့်ခန်း ၃.၈၀ - series RLC circuit တစ်ခုတွင် resistor တစ်ခု၊ capacitor တစ်ခုနှင့် inductor တစ်ခုတို့ကို ပုံ ၃.၃၆ တွင် ပြထားသည့်အတိုင်း series အလိုက် ချိတ်ဆက်ထားပါသည်။ အကယ်၍ R ၊ L ၊ နှင့် C တို့သည် resistance၊ inductance နှင့် capacitance တို့ဖြစ်ပါက၊ အစိတ်အပိုင်း သုံးခုအတွက် voltage ( v ) နှင့် current ( i ) တို့၏ ဆက်နွှယ်မှုကို အောက်ပါညီမျှခြင်းများဖြင့် ဖော်ပြနိုင်သည် - v R = i R R , v L = L d i L d t , i C = C d v C d t , နှင့် ပတ်လမ်းချိတ်ဆက်မှုများအရ ဆက်နွှယ်မှုများမှာ - i R = i L = − i C , v C = v L + v R . ဤညီမျှခြင်းများကို ပေါင်းစပ်လိုက်ပါက ပတ်လမ်း၏ အခြေအနေ (capacitor ပေါ်ရှိ voltage v C နှင့် inductor အတွင်းရှိ current i L တို့ဖြင့် ဖော်ပြသည်) ကို အောက်ပါ differential equation အစုံဖြင့် ဖော်ပြနိုင်ကြောင်း တွေ့ရပါမည် - d v C d t = − i L C , d i L d t = 1 L v C − R L i L . ဤ differential equation စနစ်ကို ကိုယ်စားပြုသော အချက်ပြစီးဆင်းမှုပုံချပ်ကို ပုံ ၃.၃၇ တွင် ပြထားပါသည်။

ပတ်လမ်း၏ ပါရာမီတာများဖြစ်သော R ၊ L ၊ နှင့် C နှင့် အချိန်တိုးမြင့်မှု (time increment) d t တို့ကို အာဂူးမန့်များအဖြစ် လက်ခံသော RLC လုပ်ထုံးလုပ်နည်းကို ရေးသားပါ။ လေ့ကျင့်ခန်း ၃.၇၃ ပါ RC လုပ်ထုံးလုပ်နည်းနှင့် ဆင်တူသော နည်းလမ်းဖြင့်၊ RLC သည် အခြေအနေပြောင်းလဲနိုင်သောကိန်း (state variables) များ၏ ကနဦးတန်ဖိုးများ ဖြစ်ကြသော v C 0 နှင့် i L 0 တို့ကို လက်ခံပြီး အခြေအနေစထရင်းများဖြစ်သော v C နှင့် i L အစုံကို (cons အသုံးပြု၍) ထုတ်လုပ်ပေးသည့် လုပ်ထုံးလုပ်နည်းတစ်ခုကို ထုတ်ပေးရပါမည်။ RLC ကို အသုံးပြု၍ R = 1 ohm၊ C = 0.2 farad၊ L = 1 henry၊ d t = 0.1 second နှင့် ကနဦးတန်ဖိုးများဖြစ်သော i L 0 = 0 amps နှင့် v C 0 = 10 volts ရှိသော series RLC circuit ၏ အပြုအမူကို ပုံဖော်သည့် စထရင်းအစုံကို ထုတ်လုပ်ပါ။

ပုံမှန်အစဉ်လိုက်အကဲဖြတ်ခြင်း (Normal-order evaluation)

ဤအပိုင်းရှိ ဥပမာများသည် delay နှင့် force ကို တိုက်ရိုက်အသုံးပြုခြင်းက ပရိုဂရမ်ရေးသားခြင်းတွင် မည်မျှပြောင်းလွယ်ပြင်လွယ်ရှိစေကြောင်း သရုပ်ဖော်ထားသော်လည်း၊ တစ်ချိန်တည်းမှာပင် ၎င်းသည် ကျွန်ုပ်တို့၏ ပရိုဂရမ်များကို မည်သို့ ပိုမိုရှုပ်ထွေးစေနိုင်ကြောင်းလည်း ပြသနေပါသည်။ ဥပမာအားဖြင့်၊ ကျွန်ုပ်တို့၏ integral လုပ်ထုံးလုပ်နည်းသစ်သည် loop များပါသော စနစ်များကို ပုံဖော်နိုင်စွမ်း ပေးသော်လည်း၊ integral ကို delayed integrand နှင့် ခေါ်ဆိုရမည်ဖြစ်ကြောင်းနှင့် integral ကို အသုံးပြုသည့် လုပ်ထုံးလုပ်နည်းတိုင်းက ဤအချက်ကို သိရှိထားရမည်ဖြစ်ကြောင်းကိုလည်း သတိပြုရပါမည်။ အမှန်စင်စစ်၊ ကျွန်ုပ်တို့သည် လုပ်ထုံးလုပ်နည်း အမျိုးအစား နှစ်မျိုးကို ဖန်တီးလိုက်သကဲ့သို့ ဖြစ်သွားပါသည် - သာမန်လုပ်ထုံးလုပ်နည်းများနှင့် delayed argument များကို လက်ခံသော လုပ်ထုံးလုပ်နည်းများ ဖြစ်ကြသည်။ ယေဘုယျအားဖြင့်၊ သီးခြားစီဖြစ်သော လုပ်ထုံးလုပ်နည်းအမျိုးအစားများကို ဖန်တီးခြင်းက ကျွန်ုပ်တို့ကို သီးခြားစီဖြစ်သော higher-order procedure များကိုလည်း ဖန်တီးရန် တွန်းအားပေးပါလိမ့်မည်။200

မတူညီသော လုပ်ထုံးလုပ်နည်းအမျိုးအစား နှစ်မျိုး လိုအပ်မှုကို ရှောင်ရှားနိုင်သည့် နည်းလမ်းတစ်ခုမှာ လုပ်ထုံးလုပ်နည်းအားလုံးကို delayed argument များ လက်ခံစေရန် ပြုလုပ်ခြင်း ဖြစ်သည်။ လုပ်ထုံးလုပ်နည်းများသို့ ပေးပို့သော အာဂူးမန့်အားလုံးကို အလိုအလျောက် delay လုပ်ထားပြီး၊ အမှန်တကယ် လိုအပ်သောအခါမှသာ (ဥပမာ - primitive operation တစ်ခုခုအတွက် လိုအပ်လာသောအခါ) ၎င်းတို့ကို force လုပ်သည့် evaluation model တစ်ခုကို ကျွန်ုပ်တို့ အသုံးပြုနိုင်ပါသည်။ ၎င်းသည် ကျွန်ုပ်တို့၏ ဘာသာစကားကို အပိုင်း ၁.၁.၅ တွင် evaluation အတွက် substitution model ကို မိတ်ဆက်စဉ်က ဖော်ပြခဲ့သော normal-order evaluation သို့ ပြောင်းလဲပေးပါလိမ့်မည်။ Normal-order evaluation သို့ ပြောင်းလဲခြင်းသည် delayed evaluation အသုံးပြုမှုကို ရိုးရှင်းစေရန် တစ်ပြေးညီဖြစ်ပြီး သေသပ်သော နည်းလမ်းကို ပေးစွမ်းနိုင်ပြီး၊ ကျွန်ုပ်တို့သည် stream processing ကိုသာ အဓိကထားမည်ဆိုပါက ၎င်းသည် အသုံးပြုသင့်သော သဘာဝကျသည့် နည်းဗျူဟာတစ်ခု ဖြစ်ပါလိမ့်မည်။ အပိုင်း ၄.၂ တွင် evaluator အကြောင်း လေ့လာပြီးနောက်၊ ကျွန်ုပ်တို့၏ ဘာသာစကားကို ဤနည်းအတိုင်း မည်သို့ပြောင်းလဲရမည်ကို တွေ့ရပါမည်။ ကံမကောင်းစွာဖြင့်၊ လုပ်ထုံးလုပ်နည်းခေါ်ဆိုမှုများတွင် delay များ ထည့်သွင်းခြင်းသည် အဖြစ်အပျက်များ၏ အစီအစဉ်အပေါ် မူတည်နေသော ပရိုဂရမ်များ (ဥပမာ - assignment အသုံးပြုသော၊ data ပြောင်းလဲသော (mutate) သို့မဟုတ် input/output လုပ်ဆောင်သော ပရိုဂရမ်များ) ကို ဒီဇိုင်းထုတ်နိုင်စွမ်းအား များစွာ ထိခိုက်စေပါသည်။ လေ့ကျင့်ခန်း ၃.၅၁ နှင့် လေ့ကျင့်ခန်း ၃.၅၂ တို့တွင် သရုပ်ဖော်ထားသည့်အတိုင်း cons-stream ထဲမှ delay တစ်ခုတည်းကပင် ကြီးမားသော ရှုပ်ထွေးမှုများကို ဖြစ်ပေါ်စေနိုင်ပါသည်။ သိသလောက်မူ၊ mutability နှင့် delayed evaluation တို့သည် ပရိုဂရမ်မင်းဘာသာစကားများတွင် ကောင်းမွန်စွာ ရောနှော၍မရဘဲ၊ ၎င်းတို့နှစ်ခုလုံးကို တစ်ပြိုင်နက် ကိုင်တွယ်နိုင်မည့် နည်းလမ်းများ ရှာဖွေဖော်ထုတ်ခြင်းသည် လက်ရှိတွင် သုတေသနပြုနေဆဲ နယ်ပယ်တစ်ခု ဖြစ်ပါသည်။

3.5.5Modularity of Functional Programs and Modularity of Objects

3.1.2 တွင် ကျွန်ုပ်တို့ မြင်တွေ့ခဲ့ရသည့်အတိုင်း assignment ကို စတင်အသုံးပြုခြင်း၏ အဓိက အကျိုးကျေးဇူးတစ်ခုမှာ စနစ်ကြီးတစ်ခု၏ state အစိတ်အပိုင်းများကို local variables များအတွင်း ထည့်သွင်းချုပ်ကိုင်ခြင်း (encapsulating) သို့မဟုတ် "ဖုံးကွယ်ခြင်း" အားဖြင့် ကျွန်ုပ်တို့၏ စနစ်များ၏ modularity ကို တိုးမြှင့်နိုင်ခြင်း ဖြစ်သည်။ Stream မော်ဒယ်များသည် assignment ကို အသုံးမပြုဘဲ တူညီသော modularity ကို ပေးစွမ်းနိုင်သည်။ ဥပမာတစ်ခုအနေဖြင့်၊ 3.1.2 တွင် ကျွန်ုပ်တို့ လေ့လာခဲ့သော π ၏ Monte Carlo ခန့်မှန်းချက်ကို stream-processing ရှုထောင့်မှ ပြန်လည်အကောင်အထည်ဖော်ကြည့်ပါမည်။

အဓိက modularity ပြဿနာမှာ random-number generator တစ်ခု၏ အတွင်းပိုင်း state ကို random numbers များ အသုံးပြုသည့် ပရိုဂရမ်များမှ ဖုံးကွယ်ထားလိုခြင်း ဖြစ်သည်။ ကျွန်ုပ်တို့သည် rand-update ဟူသော procedure ဖြင့် စတင်ခဲ့ပြီး၊ ၎င်း၏ ဆက်တိုက်ထွက်ပေါ်လာသော တန်ဖိုးများသည် ကျွန်ုပ်တို့အတွက် လိုအပ်သော random numbers များဖြစ်လာကာ ၎င်းကို အသုံးပြု၍ random-number generator တစ်ခုကို ဖန်တီးခဲ့သည်- 

(define rand
  (let ((x random-init))
    (lambda ()
      (set! x (rand-update x))
      x)))

Stream ပုံစံဖော်ဆောင်မှုတွင် သီးခြား random-number generator ဟူ၍ မရှိဘဲ၊ rand-update ကို ဆက်တိုက် ခေါ်ယူခြင်းဖြင့် ထွက်ပေါ်လာသော random numbers stream တစ်ခုသာ ရှိသည်- 

(define random-numbers
  (cons-stream random-init
               (stream-map rand-update 
                           random-numbers)))

ကျွန်ုပ်တို့သည် ဤအရာကို အသုံးပြု၍ random-numbers stream ရှိ ဆက်တိုက်ဖြစ်ပေါ်သော အတွဲများအပေါ်တွင် လုပ်ဆောင်ထားသည့် Cesàro experiment ၏ ရလဒ်များ stream ကို တည်ဆောက်သည်- 

(define cesaro-stream
  (map-successive-pairs
   (lambda (r1 r2) (= (gcd r1 r2) 1))
   random-numbers))

(define (map-successive-pairs f s)
  (cons-stream
   (f (stream-car s) 
      (stream-car (stream-cdr s)))
   (map-successive-pairs 
    f (stream-cdr (stream-cdr s)))))

ယခုအခါ cesaro-stream ကို monte-carlo procedure ထဲသို့ ထည့်သွင်းပေးလိုက်ပြီး၊ ၎င်းသည် ဖြစ်တန်စွမ်း ခန့်မှန်းချက်များ stream တစ်ခုကို ထုတ်ပေးသည်။ ထို့နောက် ထိုရလဒ်များကို π ၏ ခန့်မှန်းချက်များ stream တစ်ခုအဖြစ် ပြောင်းလဲလိုက်သည်။ ပရိုဂရမ်၏ ဤဗားရှင်းတွင် စမ်းသပ်မှု မည်မျှပြုလုပ်ရမည်ကို ပြောပြရန် parameter မလိုအပ်ပါ။ ပိုမိုကောင်းမွန်သော π ခန့်မှန်းချက်များကို (စမ်းသပ်မှုများ ပိုမိုပြုလုပ်ခြင်းဖြင့်) pi stream ထဲသို့ ပိုမိုနက်ရှိုင်းစွာ ကြည့်ရှုခြင်းဖြင့် ရရှိနိုင်သည်-

                     failed)
  (define (next passed failed)
    (cons-stream
     (/ passed (+ passed failed))
     (monte-carlo
      (stream-cdr experiment-stream) 
      passed 
      failed)))
  (if (stream-car experiment-stream)
      (next (+ passed 1) failed)
      (next passed (+ failed 1))))

(define pi
  (stream-map
   (lambda (p) (sqrt (/ 6 p)))
   (monte-carlo cesaro-stream 0 0)))

ဤနည်းလမ်းတွင် သိသာထင်ရှားသော modularity ရှိသည်၊ အကြောင်းမှာ ကျွန်ုပ်တို့သည် မည်သည့်စမ်းသပ်မှုမျိုးကိုမဆို ကိုင်တွယ်နိုင်သည့် ယေဘုယျ monte-carlo procedure တစ်ခုကို ရေးဆွဲနိုင်သေးသောကြောင့် ဖြစ်သည်။ သို့သော်လည်း ဤနေရာတွင် assignment သို့မဟုတ် local state မရှိပါ။

လေ့ကျင့်ခန်း ၃.၈၁: လေ့ကျင့်ခန်း ၃.၆ တွင် ထပ်ခါတလဲလဲ ဖြစ်ပေါ်စေနိုင်သော "random" numbers စဉ်ဆက်များကို ထုတ်ပေးနိုင်ရန် random-number sequence ကို reset လုပ်ခွင့်ပြုသည့် random-number generator အကြောင်း ဆွေးနွေးခဲ့သည်။ Random number အသစ်တစ်ခု generate လုပ်ရန် သို့မဟုတ် sequence ကို သတ်မှတ်ထားသော တန်ဖိုးတစ်ခုသို့ reset လုပ်ရန် တောင်းဆိုချက်များ (requests) input stream အပေါ်တွင် အလုပ်လုပ်ပြီး လိုချင်သော random numbers stream ကို ထုတ်ပေးသည့် ဤ generator ၏ stream ပုံစံဖော်ဆောင်မှုတစ်ခုကို ပြုလုပ်ပါ။ သင်၏ အဖြေတွင် assignment ကို အသုံးမပြုပါနှင့်။

လေ့ကျင့်ခန်း ၃.၈၂: Monte Carlo integration ဆိုင်ရာ လေ့ကျင့်ခန်း ၃.၅ ကို streams များဖြင့် ပြန်လည်ပြုလုပ်ပါ။ estimate-integral ၏ stream ဗားရှင်းတွင် စမ်းသပ်မှု မည်မျှပြုလုပ်ရမည်ကို ပြောပြသော argument ရှိမည်မဟုတ်ပါ။ ယင်းအစား၊ ၎င်းသည် ဆက်တိုက် ပိုမိုပြုလုပ်လာသော စမ်းသပ်မှုများအပေါ် အခြေခံသည့် ခန့်မှန်းချက်များ stream တစ်ခုကို ထုတ်ပေးမည် ဖြစ်သည်။

အချိန်၏ functional-programming အမြင်

ဤအခန်း၏ အစတွင် ဖော်ပြခဲ့သော objects နှင့် state ဆိုင်ရာ ကိစ္စရပ်များကို ပြန်လည်လေ့လာပြီး ရှုထောင့်အသစ်ဖြင့် ဆန်းစစ်ကြည့်ကြပါစို့။ ကျွန်ုပ်တို့သည် state ရှိသော စနစ်များကို မော်ဒယ်ပြုလုပ်သည့် ပရိုဂရမ်များကို modular နည်းလမ်းဖြင့် တည်ဆောက်နိုင်ရန် assignment နှင့် mutable objects တို့ကို ယန္တရားတစ်ခုအဖြစ် မိတ်ဆက်ခဲ့သည်။ ကျွန်ုပ်တို့သည် local state variables များပါဝင်သော computational objects များကို တည်ဆောက်ခဲ့ပြီး ထို variables များကို ပြောင်းလဲရန် assignment ကို အသုံးပြုခဲ့သည်။ ကမ္ဘာလောကရှိ objects များ၏ အချိန်နှင့်အမျှ ပြောင်းလဲသော အပြုအမူကို ၎င်းတို့နှင့် သက်ဆိုင်သော computational objects များ၏ အချိန်နှင့်အမျှ ပြောင်းလဲသော အပြုအမူဖြင့် မော်ဒယ်ပြုလုပ်ခဲ့သည်။

ယခုအခါ streams များသည် local state ရှိသော objects များကို မော်ဒယ်ပြုလုပ်ရန် အခြားရွေးချယ်စရာ နည်းလမ်းတစ်ခု ပေးစွမ်းနိုင်သည်ကို ကျွန်ုပ်တို့ တွေ့မြင်ခဲ့ရပြီ။ Object တစ်ခုခု၏ local state ကဲ့သို့သော ပြောင်းလဲနေသော ပမာဏတစ်ခုကို ဆက်တိုက်ဖြစ်ပေါ်နေသော state များ၏ အချိန်ဆိုင်ရာ သမိုင်းကြောင်းကို ကိုယ်စားပြုသည့် stream တစ်ခုကို အသုံးပြု၍ မော်ဒယ်ပြုလုပ်နိုင်သည်။ အနှစ်သာရအားဖြင့်၊ ကျွန်ုပ်တို့သည် streams များကို အသုံးပြု၍ အချိန်ကို ပြတ်ပြတ်သားသား (explicitly) ဖော်ပြသည်၊ ထို့ကြောင့် ကျွန်ုပ်တို့၏ simulated ကမ္ဘာမှ အချိန်ကို evaluation ပြုလုပ်စဉ်အတွင်း ဖြစ်ပျက်သော အဖြစ်အပျက်များ အစဉ်လိုက်နှင့် သီးခြားခွဲထုတ်လိုက်သည်။ အမှန်စင်စစ်အားဖြင့်၊ delay ရှိနေခြင်းကြောင့် မော်ဒယ်ရှိ simulated အချိန်နှင့် evaluation ပြုလုပ်စဉ်အတွင်း ဖြစ်ပျက်သော အဖြစ်အပျက်များ၏ အစီအစဉ်ကြားတွင် ဆက်စပ်မှု အနည်းငယ်သာ ရှိပေလိမ့်မည်။

မော်ဒယ်ပြုလုပ်ခြင်းဆိုင်ရာ ဤနည်းလမ်းနှစ်ခုကို နှိုင်းယှဉ်ရန်အတွက်၊ ဘဏ်အကောင့်တစ်ခုရှိ လက်ကျန်ငွေကို စောင့်ကြည့်စစ်ဆေးသော "withdrawal processor" တစ်ခု၏ အကောင်အထည်ဖော်မှုကို ပြန်လည်စဉ်းစားကြည့်ကြပါစို့။ 3.1.3 တွင် ကျွန်ုပ်တို့သည် ထိုသို့သော processor ၏ ရိုးရှင်းသော ဗားရှင်းတစ်ခုကို အကောင်အထည်ဖော်ခဲ့သည်- 

(define (make-simplified-withdraw balance)
  (lambda (amount)
    (set! balance (- balance amount))
    balance))

make-simplified-withdraw ကို ခေါ်ယူခြင်းဖြင့် computational objects များကို ထုတ်ပေးပြီး၊ တစ်ခုစီတွင် ၎င်းကို ဆက်တိုက်ခေါ်ယူမှုများကြောင့် လျော့နည်းသွားသော balance ဟူသော local state variable တစ်ခုစီ ပါရှိသည်။ ထို object သည် amount ကို argument အဖြစ် လက်ခံပြီး လက်ကျန်ငွေအသစ်ကို ပြန်ပေးသည်။ ဘဏ်အကောင့် အသုံးပြုသူတစ်ဦးသည် ထိုသို့သော object ထဲသို့ input များကို ဆက်တိုက် ရိုက်ထည့်နေပြီး display screen ပေါ်တွင် ပြသနေသော ပြန်လာသည့် တန်ဖိုးများကို ကြည့်ရှုနေသည်ဟု ကျွန်ုပ်တို့ စိတ်ကူးကြည့်နိုင်သည်။

သို့မဟုတ်ပါက၊ withdrawal processor ကို input အဖြစ် လက်ကျန်ငွေတစ်ခုနှင့် ထုတ်ယူမည့် ပမာဏများ stream တစ်ခုကို လက်ခံပြီး အကောင့်ရှိ ဆက်တိုက်ဖြစ်ပေါ်သော လက်ကျန်ငွေများ stream ကို ထုတ်ပေးသည့် procedure တစ်ခုအဖြစ် မော်ဒယ်ပြုလုပ်နိုင်သည်- 

(define (stream-withdraw balance amount-stream)
  (cons-stream
   balance
   (stream-withdraw 
    (- balance (stream-car amount-stream))
    (stream-cdr amount-stream))))

Stream-withdraw သည် ၎င်း၏ input အပေါ်တွင်သာ လုံးလုံးလျားလျား မူတည်သော သတ်မှတ်ချက် ပြည့်စုံသည့် သင်္ချာ function တစ်ခုကို အကောင်အထည်ဖော်ခြင်း ဖြစ်သည်။ သို့သော်၊ input amount-stream သည် အသုံးပြုသူ ရိုက်ထည့်လိုက်သော ဆက်တိုက်တန်ဖိုးများ stream ဖြစ်ပြီး ရလဒ်အဖြစ် ထွက်ပေါ်လာသော လက်ကျန်ငွေများ stream ကို ပြသသည်ဟု ဆိုပါစို့။ ထိုအခါ၊ တန်ဖိုးများကို ရိုက်ထည့်ပြီး ရလဒ်များကို ကြည့်ရှုနေသော အသုံးပြုသူ၏ ရှုထောင့်မှကြည့်လျှင် stream process သည် make-simplified-withdraw ဖြင့် ဖန်တီးထားသော object နှင့် တူညီသော အပြုအမူရှိသည်။ သို့သော်၊ stream ဗားရှင်းတွင် assignment မရှိ၊ local state variable မရှိသည့်အတွက် 3.1.3 တွင် ကျွန်ုပ်တို့ ကြုံတွေ့ခဲ့ရသော သီအိုရီဆိုင်ရာ အခက်အခဲများလည်း မရှိပါ။ သို့သော် စနစ်တွင် state ရှိနေဆဲပင်!

ဒါက တကယ်ကို မှတ်သားစရာပါ။ stream-withdraw သည် အပြုအမူ မပြောင်းလဲသော သတ်မှတ်ချက် ပြည့်စုံသည့် သင်္ချာ function တစ်ခုကို အကောင်အထည်ဖော်ထားသော်လည်း၊ ဤနေရာတွင် အသုံးပြုသူ၏ ခံစားသိရှိမှုမှာ ပြောင်းလဲနေသော state ရှိသည့် စနစ်တစ်ခုနှင့် ဆက်သွယ်နေရသကဲ့သို့ ဖြစ်သည်။ ဤ ဝိရောဓိ (paradox) ကို ဖြေရှင်းရန် နည်းလမ်းတစ်ခုမှာ စနစ်အပေါ်တွင် state ရှိနေသည်ဟု ဖြစ်ပေါ်စေသည်မှာ အသုံးပြုသူ၏ အချိန်နှင့်အမျှ တည်ရှိနေမှုကြောင့် ဖြစ်သည်ကို သဘောပေါက်ရန် ဖြစ်သည်။ အကယ်၍ အသုံးပြုသူသည် တစ်ကြိမ်ချင်းစီ လုပ်ဆောင်မှုများထက် လက်ကျန်ငွေများ stream အနေဖြင့် ပြန်လည်စဉ်းစားနိုင်မည်ဆိုလျှင်၊ စနစ်သည် stateless ဖြစ်နေသည်ဟု မြင်တွေ့ရမည်ဖြစ်သည်။201

ရှုပ်ထွေးသော ဖြစ်စဉ်တစ်ခု၏ အစိတ်အပိုင်းတစ်ခု၏ ရှုထောင့်မှကြည့်လျှင် အခြားအစိတ်အပိုင်းများသည် အချိန်နှင့်အမျှ ပြောင်းလဲနေသကဲ့သို့ ထင်ရသည်။ ၎င်းတို့တွင် ဖုံးကွယ်ထားသော အချိန်နှင့်အမျှ ပြောင်းလဲနေသည့် local state ရှိသည်။ ကျွန်ုပ်တို့၏ ကမ္ဘာရှိ သဘာဝအလျောက် ကွဲပြားမှုမျိုးကို (ထိုကမ္ဘာ၏ အစိတ်အပိုင်းတစ်ခုအဖြစ် ကျွန်ုပ်တို့၏ ရှုထောင့်မှ မြင်ရသည့်အတိုင်း) ကျွန်ုပ်တို့၏ ကွန်ပျူတာရှိ တည်ဆောက်ပုံများဖြင့် မော်ဒယ်ပြုလုပ်သည့် ပရိုဂရမ်များ ရေးသားလိုပါက၊ ကျွန်ုပ်တို့သည် functional မဟုတ်သော computational objects များကို တည်ဆောက်ရမည် — ၎င်းတို့သည် အချိန်နှင့်အမျှ ပြောင်းလဲရမည်ဖြစ်သည်။ ကျွန်ုပ်တို့သည် state ကို local state variables များဖြင့် မော်ဒယ်ပြုလုပ်ပြီး၊ state ပြောင်းလဲမှုများကို ထို variables များသို့ assignment များ ပြုလုပ်ခြင်းဖြင့် မော်ဒယ်ပြုလုပ်သည်။ ဤသို့ ပြုလုပ်ခြင်းဖြင့် ကျွန်ုပ်တို့သည် computation တစ်ခု၏ လုပ်ဆောင်ချိန်ကို ကျွန်ုပ်တို့ ပါဝင်နေသော ကမ္ဘာရှိ အချိန်နှင့် မော်ဒယ်ပြုလုပ်လိုက်ပြီး၊ ထို့ကြောင့် ကျွန်ုပ်တို့၏ ကွန်ပျူတာတွင် "objects" များကို ရရှိလာသည်။

Objects များဖြင့် မော်ဒယ်ပြုလုပ်ခြင်းသည် စွမ်းအားထက်မြက်ပြီး အလိုလို သိမြင်လွယ်သည်၊ အကြောင်းမှာ ၎င်းသည် ကျွန်ုပ်တို့ ပါဝင်နေသော ကမ္ဘာနှင့် ဆက်သွယ်နေရသည်ဟူသော ခံစားသိရှိမှုနှင့် ကိုက်ညီသောကြောင့် ဖြစ်သည်။ သို့သော်လည်း ဤအခန်းတစ်လျှောက်လုံး ကျွန်ုပ်တို့ အကြိမ်ကြိမ် တွေ့မြင်ခဲ့ရသည့်အတိုင်း၊ ဤမော်ဒယ်များသည် အဖြစ်အပျက်များ၏ အစီအစဉ်ကို ထိန်းချုပ်ရန်နှင့် process များစွာကို တစ်ပြိုင်နက် လုပ်ဆောင်ရန် (synchronizing) ဆိုင်ရာ ခက်ခဲသော ပြဿနာများကို ဖြစ်ပေါ်စေသည်။ ဤပြဿနာများကို ရှောင်ရှားနိုင်ခြေရှိမှုသည် assignment သို့မဟုတ် mutable data များ မပါဝင်သော functional programming languages များ ဖွံ့ဖြိုးတိုးတက်လာစေရန် လှုံ့ဆော်ပေးခဲ့သည်။ ထိုကဲ့သို့သော ဘာသာစကားတွင် procedures အားလုံးသည် ၎င်းတို့၏ arguments များအပေါ်တွင် အခြေခံ၍ အပြုအမူ မပြောင်းလဲသော သတ်မှတ်ချက် ပြည့်စုံသည့် သင်္ချာ functions များကို အကောင်အထည်ဖော်ကြသည်။ Functional ချဉ်းကပ်မှုသည် concurrent စနစ်များကို ကိုင်တွယ်ရန်အတွက် အလွန်ဆွဲဆောင်မှုရှိသည်။202

အခြားတစ်ဖက်တွင်မူ၊ ကျွန်ုပ်တို့ အနီးကပ် လေ့လာကြည့်ပါက functional မော်ဒယ်များတွင်လည်း အချိန်နှင့် ဆက်စပ်သော ပြဿနာများ တဖြည်းဖြည်း ရောက်ရှိလာသည်ကို တွေ့နိုင်သည်။ အထူးသဖြင့် အပြန်အလှန် ဆက်သွယ်သော စနစ်များ (interactive systems)၊ အထူးသဖြင့် သီးခြား entities များကြား ဆက်သွယ်မှုများကို မော်ဒယ်ပြုလုပ်သော စနစ်များကို ဒီဇိုင်းထုတ်လိုသည့်အခါ အခက်တွေ့စေသော အပိုင်းတစ်ခု ဖြစ်ပေါ်လာသည်။ ဥပမာအားဖြင့်၊ ပူးတွဲဘဏ်အကောင့်များ ခွင့်ပြုထားသော ဘဏ်လုပ်ငန်းစနစ်တစ်ခု၏ အကောင်အထည်ဖော်မှုကို နောက်တစ်ကြိမ် စဉ်းစားကြည့်ပါ။ Assignment နှင့် objects များကို အသုံးပြုသည့် သမရိုးကျ စနစ်တစ်ခုတွင်၊ Peter နှင့် Paul တို့ အကောင့်တစ်ခုတည်းကို မျှဝေသုံးစွဲကြသည်ဟူသော အချက်ကို 3.1.3 တွင် ကျွန်ုပ်တို့ မြင်တွေ့ခဲ့ရသည့်အတိုင်း Peter နှင့် Paul နှစ်ဦးစလုံးက ၎င်းတို့၏ လုပ်ဆောင်မှု တောင်းဆိုချက်များကို ဘဏ်အကောင့် object တစ်ခုတည်းသို့ ပေးပို့ခြင်းဖြင့် မော်ဒယ်ပြုလုပ်မည်ဖြစ်သည်။ သီးခြား "objects" များ မရှိသော stream ရှုထောင့်မှကြည့်လျှင် ဘဏ်အကောင့်တစ်ခုကို လုပ်ဆောင်မှု တောင်းဆိုချက်များ stream တစ်ခုအပေါ် လုပ်ဆောင်ပြီး တုံ့ပြန်မှုများ stream တစ်ခု ထုတ်ပေးသည့် process တစ်ခုအဖြစ် မော်ဒယ်ပြုလုပ်နိုင်ကြောင်း ကျွန်ုပ်တို့ ဖော်ပြခဲ့ပြီးဖြစ်သည်။ ထို့ကြောင့် Peter နှင့် Paul တို့တွင် ပူးတွဲဘဏ်အကောင့်တစ်ခု ရှိသည်ဟူသော အချက်ကို Figure 3.38 တွင် ပြထားသည့်အတိုင်း Peter ၏ လုပ်ဆောင်မှု တောင်းဆိုချက်များ stream နှင့် Paul ၏ တောင်းဆိုချက်များ stream တို့ကို ပေါင်းစည်း (merging) ပြီး ရလဒ်ကို ဘဏ်အကောင့် stream process ထဲသို့ ထည့်သွင်းပေးခြင်းဖြင့် မော်ဒယ်ပြုလုပ်နိုင်သည်။

ဤပုံစံဖော်ဆောင်မှု၏ ပြဿနာမှာ merge ဟူသော အယူအဆတွင် ရှိသည်။ Stream နှစ်ခုကို ပေါင်းစည်းရာတွင် Peter ဆီမှ တောင်းဆိုချက်တစ်ခုနှင့် Paul ဆီမှ တောင်းဆိုချက်တစ်ခုကို အလှည့်ကျ ယူခြင်းဖြင့် မရနိုင်ပါ။ Paul သည် အကောင့်ကို အလွန်ရံဖန်ရံခါမှသာ အသုံးပြုသည်ဟု ဆိုပါစို့။ Peter အနေဖြင့် ဒုတိယအကြိမ် လုပ်ဆောင်မှု မပြုလုပ်မီ Paul အကောင့်အသုံးပြုသည်အထိ စောင့်ခိုင်းရန် မဖြစ်နိုင်ပါ။ ထိုသို့သော merge ကို မည်သို့ပင် အကောင်အထည်ဖော်စေကာမူ၊ ၎င်းသည် လုပ်ဆောင်မှု stream နှစ်ခုကို Peter နှင့် Paul တို့ ခံစားသိရှိရသော "real time" ၏ ကန့်သတ်ချက်များအတွင်း တစ်ခုနှင့်တစ်ခု ကြားညှပ်၍ ပေါင်းစပ်ပေးရမည်ဖြစ်သည်။ ဆိုလိုသည်မှာ Peter နှင့် Paul တို့ တွေ့ဆုံသောအခါ၊ အချို့သော လုပ်ဆောင်မှုများသည် မတွေ့ဆုံမီက လုပ်ဆောင်ခဲ့ပြီး ဖြစ်ကြောင်းနှင့် အချို့မှာ တွေ့ဆုံပြီးမှ လုပ်ဆောင်ခဲ့ခြင်း ဖြစ်ကြောင်း ၎င်းတို့ သဘောတူညီနိုင်ရမည် ဖြစ်သည်။203 ဤအရာသည် 3.4.1 တွင် ကျွန်ုပ်တို့ ကိုင်တွယ်ခဲ့ရသော တူညီသည့် ကန့်သတ်ချက်ပင် ဖြစ်သည်၊ ထိုစဉ်က state ရှိသော objects များကို concurrent processing လုပ်ရာတွင် အဖြစ်အပျက်များ၏ အစီအစဉ် "မှန်ကန်" စေရန်အတွက် ပြတ်သားသော synchronization ကို မိတ်ဆက်ရန် လိုအပ်ကြောင်း ကျွန်ုပ်တို့ တွေ့ရှိခဲ့ရသည်။ ထို့ကြောင့်၊ functional style ကို အားပေးရန် ကြိုးပမ်းရာတွင်၊ မတူညီသော agents များထံမှ input များကို ပေါင်းစည်းရန် လိုအပ်ချက်သည် functional style က ဖယ်ရှားပစ်ရန် ရည်ရွယ်ခဲ့သော ပြဿနာများကိုပင် ပြန်လည်ရောက်ရှိစေသည်။

ကျွန်ုပ်တို့သည် ဤအခန်းကို ကျွန်ုပ်တို့ မော်ဒယ်ပြုလုပ်ရန် ကြိုးစားနေသော လက်တွေ့ကမ္ဘာ၏ တည်ဆောက်ပုံနှင့် ကိုက်ညီသည့် computational models များ တည်ဆောက်ရန် ရည်ရွယ်ချက်ဖြင့် စတင်ခဲ့သည်။ ကျွန်ုပ်တို့သည် ကမ္ဘာလောကကို state ရှိပြီး အချိန်နှင့်အမျှ တစ်ခုနှင့်တစ်ခု ဆက်သွယ်နေသော သီးခြား objects များ အစုအဝေးအဖြစ် မော်ဒယ်ပြုလုပ်နိုင်သည်၊ သို့မဟုတ်ပါက ကမ္ဘာလောကကို အချိန်မရှိ၊ state မရှိသော တစ်ခုတည်းသော အရာအဖြစ် မော်ဒယ်ပြုလုပ်နိုင်သည်။ ရှုထောင့်တစ်ခုစီတွင် အားသာချက်များစွာ ရှိသော်လည်း၊ မည်သည့်ရှုထောင့်တစ်ခုတည်းကမျှ လုံးဝပြည့်စုံမှု မရှိသေးပါ။ ကြီးကျယ်ခမ်းနားသော ပေါင်းစည်းမှုတစ်ခုသည် ပေါ်ထွက်လာရန် ကျန်ရှိနေသေးသည်။204

Hardy ၏ စကားကို ကိုးကားရလျှင်- “ ‘အပြုသဘောဆောင်သော ကိန်းပြည့်တိုင်းသည် သူ၏ သူငယ်ချင်းများထဲမှ တစ်ဦးဖြစ်သည်’ ဟု မှတ်ချက်ပြုခဲ့သူမှာ Mr. Littlewood (ကျွန်ုပ်ထင်မြင်ချက်အရ) ဖြစ်သည်။ သူ Putney တွင် ဖျားနာနေစဉ် တစ်ခါက သွားရောက်တွေ့ဆုံခဲ့သည်ကို ကျွန်ုပ်မှတ်မိသည်။ ကျွန်ုပ်သည် တက္ကစီနံပါတ် ၁၇၂၉ ဖြင့် သွားခဲ့ပြီး ထိုနံပါတ်မှာ ပျင်းစရာကောင်းသည်ဟု ထင်ရကြောင်းနှင့် ၎င်းမှာ မကောင်းသော နိမိတ်မဟုတ်ရန် မျှော်လင့်ကြောင်း ပြောကြားခဲ့သည်။ ‘မဟုတ်ပါဘူး’ ဟု သူက ပြန်လည်ဖြေကြားခဲ့သည်၊ ‘ဒါဟာ အရမ်းစိတ်ဝင်စားဖို့ကောင်းတဲ့ နံပါတ်တစ်ခုပါ၊ ဒါဟာ နည်းလမ်းနှစ်မျိုးနဲ့ ကုဗကိန်း (cube) နှစ်ခုရဲ့ ပေါင်းလဒ်အဖြစ် ဖော်ပြနိုင်တဲ့ အငယ်ဆုံးကိန်းပါပဲ။’ ” Ramanujan ကိန်းများကို ထုတ်လုပ်ရန် weighted pairs များကို အသုံးပြုသည့် နည်းလမ်းကို Charles Leiserson က ကျွန်ုပ်တို့အား ပြသခဲ့သည်။

199 ဤ procedure သည် Scheme implementation အားလုံးတွင် အလုပ်လုပ်ရန် အာမမခံသော်လည်း၊ မည်သည့် implementation အတွက်မဆို အလုပ်လုပ်နိုင်မည့် ရိုးရှင်းသော ကွဲပြားမှုတစ်ခု ရှိပါသည်။ အဆိုပါ ပြဿနာမှာ Scheme implementation များက internal definition များကို ကိုင်တွယ်ပုံချင်း မသိမသာ ကွဲပြားမှုများနှင့် သက်ဆိုင်သည်။ (ကြည့်ရန် 4.1.6။)

200 ဤသည်မှာ Pascal ကဲ့သို့သော သာမန် strongly typed ဘာသာစကားများတွင် higher-order procedure များကို ကိုင်တွယ်ရာတွင် ကြုံတွေ့ရသည့် အခက်အခဲများကို Lisp တွင် အနည်းငယ် ထင်ဟပ်ဖော်ပြနေခြင်းဖြစ်သည်။ ထိုကဲ့သို့သော ဘာသာစကားများတွင် ပရိုဂရမ်မာသည် procedure တစ်ခုစီ၏ argument များနှင့် ရလဒ်များအတွက် data type (ဥပမာ- ကိန်းဂဏန်း၊ logical တန်ဖိုး၊ sequence စသည်ဖြင့်) ကို သတ်မှတ်ပေးရမည်ဖြစ်သည်။ ထို့ကြောင့် “ပေးထားသော proc procedure တစ်ခုကို sequence တစ်ခုအတွင်းရှိ element အားလုံးအပေါ် map လုပ်ခြင်း” ကဲ့သို့သော abstraction ကို stream-map ကဲ့သို့သော single higher-order procedure တစ်ခုတည်းဖြင့် ဖော်ပြနိုင်မည်မဟုတ်ပါ။ ယင်းအစား proc အတွက် သတ်မှတ်နိုင်သည့် argument နှင့် ရလဒ် data type ပေါင်းစပ်မှု တစ်ခုစီအတွက် ကွဲပြားသော mapping procedure တစ်ခု လိုအပ်မည်ဖြစ်သည်။ higher-order procedure များ ရှိနေချိန်တွင် လက်တွေ့ကျသော “data type” အယူအဆကို ထိန်းသိမ်းခြင်းသည် ခက်ခဲသော ပြဿနာများစွာကို ဖြစ်ပေါ်စေသည်။ ဤပြဿနာကို ကိုင်တွယ်ဖြေရှင်းနည်း တစ်မျိုးကို ML ဘာသာစကား (Gordon et al. 1979) တွင် တွေ့နိုင်ပြီး၊ ၎င်း၏ “polymorphic data types” များတွင် data type များအကြား higher-order transformation များအတွက် template များ ပါဝင်သည်။ ထို့အပြင် ML ရှိ procedure အများစုအတွက် data type များကို ပရိုဂရမ်မာက တိတိကျကျ ကြေညာရန် မလိုပါ။ ယင်းအစား ML တွင် type-inferencing ယန္တရား ပါဝင်ပြီး ၎င်းသည် environment ရှိ အချက်အလက်များကို အသုံးပြု၍ အသစ်သတ်မှတ်ထားသော procedure များအတွက် data type များကို ခန့်မှန်းတွက်ချက်ပေးသည်။

201 ထို့အတူပင် ရုပ်ပိုင်းဆိုင်ရာတွင် ရွေ့လျားနေသော အမှုန်တစ်ခုကို ကျွန်ုပ်တို့ လေ့လာသောအခါ၊ ထိုအမှုန်၏ တည်နေရာ (အခြေအနေ) ပြောင်းလဲနေသည်ဟု ကျွန်ုပ်တို့ ပြောကြသည်။ သို့သော် space-time ရှိ ထိုအမှုန်၏ world line ရှုထောင့်မှကြည့်လျှင် မည်သည့်ပြောင်းလဲမှုမျှ မပါဝင်ပါ။

202 Fortran ကို တီထွင်သူ John Backus သည် ၁၉၇၈ ခုနှစ်တွင် ACM Turing award ကို ရရှိခဲ့စဉ်က functional programming ကို လူသိများလာစေခဲ့သည်။ သူ၏ ကျေးဇူးတင်စကား (Backus 1978) တွင် functional ချဉ်းကပ်ပုံကို ပြင်းပြင်းထန်ထန် ထောက်ခံအားပေးခဲ့သည်။ functional programming ၏ ကောင်းမွန်သော ခြုံငုံသုံးသပ်ချက်ကို Henderson 1980 နှင့် Darlington et al. 1982 တို့တွင် ဖော်ပြထားသည်။

203 မည်သည့် stream နှစ်ခုအတွက်မဆို လက်ခံနိုင်ဖွယ်ရှိသော interleaving အစီအစဉ်သည် ယေဘုယျအားဖြင့် တစ်ခုထက်မက ရှိနိုင်သည်ကို သတိပြုပါ။ ထို့ကြောင့် နည်းပညာအရ “merge” သည် function တစ်ခုထက် relation တစ်ခု ဖြစ်သည် - အဖြေမှာ input များအပေါ်တွင် မူတည်သော deterministic function တစ်ခု မဟုတ်ပါ။ concurrency နှင့် ကိုင်တွယ်ရာတွင် nondeterminism သည် မရှိမဖြစ်လိုအပ်ကြောင်း ကျွန်ုပ်တို့ (Footnote 167 တွင်) ဖော်ပြခဲ့ပြီးဖြစ်သည်။ merge relation သည် functional ရှုထောင့်မှနေ၍ အလားတူ အရေးကြီးသော nondeterminism ကို ဖော်ပြနေခြင်းဖြစ်သည်။ 4.3 တွင် nondeterminism ကို အခြားရှုထောင့်တစ်ခုမှ ထပ်မံလေ့လာပါမည်။

204 object model သည် ကမ္ဘာလောကကို သီးခြားအစိတ်အပိုင်းများအဖြစ် ခွဲထုတ်ခြင်းဖြင့် ခန့်မှန်းတွက်ချက်သည်။ functional model သည် object နယ်နိမိတ်များအလိုက် modularize မလုပ်ဆောင်ပါ။ “object” များ၏ ဝေမျှမထားသော အခြေအနေ (unshared state) သည် ၎င်းတို့ ဝေမျှထားသော အခြေအနေထက် များစွာ ပိုကြီးမားသောအခါတွင် object model သည် အသုံးဝင်သည်။ object ရှုထောင့် မအောင်မြင်သည့် နေရာတစ်ခု၏ ဥပမာမှာ quantum mechanics ဖြစ်ပြီး၊ အရာဝတ္ထုများကို သီးခြားအမှုန်များအဖြစ် တွေးတောခြင်းသည် ရှေ့နောက်မညီမှုများနှင့် ရှုပ်ထွေးမှုများကို ဖြစ်ပေါ်စေသည်။ object ရှုထောင့်နှင့် functional ရှုထောင့်ကို ပေါင်းစည်းခြင်းသည် programming နှင့် များစွာမသက်ဆိုင်ဘဲ အခြေခံကျသော epistemological (အသိပညာဗေဒ) ဆိုင်ရာ ကိစ္စရပ်များနှင့်သာ သက်ဆိုင်ပေလိမ့်မည်။
