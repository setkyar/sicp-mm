4.3Scheme ၏ မူကွဲများ — Nondeterministic Computing

ဤအပိုင်းတွင်၊ အလိုအလျောက် ရှာဖွေနိုင်သည့် (automatic search) စွမ်းရည်ကို evaluator ထဲတွင် ထည့်သွင်းတည်ဆောက်ခြင်းဖြင့် nondeterministic computing ဟုခေါ်သော programming paradigm တစ်ခုကို support လုပ်နိုင်ရန် Scheme evaluator ကို ကျွန်ုပ်တို့ တိုးချဲ့ပါမည်။ ၎င်းသည် အပိုင်း 4.2 ရှိ lazy evaluation ကို မိတ်ဆက်ပေးခဲ့ခြင်းထက် ပို၍ နက်ရှိုင်းသော ဘာသာစကားဆိုင်ရာ ပြောင်းလဲမှုတစ်ခု ဖြစ်သည်။

Nondeterministic computing သည် stream processing ကဲ့သို့ပင် “generate and test” applications များအတွက် အသုံးဝင်သည်။ အပေါင်းကိန်း (positive integers) list နှစ်ခုမှ စတင်ပြီး၊ ပထမ list မှ တစ်ခုနှင့် ဒုတိယ list မှ တစ်ခုစီယူ၍ ၎င်းတို့၏ ပေါင်းလဒ်သည် prime ကိန်းဖြစ်သော integer စုံတွဲတစ်တွဲကို ရှာဖွေသည့် အလုပ်ကို စဉ်းစားကြည့်ပါ။ ဤအရာကို finite sequence operations များဖြင့် 2.2.3 တွင် လည်းကောင်း၊ infinite streams များဖြင့် 3.5.3 တွင် လည်းကောင်း မည်သို့ကိုင်တွယ်ရမည်ကို ကျွန်ုပ်တို့ တွေ့ခဲ့ပြီးဖြစ်သည်။ ကျွန်ုပ်တို့၏ ချဉ်းကပ်ပုံမှာ ဖြစ်နိုင်ခြေရှိသော စုံတွဲအားလုံး၏ sequence ကို ထုတ်လုပ်ပြီးနောက် ပေါင်းလဒ် prime ဖြစ်သော စုံတွဲများကို ရွေးချယ်ရန် filter လုပ်ခြင်း ဖြစ်သည်။ အခန်း Chapter 2 တွင်ကဲ့သို့ စုံတွဲ sequence တစ်ခုလုံးကို အရင်ထုတ်လုပ်သည်ဖြစ်စေ၊ သို့မဟုတ် အခန်း Chapter 3 တွင်ကဲ့သို့ ထုတ်လုပ်ခြင်းနှင့် filter လုပ်ခြင်းကို တစ်လှည့်စီ (interleave) လုပ်သည်ဖြစ်စေ၊ တွက်ချက်မှု မည်သို့ဖွဲ့စည်းထားသည်ဟူသော အဓိကပုံရိပ်အတွက်မူ အရေးမကြီးလှပေ။

Nondeterministic ချဉ်းကပ်ပုံသည် ကွဲပြားသော ပုံရိပ်တစ်ခုကို ဖော်ဆောင်ပေးသည်။ ပထမ list မှ ကိန်းတစ်ခုနှင့် ဒုတိယ list မှ ကိန်းတစ်ခုကို (တစ်နည်းနည်းဖြင့်) ရိုးရိုးရှင်းရှင်း ရွေးချယ်ပြီး ၎င်းတို့၏ ပေါင်းလဒ်သည် prime ဖြစ်ရမည်ဟု (ယန္တရားတစ်ခုခုကို သုံး၍) သတ်မှတ်လိုက်သည်ဟု စိတ်ကူးကြည့်ပါ။ ၎င်းကို အောက်ပါ procedure ဖြင့် ဖော်ပြနိုင်သည်- 

(define (prime-sum-pair list1 list2)
  (let ((a (an-element-of list1))
        (b (an-element-of list2)))
    (require (prime? (+ a b)))
    (list a b)))

ဤ procedure သည် ပြဿနာကို ဖြေရှင်းနည်းကို သတ်မှတ်ပေးခြင်းထက် ပြဿနာကို ပြန်လည်ဖော်ပြရုံသာ ဖြစ်သည်ဟု ထင်ရပေမည်။ သို့သော်ငြားလည်း၊ ၎င်းသည် ခိုင်လုံသော nondeterministic program တစ်ခု ဖြစ်သည်။246

ဤနေရာတွင် အဓိက စိတ်ကူးမှာ nondeterministic language ရှိ expression များသည် ဖြစ်နိုင်ခြေရှိသော တန်ဖိုးတစ်ခုထက်မက ရှိနိုင်ခြင်း ဖြစ်သည်။ ဥပမာအားဖြင့်၊ an-element-of သည် ပေးထားသော list ၏ မည်သည့် element ကိုမဆို ပြန်ပေးနိုင်သည်။ ကျွန်ုပ်တို့၏ nondeterministic program evaluator သည် ဖြစ်နိုင်ခြေရှိသော တန်ဖိုးတစ်ခုကို အလိုအလျောက် ရွေးချယ်ပြီး ထိုရွေးချယ်မှုကို မှတ်သားထားခြင်းဖြင့် အလုပ်လုပ်ပါမည်။ အကယ်၍ နောက်ဆက်တွဲ လိုအပ်ချက်နှင့် မကိုက်ညီပါက၊ evaluator သည် အခြားရွေးချယ်မှုတစ်ခုကို ထပ်မံကြိုးစားမည်ဖြစ်ပြီး၊ တွက်ချက်မှု အောင်မြင်သည်အထိ သို့မဟုတ် ရွေးချယ်စရာများ ကုန်ဆုံးသွားသည်အထိ ရွေးချယ်မှုအသစ်များကို ဆက်လက်ကြိုးစားနေမည် ဖြစ်သည်။ Lazy evaluator က တန်ဖိုးများကို မည်သို့ delay လုပ်ပြီး force လုပ်သည်ဟူသော အသေးစိတ်အချက်အလက်များမှ ပရိုဂရမ်မာကို လွတ်မြောက်စေခဲ့သလိုပင်၊ nondeterministic program evaluator သည် ရွေးချယ်မှုများကို မည်သို့ပြုလုပ်သည်ဟူသော အသေးစိတ်အချက်အလက်များမှ ပရိုဂရမ်မာကို လွတ်မြောက်စေမည်ဖြစ်သည်။

Nondeterministic evaluation နှင့် stream processing တို့မှ ဖော်ဆောင်ပေးသော မတူညီသည့် အချိန်ကာလ ပုံရိပ်များကို နှိုင်းယှဉ်ကြည့်ခြင်းက အကျိုးရှိစေသည်။ Stream processing သည် ဖြစ်နိုင်ခြေရှိသော အဖြေများ၏ stream ကို စုစည်းသည့် အချိန်နှင့် အမှန်တကယ် stream element များ ထုတ်လုပ်သည့် အချိန်ကို သီးခြားစီဖြစ်စေရန် (decouple) lazy evaluation ကို အသုံးပြုသည်။ Evaluator သည် ဖြစ်နိုင်ခြေရှိသော အဖြေအားလုံးသည် ကျွန်ုပ်တို့ရှေ့တွင် အချိန်နှင့်မသက်ဆိုင်သော sequence တစ်ခုအဖြစ် ရှိနေသည်ဟူသော အသွင်အပြင်ကို ဖန်တီးပေးသည်။ Nondeterministic evaluation တွင်မူ၊ expression တစ်ခုသည် ရွေးချယ်မှုများအပေါ် မူတည်၍ ဖြစ်နိုင်ခြေရှိသော ကမ္ဘာ (worlds) အစုအဝေးကို စူးစမ်းရှာဖွေခြင်းကို ကိုယ်စားပြုသည်။ အချို့သော ဖြစ်နိုင်ခြေရှိသော ကမ္ဘာများသည် လမ်းစပျောက်သွားနိုင်ပြီး၊ အချို့မှာမူ အသုံးဝင်သော တန်ဖိုးများ ရှိကြသည်။ Nondeterministic program evaluator သည် အချိန်ကာလများ ခက်ဖြာသွားသည်ဟူသော အသွင်အပြင်ကိုလည်းကောင်း၊ ကျွန်ုပ်တို့၏ ပရိုဂရမ်များတွင် ဖြစ်နိုင်ခြေရှိသော execution ရာဇဝင် အမျိုးမျိုးရှိသည်ဟူသော အသွင်အပြင်ကိုလည်းကောင်း အထောက်အကူပြုသည်။ လမ်းစပျောက်သည့် နေရာသို့ ရောက်သောအခါ၊ ယခင်က ရွေးချယ်ခဲ့သော နေရာသို့ ပြန်သွားပြီး အခြားသော လမ်းခွဲတစ်ခုအတိုင်း ဆက်လက် လုပ်ဆောင်နိုင်သည်။

အောက်တွင် ဖော်ပြထားသော nondeterministic program evaluator ကို amb evaluator ဟု ခေါ်သည်၊ အကြောင်းမှာ ၎င်းကို amb ဟုခေါ်သော special form အသစ်တစ်ခုပေါ်တွင် အခြေခံထားသောကြောင့် ဖြစ်သည်။ အထက်ပါ prime-sum-pair ၏ definition ကို (prime?၊ an-element-of နှင့် require တို့၏ definition များနှင့်အတူ) amb evaluator driver loop တွင် အောက်ပါအတိုင်း ရိုက်ထည့်၍ run နိုင်သည်- 

;;; Amb-Eval input:
(prime-sum-pair '(1 3 5 8) '(20 35 110))

;;; Starting a new problem
;;; Amb-Eval value:
(3 20)

ပြန်ပေးလိုက်သော တန်ဖိုးကို evaluator မှ list တစ်ခုချင်းစီမှ element များကို အောင်မြင်သော ရွေးချယ်မှုတစ်ခု ရသည်အထိ အကြိမ်ကြိမ် ရွေးချယ်ပြီးနောက် ရရှိလာခြင်း ဖြစ်သည်။

အပိုင်း 4.3.1 တွင် amb ကို မိတ်ဆက်ပေးမည်ဖြစ်ပြီး evaluator ၏ အလိုအလျောက် ရှာဖွေနိုင်သည့် ယန္တရားမှတစ်ဆင့် nondeterminism ကို မည်သို့ အထောက်အကူပြုသည်ကို ရှင်းပြပါမည်။ 4.3.2 တွင် nondeterministic program များ၏ ဥပမာများကို တင်ပြထားပြီး၊ 4.3.3 တွင် ပုံမှန် Scheme evaluator ကို ပြုပြင်ခြင်းဖြင့် amb evaluator ကို မည်သို့ implement လုပ်ရမည်ဟူသော အသေးစိတ်အချက်အလက်များကို ပေးထားပါသည်။

4.3.1Amb နှင့် ရှာဖွေခြင်း (Search)

Scheme တွင် nondeterminism ကို အထောက်အပံ့ပေးရန်အတွက် amb ဟုခေါ်သော special form အသစ်တစ်ခုကို ကျွန်ုပ်တို့ မိတ်ဆက်ပေးပါမည်။247 ဖော်ပြချက် (expression) 

(amb ⟨e₁⟩ ⟨e₂⟩ … ⟨eₙ⟩)

သည် n ခုသော ဖော်ပြချက်များ ⟨ e i ⟩ ထဲမှ တစ်ခု၏ တန်ဖိုးကို "မရေမရာ" (ambiguously) ပြန်ပေးသည်။ ဥပမာအားဖြင့်၊ ဖော်ပြချက် 

(list (amb 1 2 3) (amb 'a 'b))

တွင် ဖြစ်နိုင်ခြေရှိသော တန်ဖိုး ခြောက်ခု ရှိနိုင်သည်- 

(1 a) (1 b) (2 a) (2 b) (3 a) (3 b)

ရွေးချယ်စရာ တစ်ခုတည်းသာပါသော Amb သည် သာမန် (တစ်ခုတည်းသော) တန်ဖိုးကို ထုတ်ပေးသည်။

ရွေးချယ်စရာ လုံးဝမပါသော Amb — ဖော်ပြချက် (amb) — သည် လက်ခံနိုင်သော တန်ဖိုးမရှိသည့် ဖော်ပြချက်တစ်ခု ဖြစ်သည်။ လုပ်ဆောင်ချက်အရ ကျွန်ုပ်တို့သည် (amb) ကို တွက်ချက်သည့်အခါတွင် တွက်ချက်မှု "မအောင်မြင်ခြင်း" (fail) ကို ဖြစ်ပေါ်စေသော ဖော်ပြချက်တစ်ခုအဖြစ် မှတ်ယူနိုင်သည်။ တွက်ချက်မှုသည် ရပ်တန့်သွားပြီး မည်သည့်တန်ဖိုးမျှ ထုတ်ပေးမည်မဟုတ်ပါ။ ဤအကြံဉာဏ်ကို အသုံးပြု၍ သတ်မှတ်ထားသော predicate ဖော်ပြချက် p သည် မှန်ကန်ရမည်ဟူသော လိုအပ်ချက်ကို အောက်ပါအတိုင်း ဖော်ပြနိုင်သည်- 

(define (require p)
  (if (not p) (amb)))

amb နှင့် require တို့ကို အသုံးပြု၍ အထက်တွင် အသုံးပြုခဲ့သော an-element-of procedure ကို အောက်ပါအတိုင်း အကောင်အထည်ဖော်နိုင်သည်- 

(define (an-element-of items)
  (require (not (null? items)))
  (amb (car items) 
       (an-element-of (cdr items))))

အကယ်၍ list သည် ဗလာဖြစ်နေပါက An-element-of သည် မအောင်မြင်ပါ။ သို့မဟုတ်ပါက ၎င်းသည် list ၏ ပထမဆုံး element ကို ဖြစ်စေ၊ သို့မဟုတ် list ၏ ကျန်ရှိသော အစိတ်အပိုင်းမှ ရွေးချယ်ထားသော element တစ်ခုကို ဖြစ်စေ မရေမရာ ပြန်ပေးမည် ဖြစ်သည်။

ရွေးချယ်စရာများ၏ အကန့်အသတ်မဲ့ အတိုင်းအတာများကိုလည်း ဖော်ပြနိုင်သည်။ အောက်ပါ procedure သည် ပေးထားသော n ထက် ကြီးသော သို့မဟုတ် ညီသော မည်သည့် ကိန်းပြည့် (integer) ကိုမဆို ပြန်ပေးနိုင်သည်- 

(define (an-integer-starting-from n)
  (amb n (an-integer-starting-from (+ n 1))))

၎င်းသည် 3.5.2 တွင် ဖော်ပြထားသော stream procedure integers-starting-from နှင့် ဆင်တူသော်လည်း အရေးကြီးသော ကွာခြားချက်တစ်ခု ရှိသည်- stream procedure သည် n ဖြင့် စတင်သော ကိန်းပြည့်အားလုံး၏ အစီအစဉ်ကို ကိုယ်စားပြုသည့် object တစ်ခုကို ပြန်ပေးသော်လည်း amb procedure သည် ကိန်းပြည့် တစ်ခုတည်းကိုသာ ပြန်ပေးသည်။248

စိတ္တဇအားဖြင့်၊ amb ဖော်ပြချက်တစ်ခုကို တွက်ချက်ခြင်းသည် အချိန်ကို အခက်အလက်များ (branches) အဖြစ် ခွဲထုတ်လိုက်သည်ဟု စိတ်ကူးကြည့်နိုင်သည်။ ထိုအခက်အလက်တစ်ခုစီတွင် ဖော်ပြချက်၏ ဖြစ်နိုင်ခြေရှိသော တန်ဖိုးတစ်ခုစီဖြင့် တွက်ချက်မှု ဆက်လက်လုပ်ဆောင်သည်။ amb သည် nondeterministic choice point တစ်ခုကို ကိုယ်စားပြုသည်ဟု ကျွန်ုပ်တို့ ဆိုကြသည်။ အကယ်၍ ကျွန်ုပ်တို့တွင် လိုအပ်သလို ခွဲဝေအသုံးပြုနိုင်သော (dynamically allocated) ပရိုဆက်ဆာ အရေအတွက် လုံလောက်စွာရှိသည့် စက်တစ်ခု ရှိခဲ့ပါက၊ ရှာဖွေခြင်း (search) ကို တိုက်ရိုက် အကောင်အထည်ဖော်နိုင်သည်။ လုပ်ဆောင်ချက်သည် amb ဖော်ပြချက်တစ်ခုနှင့် မတွေ့မချင်း sequential machine တစ်ခုကဲ့သို့ ဆက်လက်လုပ်ဆောင်သွားမည်ဖြစ်သည်။ ထိုအမှတ်တွင် ရွေးချယ်မှုမှ ညွှန်ပြသော ပြိုင်တူလုပ်ဆောင်ချက်များ (parallel executions) အားလုံးကို ဆက်လက်လုပ်ဆောင်ရန်အတွက် ပရိုဆက်ဆာများကို ပိုမိုခွဲဝေပေးပြီး အစပျိုးပေးမည်ဖြစ်သည်။ ပရိုဆက်ဆာတစ်ခုစီသည် ၎င်းသည် တစ်ခုတည်းသော ရွေးချယ်မှုဖြစ်သကဲ့သို့ အစီအစဉ်အတိုင်း လုပ်ဆောင်သွားမည်ဖြစ်ပြီး၊ မအောင်မြင်မှု (failure) တစ်ခုနှင့် တွေ့၍ ရပ်တန့်သွားခြင်း၊ သို့မဟုတ် ထပ်မံခွဲထွက်ခြင်း၊ သို့မဟုတ် ပြီးဆုံးသွားခြင်းတို့အထိ လုပ်ဆောင်မည်ဖြစ်သည်။249

တစ်ဖက်တွင်မူ၊ လုပ်ငန်းစဉ်တစ်ခုတည်း (သို့မဟုတ် လုပ်ငန်းစဉ်အနည်းငယ်) သာ လုပ်ဆောင်နိုင်သော စက်တစ်ခုရှိလျှင် အခြားရွေးချယ်စရာများကို အစီအစဉ်လိုက် စဉ်းစားရမည်ဖြစ်သည်။ choice point တစ်ခုနှင့် တွေ့တိုင်း လိုက်နာရမည့် အခက်အလက်တစ်ခုကို ကျပန်း (random) ရွေးချယ်ရန် evaluator တစ်ခုကို ပြင်ဆင်ခြင်းကို စိတ်ကူးကြည့်နိုင်သည်။ သို့သော် ကျပန်းရွေးချယ်မှုသည် မအောင်မြင်သော တန်ဖိုးများဆီသို့ လွယ်ကူစွာ ဦးတည်သွားနိုင်သည်။ ကျွန်ုပ်တို့သည် evaluator ကို ထပ်ခါတလဲလဲ လုပ်ဆောင်စေပြီး ကျပန်းရွေးချယ်မှုများ ပြုလုပ်ကာ မအောင်မြင်သော တန်ဖိုးတစ်ခုကို ရှာတွေ့ရန် မျှော်လင့်နိုင်သော်လည်း ဖြစ်နိုင်ခြေရှိသော execution paths အားလုံးကို စနစ်တကျ ရှာဖွေခြင်း (systematically search) က ပိုကောင်းပါသည်။ ဤကဏ္ဍတွင် ကျွန်ုပ်တို့ တည်ဆောက်မည့် amb evaluator သည် စနစ်တကျ ရှာဖွေခြင်းကို အောက်ပါအတိုင်း အကောင်အထည်ဖော်သည်- evaluator သည် amb အသုံးပြုမှုကို တွေ့သောအခါတွင် အစပိုင်း၌ ပထမဆုံး အခြားရွေးချယ်စရာကို ရွေးချယ်သည်။ ဤရွေးချယ်မှုသည် နောက်ထပ် ရွေးချယ်မှုတစ်ခုဆီသို့ ဦးတည်သွားနိုင်သည်။ evaluator သည် choice point တစ်ခုစီတွင် အမြဲတမ်း ပထမဆုံး အခြားရွေးချယ်စရာကို အရင်ရွေးချယ်မည် ဖြစ်သည်။ အကယ်၍ ရွေးချယ်မှုတစ်ခုသည် မအောင်မြင်မှု (failure) ဖြစ်စေပါက evaluator သည် အလိုအလျောက်250 ပြန်သွားပြီး (backtracks) နောက်ဆုံးတွေ့ခဲ့သော choice point သို့ ပြန်သွားပြီး နောက်ထပ် အခြားရွေးချယ်စရာတစ်ခုကို စမ်းသပ်သည်။ အကယ်၍ choice point တစ်ခုတွင် အခြားရွေးချယ်စရာများ ကုန်သွားပါက evaluator သည် ယခင် choice point သို့ ပြန်သွားပြီး ထိုနေရာမှ ပြန်လည်စတင်မည် ဖြစ်သည်။ ဤလုပ်ငန်းစဉ်သည် depth-first search သို့မဟုတ် chronological backtracking ဟု လူသိများသော ရှာဖွေရေး မဟာဗျူဟာသို့ ဦးတည်စေသည်။251

Driver loop

amb evaluator အတွက် driver loop တွင် ထူးခြားသော ဂုဏ်သတ္တိအချို့ ရှိသည်။ ၎င်းသည် ဖော်ပြချက်တစ်ခုကို ဖတ်ရှုပြီး အထက်တွင် ဖော်ပြခဲ့သော prime-sum-pair ဥပမာကဲ့သို့ ပထမဆုံး မအောင်မြင်သော လုပ်ဆောင်ချက်၏ တန်ဖိုးကို ရိုက်နှိပ်ပေးသည်။ အကယ်၍ ကျွန်ုပ်တို့သည် နောက်ထပ် အောင်မြင်သော လုပ်ဆောင်ချက်၏ တန်ဖိုးကို မြင်လိုပါက interpreter ကို backtrack လုပ်ခိုင်းပြီး ဒုတိယမြောက် မအောင်မြင်သော လုပ်ဆောင်ချက်ကို ထုတ်ပေးရန် ကြိုးစားခိုင်းနိုင်သည်။ ၎င်းကို try-again ဆိုသော သင်္ကေတကို ရိုက်နှိပ်ခြင်းဖြင့် အချက်ပြသည်။ အကယ်၍ try-again မှအပ အခြား မည်သည့် ဖော်ပြချက်ကိုမဆို ပေးပါက interpreter သည် ယခင်ပြဿနာရှိ ရှာဖွေမထားသော အခြားရွေးချယ်စရာများကို စွန့်လွှတ်ပြီး ပြဿနာအသစ်တစ်ခုကို စတင်ပါလိမ့်မည်။ ဤတွင် နမူနာ အပြန်အလှန်လုပ်ဆောင်မှုတစ်ခု ရှိသည်-

;;; Starting a new problem
;;; Amb-Eval value:
(3 20)

;;; Amb-Eval input:

;;; Amb-Eval value:
(3 110)

;;; Amb-Eval input:
try-again

;;; Amb-Eval value:

;;; There are no more values of

 (quote (1 3 5 8))

လေ့ကျင့်ခန်း ၄.၃၅: ပေးထားသော အကန့်အသတ် နှစ်ခုကြားရှိ ကိန်းပြည့်တစ်ခုကို ပြန်ပေးသော an-integer-between procedure တစ်ခုကို ရေးသားပါ။ ၎င်းကို Pythagorean triples များကို ရှာဖွေပေးသည့် procedure တစ်ခုအား အောက်ပါအတိုင်း အကောင်အထည်ဖော်ရန် အသုံးပြုနိုင်သည်၊ ဆိုလိုသည်မှာ ပေးထားသော အကန့်အသတ်များကြားရှိ i ≤ j နှင့် i 2 + j 2 = k 2 ဖြစ်သော ကိန်းပြည့် သုံးခုတွဲ ( i , j , k ) များ ဖြစ်သည်- 

(define (a-pythagorean-triple-between low high)
  (let ((i (an-integer-between low high)))
    (let ((j (an-integer-between i high)))
      (let ((k (an-integer-between j high)))
        (require (= (+ (* i i) (* j j)) 
                    (* k k)))
        (list i j k)))))

လေ့ကျင့်ခန်း ၄.၃၆: Exercise 3.69 တွင် ရှာဖွေရမည့် ကိန်းပြည့်များ၏ အရွယ်အစားအတွက် အထက်ကန့်သတ်ချက်မရှိဘဲ Pythagorean triples အားလုံး၏ stream ကို မည်သို့ ထုတ်လုပ်ရမည်ကို ဆွေးနွေးခဲ့သည်။ Exercise 4.35 ရှိ procedure တွင် an-integer-between ကို an-integer-starting-from နှင့် အစားထိုးရုံမျှဖြင့် မထူးခြားသော Pythagorean triples များကို ထုတ်လုပ်ရန်အတွက် အဘယ်ကြောင့် မလုံလောက်ရသည်ကို ရှင်းပြပါ။ ၎င်းကို အမှန်တကယ် လုပ်ဆောင်နိုင်မည့် procedure တစ်ခုကို ရေးသားပါ။ (ဆိုလိုသည်မှာ try-again ကို ထပ်ခါတလဲလဲ ရိုက်နှိပ်ခြင်းဖြင့် မူအရ Pythagorean triples အားလုံးကို နောက်ဆုံးတွင် ထုတ်ပေးနိုင်မည့် procedure တစ်ခုကို ရေးသားပါ။)

လေ့ကျင့်ခန်း ၄.၃၇: Pythagorean triples များကို ထုတ်လုပ်ရန်အတွက် အောက်ပါနည်းလမ်းသည် Exercise 4.35 ရှိ နည်းလမ်းထက် ပိုမိုထိရောက်သည်ဟု Ben Bitdiddle က ဆိုပါသည်။ သူပြောသည်မှာ မှန်ပါသလား။ (အရိပ်အမြွက်- ရှာဖွေရမည့် ဖြစ်နိုင်ခြေ အရေအတွက်ကို ထည့်သွင်းစဉ်းစားပါ။) 

(define (a-pythagorean-triple-between low high)
  (let ((i (an-integer-between low high))
        (hsq (* high high)))
    (let ((j (an-integer-between i high)))
      (let ((ksq (+ (* i i) (* j j))))
        (require (>= hsq ksq))
        (let ((k (sqrt ksq)))
          (require (integer? k))
          (list i j k))))))

4.3.2Nondeterministic ပရိုဂရမ်များ၏ ဥပမာများ

အပိုင်း 4.3.3 တွင် amb evaluator ၏ အကောင်အထည်ဖော်ပုံကို ဖော်ပြထားသည်။ သို့သော် ဦးစွာပထမ၊ ၎င်းကို မည်သို့အသုံးပြုနိုင်ပုံ ဥပမာအချို့ကို တင်ပြပါမည်။ Nondeterministic programming ၏ အားသာချက်မှာ ရှာဖွေမှု (search) လုပ်ဆောင်ပုံ အသေးစိတ်အချက်အလက်များကို ဖုံးကွယ်ထားနိုင်ပြီး၊ ထို့ကြောင့် ကျွန်ုပ်တို့၏ ပရိုဂရမ်များကို ပိုမိုမြင့်မားသော abstraction အဆင့်တွင် ဖော်ပြနိုင်ခြင်းပင် ဖြစ်သည်။

အောက်ပါပဟေဠိသည် (Dinesman 1968 မှ ကောက်နုတ်ထားသည်) ရိုးရှင်းသော logic puzzle အမျိုးအစားများထဲမှ တစ်ခုဖြစ်သည်-

Baker, Cooper, Fletcher, Miller, နှင့် Smith တို့သည် ငါးထပ်သာရှိသော တိုက်ခန်းတစ်တိုက်၏ မတူညီသော အထပ်များတွင် နေထိုင်ကြသည်။ Baker သည် အပေါ်ဆုံးထပ်တွင် မနေပါ။ Cooper သည် အောက်ဆုံးထပ်တွင် မနေပါ။ Fletcher သည် အပေါ်ဆုံးထပ်ရော အောက်ဆုံးထပ်တွင်ပါ မနေပါ။ Miller သည် Cooper ထက် ပိုမြင့်သော အထပ်တွင် နေသည်။ Smith သည် Fletcher နေသော အထပ်နှင့် ကပ်လျက်အထပ်တွင် မနေပါ။ Fletcher သည် Cooper နေသော အထပ်နှင့် ကပ်လျက်အထပ်တွင် မနေပါ။ မည်သူသည် မည်သည့်အထပ်တွင် နေထိုင်ကြသနည်း။

ဖြစ်နိုင်ခြေရှိသမျှ အားလုံးကို တစ်ခုချင်းစီ ထုတ်ဖော်ကြည့်ပြီး ပေးထားသော ကန့်သတ်ချက်များကို အသုံးချခြင်းဖြင့် မည်သူသည် မည်သည့်အထပ်တွင် နေသည်ကို တိုက်ရိုက်နည်းလမ်းဖြင့် ဆုံးဖြတ်နိုင်သည်-252 

(define (multiple-dwelling)
  (let ((baker (amb 1 2 3 4 5))
        (cooper (amb 1 2 3 4 5))
        (fletcher (amb 1 2 3 4 5))
        (miller (amb 1 2 3 4 5))
        (smith (amb 1 2 3 4 5)))
    (require
     (distinct? (list baker cooper fletcher 
                      miller smith)))
    (require (not (= baker 5)))
    (require (not (= cooper 1)))
    (require (not (= fletcher 5)))
    (require (not (= fletcher 1)))
    (require (> miller cooper))
    (require
     (not (= (abs (- smith fletcher)) 1)))
    (require 
     (not (= (abs (- fletcher cooper)) 1)))
    (list (list 'baker baker)
          (list 'cooper cooper)
          (list 'fletcher fletcher)
          (list 'miller miller)
          (list 'smith smith))))

ဤရိုးရှင်းသော လုပ်ထုံးလုပ်နည်း (procedure) သည် အလုပ်လုပ်သော်လည်း အလွန်နှေးကွေးပါသည်။ လေ့ကျင့်ခန်း 4.39 နှင့် လေ့ကျင့်ခန်း 4.40 တို့တွင် ဖြစ်နိုင်ခြေရှိသော တိုးတက်ကောင်းမွန်အောင် ပြုပြင်နည်းအချို့ကို ဆွေးနွေးထားသည်။

လေ့ကျင့်ခန်း 4.38: Smith နှင့် Fletcher တို့သည် ကပ်လျက်အထပ်များတွင် မနေရဟူသော ကန့်သတ်ချက်ကို ချန်လှပ်၍ multiple-dwelling procedure ကို ပြုပြင်ပါ။ ဤသို့ ပြုပြင်ထားသော ပဟေဠိအတွက် အဖြေပေါင်း မည်မျှရှိသနည်း။

လေ့ကျင့်ခန်း 4.39: multiple-dwelling procedure တွင် ကန့်သတ်ချက်များ၏ အစဉ်လိုက်စီထားမှုသည် အဖြေအပေါ် သက်ရောက်မှု ရှိပါသလား။ အဖြေရှာရန် ကြာမြင့်ချိန်အပေါ် သက်ရောက်မှု ရှိပါသလား။ အကယ်၍ သက်ရောက်မှုရှိသည်ဟု သင်ထင်ပါက၊ ကန့်သတ်ချက်များကို အစဉ်လိုက် ပြန်လည်စီခြင်းဖြင့် ပိုမိုမြန်ဆန်သော ပရိုဂရမ်တစ်ခုကို တင်ပြပါ။ အကယ်၍ သက်ရောက်မှုမရှိဟု သင်ထင်ပါက၊ သင်၏ အကြောင်းပြချက်ကို တင်ပြပါ။

လေ့ကျင့်ခန်း 4.40: multiple dwelling ပုစ္ဆာတွင်၊ အထပ်များ သတ်မှတ်ချက် မတူညီရဟူသော ကန့်သတ်ချက် မတိုင်မီနှင့် နောက်ပိုင်းတွင် လူများကို အထပ်များသို့ သတ်မှတ်ပေးနိုင်သော အစုအဝေး (sets of assignments) မည်မျှရှိသနည်း။ လူများကို အထပ်များသို့ သတ်မှတ်ပေးနိုင်သော ဖြစ်နိုင်ခြေအားလုံးကို ထုတ်ပေးပြီးမှ backtracking ကို အသုံးပြု၍ ဖယ်ရှားခြင်းသည် အလွန်ပင် ထိရောက်မှု မရှိပါ။ ဥပမာအားဖြင့်၊ ကန့်သတ်ချက် အများစုသည် လူ-အထပ် variable တစ်ခု သို့မဟုတ် နှစ်ခုအပေါ်တွင်သာ မူတည်ပြီး၊ ထို့ကြောင့် လူအားလုံးအတွက် အထပ်များ မရွေးချယ်မီတွင်ပင် ၎င်းတို့ကို အသုံးပြုနိုင်သည်။ ယခင်ကန့်သတ်ချက်များအရ ပယ်ဖျက်ပြီးသား ဖြစ်နိုင်ခြေများကို မဟုတ်ဘဲ၊ ကျန်ရှိသော ဖြစ်နိုင်ခြေများကိုသာ ထုတ်ပေးခြင်းဖြင့် ဤပုစ္ဆာကို ဖြေရှင်းပေးမည့် ပိုမိုထိရောက်သော nondeterministic procedure တစ်ခုကို ရေးသားပြီး သရုပ်ပြပါ။ (Hint: ဤသည်မှာ let expression များစွာ ဆင့်ကဲအသုံးပြုရန် လိုအပ်ပါလိမ့်မည်။)

လေ့ကျင့်ခန်း 4.42: အောက်ပါ "Liars" ပဟေဠိကို ဖြေရှင်းပါ (Phillips 1934 မှ):

ကျောင်းသူငါးဦး စာမေးပွဲဖြေဆိုကြသည်။ သူတို့၏ မိဘများသည် ရလဒ်အပေါ်တွင် အလွန်အမင်း စိတ်ဝင်စားမှု ပြသကြသည်ဟု သူတို့က ထင်မြင်ကြသည်။ ထို့ကြောင့် အိမ်သို့ စာမေးပွဲအကြောင်း စာရေးရာတွင်၊ မိန်းကလေး တစ်ဦးစီသည် မှန်ကန်သော အဆိုတစ်ခုနှင့် မှားယွင်းသော အဆိုတစ်ခုကို ပြုလုပ်ရန် သဘောတူခဲ့ကြသည်။ သူတို့၏ ပေးစာများမှ သက်ဆိုင်ရာ အစိတ်အပိုင်းများမှာ အောက်ပါအတိုင်း ဖြစ်သည်-

  • Betty: “Kitty က စာမေးပွဲမှာ ဒုတိယရတယ်။ ကျွန်မက တတိယပဲ ရပါတယ်။”

  • Kitty: “ကျွန်မ ဒုတိယ ရတယ်။ Mary က စတုတ္ထပဲ ရပါတယ်။”
  • Mary: “ကျွန်မ စတုတ္ထ ရပါတယ်။ ပထမနေရာကိုတော့ Betty က ယူသွားပါတယ်။”

Parsing natural language (သဘာဝဘာသာစကားကို ခွဲခြမ်းစိတ်ဖြာခြင်း)

သဘာဝဘာသာစကားကို input အဖြစ် လက်ခံရန် ဒီဇိုင်းထုတ်ထားသော ပရိုဂရမ်များသည် များသောအားဖြင့် input ကို parse လုပ်ရန် (ဆိုလိုသည်မှာ input ကို သဒ္ဒါဆိုင်ရာ ဖွဲ့စည်းပုံ တစ်ခုခုနှင့် တိုက်ဆိုင်ကြည့်ရန်) ကြိုးပမ်းခြင်းဖြင့် စတင်လေ့ရှိသည်။ ဥပမာအားဖြင့်၊ "The cat eats" ကဲ့သို့သော article တစ်ခု၊ နောက်တွင် noun တစ်ခု၊ ၎င်းနောက်တွင် verb တစ်ခု ပါဝင်သော ရိုးရှင်းသော ဝါကျများကို အသိအမှတ်ပြုရန် ကြိုးစားနိုင်သည်။ ထိုကဲ့သို့သော ခွဲခြမ်းစိတ်ဖြာမှုကို လုပ်ဆောင်နိုင်ရန်အတွက်၊ ကျွန်ုပ်တို့သည် စကားလုံး တစ်လုံးချင်းစီ၏ ဝါစင်္ဂ (parts of speech) များကို ခွဲခြားနိုင်ရပါမည်။ စကားလုံး အမျိုးမျိုးကို အမျိုးအစားခွဲထားသော list အချို့ဖြင့် စတင်နိုင်သည်-254

ကျွန်ုပ်တို့တွင် grammar (သဒ္ဒါ) တစ်ခုလည်း လိုအပ်သည်၊ ဆိုလိုသည်မှာ သဒ္ဒါဆိုင်ရာ အစိတ်အပိုင်းများကို ပိုမိုရိုးရှင်းသော အစိတ်အပိုင်းများမှ မည်သို့ပေါင်းစပ်ရမည်ကို ဖော်ပြထားသော စည်းမျဉ်းအစုအဝေး ဖြစ်သည်။ အလွန်ရိုးရှင်းသော သဒ္ဒါတစ်ခုသည် ဝါကျတစ်ခုတွင် အစိတ်အပိုင်းနှစ်ခု အမြဲပါဝင်ရမည်ဟု သတ်မှတ်နိုင်သည်—noun phrase တစ်ခုနှင့် ၎င်းနောက်တွင် verb တစ်ခု—ဖြစ်ပြီး noun phrase တစ်ခုတွင် article တစ်ခုနှင့် ၎င်းနောက်တွင် noun တစ်ခု ပါဝင်သည်။ ဤသဒ္ဒါအရ "The cat eats" ဆိုသော ဝါကျကို အောက်ပါအတိုင်း parse လုပ်နိုင်သည်- 

(sentence
 (noun-phrase (article the) (noun cat))
 (verb eats))

သဒ္ဒါဆိုင်ရာ စည်းမျဉ်းတစ်ခုစီအတွက် သီးခြား procedure များရှိသော ရိုးရှင်းသော ပရိုဂရမ်တစ်ခုဖြင့် ထိုသို့သော parse တစ်ခုကို ထုတ်ပေးနိုင်သည်။ ဝါကျတစ်ခုကို parse လုပ်ရန်အတွက်၊ ၎င်း၏ ဖွဲ့စည်းပုံ အစိတ်အပိုင်းနှစ်ခုကို ခွဲခြားပြီး ထိုအစိတ်အပိုင်း နှစ်ခုပါသော list တစ်ခုကို sentence ဟူသော သင်္ကေတဖြင့် tag လုပ်၍ ပြန်ပေးပါသည်- 

(define (parse-sentence)
  (list 'sentence
         (parse-noun-phrase)

(define (parse-noun-phrase)
  (list 'noun-phrase
        (parse-word articles)
        (parse-word nouns)))

အနိမ့်ဆုံးအဆင့်တွင်၊ parsing ဆိုသည်မှာ နောက်ထပ် parse မလုပ်ရသေးသော စကားလုံးသည် လိုအပ်သော ဝါစင်္ဂအတွက် သတ်မှတ်ထားသော စကားလုံး list ထဲတွင် ပါဝင်ခြင်း ရှိမရှိကို ထပ်ခါတလဲလဲ စစ်ဆေးခြင်းပင် ဖြစ်သည်။ ဤအရာကို အကောင်အထည်ဖော်ရန်အတွက်၊ ကျွန်ုပ်တို့သည် parse မလုပ်ရသေးသော input များဖြစ်သည့် global variable *unparsed* ကို ထိန်းသိမ်းထားသည်။ စကားလုံးတစ်လုံးကို စစ်ဆေးသည့်အခါတိုင်း၊ *unparsed* သည် အလွတ်မဖြစ်ရဘဲ ၎င်းသည် သတ်မှတ်ထားသော list ထဲမှ စကားလုံးတစ်လုံးဖြင့် စတင်ရမည်ဟု ကန့်သတ်ထားသည်။ အကယ်၍ ဟုတ်ပါက၊ ကျွန်ုပ်တို့သည် ထိုစကားလုံးကို *unparsed* ထဲမှ ဖယ်ရှားပြီး ၎င်းစကားလုံးနှင့်အတူ ၎င်း၏ ဝါစင်္ဂ (list ၏ ရှေ့ဆုံးတွင် တွေ့ရသည်) ကိုပါ တွဲ၍ ပြန်ပေးပါသည်-255 

(define (parse-word word-list)
  (require (not (null? *unparsed*)))
  (require (memq (car *unparsed*) 
                 (cdr word-list)))
  (let ((found-word (car *unparsed*)))
    (set! *unparsed* (cdr *unparsed*))
    (list (car word-list) found-word)))

Parsing စတင်ရန်အတွက်၊ ကျွန်ုပ်တို့ လုပ်ဆောင်ရန် လိုအပ်သည်မှာ *unparsed* ကို input တစ်ခုလုံးအဖြစ် သတ်မှတ်ပေးရန်၊ ဝါကျတစ်ခုကို parse လုပ်ရန် ကြိုးစားရန်နှင့် ဘာမျှကျန်မနေကြောင်း စစ်ဆေးရန်သာ ဖြစ်သည်- 

(define *unparsed* '())

  (set! *unparsed* input)
  (let ((sent (parse-sentence)))
    (require (null? *unparsed*))
    sent))

Hacker သည် ဝါကျများကို parse လုပ်ခြင်းထက် စိတ်ဝင်စားစရာကောင်းသော ဝါကျများကို ထုတ်ပေးခြင်း (generating) ကို ပိုမိုစိတ်ဝင်စားသည်။ parse-word procedure ကို "input sentence" ကို လျစ်လျူရှုပြီး အမြဲတမ်း အောင်မြင်ကာ သင့်တော်သော စကားလုံးတစ်လုံးကို ထုတ်ပေးရန် ပြောင်းလဲလိုက်ရုံဖြင့်၊ parsing အတွက် တည်ဆောက်ထားသော ပရိုဂရမ်များကို ဝါကျထုတ်ပေးရန်အတွက် အသုံးပြုနိုင်သည်ဟု သူမက ယူဆသည်။ Alyssa ၏ စိတ်ကူးကို အကောင်အထည်ဖော်ပြီး၊ ပထမဆုံး ထွက်ပေါ်လာသော ဝါကျ ခြောက်ခုခန့်ကို ပြသပါ။258

4.3.3Amb Evaluator ကို အကောင်အထည်ဖော်ခြင်း

ရိုးရိုး Scheme ဖော်ပြချက် (expression) တစ်ခုကို အကဲဖြတ်ခြင်း (evaluation) သည် တန်ဖိုးတစ်ခု ပြန်ပေးနိုင်သည်၊ သို့မဟုတ် မည်သည့်အခါမျှ မပြီးဆုံးနိုင်ပါ၊ သို့မဟုတ် အမှားတစ်ခုကို အကြောင်းကြားနိုင်ပါသည်။ nondeterministic Scheme တွင်မူ ဖော်ပြချက်တစ်ခုကို အကဲဖြတ်ခြင်းသည် လမ်းဆုံး (dead end) တစ်ခုကို တွေ့ရှိခြင်းမျိုးလည်း ဖြစ်နိုင်ပြီး၊ ထိုသို့ဖြစ်ပါက အကဲဖြတ်ခြင်းသည် ယခင်ရွေးချယ်မှုအမှတ် (choice point) သို့ နောက်ကြောင်းပြန်လှည့် (backtrack) ရမည် ဖြစ်သည်။ nondeterministic Scheme ကို အဓိပ္ပာယ်ပြန်ဆိုခြင်း (interpretation) သည် ဤအပိုအခြေအနေကြောင့် ပိုမိုရှုပ်ထွေးသွားပါသည်။

ကျွန်ုပ်တို့သည် 4.1.7 ရှိ analyzing evaluator ကို ပြင်ဆင်ခြင်းဖြင့် nondeterministic Scheme အတွက် amb evaluator ကို တည်ဆောက်ပါမည်။259 analyzing evaluator မှာကဲ့သို့ပင်၊ ဖော်ပြချက်တစ်ခုကို အကဲဖြတ်ခြင်းကို ထိုဖော်ပြချက်ကို ခွဲခြမ်းစိတ်ဖြာခြင်း (analysis) မှ ထွက်ပေါ်လာသော execution procedure တစ်ခုကို ခေါ်ဆိုခြင်းဖြင့် ဆောင်ရွက်ပါသည်။ ရိုးရိုး Scheme နှင့် nondeterministic Scheme တို့၏ အဓိပ္ပာယ်ပြန်ဆိုပုံ ကွာခြားချက်မှာ execution procedures များတွင်သာ လုံးလုံးလျားလျား ရှိပါလိမ့်မည်။

Execution procedures နှင့် continuations များ

ရိုးရိုး evaluator အတွက် execution procedures များသည် argument တစ်ခု (execution environment) ကိုသာ လက်ခံကြောင်း သတိရပါ။ ၎င်းနှင့်ဆန့်ကျင်ဘက်အနေဖြင့်၊ amb evaluator ရှိ execution procedures များသည် argument သုံးခုကို လက်ခံသည်- ၎င်းတို့မှာ environment နှင့် continuation procedures ဟု ခေါ်ဆိုသော လုပ်ထုံးလုပ်နည်း နှစ်ခုတို့ ဖြစ်ကြသည်။ ဖော်ပြချက်တစ်ခုကို အကဲဖြတ်ခြင်းသည် ဤ continuations နှစ်ခုအနက် တစ်ခုကို ခေါ်ဆိုခြင်းဖြင့် ပြီးဆုံးပါလိမ့်မည်- အကယ်၍ အကဲဖြတ်မှုမှ တန်ဖိုးတစ်ခု ရရှိခဲ့လျှင် success continuation ကို ထိုတန်ဖိုးနှင့်အတူ ခေါ်ဆိုပါသည်၊ အကယ်၍ အကဲဖြတ်မှုသည် လမ်းဆုံးတစ်ခုကို တွေ့ရှိခဲ့လျှင်မူ failure continuation ကို ခေါ်ဆိုပါသည်။ သင့်တော်သော continuations များကို တည်ဆောက်ခြင်းနှင့် ခေါ်ဆိုခြင်းသည် nondeterministic evaluator က backtracking ကို အကောင်အထည်ဖော်သည့် ယန္တရားပင် ဖြစ်သည်။

တန်ဖိုးတစ်ခုကို လက်ခံရယူပြီး တွက်ချက်မှုကို ဆက်လက်လုပ်ဆောင်ရန်မှာ success continuation ၏ တာဝန်ဖြစ်သည်။ ထိုတန်ဖိုးနှင့်အတူ success continuation ထံသို့ နောက်ထပ် failure continuation တစ်ခုကိုလည်း ထည့်ပေးလိုက်ပါသည်၊ ၎င်းမှာ အဆိုပါတန်ဖိုးကို အသုံးပြုခြင်းသည် လမ်းဆုံးသို့ ရောက်သွားပါက နောက်ပိုင်းတွင် ခေါ်ဆိုရန် ဖြစ်သည်။

nondeterministic လုပ်ငန်းစဉ်၏ အခြားလမ်းကြောင်းတစ်ခုကို စမ်းသပ်ကြည့်ရန်မှာ failure continuation ၏ တာဝန်ဖြစ်သည်။ nondeterministic ဘာသာစကား၏ အနှစ်သာရမှာ ဖော်ပြချက်များသည် ရွေးချယ်စရာများထဲမှ တစ်ခုခုကို ကိုယ်စားပြုနိုင်ခြင်းတွင် ရှိသည်။ မည်သည့်ရွေးချယ်မှုက လက်ခံနိုင်လောက်သော ရလဒ်များကို ပေးမည်ကို ကြိုတင်မသိနိုင်သော်လည်း၊ ထိုကဲ့သို့သော ဖော်ပြချက်ကို အကဲဖြတ်ရာတွင် သတ်မှတ်ထားသော ရွေးချယ်စရာများထဲမှ တစ်ခုဖြင့် ရှေ့ဆက်ရပါမည်။ ဒါကို ကိုင်တွယ်ရန်အတွက် evaluator က ရွေးချယ်စရာများထဲမှ တစ်ခုကို ရွေးချယ်ပြီး အဆိုပါတန်ဖိုးကို success continuation ထံ ပေးပို့လိုက်ပါသည်။ ဤတန်ဖိုးနှင့်အတူ၊ အခြားရွေးချယ်စရာတစ်ခုကို နောက်မှရွေးချယ်ရန်အတွက် ခေါ်ဆိုနိုင်သော failure continuation တစ်ခုကိုလည်း evaluator က တည်ဆောက်ပြီး အတူတကွ ပေးပို့လိုက်ပါသည်။

အသုံးပြုသူ၏ ပရိုဂရမ်က လက်ရှိလုပ်ဆောင်နေသော လမ်းကြောင်းကို တိတိလင်းလင်း ငြင်းပယ်လိုက်သောအခါ (ဥပမာ- require ကို ခေါ်ဆိုခြင်းသည် အမြဲတမ်းရှုံးနိမ့်သော ဖော်ပြချက်ဖြစ်သည့် (amb) ကို လုပ်ဆောင်ခြင်းသို့ ရောက်ရှိသွားနိုင်သည် — 4.3.1 ကိုကြည့်ပါ) အကဲဖြတ်နေစဉ်အတွင်း ကျရှုံးမှု (failure) တစ်ခု ဖြစ်ပေါ်လာပါသည် (ဆိုလိုသည်မှာ failure continuation ကို ခေါ်ဆိုခြင်းဖြစ်သည်)။ ထိုအချိန်တွင် ရှိနေသော failure continuation သည် လတ်တလောအကျဆုံး ရွေးချယ်မှုအမှတ် (choice point) ကို အခြားရွေးချယ်စရာတစ်ခုကို ရွေးချယ်စေပါလိမ့်မည်။ အကယ်၍ ထိုရွေးချယ်မှုအမှတ်တွင် စဉ်းစားရန် ရွေးချယ်စရာများ မကျန်တော့ပါက၊ ယခင်ရွေးချယ်မှုအမှတ်တစ်ခုတွင် ကျရှုံးမှုတစ်ခုကို ဖြစ်ပေါ်စေပါသည်၊ ဤနည်းအတိုင်း ဆက်သွားပါသည်။ ဖော်ပြချက်၏ နောက်ထပ်တန်ဖိုးတစ်ခုကို ရှာဖွေရန် try-again တောင်းဆိုမှုကို တုံ့ပြန်သည့်အနေဖြင့် driver loop ကလည်း failure continuations များကို ခေါ်ဆိုပါသည်။

ထို့အပြင်၊ ရွေးချယ်မှုတစ်ခုမှ ဖြစ်ပေါ်လာသော လုပ်ငန်းစဉ်လမ်းကြောင်းပေါ်တွင် ဘေးထွက်ဆိုးကျိုး လုပ်ဆောင်ချက်တစ်ခု (ဥပမာ- ကိန်းရှင်တစ်ခုသို့ တန်ဖိုးသတ်မှတ်ခြင်း) ဖြစ်ပေါ်ခဲ့လျှင်၊ လုပ်ငန်းစဉ်သည် လမ်းဆုံးသို့ ရောက်သောအခါ ရွေးချယ်မှုအသစ်တစ်ခု မပြုလုပ်မီ ထိုဘေးထွက်ဆိုးကျိုးကို ပြန်လည်ပြင်ဆင်ရန် (undo) လိုအပ်နိုင်ပါသည်။ ဘေးထွက်ဆိုးကျိုးကို ပြန်ပြင်ပြီး ကျရှုံးမှုကို ဆက်လက်လက်ဆင့်ကမ်းပေးသော failure continuation တစ်ခုကို ဘေးထွက်ဆိုးကျိုးလုပ်ဆောင်ချက်က ထုတ်ပေးခြင်းဖြင့် ဤအရာကို ပြီးမြောက်စေပါသည်။

အကျဉ်းချုပ်အားဖြင့် failure continuations များကို အောက်ပါတို့မှ တည်ဆောက်ပါသည် -

  • amb expressions — amb expression မှ ပြုလုပ်ထားသော လက်ရှိရွေးချယ်မှုသည် လမ်းဆုံးသို့ ရောက်သွားပါက အခြားရွေးချယ်စရာများ ပြုလုပ်နိုင်ရန် ယန္တရားတစ်ခု ပေးရန်။
  • the top-level driver — ရွေးချယ်စရာများ ကုန်ဆုံးသွားသောအခါ ကျရှုံးမှုကို အစီရင်ခံရန် ယန္တရားတစ်ခု ပေးရန်။
  • assignments — backtracking လုပ်နေစဉ်အတွင်း ကျရှုံးမှုများကို ကြားဖြတ်ဖမ်းယူရန်နှင့် assignments များကို ပြန်လည်ပြင်ဆင်ရန်။

လမ်းဆုံးတစ်ခုနှင့် ကြုံတွေ့ရသောအခါမှသာ ကျရှုံးမှုများကို စတင်ပါသည်။ ၎င်းမှာ အောက်ပါအတိုင်း ဖြစ်ပေါ်ပါသည် -

  • အသုံးပြုသူ ပရိုဂရမ်က (amb) ကို လုပ်ဆောင်လျှင်။
  • အသုံးပြုသူက top-level driver တွင် try-again ဟု ရိုက်ထည့်လျှင်။

ကျရှုံးမှုတစ်ခုကို လုပ်ဆောင်နေစဉ်အတွင်း၌လည်း failure continuations များကို ခေါ်ဆိုပါသည် -

  • assignment တစ်ခုမှ ဖန်တီးထားသော failure continuation သည် ဘေးထွက်ဆိုးကျိုးကို ပြန်ပြင်ပြီးသွားသောအခါ၊ ဤ assignment သို့ ဦးတည်ခဲ့သော ရွေးချယ်မှုအမှတ် သို့မဟုတ် top level သို့ ကျရှုံးမှုကို ပြန်လည်လက်ဆင့်ကမ်းပေးရန်အတွက် ၎င်းကြားဖြတ်ဖမ်းယူထားသော failure continuation ကို ခေါ်ဆိုပါသည်။
  • amb အတွက် failure continuation သည် ရွေးချယ်စရာများ ကုန်သွားသောအခါ၊ ယခင်ရွေးချယ်မှုအမှတ် သို့မဟုတ် top level သို့ ကျရှုံးမှုကို ပြန်လည်လက်ဆင့်ကမ်းပေးရန်အတွက် amb ထံသို့ မူလက ပေးထားသော failure continuation ကို ခေါ်ဆိုပါသည်။

amb evaluator အတွက် syntax နှင့် data-representation procedures များ၊ နှင့် အခြေခံ analyze procedure တို့သည် amb special form ကို အသိအမှတ်ပြုရန် နောက်ထပ် syntax procedures များ လိုအပ်သည်မှလွဲ၍ 4.1.7 ရှိ evaluator နှင့် ထပ်တူပင် ဖြစ်ပါသည်- 260 

(define (amb? exp) (tagged-list? exp 'amb))
(define (amb-choices exp) (cdr exp))

ကျွန်ုပ်တို့သည် ဤ special form ကို အသိအမှတ်ပြုပြီး သင့်လျော်သော execution procedure တစ်ခုကို ထုတ်ပေးမည့် clause တစ်ခုကို analyze ရှိ dispatch တွင် ထည့်သွင်းရပါမည်-

(define (ambeval exp env succeed fail)
  ((analyze exp) env succeed fail))

ကို လုပ်ဆောင်ခြင်းသည် ပေးထားသော ဖော်ပြချက်ကို အကဲဖြတ်ရန် ကြိုးပမ်းမည်ဖြစ်ပြီး၊ ဖော်ပြချက်၏ တန်ဖိုး (အကယ်၍ အကဲဖြတ်မှု အောင်မြင်လျှင်) သို့မဟုတ် failed ဟူသော သင်္ကေတ (အကယ်၍ အကဲဖြတ်မှု ကျရှုံးလျှင်) ကို ပြန်ပေးပါလိမ့်မည်။ အောက်တွင် ဖော်ပြထားသော driver loop ရှိ ambeval ကို ခေါ်ဆိုမှုသည် loop ကို ဆက်လက်လုပ်ဆောင်ပေးပြီး try-again တောင်းဆိုမှုကို အထောက်အပံ့ပေးသည့် ပိုမိုရှုပ်ထွေးသော continuation procedures များကို အသုံးပြုထားပါသည်။

amb evaluator ၏ ရှုပ်ထွေးမှု အများစုသည် execution procedures များ တစ်ခုနှင့်တစ်ခု ခေါ်ဆိုကြစဉ် continuations များကို လက်ဆင့်ကမ်းပေးသည့် ယန္တရားမှ ဖြစ်ပေါ်လာခြင်း ဖြစ်သည်။ အောက်ပါကုဒ်များကို လေ့လာရာတွင်၊ execution procedure တစ်ခုစီကို 4.1.7 တွင် ပေးထားသော ရိုးရိုး evaluator အတွက် သက်ဆိုင်ရာ procedure နှင့် နှိုင်းယှဉ်ကြည့်သင့်ပါသည်။

ရိုးရှင်းသော ဖော်ပြချက်များ (Simple expressions)

အရှင်းဆုံး ဖော်ပြချက် အမျိုးအစားများအတွက် execution procedures များသည် continuations များကို စီမံခန့်ခွဲရန် လိုအပ်သည်မှလွဲ၍ ရိုးရိုး evaluator အတွက် လုပ်ထုံးလုပ်နည်းများနှင့် အခြေခံအားဖြင့် တူညီကြပါသည်။ execution procedures များသည် ၎င်းတို့ထံ ပေးလိုက်သော failure continuation ကို လက်ဆင့်ကမ်းပေးပြီး ဖော်ပြချက်၏ တန်ဖိုးနှင့်အတူ ရိုးရှင်းစွာ အောင်မြင်သွားပါသည်။ 

(define (analyze-self-evaluating exp)
  (lambda (env succeed fail)
    (succeed exp fail)))

(define (analyze-quoted exp)
  (let ((qval (text-of-quotation exp)))
    (lambda (env succeed fail)
      (succeed qval fail))))

(define (analyze-variable exp)
  (lambda (env succeed fail)
    (succeed (lookup-variable-value exp env)
             fail)))

(define (analyze-lambda exp)
  (let ((vars (lambda-parameters exp))
        (bproc (analyze-sequence 
                (lambda-body exp))))
    (lambda (env succeed fail)
      (succeed (make-procedure vars bproc env)
               fail))))

ကိန်းရှင်တစ်ခုကို ရှာဖွေခြင်း (looking up) သည် အမြဲတမ်း 'အောင်မြင်' (succeeds) ကြောင်း သတိပြုပါ။ အကယ်၍ lookup-variable-value က ကိန်းရှင်ကို ရှာမတွေ့ပါက၊ ထုံးစံအတိုင်း အမှားတစ်ခုကို အကြောင်းကြားပါလိမ့်မည်။ ထိုကဲ့သို့သော "ကျရှုံးမှု" မျိုးသည် ပရိုဂရမ်၏ ချို့ယွင်းချက် (သတ်မှတ်မထားသော ကိန်းရှင်တစ်ခုကို ညွှန်းဆိုခြင်း) ကို ညွှန်ပြခြင်း ဖြစ်သည်။ ၎င်းသည် လက်ရှိကြိုးပမ်းနေသော ရွေးချယ်မှုအစား အခြား nondeterministic ရွေးချယ်မှုတစ်ခုကို စမ်းကြည့်သင့်သည်ဟု ညွှန်ပြခြင်း မဟုတ်ပါ။

Conditionals နှင့် sequences များ

Conditionals များကိုလည်း ရိုးရိုး evaluator မှာကဲ့သို့ပင် ဆင်တူသော နည်းလမ်းဖြင့် ကိုင်တွယ်ပါသည်။ analyze-if မှ ထုတ်ပေးလိုက်သော execution procedure သည် predicate တန်ဖိုး မှန်မမှန် စစ်ဆေးပြီး consequent သို့မဟုတ် alternative ကို ဆက်လက်လုပ်ဆောင်မည့် success continuation နှင့်အတူ predicate execution procedure pproc ကို ခေါ်ဆိုပါသည်။ အကယ်၍ pproc ကို လုပ်ဆောင်မှု ကျရှုံးပါက၊ if ဖော်ပြချက်အတွက် မူလ failure continuation ကို ခေါ်ဆိုပါသည်။ 

(define (analyze-if exp)
  (let ((pproc (analyze (if-predicate exp)))
        (cproc (analyze (if-consequent exp)))
        (aproc (analyze (if-alternative exp))))
    (lambda (env succeed fail)
      (pproc env
             ;; success continuation for evaluating
             ;; the predicate to obtain pred-value
             (lambda (pred-value fail2)
               (if (true? pred-value)
                   (cproc env succeed fail2)
                   (aproc env succeed fail2)))
             ;; failure continuation for
             ;; evaluating the predicate
             fail))))

Sequences များကိုလည်း continuations များ လက်ဆင့်ကမ်းရန် လိုအပ်သော subprocedure sequentially ရှိ လုပ်ဆောင်ချက်များမှလွဲ၍ ယခင် evaluator မှာကဲ့သို့ပင် ကိုင်တွယ်ပါသည်။ ဆိုလိုသည်မှာ a နှင့် b တို့ကို အစီအစဉ်အတိုင်း လုပ်ဆောင်ရန်အတွက်၊ ကျွန်ုပ်တို့သည် b ကို ခေါ်ဆိုမည့် success continuation နှင့်အတူ a ကို ခေါ်ဆိုပါသည်။ 

(define (analyze-sequence exps)
  (define (sequentially a b)
    (lambda (env succeed fail)
      (a env
         ;; success continuation for calling a
         (lambda (a-value fail2)
           (b env succeed fail2))
         ;; failure continuation for calling a
         fail)))
  (define (loop first-proc rest-procs)
    (if (null? rest-procs)
        first-proc
        (loop (sequentially first-proc 
                            (car rest-procs))
              (cdr rest-procs))))
  (let ((procs (map analyze exps)))
    (if (null? procs)
        (error "Empty sequence: ANALYZE"))
    (loop (car procs) (cdr procs))))
Definitions နှင့် assignments များ

သတ်မှတ်ချက်များ (Definitions) သည်လည်း continuations များကို စီမံခန့်ခွဲရန် အခက်အခဲအချို့ ရှိနိုင်သည့် အခြားကိစ္စတစ်ခု ဖြစ်သည်။ အဘယ်ကြောင့်ဆိုသော် variable အသစ်တစ်ခုကို အမှန်တကယ် မသတ်မှတ်မီ definition-value expression ကို အရင်ဆုံး evaluate လုပ်ရန် လိုအပ်သောကြောင့် ဖြစ်သည်။ ဤအချက်ကို ဆောင်ရွက်ရန်အတွက် definition-value execution procedure vproc ကို environment၊ success continuation နှင့် failure continuation တို့ဖြင့် ခေါ်ယူသည်။ အကယ်၍ vproc ကို execute လုပ်ခြင်း အောင်မြင်ပြီး သတ်မှတ်ထားသော variable အတွက် value val တစ်ခု ရရှိခဲ့လျှင် variable ကို သတ်မှတ်ပြီး success ကို ဆက်လက် ပေးပို့သည်- 

(define (analyze-definition exp)
  (let ((var (definition-variable exp))
        (vproc (analyze 
                (definition-value exp))))
    (lambda (env succeed fail)
      (vproc env
             (lambda (val fail2)
               (define-variable! var val env)
               (succeed 'ok fail2))
             fail))))

Assignments များမှာ ပို၍ စိတ်ဝင်စားစရာ ကောင်းသည်။ ဤနေရာသည် continuations များကို လက်ဆင့်ကမ်းရုံတင် မဟုတ်ဘဲ အမှန်တကယ် စတင် အသုံးပြုသည့် ပထမဆုံး နေရာဖြစ်သည်။ Assignments အတွက် execution procedure သည် definitions အတွက် procedure နှင့် ဆင်တူစွာ စတင်သည်။ ၎င်းသည် variable တွင် assign လုပ်မည့် value အသစ်ကို ရယူရန် ဦးစွာ ကြိုးပမ်းသည်။ အကယ်၍ vproc ကို evaluate လုပ်ခြင်း ကျရှုံးခဲ့လျှင် assignment သည်လည်း ကျရှုံးမည် ဖြစ်သည်။

သို့သော် vproc အောင်မြင်ပြီး assignment ကို ဆက်လက် လုပ်ဆောင်သည့်အခါ၊ ဤ computation ၏ အစိတ်အပိုင်းသည် နောက်ပိုင်းတွင် ကျရှုံးသွားနိုင်သည့် ဖြစ်နိုင်ခြေကို ထည့်သွင်းစဉ်းစားရမည်။ ထိုသို့ ကျရှုံးပါက assignment မလုပ်မီ အခြေအနေသို့ ပြန်လှည့်ရန် (backtrack လုပ်ရန်) လိုအပ်မည် ဖြစ်သည်။ ထို့ကြောင့် backtracking process ၏ အစိတ်အပိုင်းအဖြစ် assignment ကို ပြန်လည် ပယ်ဖျက်ရန် (undo လုပ်ရန်) စီစဉ်ရမည် ဖြစ်သည်။261

ဤအချက်ကို ဆောင်ရွက်ရန်အတွက် vproc ကို success continuation (အောက်တွင် “*1*” ဟု မှတ်ချက်ပေးထားသည်) တစ်ခု ပေးရသည်။ ၎င်းသည် value အသစ်ကို variable ထဲသို့ assign မလုပ်မီ variable ၏ value အဟောင်းကို သိမ်းဆည်းထားပြီးမှ assignment ကို ဆက်လက် လုပ်ဆောင်သည်။ Assignment value နှင့်အတူ ပေးပို့လိုက်သော failure continuation (အောက်တွင် “*2*” ဟု မှတ်ချက်ပေးထားသည်) သည် failure ကို ဆက်လက် မလုပ်ဆောင်မီ variable ၏ value အဟောင်းကို ပြန်လည် သတ်မှတ်ပေးသည်။ ဆိုလိုသည်မှာ အောင်မြင်သော assignment တစ်ခုသည် နောက်ဆက်တွဲ ကျရှုံးမှုတစ်ခုကို ကြားဖြတ် ဟန့်တားမည့် failure continuation တစ်ခုကို ထောက်ပံ့ပေးသည်။ အခြားနည်းဖြင့် fail2 ကို ခေါ်မည့် မည်သည့် ကျရှုံးမှုမျိုးမဆို fail2 ကို အမှန်တကယ် မခေါ်မီ assignment ကို undo လုပ်ရန်အတွက် ဤ procedure ကို အစားထိုး ခေါ်ယူမည် ဖြစ်သည်။ 

(define (analyze-assignment exp)
  (let ((var (assignment-variable exp))
        (vproc (analyze 
                (assignment-value exp))))
    (lambda (env succeed fail)
      (vproc env
             (lambda (val fail2)    ; *1*
               (let ((old-value
                      (lookup-variable-value 
                       var 
                       env)))
                 (set-variable-value!
                  var 
                  val 
                  env)
                 (succeed 
                  'ok
                  (lambda ()    ; *2*
                    (set-variable-value! 
                     var
                     old-value
                     env)
                    (fail2)))))
               fail))))
Procedure application များ

Applications များအတွက် execution procedure တွင် continuations များကို စီမံခန့်ခွဲရာ၌ တွေ့ရသော နည်းပညာဆိုင်ရာ ရှုပ်ထွေးမှုများမှလွဲ၍ အတွေးအခေါ်သစ်များ မပါဝင်ပါ။ ဤရှုပ်ထွေးမှုသည် operands များကို evaluate လုပ်နေစဉ် success နှင့် failure continuations များကို ခြေရာခံရန် လိုအပ်သောကြောင့် analyze-application တွင် ဖြစ်ပေါ်လာခြင်း ဖြစ်သည်။ ကျွန်ုပ်တို့သည် သာမန် evaluator တွင်ကဲ့သို့ ရိုးရှင်းသော map ကို အသုံးမပြုဘဲ operands စာရင်းကို evaluate လုပ်ရန် get-args procedure ကို အသုံးပြုသည်။ 

(define (analyze-application exp)
  (let ((fproc (analyze (operator exp)))
        (aprocs (map analyze (operands exp))))
    (lambda (env succeed fail)
      (fproc env
             (lambda (proc fail2)
               (get-args 
                aprocs
                env
                (lambda (args fail3)
                  (execute-application
                   proc args succeed fail3))
                fail2))
             fail))))

get-args တွင် aproc execution procedures စာရင်းကို cdr-ing လုပ်ခြင်းနှင့် ရလဒ် args စာရင်းကို consing လုပ်ခြင်းတို့ကို၊ စာရင်းထဲရှိ aproc တစ်ခုစီကို get-args ကို recursive ခေါ်ယူမည့် success continuation ဖြင့် ခေါ်ခြင်းအားဖြင့် မည်သို့ လုပ်ဆောင်သည်ကို သတိပြုပါ။ ဤ get-args သို့ recursive ခေါ်ယူမှု တစ်ခုစီတွင် success continuation တစ်ခုရှိပြီး ၎င်း၏ value မှာ အသစ်ရရှိလာသော argument ကို စုဆောင်းထားသော argument များစာရင်းသို့ cons လုပ်ထားသော value ဖြစ်သည်- 

(define (get-args aprocs env succeed fail)
  (if (null? aprocs)
      (succeed '() fail)
      ((car aprocs) 
       env
       ;; success continuation for this aproc
       (lambda (arg fail2)
         (get-args 
          (cdr aprocs)
          env
          ;; success continuation for
          ;; recursive call to get-args
          (lambda (args fail3)
            (succeed (cons arg args)
                     fail3))
          fail2))
       fail)))

execute-application မှ လုပ်ဆောင်သော အမှန်တကယ် procedure application သည် continuations များကို စီမံခန့်ခွဲရန် လိုအပ်သည်မှလွဲ၍ သာမန် evaluator တွင်ကဲ့သို့ပင် ဆောင်ရွက်သည်။ 

(define (execute-application 
         proc args succeed fail)
  (cond ((primitive-procedure? proc)
         (succeed 
          (apply-primitive-procedure 
           proc args)
          fail))
        ((compound-procedure? proc)
         ((procedure-body proc)
          (extend-environment 
           (procedure-parameters proc)
           args
           (procedure-environment proc))
          succeed
          fail))
        (else (error "Unknown procedure type: 
                      EXECUTE-APPLICATION"
                     proc))))

Driver loop

amb evaluator အတွက် driver loop သည် expression တစ်ခုကို evaluate လုပ်ရာတွင် အသုံးပြုသူမှ နောက်တစ်ကြိမ် ထပ်မံကြိုးစားနိုင်စေရန် ခွင့်ပြုထားသော လုပ်ဆောင်ချက်ကြောင့် ရှုပ်ထွေးပါသည်။ driver သည် internal-loop ဟုခေါ်သော procedure တစ်ခုကို အသုံးပြုပြီး ၎င်းသည် try-again ဟုခေါ်သော procedure တစ်ခုကို argument အဖြစ် လက်ခံပါသည်။ ရည်ရွယ်ချက်မှာ try-again ကို ခေါ်ယူလိုက်ခြင်းဖြင့် nondeterministic evaluation ရှိ နောက်ထပ် မစမ်းသပ်ရသေးသော alternative တစ်ခုသို့ ဆက်လက်လုပ်ဆောင်ရန် ဖြစ်ပါသည်။ Internal-loop သည် အသုံးပြုသူမှ driver loop တွင် try-again ဟု ရိုက်နှိပ်လိုက်ပါက try-again ကို ခေါ်ယူမည်ဖြစ်ပြီး၊ သို့မဟုတ်ပါက ambeval ကို ခေါ်ယူခြင်းဖြင့် evaluation အသစ်တစ်ခုကို စတင်မည် ဖြစ်သည်။

ဤ ambeval ခေါ်ယူမှုအတွက် failure continuation သည် နောက်ထပ် value များ မရှိတော့ကြောင်း အသုံးပြုသူကို အသိပေးပြီး driver loop ကို ပြန်လည်စတင်စေသည်။

ambeval ခေါ်ယူမှုအတွက် success continuation သည် ပိုမိုနက်နဲပါသည်။ ကျွန်ုပ်တို့သည် ရရှိလာသော value ကို print ထုတ်ပြီးနောက် နောက်ထပ် alternative တစ်ခုကို ထပ်မံစမ်းသပ်နိုင်မည့် try-again procedure ဖြင့် internal loop ကို ပြန်လည်ခေါ်ယူပါသည်။ ဤ next-alternative procedure သည် success continuation ထံသို့ ပေးပို့လိုက်သော ဒုတိယမြောက် argument ဖြစ်သည်။ ပုံမှန်အားဖြင့်၊ ဤဒုတိယမြောက် argument ကို လက်ရှိ evaluation branch သည် နောက်ပိုင်းတွင် အဆင်မပြေဖြစ်ခဲ့ပါက အသုံးပြုရန် failure continuation တစ်ခုအဖြစ် ကျွန်ုပ်တို့ ယူဆကြသည်။ သို့သော် ဤကိစ္စတွင်မူ၊ ကျွန်ုပ်တို့သည် အောင်မြင်သော evaluation တစ်ခုကို ပြီးမြောက်ခဲ့ပြီးဖြစ်သောကြောင့် အခြားသော အောင်မြင်နိုင်မည့် evaluation များကို ရှာဖွေနိုင်ရန်အတွက် “failure” alternative branch ကို ခေါ်ယူနိုင်ခြင်း ဖြစ်ပါသည်။

255 parse-word ဟာ unparsed input list ကို ပြင်ဆင်ဖို့ set! ကို အသုံးပြုထားတာကို သတိပြုပါ။ ဒါဟာ အလုပ်လုပ်နိုင်ဖို့အတွက်၊ ကျွန်ုပ်တို့ရဲ့ amb evaluator ဟာ backtrack လုပ်တဲ့အခါ set! ရဲ့ သက်ရောက်မှုတွေကို ပြန်လည်ပြင်ဆင် (undo) ပေးရပါမယ်။

256 ဒီ definition ဟာ recursive ဖြစ်နေတာကို သတိပြုပါ—verb တစ်ခုရဲ့ နောက်မှာ prepositional phrases အရေအတွက် မည်မျှမဆို လိုက်ပါနိုင်ပါတယ်။

257 ဒီလို grammar မျိုးဟာ အကန့်အသတ်မရှိ ရှုပ်ထွေးသွားနိုင်ပေမဲ့၊ စစ်မှန်တဲ့ ဘာသာစကား နားလည်နိုင်စွမ်း (real language understanding) နဲ့ ယှဉ်ရင်တော့ ကစားစရာ (toy) တစ်ခုမျှသာ ဖြစ်ပါတယ်။ ကွန်ပျူတာကနေ စစ်မှန်တဲ့ သဘာဝဘာသာစကားကို နားလည်နိုင်ဖို့အတွက် syntactic analysis နဲ့ အဓိပ္ပာယ် ဖွင့်ဆိုချက် (interpretation of meaning) တို့ကို စနစ်တကျ ပေါင်းစပ်ဖို့ လိုအပ်ပါတယ်။ အခြားတစ်ဖက်မှာတော့၊ သေးငယ်တဲ့ toy parsers လေးတွေတောင်မှ information-retrieval systems လိုမျိုး ပရိုဂရမ်တွေအတွက် လွယ်ကူပြောင်းလွယ်တဲ့ command languages တွေကို ပံ့ပိုးပေးရာမှာ အသုံးဝင်နိုင်ပါတယ်။ Winston 1992 မှာ စစ်မှန်တဲ့ ဘာသာစကား နားလည်မှုအတွက် တွက်ချက်မှုဆိုင်ရာ ချဉ်းကပ်ပုံတွေနဲ့ command languages တွေမှာ simple grammars တွေရဲ့ အသုံးချပုံတွေကို ဆွေးနွေးထားပါတယ်။

258 Alyssa ရဲ့ စိတ်ကူးဟာ ကောင်းကောင်း အလုပ်လုပ်သော်လည်း (ပြီးတော့ အံ့သြစရာကောင်းလောက်အောင်လည်း ရိုးရှင်းပါတယ်)၊ ၎င်းကနေ ထုတ်ပေးလိုက်တဲ့ ဝါကျတွေဟာ အနည်းငယ် ပျင်းစရာကောင်းပါတယ်—အဲဒီဝါကျတွေဟာ ဒီဘာသာစကားရဲ့ ဖြစ်နိုင်ခြေရှိတဲ့ ဝါကျတွေကို စိတ်ဝင်စားစရာကောင်းတဲ့ ပုံစံမျိုးနဲ့ ဖော်ပြမပေးနိုင်ပါဘူး။ တကယ်တော့ အဲဒီ grammar ဟာ နေရာတော်တော်များများမှာ recursive အလွန်ဖြစ်နေပြီး၊ Alyssa ရဲ့ နည်းစနစ်ဟာ အဲဒီ recursion တွေထဲက တစ်ခုထဲမှာ “ကျရှုံး” ပြီး ပိတ်မိသွားတတ်ပါတယ်။ ဒါကို ဖြေရှင်းဖို့ နည်းလမ်းအတွက် လေ့ကျင့်ခန်း 4.50 ကို ကြည့်ပါ။

259 အပိုင်း 4.2 မှာ lazy evaluator ကို အကောင်အထည်ဖော်ဖို့အတွက် အပိုင်း 4.1.1 ရဲ့ သာမန် metacircular evaluator ကို ပြင်ဆင်ပြီး အသုံးပြုခဲ့ပါတယ်။ ဒါနဲ့ ဆန့်ကျင်ဘက်အနေနဲ့၊ amb evaluator အတွက်တော့ အပိုင်း 4.1.7 ရဲ့ analyzing evaluator ကို အခြေခံသွားမှာ ဖြစ်ပါတယ်၊ ဘာကြောင့်လဲဆိုတော့ အဲဒီ evaluator မှာပါတဲ့ execution procedures တွေဟာ backtracking ကို အကောင်အထည်ဖော်ဖို့အတွက် အဆင်ပြေတဲ့ မူဘောင်တစ်ခုကို ပံ့ပိုးပေးထားလို့ ဖြစ်ပါတယ်။

260 Evaluator ဟာ let (အပိုင်း လေ့ကျင့်ခန်း 4.22 ကိုကြည့်ပါ) ကို ပံ့ပိုးပေးထားတယ်လို့ ကျွန်ုပ်တို့ ယူဆထားပါတယ်၊ ၎င်းကို ကျွန်ုပ်တို့ရဲ့ nondeterministic programs တွေမှာ အသုံးပြုထားပါတယ်။

261 Internal definitions တွေကို scan out လုပ်တယ်လို့ (4.1.6) ယူဆနိုင်တဲ့အတွက်၊ definitions တွေကို ပြန်ဖျက်ဖို့ (undoing) အတွက် ကျွန်ုပ်တို့ စိုးရိမ်စရာ မလိုပါဘူး။
