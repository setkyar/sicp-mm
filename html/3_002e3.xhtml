3.3ပြောင်းလဲနိုင်သော ဒေတာများဖြင့် ပုံစံထုတ်ခြင်း (Modeling with Mutable Data)

အခန်း ၂ (Chapter 2) တွင် ကဏ္ဍအသီးသီးရှိသော လက်တွေ့ကမ္ဘာမှ အရာဝတ္ထုများကို ပုံစံထုတ်ရန်အတွက် အစိတ်အပိုင်း အများအပြားပါဝင်သော တွက်ချက်မှုဆိုင်ရာ အရာဝတ္ထုများကို တည်ဆောက်သည့် နည်းလမ်းတစ်ခုအဖြစ် ပေါင်းစပ်ဒေတာ (compound data) အကြောင်းကို ကိုင်တွယ်ဖြေရှင်းခဲ့သည်။ ထိုအခန်းတွင် ဒေတာ တည်ဆောက်ပုံများကို ဒေတာအရာဝတ္ထုများကို ဖန်တီးပေးသည့် constructor များနှင့် ပေါင်းစပ်ဒေတာအရာဝတ္ထုများ၏ အစိတ်အပိုင်းများကို ရယူပေးသည့် selector များအဖြစ် သတ်မှတ်ပေးသည့် ဒေတာ အဆင့်ဆင့်ခွဲထုတ်ခြင်း (data abstraction) စည်းကမ်းကို ကျွန်ုပ်တို့ မိတ်ဆက်ခဲ့သည်။ သို့သော် အခန်း ၂ တွင် မဖော်ပြထားသော ဒေတာ၏ အခြားကဏ္ဍတစ်ခု ရှိသေးသည်ကို ယခုအခါ ကျွန်ုပ်တို့ သိရှိရပြီဖြစ်သည်။ ပြောင်းလဲနေသော အခြေအနေ (changing state) ရှိသည့် အရာဝတ္ထုများဖြင့် ဖွဲ့စည်းထားသော စနစ်များကို ပုံစံထုတ်လိုသော ဆန္ဒကြောင့် ပေါင်းစပ်ဒေတာအရာဝတ္ထုများကို တည်ဆောက်ခြင်းနှင့် ရွေးချယ်ခြင်းသာမက ၎င်းတို့ကို ပြင်ဆင်ပြောင်းလဲရန် (modify) လိုအပ်ချက်ကိုလည်း ဖြစ်ပေါ်စေသည်။ ပြောင်းလဲနေသော အခြေအနေရှိသည့် ပေါင်းစပ်အရာဝတ္ထုများကို ပုံစံထုတ်ရန်အတွက် ကျွန်ုပ်တို့သည် selector များနှင့် constructor များအပြင် ဒေတာအရာဝတ္ထုများကို ပြင်ဆင်ပြောင်းလဲပေးသည့် mutators ဟုခေါ်သော လုပ်ဆောင်ချက်များပါဝင်သော ဒေတာ အဆင့်ဆင့်ခွဲထုတ်ခြင်းများကို ဒီဇိုင်းထုတ်ပါမည်။ ဥပမာအားဖြင့် ဘဏ်လုပ်ငန်းစနစ်တစ်ခုကို ပုံစံထုတ်ရာတွင် အကောင့်လက်ကျန်ငွေများကို ပြောင်းလဲရန် လိုအပ်သည်။ ထို့ကြောင့် ဘဏ်အကောင့်များကို ကိုယ်စားပြုသည့် ဒေတာတည်ဆောက်ပုံတစ်ခုသည် အောက်ပါလုပ်ဆောင်ချက်ကို လက်ခံနိုင်သည် -

၎င်းသည် သတ်မှတ်ထားသော အကောင့်၏ လက်ကျန်ငွေကို သတ်မှတ်ထားသော တန်ဖိုးအသစ်သို့ ပြောင်းလဲပေးသည်။ mutator များ သတ်မှတ်ထားသော ဒေတာအရာဝတ္ထုများကို ပြောင်းလဲနိုင်သော ဒေတာအရာဝတ္ထုများ (mutable data objects) ဟု ခေါ်သည်။

အခန်း ၂ တွင် ပေါင်းစပ်ဒေတာများ ဖန်တီးရန်အတွက် အထွေထွေသုံး "ကော်" (glue) အဖြစ် pairs များကို မိတ်ဆက်ခဲ့သည်။ pairs များသည် ပြောင်းလဲနိုင်သော ဒေတာအရာဝတ္ထုများကို တည်ဆောက်ရန်အတွက် အခြေခံအုတ်မြစ်များအဖြစ် လုပ်ဆောင်နိုင်ရန် pairs များအတွက် အခြေခံ mutator များကို သတ်မှတ်ခြင်းဖြင့် ဤကဏ္ဍကို စတင်ပါမည်။ ဤ mutator များသည် pairs များ၏ ကိုယ်စားပြုနိုင်စွမ်းကို များစွာ မြှင့်တင်ပေးပြီး 2.2 တွင် ကျွန်ုပ်တို့ အသုံးပြုခဲ့သည့် sequences များနှင့် trees များအပြင် အခြားသော ဒေတာတည်ဆောက်ပုံများကို တည်ဆောက်နိုင်စေသည်။ ထို့အပြင် ရှုပ်ထွေးသော စနစ်များကို ဒေသန္တရ အခြေအနေ (local state) ရှိသည့် အရာဝတ္ထုများ စုစည်းမှုအဖြစ် ပုံစံထုတ်ထားသည့် သရုပ်ပြပုံစံငယ် (simulations) အချို့ကိုလည်း တင်ပြပါမည်။

pair များ၏ အခြေခံလုပ်ဆောင်ချက်များဖြစ်သော — cons၊ car၊ နှင့် cdr — တို့ကို list တည်ဆောက်ပုံများ တည်ဆောက်ရန်နှင့် list တည်ဆောက်ပုံမှ အစိတ်အပိုင်းများကို ရွေးချယ်ရန် အသုံးပြုနိုင်သော်လည်း၊ ၎င်းတို့သည် list တည်ဆောက်ပုံကို ပြောင်းလဲပြင်ဆင်နိုင်စွမ်း မရှိကြပါ။ ယခုအချိန်အထိ ကျွန်ုပ်တို့အသုံးပြုခဲ့သည့် append နှင့် list ကဲ့သို့သော list လုပ်ဆောင်ချက်များသည်လည်း ထိုနည်းအတိုင်းပင် ဖြစ်ပါသည်။ အဘယ်ကြောင့်ဆိုသော် ၎င်းတို့ကို cons၊ car၊ နှင့် cdr တို့ဖြင့် အဓိပ္ပာယ်ဖွင့်ဆိုနိုင်သောကြောင့် ဖြစ်သည်။ List တည်ဆောက်ပုံများကို ပြောင်းလဲပြင်ဆင်ရန်အတွက် ကျွန်ုပ်တို့တွင် လုပ်ဆောင်ချက်အသစ်များ လိုအပ်ပါသည်။

  (let ((new (get-new-pair)))
    (set-car! new x)
    (set-cdr! new y)
    new))

(define (assoc key records)
  (cond ((null? records) false)
        ((equal? key (caar records)) 
         (car records))
        (else (assoc key (cdr records)))))

Key အစုံတစ်ခုအောက်တွင် အချက်အလက်အသစ်တစ်ခု ထည့်သွင်းရန်အတွက်၊ ပထမ key အောက်တွင် သိမ်းဆည်းထားသော subtable ရှိမရှိ သိရန် assoc ကို အသုံးပြုပါသည်။ မရှိပါက၊ ကျွန်ုပ်တို့သည် record တစ်ခုတည်း (key-2, value) ပါဝင်သော subtable အသစ်တစ်ခုကို တည်ဆောက်ပြီး ၎င်းကို ပထမ key အောက်တွင် table ထဲသို့ ထည့်သွင်းပါသည်။ အကယ်၍ ပထမ key အတွက် subtable ရှိနှင့်ပြီးသားဖြစ်ပါက၊ အထက်တွင် ဖော်ပြခဲ့သော တစ်ဖက်မြင် table များအတွက် ထည့်သွင်းနည်းစနစ်ကို အသုံးပြု၍ ဤ subtable ထဲသို့ record အသစ်ကို ထည့်သွင်းပါသည်။





(define (insert! key-1 key-2 value table)
  (let ((subtable (assoc key-1 (cdr table))))
    (if subtable
        (let ((record 
               (assoc key-2 (cdr subtable))))
          (if record
              (set-cdr! record value)
              (set-cdr! 
               subtable
               (cons (cons key-2 value)
                     (cdr subtable)))))
        (set-cdr! 
         table
         (cons (list key-1 (cons key-2 value))
               (cdr table)))))
  'ok)

(define (make-table)
  (let ((local-table (list '*table*)))
    (define (lookup key-1 key-2)
      (let ((subtable 
             (assoc key-1 (cdr local-table))))
        (if subtable
            (let ((record 
                   (assoc key-2 
                          (cdr subtable))))
              (if record (cdr record) false))
            false)))
    (define (insert! key-1 key-2 value)
      (let ((subtable 
             (assoc key-1 (cdr local-table))))
        (if subtable
            (let ((record 
                   (assoc key-2 
                          (cdr subtable))))
              (if record
                  (set-cdr! record value)
                  (set-cdr! 
                   subtable
                   (cons (cons key-2 value)
                         (cdr subtable)))))
            (set-cdr! 
             local-table
             (cons (list key-1
                         (cons key-2 value))
                   (cdr local-table)))))
      'ok)
    (define (dispatch m)
      (cond ((eq? m 'lookup-proc) lookup)
            ((eq? m 'insert-proc!) insert!)
            (else (error "Unknown operation: 
                          TABLE" m))))
    dispatch))

3.3.4ဒစ်ဂျစ်တယ်ဆားကစ်များအတွက် စီမျူလေတာတစ်ခု



ကွန်ပျူတာများကဲ့သို့ ရှုပ်ထွေးသော ဒစ်ဂျစ်တယ်စနစ်များကို ဒီဇိုင်းထုတ်ခြင်းသည် အရေးကြီးသော အင်ဂျင်နီယာလုပ်ငန်းတစ်ခုဖြစ်သည်။ ဒစ်ဂျစ်တယ်စနစ်များကို ရိုးရှင်းသော ဒြပ်စင်များကို အပြန်အလှန်ချိတ်ဆက်ခြင်းဖြင့် တည်ဆောက်သည်။ ဤအစိတ်အပိုင်းတစ်ခုချင်းစီ၏ လုပ်ဆောင်ပုံမှာ ရိုးရှင်းသော်လည်း ၎င်းတို့၏ ကွန်ရက်များသည် အလွန်ရှုပ်ထွေးသော လုပ်ဆောင်ပုံများ ရှိနိုင်သည်။ အဆိုပြုထားသော ဆားကစ်ဒီဇိုင်းများကို ကွန်ပျူတာဖြင့် စီမျူလေးရှင်းပြုလုပ်ခြင်းသည် ဒစ်ဂျစ်တယ်စနစ် အင်ဂျင်နီယာများ အသုံးပြုသော အရေးကြီးသော ကိရိယာတစ်ခုဖြစ်သည်။ ဤအပိုင်းတွင် ကျွန်ုပ်တို့သည် ဒစ်ဂျစ်တယ်လော့ဂျစ်စီမျူလေးရှင်းများ ပြုလုပ်ရန် စနစ်တစ်ခုကို ဒီဇိုင်းထုတ်ပါမည်။ ဤစနစ်သည်

ဖြစ်ရပ်အခြေပြုစီမျူလေးရှင်း (event-driven simulation) ဟုခေါ်သော ပရိုဂရမ်အမျိုးအစားတစ်ခုဖြစ်ပြီး၊ ယင်းတွင် လုပ်ဆောင်ချက်များ (“ဖြစ်ရပ်များ”) သည် နောက်ပိုင်းအချိန်တွင် ဖြစ်ပေါ်မည့် နောက်ထပ်ဖြစ်ရပ်များကို လှုံ့ဆော်ပေးကာ ၎င်းမှတစ်ဆင့် နောက်ထပ်ဖြစ်ရပ်များကို ဆက်လက်လှုံ့ဆော်ပေးခြင်း စသည်ဖြင့် ဖြစ်ပျက်သွားသည်။



ကျွန်ုပ်တို့၏ ဆားကစ်ဆိုင်ရာ တွက်ချက်မှုမော်ဒယ်ကို ဆားကစ်တည်ဆောက်ထားသော အခြေခံအစိတ်အပိုင်းများနှင့် ကိုက်ညီသည့် အရာဝတ္ထုများဖြင့် ဖွဲ့စည်းမည်ဖြစ်သည်။
ဒစ်ဂျစ်တယ်ဆစ်ဂနယ်များကို သယ်ဆောင်ပေးသည့် 
ဝိုင်ယာကြိုးများ (wires) ရှိပြီး၊ ယင်းတို့သည် 
ဒစ်ဂျစ်တယ်ဆစ်ဂနယ်များကို သယ်ဆောင်ကြသည်။ ဒစ်ဂျစ်တယ်ဆစ်ဂနယ်တစ်ခုသည် မည်သည့်အချိန်တွင်မဆို ဖြစ်နိုင်ခြေရှိသော တန်ဖိုးနှစ်ခုဖြစ်သည့် 0 နှင့် 1 ထဲမှ တစ်ခုသာ ရှိနိုင်သည်။ အဝင်ဆစ်ဂနယ်များကို သယ်ဆောင်လာသော ဝိုင်ယာကြိုးများကို အခြားအထွက်ဝိုင်ယာကြိုးများနှင့် ချိတ်ဆက်ပေးသည့် အမျိုးမျိုးသော ဒစ်ဂျစ်တယ် 
ဖန်ရှင်ဘောက်စ်များ (function boxes) လည်း ရှိသည်။ ထိုသို့သော ဘောက်စ်များသည် ၎င်းတို့၏ အဝင်ဆစ်ဂနယ်များမှ တွက်ချက်ထားသော အထွက်ဆစ်ဂနယ်များကို ထုတ်ပေးသည်။ အထွက်ဆစ်ဂနယ်သည် ဖန်ရှင်ဘောက်စ် အမျိုးအစားပေါ်မူတည်၍ အချိန်အတိုင်းအတာတစ်ခုအထိ ကြန့်ကြာမှု (delay) ရှိသည်။ ဥပမာအားဖြင့်၊

အင်ဗာတာ (inverter) သည် ၎င်း၏ အဝင်ကို ပြောင်းပြန်လှန်ပေးသော အခြေခံဖန်ရှင်ဘောက်စ်တစ်ခု ဖြစ်သည်။ အကယ်၍ အင်ဗာတာသို့ အဝင်ဆစ်ဂနယ်သည် 0 သို့ ပြောင်းလဲသွားပါက၊ inverter-delay တစ်ခုအကြာတွင် အင်ဗာတာသည် ၎င်း၏အထွက်ဆစ်ဂနယ်ကို 1 သို့ ပြောင်းလဲပေးမည်ဖြစ်သည်။ အကယ်၍ အင်ဗာတာသို့ အဝင်ဆစ်ဂနယ်သည် 1 သို့ ပြောင်းလဲသွားပါက၊ inverter-delay တစ်ခုအကြာတွင် အင်ဗာတာသည် ၎င်း၏အထွက်ဆစ်ဂနယ်ကို 0 သို့ ပြောင်းလဲပေးမည်ဖြစ်သည်။ ကျွန်ုပ်တို့သည် အင်ဗာတာကို ပုံ ၃.၂၄ တွင်ပြထားသည့်အတိုင်း သင်္ကေတဖြင့် ဆွဲကြသည်။
ပုံ ၃.၂၄ တွင်ပင် ဖော်ပြထားသော 
and-gate သည် အဝင်နှစ်ခုနှင့် အထွက်တစ်ခုပါရှိသော အခြေခံဖန်ရှင်ဘောက်စ်တစ်ခု ဖြစ်သည်။ ၎င်းသည် ၎င်း၏ အထွက်ဆစ်ဂနယ်ကို အဝင်များ၏ 
လော့ဂျစ်ကယ် and (logical and) တန်ဖိုးဖြစ်အောင် မောင်းနှင်ပေးသည်။ ဆိုလိုသည်မှာ ၎င်း၏အဝင်ဆစ်ဂနယ် နှစ်ခုစလုံး 1 ဖြစ်လာပါက၊ and-gate-delay တစ်ခုအကြာတွင် and-gate သည် ၎င်း၏အထွက်ဆစ်ဂနယ်ကို 1 ဖြစ်ရန် တွန်းအားပေးမည်ဖြစ်ပြီး၊ မဟုတ်ပါက အထွက်သည် 0 ဖြစ်လိမ့်မည်။

or-gate သည် အဝင်နှစ်ခုပါသော အလားတူအခြေခံဖန်ရှင်ဘောက်စ်တစ်ခုဖြစ်ပြီး ၎င်းသည် ၎င်း၏အထွက်ဆစ်ဂနယ်ကို အဝင်များ၏ 
လော့ဂျစ်ကယ် or (logical or) တန်ဖိုးဖြစ်အောင် မောင်းနှင်ပေးသည်။ ဆိုလိုသည်မှာ အနည်းဆုံး အဝင်ဆစ်ဂနယ်တစ်ခုသည် 1 ဖြစ်ပါက အထွက်သည် 1 ဖြစ်လာမည်ဖြစ်ပြီး၊ မဟုတ်ပါက အထွက်သည် 0 ဖြစ်လာမည်ဖြစ်သည်။





ပိုမိုရှုပ်ထွေးသော ဖန်ရှင်များကို တည်ဆောက်ရန်အတွက် အခြေခံဖန်ရှင်များကို အတူတကွ ချိတ်ဆက်နိုင်သည်။ ယင်းကို လုပ်ဆောင်ရန်အတွက် အချို့သော ဖန်ရှင်ဘောက်စ်များ၏ အထွက်များကို အခြားဖန်ရှင်ဘောက်စ်များ၏ အဝင်များသို့ ဝိုင်ယာကြိုးဖြင့် ချိတ်ဆက်ပေးသည်။ ဥပမာအားဖြင့်၊ ပုံ ၃.၂၅ တွင် ပြထားသော 
half-adder ဆားကစ်တွင် or-gate တစ်ခု၊ and-gate နှစ်ခုနှင့် အင်ဗာတာတစ်ခု ပါဝင်သည်။ ၎င်းသည် အဝင်ဆစ်ဂနယ်နှစ်ခု A နှင့် B ကို လက်ခံပြီး အထွက်ဆစ်ဂနယ်နှစ်ခု S နှင့် C ရှိသည်။ A နှင့် B ထဲမှ တစ်ခုသာလျှင် တိတိကျကျ 1 ဖြစ်သည့်အခါတိုင်း S သည် 1 ဖြစ်လာမည်ဖြစ်ပြီး၊ A နှင့် B နှစ်ခုစလုံး 1 ဖြစ်သည့်အခါတိုင်း C သည် 1 ဖြစ်လာမည်ဖြစ်သည်။ ဆက်စပ်နေသော ကြန့်ကြာမှုများကြောင့် အထွက်များကို မတူညီသောအချိန်များတွင် ထုတ်ပေးနိုင်သည်ကို ပုံတွင် မြင်နိုင်သည်။ ဒစ်ဂျစ်တယ်ဆားကစ်များ ဒီဇိုင်းထုတ်ရာတွင် ကြုံတွေ့ရသော အခက်အခဲများစွာသည် ဤအချက်ကြောင့် ဖြစ်ပေါ်လာခြင်းဖြစ်သည်။





ကျွန်ုပ်တို့ လေ့လာလိုသော ဒစ်ဂျစ်တယ်လော့ဂျစ်ဆားကစ်များကို မော်ဒယ်ပြုလုပ်ရန် ပရိုဂရမ်တစ်ခုကို ယခု တည်ဆောက်ပါမည်။ ထိုပရိုဂရမ်သည် ဆစ်ဂနယ်များကို “ထိန်းသိမ်း” ထားမည့် ဝိုင်ယာကြိုးများကို မော်ဒယ်ပြုလုပ်ထားသော တွက်ချက်မှုဆိုင်ရာ အရာဝတ္ထုများကို တည်ဆောက်မည်ဖြစ်သည်။ ဖန်ရှင်ဘောက်စ်များကို ဆစ်ဂနယ်များအကြား မှန်ကန်သော ဆက်နွယ်မှုများကို ဖြစ်ပေါ်စေမည့် ပရိုစီဂျာများဖြင့် မော်ဒယ်ပြုလုပ်မည်ဖြစ်သည်။



ကျွန်ုပ်တို့၏ စီမျူလေးရှင်း၏ အခြေခံဒြပ်စင်တစ်ခုမှာ ဝိုင်ယာကြိုးများကို တည်ဆောက်ပေးသည့် make-wire ပရိုစီဂျာ ဖြစ်လိမ့်မည်။ ဥပမာအားဖြင့်၊ ကျွန်ုပ်တို့သည် ဝိုင်ယာကြိုး ခြောက်ခုကို အောက်ပါအတိုင်း တည်ဆောက်နိုင်သည်-





(define a (make-wire))
(define b (make-wire))
(define c (make-wire))
(define d (make-wire))
(define e (make-wire))
(define s (make-wire))



ထိုသို့သော ဘောက်စ်အမျိုးအစားကို တည်ဆောက်ပေးသည့် ပရိုစီဂျာတစ်ခုကို ခေါ်ဆိုခြင်းဖြင့် ဝိုင်ယာကြိုးအစုအဝေးတစ်ခုသို့ ဖန်ရှင်ဘောက်စ်တစ်ခုကို ချိတ်ဆက်သည်။ constructor ပရိုစီဂျာသို့ ပေးပို့သော အကြောင်းပြချက်များ (arguments) မှာ ဘောက်စ်သို့ ချိတ်ဆက်မည့် ဝိုင်ယာကြိုးများ ဖြစ်သည်။ ဥပမာအားဖြင့်၊ and-gates၊ or-gates နှင့် အင်ဗာတာများကို တည်ဆောက်နိုင်သည်ဟု ယူဆလျှင် ပုံ ၃.၂၅ တွင် ပြထားသော half-adder ကို အောက်ပါအတိုင်း ဝိုင်ယာဖြင့် ချိတ်ဆက်နိုင်သည်-





(or-gate a b d)
ok

(and-gate a b c)
ok

(inverter c e)
ok

(and-gate d e s)
ok




ထို့ထက်ပို၍ ကောင်းသည်မှာ၊ half-adder သို့ ချိတ်ဆက်ရမည့် ပြင်ပဝိုင်ယာကြိုး လေးခုကို ပေးထားပြီး ဤဆားကစ်ကို တည်ဆောက်ပေးသည့် half-adder ပရိုစီဂျာတစ်ခုကို အဓိပ္ပာယ်ဖွင့်ဆိုခြင်းဖြင့် ဤလုပ်ဆောင်ချက်ကို အတိအလင်း အမည်ပေးနိုင်သည်။





(define (half-adder a b s c)
  (let ((d (make-wire)) (e (make-wire)))
    (or-gate a b d)
    (and-gate a b c)
    (inverter c e)
    (and-gate d e s)
    'ok))



ဤကဲ့သို့ အဓိပ္ပာယ်ဖွင့်ဆိုခြင်း၏ အားသာချက်မှာ ပိုမိုရှုပ်ထွေးသော ဆားကစ်များကို ဖန်တီးရာတွင် half-adder ကိုယ်တိုင်ကို အခြေခံအစိတ်အပိုင်းတစ်ခုအဖြစ် အသုံးပြုနိုင်ခြင်းပင် ဖြစ်သည်။ ပုံ ၃.၂၆ တွင်၊ ဥပမာအားဖြင့်၊ half-adder နှစ်ခုနှင့် or-gate တစ်ခုဖြင့် ဖွဲ့စည်းထားသော 
full-adder တစ်ခုကို ပြသထားသည်။154 ကျွန်ုပ်တို့သည် full-adder တစ်ခုကို အောက်ပါအတိုင်း တည်ဆောက်နိုင်သည်-





(define (full-adder a b c-in sum c-out)
  (let ((c1 (make-wire)) 
        (c2 (make-wire))
        (s  (make-wire)))
    (half-adder b c-in s c1)
    (half-adder a s sum c2)
    (or-gate c1 c2 c-out)
    'ok))





full-adder ကို ပရိုစီဂျာတစ်ခုအဖြစ် အဓိပ္ပာယ်ဖွင့်ဆိုပြီးနောက်၊ ယင်းကို ပိုမိုရှုပ်ထွေးသော ဆားကစ်များ ဖန်တီးရန် အခြေခံအစိတ်အပိုင်းတစ်ခုအဖြစ် အသုံးပြုနိုင်သည်။ (ဥပမာအားဖြင့် လေ့ကျင့်ခန်း ၃.၃၀ ကို ကြည့်ပါ။)



အနှစ်သာရအားဖြင့်၊ ကျွန်ုပ်တို့၏ စီမျူလေတာသည် ဆားကစ်ဘာသာစကားတစ်ခုကို တည်ဆောက်ရန် ကိရိယာများကို ထောက်ပံ့ပေးသည်။ ၁.၁ တွင် Lisp လေ့လာမှုကို ချဉ်းကပ်ခဲ့သော ဘာသာစကားဆိုင်ရာ ယေဘုယျအမြင်ကို အသုံးပြုမည်ဆိုလျှင်၊ အခြေခံဖန်ရှင်ဘောက်စ်များသည် ဘာသာစကား၏ အခြေခံဒြပ်စင်များ (primitive elements) ဖြစ်ပြီး၊ ဘောက်စ်များကို အတူတကွ ချိတ်ဆက်ခြင်းသည် ပေါင်းစပ်မှုနည်းလမ်း (means of combination) ကို ဖြစ်ပေါ်စေကာ၊ ဝိုင်ယာကြိုးချိတ်ဆက်မှု ပုံစံများကို ပရိုစီဂျာများအဖြစ် သတ်မှတ်ခြင်းသည် အက်ဘ်စထရပ်ရှင်းနည်းလမ်း (means of abstraction) အဖြစ် ဆောင်ရွက်ပေးသည်ဟု ဆိုနိုင်သည်။




အခြေခံဖန်ရှင်ဘောက်စ်များ



အခြေခံဖန်ရှင်ဘောက်စ်များသည် ဝိုင်ယာကြိုးတစ်ခုပေါ်ရှိ ဆစ်ဂနယ်ပြောင်းလဲမှုက အခြားဝိုင်ယာကြိုးများပေါ်ရှိ ဆစ်ဂနယ်များအပေါ် သက်ရောက်မှုရှိစေသည့် “တွန်းအား” များကို အကောင်အထည်ဖော်ပေးသည်။ ဖန်ရှင်ဘောက်စ်များ တည်ဆောက်ရန် ဝိုင်ယာကြိုးများအပေါ် အောက်ပါလုပ်ဆောင်ချက်များကို အသုံးပြုသည်-



    

  •  (get-signal ⟨wire⟩)


    ဝိုင်ယာကြိုးပေါ်ရှိ လက်ရှိဆစ်ဂနယ်တန်ဖိုးကို ပြန်ပေးသည်။

    

  •  (set-signal! ⟨wire⟩ ⟨new value⟩)


    ဝိုင်ယာကြိုးပေါ်ရှိ ဆစ်ဂနယ်တန်ဖိုးကို တန်ဖိုးသစ်သို့ ပြောင်းလဲပေးသည်။

    

  •  (add-action! ⟨wire⟩ ⟨procedure of no arguments⟩)


    ဝိုင်ယာကြိုးပေါ်ရှိ ဆစ်ဂနယ်တန်ဖိုး ပြောင်းလဲသည့်အခါတိုင်း သတ်မှတ်ထားသော ပရိုစီဂျာကို လုပ်ဆောင်ရမည်ဟု အတည်ပြုသည်။ ထိုသို့သော ပရိုစီဂျာများသည် ဝိုင်ယာကြိုးပေါ်ရှိ ဆစ်ဂနယ်တန်ဖိုး ပြောင်းလဲမှုများကို အခြားဝိုင်ယာကြိုးများသို့ ဆက်သွယ်ပေးပို့ပေးသော ကြားခံများဖြစ်သည်။

    




ထို့အပြင်၊ အချိန်ကြန့်ကြာမှုတစ်ခုနှင့် လုပ်ဆောင်ရန် ပရိုစီဂျာတစ်ခုကို လက်ခံပြီး သတ်မှတ်ထားသော ကြန့်ကြာမှုအပြီးတွင် ပေးထားသော ပရိုစီဂျာကို လုပ်ဆောင်ပေးသည့် after-delay ပရိုစီဂျာကိုလည်း ကျွန်ုပ်တို့ အသုံးပြုပါမည်။



ဤပရိုစီဂျာများကို အသုံးပြု၍ အခြေခံဒစ်ဂျစ်တယ်လော့ဂျစ်ဖန်ရှင်များကို အဓိပ္ပာယ်ဖွင့်ဆိုနိုင်သည်။ အဝင်ကို အင်ဗာတာမှတစ်ဆင့် အထွက်သို့ ချိတ်ဆက်ရန်၊ အဝင်ဝိုင်ယာကြိုးပေါ်ရှိ ဆစ်ဂနယ်တန်ဖိုး ပြောင်းလဲသည့်အခါတိုင်း လုပ်ဆောင်မည့် ပရိုစီဂျာတစ်ခုကို အဝင်ဝိုင်ယာကြိုးနှင့် ချိတ်ဆက်ပေးရန် add-action! ကို အသုံးပြုသည်။ ထိုပရိုစီဂျာသည် အဝင်ဆစ်ဂနယ်၏ logical-not ကို တွက်ချက်ပြီးနောက် inverter-delay တစ်ခုအကြာတွင် အထွက်ဆစ်ဂနယ်ကို ဤတန်ဖိုးသစ်အဖြစ် သတ်မှတ်ပေးသည်-





(define (inverter input output)
  (define (invert-input)
    (let ((new-value 
           (logical-not (get-signal input))))
      (after-delay 
       inverter-delay
       (lambda ()
         (set-signal! output new-value)))))
  (add-action! input invert-input)
  'ok)

(define (logical-not s)
  (cond ((= s 0) 1)
        ((= s 1) 0)
        (else (error "Invalid signal" s))))



and-gate သည် အနည်းငယ် ပို၍ရှုပ်ထွေးသည်။ ဂိတ်သို့ အဝင်များထဲမှ တစ်ခုခု ပြောင်းလဲပါက action ပရိုစီဂျာကို လုပ်ဆောင်ရမည်။ ၎င်းသည် အဝင်ဝိုင်ယာကြိုးများပေါ်ရှိ ဆစ်ဂနယ်တန်ဖိုးများ၏ logical-and (logical-not နှင့် ဆင်တူသော ပရိုစီဂျာကို အသုံးပြု၍) ကို တွက်ချက်ပြီး and-gate-delay တစ်ခုအကြာတွင် အထွက်ဝိုင်ယာကြိုးပေါ်၌ တန်ဖိုးသစ်ဖြစ်ပေါ်စေရန် စီစဉ်ပေးသည်။





(define (and-gate a1 a2 output)
  (define (and-action-procedure)
    (let ((new-value
           (logical-and (get-signal a1) 
                        (get-signal a2))))
      (after-delay 
       and-gate-delay
       (lambda ()
         (set-signal! output new-value)))))
  (add-action! a1 and-action-procedure)
  (add-action! a2 and-action-procedure)
  'ok)

ကျွန်ုပ်တို့၏ simulation ထဲမှ wire တစ်ခုသည် local state variable နှစ်ခုပါဝင်သော computational object တစ်ခု ဖြစ်လိမ့်မည်။ ၎င်းတို့မှာ signal-value တစ်ခု (အစပိုင်းတွင် 0 ဟု ယူဆသည်) နှင့် signal တန်ဖိုး ပြောင်းလဲသည့်အခါ လုပ်ဆောင်ရမည့် action-procedures စုစည်းမှုတို့ ဖြစ်သည်။ ကျွန်ုပ်တို့သည် အပိုင်း ၃.၁.၁ ရှိ ရိုးရှင်းသော bank-account object တွင် ပြုလုပ်ခဲ့သကဲ့သို့ပင် wire ကို message-passing ပုံစံ အသုံးပြု၍ သင့်တော်သော local operation ကို ရွေးချယ်ပေးသည့် dispatch ပရိုစီဂျာနှင့်အတူ local procedure များ စုစည်းမှုအဖြစ် အကောင်အထည်ဖော်ပါသည်:





(define (make-wire)
  (let ((signal-value 0) 
        (action-procedures '()))
    (define (set-my-signal! new-value)
      (if (not (= signal-value new-value))
          (begin (set! signal-value new-value)
                 (call-each 
                  action-procedures))
          'done))
    (define (accept-action-procedure! proc)
      (set! action-procedures 
            (cons proc action-procedures))
      (proc))
    (define (dispatch m)
      (cond ((eq? m 'get-signal) 
             signal-value)
            ((eq? m 'set-signal!) 
             set-my-signal!)
            ((eq? m 'add-action!) 
             accept-action-procedure!)
            (else (error "Unknown operation: 
                          WIRE" m))))
    dispatch))



local procedure ဖြစ်သော set-my-signal! သည် signal တန်ဖိုးအသစ်က wire ပေါ်ရှိ signal ကို ပြောင်းလဲစေခြင်း ရှိ၊ မရှိ စစ်ဆေးပါသည်။ ပြောင်းလဲမှု ရှိပါက အောက်ပါ call-each ပရိုစီဂျာကို အသုံးပြု၍ action procedure တစ်ခုချင်းစီကို လုပ်ဆောင်စေပါသည်။ call-each သည် အာဂူးမန့် မပါသော ပရိုစီဂျာများ (no-argument procedures) စာရင်းထဲမှ တစ်ခုချင်းစီကို ခေါ်ယူပေးပါသည်။:





(define (call-each procedures)
  (if (null? procedures)
      'done
      (begin ((car procedures))
             (call-each (cdr procedures)))))



local procedure ဖြစ်သော accept-action-procedure! သည် ပေးထားသော ပရိုစီဂျာကို လုပ်ဆောင်ရမည့် ပရိုစီဂျာများ စာရင်းထဲသို့ ထည့်သွင်းပြီးနောက် ထိုပရိုစီဂျာအသစ်ကို တစ်ကြိမ် လုပ်ဆောင်စေပါသည်။ ( လေ့ကျင့်ခန်း ၃.၃၁ ကို ကြည့်ပါ။)



သတ်မှတ်ထားသည့်အတိုင်း local dispatch ပရိုစီဂျာကို တည်ဆောက်ပြီးနောက် wire များပေါ်ရှိ local operation များကို အသုံးပြုနိုင်ရန် အောက်ပါ ပရိုစီဂျာများကို ပံ့ပိုးပေးနိုင်ပါသည်။:155





(define (get-signal wire)
  (wire 'get-signal))
(define (set-signal! wire new-value)
  ((wire 'set-signal!) new-value))
(define (add-action! wire action-procedure)
  ((wire 'add-action!) action-procedure))



အချိန်နှင့်အမျှ ပြောင်းလဲနိုင်သော signal များရှိပြီး device များတွင် တစ်ဆင့်ချင်းစီ ချိတ်ဆက်နိုင်သော Wire များသည် ပြုပြင်ပြောင်းလဲနိုင်သော object (mutable objects) များ၏ ပုံစံမျိုး ဖြစ်ပါသည်။ ကျွန်ုပ်တို့သည် ၎င်းတို့ကို assignment ဖြင့် ပြုပြင်ပြောင်းလဲနိုင်သော local state variable များပါရှိသည့် procedure များအဖြစ် ပုံစံထုတ် (model) ခဲ့ပါသည်။ wire အသစ်တစ်ခု တည်ဆောက်သည့်အခါ (make-wire ရှိ let expression အားဖြင့်) state variable အသစ်များ စုစည်းမှုကို သတ်မှတ်ပေးပြီး state variable အသစ်များပါရှိသော ပတ်ဝန်းကျင်ကို သိမ်းဆည်းထားသည့် dispatch ပရိုစီဂျာအသစ်တစ်ခုကို တည်ဆောက်၍ ပြန်ပေးပါသည်။



Wire များကို ၎င်းတို့နှင့် ချိတ်ဆက်ထားသော device အမျိုးမျိုးတွင် မျှဝေအသုံးပြုပါသည်။ ထို့ကြောင့် device တစ်ခုနှင့် အပြန်အလှန် သက်ရောက်မှုကြောင့် ဖြစ်ပေါ်လာသော အပြောင်းအလဲသည် ထို wire နှင့် ချိတ်ဆက်ထားသော အခြား device များအားလုံးအပေါ် သက်ရောက်မှု ရှိလိမ့်မည်။ Wire သည် ချိတ်ဆက်မှုများ ပြုလုပ်စဉ်က ၎င်းအား ပေးထားသော action procedure များကို ခေါ်ယူခြင်းဖြင့် အပြောင်းအလဲကို ၎င်း၏ အိမ်နီးချင်းများထံသို့ အသိပေးပါသည်။




Agenda



simulator ကို ပြီးပြည့်စုံစေရန် လိုအပ်သော နောက်ဆုံးအရာမှာ after-delay ဖြစ်သည်။ ဤနေရာတွင် အဓိက အယူအဆမှာ လုပ်ဆောင်ရမည့် အရာများ၏ အစီအစဉ် (schedule) များ ပါဝင်သော 
agenda ဟု ခေါ်သော data structure တစ်ခုကို ထိန်းသိမ်းထားရှိခြင်း ဖြစ်သည်။ agenda များအတွက် အောက်ပါ operation များကို သတ်မှတ်ထားပါသည်:



    

  •  (make-agenda) သည် အလွတ်ဖြစ်သော agenda အသစ်တစ်ခုကို ပြန်ပေးသည်။


  •  (empty-agenda? ⟨agenda⟩) သည် သတ်မှတ်ထားသော agenda အလွတ်ဖြစ်နေပါက true ဖြစ်သည်။


  •  (first-agenda-item ⟨agenda⟩) သည် agenda ပေါ်ရှိ ပထမဆုံး အရာကို ပြန်ပေးသည်။


  •  (remove-first-agenda-item! ⟨agenda⟩) သည် ပထမဆုံးအရာကို ဖယ်ရှားခြင်းဖြင့် agenda ကို ပြုပြင်ပြောင်းလဲသည်။


  •  (add-to-agenda! ⟨time⟩ ⟨action⟩ ⟨agenda⟩) 
သည် ပေးထားသော action procedure ကို သတ်မှတ်ထားသော အချိန်တွင် လုပ်ဆောင်ရန် agenda ထဲသို့ ထည့်သွင်းပေးသည်။


  •  (current-time ⟨agenda⟩) သည် လက်ရှိ simulation အချိန်ကို ပြန်ပေးသည်။





ကျွန်ုပ်တို့ အသုံးပြုသည့် သီးခြား agenda ကို the-agenda ဟု သတ်မှတ်ပါသည်။ after-delay ပရိုစီဂျာသည် the-agenda ထဲသို့ အစိတ်အပိုင်းအသစ်များကို ထည့်သွင်းပေးသည်:





(define (after-delay delay action)
  (add-to-agenda! 
   (+ delay (current-time the-agenda))
   action
   the-agenda))



Simulation ကို the-agenda ပေါ်တွင် အလုပ်လုပ်ပြီး agenda ရှိ ပရိုစီဂျာ တစ်ခုချင်းစီကို အစီအစဉ်အလိုက် လုပ်ဆောင်စေသော propagate ပရိုစီဂျာဖြင့် မောင်းနှင်ပါသည်။ ယေဘုယျအားဖြင့် simulation အလုပ်လုပ်နေစဉ်အတွင်း agenda ထဲသို့ အရာအသစ်များကို ထပ်မံထည့်သွင်းသွားမည်ဖြစ်ပြီး agenda ထဲတွင် အရာများ ရှိနေသမျှ ကာလပတ်လုံး propagate သည် simulation ကို ဆက်လက်လုပ်ဆောင်နေမည် ဖြစ်သည်:





(define (propagate)
  (if (empty-agenda? the-agenda)
      'done
      (let ((first-item 
             (first-agenda-item the-agenda)))
        (first-item)
        (remove-first-agenda-item! the-agenda)
        (propagate))))

(define (accept-action-procedure! proc)

        (cons proc action-procedures)))

Agenda ကို 
time segments များဖြင့် ဖွဲ့စည်းထားသည်။ Time segment တစ်ခုစီသည် ကိန်းဂဏန်းတစ်ခု (အချိန်) နှင့် ထိုအချိန်အတွင်း လုပ်ဆောင်ရန် စီစဉ်ထားသော procedure များကို သိမ်းဆည်းထားသည့် queue တစ်ခု ( လေ့ကျင့်ခန်း ၃.၃၂ ကိုကြည့်ပါ) တို့ ပါဝင်သော pair တစ်ခုဖြစ်သည်။





(define (make-time-segment time queue)
  (cons time queue))
(define (segment-time s) (car s))
(define (segment-queue s) (cdr s))



ကျွန်ုပ်တို့သည် time-segment queue များကို ၃.၃.၂ တွင် ဖော်ပြထားသော queue operation များ အသုံးပြု၍ လုပ်ဆောင်သွားမည် ဖြစ်သည်။



Agenda ကိုယ်တိုင်သည် time segments များ၏ table တစ်ခုဖြစ်သည်။ ၎င်းသည် ၃.၃.၃ တွင် ဖော်ပြထားသော table များနှင့် မတူညီသည့်အချက်မှာ segments များကို အချိန်တိုးလာသည့်အစဉ်အတိုင်း (increasing time) စီထားခြင်း ဖြစ်သည်။ ထို့အပြင်၊ ကျွန်ုပ်တို့သည် 
လက်ရှိအချိန် (current time)
(ဆိုလိုသည်မှာ နောက်ဆုံးလုပ်ဆောင်ခဲ့သော action ၏ အချိန်) ကို agenda ၏ ထိပ်ဆုံးတွင် သိမ်းဆည်းထားသည်။ အသစ်တည်ဆောက်ထားသော agenda တစ်ခုတွင် time segments များ မရှိသေးဘဲ လက်ရှိအချိန်မှာ ၀ ဖြစ်သည်:156





(define (make-agenda) (list 0))
(define (current-time agenda) (car agenda))
(define (set-current-time! agenda time)
  (set-car! agenda time))
(define (segments agenda) (cdr agenda))
(define (set-segments! agenda segments)
  (set-cdr! agenda segments))
(define (first-segment agenda) 
  (car (segments agenda)))
(define (rest-segments agenda) 
  (cdr (segments agenda)))



Agenda တစ်ခုတွင် time segments များ မရှိပါက ၎င်းသည် ဗလာ (empty) ဖြစ်သည်။





(define (empty-agenda? agenda)
  (null? (segments agenda)))

  (define (belongs-before? segments)
    (or (null? segments)
        (< time 
           (segment-time (car segments)))))
  (define (make-new-time-segment time action)
    (let ((q (make-queue)))
      (insert-queue! q action)
      (make-time-segment time q)))
  (define (add-to-segments! segments)
    (if (= (segment-time (car segments)) time)
        (insert-queue! 
         (segment-queue (car segments))
         action)
        (let ((rest (cdr segments)))
          (if (belongs-before? rest)
              (set-cdr!
               segments
               (cons (make-new-time-segment 
                      time 
                      action)
                     (cdr segments)))
              (add-to-segments! rest)))))
  (let ((segments (segments agenda)))
    (if (belongs-before? segments)
        (set-segments!
         agenda
         (cons (make-new-time-segment 
                time 
                action)
               segments))
        (add-to-segments! segments))))

အထက်တွင် ဖော်ပြထားသော အပူချိန်တွက်ချက်မှုကို လုပ်ဆောင်ရန် constraint စနစ်ကို အသုံးပြုရန်အတွက်၊ ကျွန်ုပ်တို့သည် ဦးစွာ constructor make-connector ကို ခေါ်ယူခြင်းဖြင့် C နှင့် F ဟူသော connector နှစ်ခုကို ဖန်တီးပြီး၊ C နှင့် F ကို သင့်လျော်သော network တစ်ခုတွင် ချိတ်ဆက်ရမည်-

Network ကို ဖန်တီးပေးသော procedure ကို အောက်ပါအတိုင်း အဓိပ္ပာယ်ဖွင့်ဆိုသည်-





(define (celsius-fahrenheit-converter c f)
  (let ((u (make-connector))
        (v (make-connector))
        (w (make-connector))
        (x (make-connector))
        (y (make-connector)))
    (multiplier c w u)
    (multiplier v x u)
    (adder v y f)
    (constant 9 w)
    (constant 5 x)
    (constant 32 y)
    'ok))



ဤ procedure သည် အတွင်းပိုင်း connectors များဖြစ်သော u၊ v၊ w၊ x၊ နှင့် y တို့ကို ဖန်တီးပြီး၊ ၎င်းတို့ကို ပုံ ၃.၂၈ တွင် ပြထားသည့်အတိုင်း primitive constraint constructors များဖြစ်သော adder၊ multiplier၊ နှင့် constant တို့ကို အသုံးပြု၍ ချိတ်ဆက်ပေးသည်။ ၃.၃.၄ ရှိ digital-circuit simulator ကဲ့သို့ပင်၊ ဤအခြေခံ အစိတ်အပိုင်းများ၏ ပေါင်းစပ်မှုများကို procedures များအဖြစ် ဖော်ပြခြင်းက ကျွန်ုပ်တို့၏ ဘာသာစကားအတွက် compound objects များအတွက် abstraction ပြုလုပ်နိုင်သော နည်းလမ်းကို အလိုအလျောက် ပံ့ပိုးပေးသည်။



Network အလုပ်လုပ်ပုံကို ကြည့်ရှုရန်၊ ၃.၃.၄ တွင် wire များကို စောင့်ကြည့်ရန် အသုံးပြုခဲ့သော probe procedure နှင့် ဆင်တူသော procedure ကို အသုံးပြု၍ connectors C နှင့် F တို့တွင် probes များ ထားရှိနိုင်သည်။ Connector တစ်ခုပေါ်တွင် probe တစ်ခု ထားရှိခြင်းက connector ထံသို့ တန်ဖိုးတစ်ခု ရောက်ရှိလာတိုင်း message တစ်ခု ထုတ်ပေးစေမည်ဖြစ်သည်။





(probe "Celsius temp" C)
(probe "Fahrenheit temp" F)



နောက်တစ်ဆင့်အနေဖြင့် ကျွန်ုပ်တို့သည် C ၏ တန်ဖိုးကို ၂၅ ဟု သတ်မှတ်မည်။ (set-value! ၏ တတိယမြောက် argument က ဤညွှန်ကြားချက်သည် user ထံမှ လာခြင်းဖြစ်ကြောင်း C ကို ပြောပြသည်။)





(set-value! C 25 'user)

C ပေါ်ရှိ probe သည် နိုးလာပြီး တန်ဖိုးကို အစီရင်ခံသည်။ C သည်လည်း ၎င်း၏တန်ဖိုးကို အထက်တွင် ဖော်ပြခဲ့သည့်အတိုင်း network တစ်လျှောက် ပျံ့နှံ့စေသည်။ ၎င်းက F ကို ၇၇ အဖြစ် သတ်မှတ်ပေးပြီး F ပေါ်ရှိ probe က အစီရင်ခံသည်။



ယခု ကျွန်ုပ်တို့သည် F ကို တန်ဖိုးအသစ်တစ်ခု (ဥပမာ ၂၁၂) အဖြစ် သတ်မှတ်ရန် ကြိုးစားကြည့်နိုင်သည်-





(set-value! F 212 'user)
Error! Contradiction (77 212)




Connector က ပဋိပက္ခ (contradiction) တစ်ခု ဖြစ်နေကြောင်း တိုင်တန်းပါလိမ့်မည်- ၎င်း၏ တန်ဖိုးမှာ ၇၇ ဖြစ်နေပြီး တစ်စုံတစ်ယောက်က ၂၁၂ အဖြစ် သတ်မှတ်ရန် ကြိုးစားနေသောကြောင့် ဖြစ်သည်။ အကယ်၍ ကျွန်ုပ်တို့သည် network ကို တန်ဖိုးအသစ်များဖြင့် တကယ်ပြန်သုံးလိုပါက C ကို ၎င်း၏ တန်ဖိုးဟောင်းအား မေ့လိုက်ရန် ပြောနိုင်သည်။





(forget-value! C 'user)

C ၏ တန်ဖိုးကို မူလက သတ်မှတ်ခဲ့သော user က ထိုတန်ဖိုးကို ပြန်လည် ရုပ်သိမ်းလိုက်ကြောင်း C က သိရှိသွားသောအခါ၊ C သည် ၎င်း၏တန်ဖိုးကို စွန့်လွှတ်ရန် သဘောတူလိုက်သည် (probe တွင် ပြထားသည့်အတိုင်း)။ ထို့နောက် ဤအချက်ကို network ၏ ကျန်အစိတ်အပိုင်းများကို အသိပေးသည်။ ဤအချက်အလက်သည် နောက်ဆုံးတွင် F ထံသို့ ရောက်ရှိသွားပြီး၊ F သည်လည်း ၎င်း၏တန်ဖိုးကို ၇၇ အဖြစ် ဆက်လက် ယုံကြည်ထားရန် အကြောင်းပြချက် မရှိတော့ကြောင်း သိရှိသွားသည်။ ထို့ကြောင့် F သည်လည်း probe တွင် ပြထားသည့်အတိုင်း ၎င်း၏တန်ဖိုးကို စွန့်လွှတ်လိုက်သည်။



ယခု F တွင် တန်ဖိုးမရှိတော့သဖြင့် ကျွန်ုပ်တို့သည် ၎င်းကို ၂၁၂ အဖြစ် လွတ်လပ်စွာ သတ်မှတ်နိုင်ပြီဖြစ်သည်-





(set-value! F 212 'user)

Probe: Celsius temp = 100

ဤတန်ဖိုးအသစ်သည် network တစ်လျှောက် ပျံ့နှံ့သွားသောအခါ C ကို တန်ဖိုး ၁၀၀ ဖြစ်စေရန် တိုက်တွန်းပြီး၊ ဤအချက်ကို C ပေါ်ရှိ probe က မှတ်တမ်းတင်သည်။ တူညီသော network တစ်ခုတည်းကိုပင် F ပေးထားလျှင် C ကို တွက်ချက်ရန်နှင့် C ပေးထားလျှင် F ကို တွက်ချက်ရန် အသုံးပြုနေသည်ကို သတိပြုပါ။ ဤကဲ့သို့ တွက်ချက်မှုတွင် လားရာသတ်မှတ်ချက်မရှိခြင်း (nondirectionality) သည် constraint-based စနစ်များ၏ ထူးခြားသော လက္ခဏာဖြစ်သည်။




Constraint စနစ်ကို အကောင်အထည်ဖော်ခြင်း



Constraint စနစ်ကို ၃.၃.၄ ရှိ digital-circuit simulator နှင့် အလွန်ဆင်တူသော ပုံစံဖြင့် local state ပါရှိသော procedural objects များဖြင့် အကောင်အထည်ဖော်ထားသည်။ Constraint စနစ်၏ အခြေခံ objects များသည် အနည်းငယ် ပိုမိုရှုပ်ထွေးသော်လည်း၊ agenda များနှင့် logic delay များအတွက် စိုးရိမ်စရာမလိုသောကြောင့် စနစ်တစ်ခုလုံးမှာ ပိုမိုရိုးရှင်းပါသည်။



Connectors များပေါ်ရှိ အခြေခံ လုပ်ဆောင်ချက်များမှာ အောက်ပါအတိုင်း ဖြစ်သည်-



    

  •  (has-value? ⟨connector⟩) သည် connector တွင် တန်ဖိုးရှိမရှိ ပြောပြသည်။


  •  (get-value ⟨connector⟩) သည် connector ၏ လက်ရှိတန်ဖိုးကို ပြန်ပေးသည်။


  •  (set-value! ⟨connector⟩ ⟨new-value⟩ ⟨informant⟩)
သည် informant က connector ကို တန်ဖိုးအသစ်သို့ သတ်မှတ်ရန် တောင်းဆိုနေကြောင်း ညွှန်ပြသည်။


  •  (forget-value! ⟨connector⟩ ⟨retractor⟩) သည် retractor က connector ကို ၎င်း၏တန်ဖိုးအား မေ့လိုက်ရန် တောင်းဆိုနေကြောင်း ပြောပြသည်။


  •  (connect ⟨connector⟩ ⟨new-constraint⟩) သည် connector ကို constraint အသစ်တွင် ပါဝင်ရန် ပြောပြသည်။





Connectors များသည် constraints များနှင့် ဆက်သွယ်ရာတွင် inform-about-value (ပေးထားသော constraint အား connector တွင် တန်ဖိုးရှိကြောင်း ပြောပြသော procedure) နှင့် inform-about-no-value (constraint အား connector တွင် တန်ဖိုးမရှိတော့ကြောင်း ပြောပြသော procedure) တို့ကို အသုံးပြု၍ ဆက်သွယ်ကြသည်။



Adder သည် ပေါင်းကိန်း (summand) connectors များဖြစ်သော a1 နှင့် a2 နှင့် ပေါင်းလဒ် (sum) connector တစ်ခုတို့အကြား adder constraint တစ်ခုကို တည်ဆောက်သည်။ Adder ကို local state ပါရှိသော procedure တစ်ခု (အောက်ပါ me ဆိုသော procedure) အဖြစ် အကောင်အထည်ဖော်သည်-





(define (adder a1 a2 sum)
  (define (process-new-value)
    (cond ((and (has-value? a1) 
                (has-value? a2))
           (set-value! sum
                       (+ (get-value a1) 
                          (get-value a2))
                       me))
          ((and (has-value? a1) 
                (has-value? sum))
           (set-value! a2
                       (- (get-value sum) 
                          (get-value a1))
                       me))
          ((and (has-value? a2) 
                 (has-value? sum))
           (set-value! a1
                       (- (get-value sum) 
                          (get-value a2))
                       me))))
  (define (process-forget-value)
    (forget-value! sum me)
    (forget-value! a1 me)
    (forget-value! a2 me)
    (process-new-value))
  (define (me request)
    (cond ((eq? request 'I-have-a-value)
           (process-new-value))
          ((eq? request 'I-lost-my-value)
           (process-forget-value))
          (else (error "Unknown request: 
                        ADDER" request))))
  (connect a1 me)
  (connect a2 me)
  (connect sum me)
  me)



Adder သည် သတ်မှတ်ထားသော connector များနှင့် adder အသစ်ကို ချိတ်ဆက်ပေးပြီး ၎င်းကို တန်ဖိုးအဖြစ် ပြန်ပေးသည်။ Adder ကို ကိုယ်စားပြုသော procedure me သည် local procedure များဆီသို့ dispatch တစ်ခုအဖြစ် လုပ်ဆောင်သည်။ အောက်ပါ “syntax interface” များ ( 3.3.4 ရှိ Footnote 155 ကို ကြည့်ပါ) ကို dispatch နှင့် တွဲဖက်၍ အသုံးပြုထားသည် -





(define (inform-about-value constraint)
  (constraint 'I-have-a-value))
(define (inform-about-no-value constraint)
  (constraint 'I-lost-my-value))

  (define (process-new-value)
    (cond ((or (and (has-value? m1) 
                    (= (get-value m1) 0))
               (and (has-value? m2) 
                    (= (get-value m2) 0)))
           (set-value! product 0 me))
          ((and (has-value? m1) 
                (has-value? m2))
           (set-value! product
                       (* (get-value m1) 
                          (get-value m2))
                       me))
          ((and (has-value? product) 
                (has-value? m1))
           (set-value! m2
                       (/ (get-value product) 
                          (get-value m1))
                       me))
          ((and (has-value? product) 
                (has-value? m2))
           (set-value! m1
                       (/ (get-value product) 
                          (get-value m2))
                       me))))
  (define (process-forget-value)
    (forget-value! product me)
    (forget-value! m1 me)
    (forget-value! m2 me)
    (process-new-value))
  (define (me request)
    (cond ((eq? request 'I-have-a-value)
           (process-new-value))
          ((eq? request 'I-lost-my-value)
           (process-forget-value))
          (else
           (error "Unknown request: 
                   MULTIPLIER" 
                  request))))
  (connect m1 me)
  (connect m2 me)
  (connect product me)
  me)

(define (for-each-except exception 
                         procedure 
                         list)
  (define (loop items)
    (cond ((null? items) 'done)
          ((eq? (car items) exception) 
           (loop (cdr items)))
          (else (procedure (car items))
                (loop (cdr items)))))
  (loop list))

158 ဤနည်းဖြင့် လက်ရှိအချိန် (current time) သည် အမြဲတမ်း နောက်ဆုံးလုပ်ဆောင်ခဲ့သော action ၏ အချိန်ဖြစ်နေမည်ဖြစ်သည်။ ဤအချိန်ကို agenda ၏ ထိပ်တွင် သိမ်းဆည်းထားခြင်းက ၎င်းနှင့် သက်ဆိုင်သော time segment ကို ဖျက်လိုက်လျှင်တောင် အချိန်ကို ဆက်လက် ရရှိနိုင်ကြောင်း သေချာစေသည်။

159 Constraint propagation သည် Ivan Sutherland (1963) ၏ မယုံနိုင်လောက်အောင် အမြော်အမြင်ရှိသော SKETCHPAD system တွင် ပထမဆုံး ပေါ်ပေါက်ခဲ့သည်။ Smalltalk language ပေါ်တွင် အခြေခံထားသော လှပသည့် constraint-propagation system တစ်ခုကို Xerox Palo Alto Research Center မှ Alan Borning (1977) က တီထွင်ခဲ့သည်။ Sussman၊ Stallman နှင့် Steele တို့သည် constraint propagation ကို လျှပ်စစ်ပတ်လမ်း ဆန်းစစ်ခြင်း (Sussman and Stallman 1975; Sussman and Steele 1980) တွင် အသုံးချခဲ့ကြသည်။ TK!Solver (Konopasek and Jayaraman 1984) သည် constraint များပေါ်တွင် အခြေခံထားသော ကျယ်ပြန့်သည့် modeling environment တစ်ခု ဖြစ်သည်။

160 setter သည် constraint တစ်ခု မဟုတ်နိုင်ပါ။ ကျွန်ုပ်တို့၏ အပူချိန် ဥပမာတွင် user ကို setter အဖြစ် အသုံးပြုခဲ့သည်။

161 Expression-oriented format သည် တွက်ချက်မှုတစ်ခုရှိ ကြားခံ expression များကို အမည်ပေးရန် မလိုသောကြောင့် အဆင်ပြေသည်။ ကျွန်ုပ်တို့၏ constraint language ၏ မူလပုံစံမှာ compound data များအပေါ် operation များ လုပ်ဆောင်ရာတွင် language အများအပြား ခက်ခဲရှုပ်ထွေးသကဲ့သို့ပင် ခက်ခဲရှုပ်ထွေးသည်။ ဥပမာအားဖြင့်၊ variable များသည် vector များကို ကိုယ်စားပြုသော ( a + b ) ⋅ ( c + d ) မြှောက်လဒ်ကို တွက်ချက်လိုပါက၊ သတ်မှတ်ထားသော vector argument များ၏ တန်ဖိုးများကို သတ်မှတ်ပေးသော်လည်း vector များကိုယ်တိုင်ကို တန်ဖိုးအဖြစ် ပြန်မပေးသော procedure များကို အသုံးပြုသည့် “imperative style” ဖြင့် လုပ်ဆောင်နိုင်ပါသည်- 

(v-sum a b temp1)
(v-sum c d temp2)
(v-prod temp1 temp2 answer)

တစ်နည်းအားဖြင့် vector များကို တန်ဖိုးများအဖြစ် ပြန်ပေးသော procedure များကို အသုံးပြု၍ expression များကို ကိုင်တွယ်နိုင်ပြီး temp1 နှင့် temp2 တို့ကို တိုက်ရိုက်ဖော်ပြရန် မလိုတော့ပါ- 

(define answer 
  (v-prod (v-sum a b) (v-sum c d)))

Lisp သည် procedure များ၏ တန်ဖိုးများအဖြစ် compound object များကို ပြန်ပေးရန် ခွင့်ပြုထားသောကြောင့်၊ ဤလေ့ကျင့်ခန်းတွင် ပြထားသည့်အတိုင်း ကျွန်ုပ်တို့၏ imperative-style constraint language ကို expression-oriented style သို့ ပြောင်းလဲနိုင်သည်။ Algol၊ Basic နှင့် Pascal (Pascal pointer variable များကို တိုက်ရိုက်အသုံးမပြုပါက) ကဲ့သို့သော compound object များကို ကိုင်တွယ်ရာတွင် အားနည်းသော language များတွင် compound object များကို စီမံခန့်ခွဲသည့်အခါ များသောအားဖြင့် imperative style တွင်သာ ပိတ်မိနေတတ်သည်။ Expression-oriented format ၏ အားသာချက်ကို ထောက်ရှု၍ ဤကဏ္ဍတွင် ကျွန်ုပ်တို့ လုပ်ဆောင်ခဲ့သကဲ့သို့ စနစ်ကို imperative style ဖြင့် implement လုပ်ခဲ့ရန် အကြောင်းရင်းရှိမရှိ မေးစရာရှိပါသည်။ အကြောင်းရင်းတစ်ခုမှာ non-expression-oriented constraint language သည် connector object များသာမက constraint object များ (ဥပမာ adder procedure ၏ တန်ဖိုး) ကိုပါ ကိုင်တွယ်ရန် နည်းလမ်း (handle) ပေးထားခြင်းဖြစ်သည်။ ၎င်းသည် connector များပေါ်ရှိ operation များမှတစ်ဆင့် သွယ်ဝိုက်၍မဟုတ်ဘဲ constraint များနှင့် တိုက်ရိုက်ဆက်သွယ်သည့် operation အသစ်များဖြင့် စနစ်ကို တိုးချဲ့လိုပါက အသုံးဝင်ပါသည်။ Imperative implementation ကို အခြေခံ၍ expression-oriented style ကို implement လုပ်ရန် လွယ်ကူသော်လည်း၊ ပြောင်းပြန်လုပ်ရန်မှာ အလွန်ခက်ခဲပါသည်။
