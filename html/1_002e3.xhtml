1.3Higher-Order Procedures များဖြင့် Abstractions များ ဖော်ထုတ်ခြင်း

လုပ်ဆောင်ချက် (procedures) များသည် ကိန်းဂဏန်းအမျိုးမျိုးအပေါ်တွင် မူတည်ခြင်းမရှိဘဲ၊ ကိန်းဂဏန်းများအပေါ် ပေါင်းစပ်လုပ်ဆောင်မှုများကို ဖော်ပြသည့် abstraction များဖြစ်ကြောင်း ကျွန်ုပ်တို့ တွေ့ရှိခဲ့ပြီးဖြစ်သည်။ ဥပမာအားဖြင့်၊ ကျွန်ုပ်တို့သည် 

(define (cube x) (* x x x))

ဟု သတ်မှတ်လိုက်သောအခါ၊ သတ်မှတ်ထားသော ကိန်းဂဏန်းတစ်ခု၏ cube အကြောင်းကို ပြောနေခြင်းမဟုတ်ဘဲ၊ မည်သည့်ကိန်းဂဏန်းမဆို၏ cube ကို ရယူနိုင်သည့် နည်းလမ်းတစ်ခုအကြောင်းကို ပြောနေခြင်းဖြစ်သည်။ အမှန်စင်စစ် ဤ procedure ကို မသတ်မှတ်ဘဲ အောက်ပါအတိုင်း ဖော်ပြချက်များကို အမြဲရေးသားခြင်းဖြင့်လည်း အလုပ်ဖြစ်နိုင်ပါသည်။ 

(* 3 3 3)
(* x x x)
(* y y y)

ထိုသို့ရေးပါက cube ကို တိုက်ရိုက်ဖော်ပြရန် လိုအပ်တော့မည်မဟုတ်ပါ။ သို့သော် ဤသို့ပြုလုပ်ခြင်းသည် ကျွန်ုပ်တို့ကို ကြီးမားသော အားနည်းချက်တစ်ခုဖြစ်စေပါသည်။ အဆင့်မြင့်သော လုပ်ဆောင်ချက်များ (higher-level operations) အနေဖြင့် စဉ်းစားမည့်အစား၊ programming language တွင် ပါရှိပြီးသား primitive operation များ (ဤနေရာတွင် မြှောက်ခြင်း) အဆင့်မှာတင် အမြဲတမ်း အလုပ်လုပ်နေရစေပါသည်။ ကျွန်ုပ်တို့၏ ပရိုဂရမ်များသည် cube များကို တွက်ချက်နိုင်ပါလိမ့်မည်၊ သို့သော် ကျွန်ုပ်တို့၏ language တွင် cubing ဆိုသည့် အယူအဆ (concept) ကို ဖော်ပြနိုင်စွမ်း ရှိမည်မဟုတ်ပါ။ စွမ်းအားထက်မြက်သော programming language တစ်ခုထံမှ ကျွန်ုပ်တို့ တောင်းဆိုသင့်သည့် အချက်တစ်ခုမှာ အများသုံးပုံစံ (common patterns) များကို နာမည်များပေးခြင်းဖြင့် abstractions များ တည်ဆောက်နိုင်စွမ်းရှိရန်နှင့် ထို abstractions များအတိုင်း တိုက်ရိုက်အလုပ်လုပ်နိုင်ရန်ဖြစ်သည်။ Procedures များသည် ဤစွမ်းရည်ကို ပေးစွမ်းပါသည်။ ထို့ကြောင့် primitive အကျဆုံး language များမှလွဲ၍ ကျန်ရှိသော programming language အားလုံးတွင် procedures များကို သတ်မှတ်ရန် ယန္တရားများ ပါဝင်ကြခြင်း ဖြစ်ပါသည်။

သို့သော် ကိန်းဂဏန်းများကိုသာ parameter အဖြစ် လက်ခံနိုင်သော procedures များဖြင့်သာ ကန့်သတ်ထားမည်ဆိုပါက၊ ကိန်းဂဏန်းဆိုင်ရာ လုပ်ဆောင်ချက်များတွင်ပင် abstractions များ ဖန်တီးနိုင်စွမ်းမှာ များစွာ အကန့်အသတ်ရှိနေပါလိမ့်မည်။ တစ်ခါတစ်ရံတွင် တူညီသော programming pattern တစ်ခုတည်းကို မတူညီသော procedures များစွာနှင့် အသုံးပြုလေ့ရှိပါသည်။ ထိုကဲ့သို့သော pattern များကို concept တစ်ခုအနေဖြင့် ဖော်ပြနိုင်ရန်အတွက်၊ procedures များကို argument အဖြစ် လက်ခံနိုင်သော သို့မဟုတ် procedures များကို value အဖြစ် ပြန်ပေးနိုင်သော procedures များကို တည်ဆောက်ရန် လိုအပ်ပါလိမ့်မည်။ Procedures များကို ကိုင်တွယ်အသုံးပြုသော procedures များကို higher-order procedures ဟု ခေါ်ဆိုပါသည်။ ဤအပိုင်းတွင် higher-order procedures များသည် စွမ်းအားထက်မြက်သော abstraction mechanisms များအဖြစ် မည်သို့ဆောင်ရွက်ပေးနိုင်ကြောင်းနှင့် ကျွန်ုပ်တို့ language ၏ ဖော်ပြနိုင်စွမ်း (expressive power) ကို မည်မျှအထိ များပြားလာစေနိုင်ကြောင်း ဖော်ပြပေးသွားမည် ဖြစ်ပါသည်။

1.3.1Procedures များအား Arguments များအဖြစ် အသုံးပြုခြင်း

အောက်ပါ procedure သုံးခုကို လေ့လာကြည့်ပါ။ ပထမတစ်ခုမှာ a မှ b အထိ ကိန်းပြည့်များ၏ ပေါင်းလဒ်ကို တွက်ချက်ပေးသည်: 

(define (sum-integers a b)
  (if (> a b) 
      0 
      (+ a (sum-integers (+ a 1) b))))

ဒုတိယတစ်ခုမှာ ပေးထားသော range အတွင်းရှိ ကိန်းပြည့်များ၏ cube (သုံးထပ်ကိန်း) များ၏ ပေါင်းလဒ်ကို တွက်ချက်ပေးသည်: 

(define (sum-cubes a b)
  (if (> a b) 
      0 
      (+ (cube a) 
         (sum-cubes (+ a 1) b))))

တတိယတစ်ခုမှာ အောက်ပါ series ရှိ term များ၏ ပေါင်းလဒ်ကို တွက်ချက်ပေးသည် 1 1 ⋅ 3 + 1 5 ⋅ 7 + 1 9 ⋅ 11 + … , ၎င်းသည် π / 8 သို့ (အလွန်နှေးကွေးစွာ) ချဉ်းကပ်သွားသည်:49 

(define (pi-sum a b)
  (if (> a b)
      0
      (+ (/ 1.0 (* a (+ a 2))) 
         (pi-sum (+ a 4) b))))

ဤ procedure သုံးခုသည် တူညီသော အခြေခံ pattern တစ်ခုကို မျှဝေအသုံးပြုထားကြောင်း ထင်ရှားပါသည်။ ၎င်းတို့သည် များသောအားဖြင့် တူညီကြပြီး၊ procedure အမည်၊ ပေါင်းထည့်ရမည့် term ကို တွက်ချက်ရန် အသုံးပြုသော a ၏ function နှင့် a ၏ နောက်တန်ဖိုးကို ပေးမည့် function တို့တွင်သာ ကွဲပြားကြသည်။ ကျွန်ုပ်တို့သည် template တစ်ခုတည်းရှိ နေရာလွတ်များကို ဖြည့်စွက်ခြင်းဖြင့် ဤ procedure တစ်ခုချင်းစီကို ဖန်တီးနိုင်သည်: 

(define (⟨name⟩ a b)
  (if (> a b)
      0
      (+ (⟨term⟩ a) 
         (⟨name⟩ (⟨next⟩ a) b))))

ဤကဲ့သို့ တူညီသော pattern ရှိနေခြင်းသည် အသုံးဝင်သော abstraction တစ်ခုကို ထုတ်ဖော်ရန်ရှိနေကြောင်း ခိုင်လုံသော အထောက်အထားဖြစ်ပါသည်။ အမှန်စင်စစ်၊ သင်္ချာပညာရှင်များသည် လွန်ခဲ့သော အချိန်များစွာကတည်းက series တစ်ခု၏ ပေါင်းလဒ် (summation of a series) ဆိုင်ရာ abstraction ကို ဖော်ထုတ်ခဲ့ကြပြီး ဤအယူအဆကို ဖော်ပြရန်အတွက် ဥပမာအားဖြင့် ∑ n = a b f ( n ) = f ( a ) + ⋯ + f ( b ) , ကဲ့သို့သော “sigma notation” ကို တီထွင်ခဲ့ကြသည်။ Sigma notation ၏ အစွမ်းမှာ သင်္ချာပညာရှင်များအား သီးခြားပေါင်းလဒ်များအပြင် ပေါင်းလဒ်ဆိုသည့် အယူအဆကိုယ်တိုင်ကို ကိုင်တွယ်ဖြေရှင်းနိုင်စေခြင်းဖြစ်သည်—ဥပမာအားဖြင့်၊ ပေါင်းလဒ်တွက်နေသည့် သီးခြား series နှင့် မသက်ဆိုင်ဘဲ ပေါင်းလဒ်များ၏ ယေဘုယျရလဒ်များကို ဖော်ထုတ်ရန်ဖြစ်သည်။

အလားတူပင်၊ program designer များအနေဖြင့် ကျွန်ုပ်တို့၏ language ကို သီးခြားပေါင်းလဒ်များကို တွက်ချက်ပေးရုံသာမက ပေါင်းလဒ်ဆိုသည့် အယူအဆကိုပါ ဖော်ပြနိုင်သော procedure တစ်ခုကို ရေးသားနိုင်လောက်အောင် အစွမ်းထက်စေလိုပါသည်။ ကျွန်ုပ်တို့၏ procedural language တွင် အထက်ပါ template ကို ယူ၍ “နေရာလွတ်များ” (slots) ကို formal parameter များအဖြစ် ပြောင်းလဲခြင်းဖြင့် အလွယ်တကူ လုပ်ဆောင်နိုင်သည်: 

(define (sum term a next b)
  (if (> a b)
      0
      (+ (term a)
         (sum term (next a) next b))))

sum သည် ၎င်း၏ argument များအဖြစ် အောက်ခြေနှင့် အထက်ခြေနယ်နိမိတ်များဖြစ်သော a နှင့် b အပြင် term နှင့် next အမည်ရှိ procedure များကိုပါ လက်ခံထားသည်ကို သတိပြုပါ။ ကျွန်ုပ်တို့သည် sum ကို အခြား procedure များကဲ့သို့ပင် အသုံးပြုနိုင်သည်။ ဥပမာအားဖြင့်၊ ၎င်းကို (argument ကို 1 တိုးပေးသော inc procedure နှင့် တွဲဖက်၍) sum-cubes ကို သတ်မှတ်ရန် အသုံးပြုနိုင်သည်: 

(define (inc n) (+ n 1))

(define (sum-cubes a b)
  (sum cube a inc b))

၎င်းကို အသုံးပြု၍ ၁ မှ ၁၀ အထိ ကိန်းပြည့်များ၏ cube ပေါင်းလဒ်ကို တွက်ချက်နိုင်သည်: 

(sum-cubes 1 10)
3025

Term ကို တွက်ချက်ရန် identity procedure ၏ အကူအညီဖြင့် sum-integers ကို sum အားဖြင့် အောက်ပါအတိုင်း သတ်မှတ်နိုင်သည်: 

(define (identity x) x)

(define (sum-integers a b)
  (sum identity a inc b))

ထို့နောက် ၁ မှ ၁၀ အထိ ကိန်းပြည့်များကို ပေါင်းနိုင်သည်: 

(sum-integers 1 10)
55

pi-sum ကိုလည်း အလားတူပင် သတ်မှတ်နိုင်သည်:50 

(define (pi-sum a b)
  (define (pi-term x)
    (/ 1.0 (* x (+ x 2))))
  (define (pi-next x)
    (+ x 4))
  (sum pi-term a pi-next b))

ဤ procedure များကို အသုံးပြု၍ π ၏ ခန့်မှန်းခြေတန်ဖိုးကို တွက်ချက်နိုင်သည်: 

(* 8 (pi-sum 1 1000))
3.139592655589783

sum ကို ရရှိပြီးနောက်၊ နောက်ထပ် အယူအဆများကို ဖော်ထုတ်ရာတွင် ၎င်းကို အခြေခံအုတ်မြစ် (building block) တစ်ခုအဖြစ် အသုံးပြုနိုင်သည်။ ဥပမာအားဖြင့်၊ နယ်နိမိတ် a နှင့် b ကြားရှိ function f ၏ definite integral ကို d x ၏ တန်ဖိုးအနည်းငယ်များအတွက် အောက်ပါ formula ကို အသုံးပြု၍ ကိန်းဂဏန်းအရ ခန့်မှန်းတွက်ချက်နိုင်သည်။ ၎င်းကို procedure တစ်ခုအနေဖြင့် တိုက်ရိုက်ဖော်ပြနိုင်သည်: 

(define (integral f a b dx)
  (define (add-dx x) (+ x dx))
  (* (sum f (+ a (/ dx 2.0)) add-dx b) 
     dx))

(integral cube 0 1 0.01)
.24998750000000042

(integral cube 0 1 0.001)

ကျွန်ုပ်တို့သည် ကိန်းဂဏန်းဆိုင်ရာ လုပ်ဆောင်ချက်ပုံစံများအား သက်ဆိုင်ရာ ကိန်းဂဏန်းများအပေါ်တွင် မမှီခိုတော့ဘဲ သီးခြားဖြစ်စေရန်အတွက် ပေါင်းစပ်လုပ်ထုံးလုပ်နည်းများ (compound procedures) အကြောင်းကို 1.1.4 တွင် မိတ်ဆက်ပေးခဲ့ပါသည်။ 1.3.1 ရှိ integral လုပ်ထုံးလုပ်နည်းကဲ့သို့သော အဆင့်မြင့်လုပ်ထုံးလုပ်နည်းများ (higher-order procedures) နှင့်အတူ၊ ကျွန်ုပ်တို့သည် ပိုမိုစွမ်းအားကောင်းသော abstraction အမျိုးအစားတစ်ခုကို စတင်မြင်တွေ့လာရသည်- ၎င်းမှာ သက်ဆိုင်ရာ functions များအပေါ်တွင် မမှီခိုဘဲ ယေဘုယျတွက်ချက်မှုနည်းလမ်းများကို ဖော်ပြရန် အသုံးပြုသော လုပ်ထုံးလုပ်နည်းများ ဖြစ်ပါသည်။ ဤကဏ္ဍတွင် ကျွန်ုပ်တို့သည် ပိုမိုအသေးစိတ်ကျသော ဥပမာနှစ်ခုဖြစ်သည့်—functions များ၏ zeros နှင့် fixed points များကို ရှာဖွေသည့် ယေဘုယျနည်းလမ်းများအကြောင်းကို ဆွေးနွေးမည်ဖြစ်ပြီး၊ ဤနည်းလမ်းများကို လုပ်ထုံးလုပ်နည်းများအဖြစ် တိုက်ရိုက်မည်သို့ ဖော်ပြနိုင်သည်ကို ပြသပါမည်။

1.3.4တန်ဖိုးများအဖြစ် ပြန်ပေးသော ပရိုစီဂျာများ

အထက်ပါ ဥပမာများက ပရိုစီဂျာများကို အာဂူမန့်များအဖြစ် ပေးပို့နိုင်ခြင်းသည် ကျွန်ုပ်တို့၏ ပရိုဂရမ်မင်းဘာသာစကား၏ ဖော်ပြနိုင်စွမ်းကို မည်သို့ သိသာထင်ရှားစွာ မြှင့်တင်ပေးသည်ကို ပြသခဲ့သည်။ ပြန်ပေးသော တန်ဖိုးကိုယ်တိုင်က ပရိုစီဂျာများ ဖြစ်နေသည့် ပရိုစီဂျာများကို ဖန်တီးခြင်းဖြင့် ကျွန်ုပ်တို့သည် ပို၍ပင် အစွမ်းထက်သော ဖော်ပြနိုင်စွမ်းကို ရရှိနိုင်သည်။

1.3.3 ၏ အဆုံးတွင် ဖော်ပြခဲ့သော ကိန်းသေအမှတ် (fixed-point) ဥပမာကို ပြန်လည်ကြည့်ရှုခြင်းဖြင့် ဤစိတ်ကူးကို ရှင်းလင်းစွာ မြင်နိုင်သည်။ x သည် y ↦ x / y ဖန်ရှင်၏ ကိန်းသေအမှတ်တစ်ခု ဖြစ်သည်ဟူသော လေ့လာတွေ့ရှိချက်မှ စတင်၍ နှစ်ထပ်ကိန်းရင်း (square-root) ရှာသော ပရိုစီဂျာ၏ ဗားရှင်းအသစ်ကို ကိန်းသေအမှတ် ရှာဖွေခြင်းတစ်ခုအဖြစ် ကျွန်ုပ်တို့ ပုံဖော်ခဲ့သည်။ ထို့နောက် ခန့်မှန်းခြေတန်ဖိုးများကို ချဉ်းကပ်စေရန် average damping ကို အသုံးပြုခဲ့သည်။ Average damping သည် သူ့ဘာသာသူ အသုံးဝင်သော ယေဘုယျ နည်းစနစ်တစ်ခု ဖြစ်သည်။ ဆိုလိုသည်မှာ ဖန်ရှင် f တစ်ခု ပေးထားပါက၊ x ရှိ ၎င်း၏ တန်ဖိုးသည် x နှင့် f ( x ) တို့၏ ပျမ်းမျှတန်ဖိုးနှင့် ညီမျှသော ဖန်ရှင်ကို ကျွန်ုပ်တို့ စဉ်းစားကြသည်။

Average damping ၏ စိတ်ကူးကို အောက်ပါ ပရိုစီဂျာဖြင့် ဖော်ပြနိုင်သည်- 

(define (average-damp f)
  (lambda (x) 
    (average x (f x))))

Average-damp သည် ပရိုစီဂျာ f ကို အာဂူမန့်အဖြစ် ယူပြီး ကိန်းဂဏန်း x တွင် အသုံးချသောအခါ x နှင့် (f x) တို့၏ ပျမ်းမျှကို ထုတ်ပေးသည့် (lambda ဖြင့် ဖန်တီးထားသော) ပရိုစီဂျာတစ်ခုကို တန်ဖိုးအဖြစ် ပြန်ပေးသည်။ ဥပမာအားဖြင့်၊ average-damp ကို square ပရိုစီဂျာတွင် အသုံးချပါက ကိန်းဂဏန်း x ရှိ ၎င်း၏ တန်ဖိုးသည် x နှင့် x 2 တို့၏ ပျမ်းမျှဖြစ်သည့် ပရိုစီဂျာတစ်ခုကို ရရှိသည်။ ထိုရရှိလာသော ပရိုစီဂျာကို 10 တွင် အသုံးချပါက 10 နှင့် 100 ၏ ပျမ်းမျှဖြစ်သော 55 ကို ရရှိသည်:59 

((average-damp square) 10)
55

average-damp ကို အသုံးပြု၍ နှစ်ထပ်ကိန်းရင်းရှာသော ပရိုစီဂျာကို အောက်ပါအတိုင်း ပြန်လည် ပုံဖော်နိုင်သည်- 

(define (sqrt x)
  (fixed-point 
   (average-damp 
    (lambda (y) (/ x y)))
   1.0))

ဤပုံဖော်မှုတွင် အသုံးပြုထားသော အချက်သုံးချက်ကို မည်သို့ ရှင်းလင်းစွာ ဖော်ပြထားသည်ကို သတိပြုပါ- ကိန်းသေအမှတ် ရှာဖွေခြင်း (fixed-point search)၊ average damping နှင့် ဖန်ရှင် y ↦ x / y တို့ဖြစ်သည်။ နှစ်ထပ်ကိန်းရင်းရှာသော ဤပုံစံသစ်ကို 1.1.7 တွင် ပေးထားသော မူလဗားရှင်းနှင့် နှိုင်းယှဉ်ကြည့်ရန် သင့်လျော်သည်။ ဤပရိုစီဂျာများသည် လုပ်ငန်းစဉ် အတူတူကိုပင် ဖော်ပြနေခြင်းဖြစ်ကြောင်း သတိရပါ၊ ၎င်းလုပ်ငန်းစဉ်ကို ဤကဲ့သို့ အကြမ်းဖျင်းပုံဖော်မှုများ (abstractions) ဖြင့် ဖော်ပြသောအခါ စိတ်ကူးမှာ မည်မျှ ပိုမိုရှင်းလင်းသွားသည်ကို သတိပြုပါ။ ယေဘုယျအားဖြင့်၊ လုပ်ငန်းစဉ်တစ်ခုကို ပရိုစီဂျာအဖြစ် ပုံဖော်ရန် နည်းလမ်းများစွာ ရှိပါသည်။ အတွေ့အကြုံရှိ ပရိုဂရမ်မာများသည် ပိုမိုရှင်းလင်းထင်ရှားသော ပုံစံများကို မည်သို့ရွေးချယ်ရမည်ကို သိကြပြီး၊ လုပ်ငန်းစဉ်၏ အသုံးဝင်သော အစိတ်အပိုင်းများကို အခြားနေရာများတွင် ပြန်လည်အသုံးပြုနိုင်သည့် သီးခြားအစိတ်အပိုင်းများအဖြစ် မည်သို့ ထုတ်ဖော်ရမည်ကို သိကြသည်။ ပြန်လည်အသုံးပြုခြင်း၏ ရိုးရှင်းသော ဥပမာတစ်ခုအနေဖြင့်၊ x ၏ သုံးထပ်ကိန်းရင်းသည် y ↦ x / y 2 ဖန်ရှင်၏ ကိန်းသေအမှတ်တစ်ခု ဖြစ်ကြောင်း သတိပြုပါ၊ ထို့ကြောင့် ကျွန်ုပ်တို့၏ နှစ်ထပ်ကိန်းရင်းရှာသော ပရိုစီဂျာကို သုံးထပ်ကိန်းရင်း ထုတ်ယူသော ပရိုစီဂျာအဖြစ် ချက်ချင်း ပြောင်းလဲအသုံးပြုနိုင်သည်:60 

(define (cube-root x)
  (fixed-point 
   (average-damp 
    (lambda (y) 
      (/ x (square y))))
   1.0))
Newton ၏ နည်းလမ်း

1.1.7 တွင် နှစ်ထပ်ကိန်းရင်းရှာသော ပရိုစီဂျာကို ပထမဆုံး မိတ်ဆက်ခဲ့စဉ်က၊ ၎င်းသည် Newton ၏ နည်းလမ်း ၏ ထူးခြားသော ဖြစ်ရပ်တစ်ခု ဖြစ်ကြောင်း ဖော်ပြခဲ့သည်။ အကယ်၍ x ↦ g ( x ) သည် differentiable ဖြစ်သော ဖန်ရှင်တစ်ခု ဖြစ်ပါက၊ ညီမျှခြင်း g ( x ) = 0 ၏ အဖြေတစ်ခုသည် x ↦ f ( x ) ဖန်ရှင်၏ ကိန်းသေအမှတ်တစ်ခု ဖြစ်သည်၊ ဤနေရာတွင် f ( x ) = x − g ( x ) D g ( x ) ဖြစ်ပြီး D g ( x ) သည် x တွင် တွက်ချက်ထားသော g ၏ derivative ဖြစ်သည်။ Newton ၏ နည်းလမ်းသည် ညီမျှခြင်း၏ အဖြေကို ခန့်မှန်းရန်အတွက် ဖန်ရှင် f ၏ ကိန်းသေအမှတ်တစ်ခုကို ရှာဖွေခြင်းအားဖြင့် အထက်တွင် ကျွန်ုပ်တို့ မြင်တွေ့ခဲ့ရသော fixed-point method ကို အသုံးပြုခြင်း ဖြစ်သည်။61

ဖန်ရှင် g အများအပြားအတွက်နှင့် x အတွက် လုံလောက်သော ကောင်းမွန်သည့် အစပျိုး ခန့်မှန်းချက်များအတွက် Newton ၏ နည်းလမ်းသည် g ( x ) = 0 ၏ အဖြေဆီသို့ အလွန်လျင်မြန်စွာ ချဉ်းကပ်သွားပါသည်။62

Newton ၏ နည်းလမ်းကို ပရိုစီဂျာတစ်ခုအဖြစ် အကောင်အထည်ဖော်ရန်အတွက် ကျွန်ုပ်တို့သည် ဦးစွာ derivative ၏ စိတ်ကူးကို ဖော်ပြရပါမည်။ "derivative" သည် average damping ကဲ့သို့ပင် ဖန်ရှင်တစ်ခုကို အခြားဖန်ရှင်တစ်ခုအဖြစ် ပြောင်းလဲပေးသည့် အရာတစ်ခု ဖြစ်ကြောင်း သတိပြုပါ။ ဥပမာအားဖြင့်၊ ဖန်ရှင် x ↦ x 3 ၏ derivative သည် ဖန်ရှင် x ↦ 3 x 2 ဖြစ်သည်။ ယေဘုယျအားဖြင့်၊ အကယ်၍ g သည် ဖန်ရှင်တစ်ခု ဖြစ်ပြီး d x သည် သေးငယ်သော ကိန်းဂဏန်းတစ်ခု ဖြစ်ပါက၊ g ၏ derivative D g သည် မည်သည့်ကိန်းဂဏန်း x တွင်မဆို (သေးငယ်သော d x ၏ အကန့်အသတ်အတွင်း) ၎င်း၏ တန်ဖိုးကို အောက်ပါအတိုင်း ပေးထားသော ဖန်ရှင် ဖြစ်သည်- D g ( x ) = g ( x + d x ) − g ( x ) d x . ထို့ကြောင့်၊ derivative ၏ စိတ်ကူးကို ( d x ကို 0.00001 ဟု ယူဆ၍) အောက်ပါ ပရိုစီဂျာအဖြစ် ဖော်ပြနိုင်သည်- 

(define (deriv g)
  (lambda (x)
    (/ (- (g (+ x dx)) (g x))
       dx)))

သတ်မှတ်ချက်နှင့်အတူ- 

(define dx 0.00001)

average-damp ကဲ့သို့ပင်၊ deriv သည် ပရိုစီဂျာတစ်ခုကို အာဂူမန့်အဖြစ် ယူပြီး ပရိုစီဂျာတစ်ခုကို တန်ဖိုးအဖြစ် ပြန်ပေးသော ပရိုစီဂျာတစ်ခု ဖြစ်သည်။ ဥပမာအားဖြင့်၊ x ↦ x 3 ၏ derivative ကို 5 တွင် ခန့်မှန်းရန် (ယင်း၏ တိကျသောတန်ဖိုးမှာ 75 ဖြစ်သည်) အောက်ပါတို့ကို တွက်ချက်နိုင်သည်- 

(define (cube x) (* x x x))

deriv ၏ အကူအညီဖြင့်၊ Newton ၏ နည်းလမ်းကို fixed-point လုပ်ငန်းစဉ်တစ်ခုအဖြစ် ဖော်ပြနိုင်သည်- 

(define (newton-transform g)
  (lambda (x)
    (- x (/ (g x) 
            ((deriv g) x)))))

(define (newtons-method g guess)
  (fixed-point (newton-transform g) 
               guess))

newton-transform ပရိုစီဂျာသည် ဤအပိုင်း၏ အစတွင် ဖော်ပြခဲ့သော ဖော်မြူလာကို ဖော်ပြပြီး newtons-method ကို ၎င်း၏အပေါ်တွင် အခြေခံ၍ အလွယ်တကူ သတ်မှတ်နိုင်သည်။ ၎င်းသည် ကျွန်ုပ်တို့ ညှိနှိုင်းလိုသော ဖန်ရှင်ကို တွက်ချက်ပေးသည့် ပရိုစီဂျာနှင့် အစပျိုး ခန့်မှန်းချက်တစ်ခုတို့ကို အာဂူမန့်များအဖြစ် ယူသည်။ ဥပမာအားဖြင့်၊ x ၏ နှစ်ထပ်ကိန်းရင်းကို ရှာရန်၊ အစပျိုး ခန့်မှန်းချက် 1 မှ စတင်၍ y ↦ y 2 − x ဖန်ရှင်၏ zero (သုညတန်ဖိုး) ကို ရှာဖွေရန် Newton ၏ နည်းလမ်းကို အသုံးပြုနိုင်ပါသည်။63

ဤသည်မှာ နှစ်ထပ်ကိန်းရင်းရှာသော ပရိုစီဂျာ၏ နောက်ထပ် ပုံစံတစ်ခု ဖြစ်သည်- 

(define (sqrt x)
  (newtons-method 
   (lambda (y) 
     (- (square y) x)) 
   1.0))

နှစ်ထပ်ကိန်းရင်း တွက်ချက်မှုကို ပိုမိုယေဘုယျကျသော နည်းလမ်းတစ်ခု၏ သာဓကတစ်ခုအဖြစ် ဖော်ပြရန် နည်းလမ်းနှစ်ခုကို ကျွန်ုပ်တို့ မြင်တွေ့ခဲ့ရပြီးဖြစ်သည် - တစ်ကြိမ်မှာ ကိန်းသေအမှတ် ရှာဖွေခြင်း အနေဖြင့်ဖြစ်ပြီး နောက်တစ်ကြိမ်မှာ Newton ၏ နည်းလမ်းကို အသုံးပြု၍ ဖြစ်သည်။ Newton ၏ နည်းလမ်းကိုယ်တိုင်က ကိန်းသေအမှတ် လုပ်ငန်းစဉ်တစ်ခုအဖြစ် ဖော်ပြထားသောကြောင့်၊ အမှန်စင်စစ်တွင် ကျွန်ုပ်တို့သည် နှစ်ထပ်ကိန်းရင်းများကို ကိန်းသေအမှတ်များအဖြစ် တွက်ချက်ရန် နည်းလမ်းနှစ်ခုကို မြင်တွေ့ခဲ့ရခြင်း ဖြစ်သည်။ နည်းလမ်းတစ်ခုစီသည် ဖန်ရှင်တစ်ခုမှ စတင်ပြီး ထိုဖန်ရှင်ကို အသွင်ပြောင်းလဲမှု (transformation) တစ်ခုခု လုပ်ဆောင်ထားသော အရာ၏ ကိန်းသေအမှတ်ကို ရှာဖွေသည်။ ဤယေဘုယျ စိတ်ကူးကို ပရိုစီဂျာတစ်ခုအဖြစ် အောက်ပါအတိုင်း ဖော်ပြနိုင်သည်- 

(define (fixed-point-of-transform 
         g transform guess)
  (fixed-point (transform g) guess))

ဤအလွန်ယေဘုယျကျသော ပရိုစီဂျာသည် ဖန်ရှင်အချို့ကို တွက်ချက်ပေးသော ပရိုစီဂျာ g၊ g ကို အသွင်ပြောင်းပေးသော ပရိုစီဂျာနှင့် အစပျိုး ခန့်မှန်းချက်တစ်ခုတို့ကို အာဂူမန့်များအဖြစ် ယူသည်။ ပြန်ပေးသော ရလဒ်သည် အသွင်ပြောင်းထားသော ဖန်ရှင်၏ ကိန်းသေအမှတ် ဖြစ်သည်။

ဤအကြမ်းဖျင်းပုံဖော်မှုကို အသုံးပြု၍၊ ဤအပိုင်းမှ ပထမဆုံး နှစ်ထပ်ကိန်းရင်း တွက်ချက်မှုကို ( y ↦ x / y ၏ average-damped ဗားရှင်း၏ ကိန်းသေအမှတ်ကို ရှာဖွေခြင်း) ဤယေဘုယျနည်းလမ်း၏ သာဓကတစ်ခုအဖြစ် ပြန်လည် ပုံဖော်နိုင်သည်- 

(define (sqrt x)
  (fixed-point-of-transform 
   (lambda (y) (/ x y))
   average-damp
   1.0))

အလားတူပင်၊ ဤအပိုင်းမှ ဒုတိယမြောက် နှစ်ထပ်ကိန်းရင်း တွက်ချက်မှုကို ( y ↦ y 2 − x ၏ Newton transform ၏ ကိန်းသေအမှတ်ကို ရှာဖွေသည့် Newton ၏ နည်းလမ်း သာဓကတစ်ခု) ကို အောက်ပါအတိုင်း ဖော်ပြနိုင်သည်- 

(define (sqrt x)
  (fixed-point-of-transform 
   (lambda (y) (- (square y) x))
   newton-transform
   1.0))

ပေါင်းစပ်ပရိုစီဂျာများ (compound procedures) သည် အလွန်အရေးကြီးသော အကြမ်းဖျင်းပုံဖော်မှု ယန္တရားတစ်ခုဖြစ်ကြောင်း လေ့လာတွေ့ရှိချက်ဖြင့် အပိုင်း 1.3 ကို စတင်ခဲ့သည်၊ အကြောင်းမှာ ၎င်းတို့သည် ယေဘုယျ တွက်ချက်မှု နည်းလမ်းများကို ကျွန်ုပ်တို့၏ ပရိုဂရမ်မင်းဘာသာစကားတွင် အတိအလင်း ဖော်ပြနိုင်သော အစိတ်အပိုင်းများအဖြစ် အသုံးပြုခွင့်ပေးသောကြောင့် ဖြစ်သည်။ ယခုအခါ အဆင့်မြင့်ပရိုစီဂျာများ (higher-order procedures) သည် နောက်ထပ် အကြမ်းဖျင်းပုံဖော်မှုများကို ဖန်တီးရန် ဤယေဘုယျ နည်းလမ်းများကို မည်သို့ စီမံခန့်ခွဲခွင့်ပေးသည်ကို ကျွန်ုပ်တို့ မြင်တွေ့ခဲ့ရပြီ ဖြစ်သည်။

ပရိုဂရမ်မာများအနေဖြင့်၊ ကျွန်ုပ်တို့သည် ကျွန်ုပ်တို့၏ ပရိုဂရမ်များရှိ အခြေခံ အကြမ်းဖျင်းပုံဖော်မှုများကို ဖော်ထုတ်ရန်နှင့် ပိုမိုအစွမ်းထက်သော အကြမ်းဖျင်းပုံဖော်မှုများကို ဖန်တီးရန် ၎င်းတို့အပေါ်တွင် အခြေခံ၍ တည်ဆောက်ရန်နှင့် ယေဘုယျပြုရန် အခွင့်အလမ်းများကို အမြဲနိုးကြားနေသင့်သည်။ ဤသည်မှာ ပရိုဂရမ်များကို အမြဲတမ်း အကြမ်းဖျင်းနိုင်သမျှ အရှိဆုံးပုံစံဖြင့် ရေးသားရမည်ဟု ဆိုလိုခြင်း မဟုတ်ပါ။ အတွေ့အကြုံရှိ ပရိုဂရမ်မာများသည် ၎င်းတို့၏ လုပ်ဆောင်ချက်နှင့် ကိုက်ညီသော သင့်လျော်သည့် အကြမ်းဖျင်းပုံဖော်မှု အဆင့်ကို မည်သို့ ရွေးချယ်ရမည်ကို သိကြသည်။ သို့သော် ဤအကြမ်းဖျင်းပုံဖော်မှုများ၏ ပုံစံဖြင့် စဉ်းစားနိုင်ရန် အရေးကြီးသည်၊ သို့မှသာ ၎င်းတို့ကို အခြေအနေသစ်များတွင် အသုံးချရန် အသင့်ရှိနေမည် ဖြစ်သည်။ အဆင့်မြင့်ပရိုစီဂျာများ၏ အရေးပါမှုမှာ ဤအကြမ်းဖျင်းပုံဖော်မှုများကို ကျွန်ုပ်တို့၏ ပရိုဂရမ်မင်းဘာသာစကားတွင် အခြားသော တွက်ချက်မှုဆိုင်ရာ အစိတ်အပိုင်းများကဲ့သို့ပင် ကိုင်တွယ်နိုင်သော အစိတ်အပိုင်းများအဖြစ် အတိအလင်း ကိုယ်စားပြုနိုင်စေခြင်းပင် ဖြစ်သည်။

ယေဘုယျအားဖြင့်၊ ပရိုဂရမ်မင်းဘာသာစကားများသည် တွက်ချက်မှုဆိုင်ရာ အစိတ်အပိုင်းများကို စီမံခန့်ခွဲနိုင်သည့် နည်းလမ်းများအပေါ် ကန့်သတ်ချက်များ ချမှတ်ထားတတ်သည်။ ကန့်သတ်ချက် အနည်းဆုံးရှိသော အစိတ်အပိုင်းများကို first-class အဆင့်ရှိသည်ဟု ဆိုကြသည်။ First-class အစိတ်အပိုင်းများ၏ "အခွင့်အရေးနှင့် အခွင့်ထူးများ" အချို့မှာ အောက်ပါအတိုင်း ဖြစ်သည်:64

  • ၎င်းတို့ကို ဗေရီယေးဘယ်လ်များ (variables) ဖြင့် အမည်ပေးနိုင်သည်။
  • ၎င်းတို့ကို ပရိုစီဂျာများသို့ အာဂူမန့်များအဖြစ် ပေးပို့နိုင်သည်။
  • ၎င်းတို့ကို ပရိုစီဂျာများ၏ ရလဒ်များအဖြစ် ပြန်ပေးနိုင်သည်။
  • ၎င်းတို့ကို ဒေတာတည်ဆောက်ပုံများ (data structures) ထဲတွင် ထည့်သွင်းနိုင်သည်။65

Lisp သည် အခြားသော အသုံးများသည့် ပရိုဂရမ်မင်းဘာသာစကားများနှင့် မတူဘဲ၊ ပရိုစီဂျာများကို အပြည့်အဝ first-class အဆင့် ပေးထားသည်။ ဤသည်မှာ ထိရောက်သော အကောင်အထည်ဖော်မှုအတွက် စိန်ခေါ်မှုများ ဖြစ်စေသော်လည်း၊ ရရှိလာသော ဖော်ပြနိုင်စွမ်းမှာ အလွန်ကြီးမားပါသည်။66

62 Newton’s method သည် အဖြေတစ်ခုဆီသို့ အမြဲတမ်း ဦးတည်သွားခြင်း (converge ဖြစ်ခြင်း) မရှိသော်လည်း၊ အဆင်ပြေသော အခြေအနေများတွင် iteration တစ်ခုစီသည် အဖြေ၏ အနီးစပ်ဆုံး တိကျမှု (accuracy) ကို ဂဏန်းနေရာ အရေအတွက်အားဖြင့် နှစ်ဆ တိုးလာစေကြောင်း ပြသနိုင်သည်။ ထိုသို့သော အခြေအနေများတွင် Newton’s method သည် half-interval method ထက် ပို၍မြန်ဆန်စွာ converge ဖြစ်လိမ့်မည်။

63 Square roots ရှာဖွေရာတွင် Newton’s method သည် မည်သည့် အစပျိုးမှတ်မှမဆို မှန်ကန်သော အဖြေသို့ လျင်မြန်စွာ ဦးတည်သွား (converge ဖြစ်) သည်။
