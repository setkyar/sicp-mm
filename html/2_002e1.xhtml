2.1Data Abstraction မိတ်ဆက်

1.1.8 တွင်၊ ပိုမိုရှုပ်ထွေးသော procedure တစ်ခုကို တည်ဆောက်ရာတွင် အစိတ်အပိုင်းတစ်ခုအဖြစ် အသုံးပြုသည့် procedure တစ်ခုကို သီးခြားလုပ်ဆောင်ချက်များ စုစည်းထားမှုအဖြစ်သာမက procedural abstraction တစ်ခုအဖြစ်လည်း ရှုမြင်နိုင်ကြောင်း ကျွန်ုပ်တို့ သတိပြုမိခဲ့သည်။ ဆိုလိုသည်မှာ procedure ကို မည်သို့အကောင်အထည်ဖော်ထားသည်ဟူသော အသေးစိတ်အချက်များကို ဖုံးကွယ်ထားနိုင်ပြီး၊ ၎င်း procedure ကို တူညီသော အထွေထွေလုပ်ဆောင်ချက်ရှိသည့် အခြား မည်သည့် procedure နှင့်မဆို အစားထိုးနိုင်သည်။ အခြားနည်းဖြင့် ဆိုရသော် procedure အား အသုံးပြုမည့် ပုံစံကို ပိုမိုအခြေခံကျသော procedure များဖြင့် အကောင်အထည်ဖော်မည့် အသေးစိတ်အချက်များမှ သီးခြားခွဲထုတ်ပေးသည့် abstraction တစ်ခုကို ကျွန်ုပ်တို့ ပြုလုပ်နိုင်သည်။ compound data အတွက် အလားတူအယူအဆကို data abstraction ဟု ခေါ်သည်။ Data abstraction သည် compound data object တစ်ခုအား အသုံးပြုပုံကို ၎င်းအား ပိုမိုအခြေခံကျသော data object များဖြင့် မည်သို့တည်ဆောက်သည်ဟူသော အသေးစိတ်အချက်များမှ သီးခြားခွဲထုတ်ရန် ကူညီပေးသည့် နည်းစနစ်တစ်ခုဖြစ်သည်။

Data abstraction ၏ အခြေခံစိတ်ကူးမှာ compound data object များကို အသုံးပြုမည့် ပရိုဂရမ်များကို “abstract data” ပေါ်တွင် လုပ်ဆောင်နိုင်စေရန် ဖွဲ့စည်းတည်ဆောက်ရန်ဖြစ်သည်။ ဆိုလိုသည်မှာ ကျွန်ုပ်တို့၏ ပရိုဂရမ်များသည် လုပ်ဆောင်ရမည့် လုပ်ငန်းစဉ်အတွက် အမှန်တကယ် မလိုအပ်သော data ဆိုင်ရာ ယူဆချက်များကို မပြုလုပ်ဘဲ data ကို အသုံးပြုသင့်သည်။ တစ်ချိန်တည်းမှာပင်၊ data ကို အသုံးပြုသော ပရိုဂရမ်များနှင့် သီးခြားလွတ်လပ်သော “concrete” data representation တစ်ခုကို သတ်မှတ်သည်။ ကျွန်ုပ်တို့ စနစ်၏ ဤအပိုင်းနှစ်ခုကြားရှိ interface မှာ abstract data ကို concrete representation အဖြစ် အကောင်အထည်ဖော်ပေးသည့် selectors နှင့် constructors ဟုခေါ်သော procedure အစုအဝေးတစ်ခု ဖြစ်လိမ့်မည်။ ဤနည်းစနစ်ကို သရုပ်ဖော်ရန်အတွက် rational numbers များကို ကိုင်တွယ်အသုံးပြုရန် procedure အစုအဝေးတစ်ခုကို မည်သို့ဒီဇိုင်းထုတ်ရမည်ကို ကျွန်ုပ်တို့ စဉ်းစားပါမည်။

2.1.1ဥပမာ- ရာရှင်နယ်ကိန်းများအတွက် သင်္ချာလုပ်ဆောင်ချက်များ

ကျွန်ုပ်တို့သည် ရာရှင်နယ်ကိန်း (rational numbers) များဖြင့် သင်္ချာတွက်ချက်မှုများ ပြုလုပ်လိုသည်ဟု ဆိုပါစို့။ ၎င်းတို့ကို ပေါင်း၊ နှုတ်၊ မြှောက်၊ စား ပြုလုပ်နိုင်ရန်နှင့် ရာရှင်နယ်ကိန်းနှစ်ခု ညီ၊ မညီ စမ်းသပ်နိုင်ရန် ကျွန်ုပ်တို့ အလိုရှိသည်။

အပိုင်းဝေ (numerator) နှင့် အပိုင်းခြေ (denominator) တို့မှ ရာရှင်နယ်ကိန်းတစ်ခုကို တည်ဆောက်ရန် နည်းလမ်းတစ်ခု ကျွန်ုပ်တို့ထံတွင် ရှိပြီးသားဖြစ်သည်ဟု အစပြု၍ ယူဆကြပါစို့။ ထို့ပြင် ရာရှင်နယ်ကိန်းတစ်ခု ပေးထားပါက၊ ၎င်း၏ အပိုင်းဝေနှင့် အပိုင်းခြေတို့ကို ထုတ်ယူရန် (သို့မဟုတ် ရွေးထုတ်ရန်) နည်းလမ်းတစ်ခု ရှိသည်ဟုလည်း ယူဆမည်။ ထပ်မံ၍ constructor နှင့် selector များကို procedure များအဖြစ် ရရှိနိုင်သည်ဟု ယူဆကြပါစို့-

  • (make-rat ⟨n⟩ ⟨d⟩) သည် ကိန်းပြည့် ⟨n⟩ ကို အပိုင်းဝေအဖြစ်လည်းကောင်း၊ ကိန်းပြည့် ⟨d⟩ ကို အပိုင်းခြေအဖြစ်လည်းကောင်း အသုံးပြုထားသည့် ရာရှင်နယ်ကိန်းကို ပြန်ပေးသည်။
  • (numer ⟨x⟩) သည် ရာရှင်နယ်ကိန်း ⟨x⟩ ၏ အပိုင်းဝေကို ပြန်ပေးသည်။
  • (denom ⟨x⟩) သည် ရာရှင်နယ်ကိန်း ⟨x⟩ ၏ အပိုင်းခြေကို ပြန်ပေးသည်။

ဤနေရာတွင် ကျွန်ုပ်တို့သည် ပေါင်းစပ်ဖွဲ့စည်းခြင်းဆိုင်ရာ အစွမ်းထက်သော ဗျူဟာတစ်ခုဖြစ်သည့် wishful thinking ကို အသုံးပြုနေခြင်းဖြစ်သည်။ ရာရှင်နယ်ကိန်းတစ်ခုကို မည်သို့ ဖော်ပြမည်၊ သို့မဟုတ် numer၊ denom နှင့် make-rat procedure များကို မည်သို့ အကောင်အထည်ဖော်မည်ကို ကျွန်ုပ်တို့ မပြောရသေးပါ။ သို့သော်လည်း ကျွန်ုပ်တို့တွင် ဤ procedure သုံးခု ရှိခဲ့မည်ဆိုပါက၊ အောက်ပါ ဆက်သွယ်ချက်များကို အသုံးပြု၍ ပေါင်း၊ နှုတ်၊ မြှောက်၊ စား ပြုလုပ်နိုင်ပြီး ညီမျှမှုကို စမ်းသပ်နိုင်မည်ဖြစ်သည်- n 1 d 1 + n 2 d 2 = n 1 d 2 + n 2 d 1 d 1 d 2 , n 1 d 1 − n 2 d 2 = n 1 d 2 − n 2 d 1 d 1 d 2 , n 1 d 1 × n 2 d 2 = n 1 n 2 d 1 d 2 , n 1 / d 1 n 2 / d 2 = n 1 d 2 d 1 n 2 , n 1 d 1 = n 2 d 2   i f   a n d   o n l y   i f n 1 d 2 = n 2 d 1 . ဤစည်းမျဉ်းများကို procedure များအဖြစ် ဖော်ပြနိုင်သည်- 

(define (add-rat x y)
  (make-rat (+ (* (numer x) (denom y))
               (* (numer y) (denom x)))
            (* (denom x) (denom y))))

(define (sub-rat x y)
  (make-rat (- (* (numer x) (denom y))
               (* (numer y) (denom x)))
            (* (denom x) (denom y))))

(define (mul-rat x y)
  (make-rat (* (numer x) (numer y))
            (* (denom x) (denom y))))

(define (div-rat x y)
  (make-rat (* (numer x) (denom y))
            (* (denom x) (numer y))))

(define (equal-rat? x y)
  (= (* (numer x) (denom y))
     (* (numer y) (denom x))))

ယခုအခါ ကျွန်ုပ်တို့တွင် selector နှင့် constructor procedure များဖြစ်သော numer၊ denom နှင့် make-rat တို့ကို အသုံးပြု၍ သတ်မှတ်ထားသော ရာရှင်နယ်ကိန်းဆိုင်ရာ လုပ်ဆောင်ချက်များ ရှိပြီဖြစ်သည်။ သို့သော် ၎င်းတို့ကို ကျွန်ုပ်တို့ မသတ်မှတ်ရသေးပါ။ ကျွန်ုပ်တို့ လိုအပ်သည်မှာ ရာရှင်နယ်ကိန်းတစ်ခု ဖြစ်လာစေရန် အပိုင်းဝေနှင့် အပိုင်းခြေတို့ကို တစ်ခုတည်းဖြစ်အောင် တွဲကပ်ပေးမည့် နည်းလမ်းတစ်ခု ဖြစ်သည်။

အတွဲများ (Pairs)

ကျွန်ုပ်တို့၏ data abstraction ၏ လက်တွေ့အဆင့် (concrete level) ကို အကောင်အထည်ဖော်နိုင်ရန်၊ ကျွန်ုပ်တို့၏ programming language သည် pair ဟုခေါ်သော ပေါင်းစပ်ဖွဲ့စည်းပုံတစ်ခုကို ထောက်ပံ့ပေးထားပြီး၊ ၎င်းကို primitive procedure ဖြစ်သော cons ဖြင့် တည်ဆောက်နိုင်သည်။ ဤ procedure သည် argument နှစ်ခုကို ယူပြီး၊ ထို argument နှစ်ခုကို အစိတ်အပိုင်းများအဖြစ် ပါဝင်သော ပေါင်းစပ် data object တစ်ခုကို ပြန်ပေးသည်။ pair တစ်ခု ပေးထားပါက၊ primitive procedure များဖြစ်သော car နှင့် cdr ကို အသုံးပြု၍ အစိတ်အပိုင်းများကို ထုတ်ယူနိုင်သည်။68 ထို့ကြောင့် cons၊ car နှင့် cdr တို့ကို အောက်ပါအတိုင်း အသုံးပြုနိုင်သည်- 

(define x (cons 1 2))

(car x)
1

(cdr x)
2

pair တစ်ခုသည် primitive data object တစ်ခုကဲ့သို့ပင် အမည်ပေးနိုင်ပြီး ကိုင်တွယ်အသုံးပြုနိုင်သော data object တစ်ခုဖြစ်သည်ကို သတိပြုပါ။ ထို့ပြင် cons ကို အသုံးပြု၍ အစိတ်အပိုင်းများကိုယ်တိုင်က pair များဖြစ်နေသော အတွဲများကို တည်ဆောက်နိုင်ပြီး၊ ထိုနည်းအတိုင်း ဆက်လက် လုပ်ဆောင်သွားနိုင်သည်- 

(define x (cons 1 2))
(define y (cons 3 4))
(define z (cons x y))

(car (car z))
1

(car (cdr z))
3

2.2 တွင် pair များကို ပေါင်းစပ်နိုင်သည့် ဤစွမ်းရည်ကြောင့် pair များကို ရှုပ်ထွေးသော data structure အမျိုးမျိုး ဖန်တီးရန်အတွက် အထွေထွေသုံး အခြေခံအစိတ်အပိုင်းများအဖြစ် အသုံးပြုနိုင်ပုံကို တွေ့ရမည်ဖြစ်သည်။ cons၊ car နှင့် cdr procedure များဖြင့် အကောင်အထည်ဖော်ထားသော single compound-data primitive ဖြစ်သည့် pair သည် ကျွန်ုပ်တို့ လိုအပ်သော တစ်ခုတည်းသော တွဲကပ်ပစ္စည်း (glue) ဖြစ်သည်။ pair များမှ တည်ဆောက်ထားသော data object များကို list-structured data ဟု ခေါ်သည်။

ရာရှင်နယ်ကိန်းများကို ကိုယ်စားပြုဖော်ပြခြင်း

pair များသည် ရာရှင်နယ်ကိန်းစနစ်ကို ပြီးပြည့်စုံစေရန် သဘာဝကျသော နည်းလမ်းတစ်ခုကို ပေးစွမ်းသည်။ ရာရှင်နယ်ကိန်းတစ်ခုကို အပိုင်းဝေနှင့် အပိုင်းခြေဖြစ်သော ကိန်းပြည့်နှစ်ခုပါဝင်သည့် pair တစ်ခုအဖြစ် ဖော်ပြရုံသာ ဖြစ်သည်။ ထိုအခါ make-rat၊ numer နှင့် denom တို့ကို အောက်ပါအတိုင်း အလွယ်တကူ အကောင်အထည်ဖော်နိုင်သည်-69 

(define (make-rat n d) (cons n d))
(define (numer x) (car x))
(define (denom x) (cdr x))

ထို့ပြင် ကျွန်ုပ်တို့၏ တွက်ချက်မှု ရလဒ်များကို ပြသရန်အတွက်၊ အပိုင်းဝေ၊ slash (စောင်းစောင်းမျဉ်း) နှင့် အပိုင်းခြေတို့ကို ရိုက်နှိပ်ခြင်းဖြင့် ရာရှင်နယ်ကိန်းများကို print ထုတ်နိုင်သည်-70 

(define (print-rat x)
  (newline)
  (display (numer x))
  (display "/")
  (display (denom x)))

ယခု ကျွန်ုပ်တို့၏ ရာရှင်နယ်ကိန်းဆိုင်ရာ procedure များကို စမ်းသပ်ကြည့်နိုင်ပြီ- 

(define one-half (make-rat 1 2))
(print-rat one-half)
1/2

(define one-third (make-rat 1 3))
(print-rat
 (add-rat one-half one-third))
5/6

(print-rat
 (mul-rat one-half one-third))
1/6

(print-rat
 (add-rat one-third one-third))
6/9

နောက်ဆုံး ဥပမာတွင် ပြထားသည့်အတိုင်း၊ ကျွန်ုပ်တို့၏ ရာရှင်နယ်ကိန်း အကောင်အထည်ဖော်မှုသည် ရာရှင်နယ်ကိန်းများကို အငယ်ဆုံးပုံစံ (lowest terms) သို့ မလျှော့ချပေးပါ။ ဤအချက်ကို make-rat ကို ပြောင်းလဲခြင်းဖြင့် ပြုပြင်နိုင်သည်။ အကယ်၍ ကျွန်ုပ်တို့တွင် 1.2.5 ရှိ ကိန်းပြည့်နှစ်ခု၏ အကြီးဆုံးဘုံဆခွဲကိန်းကို ထုတ်ပေးသော gcd procedure ကဲ့သို့သော procedure တစ်ခု ရှိပါက၊ pair မတည်ဆောက်မီ အပိုင်းဝေနှင့် အပိုင်းခြေကို အငယ်ဆုံးပုံစံသို့ လျှော့ချရန် gcd ကို အသုံးပြုနိုင်သည်။ 

(define (make-rat n d)
  (let ((g (gcd n d)))
    (cons (/ n g) 
          (/ d g))))

ရရှိပြီဖြစ်သည်။ ဤပြုပြင်ပြောင်းလဲမှုကို အမှန်တကယ် လုပ်ဆောင်ချက်များကို အကောင်အထည်ဖော်သည့် procedure များ (ဥပမာ add-rat နှင့် mul-rat) ကို မပြောင်းလဲဘဲ၊ constructor make-rat ကို ပြောင်းလဲခြင်းဖြင့်သာ လုပ်ဆောင်နိုင်ခဲ့ခြင်း ဖြစ်သည်။

လေ့ကျင့်ခန်း ၂.၁- အပေါင်းကိန်းနှင့် အနှုတ်ကိန်း argument နှစ်မျိုးလုံးကို ကိုင်တွယ်နိုင်သော ပိုမိုကောင်းမွန်သည့် make-rat version တစ်ခုကို သတ်မှတ်ပါ။ make-rat သည် လက္ခဏာ (sign) ကို ပုံမှန်ညှိပေး (normalize လုပ်ပေး) ရမည်ဖြစ်ရာ ရာရှင်နယ်ကိန်းသည် အပေါင်းကိန်းဖြစ်ပါက အပိုင်းဝေနှင့် အပိုင်းခြေ နှစ်ခုလုံး အပေါင်းဖြစ်စေရန်နှင့်၊ ရာရှင်နယ်ကိန်းသည် အနှုတ်ကိန်းဖြစ်ပါက အပိုင်းဝေသာ အနှုတ်ဖြစ်စေရန် လုပ်ဆောင်ရမည်။

2.1.2အနှစ်ထုတ်ခြင်း အတားအဆီးများ (Abstraction Barriers)

စုပေါင်းဒေတာ (compound data) နှင့် ဒေတာ အနှစ်ထုတ်ခြင်း (data abstraction) ဆိုင်ရာ ဥပမာများကို ဆက်လက်မလေ့လာမီ၊ ရာရှင်နယ်ကိန်း (rational-number) ဥပမာမှ ပေါ်ထွက်လာသော အချက်အချို့ကို သုံးသပ်ကြည့်ကြပါစို့။ ကျွန်ုပ်တို့သည် ရာရှင်နယ်ကိန်း လုပ်ဆောင်ချက်များကို make-rat ဟူသော တည်ဆောက်သူ (constructor) နှင့် numer၊ denom ဟူသော ရွေးချယ်သူများ (selectors) ဖြင့် အဓိပ္ပာယ်ဖွင့်ဆိုခဲ့သည်။ ယေဘုယျအားဖြင့် ဒေတာ အနှစ်ထုတ်ခြင်း၏ အခြေခံ အယူအဆမှာ ဒေတာ အရာဝတ္ထု (data object) အမျိုးအစား တစ်ခုစီအတွက် အခြေခံ လုပ်ဆောင်ချက် အစုအဝေးတစ်ခုကို သတ်မှတ်ရန်ဖြစ်ပြီး၊ ထိုအမျိုးအစားရှိသော ဒေတာ အရာဝတ္ထုများကို ကိုင်တွယ်ခြယ်လှယ်မှု အားလုံးကို ထိုလုပ်ဆောင်ချက်များဖြင့်သာ ဖော်ပြရန်နှင့် ဒေတာကို ကိုင်တွယ်ရာတွင် ထိုလုပ်ဆောင်ချက်များကိုသာ အသုံးပြုရန် ဖြစ်သည်။

ရာရှင်နယ်ကိန်း စနစ်၏ တည်ဆောက်ပုံကို ပုံ ၂.၁ တွင် ပြထားသည့်အတိုင်း မြင်ယောင်ကြည့်နိုင်သည်။ အလျားလိုက်မျဉ်းများသည် စနစ်၏ မတူညီသော "အဆင့်များ" (levels) ကို သီးခြားခွဲထုတ်ထားသည့် အနှစ်ထုတ်ခြင်း အတားအဆီးများ (abstraction barriers) ကို ကိုယ်စားပြုသည်။ အဆင့်တစ်ခုစီတွင်၊ ထိုအတားအဆီးသည် ဒေတာ အနှစ်ထုတ်ခြင်းကို အသုံးပြုသည့် (အပေါ်ဘက်ရှိ) ပရိုဂရမ်များကို ဒေတာ အနှစ်ထုတ်ခြင်းကို အကောင်အထည်ဖော်သည့် (အောက်ဘက်ရှိ) ပရိုဂရမ်များမှ ခွဲခြားပေးထားသည်။ ရာရှင်နယ်ကိန်းများကို အသုံးပြုသည့် ပရိုဂရမ်များသည် ရာရှင်နယ်ကိန်း ပက်ကေ့ချ် (rational-number package) မှ "အများပြည်သူသုံးရန်" ပံ့ပိုးပေးထားသော လုပ်ထုံးလုပ်နည်းများဖြစ်သည့် add-rat, sub-rat, mul-rat, div-rat, နှင့် equal-rat? တို့ဖြင့်သာ ကိုင်တွယ်အသုံးပြုကြသည်။ ၎င်းတို့ကို တစ်ဖန် make-rat, numer, နှင့် denom ဟူသော တည်ဆောက်သူနှင့် ရွေးချယ်သူများဖြင့်သာ အကောင်အထည်ဖော်ထားပြီး၊ ထိုတည်ဆောက်သူနှင့် ရွေးချယ်သူများကိုယ်တိုင်ကို pair များဖြင့် အကောင်အထည်ဖော်ထားသည်။ pair များကို cons, car, နှင့် cdr တို့ဖြင့် ကိုင်တွယ်နိုင်သရွေ့ pair များကို မည်သို့ အကောင်အထည်ဖော်ထားသည်ဟူသော အသေးစိတ်အချက်အလက်များသည် ကျန်ရှိသော ရာရှင်နယ်ကိန်း ပက်ကေ့ချ်အတွက် အရေးမကြီးပါ။ လက်တွေ့တွင်၊ အဆင့်တစ်ခုစီရှိ လုပ်ထုံးလုပ်နည်းများသည် အနှစ်ထုတ်ခြင်း အတားအဆီးများကို သတ်မှတ်ပေးပြီး မတူညီသော အဆင့်များကို ချိတ်ဆက်ပေးသည့် အပြန်အလှန် ချိတ်ဆက်မှုများ (interfaces) ပင် ဖြစ်ကြသည်။

ဤရိုးရှင်းသော အယူအဆတွင် အားသာချက်များစွာ ရှိသည်။ အားသာချက်တစ်ခုမှာ ပရိုဂရမ်များကို ထိန်းသိမ်းရန်နှင့် ပြုပြင်မွမ်းမံရန် ပိုမိုလွယ်ကူစေခြင်းဖြစ်သည်။ မည်သည့် ရှုပ်ထွေးသော ဒေတာ တည်ဆောက်ပုံကိုမဆို ပရိုဂရမ်မင်းဘာသာစကားမှ ပံ့ပိုးပေးထားသော အခြေခံ ဒေတာ တည်ဆောက်ပုံများ (primitive data structures) ဖြင့် နည်းမျိုးစုံဖြင့် ကိုယ်စားပြုဖော်ပြနိုင်သည်။ ဒေတာကို ကိုယ်စားပြုဖော်ပြပုံ ရွေးချယ်မှုသည် ၎င်းအပေါ်တွင် အလုပ်လုပ်သည့် ပရိုဂရမ်များအပေါ် သက်ရောက်မှုရှိသည်မှာ ထင်ရှားပါသည်။ ထို့ကြောင့် နောက်ပိုင်းတွင် ဖော်ပြပုံစနစ်ကို ပြောင်းလဲခဲ့လျှင် ထိုပရိုဂရမ်အားလုံးကို လိုက်လျောညီထွေဖြစ်အောင် ပြုပြင်ရပေမည်။ ဖော်ပြပုံစနစ်အပေါ် မှီခိုနေရမှုကို ပရိုဂရမ် မော်ဂျူး (module) အနည်းငယ်အတွင်း၌သာ ကန့်သတ်ထားရန် စနစ်တကျ ဒီဇိုင်းမဆွဲထားပါက၊ ကြီးမားသော ပရိုဂရမ်များအတွက် ဤအလုပ်သည် အချိန်ကုန်ပြီး ကုန်ကျစရိတ် များပြားနိုင်သည်။

ဥပမာအားဖြင့်၊ ရာရှင်နယ်ကိန်းများကို အရှင်းဆုံးပုံစံ (lowest terms) သို့ လျှော့ချသည့် ပြဿနာကို ဖြေရှင်းရန် အခြားနည်းလမ်းတစ်ခုမှာ ရာရှင်နယ်ကိန်းကို တည်ဆောက်ချိန်တွင် မဟုတ်ဘဲ ၎င်း၏ အစိတ်အပိုင်းများကို ရယူအသုံးပြုသည့် အချိန်တိုင်းတွင် လျှော့ချခြင်း ဖြစ်သည်။ ဤနည်းလမ်းသည် မတူညီသော တည်ဆောက်သူနှင့် ရွေးချယ်သူ လုပ်ထုံးလုပ်နည်းများကို ဖြစ်ပေါ်စေသည်- 

(define (make-rat n d)
  (cons n d))

(define (numer x)
  (let ((g (gcd (car x) (cdr x))))
    (/ (car x) g)))

(define (denom x)
  (let ((g (gcd (car x) (cdr x))))

ဖော်ပြပုံစနစ်အပေါ် မှီခိုမှုကို အပြန်အလှန် ချိတ်ဆက်မှုဆိုင်ရာ လုပ်ထုံးလုပ်နည်း အနည်းငယ်အတွင်း၌သာ ကန့်သတ်ထားခြင်းသည် ပရိုဂရမ်များကို ဒီဇိုင်းဆွဲရာတွင်ရော၊ ပြုပြင်မွမ်းမံရာတွင်ရော အထောက်အကူပြုသည်။ အဘယ်ကြောင့်ဆိုသော် ၎င်းသည် အခြား အကောင်အထည်ဖော်မှုများကို ထည့်သွင်းစဉ်းစားနိုင်ရန် ပျော့ပြောင်းမှုကို ပေးသောကြောင့် ဖြစ်သည်။ ကျွန်ုပ်တို့၏ ရိုးရှင်းသော ဥပမာကို ဆက်လက်ပြောရလျှင်၊ ကျွန်ုပ်တို့သည် ရာရှင်နယ်ကိန်း ပက်ကေ့ချ်တစ်ခုကို ဒီဇိုင်းဆွဲနေပြီး gcd ကို တည်ဆောက်ချိန်တွင် တွက်မည်လား သို့မဟုတ် ရွေးချယ်ချိန်တွင် တွက်မည်လားဆိုသည်ကို အစပိုင်းတွင် မဆုံးဖြတ်နိုင်သေးဟု ဆိုပါစို့။ ဒေတာ အနှစ်ထုတ်ခြင်း နည်းစနစ်သည် စနစ်၏ ကျန်ရှိသော အစိတ်အပိုင်းများကို ဆက်လက်လုပ်ဆောင်နိုင်စွမ်း မဆုံးရှုံးစေဘဲ ထိုဆုံးဖြတ်ချက်ကို ရွှေ့ဆိုင်းထားနိုင်ရန် လမ်းစတစ်ခု ပေးထားသည်။

လေ့ကျင့်ခန်း ၂.၂: ပြင်ညီ (plane) တစ်ခုပေါ်ရှိ မျဉ်းပိုင်း (line segment) များကို ကိုယ်စားပြုဖော်ပြသည့် ပြဿနာကို သုံးသပ်ကြည့်ပါ။ မျဉ်းပိုင်းတစ်ခုစီကို အမှတ်နှစ်ခု (စမှတ်နှင့် ဆုံးမှတ်) ပါဝင်သော pair တစ်ခုအဖြစ် ဖော်ပြသည်။ မျဉ်းပိုင်းများကို အမှတ်များဖြင့် ဖော်ပြပုံကို သတ်မှတ်ပေးသည့် make-segment ဟူသော တည်ဆောက်သူနှင့် start-segment၊ end-segment ဟူသော ရွေးချယ်သူများကို အဓိပ္ပာယ်ဖွင့်ဆိုပါ။ ထို့အပြင်၊ အမှတ်တစ်ခုကို ကိန်းဂဏန်းနှစ်ခု ( x ကိုဩဒိနိတ် နှင့် y ကိုဩဒိနိတ်) ပါဝင်သော pair တစ်ခုအဖြစ် ဖော်ပြနိုင်သည်။ ထိုသို့ဖော်ပြရန်အတွက် make-point ဟူသော တည်ဆောက်သူနှင့် x-point၊ y-point ဟူသော ရွေးချယ်သူများကို သတ်မှတ်ပါ။ နောက်ဆုံးတွင်၊ သင်၏ ရွေးချယ်သူများနှင့် တည်ဆောက်သူများကို အသုံးပြု၍ မျဉ်းပိုင်းတစ်ခုကို အခြေခံကိန်း (argument) အဖြစ် လက်ခံပြီး ၎င်း၏ အလယ်မှတ် (midpoint - အစွန်းမှတ်များ၏ ကိုဩဒိနိတ်များ ပျမ်းမျှတန်ဖိုးရှိသော အမှတ်) ကို ပြန်ပေးသည့် midpoint-segment ဟူသော လုပ်ထုံးလုပ်နည်းကို အဓိပ္ပာယ်ဖွင့်ဆိုပါ။ သင်၏ လုပ်ထုံးလုပ်နည်းများကို စမ်းသပ်ရန်အတွက် အမှတ်များကို ပုံနှိပ်ထုတ်ရန် နည်းလမ်းတစ်ခု လိုအပ်ပါလိမ့်မည်-

2.1.3ဒေတာ (Data) ဆိုသည်မှာ အဘယ်နည်း?

သို့သော် data ဆိုသည်မှာ အတိအကျအားဖြင့် အဘယ်နည်း? "ပေးထားသော selector များနှင့် constructor များဖြင့် အကောင်အထည်ဖော်ထားသမျှ" ဟု ပြောရုံနှင့် မလုံလောက်ပါ။ ရှင်းလင်းစွာပင်၊ ကြုံရာ procedure သုံးခုအစုတိုင်းသည် rational-number အကောင်အထည်ဖော်မှုအတွက် သင့်လျော်သော အခြေခံအဖြစ် မဆောင်ရွက်နိုင်ပါ။ အကယ်၍ ကျွန်ုပ်တို့သည် integer n နှင့် d အစုံမှ rational number x ကို တည်ဆောက်ပါက၊ x ၏ numer နှင့် denom ကို ထုတ်ယူပြီး ၎င်းတို့ကို စားခြင်းသည် n ကို d နှင့် စားခြင်းနှင့် တူညီသော ရလဒ်ကို ရရှိရန် အာမခံချက်ရှိရန် လိုအပ်သည်။ တစ်နည်းအားဖြင့်ဆိုသော်၊ make-rat၊ numer နှင့် denom တို့သည် အောက်ပါအခြေအနေကို ကျေနပ်ရမည်- မည်သည့် integer n နှင့် သုညမဟုတ်သော integer d အတွက်မဆို၊ အကယ်၍ x သည် (make-rat n d) ဖြစ်ပါက၊ (numer x) (denom x) = n d . အမှန်စင်စစ်၊ ဤသည်မှာ rational-number ကို ကိုယ်စားပြုရန် (representation) အတွက် သင့်လျော်သော အခြေခံတစ်ခုဖြစ်စေရန် make-rat၊ numer နှင့် denom တို့ ဖြည့်ဆည်းရမည့် တစ်ခုတည်းသော အခြေအနေဖြစ်သည်။ ယေဘုယျအားဖြင့်၊ ဒေတာကို selector များနှင့် constructor များ အစုအဝေးတစ်ခုအပြင်၊ မှန်ကန်သော ကိုယ်စားပြုမှု (representation) ဖြစ်ရန်အတွက် ဤ procedure များ ဖြည့်ဆည်းရမည့် သတ်မှတ်ထားသော အခြေအနေများဖြင့် အဓိပ္ပာယ်ဖွင့်ဆိုထားသည်ဟု ကျွန်ုပ်တို့ ယူဆနိုင်သည်။71

ဤအမြင်သည် rational number ကဲ့သို့သော "high-level" data object များသာမက၊ lower-level object များကိုပါ အဓိပ္ပာယ်ဖွင့်ဆိုရန် အသုံးဝင်သည်။ ကျွန်ုပ်တို့၏ rational number များကို အဓိပ္ပာယ်ဖွင့်ဆိုရန် အသုံးပြုခဲ့သော pair (အစုံ) ဆိုသည့် အယူအဆကို စဉ်းစားကြည့်ပါ။ pair ဆိုသည်မှာ အဘယ်နည်းဆိုသည်ကို ကျွန်ုပ်တို့ အမှန်တကယ် မပြောခဲ့ဘဲ၊ language မှ pair များကို လုပ်ဆောင်ရန်အတွက် cons၊ car နှင့် cdr ဟူသော procedure များကို ပံ့ပိုးပေးထားကြောင်းသာ ပြောခဲ့သည်။ သို့သော် ဤလုပ်ဆောင်ချက် သုံးခုနှင့် ပတ်သက်၍ ကျွန်ုပ်တို့ သိရန်လိုအပ်သော တစ်ခုတည်းသော အချက်မှာ အကယ်၍ ကျွန်ုပ်တို့သည် cons ကို အသုံးပြု၍ object နှစ်ခုကို ပေါင်းစည်းပါက၊ car နှင့် cdr ကို အသုံးပြု၍ ထို object များကို ပြန်လည်ရယူနိုင်သည်ဟူသော အချက်ဖြစ်သည်။ ဆိုလိုသည်မှာ၊ မည်သည့် object x နှင့် y အတွက်မဆို၊ အကယ်၍ z သည် (cons x y) ဖြစ်ပါက (car z) သည် x ဖြစ်ပြီး (cdr z) သည် y ဖြစ်သည်ဟူသော အခြေအနေကို ဤလုပ်ဆောင်ချက်များက ကျေနပ်စေသည်။ အမှန်စင်စစ်၊ ဤ procedure သုံးခုကို ကျွန်ုပ်တို့၏ language တွင် primitive များအဖြစ် ထည့်သွင်းထားကြောင်း ကျွန်ုပ်တို့ ဖော်ပြခဲ့သည်။ သို့သော်၊ အထက်ပါအခြေအနေကို ကျေနပ်စေသော မည်သည့် procedure သုံးခုအစုကိုမဆို pair များကို အကောင်အထည်ဖော်ရန် အခြေခံအဖြစ် အသုံးပြုနိုင်သည်။ မည်သည့် data structure ကိုမျှ အသုံးမပြုဘဲ procedure များကိုသာ အသုံးပြု၍ cons၊ car နှင့် cdr တို့ကို အကောင်အထည်ဖော်နိုင်သည်ဟူသော အချက်က ဤအချက်ကို ထင်ရှားစွာ သရုပ်ဖော်နေပါသည်။ အဓိပ္ပာယ်ဖွင့်ဆိုချက်များမှာ အောက်ပါအတိုင်းဖြစ်သည်- 

(define (cons x y)
  (define (dispatch m)
    (cond ((= m 0) x)
          ((= m 1) y)
          (else 
           (error "Argument not 0 or 1:
                   CONS" m))))
  dispatch)

(define (car z) (z 0))
(define (cdr z) (z 1))

ဤကဲ့သို့ procedure များကို အသုံးပြုခြင်းသည် ဒေတာသည် မည်သို့ဖြစ်သင့်သည်ဟူသော ကျွန်ုပ်တို့၏ ပင်ကိုယ်အသိစိတ် (intuitive notion) နှင့် လုံးဝမတူညီပါ။ သို့သော်လည်း၊ ဤသည်မှာ pair များကို ကိုယ်စားပြုရန် မှန်ကန်သောနည်းလမ်းဖြစ်ကြောင်း ပြသရန်အတွက် ဤ procedure များသည် အထက်တွင်ဖော်ပြထားသော အခြေအနေကို ကျေနပ်မှုရှိမရှိ စစ်ဆေးရန်သာ လိုအပ်သည်။

သတိပြုရန် သိမ်မွေ့သော အချက်မှာ (cons x y) မှ ပြန်ပေးသော တန်ဖိုး (value) သည် procedure တစ်ခု ဖြစ်နေခြင်းပင်ဖြစ်သည် — ဆိုလိုသည်မှာ အတွင်း၌ အဓိပ္ပာယ်ဖွင့်ဆိုထားသော dispatch အမည်ရှိ procedure ဖြစ်ပြီး၊ ၎င်းသည် argument တစ်ခုကို လက်ခံကာ ထို argument သည် 0 သို့မဟုတ် 1 ဖြစ်မှုအပေါ် မူတည်၍ x သို့မဟုတ် y ကို ပြန်ပေးသည်။ ထို့အတူပင်၊ (car z) ကို z အား 0 နှင့် အသုံးပြုရန် (apply) အဓိပ္ပာယ်ဖွင့်ဆိုထားသည်။ ထို့ကြောင့်၊ အကယ်၍ z သည် (cons x y) မှ ဖွဲ့စည်းထားသော procedure ဖြစ်ပါက၊ z ကို 0 နှင့် အသုံးပြုခြင်းသည် x ကို ရရှိစေမည်ဖြစ်သည်။ ထို့ကြောင့် (car (cons x y)) သည် လိုလားချက်အတိုင်း x ကို ရရှိစေကြောင်း ကျွန်ုပ်တို့ ပြသခဲ့ပြီးဖြစ်သည်။ ထို့အတူပင်၊ (cdr (cons x y)) သည် (cons x y) မှ ပြန်ပေးသော procedure ကို 1 နှင့် အသုံးပြုပြီး y ကို ပြန်ပေးသည်။ ထို့ကြောင့်၊ ဤကဲ့သို့ procedure ဖြင့် pair များကို အကောင်အထည်ဖော်ခြင်းသည် မှန်ကန်သော အကောင်အထည်ဖော်မှုတစ်ခုဖြစ်ပြီး၊ ကျွန်ုပ်တို့သည် cons၊ car နှင့် cdr တို့ကိုသာ အသုံးပြု၍ pair များကို အသုံးပြုမည်ဆိုပါက ဤအကောင်အထည်ဖော်မှုကို "အစစ်အမှန်" data structure များ အသုံးပြုထားသည့် အကောင်အထည်ဖော်မှုနှင့် ခွဲခြားနိုင်မည်မဟုတ်ပေ။

pair များကို procedure ဖြင့် ကိုယ်စားပြုပြရခြင်း၏ ရည်ရွယ်ချက်မှာ ကျွန်ုပ်တို့၏ language က ဤသို့ အလုပ်လုပ်နေသောကြောင့် မဟုတ်ဘဲ (Scheme နှင့် ယေဘုယျအားဖြင့် Lisp system များသည် ပိုမိုမြန်ဆန်စေရန်အတွက် pair များကို တိုက်ရိုက် အကောင်အထည်ဖော်ကြသည်)၊ ဤသို့ အလုပ်လုပ်နိုင်ကြောင်း ပြသရန်ဖြစ်သည်။ Procedure ဖြင့် ကိုယ်စားပြုခြင်းသည် နားလည်ရန် ခက်ခဲနိုင်သော်လည်း၊ pair များ ဖြည့်ဆည်းရန်လိုအပ်သည့် တစ်ခုတည်းသော အခြေအနေများကို ဖြည့်ဆည်းပေးနိုင်သောကြောင့် pair များကို ကိုယ်စားပြုရန်အတွက် လုံးဝလုံလောက်သော နည်းလမ်းတစ်ခုဖြစ်သည်။ ဤဥပမာသည် procedure များကို object များအဖြစ် ကိုင်တွယ်နိုင်စွမ်းရှိခြင်းက compound data များကို ကိုယ်စားပြုနိုင်စွမ်းကို အလိုအလျောက် ပံ့ပိုးပေးကြောင်းကိုလည်း သက်သေပြနေသည်။ ဤအရာသည် ယခုအချိန်တွင် ထူးဆန်းနေပုံရနိုင်သော်လည်း၊ ဒေတာကို procedure ဖြင့် ကိုယ်စားပြုခြင်းများသည် ကျွန်ုပ်တို့၏ programming ပညာရပ်တွင် အဓိကကျသော အခန်းကဏ္ဍမှ ပါဝင်လာမည်ဖြစ်သည်။ ဤ programming ပုံစံကို message passing ဟု ခေါ်လေ့ရှိပြီး၊ ကျွန်ုပ်တို့သည် modeling နှင့် simulation ဆိုင်ရာ ကိစ္စရပ်များကို ကိုင်တွယ်မည့် Chapter 3 တွင် ၎င်းကို အခြေခံကိရိယာတစ်ခုအဖြစ် အသုံးပြုမည်ဖြစ်သည်။

Hoare (1972) မှ စတင်လမ်းဖောက်ခဲ့သည့် နည်းလမ်းတစ်ခုကို abstract models နည်းလမ်းဟု လူသိများသည်။ ၎င်းသည် အထက်ပါ rational-number ဥပမာတွင် ဖော်ပြထားသည့် “လုပ်ထုံးလုပ်နည်းများ နှင့် အခြေအနေများ” (procedures plus conditions) သတ်မှတ်ချက်ကို ပုံစံချပေးသည်။ rational-number ဖော်ပြမှုအပေါ်ရှိ အခြေအနေကို ကိန်းပြည့်များ (integers) ဆိုင်ရာ အချက်အလက်များ (ညီမျှခြင်းနှင့် စားခြင်း) ဖြင့် ဖော်ပြခဲ့သည်ကို သတိပြုပါ။ ယေဘုယျအားဖြင့်၊ abstract models များသည် အသစ်သော ဒေတာအရာဝတ္ထု (data objects) အမျိုးအစားများကို ယခင်က သတ်မှတ်ပြီးသား ဒေတာအရာဝတ္ထု အမျိုးအစားများဖြင့် သတ်မှတ်ကြသည်။ ထို့ကြောင့် ဒေတာအရာဝတ္ထုများဆိုင်ရာ အဆိုပြုချက်များကို (assertions) ယခင်က သတ်မှတ်ထားသော ဒေတာအရာဝတ္ထုများဆိုင်ရာ အဆိုပြုချက်များသို့ လျှော့ချခြင်းဖြင့် စစ်ဆေးနိုင်သည်။ MIT မှ Zilles၊ IBM မှ Goguen, Thatcher, Wagner, နှင့် Wright ( Thatcher et al. 1978 ကိုကြည့်ပါ) နှင့် Toronto မှ Guttag (Guttag 1977 ကိုကြည့်ပါ) တို့မှ တင်ပြခဲ့သည့် အခြားသော ချဉ်းကပ်မှုတစ်ခုကို algebraic specification ဟု ခေါ်ဆိုသည်။ ၎င်းသည် “လုပ်ထုံးလုပ်နည်းများ” ကို abstract algebraic system တစ်ခု၏ အစိတ်အပိုင်းများအဖြစ် မှတ်ယူပြီး၊ ၎င်းတို့၏ လုပ်ဆောင်ချက်များကို ကျွန်ုပ်တို့၏ “အခြေအနေများ” နှင့် ကိုက်ညီသော axioms (အမှန်တရားများ) ဖြင့် သတ်မှတ်သည်။ ထို့နောက် ဒေတာအရာဝတ္ထုများဆိုင်ရာ အဆိုပြုချက်များကို စစ်ဆေးရန် abstract algebra နည်းစနစ်များကို အသုံးပြုသည်။ ဤနည်းလမ်းနှစ်ခုလုံးကို Liskov and Zilles (1975) ၏ စာတမ်းတွင် လေ့လာတင်ပြထားပါသည်။
