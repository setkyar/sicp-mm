3.2The Environment Model of Evaluation

အခန်း ၁ မှာ compound procedure တွေကို စတင်မိတ်ဆက်တုန်းက၊ procedure တစ်ခုကို argument တွေနဲ့ ဘယ်လို apply လုပ်သလဲဆိုတာကို အဓိပ္ပာယ်ဖွင့်ဆိုဖို့ substitution model of evaluation (1.1.5) ကို အသုံးပြုခဲ့ပါတယ်။

  • compound procedure တစ်ခုကို argument တွေနဲ့ apply လုပ်ဖို့ဆိုတာ၊ procedure ရဲ့ formal parameter တစ်ခုချင်းစီကို သက်ဆိုင်ရာ argument တွေနဲ့ အစားထိုးပြီး procedure ရဲ့ body ကို evaluate လုပ်ခြင်း ဖြစ်ပါတယ်။

ကျွန်ုပ်တို့ရဲ့ programming language ထဲကို assignment ကို ထည့်သွင်းလိုက်တာနဲ့ အဲဒီလို အဓိပ္ပာယ်ဖွင့်ဆိုချက်ဟာ လုံလောက်မှု မရှိတော့ပါဘူး။ အထူးသဖြင့်၊ assignment ရှိနေတဲ့အခါ variable တစ်ခုဟာ တန်ဖိုးတစ်ခုအတွက် အမည်တစ်ခုသက်သက် မဟုတ်တော့ဘူးလို့ 3.1.3 မှာ အငြင်းပွားခဲ့ပါတယ်။ အဲဒီအစား၊ variable တစ်ခုဟာ တန်ဖိုးတွေကို သိမ်းဆည်းထားနိုင်တဲ့ "နေရာ" (place) တစ်ခုကို တစ်နည်းနည်းနဲ့ ညွှန်းဆိုရပါမယ်။ ကျွန်ုပ်တို့ရဲ့ evaluation model အသစ်မှာတော့ ဒီနေရာတွေကို environments လို့ခေါ်တဲ့ structure တွေထဲမှာ ထိန်းသိမ်းထားမှာ ဖြစ်ပါတယ်။

environment ဆိုတာ frames တွေရဲ့ အစီအစဉ် (sequence) တစ်ခု ဖြစ်ပါတယ်။ frame တစ်ခုစီဟာ variable နာမည်တွေကို ၎င်းတို့နဲ့ သက်ဆိုင်တဲ့ တန်ဖိုးတွေနဲ့ ချိတ်ဆက်ပေးထားတဲ့ bindings တွေရဲ့ ဇယား (ဗလာ ဖြစ်နေနိုင်ပါတယ်) တစ်ခု ဖြစ်ပါတယ်။ (frame တစ်ခုထဲမှာ variable တစ်ခုအတွက် အများဆုံး binding တစ်ခုသာ ရှိနိုင်ပါတယ်။) frame တစ်ခုစီမှာ ၎င်းရဲ့ enclosing environment ဆီကို ညွှန်ပြတဲ့ pointer တစ်ခုလည်း ရှိပါတယ်။ ဆွေးနွေးမှု ရည်ရွယ်ချက်အရ အဲဒီ frame ဟာ global လို့ သတ်မှတ်မထားဘူးဆိုရင်ပေါ့။ environment တစ်ခုနဲ့ သက်ဆိုင်တဲ့ variable တစ်ခုရဲ့ တန်ဖိုး ဆိုတာ အဲဒီ environment ရဲ့ ပထမဆုံး frame ကစပြီး ရှာဖွေရာမှာ အဲဒီ variable အတွက် binding တွေ့တဲ့ ပထမဆုံး frame က ပေးထားတဲ့ တန်ဖိုး ဖြစ်ပါတယ်။ အကယ်၍ sequence ထဲက ဘယ် frame မှာမှ အဲဒီ variable အတွက် binding မသတ်မှတ်ထားဘူးဆိုရင်၊ အဲဒီ variable ဟာ အဲဒီ environment မှာ unbound ဖြစ်နေတယ်လို့ ဆိုပါတယ်။

ပုံ ၃.၁ မှာ I, II, နဲ့ III လို့ အမည်တပ်ထားတဲ့ frame သုံးခုနဲ့ ဖွဲ့စည်းထားတဲ့ ရိုးရှင်းတဲ့ environment structure တစ်ခုကို ပြသထားပါတယ်။ ပုံမှာ A, B, C, နဲ့ D တို့ဟာ environment တွေကို ညွှန်ပြတဲ့ pointer တွေ ဖြစ်ပါတယ်။ C နဲ့ D ဟာ environment တစ်ခုတည်းကိုပဲ ညွှန်ပြနေတာပါ။ variable z နဲ့ x တို့ကို frame II မှာ bind လုပ်ထားပြီး၊ y နဲ့ x တို့ကိုတော့ frame I မှာ bind လုပ်ထားပါတယ်။ environment D မှာ x ရဲ့ တန်ဖိုးဟာ ၃ ဖြစ်ပါတယ်။ environment B နဲ့ သက်ဆိုင်တဲ့ x ရဲ့ တန်ဖိုးဟာလည်း ၃ ပဲ ဖြစ်ပါတယ်။ ဒါကို အခုလို ဆုံးဖြတ်ပါတယ်- ကျွန်ုပ်တို့ဟာ sequence ထဲက ပထမဆုံး frame (frame III) ကို စစ်ဆေးပြီး x အတွက် binding မတွေ့တဲ့အတွက်၊ အပြင်ဘက်က enclosing environment D ကို ဆက်သွားပြီး frame I မှာ binding ကို တွေ့ရှိပါတယ်။ အခြားတစ်ဖက်မှာတော့ environment A မှာ x ရဲ့ တန်ဖိုးဟာ ၇ ဖြစ်ပါတယ်၊ ဘာကြောင့်လဲဆိုတော့ sequence ရဲ့ ပထမဆုံး frame (frame II) မှာ x ကို ၇ နဲ့ bind လုပ်ထားတာ တွေ့ရလို့ပါ။ environment A နဲ့ ပတ်သက်ရင်၊ frame II မှာ x ကို ၇ နဲ့ bind လုပ်ထားတာဟာ frame I မှာ x ကို ၃ နဲ့ bind လုပ်ထားတာကို shadow လုပ်ထားတယ် (လွှမ်းမိုးထားတယ်) လို့ ဆိုပါတယ်။

environment ဟာ evaluation process အတွက် အလွန်အရေးကြီးပါတယ်၊ ဘာကြောင့်လဲဆိုတော့ သူဟာ expression တစ်ခုကို ဘယ် context (နောက်ခံအခြေအနေ) မှာ evaluate လုပ်ရမလဲဆိုတာကို ဆုံးဖြတ်ပေးလို့ ဖြစ်ပါတယ်။ တကယ်တော့ programming language တစ်ခုက expression တွေဟာ သူတို့ဘာသာသူတို့ ဘယ်လိုအဓိပ္ပာယ်မှ မရှိဘူးလို့တောင် ပြောနိုင်ပါတယ်။ အဲဒီအစား၊ expression တစ်ခုဟာ ၎င်းကို evaluate လုပ်မယ့် environment တစ်ခုခုနဲ့ သက်ဆိုင်မှသာ အဓိပ္ပာယ်ရှိလာတာ ဖြစ်ပါတယ်။ (+ 1 1) လို ရိုးရှင်းတဲ့ expression တစ်ခုရဲ့ interpretation တောင်မှ + ဆိုတာ ပေါင်းခြင်းအတွက် သင်္ကေတဖြစ်တယ်ဆိုတဲ့ context တစ်ခုမှာ လုပ်ဆောင်နေတယ်ဆိုတဲ့ နားလည်မှုပေါ်မှာ မူတည်နေပါတယ်။ ဒါကြောင့် ကျွန်ုပ်တို့ရဲ့ evaluation model မှာ expression တစ်ခုကို environment တစ်ခုခုနဲ့ သက်ဆိုင်ပြီး evaluate လုပ်ခြင်းအကြောင်း အမြဲပြောမှာ ဖြစ်ပါတယ်။ interpreter နဲ့ ဆက်သွယ်ဆောင်ရွက်မှုတွေကို ဖော်ပြဖို့အတွက်၊ primitive procedure တွေနဲ့ သက်ဆိုင်တဲ့ သင်္ကေတတွေအတွက် တန်ဖိုးတွေပါဝင်တဲ့ (အပြင်ဘက် environment မရှိတဲ့) frame တစ်ခုတည်းပါတဲ့ global environment တစ်ခုရှိတယ်လို့ ယူဆပါမယ်။ ဥပမာအားဖြင့်၊ + ဟာ ပေါင်းခြင်းအတွက် သင်္ကေတဖြစ်တယ်ဆိုတဲ့ အယူအဆကို သင်္ကေတ + ဟာ global environment မှာ primitive addition procedure နဲ့ bind လုပ်ထားတယ်လို့ ပြောခြင်းဖြင့် ဖော်ပြပါတယ်။

3.2.1The Rules for Evaluation

interpreter က combination တစ်ခုကို ဘယ်လို evaluate လုပ်သလဲဆိုတဲ့ ယေဘုယျ သတ်မှတ်ချက်ဟာ 1.1.3 မှာ ပထမဆုံး မိတ်ဆက်ခဲ့တုန်းကအတိုင်းပဲ ရှိနေပါသေးတယ်-

  • combination တစ်ခုကို evaluate လုပ်ရန်-
  1. combination ရဲ့ subexpression တွေကို evaluate လုပ်ပါ။140
  2. operator subexpression ရဲ့ တန်ဖိုးကို operand subexpression တွေရဲ့ တန်ဖိုးတွေပေါ်မှာ apply လုပ်ပါ။

environment model of evaluation ဟာ compound procedure တစ်ခုကို argument တွေနဲ့ apply လုပ်တာ ဘာကိုဆိုလိုသလဲဆိုတဲ့ သတ်မှတ်ချက်မှာ substitution model ရဲ့ နေရာမှာ အစားထိုးဝင်ရောက်လာပါတယ်။

environment model of evaluation မှာ procedure တစ်ခုဆိုတာ code အချို့နဲ့ environment တစ်ခုကို ညွှန်ပြတဲ့ pointer တစ်ခုတို့ ပါဝင်တဲ့ pair တစ်ခု အမြဲဖြစ်ပါတယ်။ procedure တွေကို နည်းလမ်းတစ်ခုတည်းနဲ့ပဲ ဖန်တီးနိုင်ပါတယ်- အဲဒါကတော့ λ-expression တစ်ခုကို evaluate လုပ်ခြင်းဖြင့် ဖြစ်ပါတယ်။ ဒါဟာ λ-expression ရဲ့ စာသားကနေ ရရှိတဲ့ code ပါဝင်ပြီး၊ အဲဒီ procedure ကို ဖန်တီးဖို့ evaluate လုပ်ခဲ့တဲ့ environment ကို environment အဖြစ် သတ်မှတ်ထားတဲ့ procedure တစ်ခုကို ထွက်ပေါ်စေပါတယ်။ ဥပမာအားဖြင့်၊ 

(define (square x)
  (* x x))

ဆိုတဲ့ procedure definition ကို global environment မှာ evaluate လုပ်တာကို စဉ်းစားကြည့်ပါ။ procedure definition syntax ဟာ အောက်ခြေမှာရှိနေတဲ့ implicit λ-expression အတွက် syntactic sugar သက်သက်သာ ဖြစ်ပါတယ်။ အောက်ပါအတိုင်း အသုံးပြုတာနဲ့ အတူတူပဲ ဖြစ်မှာပါ- 

(define square
  (lambda (x) (* x x)))

ဒါဟာ (lambda (x) (* x x)) ကို evaluate လုပ်ပြီး၊ ရရှိလာတဲ့ တန်ဖိုးကို square နဲ့ bind လုပ်လိုက်တာ ဖြစ်ပါတယ်၊ ဒါတွေအားလုံးဟာ global environment ထဲမှာ ဖြစ်ပျက်တာပါ။

ပုံ ၃.၂ မှာ ဒီ define expression ကို evaluate လုပ်လိုက်တဲ့ ရလဒ်ကို ပြသထားပါတယ်။ procedure object ဆိုတာ code ကနေ ၎င်း procedure မှာ formal parameter တစ်ခု (x) ရှိပြီး procedure body ကတော့ (* x x) ဖြစ်တယ်လို့ သတ်မှတ်ထားတဲ့ pair တစ်ခု ဖြစ်ပါတယ်။ procedure ရဲ့ environment အပိုင်းကတော့ global environment ကို ညွှန်ပြတဲ့ pointer တစ်ခု ဖြစ်ပါတယ်၊ ဘာကြောင့်လဲဆိုတော့ procedure ကို ဖန်တီးဖို့ λ-expression ကို evaluate လုပ်ခဲ့တဲ့ environment ဟာ global environment ဖြစ်လို့ပါ။ procedure object ကို သင်္ကေတ square နဲ့ ချိတ်ဆက်ပေးတဲ့ binding အသစ်တစ်ခုကို global frame ထဲမှာ ထည့်သွင်းလိုက်ပါတယ်။ ယေဘုယျအားဖြင့်၊ define ဟာ frame တွေထဲမှာ binding တွေ ထပ်ထည့်ခြင်းအားဖြင့် definition တွေကို ဖန်တီးပေးပါတယ်။

အခု procedure တွေ ဘယ်လိုဖန်တီးသလဲဆိုတာကို မြင်တွေ့ခဲ့ပြီးပြီဆိုတော့၊ procedure တွေကို ဘယ်လို apply လုပ်သလဲဆိုတာကို ဖော်ပြနိုင်ပါပြီ။ environment model က အခုလို သတ်မှတ်ပါတယ်- procedure တစ်ခုကို argument တွေနဲ့ apply လုပ်ဖို့၊ parameter တွေကို argument တွေရဲ့ တန်ဖိုးတွေနဲ့ bind လုပ်ထားတဲ့ frame တစ်ခုပါဝင်တဲ့ environment အသစ်တစ်ခုကို ဖန်တီးပါ။ အဲဒီ frame ရဲ့ enclosing environment ဟာ procedure မှာ သတ်မှတ်ထားတဲ့ environment ဖြစ်ပါတယ်။ အခု၊ အဲဒီ environment အသစ်အတွင်းမှာ procedure body ကို evaluate လုပ်ပါ။

ဒီ rule ကို ဘယ်လိုလိုက်နာသလဲဆိုတာ ပြသဖို့၊ ပုံ ၃.၃ မှာ global environment ထဲမှာ (square 5) expression ကို evaluate လုပ်လို့ ထွက်ပေါ်လာတဲ့ environment structure ကို ဖော်ပြထားပါတယ်။ ဒီမှာ square ဆိုတာ ပုံ ၃.၂ မှာ ဖန်တီးခဲ့တဲ့ procedure ဖြစ်ပါတယ်။ procedure ကို apply လုပ်လိုက်တဲ့အခါ ပုံထဲမှာ E1 လို့ အမည်တပ်ထားတဲ့ environment အသစ်တစ်ခု ဖန်တီးမှုကို ဖြစ်ပေါ်စေပါတယ်၊ အဲဒီ environment ဟာ procedure ရဲ့ formal parameter ဖြစ်တဲ့ x ကို argument 5 နဲ့ bind လုပ်ထားတဲ့ frame နဲ့ စတင်ပါတယ်။ ဒီ frame ကနေ အပေါ်ကို တက်သွားတဲ့ pointer က ဒီ frame ရဲ့ enclosing environment ဟာ global environment ဖြစ်ကြောင်း ပြသနေပါတယ်။ ဒီနေရာမှာ global environment ကို ရွေးချယ်ရတာဟာ square procedure object ရဲ့ အစိတ်အပိုင်းအဖြစ် ညွှန်ပြထားတဲ့ environment ဖြစ်လို့ပါ။ E1 အတွင်းမှာ၊ procedure ရဲ့ body ဖြစ်တဲ့ (* x x) ကို ကျွန်ုပ်တို့ evaluate လုပ်ပါတယ်။ E1 မှာ x ရဲ့ တန်ဖိုးဟာ 5 ဖြစ်တဲ့အတွက်၊ ရလဒ်ဟာ (* 5 5) သို့မဟုတ် 25 ဖြစ်ပါတယ်။

procedure application ရဲ့ environment model ကို စည်းမျဉ်း နှစ်ခုနဲ့ အကျဉ်းချုပ်နိုင်ပါတယ်-

  • procedure object တစ်ခုကို argument အစုအဝေးတစ်ခုပေါ်မှာ apply လုပ်ရာမှာ၊ frame တစ်ခုကို တည်ဆောက်ပြီး procedure ရဲ့ formal parameter တွေကို call မှာပါတဲ့ argument တွေနဲ့ bind လုပ်ရပါတယ်၊ ပြီးရင်တော့ အဲဒီ တည်ဆောက်လိုက်တဲ့ environment အသစ်ရဲ့ context မှာ procedure ရဲ့ body ကို evaluate လုပ်ရပါတယ်။ environment အသစ်မှာ ပါဝင်တဲ့ frame ရဲ့ enclosing environment ဟာ apply လုပ်ခံရတဲ့ procedure object ထဲမှာ ပါဝင်တဲ့ environment ဖြစ်ပါတယ်။
  • procedure တစ်ခုကို ပေးထားတဲ့ environment တစ်ခုနဲ့ သက်ဆိုင်တဲ့ λ-expression ကို evaluate လုပ်ခြင်းဖြင့် ဖန်တီးပါတယ်။ ရရှိလာတဲ့ procedure object ဆိုတာ λ-expression ရဲ့ စာသားနဲ့ အဲဒီ procedure ကို ဖန်တီးခဲ့တဲ့ environment ကို ညွှန်ပြတဲ့ pointer တို့ ပါဝင်တဲ့ pair တစ်ခု ဖြစ်ပါတယ်။

ကျွန်ုပ်တို့ဟာ define ကို အသုံးပြုပြီး သင်္ကေတတစ်ခုကို သတ်မှတ်ခြင်းဟာ လက်ရှိ environment frame မှာ binding တစ်ခုကို ဖန်တီးပြီး၊ အဲဒီသင်္ကေတကို သတ်မှတ်ထားတဲ့ တန်ဖိုး ပေးအပ်ခြင်း ဖြစ်တယ်လို့လည်း သတ်မှတ်ပါတယ်။141 နောက်ဆုံးအနေနဲ့၊ ကျွန်ုပ်တို့ကို environment model အသုံးပြုဖို့ တွန်းအားပေးခဲ့တဲ့ set! ရဲ့ လုပ်ဆောင်ပုံကို သတ်မှတ်ပါမယ်။ environment တစ်ခုခုမှာ (set! ⟨variable⟩ ⟨value⟩) expression ကို evaluate လုပ်ခြင်းဟာ အဲဒီ environment ထဲမှာ variable ရဲ့ binding ကို ရှာဖွေပြီး၊ အဲဒီ binding ကို တန်ဖိုးအသစ်ဖြစ်အောင် ပြောင်းလဲလိုက်ခြင်း ဖြစ်ပါတယ်။ ဆိုလိုတာကတော့၊ environment ထဲမှာ variable အတွက် binding ပါဝင်တဲ့ ပထမဆုံး frame ကို ရှာပြီး အဲဒီ frame ကို ပြင်ဆင်လိုက်တာ ဖြစ်ပါတယ်။ အကယ်၍ variable ဟာ environment မှာ unbound ဖြစ်နေရင်၊ set! ဟာ error တစ်ခုကို ထုတ်ပေးပါလိမ့်မယ်။

ဒီ evaluation rule တွေဟာ substitution model ထက် ပိုမိုရှုပ်ထွေးပေမယ့်၊ အသင့်အတင့်တော့ ရိုးရှင်းနေပါသေးတယ်။ ဒါ့အပြင်၊ evaluation model ဟာ abstract ဖြစ်ပေမယ့် interpreter က expression တွေကို ဘယ်လို evaluate လုပ်သလဲဆိုတာကို မှန်ကန်တဲ့ ဖော်ပြချက်တစ်ခု ပေးစွမ်းပါတယ်။ အခန်း ၄ မှာတော့ ဒီ model ဟာ လက်တွေ့အလုပ်လုပ်တဲ့ interpreter တစ်ခုကို implement လုပ်ဖို့အတွက် ဘယ်လို blueprint အဖြစ် အသုံးဝင်မလဲဆိုတာကို မြင်တွေ့ရပါလိမ့်မယ်။ နောက်လာမယ့် အပိုင်းတွေမှာတော့ သရုပ်ပြ ပရိုဂရမ်အချို့ကို analysis လုပ်ခြင်းဖြင့် ဒီ model ရဲ့ အသေးစိတ်တွေကို ရှင်းလင်းတင်ပြသွားမှာ ဖြစ်ပါတယ်။

ကျွန်ုပ်တို့ 1.1.5 တွင် substitution model ကို မိတ်ဆက်ခဲ့စဉ်က၊ အောက်ပါ procedure definition များအရ (f 5) ဆိုသော combination သည် မည်ကဲ့သို့ 136 အဖြစ် evaluate ဖြစ်သွားသည်ကို ပြသခဲ့ပြီး ဖြစ်သည်- 

(define (square x)
  (* x x))
(define (sum-of-squares x y)
  (+ (square x) (square y)))
(define (f a)
  (sum-of-squares (+ a 1) (* a 2)))

ယခု ဤဥပမာကိုပင် environment model ကို အသုံးပြု၍ analyze လုပ်ကြည့်ပါမည်။ Figure 3.4 တွင် global environment ထဲ၌ f၊ square နှင့် sum-of-squares တို့၏ definition များကို evaluate လုပ်ခြင်းဖြင့် ဖန်တီးလိုက်သော procedure object သုံးခုကို ပြထားပါသည်။ procedure object တစ်ခုစီတွင် code အချို့နှင့်အတူ global environment သို့ ညွှန်းထားသော pointer တစ်ခု ပါဝင်ပါသည်။

Figure 3.5 တွင် (f 5) expression ကို evaluate လုပ်ခြင်းဖြင့် ဖန်တီးလိုက်သော environment structure ကို တွေ့နိုင်ပါသည်။ f ကို ခေါ်ယူလိုက်ခြင်းသည် environment အသစ် E1 ကို ဖန်တီးပေးပြီး၊ ၎င်းသည် f ၏ formal parameter ဖြစ်သော a ကို argument 5 နှင့် ချည်နှောင်ထားသည့် frame တစ်ခုနှင့် စတင်ပါသည်။ E1 ထဲတွင် f ၏ body ကို ကျွန်ုပ်တို့ evaluate လုပ်ပါသည်- 

(sum-of-squares (+ a 1) (* a 2))

ဤ combination ကို evaluate လုပ်ရန် ပထမဦးစွာ subexpression များကို evaluate လုပ်ပါသည်။ ပထမ subexpression ဖြစ်သော sum-of-squares တွင် တန်ဖိုးတစ်ခု ရှိပြီး ၎င်းသည် procedure object တစ်ခု ဖြစ်ပါသည်။ (ဤတန်ဖိုးကို မည်သို့ရှာဖွေသည်ကို သတိပြုပါ- ပထမဦးစွာ E1 ၏ ပထမ frame ထဲတွင် ရှာပါသည်၊ ထိုနေရာတွင် sum-of-squares အတွက် binding မရှိပါ။ ထို့နောက် ၎င်းကို ဝန်းရံထားသော environment ဖြစ်သည့် global environment သို့ ဆက်လက်ရှာဖွေရာ Figure 3.4 တွင် ပြထားသော binding ကို တွေ့ရှိရပါသည်။) အခြား subexpression နှစ်ခုကိုမူ primitive operation များဖြစ်သော + နှင့် * တို့ကို အသုံးပြု၍ (+ a 1) နှင့် (* a 2) combination တို့ကို evaluate လုပ်ပြီး 6 နှင့် 10 တို့ကို အသီးသီး ရရှိပါသည်။

ယခု sum-of-squares procedure object ကို argument 6 နှင့် 10 တို့ဖြင့် apply လုပ်ပါမည်။ ၎င်းသည် environment အသစ် E2 ကို ရရှိစေပြီး ထိုနေရာတွင် formal parameter များဖြစ်သော x နှင့် y တို့ကို argument များဖြင့် ချည်နှောင်ထားပါသည်။ E2 အတွင်းတွင် combination (+ (square x) (square y)) ကို evaluate လုပ်ပါသည်။ ၎င်းသည် ကျွန်ုပ်တို့အား (square x) ကို evaluate လုပ်ရန် လမ်းညွှန်ပါသည်၊ ထိုနေရာတွင် square ကို global frame တွင် တွေ့ရပြီး x သည် 6 ဖြစ်ပါသည်။ ထပ်မံ၍ environment အသစ် E3 တစ်ခုကို တည်ဆောက်ပြီး x ကို 6 နှင့် ချည်နှောင်ကာ ၎င်းအတွင်း၌ square ၏ body ဖြစ်သော (* x x) ကို evaluate လုပ်ပါသည်။ ထို့အပြင် sum-of-squares ကို apply လုပ်ခြင်း၏ အစိတ်အပိုင်းတစ်ခုအနေဖြင့် subexpression (square y) ကိုလည်း evaluate လုပ်ရပါမည်၊ ဤနေရာတွင် y သည် 10 ဖြစ်ပါသည်။ ဤဒုတိယအကြိမ် square ကို ခေါ်ယူခြင်းသည် အခြား environment E4 တစ်ခုကို ဖန်တီးပြီး ၎င်းထဲတွင် square ၏ formal parameter ဖြစ်သော x ကို 10 နှင့် ချည်နှောင်ပါသည်။ E4 အတွင်းတွင်လည်း (* x x) ကို evaluate လုပ်ရမည် ဖြစ်ပါသည်။

သတိပြုရန် အရေးကြီးသော အချက်မှာ square ကို ခေါ်ယူလိုက်တိုင်း x အတွက် binding တစ်ခုပါဝင်သော environment အသစ်တစ်ခုကို ဖန်တီးခြင်း ဖြစ်ပါသည်။ x ဟု အမည်ပေးထားသော မတူညီသည့် local variable အားလုံးကို frame အသီးသီးက မည်ကဲ့သို့ သီးခြားစီ ထိန်းသိမ်းထားသည်ကို ဤနေရာတွင် မြင်နိုင်ပါသည်။ square ကြောင့် ဖန်တီးလိုက်သော frame တစ်ခုစီသည် global environment ကို ညွှန်းနေသည်ကို သတိပြုပါ၊ အဘယ်ကြောင့်ဆိုသော် ၎င်းသည် square procedure object က ညွှန်ပြထားသော environment ဖြစ်သောကြောင့် ဖြစ်ပါသည်။

subexpression များကို evaluate လုပ်ပြီးနောက် ရလဒ်များကို ပြန်ပေး (return) ပါသည်။ square ကို နှစ်ကြိမ်ခေါ်ယူခြင်းမှ ထွက်ပေါ်လာသော တန်ဖိုးများကို sum-of-squares က ပေါင်းစပ်ပေးပြီး ထိုရလဒ်ကို f က return လုပ်ပါသည်။ ကျွန်ုပ်တို့သည် ဤနေရာတွင် environment structure များကိုသာ အဓိကထားသောကြောင့် ဤ return လုပ်လိုက်သော တန်ဖိုးများသည် call တစ်ခုမှ တစ်ခုသို့ မည်သို့ စီးဆင်းသွားသည်ကို အသေးစိတ် မဆွေးနွေးတော့ပါ။ သို့သော် ၎င်းသည်လည်း evaluation process ၏ အရေးကြီးသော ကဏ္ဍတစ်ခု ဖြစ်ပြီး Chapter 5 တွင် အသေးစိတ် ပြန်လည် ဆွေးနွေးပါမည်။

Exercise 3.9: 1.2.1 တွင် factorial များ တွက်ချက်ရန် procedure နှစ်ခုကို analyze လုပ်ဖို့ substitution model ကို အသုံးပြုခဲ့ပါသည်။ recursive version မှာ- 

(define (factorial n)
  (if (= n 1)
      1
      (* n (factorial (- n 1)))))

နှင့် iterative version မှာ- 

(define (factorial n)
  (fact-iter 1 1 n))

(define (fact-iter product 
                   counter 
                   max-count)
  (if (> counter max-count)
      product
      (fact-iter (* counter product)
                 (+ counter 1)
                 max-count)))

လုပ်ဆောင်ချက် (procedures) များနှင့် တန်ဖိုးသတ်မှတ်ခြင်း (assignment) တို့ကို အသုံးပြု၍ local state ရှိသော objects များကို မည်သို့ကိုယ်စားပြုနိုင်သည်ကို သိရှိရန် environment model ကို လေ့လာနိုင်သည်။ ဥပမာအနေဖြင့် 3.1.1 ရှိ အောက်ပါ procedure ကို ခေါ်ယူခြင်းဖြင့် တည်ဆောက်ထားသော “withdrawal processor” ကို ကြည့်ပါ။

(define (make-withdraw initial-amount)

          "Insufficient funds"))))

(let ((⟨var⟩ ⟨exp⟩)) ⟨body⟩)

((lambda (⟨var⟩) ⟨body⟩) ⟨exp⟩)

3.2.4Internal Definitions

အပိုင်း 1.1.8 တွင် procedures များ၌ internal definitions များ ရှိနိုင်သည်ဟူသော အယူအဆကို မိတ်ဆက်ခဲ့ပြီး၊ ၎င်းသည် အောက်ပါ square roots တွက်ချက်သည့် procedure တွင်ကဲ့သို့ block structure တစ်ခုကို ဖြစ်ပေါ်စေသည်- 

(define (sqrt x)
  (define (good-enough? guess)
    (< (abs (- (square guess) x)) 0.001))
  (define (improve guess)
    (average guess (/ x guess)))
  (define (sqrt-iter guess)
    (if (good-enough? guess)
        guess
        (sqrt-iter (improve guess))))
  (sqrt-iter 1.0))

ယခုအခါ ဤ internal definitions များသည် အဘယ်ကြောင့် မျှော်လင့်ထားသည့်အတိုင်း အလုပ်လုပ်သည်ကို သိမြင်နိုင်ရန် environment model ကို အသုံးပြုနိုင်ပြီဖြစ်သည်။ Figure 3.11 တွင် (sqrt 2) expression ကို evaluation ပြုလုပ်နေစဉ် good-enough? ဆိုသည့် internal procedure ကို guess တန်ဖိုး 1 ဖြင့် ပထမဆုံးအကြိမ် ခေါ်ဆိုလိုက်သည့် အချိန်ကို ပြသထားသည်။

Environment ၏ တည်ဆောက်ပုံကို လေ့လာကြည့်ပါ။ Sqrt သည် global environment ထဲရှိ symbol တစ်ခုဖြစ်ပြီး၊ ၎င်းနှင့် ဆက်နွှယ်နေသော environment မှာ global environment ဖြစ်သည့် procedure object တစ်ခုနှင့် bound ဖြစ်နေသည်။ sqrt ကို ခေါ်ဆိုလိုက်သောအခါ၊ global environment ၏ လက်အောက်ခံဖြစ်သော environment အသစ် E1 ကို ဖန်တီးလိုက်ပြီး၊ ထိုပတ်ဝန်းကျင်တွင် parameter x သည် 2 နှင့် bound ဖြစ်သွားသည်။ ထို့နောက် sqrt ၏ body ကို E1 ထဲတွင် evaluation ပြုလုပ်သည်။ sqrt ၏ body ထဲရှိ ပထမဆုံး expression မှာ- 

(define (good-enough? guess)
  (< (abs (- (square guess) x)) 0.001))

ဤ expression ကို evaluation ပြုလုပ်ခြင်းဖြင့် good-enough? procedure ကို environment E1 ထဲတွင် define လုပ်လိုက်သည်။ ပို၍ တိတိကျကျ ပြောရလျှင်၊ good-enough? symbol ကို E1 ၏ ပထမဆုံး frame ထဲသို့ ထည့်သွင်းလိုက်ပြီး၊ ၎င်းနှင့် ဆက်နွှယ်နေသော environment မှာ E1 ဖြစ်သည့် procedure object တစ်ခုနှင့် bound ဖြစ်စေလိုက်သည်။ ထို့အတူ improve နှင့် sqrt-iter တို့ကိုလည်း E1 ထဲတွင် procedures များအဖြစ် define လုပ်ခဲ့သည်။ ပုံကို ကျစ်လျစ်စေရန်အတွက် Figure 3.11 တွင် good-enough? အတွက် procedure object ကိုသာ ပြသထားသည်။

Local procedures များကို define လုပ်ပြီးနောက်၊ (sqrt-iter 1.0) expression ကို environment E1 ထဲ၌ပင် evaluation ပြုလုပ်သည်။ ထို့ကြောင့် E1 ထဲတွင် sqrt-iter နှင့် bound ဖြစ်နေသော procedure object ကို argument 1 ဖြင့် ခေါ်ဆိုလိုက်သည်။ ၎င်းသည် environment E2 ကို ဖန်တီးပေးပြီး၊ ထိုပတ်ဝန်းကျင်တွင် sqrt-iter ၏ parameter ဖြစ်သော guess သည် 1 နှင့် bound ဖြစ်နေသည်။ Sqrt-iter သည် တစ်ဖန် good-enough? ကို ခေါ်ဆိုရာတွင် (E2 မှ) guess ၏ တန်ဖိုးကို good-enough? အတွက် argument အဖြစ် အသုံးပြုသည်။ ၎င်းသည် အခြား environment တစ်ခုဖြစ်သော E3 ကို တည်ဆောက်ပေးပြီး၊ ထိုပတ်ဝန်းကျင်တွင် guess (good-enough? ၏ parameter) သည် 1 နှင့် bound ဖြစ်နေသည်။ sqrt-iter နှင့် good-enough? နှစ်ခုလုံးတွင် guess အမည်ရှိသော parameter တစ်ခုစီ ပါရှိသော်လည်း၊ ၎င်းတို့သည် မတူညီသော frames များတွင် တည်ရှိသည့် သီးခြား local variables နှစ်ခုဖြစ်သည်။ ထို့ပြင် E2 နှင့် E3 နှစ်ခုလုံး၏ enclosing environment မှာ E1 ဖြစ်သည်၊ အကြောင်းမှာ sqrt-iter နှင့် good-enough? procedures နှစ်ခုလုံး၏ environment part မှာ E1 ဖြစ်သောကြောင့်ဖြစ်သည်။ ယင်း၏ ရလဒ်တစ်ခုမှာ good-enough? ၏ body ထဲတွင် ပါရှိသော symbol x သည် E1 ထဲတွင် ရှိနေသော x ၏ binding (မူလ sqrt procedure ကို ခေါ်ဆိုခဲ့စဉ်က x ၏ တန်ဖိုး) ကို ရည်ညွှန်းမည် ဖြစ်သည်။

Environment model သည် ပရိုဂရမ်များကို modularizing ပြုလုပ်ရာတွင် local procedure definitions များကို အသုံးဝင်သော နည်းစနစ်တစ်ခု ဖြစ်စေသည့် အဓိက ဂုဏ်သတ္တိနှစ်ခုကို ရှင်းပြပေးသည်-

  • Local procedures များ၏ အမည်များသည် ၎င်းတို့ကို ဝန်းရံထားသော procedure ၏ အပြင်ဘက်ရှိ အမည်များကို အနှောင့်အယှက် မဖြစ်စေပါ။ အကြောင်းမှာ local procedure အမည်များကို global environment တွင် bound ဖြစ်စေမည့်အစား၊ ၎င်းကို run သည့်အခါ ထို procedure မှ ဖန်တီးလိုက်သော frame ထဲတွင်သာ bound ဖြစ်စေမည် ဖြစ်သောကြောင့်ဖြစ်သည်။
  • Local procedures များသည် ၎င်းတို့ကို ဝန်းရံထားသော procedure ၏ arguments များကို parameter အမည်များအား free variables များအဖြစ် အသုံးပြုခြင်းဖြင့် အလွယ်တကူ ရယူနိုင်သည်။ အကြောင်းမှာ local procedure ၏ body ကို evaluation ပြုလုပ်သည့် environment သည် ဝန်းရံထားသော procedure အတွက် evaluation environment ၏ လက်အောက်ခံ ဖြစ်နေသောကြောင့် ဖြစ်သည်။

Exercise 3.11: အပိုင်း 3.2.3 တွင် local state ပါဝင်သော procedures များ၏ လုပ်ဆောင်ပုံကို environment model က မည်သို့ ဖော်ပြသည်ကို ကျွန်ုပ်တို့ တွေ့မြင်ခဲ့ရသည်။ ယခုအခါ internal definitions များ မည်သို့ အလုပ်လုပ်သည်ကိုလည်း တွေ့မြင်ပြီးဖြစ်သည်။ ပုံမှန် message-passing procedure တစ်ခုတွင် ဤအချက်နှစ်ခုလုံး ပါဝင်လေ့ရှိသည်။ အပိုင်း 3.1.1 မှ ဘဏ်အကောင့် (bank account) procedure ကို လေ့လာကြည့်ပါ- 

(define (make-account balance)
  (define (withdraw amount)
    (if (>= balance amount)
        (begin (set! balance 
                     (- balance 
                        amount))
               balance)
        "Insufficient funds"))
  (define (deposit amount)
    (set! balance (+ balance amount))
    balance)
  (define (dispatch m)
    (cond ((eq? m 'withdraw) withdraw)
          ((eq? m 'deposit) deposit)
          (else (error "Unknown request: 
                        MAKE-ACCOUNT" 
                       m))))
  dispatch)

အောက်ပါ အပြန်အလှန် လုပ်ဆောင်မှုများကြောင့် ဖြစ်ပေါ်လာသော environment structure ကို ဖော်ပြပါ- 

(define acc (make-account 50))

((acc 'deposit) 40)
90

((acc 'withdraw) 60)
30

acc အတွက် local state ကို မည်သည့်နေရာတွင် သိမ်းဆည်းထားသနည်း။ အခြား အကောင့်တစ်ခုကို define လုပ်သည်ဟု ယူဆပါ- 

(define acc2 (make-account 100))

အကောင့်နှစ်ခုလုံးအတွက် local states များကို မည်သို့ သီးခြားစီ ခွဲခြားထားသနည်း။ Environment structure ၏ မည်သည့် အစိတ်အပိုင်းများကို acc နှင့် acc2 တို့အကြား မျှဝေအသုံးပြုထားသနည်း။
