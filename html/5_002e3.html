5.3သိမ်းဆည်းမှု နေရာချထားခြင်းနှင့် Garbage Collection

အပိုင်း 5.4 တွင် Scheme evaluator တစ်ခုကို register machine တစ်ခုအဖြစ် မည်သို့ အကောင်အထည်ဖော်ရမည်ကို ကျွန်ုပ်တို့ ပြသပါမည်။ ဆွေးနွေးမှုကို ရိုးရှင်းစေရန်အတွက် ကျွန်ုပ်တို့၏ register machine များတွင် စာရင်းပုံစံ ဖွဲ့စည်းထားသော မှတ်ဉာဏ် (list-structured memory) တစ်ခု ပါရှိသည်ဟု ယူဆပါမည်။ ယင်းမှတ်ဉာဏ်တွင် စာရင်းပုံစံ ဒေတာများကို ကိုင်တွယ်ဖြေရှင်းရန် အခြေခံ လုပ်ဆောင်ချက်များသည် primitive များ ဖြစ်ကြသည်။ ထိုကဲ့သို့သော မှတ်ဉာဏ်တစ်ခု ရှိသည်ဟု ယူဆခြင်းသည် Scheme interpreter တစ်ခု၏ ထိန်းချုပ်မှု လုပ်ငန်းစဉ်များအပေါ် အာရုံစိုက်နေချိန်တွင် အသုံးဝင်သော abstraction တစ်ခု ဖြစ်သော်လည်း၊ ၎င်းသည် ခေတ်ပေါ် ကွန်ပျူတာများ၏ အမှန်တကယ် ဖြစ်ပျက်နေသော အခြေခံ ဒေတာ လုပ်ဆောင်ချက်များကို ထင်ဟပ်ခြင်း မရှိပေ။ Lisp စနစ်တစ်ခု မည်သို့ အလုပ်လုပ်ပုံကို ပိုမို ပြည့်စုံစွာ နားလည်စေရန်အတွက်၊ list structure များကို သာမန် ကွန်ပျူတာ မှတ်ဉာဏ်များနှင့် ကိုက်ညီသော နည်းလမ်းဖြင့် မည်သို့ ဖော်ပြနိုင်သည်ကို ကျွန်ုပ်တို့ လေ့လာစုံစမ်းရမည် ဖြစ်သည်။

list structure ကို အကောင်အထည်ဖော်ရာတွင် ထည့်သွင်းစဉ်းစားရမည့် အချက် နှစ်ချက် ရှိသည်။ ပထမအချက်မှာ ဖော်ပြခြင်း (representation) ဆိုင်ရာ ကိစ္စရပ်သက်သက် ဖြစ်သည်- သာမန် ကွန်ပျူတာ မှတ်ဉာဏ်များ၏ သိမ်းဆည်းနိုင်စွမ်းနှင့် လိပ်စာသတ်မှတ်နိုင်စွမ်း (addressing capabilities) များကိုသာ အသုံးပြု၍ Lisp pairs များ၏ “box-and-pointer” ဖွဲ့စည်းပုံကို မည်သို့ ဖော်ပြမည်နည်း ဟူသည့် အချက်ဖြစ်သည်။ ဒုတိယအချက်မှာ တွက်ချက်မှုများ လုပ်ဆောင်နေစဉ်အတွင်း မှတ်ဉာဏ်ကို စီမံခန့်ခွဲခြင်းနှင့် သက်ဆိုင်သည်။ Lisp စနစ်တစ်ခု၏ လုပ်ဆောင်ချက်သည် ဒေတာ အရာဝတ္ထု (data objects) အသစ်များကို အဆက်မပြတ် ဖန်တီးနိုင်စွမ်းပေါ်တွင် များစွာ မူတည်နေသည်။ ၎င်းတို့တွင် interpret လုပ်ခြင်း ခံနေရသည့် Lisp procedures များမှ တိုက်ရိုက် ဖန်တီးလိုက်သော အရာဝတ္ထုများသာမက environments နှင့် argument lists များကဲ့သို့ interpreter ကိုယ်တိုင်မှ ဖန်တီးသော တည်ဆောက်ပုံများလည်း ပါဝင်သည်။ ဒေတာ အရာဝတ္ထု အသစ်များကို အဆက်မပြတ် ဖန်တီးနေခြင်းသည် အကန့်အသတ်မရှိသော၊ လျင်မြန်စွာ အသုံးပြုနိုင်သော မှတ်ဉာဏ်ရှိသည့် ကွန်ပျူတာတစ်လုံးတွင် ပြဿနာ မရှိနိုင်သော်လည်း၊ ကွန်ပျူတာ မှတ်ဉာဏ်များသည် အကန့်အသတ်ရှိသော အရွယ်အစားဖြင့်သာ ရရှိနိုင်ပါသည် (စိတ်မကောင်းစရာပင်)။ ထို့ကြောင့် Lisp စနစ်များသည် အကန့်အသတ်မရှိသော မှတ်ဉာဏ်ရှိသည်ဟု ထင်ယောင်ထင်မှား ဖြစ်စေရန်အတွက် အလိုအလျောက် သိမ်းဆည်းမှု နေရာချထားခြင်း (automatic storage allocation) စနစ်ကို ထောက်ပံ့ပေးထားသည်။ ဒေတာ အရာဝတ္ထုတစ်ခုကို အသုံးမလိုတော့သည့် အခါတွင်၊ ၎င်းအတွက် ဖယ်ထားသော မှတ်ဉာဏ်ကို အလိုအလျောက် ပြန်လည် သိမ်းဆည်းပြီး ဒေတာ အရာဝတ္ထု အသစ်များ တည်ဆောက်ရန်အတွက် အသုံးပြုသည်။ ထိုကဲ့သို့သော အလိုအလျောက် သိမ်းဆည်းမှု နေရာချထားပေးခြင်းအတွက် နည်းလမ်း အမျိုးမျိုး ရှိသည်။ ဤအပိုင်းတွင် ကျွန်ုပ်တို့ ဆွေးနွေးမည့် နည်းလမ်းကို garbage collection ဟု ခေါ်ဆိုသည်။

5.3.1ဗက်တာများအဖြစ် မှတ်ဉာဏ် (Memory as Vectors)

ရိုးရိုး ကွန်ပျူတာ မှတ်ဉာဏ် (memory) တစ်ခုကို သတင်းအချက်အလက် တစ်ခုစီ သိမ်းဆည်းထားနိုင်သည့် အခန်းငယ်များစွာ ပါဝင်သော array တစ်ခုအဖြစ် ယူဆနိုင်သည်။ အခန်းငယ် တစ်ခုစီတွင် သီးခြား အမည်တစ်ခုစီ ရှိပြီး ၎င်းကို address သို့မဟုတ် location ဟု ခေါ်သည်။ ပုံမှန် memory စနစ်များသည် အခြေခံ လုပ်ဆောင်ချက် နှစ်ခုကို ပံ့ပိုးပေးသည်- တစ်ခုမှာ သတ်မှတ်ထားသော တည်နေရာတွင် သိမ်းဆည်းထားသည့် ဒေတာကို ဆွဲထုတ်ယူခြင်း (fetch) ဖြစ်ပြီး နောက်တစ်ခုမှာ သတ်မှတ်ထားသော တည်နေရာတွင် ဒေတာ အသစ်တစ်ခုကို ထည့်သွင်း သတ်မှတ်ခြင်း (assign) ဖြစ်သည်။ Memory address များကို အခန်းငယ်များ၏ အစုအဝေးတစ်ခုကို အစဉ်လိုက် အသုံးပြုနိုင်ရန် (sequential access) အတွက် တစ်ဆင့်ချင်း တိုးမြှင့်နိုင်သည်။ ပို၍ ယေဘုယျအားဖြင့် ဆိုရလျှင် အရေးကြီးသော ဒေတာ လုပ်ဆောင်ချက် အများအပြားသည် memory address များကို ဒေတာများသဖွယ် သဘောထားရန် လိုအပ်ပြီး ၎င်းတို့ကို memory တည်နေရာများတွင် သိမ်းဆည်းနိုင်ကာ machine register များတွင် ကိုင်တွယ် အသုံးပြုနိုင်သည်။ List structure ကို ကိုယ်စားပြု ဖော်ပြခြင်းသည် ထိုကဲ့သို့သော address arithmetic ကို အသုံးပြုသည့် နယ်ပယ်တစ်ခု ဖြစ်သည်။

ကွန်ပျူတာ memory ကို ပုံဖော်ရန်အတွက် ကျွန်ုပ်တို့သည် vector ဟုခေါ်သော ဒေတာ တည်ဆောက်ပုံ (data structure) အမျိုးအစားသစ် တစ်ခုကို အသုံးပြုသည်။ သဘောတရားအရ vector ဆိုသည်မှာ ၎င်း၏ အစိတ်အပိုင်း တစ်ခုချင်းစီကို index ကိန်းပြည့်တစ်ခု အသုံးပြု၍ index ပေါ်တွင် မူတည်ခြင်းမရှိသော အချိန်အတွင်း ဝင်ရောက်ကြည့်ရှုနိုင်သည့် ပေါင်းစပ် ဒေတာ အရာဝတ္ထု (compound data object) တစ်ခု ဖြစ်သည်။290 Memory လုပ်ဆောင်ချက်များကို ဖော်ပြရန်အတွက် ကျွန်ုပ်တို့သည် vector များကို ကိုင်တွယ်ရန် အခြေခံ Scheme procedure နှစ်ခုကို အသုံးပြုသည်-

  • (vector-ref ⟨vector⟩ ⟨n⟩) သည် vector ၏ n ခုမြောက် အစိတ်အပိုင်းကို ပြန်ပေးသည်။
  • (vector-set! ⟨vector⟩ ⟨n⟩ ⟨value⟩) သည် vector ၏ n ခုမြောက် အစိတ်အပိုင်းကို သတ်မှတ်ထားသော တန်ဖိုးသို့ ပြောင်းလဲပေးသည်။

ဥပမာအားဖြင့် v သည် vector တစ်ခု ဖြစ်ပါက (vector-ref v 5) သည် vector v ၏ ပဉ္စမမြောက် အစိတ်အပိုင်းကို ရယူပြီး (vector-set! v 5 7) သည် vector v ၏ ပဉ္စမမြောက် အစိတ်အပိုင်း၏ တန်ဖိုးကို 7 သို့ ပြောင်းလဲပေးသည်။291 ကွန်ပျူတာ memory အတွက်မူ ဤကဲ့သို့ ဝင်ရောက်ကြည့်ရှုခြင်းကို memory ထဲရှိ vector ၏ အစ တည်နေရာကို သတ်မှတ်သည့် base address နှင့် vector ၏ သီးခြား အစိတ်အပိုင်း တစ်ခု၏ offset ကို သတ်မှတ်သည့် index တို့ကို လိပ်စာ သင်္ချာ (address arithmetic) အသုံးပြုကာ ပေါင်းစပ်ခြင်းဖြင့် အကောင်အထည်ဖော်နိုင်သည်။

နှစ်ခုအဖြစ် ပိုင်းခြားထားသည်ဟု မြင်ယောင်ကြည့်ပါ။ ကျွန်ုပ်တို့သည် list structure ကို အောက်ပါအတိုင်း ကိုယ်စားပြုပါမည်- pair တစ်ခုသို့ ညွှန်ပြသော ပွိုင်တာ (pointer) ဆိုသည်မှာ အဆိုပါ vector နှစ်ခုလုံး၏ မဟုတ်သော အရာဝတ္ထုများ (ဥပမာ- ကိန်းဂဏန်းများနှင့် သင်္ကေတများ) အတွက် ကိုယ်စားပြုချက် တစ်ခုနှင့် ဒေတာ အမျိုးအစား တစ်ခုကို တစ်ခုနှင့်တစ်ခု ခွဲခြားရန် နည်းလမ်းတစ်ခုလည်း လိုအပ်ပါသည်။ ဤအရာကို လုပ်ဆောင်နိုင်သည့် နည်းလမ်းများစွာ ရှိသော်လည်း ၎င်းတို့အားလုံးသည် typed pointers များကို အသုံးပြုခြင်းသို့သာ ဦးတည်သွားပါသည်၊ ဆိုလိုသည်မှာ “pointer” ဆိုသည့် အယူအဆတွင် ဒေတာ အမျိုးအစားဆိုင်ရာ သတင်းအချက်အလက်များကိုပါ ထည့်သွင်း ချဲ့ထွင်ခြင်း ဖြစ်သည်။292 ဒေတာ အမျိုးအစားသည် pair တစ်ခုသို့ ညွှန်ပြသော ပွိုင်တာ (၎င်းတွင် “pair” ဒေတာ အမျိုးအစားနှင့် memory vector များထဲသို့ ညွှန်းဆိုသည့် index တစ်ခု ပါဝင်သည်) နှင့် အခြားသော ဒေတာ အမျိုးအစားများသို့ ညွှန်ပြသော ပွိုင်တာများ (၎င်းတို့တွင် အခြားသော ဒေတာ အမျိုးအစားနှင့် ထိုအမျိုးအစားကို ကိုယ်စားပြုရန် အသုံးပြုသည့် အရာတစ်ခုခု ပါဝင်သည်) ကို စနစ်က ခွဲခြားနိုင်စေသည်။ ဒေတာ အရာဝတ္ထု နှစ်ခု၏ ပွိုင်တာများသည် ထပ်တူညီနေပါက ၎င်းတို့ကို တူညီသည် (eq?) ဟု သတ်မှတ်သည်။293 Figure 5.14 တွင် list ((1 2) 3 4) ကို ကိုယ်စားပြုရန် ဤနည်းလမ်းကို အသုံးပြုပုံနှင့် box-and-pointer diagram ကို ပြသထားသည်။ ဒေတာ အမျိုးအစား သတင်းအချက်အလက်ကို ဖော်ပြရန် အက္ခရာ ရှေ့ဆက် (letter prefix) များကို အသုံးပြုသည်။ ထို့ကြောင့် index 5 ရှိသော pair သို့ ညွှန်ပြသည့် ပွိုင်တာကို p5 ဟု ဖော်ပြပြီး၊ လစ်ဟာနေသော list ကို pointer e0 ဖြင့် လည်းကောင်း၊ ကိန်းဂဏန်း 4 သို့ ညွှန်ပြသည့် ပွိုင်တာကို n4 ဟု လည်းကောင်း ဖော်ပြသည်။ Box-and-pointer diagram တွင် pair တစ်ခုစီ၏ car နှင့် cdr တို့ကို မည်သည့်နေရာတွင် သိမ်းဆည်းထားသည်ကို ဖော်ပြသည့် vector index ကို pair တစ်ခုစီ၏ ဘယ်ဘက် အောက်ခြေတွင် ဖော်ပြထားသည်။ the-cars နှင့် the-cdrs ရှိ လွတ်နေသော တည်နေရာများတွင် အခြားသော list structure များ၏ အစိတ်အပိုင်းများ ရှိနေနိုင်သည် (ဤနေရာတွင် ထည့်သွင်းစဉ်းစားရန် မလိုပါ)။

n4 ကဲ့သို့သော ကိန်းဂဏန်း တစ်ခုသို့ ညွှန်ပြသည့် ပွိုင်တာတွင် ကိန်းဂဏန်း ဒေတာ ဖြစ်ကြောင်း ဖော်ပြသည့် အမျိုးအစားနှင့်အတူ ကိန်းဂဏန်း 4 ၏ အမှန်တကယ် ကိုယ်စားပြုချက်တို့ ပါဝင်နိုင်သည်။294 ပွိုင်တာ တစ်ခုအတွက် သတ်မှတ်ထားသော ကန့်သတ် နေရာလွတ်အတွင်း ကိုယ်စားမပြုနိုင်လောက်အောင် ကြီးမားသော ကိန်းဂဏန်းများအတွက်မူ bignum ဟုခေါ်သော သီးခြား ဒေတာ အမျိုးအစား တစ်ခုကို အသုံးပြုနိုင်ပြီး၊ ထိုအခါ ပွိုင်တာသည် ကိန်းဂဏန်း၏ အစိတ်အပိုင်းများကို သိမ်းဆည်းထားသည့် list တစ်ခုကို ညွှန်ပြမည် ဖြစ်သည်။295

သင်္ကေတ (symbol) တစ်ခုကို symbol ၏ ပုံနှိပ် ဖော်ပြချက် ဖြစ်စေသည့် စာလုံး အစဉ်အတန်း တစ်ခုကို ညွှန်ပြသော typed pointer တစ်ခုအဖြစ် ကိုယ်စားပြုနိုင်သည်။ ဤစာလုံး အစဉ်အတန်းကို input တွင် character string အား စတင် တွေ့ရှိသည့်အခါ Lisp reader က တည်ဆောက်သည်။ Symbol တစ်ခု၏ ဖြစ်ရပ် (instance) နှစ်ခုကို eq? အသုံးပြု၍ “တူညီသော” symbol များအဖြစ် အသိအမှတ်ပြုစေလိုပြီး eq? ကို ပွိုင်တာများ တူညီမှု ရှိမရှိ စစ်ဆေးသည့် ရိုးရှင်းသော စမ်းသပ်ချက် တစ်ခုအဖြစ် အသုံးပြုလိုသောကြောင့်၊ reader သည် တူညီသော character string ကို နှစ်ကြိမ် တွေ့ရှိပါက ဖြစ်ရပ် နှစ်ခုလုံးအတွက် (တူညီသော စာလုံး အစဉ်အတန်းသို့ ညွှန်ပြသည့်) တူညီသော ပွိုင်တာကို အသုံးပြုရန် သေချာအောင် လုပ်ဆောင်ရမည်။ ဤအရာကို လုပ်ဆောင်နိုင်ရန် reader သည် ၎င်းတွေ့ရှိခဲ့ဖူးသမျှသော symbol အားလုံး၏ ဇယားတစ်ခုကို ထိန်းသိမ်းထားပြီး ၎င်းကို ထုံးစံအားဖြင့် obarray ဟု ခေါ်သည်။ Reader သည် character string တစ်ခုကို တွေ့ရှိပြီး symbol တစ်ခု တည်ဆောက်တော့မည့်အခါ အဆိုပါ character string ကို ယခင်က တွေ့ရှိခဲ့ဖူးခြင်း ရှိမရှိ obarray တွင် စစ်ဆေးသည်။ မတွေ့ဖူးသေးပါက ၎င်းစာလုံးများကို အသုံးပြု၍ symbol အသစ်တစ်ခု (စာလုံး အစဉ်အတန်း အသစ်သို့ ညွှန်ပြသည့် typed pointer တစ်ခု) ကို တည်ဆောက်ပြီး ဤပွိုင်တာကို obarray ထဲသို့ ထည့်သွင်းသည်။ Reader သည် အဆိုပါ string ကို ယခင်က တွေ့ဖူးပြီးသား ဖြစ်ပါက obarray တွင် သိမ်းဆည်းထားသော symbol pointer ကို ပြန်ပေးသည်။ Character string များကို သီးခြား ပွိုင်တာများဖြင့် အစားထိုးသည့် ဤလုပ်ငန်းစဉ်ကို symbol များကို interning လုပ်ခြင်းဟု ခေါ်သည်။

Implementing the primitive list operations

အထက်ဖော်ပြပါ ကိုယ်စားပြု နည်းစနစ်ကို အသုံးပြု၍ register machine တစ်ခု၏ “အခြေခံ” (primitive) list လုပ်ဆောင်ချက် တစ်ခုစီကို အခြေခံ vector လုပ်ဆောင်ချက် တစ်ခု သို့မဟုတ် တစ်ခုထက်ပိုသော လုပ်ဆောင်ချက်များဖြင့် အစားထိုးနိုင်သည်။ Memory vector များကို ခွဲခြားသတ်မှတ်ရန် the-cars နှင့် the-cdrs ဟူသော register နှစ်ခုကို အသုံးပြုမည်ဖြစ်ပြီး vector-ref နှင့် vector-set! တို့ကို အခြေခံ လုပ်ဆောင်ချက်များအဖြစ် အသုံးပြုနိုင်သည်ဟု ယူဆပါမည်။ ပွိုင်တာများပေါ်တွင် လုပ်ဆောင်သော ကိန်းဂဏန်း လုပ်ဆောင်ချက်များ (ဥပမာ- ပွိုင်တာတစ်ခုကို တိုးမြှင့်ခြင်း၊ vector တစ်ခုကို ညွှန်းဆိုရန် pair pointer ကို အသုံးပြုခြင်း သို့မဟုတ် ကိန်းဂဏန်း နှစ်ခုကို ပေါင်းခြင်း) သည် typed pointer ၏ index အစိတ်အပိုင်းကိုသာ အသုံးပြုသည်ဟုလည်း ယူဆပါမည်။

ဥပမာအားဖြင့် ကျွန်ုပ်တို့သည် register machine တစ်ခုကို အောက်ပါ ညွှန်ကြားချက်များ (instructions) လုပ်ဆောင်နိုင်ရန် ဖန်တီးနိုင်သည် - 

(assign ⟨reg₁⟩ (op car) (reg ⟨reg₂⟩))
(assign ⟨reg₁⟩ (op cdr) (reg ⟨reg₂⟩))

အကယ်၍ ၎င်းတို့ကို အသီးသီး အောက်ပါအတိုင်း အကောင်အထည်ဖော်ပါက - 

(assign ⟨reg₁⟩ 
        (op vector-ref)
        (reg the-cars)
        (reg ⟨reg₂⟩))
(assign ⟨reg₁⟩
        (op vector-ref)
        (reg the-cdrs)
        (reg ⟨reg₂⟩))

အောက်ပါ ညွှန်ကြားချက်များကို 

(perform (op set-car!) (reg ⟨reg₁⟩) (reg ⟨reg₂⟩))
(perform (op set-cdr!) (reg ⟨reg₁⟩) (reg ⟨reg₂⟩))

         (reg the-cars)
         (reg ⟨reg₁⟩)
         (reg ⟨reg₂⟩))
(perform (op vector-set!)
         (reg the-cdrs)
         (reg ⟨reg₁⟩)

Cons လုပ်ဆောင်ချက်ကို အသုံးမပြုရသေးသော index တစ်ခုကို သတ်မှတ်ခြင်း (allocating) နှင့် cons ၏ argument များကို the-cars နှင့် the-cdrs ရှိ အဆိုပါ index ရှိသော vector တည်နေရာတွင် သိမ်းဆည်းခြင်းဖြင့် လုပ်ဆောင်သည်။ နောက်ထပ် ရရှိနိုင်မည့် index ပါဝင်သော pair pointer ကို အမြဲတမ်း သိမ်းဆည်းထားသည့် free ဟုခေါ်သော အထူး register တစ်ခု ရှိသည်ဟု ကျွန်ုပ်တို့ ယူဆပြီး နောက်ထပ် လွတ်နေသော တည်နေရာကို ရှာဖွေရန် ထိုပွိုင်တာ၏ index အစိတ်အပိုင်းကို တိုးမြှင့်နိုင်သည်ဟု ယူဆသည်။296 ဥပမာအားဖြင့် ညွှန်ကြားချက် 

(assign ⟨reg₁⟩
        (op cons)
        (reg ⟨reg₂⟩)
        (reg ⟨reg₃⟩))

ကို အောက်ပါ vector လုပ်ဆောင်ချက် အစီအစဉ်အတိုင်း အကောင်အထည်ဖော်သည် -297 

(perform (op vector-set!)
         (reg the-cars)
         (reg free)
         (reg ⟨reg₂⟩))

         (reg the-cdrs)

         (reg ⟨reg₃⟩))

Implementing stacks

ကျွန်ုပ်တို့၏ register machine များသည် stack များကို အသုံးပြုသော်လည်း stack များကို list များဖြင့် ပုံဖော်နိုင်သောကြောင့် ဤနေရာတွင် ထူးထူးခြားခြား လုပ်ဆောင်ရန် မလိုပါ။ Stack သည် သိမ်းဆည်းထားသော တန်ဖိုးများ၏ list တစ်ခု ဖြစ်နိုင်ပြီး ၎င်းကို အထူး register ဖြစ်သော the-stack က ညွှန်ပြထားမည်။ ထို့ကြောင့် (save ⟨reg⟩) ကို အောက်ပါအတိုင်း အကောင်အထည်ဖော်နိုင်သည် - 

(assign the-stack 
        (op cons)
        (reg ⟨reg⟩)
        (reg the-stack))

အလားတူပင် (restore ⟨reg⟩) ကို အောက်ပါအတိုင်း အကောင်အထည်ဖော်နိုင်သည် - 

(assign ⟨reg⟩ (op car) (reg the-stack))
(assign the-stack (op cdr) (reg the-stack))

ယခု ကျွန်ုပ်တို့၏ register-machine ဘာသာစကားကို အသုံးပြု၍ stop-and-copy algorithm ကို အသေးစိတ် ဖော်ပြပါမည်။ ရောက်ရှိနိုင်သော (accessible) data အားလုံးကို နောက်ဆုံးတွင် ညွှန်ပြမည့် structure တစ်ခုသို့ pointer သိမ်းဆည်းထားသော root ဟုခေါ်သည့် register တစ်ခု ရှိသည်ဟု ကျွန်ုပ်တို့ ယူဆပါမည်။ Garbage collection မစတင်မီ စက်၏ register အားလုံးရှိ အကြောင်းအရာများကို root က ညွှန်ပြနေသော ကြိုတင်သတ်မှတ်ထားသည့် list တစ်ခုတွင် သိမ်းဆည်းခြင်းဖြင့် ဤအစီအစဉ်ကို လုပ်ဆောင်နိုင်သည်။301 လက်ရှိ working memory အပြင်၊ အသုံးဝင်သော data များကို ကူးယူနိုင်ရန်အတွက် ရရှိနိုင်သော free memory ရှိသည်ဟုလည်း ကျွန်ုပ်တို့ ယူဆသည်။ လက်ရှိ working memory တွင် base address များကို the-cars နှင့် the-cdrs ဟုခေါ်သော register များတွင် သိမ်းဆည်းထားသည့် vector များ ပါဝင်ပြီး၊ free memory မှာ new-cars နှင့် new-cdrs ဟုခေါ်သော register များတွင် ရှိသည်။

လက်ရှိ working memory ထဲရှိ free cell များ ကုန်ဆုံးသွားသောအခါ၊ ဆိုလိုသည်မှာ cons လုပ်ဆောင်ချက်သည် free pointer ကို memory vector ၏ အဆုံးထက် ကျော်လွန်၍ တိုးမြှင့်ရန် ကြိုးပမ်းသောအခါ garbage collection စတင်လုပ်ဆောင်သည်။ Garbage-collection ဖြစ်စဉ် ပြီးဆုံးသောအခါ၊ root pointer သည် memory အသစ်ထဲသို့ ညွှန်ပြနေမည်ဖြစ်ပြီး၊ root မှတစ်ဆင့် ရောက်ရှိနိုင်သော object အားလုံးကို memory အသစ်သို့ ပြောင်းရွှေ့ပြီး ဖြစ်လိမ့်မည်၊ ထို့ပြင် free pointer သည် memory အသစ်တွင် pair အသစ်တစ်ခု နေရာချပေးနိုင်မည့် နောက်ထပ်နေရာကို ညွှန်ပြနေမည် ဖြစ်သည်။ ထို့အပြင်၊ working memory နှင့် memory အသစ်တို့၏ အခန်းကဏ္ဍများကို အပြန်အလှန် လဲလှယ်ပြီး ဖြစ်လိမ့်မည် — pair အသစ်များကို memory အသစ်တွင် free က ညွှန်ပြသော နေရာမှ စတင်၍ တည်ဆောက်မည်ဖြစ်ပြီး၊ (ယခင်) working memory သည် နောက်တစ်ကြိမ် garbage collection အတွက် memory အသစ်အဖြစ် အသုံးပြုနိုင်မည် ဖြစ်သည်။ ပုံ ၅.၁၅ တွင် garbage collection မတိုင်မီနှင့် ပြီးနောက် memory စီစဉ်ပုံကို ပြသထားသည်။

Garbage-collection ဖြစ်စဉ်၏ အခြေအနေကို free နှင့် scan ဟူသော pointer နှစ်ခုကို ထိန်းသိမ်းခြင်းဖြင့် ထိန်းချုပ်သည်။ ၎င်းတို့ကို memory အသစ်၏ အစသို့ ညွှန်ပြရန် အစပျိုး (initialize) သတ်မှတ်သည်။ Algorithm သည် root က ညွှန်ပြနေသော pair ကို memory အသစ်၏ အစသို့ ပြောင်းရွှေ့ခြင်းဖြင့် စတင်သည်။ Pair ကို ကူးယူသည်၊ root pointer ကို နေရာအသစ်သို့ ညွှန်ပြရန် ညှိနှိုင်းသည်၊ ထို့နောက် free pointer ကို တိုးမြှင့်သည်။ ထို့အပြင်၊ pair ၏ မူလနေရာကို ၎င်း၏ အကြောင်းအရာများ ရွှေ့ပြောင်းပြီးဖြစ်ကြောင်း ပြသရန် အမှတ်အသား ပြုလုပ်သည်။ ဤအမှတ်အသားကို အောက်ပါအတိုင်း ပြုလုပ်သည်- car တည်နေရာတွင်၊ ဤအရာသည် ရွှေ့ပြောင်းပြီးသား object တစ်ခုဖြစ်ကြောင်း အချက်ပြသည့် အထူး tag တစ်ခုကို ထားရှိသည်။ (ထိုသို့သော object ကို ထုံးစံအားဖြင့် broken heart ဟု ခေါ်သည်။)302 cdr တည်နေရာတွင်မူ object ကို ရွှေ့ပြောင်းထားသည့် နေရာသို့ ညွှန်ပြသော forwarding address ကို ထားရှိသည်။

Root ကို ပြောင်းရွှေ့ပြီးနောက်၊ garbage collector သည် ၎င်း၏ အခြေခံ စက်ဝန်း (cycle) ထဲသို့ ဝင်ရောက်သည်။ Algorithm ၏ အဆင့်တိုင်းတွင်၊ scan pointer (အစပိုင်းတွင် ပြောင်းရွှေ့ထားသော root ကို ညွှန်ပြနေသည်) သည် memory အသစ်သို့ ပြောင်းရွှေ့ထားသော်လည်း ၎င်း၏ car နှင့် cdr pointer များသည် memory အဟောင်းထဲရှိ object များကို ညွှန်ပြနေဆဲဖြစ်သော pair တစ်ခုကို ညွှန်ပြနေသည်။ ဤ object တစ်ခုစီကို ပြောင်းရွှေ့ပြီး scan pointer ကို တိုးမြှင့်သည်။ Object တစ်ခုကို ပြောင်းရွှေ့ရန် (ဥပမာ၊ ကျွန်ုပ်တို့ scan ဖတ်နေသော pair ၏ car pointer က ညွှန်ပြသော object) ၎င်း object သည် ရွှေ့ပြောင်းပြီးသား ဟုတ်မဟုတ် စစ်ဆေးသည် (object ၏ car တည်နေရာတွင် broken-heart tag ရှိမရှိကို ကြည့်၍ စစ်ဆေးခြင်းဖြစ်သည်)။ အကယ်၍ object သည် မရွှေ့ပြောင်းရသေးပါက၊ ၎င်းကို free က ညွှန်ပြသော နေရာသို့ ကူးယူသည်၊ free ကို update လုပ်သည်၊ object ၏ မူလနေရာတွင် broken heart တစ်ခု သတ်မှတ်သည်၊ ထို့နောက် object ဆီသို့ ညွှန်ပြသော pointer ကို (ဤဥပမာတွင် ကျွန်ုပ်တို့ scan ဖတ်နေသော pair ၏ car pointer) နေရာအသစ်သို့ ညွှန်ပြရန် update လုပ်သည်။ အကယ်၍ object သည် ရွှေ့ပြောင်းပြီးသားဖြစ်ပါက၊ ၎င်း၏ forwarding address ကို (broken heart ၏ cdr တည်နေရာတွင် တွေ့ရသည်) scan ဖတ်နေသော pair ရှိ pointer ၏ အစားထိုးမှုအဖြစ် အသုံးပြုသည်။ နောက်ဆုံးတွင်၊ ရောက်ရှိနိုင်သော object အားလုံးကို ပြောင်းရွှေ့ပြီး scan ဖတ်ပြီးဖြစ်လိမ့်မည်၊ ထိုအချိန်တွင် scan pointer သည် free pointer ကို ကျော်လွန်သွားပြီး ဖြစ်စဉ် ပြီးဆုံးသွားမည် ဖြစ်သည်။

stop-and-copy algorithm ကို register machine အတွက် ညွှန်ကြားချက်များ အစီအစဉ်တစ်ခုအဖြစ် ဖော်ပြနိုင်သည်။ Object တစ်ခုကို ပြောင်းရွှေ့ခြင်း၏ အခြေခံအဆင့်ကို relocate-old-result-in-new ဟုခေါ်သော subroutine တစ်ခုက လုပ်ဆောင်သည်။ ဤ subroutine သည် ပြောင်းရွှေ့ရမည့် object သို့ pointer ဖြစ်သော ၎င်း၏ argument ကို old ဟုခေါ်သော register မှ ရယူသည်။ ၎င်းသည် သတ်မှတ်ထားသော object ကို ပြောင်းရွှေ့သည် (ထိုဖြစ်စဉ်တွင် free ကို တိုးမြှင့်သည်)၊ ပြောင်းရွှေ့ထားသော object သို့ pointer ကို new ဟုခေါ်သော register ထဲသို့ ထည့်သည်၊ ထို့နောက် relocate-continue register ထဲတွင် သိမ်းဆည်းထားသော entry point သို့ ခုန်ကူးခြင်းဖြင့် ပြန်သွားသည်။ Garbage collection စတင်ရန်၊ free နှင့် scan ကို အစပျိုးပြီးနောက် root pointer ကို ပြောင်းရွှေ့ရန်အတွက် ဤ subroutine ကို ခေါ်ယူသည်။ root ကို ပြောင်းရွှေ့ခြင်း ပြီးဆုံးသောအခါ၊ pointer အသစ်ကို root အသစ်အဖြစ် သတ်မှတ်ပြီး garbage collector ၏ main loop ထဲသို့ ဝင်ရောက်သည်။ 

begin-garbage-collection
  (assign free (const 0))
  (assign scan (const 0))
  (assign old (reg root))
  (assign relocate-continue 
          (label reassign-root))
  (goto (label relocate-old-result-in-new))
reassign-root
  (assign root (reg new))
  (goto (label gc-loop))

Garbage collector ၏ main loop တွင် scan ဖတ်ရန် object များ ကျန်ရှိသေးသလားဆိုသည်ကို ကျွန်ုပ်တို့ ဆုံးဖြတ်ရမည်။ scan pointer နှင့် free pointer တို့ တူညီနေသလားဆိုသည်ကို စမ်းသပ်ခြင်းဖြင့် ဤအရာကို လုပ်ဆောင်သည်။ အကယ်၍ pointer များ တူညီနေပါက၊ ရောက်ရှိနိုင်သော object အားလုံးကို ပြောင်းရွှေ့ပြီးဖြစ်သည်၊ ထို့ကြောင့် ရပ်တန့်ထားသော တွက်ချက်မှုကို ဆက်လက်လုပ်ဆောင်နိုင်ရန် ရှင်းလင်းရေးလုပ်ဆောင်သည့် gc-flip သို့ ခုန်ကူးသည်။ Scan ဖတ်ရန် pair များ ကျန်ရှိနေသေးပါက၊ နောက်ထပ် pair ၏ car ကို ပြောင်းရွှေ့ရန်အတွက် relocate subroutine ကို ခေါ်သည် (car pointer ကို old ထဲသို့ ထည့်သွင်းခြင်းဖြင့် ဖြစ်သည်)။ relocate-continue register ကို သတ်မှတ်ထားသောကြောင့် subroutine သည် car pointer ကို update လုပ်ရန် ပြန်လာမည် ဖြစ်သည်။ 

gc-loop
  (test (op =) (reg scan) (reg free))
  (branch (label gc-flip))
  (assign old 
          (op vector-ref)
          (reg new-cars)
          (reg scan))
  (assign relocate-continue 
          (label update-car))
  (goto (label relocate-old-result-in-new))

update-car တွင်၊ ကျွန်ုပ်တို့သည် scan ဖတ်နေသော pair ၏ car pointer ကို ပြင်ဆင်ပြီး၊ ထို pair ၏ cdr ကို ပြောင်းရွှေ့ရန် ဆက်လက် လုပ်ဆောင်သည်။ ထိုသို့ ပြောင်းရွှေ့ခြင်း ပြီးဆုံးသောအခါ update-cdr သို့ ပြန်လာသည်။ cdr ကို ပြောင်းရွှေ့ပြီး update လုပ်ပြီးနောက်၊ ထို pair ကို scan ဖတ်ခြင်း ပြီးဆုံးပြီဖြစ်သဖြင့် main loop ကို ဆက်လက် လုပ်ဆောင်သည်။

           (reg new-cars)
           (reg scan)
           (reg new))

290 ကျွန်ုပ်တို့သည် မှတ်ဉာဏ် (memory) ကို အရာဝတ္ထုများ၏ စာရင်း (lists) များအဖြစ် ကိုယ်စားပြုနိုင်သည်။ သို့သော်၊ စာရင်းတစ်ခု၏ n th အစိတ်အပိုင်းကို ရယူရန် n − 1 cdr လုပ်ဆောင်ချက်များ လိုအပ်သောကြောင့်၊ ရယူအသုံးပြုချိန် (access time) သည် အညွှန်းကိန်း (index) အပေါ်တွင် မှီခိုမှုမရှိဘဲ ဖြစ်နေမည်မဟုတ်ပေ။

293 အမျိုးအစား အချက်အလက် (type information) ကို Lisp စနစ် အကောင်အထည်ဖော်မည့် စက်၏ အသေးစိတ်အချက်အလက်များအပေါ် မူတည်၍ နည်းလမ်းအမျိုးမျိုးဖြင့် ကုဒ်ရေးသွင်းနိုင်သည်။ Lisp ပရိုဂရမ်များ၏ လုပ်ဆောင်မှု စွမ်းဆောင်ရည်သည် ဤရွေးချယ်မှုကို မည်မျှ လိမ္မာပါးနပ်စွာ ပြုလုပ်သည်ဆိုသည့်အပေါ်တွင် များစွာမူတည်လိမ့်မည်ဖြစ်သော်လည်း၊ ကောင်းမွန်သော ရွေးချယ်မှုများအတွက် ယေဘုယျ ဒီဇိုင်းစည်းမျဉ်းများကို ဖော်ထုတ်ရန် ခက်ခဲပါသည်။ အမျိုးအစားသတ်မှတ်ထားသော pointer (typed pointers) များကို အကောင်အထည်ဖော်ရန် အရိုးရှင်းဆုံးနည်းလမ်းမှာ ဒေတာအမျိုးအစားကို ကုဒ်အဖြစ်ပြောင်းလဲပေးသည့် type field တစ်ခုအဖြစ် pointer တစ်ခုစီတွင် သတ်မှတ်ထားသော bit အရေအတွက်တစ်ခုကို ခွဲဝေပေးရန်ဖြစ်သည်။ ထိုသို့သော ကိုယ်စားပြုမှုကို ဒီဇိုင်းဆွဲရာတွင် ဖြေရှင်းရမည့် အရေးကြီးသော မေးခွန်းများတွင် အောက်ပါတို့ ပါဝင်သည်- အမျိုးအစား bit မည်မျှ လိုအပ်သနည်း? vector အညွှန်းကိန်း (indices) များ မည်မျှ ကြီးမားရမည်နည်း? pointer များ၏ type field များကို ကိုင်တွယ်ရန် အခြေခံ စက်ညွှန်ကြားချက် (primitive machine instructions) များကို မည်မျှ ထိရောက်စွာ အသုံးပြုနိုင်သနည်း? type field များကို ထိရောက်စွာ ကိုင်တွယ်ရန်အတွက် အထူး ဟာ့ဒ်ဝဲများ ပါဝင်သော စက်များကို tagged architectures ရှိသည်ဟု ဆိုကြသည်။

296 လွတ်လပ်သော သိမ်းဆည်းရာနေရာ (free storage) ကို ရှာဖွေရန် အခြားနည်းလမ်းများလည်း ရှိပါသည်။ ဥပမာအားဖြင့်၊ အသုံးမပြုရသေးသော pair အားလုံးကို free list တစ်ခုအဖြစ် ချိတ်ဆက်ထားနိုင်သည်။ ကျွန်ုပ်တို့၏ လွတ်လပ်သောနေရာများသည် တစ်ခုနှင့်တစ်ခု ဆက်တိုက်ဖြစ်နေသည် (ထို့ကြောင့် pointer တစ်ခုကို တိုးမြှင့်ခြင်းဖြင့် ၎င်းတို့ကို ရယူနိုင်သည်)၊ အကြောင်းမှာ ကျွန်ုပ်တို့သည် 5.3.2 တွင် တွေ့ရမည့်အတိုင်း compacting garbage collector ကို အသုံးပြုနေသောကြောင့် ဖြစ်ပါသည်။

297 ဒါဟာ အခြေခံအားဖြင့် 3.3.1 တွင် ဖော်ပြထားသည့်အတိုင်း cons ကို set-car! နှင့် set-cdr! တို့ဖြင့် အကောင်အထည်ဖော်ခြင်း ဖြစ်သည်။ ထိုအကောင်အထည်ဖော်မှုတွင် အသုံးပြုထားသော get-new-pair လုပ်ဆောင်ချက်ကို ဤနေရာတွင် free pointer ဖြင့် ဖော်ဆောင်ထားပါသည်။

298 မှတ်ဉာဏ်များ (memories) သည် ကွန်ပျူတာ၏ သက်တမ်းတစ်လျှောက်တွင် လွတ်လပ်သော မှတ်ဉာဏ်ကုန်သွားရန် မဖြစ်နိုင်လောက်အောင် ကြီးမားလာနိုင်သောကြောင့် ရေရှည်တွင် ဤအချက်သည် မှန်ကန်ချင်မှ မှန်ကန်ပါလိမ့်မည်။ ဥပမာအားဖြင့်၊ တစ်နှစ်တွင် 3 ⋅ 10 13 မိုက်ခရိုစက္ကန့် ခန့် ရှိသည်၊ ထို့ကြောင့် ကျွန်ုပ်တို့သည် တစ်မိုက်ခရိုစက္ကန့်လျှင် တစ်ကြိမ် cons ပြုလုပ်မည်ဆိုပါက၊ မှတ်ဉာဏ်မကုန်ဘဲ နှစ်ပေါင်း 30 ခန့် လည်ပတ်နိုင်သော စက်တစ်လုံး တည်ဆောက်ရန် မှတ်ဉာဏ်ဆဲလ် (cells) ပေါင်း 10 15 ခန့် လိုအပ်မည်ဖြစ်သည်။ ထိုမျှလောက်သော မှတ်ဉာဏ်ပမာဏသည် ယနေ့ခေတ် စံနှုန်းများအရ အဓိပ္ပာယ်မရှိလောက်အောင် ကြီးမားသည်ဟု ထင်ရသော်လည်း၊ ၎င်းသည် ရုပ်ပိုင်းဆိုင်ရာအရ မဖြစ်နိုင်သည်တော့ မဟုတ်ပါ။ အခြားတစ်ဖက်တွင်မူ၊ ပရိုဆက်ဆာများသည် ပိုမိုမြန်ဆန်လာပြီး အနာဂတ်ကွန်ပျူတာတစ်လုံးတွင် မှတ်ဉာဏ်တစ်ခုတည်းအပေါ် ပြိုင်တူ (parallel) အလုပ်လုပ်သော ပရိုဆက်ဆာအမြောက်အမြား ရှိလာနိုင်သောကြောင့်၊ ကျွန်ုပ်တို့ ခန့်မှန်းထားသည်ထက် ပိုမိုမြန်ဆန်စွာ မှတ်ဉာဏ်ကို အသုံးပြုပစ်နိုင်ခြေ ရှိပါသည်။

299 ကျွန်ုပ်တို့သည် ဤနေရာတွင် stack ကို 5.3.1 တွင် ဖော်ပြထားသည့်အတိုင်း စာရင်း (list) တစ်ခုအဖြစ် ကိုယ်စားပြုထားသည်ဟု ယူဆပါသည်၊ ထို့ကြောင့် stack ပေါ်ရှိ အရာဝတ္ထုများကို stack register ရှိ pointer မှတစ်ဆင့် ရယူနိုင်သည်။

300 ဤအယူအဆကို Minsky မှ စတင်ဖော်ထုတ်ခဲ့ပြီး MIT Research Laboratory of Electronics ရှိ PDP-1 အတွက် Lisp အကောင်အထည်ဖော်မှု၏ အစိတ်အပိုင်းတစ်ခုအဖြစ် ပထမဆုံး စတင်အသုံးပြုခဲ့သည်။ ၎င်းကို Multics time-sharing system အတွက် Lisp အကောင်အထည်ဖော်မှုတွင် အသုံးပြုရန် Fenichel and Yochelson (1969) မှ ထပ်မံ တိုးတက်အောင် ပြုလုပ်ခဲ့သည်။ နောက်ပိုင်းတွင် Baker (1978) သည် garbage collection လုပ်ဆောင်နေစဉ်အတွင်း တွက်ချက်မှုများ ရပ်တန့်ရန် မလိုသော ဤနည်းလမ်း၏ “real-time” ဗားရှင်းကို ဖော်ထုတ်ခဲ့သည်။ Baker ၏ အယူအဆကို အချို့သော တည်ဆောက်ပုံများသည် ပိုမိုပျက်စီးလွယ်ပြီး အခြားတည်ဆောက်ပုံများသည် ပိုမိုခိုင်မြဲသည်ဟူသော အချက်ကို အသုံးချနိုင်ရန် Hewitt, Lieberman, နှင့် Moon (ကြည့်ပါ Lieberman and Hewitt 1983) တို့မှ ထပ်မံတိုးချဲ့ခဲ့သည်။

အခြား အသုံးများသော garbage-collection နည်းလမ်းတစ်ခုမှာ mark-sweep နည်းလမ်း ဖြစ်သည်။ ၎င်းတွင် စက်၏ register များမှ ရယူနိုင်သော တည်ဆောက်ပုံအားလုံးကို လိုက်လံစစ်ဆေးခြင်း (tracing) နှင့် ကျွန်ုပ်တို့ရောက်ရှိသည့် pair တစ်ခုစီကို မှတ်သားခြင်း (marking) တို့ ပါဝင်သည်။ ထို့နောက် ကျွန်ုပ်တို့သည် မှတ်ဉာဏ်အားလုံးကို စကင်ဖတ် (scan) ပြီး၊ အမှတ်အသားမပါသော မည်သည့်နေရာကိုမဆို အမှိုက် (garbage) အဖြစ် "သိမ်းကျုံးယူငင်" (swept up) ကာ ပြန်လည်အသုံးပြုနိုင်ရန် ပြုလုပ်ပေးပါသည်။ mark-sweep နည်းလမ်းအကြောင်း အပြည့်အစုံကို Allen 1978 တွင် ဖတ်ရှုနိုင်သည်။

Minsky-Fenichel-Yochelson algorithm သည် မှတ်ဉာဏ်၏ အသုံးဝင်သော အစိတ်အပိုင်းကိုသာ စစ်ဆေးသောကြောင့် မှတ်ဉာဏ်အများအပြားရှိသော စနစ်များတွင် အဓိကအသုံးပြုသည့် algorithm ဖြစ်သည်။ ၎င်းသည် sweep အဆင့်တွင် မှတ်ဉာဏ်အားလုံးကို စစ်ဆေးရသည့် mark-sweep နှင့် ကွာခြားပါသည်။ stop-and-copy ၏ ဒုတိယမြောက် အားသာချက်မှာ ၎င်းသည် compacting garbage collector တစ်ခု ဖြစ်ခြင်းပင် ဖြစ်သည်။ ဆိုလိုသည်မှာ garbage-collection အဆင့်၏ အဆုံးတွင် အသုံးဝင်သော ဒေတာများကို အမှိုက် pair များအားလုံး ဖယ်ရှားလျက် ဆက်တိုက်ဖြစ်သော မှတ်ဉာဏ်နေရာများသို့ ရွှေ့ပြောင်းပေးမည် ဖြစ်သည်။ အလွန်ဝေးကွာသော မှတ်ဉာဏ်လိပ်စာများကို ရယူခြင်းအတွက် အပို paging လုပ်ဆောင်ချက်များ လိုအပ်နိုင်သော virtual memory ရှိသည့် စက်များတွင် ၎င်းသည် အလွန်အရေးကြီးသော စွမ်းဆောင်ရည်ဆိုင်ရာ ထည့်သွင်းစဉ်းစားစရာ ဖြစ်နိုင်ပါသည်။

301 ဤ register စာရင်းတွင် storage-allocation စနစ်မှ အသုံးပြုသော register များ ဖြစ်သည့် — root၊ the-cars၊ the-cdrs နှင့် ဤကဏ္ဍတွင် မိတ်ဆက်မည့် အခြား register များ မပါဝင်ပါ။

302 broken heart ဟူသော ဝေါဟာရကို David Cressey က စတင်သုံးစွဲခဲ့ခြင်းဖြစ်ပြီး၊ သူသည် ၁၉၇၀ ပြည့်လွန်နှစ်များ အစောပိုင်းက MIT တွင် ဖော်ထုတ်ခဲ့သော Lisp ၏ dialect တစ်ခုဖြစ်သည့် MDL အတွက် garbage collector ကို ရေးသားခဲ့သူဖြစ်သည်။

303 Garbage collector သည် စာရင်းတည်ဆောက်ပုံဆိုင်ရာ pair? လုပ်ဆောင်ချက်အစား အနိမ့်တန်း ပရီဒီကိတ် (low-level predicate) pointer-to-pair? ကို အသုံးပြုပါသည်၊ အကြောင်းမှာ အစစ်အမှန် စနစ်တစ်ခုတွင် garbage-collection ရည်ရွယ်ချက်အတွက် pair များအဖြစ် သတ်မှတ်ထားသော အရာမျိုးစုံ ရှိနိုင်သောကြောင့် ဖြစ်သည်။ ဥပမာအားဖြင့်၊ IEEE စံနှုန်းနှင့် ကိုက်ညီသော Scheme စနစ်တစ်ခုတွင် procedure object တစ်ခုကို pair? ပရီဒီကိတ်နှင့် မကိုက်ညီသော အထူး “pair” အမျိုးအစားတစ်ခုအဖြစ် အကောင်အထည်ဖော်နိုင်သည်။ စမ်းသပ်တုပမှု (simulation) ရည်ရွယ်ချက်အတွက် pointer-to-pair? ကို pair? အဖြစ် အကောင်အထည်ဖော်နိုင်သည်။
