4.2Scheme အပေါ် မူကွဲများ — Lazy Evaluation

ယခုအခါ ကျွန်ုပ်တို့တွင် Lisp ပရိုဂရမ်တစ်ခုအဖြစ် ဖော်ပြထားသည့် evaluator တစ်ခု ရှိနေပြီဖြစ်သဖြင့်၊ evaluator ကို ပြင်ဆင်ရုံဖြင့် ဘာသာစကား ဒီဇိုင်းတွင် အခြားရွေးချယ်စရာများကို စမ်းသပ်ကြည့်နိုင်ပါသည်။ အမှန်စင်စစ်၊ ဘာသာစကားအသစ်များကို လက်ရှိရှိနေသော အဆင့်မြင့်ဘာသာစကားတစ်ခုအတွင်း ဘာသာစကားအသစ်ကို ထည့်သွင်းထားသည့် evaluator တစ်ခုကို ဦးစွာရေးသားခြင်းဖြင့် တီထွင်လေ့ရှိကြသည်။ ဥပမာအားဖြင့်၊ Lisp အသိုင်းအဝိုင်းမှ အခြားအဖွဲ့ဝင်တစ်ဦးနှင့် Lisp တွင် ပြုလုပ်ရန် အဆိုပြုထားသော ပြင်ဆင်မှုအချို့ကို ဆွေးနွေးလိုပါက၊ ထိုအပြောင်းအလဲကို ထည့်သွင်းထားသည့် evaluator တစ်ခုကို ပေးပို့နိုင်ပါသည်။ လက်ခံရရှိသူသည် ထို evaluator အသစ်ဖြင့် စမ်းသပ်နိုင်ပြီး နောက်ထပ် ပြင်ဆင်မှုများအဖြစ် မှတ်ချက်များ ပြန်လည်ပေးပို့နိုင်ပါသည်။ အဆင့်မြင့် အကောင်အထည်ဖော်မှု အခြေခံသည် evaluator ကို စမ်းသပ်ရန်နှင့် အမှားရှာရန် ပိုမိုလွယ်ကူစေရုံသာမက၊ ကျွန်ုပ်တို့၏ embedded Lisp evaluator သည် အောက်ခံ Lisp မှ primitives နှင့် control structure များကို အသုံးပြုသကဲ့သို့ပင်၊ ဒီဇိုင်နာအနေဖြင့် အောက်ခံဘာသာစကားမှ စွမ်းဆောင်ရည်များကို snarf235 လုပ်ရန်လည်း အခွင့်အလမ်း ပေးပါသည်။ နောင်အခါမှသာ (လိုအပ်ပါက) ဒီဇိုင်နာသည် အဆင့်နိမ့်ဘာသာစကား သို့မဟုတ် ဟာ့ဒ်ဝဲတွင် အလုံးစုံ အကောင်အထည်ဖော်မှုကို တည်ဆောက်ရန် ကြိုးပမ်းရမည်ဖြစ်သည်။ ဤအပိုင်းနှင့် နောက်အပိုင်းတွင် သိသာထင်ရှားသော အပိုဆောင်း expressive power ကို ပေးစွမ်းနိုင်သော Scheme အပေါ် မူကွဲအချို့ကို လေ့လာပါမည်။

4.2.1Normal Order နှင့် Applicative Order

1.1 တွင်၊ ကျွန်ုပ်တို့သည် evaluation model များအကြောင်း စတင်ဆွေးနွေးခဲ့ရာ၌ Scheme သည် applicative-order ဘာသာစကားတစ်ခု ဖြစ်ကြောင်း သတိပြုခဲ့ကြသည်၊ ဆိုလိုသည်မှာ Scheme procedure များသို့ ပေးပို့သော argument အားလုံးကို procedure ကို အသုံးချချိန်တွင် အကဲဖြတ်ပါသည်။ ၎င်းနှင့် ဆန့်ကျင်ဘက်ဖြစ်သော normal-order ဘာသာစကားများသည် procedure argument များ၏ အကဲဖြတ်မှုကို အမှန်တကယ် လိုအပ်လာသည့် တန်ဖိုးများအထိ ဆိုင်းငံ့ထားပါသည်။ Procedure argument များကို အကဲဖြတ်ခြင်းကို နောက်ဆုံးဖြစ်နိုင်သော အချိန်အထိ (ဥပမာ- primitive operation တစ်ခုမှ လိုအပ်လာသည့်အချိန်အထိ) ဆိုင်းငံ့ထားခြင်းကို lazy evaluation ဟု ခေါ်ဆိုပါသည်။236 အောက်ပါ procedure ကို လေ့လာကြည့်ပါ- 

(define (try a b)
  (if (= a 0) 1 b))

(try 0 (/ 1 0)) ကို အကဲဖြတ်ခြင်းသည် Scheme တွင် error တစ်ခု ဖြစ်ပေါ်စေပါသည်။ Lazy evaluation ဖြင့်ဆိုပါက error ရှိမည်မဟုတ်ပါ။ ထိုဖော်ပြချက်ကို အကဲဖြတ်ပါက 1 ကို ပြန်ပေးလိမ့်မည်၊ အကြောင်းမှာ argument ဖြစ်သော (/ 1 0) ကို မည်သည့်အခါမျှ အကဲဖြတ်မည် မဟုတ်သောကြောင့် ဖြစ်သည်။

Lazy evaluation ကို အသုံးပြုထားသော ဥပမာတစ်ခုမှာ procedure unless ၏ အဓိပ္ပာယ်သတ်မှတ်ချက် ဖြစ်သည်- 

(define (unless condition 
                usual-value 
                exceptional-value)
  (if condition 
      exceptional-value 
      usual-value))

၎င်းကို အောက်ပါကဲ့သို့သော ဖော်ပြချက်များတွင် အသုံးပြုနိုင်သည်- 

(unless (= b 0)
        (/ a b)
        (begin 
          (display "exception: returning 0")
          0))

၎င်းသည် applicative-order ဘာသာစကားတွင် အလုပ်လုပ်မည်မဟုတ်ပါ၊ အကြောင်းမှာ unless ကို မခေါ်ဆိုမီ usual value နှင့် exceptional value နှစ်ခုလုံးကို အကဲဖြတ်မည် ဖြစ်သောကြောင့်ဖြစ်သည် (Exercise 1.6 နှင့် နှိုင်းယှဉ်ကြည့်ပါ)။ Lazy evaluation ၏ အားသာချက်တစ်ခုမှာ unless ကဲ့သို့သော procedure အချို့သည် ၎င်းတို့၏ argument အချို့ကို အကဲဖြတ်ရာတွင် အမှားများ ဖြစ်ပေါ်စေခြင်း သို့မဟုတ် အဆုံးမရှိခြင်း ဖြစ်နေစေကာမူ အသုံးဝင်သော တွက်ချက်မှုများကို လုပ်ဆောင်နိုင်ခြင်း ဖြစ်သည်။

အကယ်၍ procedure တစ်ခု၏ body သို့ မဝင်မီ argument တစ်ခုကို အကဲမဖြတ်ရသေးပါက၊ ထို procedure သည် ထို argument အပေါ်တွင် non-strict ဖြစ်သည်ဟု ကျွန်ုပ်တို့ ပြောဆိုပါသည်။ အကယ်၍ procedure ၏ body သို့ မဝင်မီ argument ကို အကဲဖြတ်ပြီး ဖြစ်ပါက၊ ထို procedure သည် ထို argument အပေါ်တွင် strict ဖြစ်သည်ဟု ကျွန်ုပ်တို့ ပြောဆိုပါသည်။237 စစ်မှန်သော applicative-order ဘာသာစကားတွင်၊ procedure အားလုံးသည် argument တစ်ခုစီအပေါ်တွင် strict ဖြစ်ကြသည်။ စစ်မှန်သော normal-order ဘာသာစကားတွင်၊ compound procedure အားလုံးသည် argument တစ်ခုစီအပေါ်တွင် non-strict ဖြစ်ကြပြီး၊ primitive procedure များသည် strict သို့မဟုတ် non-strict ဖြစ်နိုင်သည်။ ပရိုဂရမ်မာများအား ၎င်းတို့သတ်မှတ်သော procedure များ၏ strictness အပေါ် အသေးစိတ် ထိန်းချုပ်နိုင်ခွင့်ပေးထားသော ဘာသာစကားများလည်း ရှိပါသည် (Exercise 4.31 ကို ကြည့်ပါ)။

Non-strict အဖြစ် အကျိုးရှိရှိ ပြုလုပ်နိုင်သော procedure ၏ ထင်ရှားသော ဥပမာတစ်ခုမှာ cons (သို့မဟုတ် ယေဘုယျအားဖြင့် ဒေတာ တည်ဆောက်ပုံများအတွက် constructor အားလုံးနီးပါး) ဖြစ်သည်။ Element များ၏ တန်ဖိုးများကို မသိရှိရသေးသော်လည်း၊ element များကို ပေါင်းစပ်၍ ဒေတာ တည်ဆောက်ပုံများ ပြုလုပ်ခြင်းနှင့် ရရှိလာသော ဒေတာ တည်ဆောက်ပုံများအပေါ် လုပ်ဆောင်ခြင်းဖြင့် အသုံးဝင်သော တွက်ချက်မှုများကို လုပ်ဆောင်နိုင်သည်။ ဥပမာအားဖြင့်၊ list တစ်ခုအတွင်းရှိ တစ်ခုချင်းစီသော element များ၏ တန်ဖိုးကို မသိဘဲ list ၏ အလျားကို တွက်ချက်ခြင်းသည် အဓိပ္ပာယ်ရှိလှပါသည်။ ကျွန်ုပ်တို့သည် 4.2.3 တွင် Chapter 3 မှ streams များကို non-strict cons pairs များဖြင့် ဖွဲ့စည်းထားသော list များအဖြစ် အကောင်အထည်ဖော်ရန်အတွက် ဤအကြံဉာဏ်ကို အသုံးပြုပါမည်။

Exercise 4.25: (သာမန် applicative-order Scheme တွင်) အထက်တွင် ဖော်ပြထားသည့်အတိုင်း unless ကို သတ်မှတ်ပြီး၊ factorial ကို unless ကို အသုံးပြု၍ အောက်ပါအတိုင်း သတ်မှတ်သည်ဟု ဆိုကြပါစို့။ 

(define (factorial n)
  (unless (= n 1)
          (* n (factorial (- n 1)))
          1))

အကယ်၍ ကျွန်ုပ်တို့သည် (factorial 5) ကို အကဲဖြတ်ရန် ကြိုးစားပါက ဘာဖြစ်မည်နည်း။ ကျွန်ုပ်တို့၏ သတ်မှတ်ချက်များသည် normal-order ဘာသာစကားတွင် အလုပ်လုပ်ပါမည်လား။

Exercise 4.26: Ben Bitdiddle နှင့် Alyssa P. Hacker တို့သည် unless ကဲ့သို့သော အရာများကို အကောင်အထည်ဖော်ရန်အတွက် lazy evaluation ၏ အရေးပါမှုအပေါ် သဘောထားကွဲလွဲနေကြသည်။ Ben က unless ကို applicative order တွင် special form တစ်ခုအဖြစ် အကောင်အထည်ဖော်ရန် ဖြစ်နိုင်ကြောင်း ထောက်ပြသည်။ Alyssa က အကယ်၍ ထိုသို့လုပ်ဆောင်ပါက unless သည် syntax တစ်ခုသာ ဖြစ်သွားမည်ဖြစ်ပြီး၊ higher-order procedures များနှင့် တွဲဖက်အသုံးပြုနိုင်သော procedure တစ်ခု ဖြစ်တော့မည်မဟုတ်ကြောင်း ပြန်လည်ချေပသည်။ နှစ်ဖက်စလုံး၏ ငြင်းခုံမှုဆိုင်ရာ အသေးစိတ်အချက်အလက်များကို ဖြည့်စွက်ပါ။ unless ကို (cond သို့မဟုတ် let ကဲ့သို့သော) derived expression တစ်ခုအဖြစ် မည်သို့ အကောင်အထည်ဖော်ရမည်ကို ပြသပါ၊ ထို့ပြင် unless ကို special form အဖြစ် မဟုတ်ဘဲ procedure တစ်ခုအဖြစ် ရရှိနိုင်ခြင်းက အသုံးဝင်မည့် အခြေအနေတစ်ခု၏ ဥပမာကို ပေးပါ။

4.2.2Lazy Evaluation ပါဝင်သော Interpreter တစ်ခု

ဤအခန်းတွင် ကျွန်ုပ်တို့သည် compound procedure များ၏ argument တစ်ခုစီတိုင်းတွင် non-strict ဖြစ်သည်မှအပ Scheme နှင့် တူညီသော normal-order ဘာသာစကားတစ်ခုကို အကောင်အထည်ဖော်မည် ဖြစ်သည်။ Primitive procedure များသည် strict ဖြစ်နေဆဲ ဖြစ်လိမ့်မည်။ ၎င်းအနက်ဖွင့်ဆိုသည့် ဘာသာစကားသည် ဤနည်းအတိုင်း လုပ်ဆောင်နိုင်စေရန် 4.1.1 ရှိ evaluator ကို ပြုပြင်မွမ်းမံရန် မခက်ခဲပါ။ လိုအပ်သော ပြောင်းလဲမှုအားလုံးနီးပါးသည် procedure application ပေါ်တွင် အဓိကမူတည်သည်။

အခြေခံစိတ်ကူးမှာ procedure တစ်ခုကို apply လုပ်သောအခါ၊ မည်သည့် argument များကို evaluate လုပ်မည်၊ မည်သည့်အရာများကို delay လုပ်မည်ကို interpreter က ဆုံးဖြတ်ရမည် ဖြစ်သည်။ ဆိုင်းငံ့ထားသော argument များကို evaluate မလုပ်ဘဲ၊ ၎င်းတို့အစား thunks ဟုခေါ်သော object များအဖြစ် ပြောင်းလဲလိုက်သည်။238 Thunk တွင် ၎င်းကို လိုအပ်သောအခါ argument ၏ value ကို ထုတ်ပေးနိုင်ရန် လိုအပ်သော အချက်အလက်များ ပါဝင်ရမည်ဖြစ်ပြီး၊ ၎င်းသည် application ပြုလုပ်ချိန်၌ evaluate လုပ်ခဲ့သကဲ့သို့ ဖြစ်ရမည်။ ထို့ကြောင့် thunk တွင် argument expression နှင့် procedure application ကို evaluate လုပ်နေသော environment တို့ ပါဝင်ရမည်။

Thunk တစ်ခုအတွင်းရှိ expression ကို evaluate လုပ်ခြင်း လုပ်ငန်းစဉ်ကို forcing ဟု ခေါ်သည်။239 ယေဘုယျအားဖြင့် thunk တစ်ခုကို ၎င်း၏ value လိုအပ်မှသာ force လုပ်လိမ့်မည်- ၎င်းကို thunk ၏ value ကို အသုံးပြုမည့် primitive procedure ထံ ပေးပို့သောအခါ၊ ၎င်းသည် conditional တစ်ခု၏ predicate value ဖြစ်နေသောအခါ နှင့် ၎င်းသည် procedure အဖြစ် apply လုပ်တော့မည့် operator ၏ value ဖြစ်နေသောအခါမျိုးတွင် ဖြစ်သည်။ ကျွန်ုပ်တို့တွင် ရရှိနိုင်သော ဒီဇိုင်းရွေးချယ်မှုတစ်ခုမှာ memoize လုပ်မလား၊ မလုပ်ဘူးလား ဆိုသည်ပင် ဖြစ်သည်၊ ၎င်းကို 3.5.1 ရှိ delayed object များတွင် လုပ်ဆောင်ခဲ့သကဲ့သို့ပင် ဖြစ်သည်။ Memoization ဖြင့် thunk ကို ပထမဆုံးအကြိမ် force လုပ်သောအခါ၊ တွက်ချက်ထားသော value ကို သိမ်းဆည်းထားသည်။ နောက်ပိုင်း force လုပ်မှုများသည် တွက်ချက်မှုကို ထပ်မလုပ်တော့ဘဲ သိမ်းဆည်းထားသော value ကိုသာ ပြန်ပေးမည် ဖြစ်သည်။ အပလီကေးရှင်း အတော်များများအတွက် ပိုမိုထိရောက်သောကြောင့် ကျွန်ုပ်တို့၏ interpreter ကို memoize လုပ်ရန် ဖန်တီးပါမည်။ သို့သော် ဤနေရာတွင် ခက်ခဲသော ထည့်သွင်းစဉ်းစားစရာများ ရှိပါသည်။240

Evaluator ကို ပြုပြင်မွမ်းမံခြင်း

Lazy evaluator နှင့် 4.1 ရှိ evaluator တို့၏ အဓိကကွာခြားချက်မှာ eval နှင့် apply တို့တွင် procedure application များကို ကိုင်တွယ်ပုံဖြစ်သည်။

eval ၏ application? clause သည် အောက်ပါအတိုင်း ဖြစ်လာသည် 

((application? exp)
 (apply (actual-value (operator exp) env)
        (operands exp)
        env))

၎င်းသည် 4.1.1 ရှိ eval ၏ application? clause နှင့် အတော်လေး ဆင်တူပါသည်။ သို့သော် lazy evaluation အတွက်မူ ကျွန်ုပ်တို့သည် apply ကို operand expression များဖြင့်သာ ခေါ်ဆိုပြီး ၎င်းတို့ကို evaluate လုပ်ရာမှ ရရှိလာသော argument များဖြင့် မဟုတ်ပါ။ အကယ်၍ argument များကို delay လုပ်မည်ဆိုလျှင် thunk များ တည်ဆောက်ရန် environment လိုအပ်သောကြောင့် ၎င်းကိုလည်း ထည့်ပေးရမည်ဖြစ်သည်။ apply သည် ၎င်း၏ အမျိုးအစား (primitive သို့မဟုတ် compound) ပေါ် မူတည်၍ ခွဲခြားလုပ်ဆောင်ရန်အတွက် အမှန်တကယ် apply လုပ်မည့် procedure ကို လိုအပ်သောကြောင့် operator ကိုမူ evaluate လုပ်နေဆဲဖြစ်သည်။

Expression တစ်ခု၏ အမှန်တကယ် value လိုအပ်သည့် အခါတိုင်း၊ ကျွန်ုပ်တို့သည် အောက်ပါအတိုင်း အသုံးပြုသည် 

(define (actual-value exp env)
  (force-it (eval exp env)))

eval တစ်ခုတည်း အသုံးပြုမည့်အစား အကယ်၍ expression ၏ value သည် thunk ဖြစ်နေပါက ၎င်းကို force လုပ်နိုင်ရန်အတွက် ဖြစ်သည်။

ကျွန်ုပ်တို့၏ apply version အသစ်သည်လည်း 4.1.1 ရှိ version နှင့် အတော်လေး ဆင်တူပါသည်။ ကွာခြားချက်မှာ eval က evaluate မလုပ်ရသေးသော operand expression များကို ထည့်ပေးလိုက်ခြင်း ဖြစ်သည်- Primitive procedure များအတွက် (strict ဖြစ်သော) primitive ကို apply မလုပ်မီ argument အားလုံးကို evaluate လုပ်သည်; compound procedure များအတွက် (non-strict ဖြစ်သော) procedure ကို apply မလုပ်မီ argument အားလုံးကို delay လုပ်သည်။ 

(define (apply procedure arguments env)
  (cond ((primitive-procedure? procedure)
         (apply-primitive-procedure
          procedure
          (list-of-arg-values 
           arguments 
           env)))  ; changed
        ((compound-procedure? procedure)
         (eval-sequence
          (procedure-body procedure)
          (extend-environment
           (procedure-parameters procedure)
           (list-of-delayed-args 
            arguments 
            env)   ; changed
           (procedure-environment procedure))))
        (else (error "Unknown procedure 
                      type: APPLY" 
                     procedure))))

Argument များကို လုပ်ဆောင်သော procedure များသည် 4.1.1 မှ list-of-values နှင့် တူညီကြသော်လည်း list-of-delayed-args သည် argument များကို evaluate လုပ်မည့်အစား delay လုပ်ခြင်းနှင့် list-of-arg-values သည် eval အစား actual-value ကို အသုံးပြုခြင်းတို့သာ ကွာခြားသည်- 

(define (list-of-arg-values exps env)
  (if (no-operands? exps)
      '()
      (cons (actual-value 
             (first-operand exps) 
             env)
            (list-of-arg-values 
             (rest-operands exps)
             env))))

(define (list-of-delayed-args exps env)
  (if (no-operands? exps)
      '()
      (cons (delay-it 
             (first-operand exps) 
             env)
            (list-of-delayed-args 
             (rest-operands exps)
             env))))

Evaluator တွင် ကျွန်ုပ်တို့ ပြောင်းလဲရမည့် နောက်တစ်နေရာမှာ if ကို ကိုင်တွယ်ပုံတွင် ဖြစ်သည်။ ထိုနေရာတွင် predicate expression သည် true သို့မဟုတ် false ဖြစ်မဖြစ် မစစ်ဆေးမီ ၎င်း၏ value ကို ရယူရန် eval အစား actual-value ကို အသုံးပြုရမည်ဖြစ်သည်- 

(define (eval-if exp env)
  (if (true? (actual-value (if-predicate exp) 
                           env))
      (eval (if-consequent exp) env)
      (eval (if-alternative exp) env)))

နောက်ဆုံးအနေဖြင့်၊ ကျွန်ုပ်တို့သည် driver-loop procedure (4.1.4) ကို eval အစား actual-value အသုံးပြုရန် ပြောင်းလဲရမည်။ သို့မှသာ delayed value တစ်ခုသည် read-eval-print loop သို့ ပြန်ရောက်လာပါက ၎င်းကို print မထုတ်မီ force လုပ်မည်ဖြစ်သည်။ ၎င်းသည် lazy evaluator ဖြစ်ကြောင်း ဖော်ပြရန် prompt များကိုလည်း ပြောင်းလဲပါမည်- 

(define input-prompt  ";;; L-Eval input:")
(define output-prompt ";;; L-Eval value:")

(define (driver-loop)
  (prompt-for-input input-prompt)
  (let ((input (read)))
    (let ((output (actual-value 
                   input 
                   the-global-environment)))
      (announce-output output-prompt)
      (user-print output)))
  (driver-loop))

ဤအပြောင်းအလဲများ ပြုလုပ်ပြီးနောက်၊ ကျွန်ုပ်တို့သည် evaluator ကို စတင်၍ စမ်းသပ်နိုင်သည်။ 4.2.1 တွင် ဆွေးနွေးခဲ့သော try expression ကို အောင်မြင်စွာ evaluate လုပ်နိုင်ခြင်းက interpreter သည် lazy evaluation ကို လုပ်ဆောင်နေကြောင်း ညွှန်ပြနေသည်- 

(define the-global-environment 
  (setup-environment))

(driver-loop)

;;; L-Eval input:
(define (try a b) (if (= a 0) 1 b))

;;; L-Eval value:
ok

;;; L-Eval input:
(try 0 (/ 1 0))

;;; L-Eval value:
1
Thunk များကို ဖော်ပြခြင်း

ကျွန်ုပ်တို့၏ evaluator သည် procedure များကို argument များထံ apply လုပ်သောအခါ thunk များ ဖန်တီးရန်နှင့် ထို thunk များကို နောက်မှ force လုပ်ရန် စီစဉ်ရမည်။ Thunk တစ်ခုသည် argument ကို နောက်မှ ထုတ်ပေးနိုင်ရန်အတွက် expression ကို environment နှင့်အတူ ထုပ်ပိုးထားရမည်။ Thunk ကို force လုပ်ရန်၊ ကျွန်ုပ်တို့သည် thunk ထဲမှ expression နှင့် environment ကို ထုတ်ယူပြီး environment ထဲတွင် expression ကို evaluate လုပ်ရုံသာဖြစ်သည်။ Expression ၏ value သည် ကိုယ်တိုင် thunk ဖြစ်နေခဲ့လျှင် ၎င်းကိုပါ force လုပ်နိုင်ရန်နှင့် thunk မဟုတ်သော အရာတစ်ခု ရောက်သည်အထိ ဆက်လက်လုပ်ဆောင်နိုင်ရန် eval ထက် actual-value ကို အသုံးပြုသည်။ 

(define (force-it obj)
  (if (thunk? obj)
      (actual-value (thunk-exp obj) 
                    (thunk-env obj))
      obj))

Expression တစ်ခုကို environment နှင့် ထုပ်ပိုးရန် လွယ်ကူသော နည်းလမ်းတစ်ခုမှာ expression နှင့် environment ပါဝင်သော list တစ်ခု ပြုလုပ်ခြင်း ဖြစ်သည်။ ထို့ကြောင့် ကျွန်ုပ်တို့သည် thunk တစ်ခုကို အောက်ပါအတိုင်း ဖန်တီးသည်- 

(define (delay-it exp env)
  (list 'thunk exp env))
(define (thunk? obj) (tagged-list? obj 'thunk))
(define (thunk-exp thunk) (cadr thunk))
(define (thunk-env thunk) (caddr thunk))

အမှန်တကယ်တွင် ကျွန်ုပ်တို့၏ interpreter အတွက် လိုချင်သည်မှာ ဤအရာမျိုး မဟုတ်ဘဲ memoize လုပ်ထားသော thunk များ ဖြစ်သည်။ Thunk တစ်ခုကို force လုပ်သောအခါ၊ သိမ်းဆည်းထားသော expression ကို ၎င်း၏ value ဖြင့် အစားထိုးပြီး thunk tag ကို ပြောင်းလဲခြင်းအားဖြင့် ၎င်းကို evaluate လုပ်ပြီးသား thunk တစ်ခုအဖြစ် ပြောင်းလဲမည် ဖြစ်သည်။ သို့မှသာ ၎င်းကို evaluate လုပ်ပြီးသားအဖြစ် မှတ်မိနိုင်မည် ဖြစ်သည်။241 

(define (evaluated-thunk? obj)
  (tagged-list? obj 'evaluated-thunk))

(define (thunk-value evaluated-thunk) 
  (cadr evaluated-thunk))

(define (force-it obj)
  (cond ((thunk? obj)
         (let ((result
                (actual-value 
                 (thunk-exp obj)
                 (thunk-env obj))))
           (set-car! obj 'evaluated-thunk)
           ;; replace exp with its value:
           (set-car! (cdr obj) result) 
           ;; forget unneeded env:
           (set-cdr! (cdr obj) '()) 
           result))
        ((evaluated-thunk? obj)
         (thunk-value obj))
        (else obj)))

244 ဤသည်မှာ လေ့ကျင့်ခန်း ၂.၄ တွင် ဖော်ပြထားသော procedural representation ဖြစ်သည်။ အခြေခံအားဖြင့် မည်သည့် procedural representation (ဥပမာ- message-passing implementation) မဆို ကောင်းမွန်စွာ လုပ်ဆောင်နိုင်ပါလိမ့်မည်။ ဤ definition များကို driver loop တွင် ရိုက်ထည့်ရုံဖြင့် lazy evaluator တွင် ထည့်သွင်းနိုင်သည်ကို သတိပြုပါ။ အကယ်၍ ကျွန်ုပ်တို့သည် မူလက cons၊ car နှင့် cdr တို့ကို global environment တွင် primitive များအဖြစ် ထည့်သွင်းခဲ့ပါက၊ ၎င်းတို့ကို ပြန်လည်သတ်မှတ် (redefined) ပါလိမ့်မည်။ (လေ့ကျင့်ခန်း ၄.၃၃ နှင့် လေ့ကျင့်ခန်း ၄.၃၄ ကိုလည်း ကြည့်ပါ။)

245 ၎င်းသည် ကျွန်ုပ်တို့အား sequence များသာမက ပိုမိုယေဘုယျကျသော list structure အမျိုးအစားများ၏ delayed version များကို ဖန်တီးနိုင်စေသည်။ Hughes 1990 တွင် “lazy trees” ၏ အသုံးချပုံအချို့ကို ဆွေးနွေးထားသည်။
