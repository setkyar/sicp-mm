5.4The Explicit-Control Evaluator

5.1 တွင် ရိုးရှင်းသော Scheme ပရိုဂရမ်များကို register machines ဖော်ပြချက်များအဖြစ် မည်သို့ပြောင်းလဲရမည်ကို ကျွန်ုပ်တို့ မြင်တွေ့ခဲ့ရသည်။ ယခုအခါ ပိုမိုရှုပ်ထွေးသော ပရိုဂရမ်တစ်ခုဖြစ်သည့် 4.1.1–4.1.4 ရှိ metacircular evaluator အပေါ် ဤသို့ပြောင်းလဲခြင်းကို လုပ်ဆောင်ပါမည်။ ၎င်းသည် Scheme interpreter တစ်ခု၏ လုပ်ဆောင်ပုံကို eval နှင့် apply procedure များဖြင့် မည်သို့ဖော်ပြနိုင်သည်ကို ပြသထားသည်။ ဤအပိုင်းတွင် ကျွန်ုပ်တို့ ဖော်ထုတ်မည့် explicit-control evaluator သည် evaluation လုပ်ငန်းစဉ်တွင် အသုံးပြုသည့် အခြေခံ procedure-calling နှင့် argument-passing ယန္တရားများကို registers နှင့် stacks များပေါ်ရှိ operations များဖြင့် မည်သို့ဖော်ပြနိုင်သည်ကို ပြသပါသည်။ ထို့အပြင် explicit-control evaluator သည် သမားရိုးကျ ကွန်ပျူတာများ၏ native machine language နှင့် အလွန်ဆင်တူသော ဘာသာစကားဖြင့် ရေးသားထားသော Scheme interpreter တစ်ခု၏ implementation အဖြစ်လည်း အသုံးတော်ခံနိုင်ပါသည်။ ဤ evaluator ကို 5.2 ရှိ register-machine simulator ဖြင့် run နိုင်ပါသည်။ သို့မဟုတ်ပါက ၎င်းကို Scheme evaluator ၏ machine-language implementation တစ်ခု တည်ဆောက်ရန်အတွက် သော်လည်းကောင်း၊ Scheme expressions များကို evaluate လုပ်ရန် သီးသန့်ရည်ရွယ်သော machine တစ်ခု တည်ဆောက်ရန်အတွက်သော်လည်းကောင်း အစပျိုးအမှတ်အဖြစ် အသုံးပြုနိုင်ပါသည်။ Figure 5.16 တွင် ထိုသို့သော hardware implementation တစ်ခုဖြစ်သည့် Scheme အတွက် evaluator အဖြစ် လုပ်ဆောင်သော silicon chip တစ်ခုကို ပြသထားသည်။ chip ဒီဇိုင်းနာများသည် ဤအပိုင်းတွင် ဖော်ပြထားသော evaluator နှင့် ဆင်တူသော register machine တစ်ခုအတွက် data-path နှင့် controller သတ်မှတ်ချက်များမှ စတင်ခဲ့ပြီး၊ integrated-circuit layout ကို တည်ဆောက်ရန် design automation programs များကို အသုံးပြုခဲ့ကြသည်။304

Registers နှင့် operations များ

explicit-control evaluator ကို ဒီဇိုင်းဆွဲရာတွင် ကျွန်ုပ်တို့၏ register machine ၌ အသုံးပြုမည့် operations များကို သတ်မှတ်ရပါမည်။ ကျွန်ုပ်တို့သည် metacircular evaluator ကို quoted? နှင့် make-procedure ကဲ့သို့သော procedure များကို အသုံးပြု၍ abstract syntax အနေဖြင့် ဖော်ပြခဲ့သည်။ register machine ကို အကောင်အထည်ဖော်ရာတွင် ဤ procedure များကို အခြေခံ list-structure memory operations အစီအစဉ်များအဖြစ် ချဲ့ထွင်နိုင်ပြီး၊ ဤ operations များကို ကျွန်ုပ်တို့၏ register machine ပေါ်တွင် အကောင်အထည်ဖော်နိုင်ပါသည်။ သို့သော် ဤသို့လုပ်ဆောင်ခြင်းက ကျွန်ုပ်တို့၏ evaluator ကို အလွန်ရှည်လျားသွားစေပြီး အခြေခံဖွဲ့စည်းပုံကို အသေးစိတ်အချက်အလက်များဖြင့် ဖုံးကွယ်သွားစေပါလိမ့်မည်။ တင်ပြချက်ကို ရှင်းလင်းစေရန်အတွက် 4.1.2 တွင် ပေးထားသော syntax procedure များနှင့် အပိုင်း 4.1.3 နှင့် 4.1.4 တွင် ပေးထားသော environments နှင့် အခြား run-time data များကို ကိုယ်စားပြုရန် procedure များကို register machine ၏ primitive operations များအဖြစ် ထည့်သွင်းပါမည်။ low-level machine language ဖြင့် ပရိုဂရမ်ရေးနိုင်သော သို့မဟုတ် hardware ဖြင့် အကောင်အထည်ဖော်နိုင်သော evaluator တစ်ခုကို အပြည့်အစုံ သတ်မှတ်ရန်အတွက်၊ 5.3 တွင် ကျွန်ုပ်တို့ ဖော်ပြခဲ့သော list-structure implementation ကို အသုံးပြု၍ ဤ operations များကို ပိုမိုအခြေခံကျသော operations များဖြင့် အစားထိုးရပါမည်။

ကျွန်ုပ်တို့၏ Scheme evaluator register machine တွင် stack တစ်ခုနှင့် register ခုနစ်ခု ပါဝင်သည်- exp, env, val, continue, proc, argl, နှင့် unev တို့ ဖြစ်ကြသည်။ Exp ကို evaluate လုပ်ရမည့် expression ကို သိမ်းဆည်းရန် အသုံးပြုပြီး၊ env တွင် evaluation လုပ်ဆောင်မည့် environment ပါဝင်သည်။ evaluation အဆုံးတွင် သတ်မှတ်ထားသော environment ထဲ၌ expression ကို evaluate လုပ်ခြင်းဖြင့် ရရှိလာသော value ကို val တွင် သိမ်းဆည်းထားသည်။ continue register ကို 5.1.4 တွင် ရှင်းပြထားသည့်အတိုင်း recursion ကို အကောင်အထည်ဖော်ရန် အသုံးပြုပါသည်။ (expression တစ်ခုကို evaluate လုပ်ရန် ၎င်း၏ subexpressions များကို evaluate လုပ်ရန် လိုအပ်သောကြောင့် evaluator သည် မိမိကိုယ်ကို recursive အနေဖြင့် ခေါ်ယူရန် လိုအပ်ပါသည်။) proc, argl, နှင့် unev register များကို combinations များကို evaluate လုပ်ရာတွင် အသုံးပြုပါသည်။

evaluator ၏ registers နှင့် operations များ မည်သို့ချိတ်ဆက်ထားသည်ကို ပြသရန် data-path diagram ကို ပေးမည်မဟုတ်သလို၊ machine operations စာရင်းအပြည့်အစုံကိုလည်း ပေးမည်မဟုတ်ပါ။ ဤအရာများသည် အသေးစိတ် တင်ပြသွားမည့် evaluator ၏ controller ထဲတွင် သွယ်ဝိုက်၍ ပါဝင်ပြီးသား ဖြစ်ပါသည်။

5.4.1Explicit-Control Evaluator ၏ အဓိက အစိတ်အပိုင်း

အကဲဖြတ်စနစ် (evaluator) ၏ ဗဟိုချက်မှာ eval-dispatch မှ စတင်သော instruction အစီအစဉ်များ ဖြစ်သည်။ ၎င်းသည် 4.1.1 တွင် ဖော်ပြထားသော metacircular evaluator ၏ eval procedure နှင့် ကိုက်ညီသည်။ Controller သည် eval-dispatch မှ စတင်သောအခါ၊ env တွင် သတ်မှတ်ထားသော ပတ်ဝန်းကျင် (environment) ၌ exp တွင် သတ်မှတ်ထားသော ဖော်ပြချက် (expression) ကို အကဲဖြတ်သည်။ အကဲဖြတ်ခြင်း ပြီးဆုံးသောအခါ၊ controller သည် continue တွင် သိမ်းဆည်းထားသော entry point သို့ သွားမည်ဖြစ်ပြီး၊ val register တွင် expression ၏ တန်ဖိုး ရှိနေမည်ဖြစ်သည်။ Metacircular eval ကဲ့သို့ပင်၊ eval-dispatch ၏ တည်ဆောက်ပုံမှာ အကဲဖြတ်မည့် expression ၏ syntactic type ပေါ်တွင် မူတည်၍ ခွဲခြမ်းစိတ်ဖြာခြင်း (case analysis) ဖြစ်သည်။305

Procedure application များကို အကဲဖြတ်ခြင်း

Procedure application တစ်ခုကို operator နှင့် operands များ ပါဝင်သော combination တစ်ခုဖြင့် သတ်မှတ်သည်။ Operator သည် တန်ဖိုးမှာ procedure တစ်ခု ဖြစ်လာမည့် subexpression ဖြစ်ပြီး၊ operands များသည် အဆိုပါ procedure ကို apply လုပ်မည့် arguments များ ဖြစ်လာမည့် subexpression များ ဖြစ်ကြသည်။ Metacircular eval သည် combination ၏ အစိတ်အပိုင်းတစ်ခုချင်းစီကို အကဲဖြတ်ရန် မိမိကိုယ်ကို recursive စနစ်ဖြင့် ပြန်လည်ခေါ်ယူခြင်းဖြင့် application များကို ကိုင်တွယ်ပြီး၊ ရလဒ်များကို အမှန်တကယ် procedure application လုပ်ဆောင်ပေးသည့် apply ထံသို့ ပေးပို့သည်။ Explicit-control evaluator သည်လည်း ထိုအတိုင်းပင် လုပ်ဆောင်သည်။ ဤ recursive ခေါ်ဆိုမှုများကို goto instruction များဖြင့် အကောင်အထည်ဖော်ထားပြီး၊ recursive call မှ ပြန်မလာမီ ပြန်လည် ထုတ်ယူရမည့် (restore လုပ်ရမည့်) register များကို သိမ်းဆည်းရန် stack ကို အသုံးပြုသည်။ ခေါ်ဆိုမှုတစ်ခုစီ မပြုလုပ်မီ၊ မည်သည့် register များကို သိမ်းဆည်းရန် လိုအပ်သည်ကို (၎င်းတို့၏ တန်ဖိုးများကို နောက်ပိုင်းတွင် လိုအပ်မည်ဖြစ်သောကြောင့်) ဂရုတစိုက် သတ်မှတ်မည်ဖြစ်သည်။306

ကျွန်ုပ်တို့သည် application တစ်ခုကို အကဲဖြတ်ရာတွင် procedure တစ်ခု ရရှိရန်အတွက် operator ကို အရင်ဆုံး အကဲဖြတ်ခြင်းဖြင့် စတင်သည်။ ၎င်းကို နောက်ပိုင်းတွင် အကဲဖြတ်ထားသော operands များ၌ apply လုပ်မည်ဖြစ်သည်။ Operator ကို အကဲဖြတ်ရန်၊ ၎င်းကို exp register သို့ ရွှေ့ပြီး eval-dispatch သို့ သွားသည်။ env register ရှိ environment သည် operator ကို အကဲဖြတ်ရန်အတွက် မှန်ကန်သော environment ဖြစ်နေပြီဖြစ်သည်။ သို့သော်လည်း operands များကို အကဲဖြတ်ရန်အတွက် နောက်ပိုင်းတွင် လိုအပ်မည်ဖြစ်သောကြောင့် env ကို သိမ်းဆည်းထားသည်။ Operands များကိုလည်း unev ထဲသို့ ထုတ်ယူပြီး stack ပေါ်တွင် သိမ်းဆည်းသည်။ Operator ကို အကဲဖြတ်ပြီးနောက် eval-dispatch သည် ev-appl-did-operator တွင် ပြန်လည်စတင်နိုင်ရန် continue ကို သတ်မှတ်သည်။ သို့သော် ပထမဦးစွာ၊ application ပြီးနောက် မည်သည့်နေရာတွင် ဆက်လက်လုပ်ဆောင်ရမည်ကို controller အား အသိပေးသည့် continue ၏ တန်ဖိုးဟောင်းကို သိမ်းဆည်းထားသည်။

iterative process တစ်ခုဖြစ်ကြောင်း ကျွန်ုပ်တို့ ပြောခဲ့ပါသည်။ အဆိုပါ procedure သည် ပုံသဏ္ဌာန်အားဖြင့် recursive ဖြစ်နေသော်လည်း (၎င်းကိုယ်တိုင်အား ပြန်လည်အသုံးပြု၍ အဓိပ္ပာယ်ဖွင့်ဆိုထားသော်လည်း)၊ evaluator တစ်ခုအနေဖြင့် sqrt-iter တစ်ကြိမ်ခေါ်ဆိုမှုမှ နောက်တစ်ကြိမ်သို့ ကူးပြောင်းရာတွင် အချက်အလက်များကို သိမ်းဆည်းထားရန် ယုတ္တိဗေဒအရ မလိုအပ်ပါ။310 sqrt-iter ကဲ့သို့သော procedure တစ်ခုကို မိမိကိုယ်ကိုယ် ဆက်တိုက်ခေါ်ဆိုနေသော်လည်း လိုအပ်သော storage ပမာဏ ပိုမိုများပြားလာခြင်း မရှိဘဲ လုပ်ဆောင်နိုင်သော evaluator ကို 
tail-recursive evaluator ဟု ခေါ်ဆိုပါသည်။ Chapter 4 ရှိ evaluator ၏ metacircular implementation တွင် evaluator သည် tail-recursive ဖြစ်မဖြစ်ကို သတ်မှတ်ထားခြင်း မရှိပါ။ အကြောင်းမှာ ထို evaluator သည် ၎င်း၏ အခြေအနေ သိမ်းဆည်းမှု ယန္တရားကို အခြေခံထားသော Scheme ထံမှ ဆက်ခံထားသောကြောင့် ဖြစ်ပါသည်။ သို့သော် explicit-control evaluator တွင်မူ procedure ခေါ်ဆိုမှုများသည် stack ပေါ်တွင် အချက်အလက်များ စုပုံလာခြင်းကို မည်သည့်အချိန်တွင် ဖြစ်ပေါ်စေသည်ကို အကဲဖြတ်မှု လုပ်ငန်းစဉ်တစ်လျှောက် လိုက်လံကြည့်ရှုနိုင်ပါသည်။



ကျွန်ုပ်တို့၏ evaluator သည် tail-recursive ဖြစ်ပါသည်။ အကြောင်းမှာ sequence တစ်ခု၏ နောက်ဆုံး expression ကို အကဲဖြတ်ရန်အတွက် stack ပေါ်တွင် မည်သည့်အချက်အလက်ကိုမျှ သိမ်းဆည်းခြင်းမပြုဘဲ eval-dispatch သို့ တိုက်ရိုက် လွှဲပြောင်းပေးသောကြောင့် ဖြစ်ပါသည်။ ထို့ကြောင့် sequence တစ်ခုရှိ နောက်ဆုံး expression ကို အကဲဖြတ်ခြင်းသည် procedure call တစ်ခု ဖြစ်နေလျှင်ပင် (ဥပမာ sqrt-iter တွင် procedure body ၏ နောက်ဆုံး expression ဖြစ်သော if expression သည် sqrt-iter ကို ခေါ်ဆိုမှုအဖြစ်သို့ ပြောင်းလဲသွားသည့်တိုင်အောင်) stack ပေါ်တွင် မည်သည့်အချက်အလက်မျှ စုပုံလာစေမည် မဟုတ်ပါ။311



ဤအခြေအနေတွင် အချက်အလက်များ သိမ်းဆည်းရန် မလိုအပ်ခြင်းကို အခွင့်ကောင်းမယူမိခဲ့ပါက၊ sequence ထဲရှိ expression အားလုံးကို ပုံစံတူ သတ်မှတ်ပြီး register များကို သိမ်းဆည်းခြင်း၊ expression ကို အကဲဖြတ်ခြင်း၊ register များကို ပြန်လည်ဖော်ထုတ်ရန် ပြန်လာခြင်း စသည်တို့ကို expression အားလုံး အကဲဖြတ်ပြီးသည်အထိ အကြိမ်ကြိမ် ပြုလုပ်ခြင်းဖြင့် eval-sequence ကို အကောင်အထည်ဖော်ခဲ့မိနိုင်ပါသည်။312

stack မပါသော register machine တစ်ခုဖြင့် ထိုကဲ့သို့သော process တစ်ခုကို
မည်သို့ အကောင်အထည်ဖော်ရမည်ကို 5.1 တွင် ကျွန်ုပ်တို့ တွေ့ခဲ့ပြီးဖြစ်သည်။
process ၏ အခြေအနေ (state) ကို သတ်မှတ်ထားသော register များ အစုအဝေးတစ်ခုတွင် သိမ်းဆည်းထားခဲ့သည်။




311
ev-sequence တွင် tail recursion ကို အကောင်အထည်ဖော်ထားသည့်
ဤနည်းလမ်းသည် compiler အများအပြားတွင် အသုံးပြုသော လူသိများသည့် optimization technique တစ်မျိုးဖြစ်သည်။
procedure call နှင့် အဆုံးသတ်ထားသော procedure တစ်ခုကို compile လုပ်ရာတွင်၊
ထို call ကို ခေါ်ဆိုလိုက်သော procedure ၏ entry point သို့ jump လုပ်ခြင်းဖြင့် အစားထိုးနိုင်သည်။
ဤဗျူဟာကို ကျွန်ုပ်တို့ ဤအပိုင်းတွင် ပြုလုပ်ခဲ့သကဲ့သို့ interpreter ထဲတွင် တည်ဆောက်ထည့်သွင်းခြင်းဖြင့်
ဘာသာစကားတစ်ခုလုံးတွင် optimization ကို တစ်ပြေးညီ ရရှိစေသည်။




312
ကျွန်ုပ်တို့သည် no-more-exps? ကို အောက်ပါအတိုင်း သတ်မှတ်နိုင်သည်-





(define (no-more-exps? seq) (null? seq))




313
၎င်းသည် အမှန်တကယ် လိမ်လည်ခြင်း (cheating) မဟုတ်ပါ။
အစမှအဆုံး တည်ဆောက်ထားသော လက်တွေ့အကောင်အထည်ဖော်မှုတစ်ခုတွင်၊
execution မတိုင်မီ syntax phase တစ်ခု၌ cond->if ကဲ့သို့သော source-level
transformation များကို လုပ်ဆောင်သည့် Scheme program တစ်ခုကို interpret လုပ်ရန်
ကျွန်ုပ်တို့၏ explicit-control evaluator ကို အသုံးပြုမည်ဖြစ်သည်။




314
read နှင့် အမျိုးမျိုးသော ပုံနှိပ်ထုတ်ဝေသည့် လုပ်ဆောင်ချက် (printing operations) များသည်
primitive machine operation များအဖြစ် ရရှိနိုင်သည်ဟု ကျွန်ုပ်တို့ ဤနေရာတွင် ယူဆထားပါသည်။
၎င်းသည် ကျွန်ုပ်တို့၏ simulation အတွက် အသုံးဝင်သော်လည်း လက်တွေ့တွင်မူ လုံးဝ မဖြစ်နိုင်ပေ။
၎င်းတို့သည် အမှန်တကယ်တွင် အလွန်ရှုပ်ထွေးသော လုပ်ဆောင်ချက်များ ဖြစ်သည်။ လက်တွေ့တွင်မူ ၎င်းတို့ကို
low-level input-output လုပ်ဆောင်ချက်များကို အသုံးပြု၍ အကောင်အထည်ဖော်မည် ဖြစ်သည်။



get-global-environment လုပ်ဆောင်ချက်ကို ပံ့ပိုးရန်အတွက် ကျွန်ုပ်တို့သည် အောက်ပါအတိုင်း သတ်မှတ်သည်-





(define the-global-environment
  (setup-environment))

(define (get-global-environment)
  the-global-environment)




315
interpreter မှ ကိုင်တွယ်ဖြေရှင်းစေလိုသော အခြား error များလည်း ရှိသေးသော်လည်း
၎င်းတို့သည် ထိုမျှ မရိုးရှင်းပါ။ Exercise 5.30 ကို ကြည့်ပါ။




316
stack initialization ကို error များ ဖြစ်ပေါ်ပြီးမှသာ လုပ်ဆောင်နိုင်သော်လည်း၊
driver loop တွင် လုပ်ဆောင်ခြင်းက အောက်တွင် ဖော်ပြထားသည့်အတိုင်း evaluator ၏ စွမ်းဆောင်ရည်ကို
စောင့်ကြည့်ရန်အတွက် ပိုမိုအဆင်ပြေစေမည် ဖြစ်သည်။




317
ဝမ်းနည်းစရာမှာ၊ ၎င်းသည် C ကဲ့သို့သော သမရိုးကျ compiler-based language စနစ်များတွင်
ပုံမှန်အခြေအနေပင် ဖြစ်သည်။ UNIX(tm) တွင် စနစ်သည် “dumps core” ဖြစ်သွားပြီး၊
DOS/Windows(tm) တွင်မူ တုန့်ပြန်မှု မရှိတော့ဘဲ ရပ်ဆိုင်းသွားသည်။
Macintosh(tm) တွင် ပေါက်ကွဲနေသော ဗုံးပုံတစ်ပုံကို ပြသပြီး—သင် ကံကောင်းပါက—ကွန်ပျူတာကို
reboot လုပ်ရန် အခွင့်အရေး ပေးသည်။
