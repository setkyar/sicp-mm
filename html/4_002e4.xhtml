4.4Logic Programming

အခန်း ၁ တွင် ကွန်ပျူတာသိပ္ပံသည် အမိန့်ပေးစေခိုင်းမှု (imperative - မည်သို့လုပ်ဆောင်ရမည်နည်း) ဆိုင်ရာ အသိပညာကို ကိုင်တွယ်ဖြေရှင်းပြီး၊ သင်္ချာသည် ဖော်ပြချက် (declarative - ဘာဖြစ်သည်) ဆိုင်ရာ အသိပညာကို ကိုင်တွယ်ဖြေရှင်းကြောင်း ကျွန်ုပ်တို့ အလေးပေးဖော်ပြခဲ့သည်။ အမှန်စင်စစ်၊ ပရိုဂရမ်မင်းဘာသာစကားများသည် ပရိုဂရမ်မာအနေဖြင့် ပြဿနာတစ်ခုချင်းစီကို ဖြေရှင်းရန် အဆင့်ဆင့်သော နည်းလမ်းများကို ဖော်ပြသည့် ပုံစံဖြင့် အသိပညာကို ဖော်ပြရန် လိုအပ်သည်။ အခြားတစ်ဖက်တွင်မူ၊ အဆင့်မြင့်ဘာသာစကားများသည် ဘာသာစကား၏ အကောင်အထည်ဖော်မှုအပိုင်းအဖြစ် နည်းစနစ်ကျသော အသိပညာ အမြောက်အမြားကို ထောက်ပံ့ပေးထားပြီး၊ ၎င်းသည် သတ်မှတ်ထားသော တွက်ချက်မှုတစ်ခု မည်သို့တိုးတက်သွားမည်ဆိုသည့် အသေးစိတ်အချက်အလက်များစွာကို အသုံးပြုသူမှ စိတ်ပူစရာမလိုအောင် လွတ်လပ်ခွင့်ပေးသည်။

Lisp အပါအဝင် ပရိုဂရမ်မင်းဘာသာစကားအများစုကို သင်္ချာဆိုင်ရာ ဖန်ရှင်များ၏ တန်ဖိုးများကို တွက်ချက်ခြင်းအပေါ် အခြေခံ၍ ဖွဲ့စည်းထားသည်။ Expression-oriented languages များ (Lisp, Fortran နှင့် Algol ကဲ့သို့သော) သည် ဖန်ရှင်တစ်ခု၏ တန်ဖိုးကို ဖော်ပြသည့် expression တစ်ခုကို ထိုတန်ဖိုးအား တွက်ချက်ရန် နည်းလမ်းတစ်ခုအဖြစ်လည်း အဓိပ္ပာယ်ဖွင့်ဆိုနိုင်သည်ဟူသော "ဝေါဟာရ အနက်နှစ်မျိုးထွက်မှု" (pun) ကို အသုံးချကြသည်။ ထို့ကြောင့် ပရိုဂရမ်မင်းဘာသာစကားအများစုသည် unidirectional computations (တိကျစွာသတ်မှတ်ထားသော input နှင့် output များရှိသည့် တွက်ချက်မှုများ) ဆီသို့ များစွာ တိမ်းညွှတ်နေကြသည်။ သို့သော်လည်း၊ ဤတိမ်းညွှတ်မှုကို လျှော့ချပေးသည့် လုံးဝကွဲပြားသော ပရိုဂရမ်မင်းဘာသာစကားများ ရှိပါသည်။ တွက်ချက်မှု၏ အရာဝတ္ထုများမှာ arithmetic constraints များဖြစ်ခဲ့သည့် ၃.၃.၅ တွင် ထိုကဲ့သို့သော ဥပမာတစ်ခုကို ကျွန်ုပ်တို့ တွေ့ခဲ့ရသည်။ constraint system တစ်ခုတွင် တွက်ချက်မှု၏ ဦးတည်ချက်နှင့် အစီအစဉ်ကို တိကျစွာ သတ်မှတ်ထားခြင်းမရှိပါ။ ထို့ကြောင့် တွက်ချက်မှုကို လုပ်ဆောင်ရာတွင် စနစ်သည် သာမန်သင်္ချာတွက်ချက်မှုထက် ပိုမိုအသေးစိတ်သော “မည်သို့လုပ်ဆောင်ရမည်နည်း” (how to) အသိပညာကို ပံ့ပိုးပေးရမည်ဖြစ်သည်။ သို့သော်လည်း၊ ၎င်းသည် အသုံးပြုသူအနေဖြင့် အမိန့်ပေးစေခိုင်းမှုဆိုင်ရာ အသိပညာကို ပံ့ပိုးပေးရန် တာဝန်မှ လုံးဝကင်းလွတ်သွားသည်ဟု မဆိုလိုပါ။ တူညီသော constraint အစုအဝေးကို အကောင်အထည်ဖော်သည့် constraint network များစွာရှိပြီး၊ အသုံးပြုသူသည် သင်္ချာအရ တူညီသော network များထဲမှ သတ်မှတ်ထားသော တွက်ချက်မှုတစ်ခုအတွက် သင့်လျော်သော network ကို ရွေးချယ်ရမည်ဖြစ်သည်။

၄.၃ ရှိ nondeterministic program evaluator သည်လည်း ပရိုဂရမ်ရေးသားခြင်းဆိုသည်မှာ တစ်ဖက်သတ်ဖန်ရှင်များကို တွက်ချက်ရန်အတွက် အယ်လ်ဂိုရီသမ်များကို တည်ဆောက်ခြင်းဖြစ်သည်ဟူသော အမြင်မှ ခွဲထွက်လာသည်။ nondeterministic ဘာသာစကားတစ်ခုတွင် expression များသည် တန်ဖိုးတစ်ခုထက်မက ရှိနိုင်ပြီး၊ ရလဒ်အနေဖြင့် တွက်ချက်မှုသည် တန်ဖိုးတစ်ခုတည်းရှိသော ဖန်ရှင်များထက် ဆက်နွယ်မှုများ (relations) ကို ကိုင်တွယ်ဖြေရှင်းခြင်း ဖြစ်လာသည်။ Logic programming သည် ပရိုဂရမ်ရေးသားခြင်း၏ ဆက်နွယ်မှုဆိုင်ရာ အမြင်ကို unification ဟုခေါ်သော အစွမ်းထက်သည့် symbolic pattern matching အမျိုးအစားတစ်ခုနှင့် ပေါင်းစပ်ခြင်းဖြင့် ဤစိတ်ကူးကို တိုးချဲ့ထားသည်။262

ဤချဉ်းကပ်ပုံသည် အလုပ်ဖြစ်သည့်အခါ ပရိုဂရမ်များရေးသားရန် အလွန်အစွမ်းထက်သော နည်းလမ်းတစ်ခု ဖြစ်နိုင်သည်။ ထိုအစွမ်း၏ တစ်စိတ်တစ်ပိုင်းမှာ “ဘာဖြစ်သည်” (what is) ဟူသော အချက်အလက်တစ်ခုတည်းကိုပင် မတူညီသော “မည်သို့လုပ်ဆောင်ရမည်နည်း” (how to) အစိတ်အပိုင်းများ ရှိနိုင်သည့် မတူညီသော ပြဿနာများစွာကို ဖြေရှင်းရန် အသုံးပြုနိုင်ခြင်းကြောင့် ဖြစ်သည်။ ဥပမာအားဖြင့်၊ list နှစ်ခုကို argument အဖြစ်ယူပြီး ၎င်းတို့၏ element များကို ပေါင်းစပ်ကာ list တစ်ခုတည်းအဖြစ် ဖွဲ့စည်းပေးသော append လုပ်ဆောင်ချက်ကို စဉ်းစားကြည့်ပါ။ Lisp ကဲ့သို့သော procedural ဘာသာစကားတွင်၊ ၂.၂.၁ တွင် ကျွန်ုပ်တို့ လုပ်ဆောင်ခဲ့သလို အခြေခံ list တည်ဆောက်သူ cons ကို အသုံးပြု၍ append ကို အောက်ပါအတိုင်း အဓိပ္ပာယ်ဖွင့်ဆိုနိုင်သည်- 

(define (append x y)
  (if (null? x) 
      y 
      (cons (car x) (append (cdr x) y))))

ဤလုပ်ထုံးလုပ်နည်းကို အောက်ပါစည်းမျဉ်းနှစ်ခုအား Lisp သို့ ဘာသာပြန်ခြင်းအဖြစ် မှတ်ယူနိုင်သည်။ ပထမစည်းမျဉ်းသည် ပထမ list အလွတ်ဖြစ်သည့် အခြေအနေကို လွှမ်းခြုံထားပြီး၊ ဒုတိယစည်းမျဉ်းသည် အစိတ်အပိုင်းနှစ်ခု၏ cons ဖြစ်သော အလွတ်မဟုတ်သော list အခြေအနေကို ကိုင်တွယ်သည်-

  • မည်သည့် list y အတွက်မဆို၊ အလွတ် list နှင့် y တို့ကို append လုပ်လျှင် y ရရှိသည်။
  • မည်သည့် u၊ v၊ y နှင့် z အတွက်မဆို၊ v နှင့် y တို့ကို append လုပ်၍ z ဖြစ်ပေါ်လာပါက (cons u v) နှင့် y တို့ကို append လုပ်လျှင် (cons u z) ဖြစ်ပေါ်လာသည်။263

append လုပ်ထုံးလုပ်နည်းကို အသုံးပြု၍ အောက်ပါကဲ့သို့သော မေးခွန်းများကို ဖြေကြားနိုင်သည်-

(a b) နှင့် (c d) တို့၏ append ရလဒ်ကို ရှာပါ။

သို့သော် အဆိုပါ စည်းမျဉ်းနှစ်ခုတည်းဖြင့်ပင် လုပ်ထုံးလုပ်နည်းက မဖြေကြားနိုင်သော အောက်ပါမေးခွန်းမျိုးများကိုလည်း ဖြေကြားရန် လုံလောက်သည်-

(a b) နှင့် append လုပ်လျှင် (a b c d) ရရှိမည့် list y ကို ရှာပါ။

(a b c d) ဖြစ်ပေါ်လာရန် append လုပ်ရမည့် x နှင့် y အားလုံးကို ရှာပါ။

logic programming ဘာသာစကားတစ်ခုတွင်၊ ပရိုဂရမ်မာသည် အထက်တွင်ဖော်ပြထားသော append နှင့်ပတ်သက်သည့် စည်းမျဉ်းနှစ်ခုကို ဖော်ပြခြင်းဖြင့် append “လုပ်ထုံးလုပ်နည်း” တစ်ခုကို ရေးသားသည်။ ဤစည်းမျဉ်းအတွဲတစ်ခုတည်းကို အသုံးပြု၍ append နှင့်ပတ်သက်သော မေးခွန်းအမျိုးအစား သုံးခုလုံးကို ဖြေကြားနိုင်ရန် “မည်သို့လုပ်ဆောင်ရမည်နည်း” (how to) အသိပညာကို interpreter မှ အလိုအလျောက် ပံ့ပိုးပေးပါသည်။264

မျက်မှောက်ခေတ် logic programming ဘာသာစကားများသည် (ကျွန်ုပ်တို့ ဤနေရာတွင် အကောင်အထည်ဖော်မည့် ဘာသာစကား အပါအဝင်) ၎င်းတို့၏ ယေဘုယျ “မည်သို့လုပ်ဆောင်ရမည်နည်း” (how to) နည်းလမ်းများသည် မှားယွင်းသော အဆုံးမရှိလည်ပတ်မှုများ (infinite loops) သို့မဟုတ် အခြားမလိုလားအပ်သော အပြုအမူများဆီသို့ ဦးတည်သွားစေနိုင်သည့် အားနည်းချက်များစွာ ရှိနေပါသည်။ Logic programming သည် ကွန်ပျူတာသိပ္ပံနယ်ပယ်တွင် တက်ကြွစွာ သုတေသနပြုနေဆဲ နယ်ပယ်တစ်ခုဖြစ်သည်။265

ဤအခန်း၏ အစောပိုင်းတွင် ကျွန်ုပ်တို့သည် interpreter များကို အကောင်အထည်ဖော်ခြင်းဆိုင်ရာ နည်းပညာကို လေ့လာခဲ့ကြပြီး Lisp ကဲ့သို့သော ဘာသာစကားတစ်ခုအတွက် (အမှန်စင်စစ် မည်သည့် သာမန်ဘာသာစကားအတွက်မဆို) interpreter တစ်ခုအတွက် မရှိမဖြစ်လိုအပ်သော ဒြပ်စင်များကို ဖော်ပြခဲ့သည်။ ယခု ကျွန်ုပ်တို့သည် logic programming ဘာသာစကားတစ်ခုအတွက် interpreter တစ်ခုအကြောင်း ဆွေးနွေးရန် ဤစိတ်ကူးများကို အသုံးချပါမည်။ ဤဘာသာစကားကို query language ဟု ခေါ်ဆိုပါသည်။ အဘယ်ကြောင့်ဆိုသော် ၎င်းသည် ဤဘာသာစကားဖြင့် ဖော်ပြထားသော queries များ သို့မဟုတ် မေးခွန်းများကို ဖော်ထုတ်ခြင်းဖြင့် ဒေတာဘေ့စ်များမှ အချက်အလက်များကို ထုတ်ယူရာတွင် အလွန်အသုံးဝင်သောကြောင့် ဖြစ်သည်။ query language သည် Lisp နှင့် အလွန်ကွဲပြားသော်လည်း၊ ကျွန်ုပ်တို့ အစဉ်တစိုက် အသုံးပြုလာခဲ့သည့် ယေဘုယျ မူဘောင်အတိုင်းပင် ဤဘာသာစကားကို ဖော်ပြရန် အဆင်ပြေသည်ကို တွေ့ရပါလိမ့်မည်- ၎င်းမှာ အခြေခံဒြပ်စင်များ (primitive elements) စုစည်းမှု၊ ရိုးရှင်းသော ဒြပ်စင်များကို ပေါင်းစပ်ပြီး ပိုမိုရှုပ်ထွေးသော ဒြပ်စင်များ ဖန်တီးနိုင်စေသည့် ပေါင်းစပ်မှုနည်းလမ်းများ (means of combination) နှင့် ရှုပ်ထွေးသော ဒြပ်စင်များကို တစ်ခုတည်းသော အယူအဆဆိုင်ရာ ယူနစ်များအဖြစ် မှတ်ယူနိုင်စေသည့် အနှစ်ထုတ်မှုနည်းလမ်းများ (means of abstraction) အဖြစ် ဖြစ်သည်။ logic programming ဘာသာစကားတစ်ခုအတွက် interpreter သည် Lisp ကဲ့သို့သော ဘာသာစကားတစ်ခုအတွက် interpreter ထက် သိသိသာသာ ပိုမိုရှုပ်ထွေးပါသည်။ သို့သော်လည်း၊ ကျွန်ုပ်တို့၏ query-language interpreter တွင် အပိုင်း ၄.၁ ရှိ interpreter တွင် တွေ့ရသည့် အစိတ်အပိုင်း အများအပြား ပါဝင်နေသည်ကို တွေ့ရပါမည်။ အထူးသဖြင့်၊ expression များကို အမျိုးအစားအလိုက် ခွဲခြားပေးသည့် “eval” အပိုင်းနှင့် ဘာသာစကား၏ abstraction ယန္တရားကို အကောင်အထည်ဖော်သည့် “apply” အပိုင်း (Lisp တွင် procedures များဖြစ်ပြီး၊ logic programming တွင် rules များဖြစ်သည်) တို့ ပါဝင်မည်ဖြစ်သည်။ ထို့ပြင် သင်္ကေတများနှင့် ၎င်းတို့နှင့် သက်ဆိုင်သော တန်ဖိုးများအကြား ဆက်နွယ်မှုကို ဆုံးဖြတ်ပေးသည့် frame data structure သည်လည်း အကောင်အထည်ဖော်မှုတွင် အဓိကအခန်းကဏ္ဍမှ ပါဝင်ပါသည်။ ကျွန်ုပ်တို့၏ query-language အကောင်အထည်ဖော်မှု၏ နောက်ထပ် စိတ်ဝင်စားစရာကောင်းသော အချက်တစ်ခုမှာ အခန်း ၃ တွင် မိတ်ဆက်ခဲ့သော streams များကို အကြီးအကျယ် အသုံးပြုထားခြင်း ဖြစ်သည်။

4.4.1ဆင်ခြင်တွေးခေါ်မှုဖြင့် အချက်အလက် ရှာဖွေထုတ်ယူခြင်း (Deductive Information Retrieval)

Logic programming သည် အချက်အလက် ရှာဖွေထုတ်ယူခြင်း (information retrieval) အတွက် data base များသို့ interface များ ပံ့ပိုးပေးရာတွင် အလွန်ထူးချွန်ပါသည်။ ဤအခန်းတွင် ကျွန်ုပ်တို့ အကောင်အထည်ဖော်မည့် query language ကို ဤနည်းလမ်းဖြင့် အသုံးပြုရန် ဒီဇိုင်းထုတ်ထားခြင်း ဖြစ်ပါသည်။

query system က မည်သို့လုပ်ဆောင်သည်ကို သရုပ်ဖော်ရန်အတွက်၊ Boston ဧရိယာရှိ အောင်မြင်ဖြစ်ထွန်းနေသော နည်းပညာမြင့်ကုမ္ပဏီတစ်ခုဖြစ်သည့် Microshaft ၏ ဝန်ထမ်းမှတ်တမ်း (personnel records) data base ကို စီမံခန့်ခွဲရာတွင် ၎င်းကို မည်သို့အသုံးပြုနိုင်သည်ကို ကျွန်ုပ်တို့ ပြသပါမည်။ ဤဘာသာစကားသည် ဝန်ထမ်းအချက်အလက်များကို pattern-directed access (ပုံစံကို အခြေခံ၍ ဝင်ရောက်ကြည့်ရှုခြင်း) ပံ့ပိုးပေးပြီး ယုတ္တိဗေဒဆိုင်ရာ ကောက်ချက်ချမှုများ (logical deductions) ပြုလုပ်နိုင်ရန် ယေဘုယျ စည်းမျှဉ်းများကိုလည်း အသုံးချနိုင်ပါသည်။

Query language ၏ ရိုးရှင်းသော query များ လုပ်ဆောင်ပုံကို အောက်ပါအတိုင်း ဖော်ပြနိုင်သည်-

  • စနစ်သည် query pattern ကို ကျေနပ်စေသော (satisfy) variable များအတွက် တန်ဖိုးသတ်မှတ်ချက်အားလုံးကို ရှာဖွေပေးသည်—ဆိုလိုသည်မှာ variable များအတွက် တန်ဖိုးများအားလုံးကို ရှာဖွေရာတွင်၊ အကယ်၍ pattern variable များကို ထိုတန်ဖိုးများဖြင့် အစားထိုးလိုက်လျှင် (instantiated with) ရလဒ်သည် data base ထဲတွင် ရှိနေရမည် ဖြစ်သည်။
  • စနစ်သည် query pattern ကို ကျေနပ်စေသော variable တန်ဖိုးများဖြင့် အစားထိုးထားသည့် instantiation အားလုံးကို စာရင်းပြုစုပြီး query ကို တုံ့ပြန်သည်။

အကယ်၍ pattern တွင် variable များ မပါဝင်ပါက၊ query သည် ထို pattern သည် data base ထဲတွင် ရှိမရှိ ဆုံးဖြတ်ခြင်းသို့ လျော့ကျသွားမည် ဖြစ်သည်။ အကယ်၍ ရှိခဲ့လျှင်၊ variable များသို့ တန်ဖိုးသတ်မှတ်ခြင်းမရှိသော ဗလာတန်ဖိုးသတ်မှတ်ချက် (empty assignment) သည် ထို data base အတွက် ထို pattern ကို ကျေနပ်စေသည်။

Compound queries

ရိုးရှင်းသော query များသည် query language ၏ primitive operation များ ဖြစ်ကြသည်။ Compound operation များ ဖန်တီးနိုင်ရန်အတွက် query language က ပေါင်းစပ်မှုနည်းလမ်းများ (means of combination) ကို ပံ့ပိုးပေးထားသည်။ Query language ကို logic programming language တစ်ခု ဖြစ်စေသည့် အချက်တစ်ခုမှာ ပေါင်းစပ်မှုနည်းလမ်းများသည် logical expression များ ဖန်တီးရာတွင် အသုံးပြုသည့် and, or, နှင့် not တို့ကို တုပထားခြင်းကြောင့် ဖြစ်သည်။ (ဤနေရာတွင် and, or, နှင့် not တို့သည် Lisp primitive များ မဟုတ်ဘဲ၊ query language တွင် ထည့်သွင်းတည်ဆောက်ထားသော operation များသာ ဖြစ်သည်။)

Rules

Primitive query များနှင့် compound query များအပြင်၊ query language သည် query များကို abstract လုပ်နိုင်မည့် နည်းလမ်းများကိုလည်း ပံ့ပိုးပေးထားသည်။ ၎င်းတို့ကို rules များဖြင့် ပေးထားသည်။ Rule တစ်ခုမှာ 

(rule (lives-near ?person-1 ?person-2)
      (and (address ?person-1 
                    (?town . ?rest-1))
           (address ?person-2 
                    (?town . ?rest-2))
           (not (same ?person-1 ?person-2))))

4.4.2ကွာရီစနစ် အလုပ်လုပ်ပုံ (How the Query System Works)

အပိုင်း 4.4.4 တွင် ကွာရီ အင်တာပရီတာ (query interpreter) ကို လုပ်ထုံးလုပ်နည်းများ စုစည်းမှုအဖြစ် အကောင်အထည်ဖော်ပုံကို တင်ပြပါမည်။ ဤအပိုင်းတွင်မူ အနိမ့်ပိုင်း အကောင်အထည်ဖော်မှု အသေးစိတ်များနှင့် မသက်ဆိုင်ဘဲ စနစ်၏ ယေဘုယျ တည်ဆောက်ပုံကို ရှင်းပြထားသည့် ခြုံငုံသုံးသပ်ချက်ကို ပေးပါမည်။ အင်တာပရီတာ၏ အကောင်အထည်ဖော်မှုကို ဖော်ပြပြီးနောက်၊ ၎င်း၏ ကန့်သတ်ချက်အချို့နှင့် ကွာရီဘာသာစကား၏ လော့ဂျစ်လုပ်ဆောင်ချက်များသည် သင်္ချာလော့ဂျစ် (mathematical logic) ၏ လုပ်ဆောင်ချက်များနှင့် ကွာခြားသည့် သိမ်မွေ့သော နည်းလမ်းအချို့ကို ကျွန်ုပ်တို့ နားလည်နိုင်မည် ဖြစ်ပါသည်။

ဒေတာဘေ့စ်ရှိ အချက်အလက်များနှင့် စည်းမျဉ်းများ (rules) ကို ကွာရီများနှင့် တိုက်ဆိုင်စစ်ဆေးရန်အတွက် ကွာရီ အကဲဖြတ်စနစ် (query evaluator) သည် တစ်မျိုးမျိုးသော ရှာဖွေမှုကို လုပ်ဆောင်ရမည်မှာ သိသာထင်ရှားပါသည်။ ဤအရာကို လုပ်ဆောင်ရန် နည်းလမ်းတစ်ခုမှာ 4.3 မှ amb အကဲဖြတ်စနစ်ကို အသုံးပြု၍ ကွာရီစနစ်ကို nondeterministic ပရိုဂရမ်တစ်ခုအဖြစ် အကောင်အထည်ဖော်ခြင်း ဖြစ်သည် (လေ့ကျင့်ခန်း 4.78 ကို ကြည့်ပါ)။ အခြားဖြစ်နိုင်ခြေတစ်ခုမှာ စထရင်းများ (streams) ၏ အကူအညီဖြင့် ရှာဖွေမှုကို စီမံခန့်ခွဲခြင်း ဖြစ်သည်။ ကျွန်ုပ်တို့၏ အကောင်အထည်ဖော်မှုသည် ဤဒုတိယချဉ်းကပ်မှုကို လိုက်နာပါသည်။

ကွာရီစနစ်ကို ပက်တန် တိုက်ဆိုင်စစ်ဆေးခြင်း (pattern matching) နှင့် ယူနီဖီကေးရှင်း (unification) ဟုခေါ်သော ဗဟိုလုပ်ဆောင်ချက် နှစ်ခုဖြင့် ဖွဲ့စည်းထားပါသည်။ ဦးစွာ ပက်တန် တိုက်ဆိုင်စစ်ဆေးခြင်းအကြောင်းကို ဖော်ပြပြီး ဤလုပ်ဆောင်ချက်သည် ဖရိမ်စထရင်းများ (streams of frames) နှင့် ပေါင်းစပ်ကာ ရိုးရှင်းသော ကွာရီများနှင့် ပေါင်းစပ်ကွာရီများ နှစ်မျိုးလုံးကို မည်သို့ အကောင်အထည်ဖော်နိုင်စေကြောင်း ရှင်းပြပါမည်။ ထို့နောက် စည်းမျဉ်းများကို အကောင်အထည်ဖော်ရန် လိုအပ်သော ပက်တန် တိုက်ဆိုင်စစ်ဆေးခြင်း၏ ယေဘုယျပုံစံဖြစ်သော ယူနီဖီကေးရှင်းအကြောင်း ဆွေးနွေးပါမည်။ နောက်ဆုံးတွင်၊ အပိုင်း 4.1 ၌ ဖော်ပြထားသော အင်တာပရီတာအတွက် eval က အသုံးအနှုန်းများကို အမျိုးအစားခွဲသကဲ့သို့ပင်၊ အသုံးအနှုန်းများကို အမျိုးအစားခွဲသည့် လုပ်ထုံးလုပ်နည်းတစ်ခုမှတစ်ဆင့် ကွာရီ အင်တာပရီတာ တစ်ခုလုံး မည်သို့ အံဝင်ခွင်ကျဖြစ်သွားသည်ကို ပြသပါမည်။

ပက်တန် တိုက်ဆိုင်စစ်ဆေးခြင်း (Pattern matching)

ပက်တန် မက်ချာ (pattern matcher) ဆိုသည်မှာ ဒေတာတစ်ခုသည် သတ်မှတ်ထားသော ပက်တန်တစ်ခုနှင့် ကိုက်ညီမှု ရှိမရှိ စစ်ဆေးသည့် ပရိုဂရမ်တစ်ခု ဖြစ်သည်။ ဥပမာအားဖြင့်၊ ဒေတာစာရင်း ((a b) c (a b)) သည် ပက်တန် (?x c ?x) နှင့် ကိုက်ညီပြီး ပက်တန်ကိန်းရှင် (pattern variable) ?x သည် (a b) နှင့် ချိတ်ဆက် (bound) နေပါသည်။ ထိုဒေတာစာရင်းသည် ပက်တန် (?x ?y ?z) နှင့်လည်း ကိုက်ညီပြီး ?x နှင့် ?z နှစ်ခုလုံးသည် (a b) နှင့် ချိတ်ဆက်ကာ ?y သည် c နှင့် ချိတ်ဆက်နေပါသည်။ ၎င်းသည် ပက်တန် ((?x ?y) c (?x ?y)) နှင့်လည်း ကိုက်ညီပြီး ?x သည် a နှင့် ?y သည် b နှင့် ချိတ်ဆက်နေပါသည်။ သို့သော်၊ ၎င်းသည် ပက်တန် (?x a ?y) နှင့် မကိုက်ညီပါ၊ အကြောင်းမှာ ထိုပက်တန်သည် ဒုတိယမြောက် အစိတ်အပိုင်းအဖြစ် သင်္ကေတ a ပါရှိသော စာရင်းတစ်ခု ဖြစ်ရမည်ဟု သတ်မှတ်ထားသောကြောင့် ဖြစ်သည်။

ကွာရီစနစ်တွင် အသုံးပြုသော ပက်တန် မက်ချာသည် ထည့်သွင်းချက် (input) အဖြစ် ပက်တန်တစ်ခု၊ ဒေတာတစ်ခု၊ နှင့် အမျိုးမျိုးသော ပက်တန်ကိန်းရှင်များအတွက် ချိတ်ဆက်မှုများကို သတ်မှတ်ပေးသည့် ဖရိမ် (frame) တစ်ခုတို့ကို ယူပါသည်။ ၎င်းသည် ဖရိမ်ထဲတွင် ရှိနှင့်ပြီးသား ချိတ်ဆက်မှုများနှင့် ကိုက်ညီသော နည်းလမ်းဖြင့် ဒေတာသည် ပက်တန်နှင့် ကိုက်ညီမှု ရှိမရှိ စစ်ဆေးပါသည်။ အကယ်၍ ကိုက်ညီပါက၊ တိုက်ဆိုင်စစ်ဆေးမှုမှ ဆုံးဖြတ်လိုက်သော ချိတ်ဆက်မှုအသစ်များ ဖြည့်စွက်ထားသည့် ပေးထားသော ဖရိမ်ကို ပြန်ပေးပါသည်။ မဟုတ်ပါက၊ တိုက်ဆိုင်စစ်ဆေးမှု ရှုံးနိမ့်ကြောင်း ညွှန်ပြပါသည်။

ဥပမာအားဖြင့်၊ ဗလာဖရိမ် (empty frame) တစ်ခုပေးထားပြီး ပက်တန် (?x ?y ?x) ကို အသုံးပြု၍ (a b a) နှင့် တိုက်ဆိုင်စစ်ဆေးပါက ?x ကို a နှင့် ချိတ်ဆက်ပြီး ?y ကို b နှင့် ချိတ်ဆက်ထားသော ဖရိမ်တစ်ခုကို ပြန်ပေးပါမည်။ တူညီသော ပက်တန်၊ တူညီသော ဒေတာနှင့် ?y ကို a နှင့် ချိတ်ဆက်ထားသော ဖရိမ်တစ်ခုဖြင့် တိုက်ဆိုင်စစ်ဆေးရန် ကြိုးပမ်းပါက ရှုံးနိမ့်ပါမည်။ တူညီသော ပက်တန်၊ တူညီသော ဒေတာနှင့် ?y ကို b နှင့် ချိတ်ဆက်ထားပြီး ?x သည် ချိတ်ဆက်မှုမရှိသေးသော ဖရိမ်တစ်ခုဖြင့် စမ်းသပ်ပါက ?x ကို a နှင့် ချိတ်ဆက်မှု ဖြည့်စွက်ထားသော ပေးထားသည့် ဖရိမ်ကို ပြန်ပေးပါမည်။

ပက်တန် မက်ချာသည် စည်းမျဉ်းများ (rules) မပါဝင်သော ရိုးရှင်းသော ကွာရီများကို လုပ်ဆောင်ရန် လိုအပ်သည့် ယန္တရားအားလုံး ဖြစ်ပါသည်။ ဥပမာအားဖြင့်၊ အောက်ပါ ကွာရီကို လုပ်ဆောင်ရန်

ကျွန်ုပ်တို့သည် ဒေတာဘေ့စ်ရှိ အခိုင်အမာဖော်ပြချက် (assertions) အားလုံးကို ရှာဖွေပြီး အစောပိုင်းက ဗလာဖြစ်နေသော ဖရိမ်တစ်ခုနှင့် ပတ်သက်၍ ပက်တန်နှင့် ကိုက်ညီသော အရာများကို ရွေးချယ်ပါသည်။ တွေ့ရှိရသော တိုက်ဆိုင်မှုတစ်ခုစီအတွက်၊ ?x အတွက် တန်ဖိုးတစ်ခုဖြင့် ပက်တန်ကို အစားထိုးရန် တိုက်ဆိုင်စစ်ဆေးမှုမှ ပြန်ပေးသော ဖရိမ်ကို အသုံးပြုပါသည်။

ဖရိမ် စထရင်းများ (Streams of frames)

ဖရိမ်များနှင့် ပက်တန်များကို စမ်းသပ်ခြင်းကို စထရင်းများကို အသုံးပြု၍ ဖွဲ့စည်းထားပါသည်။ ဖရိမ်တစ်ခုတည်း ပေးထားလျှင်၊ တိုက်ဆိုင်စစ်ဆေးမှု လုပ်ငန်းစဉ်သည် ဒေတာဘေ့စ်ရှိ အချက်အလက်များကို တစ်ခုပြီးတစ်ခု စစ်ဆေးသွားပါသည်။ ဒေတာဘေ့စ် အချက်အလက်တစ်ခုစီအတွက်၊ မက်ချာသည် တိုက်ဆိုင်မှုရှုံးနိမ့်ကြောင်း ပြသသည့် အထူးသင်္ကေတတစ်ခု သို့မဟုတ် ဖရိမ်၏ တိုးချဲ့မှုတစ်ခုကို ထုတ်ပေးပါသည်။ ဒေတာဘေ့စ် အချက်အလက်အားလုံးအတွက် ရလဒ်များကို စထရင်းတစ်ခုထဲသို့ စုစည်းပြီး၊ ရှုံးနိမ့်မှုများကို ဖယ်ရှားရန် စစ်ထုတ်ကိရိယာ (filter) တစ်ခုမှတစ်ဆင့် ဖြတ်သန်းစေပါသည်။ ရလဒ်မှာ ဒေတာဘေ့စ်ရှိ အချို့သော အခိုင်အမာဖော်ပြချက်များနှင့် တိုက်ဆိုင်စစ်ဆေးမှုမှတစ်ဆင့် ပေးထားသော ဖရိမ်ကို တိုးချဲ့ထားသည့် ဖရိမ်အားလုံး၏ စထရင်းတစ်ခု ဖြစ်လာပါသည်။271

ကျွန်ုပ်တို့၏ စနစ်တွင်၊ ကွာရီတစ်ခုသည် ဖရိမ်များ၏ ထည့်သွင်းစထရင်း (input stream) တစ်ခုကို ယူပြီး စထရင်းရှိ ဖရိမ်တိုင်းအတွက် အထက်ပါ တိုက်ဆိုင်စစ်ဆေးမှု လုပ်ဆောင်ချက်ကို လုပ်ဆောင်ပါသည်၊ ၎င်းကို ပုံ 4.4 တွင် ပြထားပါသည်။ ဆိုလိုသည်မှာ၊ ထည့်သွင်းစထရင်းရှိ ဖရိမ်တစ်ခုစီအတွက်၊ ကွာရီသည် ဒေတာဘေ့စ်ရှိ အခိုင်အမာဖော်ပြချက်များနှင့် တိုက်ဆိုင်စစ်ဆေးခြင်းဖြင့် ရရှိသော ထိုဖရိမ်၏ တိုးချဲ့မှုအားလုံး ပါဝင်သော စထရင်းအသစ်တစ်ခုကို ထုတ်ပေးပါသည်။ ထို့နောက် ဤစထရင်းအားလုံးကို ပေါင်းစပ်၍ ထည့်သွင်းစထရင်းရှိ ဖရိမ်တိုင်း၏ ဖြစ်နိုင်သမျှ တိုးချဲ့မှုအားလုံး ပါဝင်သော ဧရာမ စထရင်းကြီးတစ်ခုအဖြစ် ဖွဲ့စည်းပါသည်။ ဤစထရင်းသည် ကွာရီ၏ ထွက်ပေါ်လာသော ရလဒ် (output) ဖြစ်ပါသည်။

ရိုးရှင်းသော ကွာရီတစ်ခုကို ဖြေကြားရန်၊ ကျွန်ုပ်တို့သည် ဗလာဖရိမ်တစ်ခုတည်း ပါဝင်သော ထည့်သွင်းစထရင်းဖြင့် ကွာရီကို အသုံးပြုပါသည်။ ရရှိလာသော ထွက်ပေါ်လာသည့် စထရင်းတွင် ဗလာဖရိမ်၏ တိုးချဲ့မှုအားလုံး (ဆိုလိုသည်မှာ ကျွန်ုပ်တို့၏ ကွာရီအတွက် အဖြေအားလုံး) ပါဝင်ပါသည်။ ထို့နောက် ဤဖရိမ်စထရင်းကို အသုံးပြု၍ ဖရိမ်တစ်ခုစီရှိ တန်ဖိုးများဖြင့် ကိန်းရှင်များကို အစားထိုးထားသော မူရင်းကွာရီပက်တန်၏ မိတ္တူစထရင်းတစ်ခုကို ထုတ်ပေးပြီး၊ ၎င်းသည် နောက်ဆုံးတွင် ပုံနှိပ်ထုတ်ပေးသော စထရင်း ဖြစ်ပါသည်။

ပေါင်းစပ်ကွာရီများ (Compound queries)

ပုံ 4.6 တွင် ကွာရီနှစ်ခု၏ or ကို တွက်ချက်ရန်အတွက် အစိတ်အပိုင်း ကွာရီနှစ်ခုကို ပြိုင်လျက်ပေါင်းစပ်မှု (parallel combination) အဖြစ် လုပ်ဆောင်သော အလားတူနည်းလမ်းကို ပြသထားပါသည်။ ဖရိမ်များ၏ ထည့်သွင်းစထရင်းကို ကွာရီတစ်ခုစီက သီးခြားစီ တိုးချဲ့ပါသည်။ ထို့နောက် ထွက်ပေါ်လာသော စထရင်းနှစ်ခုကို ပေါင်းစပ် (merge) ၍ နောက်ဆုံးရလဒ် စထရင်းကို ထုတ်ပေးပါသည်။

ဤအဆင့်မြင့် ဖော်ပြချက်အရပင် ပေါင်းစပ်ကွာရီများကို လုပ်ဆောင်ခြင်းသည် နှေးကွေးနိုင်ကြောင်း သိသာပါသည်။ ဥပမာအားဖြင့်၊ ကွာရီတစ်ခုသည် ထည့်သွင်းဖရိမ်တစ်ခုစီအတွက် ထွက်ပေါ်လာသော ဖရိမ်တစ်ခုထက်ပို၍ ထုတ်ပေးနိုင်ပြီး၊ and ထဲရှိ ကွာရီတစ်ခုစီသည် ၎င်း၏ ထည့်သွင်းဖရိမ်များကို ယခင်ကွာရီမှ ရရှိသောကြောင့်၊ အဆိုးရွားဆုံး အခြေအနေတွင် and ကွာရီတစ်ခုသည် ကွာရီအရေအတွက်အလိုက် ထပ်ကိန်းအလိုက် (exponential) တိုးပွားသော တိုက်ဆိုင်စစ်ဆေးမှု အရေအတွက်ကို လုပ်ဆောင်ရနိုင်ပါသည် (လေ့ကျင့်ခန်း 4.76 ကို ကြည့်ပါ)။272 ရိုးရှင်းသော ကွာရီများကိုသာ ကိုင်တွယ်သည့် စနစ်များသည် အလွန်လက်တွေ့ကျသော်လည်း၊ ရှုပ်ထွေးသော ကွာရီများကို ကိုင်တွယ်ရန်မှာ အလွန်ခက်ခဲပါသည်။273

lisp-value အထူးပုံစံ (special form) ကို ဖရိမ်စထရင်းများပေါ်တွင် အလားတူ စစ်ထုတ်ကိရိယာအဖြစ် အကောင်အထည်ဖော်ထားပါသည်။ ကျွန်ုပ်တို့သည် စထရင်းရှိ ဖရိမ်တစ်ခုစီကို အသုံးပြု၍ ပက်တန်ရှိ ကိန်းရှင်များကို အစားထိုးပြီးနောက် Lisp ပရက်ဒီကိတ် (predicate) ကို အသုံးချပါသည်။ ပရက်ဒီကိတ် ရှုံးနိမ့်သော ဖရိမ်အားလုံးကို ထည့်သွင်းစထရင်းမှ ဖယ်ရှားပါသည်။

ယူနီဖီကေးရှင်း (Unification)

ကွာရီဘာသာစကားတွင် စည်းမျဉ်းများကို ကိုင်တွယ်ရန်အတွက်၊ ပေးထားသော ကွာရီပက်တန်နှင့် ကိုက်ညီသော နိဂုံးချုပ်ချက် (conclusions) ပါရှိသည့် စည်းမျဉ်းများကို ရှာဖွေနိုင်ရပါမည်။ စည်းမျဉ်း နိဂုံးချုပ်ချက်များသည် ကိန်းရှင်များ ပါဝင်နိုင်သည်မှလွဲ၍ အခိုင်အမာဖော်ပြချက်များ (assertions) နှင့် တူညီကြသည်၊ ထို့ကြောင့် ကျွန်ုပ်တို့သည် "ပက်တန်" နှင့် "ဒေတာ" နှစ်ခုလုံးတွင် ကိန်းရှင်များ ပါဝင်နိုင်သည့် ပက်တန် တိုက်ဆိုင်စစ်ဆေးခြင်း၏ ယေဘုယျပုံစံ — ယူနီဖီကေးရှင်း (unification) — လိုအပ်ပါလိမ့်မည်။

ယူနီဖိုင်ယာ (unifier) သည် ကိန်းသေများနှင့် ကိန်းရှင်များ ပါဝင်သော ပက်တန်နှစ်ခုကို ယူပြီး၊ ထိုပက်တန်နှစ်ခုကို တူညီသွားစေရန် ကိန်းရှင်များထံ တန်ဖိုးများ သတ်မှတ်ပေးရန် ဖြစ်နိုင်မှု ရှိမရှိ ဆုံးဖြတ်ပါသည်။ အကယ်၍ ဖြစ်နိုင်ပါက၊ ဤချိတ်ဆက်မှုများ ပါဝင်သော ဖရိမ်တစ်ခုကို ပြန်ပေးပါသည်။ ဥပမာအားဖြင့်၊ (?x a ?y) နှင့် (?y ?z a) ကို ယူနီဖိုင်းလုပ်ခြင်း (unifying) သည် ?x၊ ?y၊ နှင့် ?z အားလုံးကို a နှင့် ချိတ်ဆက်ရမည့် ဖရိမ်တစ်ခုကို သတ်မှတ်ပေးပါလိမ့်မည်။ အခြားတစ်ဖက်တွင်၊ (?x ?y a) နှင့် (?x b ?y) ကို ယူနီဖိုင်းလုပ်ခြင်းသည် ရှုံးနိမ့်ပါလိမ့်မည်၊ အကြောင်းမှာ ပက်တန်နှစ်ခုကို တူညီစေမည့် ?y အတွက် တန်ဖိုးမရှိသောကြောင့် ဖြစ်သည်။ (ပက်တန်များ၏ ဒုတိယမြောက် အစိတ်အပိုင်းများ တူညီရန်အတွက် ?y သည် b ဖြစ်ရမည်၊ သို့သော် တတိယမြောက် အစိတ်အပိုင်းများ တူညီရန်အတွက်မူ ?y သည် a ဖြစ်ရမည်။) ကွာရီစနစ်တွင် အသုံးပြုသော ယူနီဖိုင်ယာသည် ပက်တန် မက်ချာကဲ့သို့ပင် ဖရိမ်တစ်ခုကို ထည့်သွင်းချက်အဖြစ် ယူပြီး ဤဖရိမ်နှင့် ကိုက်ညီသော ယူနီဖီကေးရှင်းများကို လုပ်ဆောင်ပါသည်။

ယူနီဖီကေးရှင်း အယ်လ်ဂိုရီသမ် (unification algorithm) သည် ကွာရီစနစ်၏ နည်းပညာအရ အခက်ခဲဆုံး အစိတ်အပိုင်း ဖြစ်သည်။ ရှုပ်ထွေးသော ပက်တန်များဖြင့် ယူနီဖီကေးရှင်းကို လုပ်ဆောင်ခြင်းသည် ဆင်ခြင်တုံတရားဖြင့် ကောက်ချက်ချမှု (deduction) လိုအပ်ပုံပေါ်နိုင်ပါသည်။ ဥပမာအားဖြင့် (?x ?x) နှင့် ((a ?y c) (a b ?z)) ကို ယူနီဖိုင်းလုပ်ရန်၊ အယ်လ်ဂိုရီသမ်သည် ?x သည် (a b c) ဖြစ်သင့်ကြောင်း၊ ?y သည် b ဖြစ်သင့်ကြောင်း၊ နှင့် ?z သည် c ဖြစ်သင့်ကြောင်း ကောက်ချက်ချရမည်။ ဤလုပ်ငန်းစဉ်ကို ပက်တန်အစိတ်အပိုင်းများအကြား ညီမျှခြင်းအစုအဝေး (set of equations) တစ်ခုကို ဖြေရှင်းခြင်းအဖြစ် ကျွန်ုပ်တို့ ယူဆနိုင်ပါသည်။ ယေဘုယျအားဖြင့်၊ ၎င်းတို့သည် ဖြေရှင်းရန်အတွက် သိသာထင်ရှားသော ကိုင်တွယ်မှုများ လိုအပ်နိုင်သည့် တစ်ပြိုင်နက်ဖြစ်သော ညီမျှခြင်းများ (simultaneous equations) ဖြစ်ကြသည်။275 ဥပမာအားဖြင့်၊ (?x ?x) နှင့် ((a ?y c) (a b ?z)) ကို ယူနီဖိုင်းလုပ်ခြင်းအား အောက်ပါ တစ်ပြိုင်နက်ဖြစ်သော ညီမျှခြင်းများကို သတ်မှတ်ခြင်းအဖြစ် ယူဆနိုင်သည်

စည်းမျဉ်းများ အသုံးချခြင်း (Applying rules)

ယူနီဖီကေးရှင်းသည် စည်းမျဉ်းများမှ ကောက်ချက်များထုတ်ပေးသော ကွာရီစနစ်၏ အစိတ်အပိုင်းအတွက် အဓိကသော့ချက် ဖြစ်ပါသည်။ ဤအရာကို မည်သို့ လုပ်ဆောင်သည်ကို သိမြင်ရန်၊ စည်းမျဉ်းတစ်ခုကို အသုံးချရန် လိုအပ်သော ကွာရီတစ်ခုကို လုပ်ဆောင်ခြင်းကို စဉ်းစားကြည့်ပါ၊ ဥပမာ 

(lives-near ?x (Hacker Alyssa P))

ဤကွာရီကို လုပ်ဆောင်ရန်၊ ကျွန်ုပ်တို့သည် ဤပက်တန်နှင့် ကိုက်ညီသော ဒေတာဘေ့စ်ရှိ အခိုင်အမာဖော်ပြချက်များ ရှိမရှိ သိရန် အထက်တွင် ဖော်ပြထားသော သာမန် ပက်တန်-မက်ချ် လုပ်ထုံးလုပ်နည်းကို ဦးစွာ အသုံးပြုပါသည်။ (ဤကိစ္စတွင် မည်သည့်အရာမျှ ရှိမည်မဟုတ်ပါ၊ အကြောင်းမှာ ကျွန်ုပ်တို့၏ ဒေတာဘေ့စ်တွင် မည်သူက မည်သူနှင့် နီးကပ်စွာ နေထိုင်သည်ဟူသော တိုက်ရိုက် အခိုင်အမာဖော်ပြချက်များ မပါဝင်သောကြောင့် ဖြစ်သည်။) နောက်တစ်ဆင့်မှာ ကွာရီပက်တန်ကို စည်းမျဉ်းတစ်ခုစီ၏ နိဂုံးချုပ်ချက်နှင့် ယူနီဖိုင်းလုပ်ရန် ကြိုးပမ်းခြင်း ဖြစ်သည်။ ပက်တန်သည် အောက်ပါ စည်းမျဉ်း၏ နိဂုံးချုပ်ချက်နှင့် ယူနီဖိုင်းဖြစ်သည်ကို ကျွန်ုပ်တို့ တွေ့ရှိရပါသည် 

(rule (lives-near ?person-1 ?person-2)
      (and (address ?person-1 
                    (?town . ?rest-1))
           (address ?person-2 
                    (?town . ?rest-2))
           (not (same ?person-1 ?person-2))))

ရလဒ်အနေဖြင့် ?person-2 ကို (Hacker Alyssa P) နှင့် ချိတ်ဆက်ထားပြီး ?x ကို ?person-1 နှင့် ချိတ်ဆက်ရမည် (တန်ဖိုးတူညီရမည်) ဟု သတ်မှတ်ထားသော ဖရိမ်တစ်ခု ရရှိလာပါသည်။ ယခု၊ ဤဖရိမ်နှင့် ဆက်စပ်၍ စည်းမျဉ်း၏ ကိုယ်ထည် (body) မှ ပေးထားသော ပေါင်းစပ်ကွာရီကို အကဲဖြတ်ပါသည်။ အောင်မြင်သော တိုက်ဆိုင်မှုများသည် ?person-1 အတွက် ချိတ်ဆက်မှုတစ်ခု ပေးခြင်းဖြင့် ဤဖရိမ်ကို တိုးချဲ့ပါလိမ့်မည်၊ ထို့ကြောင့် ?x အတွက် တန်ဖိုးတစ်ခု ရရှိလာမည် ဖြစ်ပြီး၊ ၎င်းကို မူရင်း ကွာရီပက်တန်တွင် အစားထိုး အသုံးပြုနိုင်ပါသည်။

ယေဘုယျအားဖြင့်၊ ကွာရီ အကဲဖြတ်စနစ်သည် ပက်တန်ကိန်းရှင် အချို့အတွက် ချိတ်ဆက်မှုများကို သတ်မှတ်ပေးထားသော ဖရိမ်တစ်ခုတွင် ကွာရီပက်တန်တစ်ခုကို တည်ထောင်ရန် ကြိုးပမ်းသောအခါ စည်းမျဉ်းတစ်ခုကို အသုံးချရန် အောက်ပါနည်းလမ်းကို အသုံးပြုပါသည်။

  • အကယ်၍ အောင်မြင်ပါက၊ မူရင်းဖရိမ်၏ တိုးချဲ့မှုတစ်ခုကို ဖွဲ့စည်းရန် ကွာရီကို စည်းမျဉ်း၏ နိဂုံးချုပ်ချက်နှင့် ယူနီဖိုင်းလုပ်ပါ။
  • တိုးချဲ့ထားသော ဖရိမ်နှင့် ဆက်စပ်၍ စည်းမျဉ်း၏ ကိုယ်ထည်မှ ဖွဲ့စည်းထားသော ကွာရီကို အကဲဖြတ်ပါ။

ဤအရာသည် Lisp အတွက် eval/apply အကဲဖြတ်စနစ်တွင် လုပ်ထုံးလုပ်နည်းတစ်ခုကို အသုံးချသည့် နည်းလမ်းနှင့် မည်မျှ ဆင်တူသည်ကို သတိပြုပါ-

  • မူရင်း လုပ်ထုံးလုပ်နည်း ပတ်ဝန်းကျင်ကို တိုးချဲ့ထားသော ဖရိမ်တစ်ခု ဖွဲ့စည်းရန် လုပ်ထုံးလုပ်နည်း၏ ကန့်သတ်ချက် (parameters) များကို ၎င်း၏ အငြင်းအခုံ (arguments) များနှင့် ချိတ်ဆက်ပါ။
  • တိုးချဲ့ထားသော ပတ်ဝန်းကျင်နှင့် ဆက်စပ်၍ လုပ်ထုံးလုပ်နည်း၏ ကိုယ်ထည်မှ ဖွဲ့စည်းထားသော အသုံးအနှုန်းကို အကဲဖြတ်ပါ။

အကဲဖြတ်စနစ် နှစ်ခုကြားရှိ ဆင်တူမှုမှာ အံ့သြစရာမဟုတ်ပါ။ လုပ်ထုံးလုပ်နည်း အဓိပ္ပာယ်ဖွင့်ဆိုချက်များသည် Lisp တွင် အနှစ်ချုပ်ခြင်း (abstraction) ၏ နည်းလမ်းများ ဖြစ်သကဲ့သို့၊ စည်းမျဉ်း အဓိပ္ပာယ်ဖွင့်ဆိုချက်များသည် ကွာရီဘာသာစကားတွင် အနှစ်ချုပ်ခြင်း၏ နည်းလမ်းများ ဖြစ်သည်။ ကိစ္စရပ်တစ်ခုစီတွင်၊ သင့်လျော်သော ချိတ်ဆက်မှုများကို ဖန်တီးပြီး ၎င်းတို့နှင့် ဆက်စပ်၍ စည်းမျဉ်း သို့မဟုတ် လုပ်ထုံးလုပ်နည်း ကိုယ်ထည်ကို အကဲဖြတ်ခြင်းဖြင့် အနှစ်ချုပ်မှုကို ကျွန်ုပ်တို့ ဖြေရှင်းပါသည်။

ရိုးရှင်းသော ကွာရီများ (Simple queries)

စည်းမျဉ်းများ မရှိသည့်အခါ ရိုးရှင်းသော ကွာရီများကို မည်သို့ အကဲဖြတ်ရမည်ကို ဤအပိုင်း၏ အစောပိုင်းတွင် ကျွန်ုပ်တို့ မြင်တွေ့ခဲ့ရသည်။ ယခု စည်းမျဉ်းများကို မည်သို့ အသုံးချရမည်ကို သိမြင်ပြီးဖြစ်၍၊ စည်းမျဉ်းများနှင့် အခိုင်အမာဖော်ပြချက်များ နှစ်မျိုးလုံးကို အသုံးပြု၍ ရိုးရှင်းသော ကွာရီများကို မည်သို့ အကဲဖြတ်ရမည်ကို ဖော်ပြနိုင်ပြီ ဖြစ်သည်။

ကွာရီပက်တန်နှင့် ဖရိမ်စထရင်းတစ်ခု ပေးထားလျှင်၊ ထည့်သွင်းစထရင်းရှိ ဖရိမ်တစ်ခုစီအတွက် စထရင်းနှစ်ခုကို ကျွန်ုပ်တို့ ထုတ်လုပ်ပါသည်-

  • (ပက်တန် မက်ချာကို အသုံးပြု၍) ဒေတာဘေ့စ်ရှိ အခိုင်အမာဖော်ပြချက် အားလုံးနှင့် ပက်တန်ကို တိုက်ဆိုင်စစ်ဆေးခြင်းဖြင့် ရရှိသော တိုးချဲ့ဖရိမ်များ၏ စထရင်းတစ်ခု၊ နှင့်
  • (ယူနီဖိုင်ယာကို အသုံးပြု၍) ဖြစ်နိုင်သမျှ စည်းမျဉ်းအားလုံးကို အသုံးချခြင်းဖြင့် ရရှိသော တိုးချဲ့ဖရိမ်များ၏ စထရင်းတစ်ခု။277

ဤစထရင်းနှစ်ခုကို ပေါင်းစည်း (appending) ခြင်းဖြင့် ပေးထားသော ပက်တန်ကို မူရင်းဖရိမ်နှင့် ကိုက်ညီသော နည်းလမ်းဖြင့် ကျေနပ်စေနိုင်သည့် နည်းလမ်းအားလုံး ပါဝင်သော စထရင်းတစ်ခုကို ထုတ်လုပ်ပေးပါသည်။ ဤစထရင်းများ (ထည့်သွင်းစထရင်းရှိ ဖရိမ်တစ်ခုစီအတွက် တစ်ခုစီ) ကို ယခုအခါ ပေါင်းစပ်၍ ဧရာမ စထရင်းကြီးတစ်ခုအဖြစ် ဖွဲ့စည်းလိုက်ပါသည်၊ ထို့ကြောင့် ၎င်းတွင် မူရင်းထည့်သွင်းစထရင်းရှိ မည်သည့်ဖရိမ်ကိုမဆို ပေးထားသော ပက်တန်နှင့် တိုက်ဆိုင်မှုတစ်ခု ထုတ်လုပ်ရန် တိုးချဲ့နိုင်သည့် နည်းလမ်းအားလုံး ပါဝင်ပါသည်။

ကွာရီ အကဲဖြတ်စနစ်နှင့် မောင်းနှင်သည့် ပတ်လမ်း (The query evaluator and the driver loop)

အခြေခံ တိုက်ဆိုင်စစ်ဆေးခြင်း လုပ်ဆောင်ချက်များ၏ ရှုပ်ထွေးမှုရှိသော်လည်း၊ စနစ်ကို မည်သည့်ဘာသာစကားအတွက်မဆို အကဲဖြတ်စနစ်တစ်ခုကဲ့သို့ပင် ဖွဲ့စည်းထားပါသည်။ တိုက်ဆိုင်စစ်ဆေးခြင်း လုပ်ဆောင်ချက်များကို ညှိနှိုင်းဆောင်ရွက်ပေးသော လုပ်ထုံးလုပ်နည်းကို qeval ဟုခေါ်ပြီး၊ ၎င်းသည် Lisp အတွက် eval လုပ်ထုံးလုပ်နည်းနှင့် ဆင်တူသော အခန်းကဏ္ဍတွင် ပါဝင်ပါသည်။ Qeval သည် ထည့်သွင်းချက်အဖြစ် ကွာရီတစ်ခုနှင့် ဖရိမ်စထရင်းတစ်ခုကို ယူပါသည်။ ၎င်း၏ ရလဒ်မှာ ပုံ 4.4 တွင် ပြထားသည့်အတိုင်း ထည့်သွင်းစထရင်းရှိ အချို့သော ဖရိမ်များကို တိုးချဲ့ထားသည့်၊ ကွာရီပက်တန်နှင့် အောင်မြင်သော တိုက်ဆိုင်မှုများနှင့် ကိုက်ညီသော ဖရိမ်စထရင်းတစ်ခု ဖြစ်ပါသည်။ eval ကဲ့သို့ပင်၊ qeval သည် မတူညီသော အသုံးအနှုန်း (ကွာရီ) အမျိုးအစားများကို ခွဲခြားပြီး တစ်ခုစီအတွက် သင့်လျော်သော လုပ်ထုံးလုပ်နည်းသို့ ပေးပို့ (dispatch) ပါသည်။ အထူးပုံစံ တစ်ခုစီ (and, or, not, နှင့် lisp-value) အတွက် လုပ်ထုံးလုပ်နည်းတစ်ခုစီနှင့် ရိုးရှင်းသော ကွာရီများအတွက် တစ်ခု ရှိပါသည်။

ဤအခန်းရှိ အခြားသော အကဲဖြတ်စနစ်များအတွက် driver-loop လုပ်ထုံးလုပ်နည်းနှင့် ဆင်တူသော မောင်းနှင်သည့် ပတ်လမ်း (driver loop) သည် တာမီနယ်မှ ကွာရီများကို ဖတ်ပါသည်။ ကွာရီတစ်ခုစီအတွက်၊ ၎င်းသည် ကွာရီနှင့် ဗလာဖရိမ်တစ်ခုတည်း ပါဝင်သော စထရင်းတစ်ခုဖြင့် qeval ကို ခေါ်ဆိုပါသည်။ ၎င်းသည် ဖြစ်နိုင်သမျှ တိုက်ဆိုင်မှုအားလုံး (ဗလာဖရိမ်၏ ဖြစ်နိုင်သမျှ တိုးချဲ့မှုအားလုံး) ၏ စထရင်းကို ထုတ်လုပ်ပါလိမ့်မည်။ ရရှိလာသော စထရင်းရှိ ဖရိမ်တစ်ခုစီအတွက်၊ ၎င်းသည် ဖရိမ်ထဲတွင် တွေ့ရှိရသော ကိန်းရှင်များ၏ တန်ဖိုးများကို အသုံးပြု၍ မူရင်းကွာရီကို အစားထိုး (instantiate) လုပ်ဆောင်ပါသည်။ ထို့နောက် ဤ အစားထိုးထားသော ကွာရီများ၏ စထရင်းကို ပုံနှိပ်ထုတ်ပေးပါသည်။278

မောင်းနှင်သည့် ပတ်လမ်းသည် အထူးအမိန့် assert! ကိုလည်း စစ်ဆေးပါသည်၊ ၎င်းသည် ထည့်သွင်းချက်မှာ ကွာရီတစ်ခုမဟုတ်ဘဲ ဒေတာဘေ့စ်သို့ ထည့်သွင်းရန် အခိုင်အမာဖော်ပြချက် သို့မဟုတ် စည်းမျဉ်းတစ်ခု ဖြစ်ကြောင်း အချက်ပြပါသည်။ ဥပမာအားဖြင့်၊

4.4.3Logic Programming သည် သင်္ချာဆိုင်ရာ ယုတ္တိဗေဒ (Mathematical Logic) လော?

query language တွင် အသုံးပြုထားသော ပေါင်းစပ်မှုနည်းလမ်းများသည် အစပိုင်းတွင် သင်္ချာဆိုင်ရာ ယုတ္တိဗေဒ၏ and၊ or နှင့် not လုပ်ဆောင်ချက်များနှင့် ထပ်တူကျသည်ဟု ထင်ရနိုင်ပြီး၊ query-language စည်းမျဉ်းများကို အသုံးချခြင်းသည် အမှန်တကယ်တွင် ခိုင်လုံသော inference နည်းလမ်းတစ်ခုဖြင့် လုပ်ဆောင်ခြင်း ဖြစ်သည်။279 သို့သော် query language ကို သင်္ချာဆိုင်ရာ ယုတ္တိဗေဒနှင့် ဤကဲ့သို့ တရားသေ သတ်မှတ်ခြင်းသည် အမှန်တကယ်တွင် မမှန်ကန်ပေ။ အကြောင်းမှာ query language သည် logical statements များကို procedural အတိုင်း အဓိပ္ပာယ်ဖွင့်ဆိုပေးသည့် control structure တစ်ခုကို ပံ့ပိုးပေးထားသောကြောင့် ဖြစ်သည်။ ကျွန်ုပ်တို့သည် ဤ control structure ကို မကြာခဏ အသုံးချနိုင်သည်။ ဥပမာအားဖြင့်၊ programmer များ၏ supervisor အားလုံးကို ရှာဖွေရန် ယုတ္တိဗေဒအရ တူညီသော ပုံစံနှစ်မျိုးအနက်မှ တစ်ခုခုဖြင့် query ကို ရေးသားနိုင်သည်- 

(and (job ?x (computer programmer))
     (supervisor ?x ?y))

သို့မဟုတ် 

(and (supervisor ?x ?y)
     (job ?x (computer programmer)))

အကယ်၍ ကုမ္ပဏီတစ်ခုတွင် programmer များထက် supervisor များ ပိုမိုများပြားနေပါက (ပုံမှန်အားဖြင့် ထိုသို့ပင် ဖြစ်တတ်သည်)၊ ဒုတိယပုံစံထက် ပထမပုံစံကို အသုံးပြုခြင်းက ပိုမိုကောင်းမွန်သည်။ အကြောင်းမှာ and ၏ ပထမ clause မှ ထွက်ပေါ်လာသော ကြားဖြတ်ရလဒ် (frame) တစ်ခုစီအတွက် data base ကို scan ဖတ်ရမည် ဖြစ်သောကြောင့် ဖြစ်သည်။

Logic programming ၏ ရည်ရွယ်ချက်မှာ တွက်ချက်မှုဆိုင်ရာ ပြဿနာတစ်ခုကို သီးခြားပြဿနာနှစ်ခုအဖြစ် ခွဲထုတ်နိုင်မည့် နည်းစနစ်များကို programmer ထံ ပံ့ပိုးပေးရန်ဖြစ်သည်- “ဘာကို” တွက်ချက်မည်နည်းနှင့် ၎င်းကို “ဘယ်လို” တွက်ချက်ရမည်နည်း တို့ဖြစ်သည်။ ၎င်းကို တွက်ချက်လိုသမျှ အရာအားလုံးကို ဖော်ပြနိုင်လောက်အောင် အစွမ်းထက်သော်လည်း၊ ထိန်းချုပ်နိုင်သော procedural interpretation ရှိရန်အတွက်မူ လုံလောက်စွာ အားနည်းသော သင်္ချာဆိုင်ရာ ယုတ္တိဗေဒ၏ subset တစ်ခုကို ရွေးချယ်ခြင်းဖြင့် အောင်မြင်အောင် လုပ်ဆောင်သည်။ ဤနေရာတွင် ရည်ရွယ်ချက်မှာ တစ်ဖက်တွင် logic programming language ဖြင့် သတ်မှတ်ထားသော program တစ်ခုသည် ကွန်ပျူတာဖြင့် လုပ်ဆောင်နိုင်သော ထိရောက်သည့် program တစ်ခု ဖြစ်ရမည်။ Control (“ဘယ်လို” တွက်ချက်မည်နည်း) ကို language ၏ evaluation order ကို အသုံးပြု၍ လုပ်ဆောင်သည်။ ကျွန်ုပ်တို့သည် clause များ၏ အစီအစဉ်နှင့် clause တစ်ခုစီအတွင်းရှိ subgoal များ၏ အစီအစဉ်ကို တွက်ချက်မှုကို ထိရောက်ပြီး စွမ်းဆောင်ရည်ကောင်းမွန်သော အစီအစဉ်အတိုင်း လုပ်ဆောင်နိုင်ရန် စီစဉ်နိုင်ရမည်။ တစ်ချိန်တည်းမှာပင်၊ တွက်ချက်မှု၏ ရလဒ် (“ဘာကို” တွက်ချက်မည်နည်း) ကို ယုတ္တိဗေဒ ဥပဒေသများ၏ ရိုးရှင်းသော အကျိုးဆက်တစ်ခုအဖြစ် ရှုမြင်နိုင်ရမည်။

ကျွန်ုပ်တို့၏ query language ကို သင်္ချာဆိုင်ရာ ယုတ္တိဗေဒ၏ ထိုကဲ့သို့သော procedural အတိုင်း အဓိပ္ပာယ်ဖွင့်ဆိုနိုင်သည့် subset တစ်ခုအဖြစ် မှတ်ယူနိုင်သည်။ assertion တစ်ခုသည် ရိုးရှင်းသော အချက်အလက် (atomic proposition) တစ်ခုကို ကိုယ်စားပြုသည်။ rule တစ်ခုသည် rule conclusion မှန်ကန်ရန်အတွက် rule body မှန်ကန်ရမည်ဟူသော implication ကို ကိုယ်စားပြုသည်။ rule တစ်ခုတွင် သဘာဝကျသော procedural interpretation ရှိသည်- rule ၏ conclusion ကို တည်ဆောက်ရန်၊ rule ၏ body ကို တည်ဆောက်ပါ။ ထို့ကြောင့် rule များသည် တွက်ချက်မှုများကို သတ်မှတ်ပေးသည်။ သို့သော် rule များကို သင်္ချာဆိုင်ရာ ယုတ္တိဗေဒ၏ ဖော်ပြချက်များအဖြစ်လည်း မှတ်ယူနိုင်သောကြောင့်၊ logic program တစ်ခုမှ လုပ်ဆောင်သော မည်သည့် “inference” ကိုမဆို သင်္ချာဆိုင်ရာ ယုတ္တိဗေဒအတွင်း၌သာ အလုံးစုံ လုပ်ဆောင်ခြင်းဖြင့်လည်း တူညီသော ရလဒ်ကို ရရှိနိုင်သည်ဟု အခိုင်အမာဆိုကာ ကြောင်းကျိုးပြနိုင်သည်။280

Infinite loops (အဆုံးမရှိသော ပတ်လမ်းများ)

Logic program များ၏ procedural interpretation ၏ အကျိုးဆက်တစ်ခုမှာ အချို့သော ပြဿနာများကို ဖြေရှင်းရန်အတွက် လုံးဝ စွမ်းဆောင်ရည်မရှိသော program များကို တည်ဆောက်မိနိုင်ခြင်း ဖြစ်သည်။ စနစ်သည် deductions များ ပြုလုပ်ရာတွင် infinite loop များအတွင်းသို့ ကျရောက်သွားသောအခါ စွမ်းဆောင်ရည်မရှိမှု၏ အဆိုးရွားဆုံး အခြေအနေ ဖြစ်ပေါ်သည်။ ရိုးရှင်းသော ဥပမာတစ်ခုအနေဖြင့်၊ ကျွန်ုပ်တို့သည် အောက်ပါတို့အပါအဝင် ထင်ရှားသော အိမ်ထောင်ရေးများဆိုင်ရာ data base တစ်ခုကို တည်ဆောက်နေသည်ဟု ဆိုကြပါစို့။ 

(assert! (married Minnie Mickey))

အကယ်၍ ကျွန်ုပ်တို့က အခုလို မေးမြန်းပါက 

(married Mickey ?who)

ကျွန်ုပ်တို့သည် မည်သည့်တုံ့ပြန်မှုကိုမျှ ရရှိမည်မဟုတ်ပါ။ အကြောင်းမှာ အကယ်၍ A သည် B နှင့် လက်ထပ်ထားပါက၊ B သည်လည်း A နှင့် လက်ထပ်ထားသည်ဖြစ်ကြောင်း စနစ်က မသိရှိသောကြောင့် ဖြစ်သည်။ ထို့ကြောင့် ကျွန်ုပ်တို့သည် rule ကို assert လုပ်သည်- 

(assert! (rule (married ?x ?y)
               (married ?y ?x)))

ထို့နောက် တစ်ဖန် query ပြန်လုပ်သည် 

(married Mickey ?who)

ကံမကောင်းစွာဖြင့်၊ ၎င်းသည် စနစ်ကို အောက်ပါအတိုင်း infinite loop တစ်ခုအတွင်းသို့ ရောက်ရှိသွားစေလိမ့်မည်-

  • စနစ်သည် married rule ကို အသုံးပြုနိုင်ကြောင်း တွေ့ရှိရသည်၊ ဆိုလိုသည်မှာ rule conclusion ဖြစ်သော (married ?x ?y) သည် query pattern ဖြစ်သော (married Mickey ?who) နှင့် အောင်မြင်စွာ unify ဖြစ်ပြီး၊ ?x ကို Mickey နှင့် ?y ကို ?who နှင့် bound ထားသော frame တစ်ခုကို ထုတ်ပေးသည်။ ထို့ကြောင့် interpreter သည် ဤ frame တွင် rule body (married ?y ?x) ကို ဆက်လက် evaluate လုပ်သည် — လက်တွေ့တွင်မူ (married ?who Mickey) query ကို လုပ်ဆောင်ခြင်းဖြစ်သည်။
  • အဖြေတစ်ခုမှာ data base တွင် assertion တစ်ခုအဖြစ် တိုက်ရိုက်ပေါ်လာသည်- (married Minnie Mickey).
  • married rule ကိုလည်း အသုံးပြုနိုင်ပြန်သဖြင့်၊ interpreter သည် rule body ကို ထပ်မံ evaluate ပြုလုပ်သည်၊ ဤတစ်ကြိမ်တွင် (married Mickey ?who) နှင့် ထပ်တူဖြစ်သည်။

စနစ်သည် ယခုအခါ infinite loop တစ်ခုထဲသို့ ရောက်ရှိနေပြီဖြစ်သည်။ အမှန်စင်စစ်၊ loop ထဲမဝင်မီ (married Minnie Mickey) ဟူသော ရိုးရှင်းသည့် အဖြေကို စနစ်က တွေ့ရှိနိုင်ခြင်း ရှိမရှိမှာ စနစ်က data base ရှိ item များကို စစ်ဆေးသည့် order နှင့် ပတ်သက်သော implementation အသေးစိတ်အချက်အလက်များပေါ်တွင် မူတည်သည်။ ၎င်းသည် ဖြစ်ပွားနိုင်သော loop အမျိုးအစားများ၏ အလွန်ရိုးရှင်းသော ဥပမာတစ်ခုမျှသာ ဖြစ်သည်။ အပြန်အလှန် ဆက်နွှယ်နေသော rule များ စုစည်းမှုသည် ခန့်မှန်းရ ပိုမိုခက်ခဲသော loop များဆီသို့ ဦးတည်သွားစေနိုင်ပြီး၊ loop တစ်ခု ဖြစ်ပေါ်လာခြင်းသည် and အတွင်းရှိ clause များ၏ order (Exercise 4.64 ကိုကြည့်ပါ) သို့မဟုတ် စနစ်က query များကို လုပ်ဆောင်သည့် order နှင့် ပတ်သက်သော low-level details များပေါ်တွင် မူတည်နိုင်သည်။281

not နှင့် ပတ်သက်သော ပြဿနာများ

Query system ထဲက နောက်ထပ် ထူးခြားချက်တစ်ခုကတော့ not နဲ့ သက်ဆိုင်ပါတယ်။ 4.4.1 ရဲ့ data base ကို အခြေခံပြီး အောက်ပါ query နှစ်ခုကို လေ့လာကြည့်ရအောင်- 

(and (supervisor ?x ?y)
     (not (job ?x (computer programmer))))

(and (not (job ?x (computer programmer)))
     (supervisor ?x ?y))

ဤ query နှစ်ခုသည် တူညီသော ရလဒ်ကို မထုတ်ပေးပါ။ ပထမ query သည် (supervisor ?x ?y) နှင့် ကိုက်ညီသော data base ရှိ entry အားလုံးကို ရှာဖွေခြင်းဖြင့် စတင်ပြီး၊ ထို့နောက် ထွက်ပေါ်လာသော frame များအနက်မှ ?x ၏ တန်ဖိုးက (job ?x (computer programmer)) ကို ကျေနပ်စေသော frame များကို ဖယ်ရှားခြင်းဖြင့် filter လုပ်သည်။ ဒုတိယ query သည် ဝင်ရောက်လာသော frame များကို (job ?x (computer programmer)) ကို ကျေနပ်စေနိုင်သော frame များ ဖယ်ရှားရန် filter လုပ်ခြင်းဖြင့် စတင်သည်။ ဝင်ရောက်လာသော တစ်ခုတည်းသော frame မှာ empty ဖြစ်နေသဖြင့်၊ ၎င်းသည် (job ?x (computer programmer)) ကို ကျေနပ်စေမည့် pattern များ ရှိမရှိ data base တွင် စစ်ဆေးသည်။ ယေဘုယျအားဖြင့် ထိုကဲ့သို့သော entry များ ရှိနေတတ်သဖြင့်၊ not clause သည် empty frame ကို filter လုပ်ပြီး ဖယ်ထုတ်ပစ်လိုက်ကာ empty frames stream တစ်ခုကို ပြန်ပေးသည်။ ရလဒ်အနေဖြင့် compound query တစ်ခုလုံးသည် empty stream တစ်ခုကို ပြန်ပေးသည်။

အခက်အခဲမှာ ကျွန်ုပ်တို့၏ not implementation သည် variable များအတွက် တန်ဖိုးများကို filter တစ်ခုအဖြစ် အသုံးပြုရန်သာ ရည်ရွယ်ခြင်း ဖြစ်သည်။ အကယ်၍ not clause တစ်ခုကို အချို့သော variable များမှာ unbound ဖြစ်နေဆဲ (အထက်ပါ ဥပမာရှိ ?x ကဲ့သို့) frame တစ်ခုနှင့် လုပ်ဆောင်ပါက၊ စနစ်သည် မမျှော်လင့်ထားသော ရလဒ်များကို ထုတ်ပေးလိမ့်မည်။ lisp-value ကို အသုံးပြုရာတွင်လည်း အလားတူ ပြဿနာများ ဖြစ်ပွားတတ်သည် — အကယ်၍ Lisp predicate ၏ argument အချို့မှာ unbound ဖြစ်နေပါက ၎င်းသည် အလုပ်မလုပ်နိုင်ပါ။ Exercise 4.77 ကို ကြည့်ပါ။

Query language ၏ not သည် သင်္ချာဆိုင်ရာ ယုတ္တိဗေဒ၏ not နှင့် ကွဲပြားသော ပိုမိုကြီးမားသည့် နည်းလမ်းတစ်ခုလည်း ရှိသေးသည်။ ယုတ္တိဗေဒတွင် ကျွန်ုပ်တို့သည် “not P ” ဆိုသည့် ဖော်ပြချက်ကို P သည် မမှန်ကန်ကြောင်း အဓိပ္ပာယ်ဖွင့်ဆိုသည်။ သို့သော် query system တွင်မူ “not P ” ဆိုသည်မှာ data base ရှိ အသိပညာများမှ P ကို deducible မဖြစ်ခြင်းကို ဆိုလိုသည်။ ဥပမာအားဖြင့်၊ 4.4.1 ရှိ ဝန်ထမ်းအချက်အလက် data base ကို အခြေခံလျှင် Ben Bitdiddle သည် ဘေ့စ်ဘော ဝါသနာရှင် မဟုတ်ကြောင်း၊ အပြင်မှာ မိုးမရွာနေကြောင်းနှင့် 2 + 2 သည် 4 မဟုတ်ကြောင်း စသည့် not ဖော်ပြချက် အမျိုးမျိုးကို စနစ်က ကောက်ချက်ချပေလိမ့်မည်။282 တစ်နည်းအားဖြင့်ဆိုသော် logic programming language များ၏ not သည် သက်ဆိုင်ရာ အချက်အလက်အားလုံးကို data base ထဲတွင် ထည့်သွင်းထားပြီးဖြစ်သည်ဟူသော closed world assumption ဟု ခေါ်ဆိုသည့် အယူအဆကို ထင်ဟပ်စေသည်။283

4.4.4Query System ကို အကောင်အထည်ဖော်ခြင်း

အပိုင်း 4.4.2 တွင် query system မည်သို့အလုပ်လုပ်သည်ကို ဖော်ပြခဲ့ပြီးဖြစ်သည်။ ယခုအခါ system ၏ အလုံးစုံအကောင်အထည်ဖော်ပုံကို တင်ပြခြင်းဖြင့် အသေးစိတ်အချက်အလက်များကို ဖြည့်စွက်ပါမည်။

4.4.4.1Driver Loop နှင့် Instantiation

query system အတွက် driver loop သည် input expression များကို ထပ်ခါတလဲလဲ ဖတ်ရှုသည်။ အကယ်၍ expression သည် data base ထဲသို့ ထည့်သွင်းရမည့် rule သို့မဟုတ် assertion တစ်ခုဖြစ်ပါက၊ အချက်အလက်ကို ထည့်သွင်းပေးသည်။ သို့မဟုတ်ပါက expression ကို query တစ်ခုအဖြစ် ယူဆသည်။ Driver သည် ဤ query ကို evaluator ဖြစ်သော qeval ထံသို့ frame အလွတ်တစ်ခုတည်းသာပါဝင်သော အစဦး frame stream နှင့်အတူ ပေးပို့သည်။ evaluation ၏ ရလဒ်မှာ data base ထဲတွင် တွေ့ရှိရသော variable တန်ဖိုးများဖြင့် query ကို ပြည့်စုံစေခြင်းဖြင့် ထွက်ပေါ်လာသော frame stream တစ်ခု ဖြစ်သည်။ ဤ frame များကို မူလ query ၏ copy များပါဝင်သော stream အသစ်တစ်ခု ပြုလုပ်ရန် အသုံးပြုပြီး၊ ထို copy များတွင် variable များကို frame stream မှ ရရှိသော တန်ဖိုးများဖြင့် instantiate လုပ်ပေးသည်။ ထို့နောက် ဤနောက်ဆုံး stream ကို terminal တွင် print ထုတ်ပေးသည်- 

(define input-prompt  ";;; Query input:")
(define output-prompt ";;; Query results:")

(define (query-driver-loop)
  (prompt-for-input input-prompt)
  (let ((q (query-syntax-process (read))))
    (cond ((assertion-to-be-added? q)
           (add-rule-or-assertion! 
            (add-assertion-body q))
           (newline)
           (display 
            "Assertion added to data base.")
           (query-driver-loop))
          (else
           (newline)
           (display output-prompt)
           (display-stream
            (stream-map
             (lambda (frame)
               (instantiate
                q
                frame
                (lambda (v f)
                  (contract-question-mark v))))
             (qeval q (singleton-stream '()))))
           (query-driver-loop)))))

ဤနေရာတွင်၊ ဤအခန်းရှိ အခြား evaluator များတွင်ကဲ့သို့ပင်၊ ကျွန်ုပ်တို့သည် query language ၏ expression များအတွက် abstract syntax ကို အသုံးပြုသည်။ assertion-to-be-added? ဆိုသော predicate နှင့် add-assertion-body ဆိုသော selector တို့အပါအဝင် expression syntax ၏ အကောင်အထည်ဖော်မှုကို 4.4.4.7 တွင် ပေးထားသည်။ Add-rule-or-assertion! ကိုမူ 4.4.4.5 တွင် အဓိပ္ပာယ်ဖွင့်ဆိုထားသည်။

input expression အပေါ်တွင် မည်သည့်လုပ်ဆောင်ချက်မျှ မလုပ်မီ၊ driver loop သည် ၎င်းကို ပိုမိုထိရောက်စွာ လုပ်ဆောင်နိုင်ရန် syntactic ပုံစံတစ်ခုသို့ ပြောင်းလဲပေးသည်။ ၎င်းတွင် pattern variable များ၏ ကိုယ်စားပြုပုံကို ပြောင်းလဲခြင်း ပါဝင်သည်။ query ကို instantiate လုပ်သောအခါ၊ unbound ဖြစ်ကျန်နေသော မည်သည့် variable များကိုမဆို print မထုတ်မီ input ကိုယ်စားပြုပုံသို့ ပြန်လည်ပြောင်းလဲပေးသည်။ ဤအသွင်ပြောင်းမှုများကို query-syntax-process နှင့် contract-question-mark (4.4.4.7) ဟူသော procedure နှစ်ခုက လုပ်ဆောင်ပေးသည်။

expression တစ်ခုကို instantiate လုပ်ရန်၊ ကျွန်ုပ်တို့သည် ၎င်းကို copy ကူးပြီး expression ထဲရှိ မည်သည့် variable ကိုမဆို ပေးထားသော frame ထဲရှိ ၎င်းတို့၏ တန်ဖိုးများဖြင့် အစားထိုးသည်။ variable များ ကိုယ်တိုင်တွင်လည်း variable များ ပါဝင်နိုင်သောကြောင့် ထိုတန်ဖိုးများကိုလည်း instantiate လုပ်သည် (ဥပမာအားဖြင့်၊ exp ထဲရှိ ?x သည် unification ၏ ရလဒ်အဖြစ် ?y နှင့် တွဲဖက်ထားပြီး ?y သည် တစ်ဖန် 5 နှင့် တွဲဖက်ထားလျှင် ဖြစ်သည်)။ အကယ်၍ variable တစ်ခုကို instantiate မလုပ်နိုင်ပါက လုပ်ဆောင်ရမည့် အစီအစဉ်ကို instantiate သို့ procedural argument တစ်ခုအဖြစ် ပေးပို့သည်။ 

(define (instantiate 
         exp frame unbound-var-handler)
  (define (copy exp)
    (cond ((var? exp)
           (let ((binding 
                  (binding-in-frame 
                   exp frame)))
             (if binding
                 (copy 
                  (binding-value binding))
                 (unbound-var-handler 
                  exp frame))))
          ((pair? exp)
           (cons (copy (car exp)) 
                 (copy (cdr exp))))
          (else exp)))
  (copy exp))

binding များကို ကိုင်တွယ်ဆောင်ရွက်သည့် procedure များကို 4.4.4.8 တွင် အဓိပ္ပာယ်ဖွင့်ဆိုထားသည်။

4.4.4.2Evaluator

query-driver-loop မှ ခေါ်ယူသော qeval procedure သည် query system ၏ အခြေခံ evaluator ဖြစ်သည်။ ၎င်းသည် query တစ်ခုနှင့် frame stream တစ်ခုတို့ကို input အဖြစ် လက်ခံပြီး၊ extended frame stream တစ်ခုကို ပြန်ပေးသည်။ အခန်း ၂ တွင် generic operation များကို အကောင်အထည်ဖော်ခဲ့စဉ်ကကဲ့သို့ပင်၊ ၎င်းသည် get နှင့် put ကို အသုံးပြု၍ data-directed dispatch ဖြင့် special form များကို ခွဲခြားသတ်မှတ်သည်။ special form အဖြစ် ခွဲခြားမထားသော မည်သည့် query ကိုမဆို simple query တစ်ခုအဖြစ် ယူဆပြီး simple-query က လုပ်ဆောင်ပေးမည် ဖြစ်သည်။ 

(define (qeval query frame-stream)
  (let ((qproc (get (type query) 'qeval)))
    (if qproc
        (qproc (contents query) frame-stream)
        (simple-query query frame-stream))))

ရိုးရှင်းသော query များ

simple-query procedure သည် ရိုးရှင်းသော query များကို ကိုင်တွယ်သည်။ ၎င်းသည် argument အဖြစ် simple query (pattern တစ်ခု) နှင့် frame stream တစ်ခုတို့ကို လက်ခံပြီး၊ query နှင့် data-base တိုက်ဆိုင်မှုအားလုံးဖြင့် frame တစ်ခုစီကို တိုးချဲ့ခြင်းဖြင့် ဖြစ်ပေါ်လာသော stream ကို ပြန်ပေးသည်။ 

(define (simple-query query-pattern 
                      frame-stream)
  (stream-flatmap
   (lambda (frame)
     (stream-append-delayed
      (find-assertions query-pattern frame)
      (delay 
        (apply-rules query-pattern frame))))
   frame-stream))

input stream ထဲရှိ frame တစ်ခုစီအတွက်၊ ကျွန်ုပ်တို့သည် pattern ကို data base ထဲရှိ assertion အားလုံးနှင့် တိုက်ဆိုင်စစ်ဆေးရန် find-assertions (4.4.4.3) ကို အသုံးပြုပြီး extended frame stream တစ်ခုကို ထုတ်ပေးသည်။ ထို့ပြင် ဖြစ်နိုင်သမျှ rule အားလုံးကို အသုံးချရန် apply-rules (4.4.4.4) ကို အသုံးပြုပြီး အခြား extended frame stream တစ်ခုကိုလည်း ထုတ်ပေးသည်။ ဤ stream နှစ်ခုကို ပေါင်းစပ်ပြီး (stream-append-delayed၊ 4.4.4.6 ကို အသုံးပြု၍) ပေးထားသော pattern သည် မူလ frame နှင့် ကိုက်ညီစွာ ပြည့်စုံနိုင်သည့် နည်းလမ်းအားလုံး၏ stream တစ်ခုကို ပြုလုပ်သည် (လေ့ကျင့်ခန်း 4.71 ကို ကြည့်ပါ)။ input frame တစ်ခုစီအတွက် stream များကို stream-flatmap (4.4.4.6) အသုံးပြု၍ ပေါင်းစပ်ကာ မူလ input stream ထဲရှိ မည်သည့် frame ကိုမဆို ပေးထားသော pattern နှင့် ကိုက်ညီမှုရှိစေရန် တိုးချဲ့နိုင်သည့် နည်းလမ်းအားလုံးပါဝင်သော stream ကြီးတစ်ခုကို ဖွဲ့စည်းသည်။

ပေါင်းစပ်ထားသော query များ

And query များကို ပုံ 4.5 တွင် ဖော်ပြထားသည့်အတိုင်း conjoin procedure ဖြင့် ကိုင်တွယ်သည်။ Conjoin သည် input အဖြစ် conjunct များနှင့် frame stream ကို လက်ခံပြီး extended frame stream ကို ပြန်ပေးသည်။ ပထမဦးစွာ၊ conjoin သည် conjunction ထဲရှိ ပထမဆုံး query ကို ပြည့်စုံစေသော ဖြစ်နိုင်သမျှ frame extension အားလုံး၏ stream ကို ရှာဖွေရန် frame stream ကို လုပ်ဆောင်သည်။ ထို့နောက်၊ ၎င်းကို frame stream အသစ်အဖြစ် အသုံးပြုပြီး ကျန်ရှိသော query များအပေါ် conjoin ကို recursive နည်းဖြင့် အသုံးချသည်။ 

(define (conjoin conjuncts frame-stream)
  (if (empty-conjunction? conjuncts)
      frame-stream
      (conjoin (rest-conjuncts conjuncts)
               (qeval 
                (first-conjunct conjuncts)
                frame-stream))))

ဤ expression

သည် and form နှင့် တွေ့သောအခါ conjoin သို့ dispatch လုပ်ရန် qeval ကို သတ်မှတ်ပေးသည်။

Or query များကိုလည်း ပုံ 4.6 တွင် ပြထားသည့်အတိုင်း အလားတူပင် ကိုင်တွယ်သည်။ or ၏ အမျိုးမျိုးသော disjunct များအတွက် output stream များကို သီးခြားစီ တွက်ချက်ပြီး 4.4.4.6 ရှိ interleave-delayed procedure ကို အသုံးပြု၍ ပေါင်းစပ်သည်။ (လေ့ကျင့်ခန်း 4.71 နှင့် လေ့ကျင့်ခန်း 4.72 ကို ကြည့်ပါ) 

(define (disjoin disjuncts frame-stream)
  (if (empty-disjunction? disjuncts)
      the-empty-stream
      (interleave-delayed
       (qeval (first-disjunct disjuncts) 
              frame-stream)
       (delay (disjoin 
               (rest-disjuncts disjuncts)
               frame-stream)))))
(put 'or 'qeval disjoin)

(define (negate operands frame-stream)
  (stream-flatmap
   (lambda (frame)

(define (lisp-value call frame-stream)

   (lambda (frame)
     (if (execute
          (instantiate
           call
           frame
           (lambda (v f)
             (error 
              "Unknown pat var: LISP-VALUE"

predicate ကို argument များသို့ သက်ရောက်စေသော Execute သည်၊ apply လုပ်ရမည့် procedure ကို ရရှိရန် predicate expression ကို eval လုပ်ရမည် ဖြစ်သည်။ သို့သော် ၎င်းသည် argument များကို evaluate မလုပ်ရပါ။ အကြောင်းမှာ ၎င်းတို့သည် အမှန်တကယ် argument များ (actual arguments) ဖြစ်နေပြီးဖြစ်သောကြောင့်ဖြစ်သည်၊ ၎င်းတို့သည် (Lisp တွင်) evaluate လုပ်ပါက argument များ ထွက်လာမည့် expression များ မဟုတ်ကြပါ။ execute ကို အရင်းခံ Lisp စနစ်ရှိ eval နှင့် apply တို့ကို အသုံးပြု၍ အကောင်အထည်ဖော်ထားသည်ကို သတိပြုပါ။ 

(define (execute exp)
  (apply (eval (predicate exp) 
               user-initial-environment)
         (args exp)))

(define (always-true ignore frame-stream)

4.4.4.3Pattern Matching အသုံးပြု၍ Assertion များကို ရှာဖွေခြင်း

simple-query (4.4.4.2) မှ ခေါ်ယူသော Find-assertions သည် input အဖြစ် pattern တစ်ခုနှင့် frame တစ်ခုကို ယူသည်။ ၎င်းသည် ပေးထားသော pattern အား data-base နှင့် တိုက်ဆိုင်စစ်ဆေးမှု (match) အပေါ် မူတည်၍ ပေးထားသော frame ကို တိုးချဲ့ထားသည့် frame stream တစ်ခုကို ပြန်ပေးသည်။ ၎င်းသည် pattern နှင့် frame တို့ကို တိုက်ဆိုင်စစ်ဆေးရန် လိုအပ်သော data base ထဲရှိ assertion အားလုံး၏ stream ကို ရယူရန် fetch-assertions (4.4.4.5) ကို အသုံးပြုသည်။ ဤနေရာတွင် fetch-assertions ကို အသုံးပြုရခြင်း အကြောင်းရင်းမှာ အောင်မြင်စွာ တိုက်ဆိုင်စစ်ဆေးနိုင်မည့် အလားအလာရှိသော အစုအဝေးထဲမှ data base entry အများအပြားကို ဖယ်ထုတ်ပစ်နိုင်မည့် ရိုးရှင်းသော စမ်းသပ်မှုများကို ကျွန်ုပ်တို့ မကြာခဏ အသုံးချနိုင်သောကြောင့် ဖြစ်သည်။ အကယ်၍ ကျွန်ုပ်တို့သည် fetch-assertions ကို ဖယ်ထုတ်ပြီး data base ရှိ assertion အားလုံး၏ stream ကို ရိုးရိုးရှင်းရှင်း စစ်ဆေးမည်ဆိုပါက စနစ်သည် အလုပ်လုပ်နေဦးမည် ဖြစ်သော်လည်း၊ matcher ကို အကြိမ်ပေါင်းများစွာ ပိုမိုခေါ်ယူရမည် ဖြစ်သောကြောင့် တွက်ချက်မှုမှာ ထိရောက်မှု (efficiency) လျော့နည်းသွားမည် ဖြစ်သည်။ 

(define (find-assertions pattern frame)
  (stream-flatmap 
    (lambda (datum) 
      (check-an-assertion datum pattern frame))
    (fetch-assertions pattern frame)))

Check-an-assertion သည် pattern တစ်ခု၊ data object (assertion) တစ်ခုနှင့် frame တစ်ခုတို့ကို argument အဖြစ် ယူပြီး၊ တိုးချဲ့ထားသော frame ပါဝင်သည့် element တစ်ခုတည်းပါသော stream သို့မဟုတ် တိုက်ဆိုင်စစ်ဆေးမှု မအောင်မြင်ပါက the-empty-stream ကို ပြန်ပေးသည်။ 

(define (check-an-assertion 
         assertion query-pat query-frame)
  (let ((match-result
         (pattern-match 
          query-pat assertion query-frame)))
    (if (eq? match-result 'failed)
        the-empty-stream
        (singleton-stream match-result))))

အခြေခံ pattern matcher သည် symbol failed သို့မဟုတ် ပေးထားသော frame ၏ တိုးချဲ့ချက် (extension) တစ်ခုကို ပြန်ပေးသည်။ matcher ၏ အခြေခံ အယူအဆမှာ pattern ကို data နှင့် တစ်ခုချင်းစီ တိုက်ဆိုင်စစ်ဆေးရန်ဖြစ်ပြီး၊ pattern variable များအတွက် binding များကို စုဆောင်းရန်ဖြစ်သည်။ အကယ်၍ pattern နှင့် data object တို့သည် တူညီနေပါက တိုက်ဆိုင်စစ်ဆေးမှု အောင်မြင်ပြီး ယခုအချိန်အထိ စုဆောင်းထားသော binding frame ကို ပြန်ပေးသည်။ သို့မဟုတ်ဘဲ pattern သည် variable တစ်ခုဖြစ်ပါက၊ ၎င်းသည် frame ထဲတွင် ရှိပြီးသား binding များနှင့် ကိုက်ညီမှု (consistent) ရှိနေသရွေ့ variable ကို data နှင့် bind လုပ်ခြင်းဖြင့် လက်ရှိ frame ကို တိုးချဲ့သည်။ အကယ်၍ pattern နှင့် data နှစ်ခုလုံးသည် pair များဖြစ်နေပါက၊ pattern ၏ car ကို data ၏ car နှင့် (recursive နည်းလမ်းဖြင့်) တိုက်ဆိုင်စစ်ဆေး၍ frame တစ်ခု ထုတ်ပေးသည်။ ထို frame ထဲတွင် pattern ၏ cdr ကို data ၏ cdr နှင့် ဆက်လက် တိုက်ဆိုင်စစ်ဆေးသည်။ အကယ်၍ ဤအခြေအနေများထဲမှ တစ်ခုမှ သက်ဆိုင်ခြင်းမရှိပါက တိုက်ဆိုင်စစ်ဆေးမှု မအောင်မြင်ဘဲ symbol failed ကို ပြန်ပေးသည်။ 

(define (pattern-match pat dat frame)
  (cond ((eq? frame 'failed) 'failed)
        ((equal? pat dat) frame)
        ((var? pat) 
         (extend-if-consistent 
          pat dat frame))
        ((and (pair? pat) (pair? dat))
         (pattern-match 
          (cdr pat) 
          (cdr dat)
          (pattern-match
           (car pat) (car dat) frame)))
        (else 'failed)))

လက်ရှိ frame ထဲတွင် ရှိပြီးသား binding များနှင့် ကိုက်ညီမှုရှိပါက binding အသစ်တစ်ခု ထည့်သွင်းခြင်းဖြင့် frame ကို တိုးချဲ့ပေးသော procedure မှာ အောက်ပါအတိုင်း ဖြစ်သည် - 

(define (extend-if-consistent var dat frame)
  (let ((binding (binding-in-frame var frame)))
    (if binding
        (pattern-match 
         (binding-value binding) dat frame)
        (extend var dat frame))))

အကယ်၍ frame ထဲတွင် variable အတွက် binding မရှိပါက၊ variable ကို data နှင့် bind လုပ်ခြင်းကို ရိုးရှင်းစွာ ထည့်သွင်းသည်။ သို့မဟုတ်ပါက frame ထဲရှိ variable ၏ တန်ဖိုးနှင့် data ကို frame ထဲတွင် တိုက်ဆိုင်စစ်ဆေးသည်။ အကယ်၍ သိမ်းဆည်းထားသော တန်ဖိုးတွင် ကိန်းသေ (constants) များသာ ပါဝင်ပါက ( extend-if-consistent ဖြင့် pattern matching လုပ်စဉ် သိမ်းဆည်းခဲ့လျှင် ထိုသို့ဖြစ်ရမည်)၊ ထိုအခါ တိုက်ဆိုင်စစ်ဆေးမှုသည် သိမ်းဆည်းထားသော တန်ဖိုးနှင့် တန်ဖိုးအသစ်တို့ တူညီခြင်း ရှိ၊ မရှိကိုသာ စမ်းသပ်သည်။ တူညီပါက မပြုပြင်ရသေးသော frame ကို ပြန်ပေးပြီး၊ မတူညီပါက မအောင်မြင်ကြောင်း ပြသသည်။ သို့သော် သိမ်းဆည်းထားသော တန်ဖိုးကို unification ( 4.4.4.4 ကို ကြည့်ပါ) လုပ်စဉ် သိမ်းဆည်းခဲ့ခြင်း ဖြစ်ပါက pattern variable များ ပါဝင်နေနိုင်သည်။ သိမ်းဆည်းထားသော pattern နှင့် data အသစ်တို့ကို recursive တိုက်ဆိုင်စစ်ဆေးခြင်းသည် ဤ pattern ထဲရှိ variable များအတွက် binding များကို ထည့်သွင်းခြင်း သို့မဟုတ် စစ်ဆေးခြင်း ပြုလုပ်လိမ့်မည်။ ဥပမာအားဖြင့်၊ ?x ကို (f ?y) နှင့် bind လုပ်ထားပြီး ?y သည် unbound ဖြစ်နေသော frame တစ်ခု ရှိသည် ဆိုပါစို့။ ကျွန်ုပ်တို့သည် ဤ frame ကို ?x နှင့် (f b) binding ဖြင့် ဖြည့်စွက်လိုသည်။ ကျွန်ုပ်တို့သည် ?x ကို ရှာဖွေသောအခါ ၎င်းကို (f ?y) နှင့် bind လုပ်ထားကြောင်း တွေ့ရသည်။ ၎င်းသည် ကျွန်ုပ်တို့အား (f ?y) နှင့် အဆိုပြုထားသော တန်ဖိုးအသစ် (f b) ကို ထို frame ထဲမှာပင် တိုက်ဆိုင်စစ်ဆေးစေသည်။ နောက်ဆုံးတွင် ဤတိုက်ဆိုင်စစ်ဆေးမှုသည် ?y ကို b နှင့် bind လုပ်ခြင်းဖြင့် frame ကို တိုးချဲ့သွားသည်။ ?X သည် (f ?y) နှင့် ဆက်လက် bind လုပ်ထားဆဲ ဖြစ်သည်။ ကျွန်ုပ်တို့သည် သိမ်းဆည်းထားသော binding တစ်ခုကို မမည်သည့်အခါမျှ မပြုပြင်သကဲ့သို့ ပေးထားသော variable တစ်ခုအတွက် binding တစ်ခုထက်ပို၍လည်း မည်သည့်အခါမျှ သိမ်းဆည်းခြင်း မရှိပါ။

Apply-rules သည် find-assertions (4.4.4.3) ၏ rule ဆိုင်ရာ အလားတူ လုပ်ဆောင်ချက်ဖြစ်သည်။ ၎င်းသည် input အဖြစ် pattern တစ်ခုနှင့် frame တစ်ခုကို ယူပြီး၊ data base ရှိ rule များကို အသုံးချခြင်းဖြင့် extension frame များ၏ stream တစ်ခုကို ဖွဲ့စည်းသည်။ Stream-flatmap သည် apply-a-rule ကို အသုံးချနိုင်ဖွယ်ရှိသော rule များ၏ stream (fetch-rules, 4.4.4.5 မှ ရွေးချယ်ထားသော) ပေါ်သို့ map လုပ်ပြီး၊ ရရှိလာသော frame stream များကို ပေါင်းစပ်ပေးသည်။ 

(define (apply-rules pattern frame)
  (stream-flatmap 
   (lambda (rule)
     (apply-a-rule rule pattern frame))
   (fetch-rules pattern frame)))

Apply-a-rule သည် 4.4.2 တွင် အကြမ်းဖျင်း ဖော်ပြထားသော နည်းလမ်းကို အသုံးပြု၍ rule များကို အသုံးချသည်။ ၎င်းသည် ပထမဦးစွာ rule conclusion ကို ပေးထားသော frame ထဲရှိ pattern နှင့် unify လုပ်ခြင်းဖြင့် ၎င်း၏ argument frame ကို ဖြည့်စွက်သည်။ အကယ်၍ ဤအဆင့် အောင်မြင်ပါက၊ ၎င်းသည် ဤ frame အသစ်ထဲတွင် rule body ကို evaluate လုပ်သည်။

သို့သော် ဤအရာများ မလုပ်ဆောင်မီ၊ program သည် rule ထဲရှိ variable အားလုံးကို ထူးခြားသော နာမည်အသစ်များဖြင့် နာမည်ပြောင်းလဲပေးသည်။ ဤသို့ ပြုလုပ်ရခြင်း အကြောင်းရင်းမှာ ကွဲပြားသော rule application များအတွက် variable များ အချင်းချင်း ရောထွေးသွားခြင်းကို တားဆီးရန် ဖြစ်သည်။ ဥပမာအားဖြင့်၊ rule နှစ်ခုလုံးက ?x ဟု အမည်ပေးထားသော variable ကို အသုံးပြုပါက၊ rule တစ်ခုချင်းစီကို အသုံးချသည့်အခါ frame ထဲသို့ ?x အတွက် binding တစ်ခုစီ ထည့်သွင်းနိုင်သည်။ ဤ ?x နှစ်ခုသည် အချင်းချင်း မည်သို့မျှ မသက်ဆိုင်ကြပါ၊ ထို့ကြောင့် ထို binding နှစ်ခုသည် ကိုက်ညီမှု (consistent) ရှိရမည်ဟု ကျွန်ုပ်တို့ မမှားယွင်းသင့်ပါ။ variable များကို နာမည်ပြောင်းမည့်အစား ပိုမိုကောင်းမွန်သော environment structure တစ်ခုကို ကျွန်ုပ်တို့ တီထွင်နိုင်သော်လည်း၊ ကျွန်ုပ်တို့ ဤနေရာတွင် ရွေးချယ်ထားသော နာမည်ပြောင်းလဲခြင်း ချဉ်းကပ်မှုသည် အထိရောက်ဆုံး (efficient) မဟုတ်သော်ငြားလည်း အရှင်းလင်းဆုံး ဖြစ်သည်။ (Exercise 4.79 ကို ကြည့်ပါ။) apply-a-rule procedure မှာ အောက်ပါအတိုင်း ဖြစ်သည် - 

(define (apply-a-rule rule
                      query-pattern
                      query-frame)
  (let ((clean-rule 
         (rename-variables-in rule)))
    (let ((unify-result
           (unify-match query-pattern
                        (conclusion clean-rule)
                        query-frame)))
      (if (eq? unify-result 'failed)
          the-empty-stream
          (qeval (rule-body clean-rule)
                 (singleton-stream 
                  unify-result))))))

rule တစ်ခု၏ အစိတ်အပိုင်းများကို ထုတ်ယူပေးသော selector များ ဖြစ်သည့် rule-body နှင့် conclusion တို့ကို 4.4.4.7 တွင် အဓိပ္ပာယ်ဖွင့်ဆိုထားပါသည်။

ကျွန်ုပ်တို့သည် rule application တစ်ခုချင်းစီအတွက် သီးခြား identifier (ဥပမာ ကိန်းဂဏန်းတစ်ခု) ကို သတ်မှတ်ပေးပြီး ၎င်း identifier ကို မူရင်း variable အမည်များနှင့် ပေါင်းစပ်ခြင်းဖြင့် သီးခြား variable အမည်များကို ထုတ်လုပ်ပေးပါသည်။ ဥပမာအားဖြင့်၊ rule-application identifier သည် 7 ဖြစ်ပါက rule ထဲရှိ ?x တစ်ခုချင်းစီကို ?x-7 အဖြစ်လည်းကောင်း၊ ?y တစ်ခုချင်းစီကို ?y-7 အဖြစ်လည်းကောင်း ပြောင်းလဲနိုင်ပါသည်။ (Make-new-variable နှင့် new-rule-application-id တို့ကို 4.4.4.7 ရှိ syntax procedures များတွင် ထည့်သွင်းထားပါသည်။) 

(define (rename-variables-in rule)
  (let ((rule-application-id 
         (new-rule-application-id)))
    (define (tree-walk exp)
      (cond ((var? exp)
             (make-new-variable 
              exp 
              rule-application-id))
            ((pair? exp)
             (cons (tree-walk (car exp))
                   (tree-walk (cdr exp))))
            (else exp)))
    (tree-walk rule)))

Unification algorithm ကို pattern နှစ်ခုနှင့် frame တစ်ခုကို input အဖြစ် လက်ခံပြီး၊ တိုးချဲ့ထားသော frame သို့မဟုတ် symbol failed ကို return ပြန်ပေးသည့် procedure တစ်ခုအဖြစ် အကောင်အထည်ဖော်ထားပါသည်။ Unifier သည် pattern matcher နှင့် ဆင်တူသော်လည်း ၎င်းသည် symmetrical ဖြစ်သည်—ဆိုလိုသည်မှာ match လုပ်ရာတွင် နှစ်ဖက်စလုံး၌ variable များ ရှိနေနိုင်ပါသည်။ Unify-match သည် အခြေခံအားဖြင့် pattern-match နှင့် အတူတူပင်ဖြစ်သော်လည်း၊ match ၏ ညာဘက်ခြမ်းရှိ object မှာ variable တစ်ခုဖြစ်နေသည့် အခြေအနေကို ကိုင်တွယ်ရန်အတွက် အပို code (အောက်တွင် “***” ဟု အမှတ်အသားပြုထားသည်) ပါရှိပါသည်။ 

(define (unify-match p1 p2 frame)
  (cond ((eq? frame 'failed) 'failed)
        ((equal? p1 p2) frame)
        ((var? p1)
         (extend-if-possible p1 p2 frame))
        ((var? p2)
         (extend-if-possible 
          p2 
          p1 
          frame))        ; ***
        ((and (pair? p1) 
              (pair? p2))
         (unify-match 
          (cdr p1) 
          (cdr p2)
          (unify-match 
           (car p1)
           (car p2)
           frame)))
        (else 'failed)))

Unification တွင်လည်း one-sided pattern matching မှာကဲ့သို့ပင်၊ အဆိုပြုထားသော frame ၏ extension သည် လက်ရှိ bindings များနှင့် ကိုက်ညီမှုရှိမှသာ လက်ခံလိုပါသည်။ Unification တွင် အသုံးပြုသည့် extend-if-possible procedure သည် pattern matching တွင် အသုံးပြုသည့် extend-if-consistent နှင့် အတူတူပင် ဖြစ်သော်လည်း အောက်ပါ program တွင် “***” ဟု အမှတ်အသားပြုထားသော အထူးစစ်ဆေးချက် နှစ်ခု ပါဝင်ပါသည်။ ပထမအခြေအနေမှာ၊ ကျွန်ုပ်တို့ match လုပ်ရန် ကြိုးစားနေသော variable သည် bound မဖြစ်သေးသော်လည်း၊ ၎င်းနှင့် match လုပ်ရန် ကြိုးစားနေသည့် တန်ဖိုးကိုယ်တိုင်မှာ (အခြား) variable တစ်ခု ဖြစ်နေပါက၊ ထိုတန်ဖိုးသည် bound ဖြစ်နေခြင်း ရှိမရှိ စစ်ဆေးရန် လိုအပ်ပြီး၊ ဖြစ်နေပါက ၎င်း၏တန်ဖိုးနှင့် match လုပ်ရမည်ဖြစ်သည်။ အကယ်၍ match လုပ်မည့် အစိတ်အပိုင်း နှစ်ခုစလုံးသည် unbound ဖြစ်နေပါက၊ တစ်ခုကို တစ်ခုနှင့် bind လုပ်နိုင်ပါသည်။

ဒုတိယစစ်ဆေးချက်မှာ variable တစ်ခုကို ၎င်း variable ပါဝင်နေသည့် pattern တစ်ခုနှင့် bind လုပ်ရန် ကြိုးပမ်းမှုကို ကိုင်တွယ်ခြင်း ဖြစ်သည်။ ဤသို့သော အခြေအနေမျိုးသည် pattern နှစ်ခုလုံးတွင် variable တစ်ခုတည်း ထပ်ခါတလဲလဲ ပါဝင်နေသည့်အခါမျိုးတွင် ဖြစ်ပေါ်နိုင်သည်။ ဥပမာအားဖြင့်၊ ?x နှင့် ?y နှစ်ခုလုံး unbound ဖြစ်နေသော frame တစ်ခုတွင် (?x ?x) နှင့် (?y ⟨?y ပါဝင်သော expression⟩) ဟူသော pattern နှစ်ခုကို unify လုပ်သည်ဟု ဆိုပါစို့။ ပထမဦးစွာ ?x ကို ?y နှင့် match လုပ်ပြီး ?x ကို ?y နှင့် binding ပြုလုပ်သည်။ ထို့နောက် တူညီသော ?x ကိုပင် ပေးထားသော ?y ပါဝင်သည့် expression နှင့် match လုပ်သည်။ ?x သည် ?y နှင့် bound ဖြစ်ပြီးသား ဖြစ်သောကြောင့်၊ ရလဒ်အနေဖြင့် ?y ကို ထို expression နှင့် match လုပ်ခြင်း ဖြစ်သွားသည်။ အကယ်၍ unifier ကို pattern များ တူညီသွားစေရန် pattern variable များအတွက် တန်ဖိုးများ ရှာဖွေပေးခြင်းဟု ယူဆပါက၊ ဤ pattern များသည် ?y သည် ၎င်း ?y ပါဝင်သော expression နှင့် တူညီစေမည့် ?y ကို ရှာဖွေရန် ညွှန်ကြားချက်များဟု ဆိုလိုရာရောက်သည်။ ထိုသို့သော equation များကို ဖြေရှင်းရန် ယေဘုယျနည်းလမ်း မရှိသောကြောင့် ကျွန်ုပ်တို့သည် ထိုသို့သော binding များကို ပယ်ချပါသည်။ ဤအခြေအနေများကို depends-on? predicate ဖြင့် ခွဲခြားသိရှိနိုင်ပါသည်။284 အခြားတစ်ဖက်တွင်မူ variable တစ်ခုကို ၎င်းကိုယ်တိုင်နှင့် ပြန်လည် bind လုပ်ရန် ကြိုးပမ်းမှုများကို ကျွန်ုပ်တို့ မပယ်ချလိုပါ။ ဥပမာအားဖြင့် (?x ?x) နှင့် (?y ?y) ကို unify လုပ်သည်ကို ကြည့်ပါ။ ?x ကို ?y နှင့် ဒုတိယအကြိမ် bind လုပ်ရန် ကြိုးပမ်းမှုသည် ?y (သိမ်းဆည်းထားသော ?x ၏ တန်ဖိုး) ကို ?y (?x ၏ တန်ဖိုးသစ်) နှင့် match လုပ်ခြင်း ဖြစ်သည်။ ဤအချက်ကို unify-match ၏ equal? clause ဖြင့် ကိုင်တွယ်ဖြေရှင်းထားပါသည်။ 

(define (extend-if-possible var val frame)
  (let ((binding (binding-in-frame var frame)))
    (cond (binding
           (unify-match
            (binding-value binding) val frame))
          ((var? val)                   ; ***
           let ((binding 
                  (binding-in-frame 
                   val
                   frame)))
             (if binding
                 (unify-match
                  var 
                  (binding-value binding) 
                  frame)
                 (extend var val frame))))
          ((depends-on? val var frame)  ; ***
           'failed)
          (else (extend var val frame)))))

(define (depends-on? exp var frame)
  (define (tree-walk e)
    (cond ((var? e)
           (if (equal? var e)
               true
               (let
                 ((b (binding-in-frame 
                      e 
                      frame)))
                  (if b
                      (tree-walk 
                       (binding-value b))
                      false))))
          ((pair? e)
           (or (tree-walk (car e))
               (tree-walk (cdr e))))
          (else false)))
  (tree-walk exp))

ဖြစ်ပြီး qeval ဟာ type (car) က unique ဆိုတဲ့ symbol ဖြစ်တဲ့ query တိုင်းအတွက် ဒီ procedure ဆီကို dispatch လုပ်ပေးပါလိမ့်မယ်။

တကယ့်အခက်အခဲက procedure uniquely-asserted ကို ရေးဖို့ ဖြစ်ပါတယ်။ ဒါဟာ unique query ရဲ့ contents (cdr) နဲ့အတူ frames stream တစ်ခုကို input အဖြစ် ယူရပါမယ်။ stream ထဲက frame တစ်ခုချင်းစီအတွက်၊ ပေးထားတဲ့ query နဲ့ ကိုက်ညီတဲ့ frame ရဲ့ extension stream အားလုံးကို ရှာဖို့ qeval ကို အသုံးပြုရပါမယ်။ item အတိအကျ တစ်ခုတည်း မရှိတဲ့ stream တွေကို ဖယ်ထုတ်ပစ်ရပါမယ်။ ကျန်ရှိတဲ့ stream တွေကို unique query ရဲ့ ရလဒ်ဖြစ်တဲ့ big stream တစ်ခုအဖြစ် စုစည်းဖို့ ပြန်ပေးရပါမယ်။ ဒါဟာ not special form ရဲ့ အကောင်အထည်ဖော်မှုနဲ့ ဆင်တူပါတယ်။

လူတစ်ဦးတည်းကိုသာ ကြီးကြပ်ကွပ်ကဲတဲ့ လူအားလုံးကို စာရင်းပြုစုတဲ့ query တစ်ခု ဖန်တီးပြီး သင့်ရဲ့ အကောင်အထည်ဖော်မှုကို စမ်းသပ်ပါ။

လေ့ကျင့်ခန်း ၄.၇၆: queries တွေကို series အဖြစ် ပေါင်းစပ်ထားတဲ့ ကျွန်ုပ်တို့ရဲ့ and အကောင်အထည်ဖော်မှုဟာ (ပုံ ၄.၅) ပြေပြစ်လှပပေမဲ့ ထိရောက်မှု မရှိပါဘူး။ ဘာကြောင့်လဲဆိုတော့ and ရဲ့ ဒုတိယ query ကို လုပ်ဆောင်တဲ့အခါ ပထမ query က ထုတ်ပေးတဲ့ frame တစ်ခုချင်းစီအတွက် data base ကို scan ဖတ်ရလို့ ဖြစ်ပါတယ်။ အကယ်၍ data base မှာ element n ခု ရှိပြီး၊ ပုံမှန် query တစ်ခုက n နဲ့ အချိုးကျတဲ့ output frames အရေအတွက် (ဥပမာ n / k ) ထုတ်ပေးတယ်ဆိုရင်၊ ပထမ query က ထုတ်ပေးတဲ့ frame တစ်ခုချင်းစီအတွက် data base ကို scan ဖတ်ဖို့ pattern matcher ကို n 2 / k ကြိမ် ခေါ်ဆိုဖို့ လိုအပ်ပါလိမ့်မယ်။ နောက်ထပ် ချဉ်းကပ်မှုတစ်ခုကတော့ and ရဲ့ အပိုင်းနှစ်ခုကို သီးခြားစီ လုပ်ဆောင်ပြီး၊ ကိုက်ညီမှုရှိတဲ့ (compatible ဖြစ်တဲ့) output frame pair အားလုံးကို ရှာဖွေဖို့ ဖြစ်ပါတယ်။ အကယ်၍ query တစ်ခုချင်းစီက n / k output frames တွေ ထုတ်ပေးမယ်ဆိုရင်၊ compatibility စစ်ဆေးမှု n 2 / k 2 ကြိမ် ပြုလုပ်ရမှာ ဖြစ်ပါတယ်။ ဒါဟာ လက်ရှိနည်းလမ်းမှာ လိုအပ်တဲ့ matches အရေအတွက်ထက် factor k ပမာဏ ပိုနည်းပါတယ်။

ဒီဗျူဟာကို အသုံးပြုတဲ့ and အကောင်အထည်ဖော်မှုတစ်ခုကို ကြံဆပါ။ frames နှစ်ခုကို input အဖြစ် ယူပြီး၊ frame တွေထဲက bindings တွေဟာ compatible ဖြစ်မဖြစ် စစ်ဆေးကာ၊ ဖြစ်တယ်ဆိုရင် bindings အစုနှစ်ခုကို ပေါင်းစပ်ထားတဲ့ frame တစ်ခု ထုတ်ပေးတဲ့ procedure တစ်ခုကို သင် အကောင်အထည်ဖော်ရပါမယ်။ ဒီလုပ်ဆောင်ချက်ဟာ unification နဲ့ ဆင်တူပါတယ်။

လေ့ကျင့်ခန်း ၄.၇၇: ၄.၄.၃ မှာ variables တွေဟာ unbound ဖြစ်နေတဲ့ frames တွေမှာ filtering operations တွေကို အသုံးချမယ်ဆိုရင် not နဲ့ lisp-value တို့က query language ကို "မှားယွင်းတဲ့" အဖြေတွေ ပေးနိုင်တာကို တွေ့ခဲ့ရပါတယ်။ ဒီအားနည်းချက်ကို ပြင်ဆင်ဖို့ နည်းလမ်းတစ်ခု ကြံဆပါ။ စိတ်ကူးတစ်ခုကတော့ operation ကို လုပ်ဆောင်နိုင်ဖို့ လုံလောက်တဲ့ variables တွေ bound ဖြစ်သွားတဲ့အခါမှသာ ဖြည့်ဆည်းပေးမယ့် "promise" တစ်ခုကို frame မှာ တွဲပေးခြင်းအားဖြင့် filtering ကို "နှောင့်နှေးသော (delayed)" နည်းလမ်းနဲ့ လုပ်ဆောင်ဖို့ ဖြစ်ပါတယ်။ အခြား operations အားလုံး လုပ်ဆောင်ပြီးတဲ့အထိ filtering ကို စောင့်ဆိုင်းနိုင်ပါတယ်။ ဒါပေမဲ့ ထိရောက်မှုရှိစေဖို့အတွက် ကြားခံ frames အရေအတွက်ကို လျှော့ချနိုင်ဖို့ filtering ကို တတ်နိုင်သမျှ မြန်မြန် လုပ်ဆောင်လိုပါတယ်။

လေ့ကျင့်ခန်း ၄.၇၈: query language ကို stream process အဖြစ် မဟုတ်ဘဲ၊ ၄.၃ ရဲ့ evaluator ကို အသုံးပြုပြီး အကောင်အထည်ဖော်မယ့် nondeterministic program တစ်ခုအဖြစ် ပြန်လည်ဒီဇိုင်းထုတ်ပါ။ ဒီချဉ်းကပ်မှုမှာ query တစ်ခုချင်းစီက (အဖြေအားလုံးရဲ့ stream အစား) အဖြေတစ်ခုတည်းကိုသာ ထုတ်ပေးမှာဖြစ်ပြီး အဖြေတွေ ထပ်မံကြည့်ရှုဖို့ user က try-again လို့ ရိုက်နှိပ်နိုင်ပါတယ်။ ဒီအပိုင်းမှာ ကျွန်ုပ်တို့ တည်ဆောက်ခဲ့တဲ့ ယန္တရား အတော်များများဟာ nondeterministic search နဲ့ backtracking တို့ရဲ့အောက်မှာ အကျုံးဝင်သွားတာကို သင် တွေ့ရပါလိမ့်မယ်။ ဒါပေမဲ့ သင့်ရဲ့ query language အသစ်ဟာ ဒီမှာ အကောင်အထည်ဖော်ထားတာနဲ့ အပြုအမူပိုင်းမှာ အနည်းငယ် ကွာခြားချက်တွေ ရှိနေတာကိုလည်း သင် တွေ့နိုင်ပါတယ်။ ဒီကွာခြားချက်ကို ဖော်ပြတဲ့ ဥပမာတွေကို သင် ရှာတွေ့နိုင်သလား?

လေ့ကျင့်ခန်း ၄.၇၉: ၄.၁ မှာ Lisp evaluator ကို အကောင်အထည်ဖော်တဲ့အခါ procedures တွေရဲ့ parameters တွေကြားမှာ နာမည်ပဋိပက္ခ (name conflicts) တွေကို ရှောင်ရှားဖို့ local environments တွေကို ဘယ်လိုသုံးရမလဲဆိုတာ တွေ့ခဲ့ရပါတယ်။ ဥပမာအားဖြင့်၊ 

(define (square x) 
  (* x x))

(define (sum-of-squares x y)
  (+ (square x) (square y)))

(sum-of-squares 3 4)

ကို evaluate လုပ်တဲ့အခါ၊ square ထဲက x နဲ့ sum-of-squares ထဲက x ကြားမှာ ရှုပ်ထွေးမှု မရှိပါဘူး။ ဘာကြောင့်လဲဆိုတော့ procedure တစ်ခုချင်းစီရဲ့ body ကို local variables တွေအတွက် bindings တွေ ပါဝင်အောင် အထူးတည်ဆောက်ထားတဲ့ environment တစ်ခုထဲမှာ evaluate လုပ်လို့ ဖြစ်ပါတယ်။ query system မှာတော့ rules တွေကို အသုံးချတဲ့အခါ နာမည်ပဋိပက္ခတွေကို ရှောင်ရှားဖို့ မတူညီတဲ့ ဗျူဟာတစ်ခုကို အသုံးပြုခဲ့ပါတယ်။ rule တစ်ခုကို အသုံးချတိုင်း variables တွေကို သီးသန့်ဖြစ်မယ်လို့ အာမခံချက်ရှိတဲ့ နာမည်အသစ်တွေနဲ့ ပြန်ပြီး နာမည်ပေးပါတယ်။ Lisp evaluator အတွက် အလားတူဗျူဟာကတော့ local environments တွေကို ဖယ်ရှားပြီး procedure တစ်ခုကို အသုံးချတိုင်း body ထဲက variables တွေကို နာမည်အသစ်ပေးတာပဲ ဖြစ်ပါလိမ့်မယ်။

renaming အစား environments ကို အသုံးပြုတဲ့ rule-application နည်းလမ်းကို query language အတွက် အကောင်အထည်ဖော်ပါ။ block-structured procedures ရဲ့ rule analog ကဲ့သို့သော စနစ်ကြီးတွေကို ကိုင်တွယ်ဖို့ constructs တွေကို query language မှာ ဖန်တီးဖို့ သင့်ရဲ့ environment structure ပေါ်မှာ အခြေခံပြီး တည်ဆောက်နိုင်မလား ကြည့်ပါ။ ဒါတွေကို context တစ်ခုမှာ အနုမာနနည်းနဲ့ ကောက်ချက်ချခြင်း (deductions) ပြုလုပ်တဲ့ ပြဿနာနဲ့ သင် ဆက်စပ်နိုင်မလား? (ဥပမာ "အကယ်၍ P က မှန်ကန်တယ်လို့ ယူဆရင်၊ ကျွန်ုပ် A နဲ့ B ကို ကောက်ချက်ချနိုင်ပါလိမ့်မယ်။") (ဒီပုစ္ဆာက အဆုံးသတ်မရှိ (open-ended) ဖြစ်ပါတယ်။ ကောင်းမွန်တဲ့ အဖြေတစ်ခုဟာ Ph.D. ဘွဲ့တစ်ခုနဲ့ ထိုက်တန်နိုင်ပါတယ်။)

262 Logic programming သည် အလိုအလျောက် သီအိုရမ် သက်သေပြခြင်း (automatic theorem proving) ဆိုင်ရာ သုတေသန၏ ရှည်လျားသော သမိုင်းကြောင်းမှ ပေါ်ထွက်လာခြင်း ဖြစ်သည်။ အစောပိုင်း သီအိုရမ် သက်သေပြသည့် ပရိုဂရမ်များသည် ဖြစ်နိုင်ခြေရှိသော သက်သေပြချက်အားလုံးကို အကုန်အစင် ရှာဖွေသောကြောင့် အောင်မြင်မှု အနည်းငယ်သာ ရရှိခဲ့သည်။ ထိုသို့ ရှာဖွေမှုကို ဖြစ်နိုင်ခြေရှိစေခဲ့သော အဓိက အောင်မြင်မှုမှာ ၁၉၆၀ ပြည့်လွန်နှစ်များ အစောပိုင်းတွင် ရှာဖွေတွေ့ရှိခဲ့သော unification algorithm နှင့် resolution principle (Robinson 1965) တို့ ဖြစ်သည်။ ဥပမာအားဖြင့် Resolution ကို Green and Raphael (1968) ( Green 1969 ကိုလည်း ကြည့်ပါ) က deductive question-answering system တစ်ခုအတွက် အခြေခံအဖြစ် အသုံးပြုခဲ့သည်။ ဤကာလအများစုတွင် သုတေသီများသည် သက်သေပြချက် ရှိပါက ၎င်းကို ရှာဖွေတွေ့ရှိရန် အာမခံချက်ပေးနိုင်သော algorithm များအပေါ် အာရုံစိုက်ခဲ့ကြသည်။ ထိုသို့သော algorithm များသည် ထိန်းချုပ်ရန်နှင့် သက်သေပြချက်တစ်ခုဆီသို့ ဦးတည်ရန် ခက်ခဲခဲ့သည်။ Hewitt (1969) သည် programming language တစ်ခု၏ control structure ကို logic-manipulation system ၏ လုပ်ဆောင်ချက်များနှင့် ပေါင်းစပ်နိုင်ခြေကို သတိပြုမိခဲ့ပြီး၊ ၎င်းသည် 4.3.1 (Footnote 250) တွင် ဖော်ပြထားသော automatic search ဆိုင်ရာ လုပ်ဆောင်ချက်များဆီသို့ ဦးတည်သွားစေခဲ့သည်။ ဤသို့ လုပ်ဆောင်နေချိန်၌ပင် Marseille ရှိ Colmerauer သည် natural language ကို ကိုင်တွယ်ရန် rule-based system များကို ဖော်ဆောင်နေခဲ့သည် ( Colmerauer et al. 1973 ကို ကြည့်ပါ)။ သူသည် ထိုစည်းမျဉ်းများကို ကိုယ်စားပြုရန် Prolog ဟုခေါ်သော programming language တစ်ခုကို တီထွင်ခဲ့သည်။ Kowalski (1973; 1979)၊ Edinburgh ၌ Prolog ပရိုဂရမ်တစ်ခုအား အကောင်အထည်ဖော်ခြင်းကို သီအိုရမ်များကို သက်သေပြခြင်း (linear Horn-clause resolution ဟုခေါ်သော သက်သေပြခြင်း နည်းစနစ်ကို အသုံးပြု၍) အဖြစ် အဓိပ္ပာယ်ဖွင့်ဆိုနိုင်ကြောင်း သတိပြုမိခဲ့သည်။ နောက်ဆုံးအချက်နှစ်ခု ပေါင်းစပ်မှုသည် logic-programming လှုပ်ရှားမှုဆီသို့ ဦးတည်သွားစေခဲ့သည်။ ထို့ကြောင့် logic programming ဖွံ့ဖြိုးတိုးတက်မှုအတွက် ဂုဏ်ပြုရာတွင် ပြင်သစ်တို့က University of Marseille မှ Prolog ၏ အစပျိုးမှုကို ညွှန်ပြနိုင်ပြီး၊ ဗြိတိသျှတို့က University of Edinburgh ရှိ လုပ်ဆောင်ချက်များကို အသားပေး ဖော်ပြနိုင်ကြသည်။ MIT မှ လူများ၏ ပြောကြားချက်အရ logic programming ကို ဤအုပ်စုများက Hewitt ၏ ထက်မြက်သော်လည်း နားလည်ရန်ခက်ခဲလှသော Ph.D. စာတမ်းတွင် ပြောဆိုထားသည်များကို နားလည်သဘောပေါက်ရန် ကြိုးပမ်းရင်း ဖော်ဆောင်ခဲ့ခြင်း ဖြစ်သည်။ Logic programming ၏ သမိုင်းကြောင်းအတွက် Robinson 1983 ကို ကြည့်ပါ။

263 စည်းမျဉ်းများ (rules) နှင့် လုပ်ထုံးလုပ်နည်း (procedure) အကြား ဆက်စပ်မှုကို သိနိုင်ရန်၊ procedure ထဲရှိ x ( x သည် အလွတ်မဟုတ်သောအခါ) ကို rule ထဲရှိ (cons u v) နှင့် တူညီသည်ဟု သတ်မှတ်ပါ။ ထို့နောက် rule ထဲရှိ z သည် (cdr x) နှင့် y တို့ကို append လုပ်ခြင်းနှင့် တူညီသည်။

264 ၎င်းသည် အဖြေကို မည်သို့ တွက်ချက်ရမည်ဟူသော ပြဿနာတစ်ခုလုံးမှ အသုံးပြုသူကို သက်သာရာမရစေသည်မှာ အမှန်ပင်ဖြစ်သည်။ append relation ကို ဖော်ဆောင်ရန် သင်္ချာနည်းအရ တူညီသော rule set များစွာ ရှိသော်လည်း၊ အချို့ကိုသာ မည်သည့်ဘက်ကမဆို တွက်ချက်နိုင်သော ထိရောက်သည့် ကိရိယာများအဖြစ် ပြောင်းလဲနိုင်သည်။ ထို့အပြင်၊ တစ်ခါတစ်ရံတွင် “ဘာဖြစ်သလဲ” (what is) ဟူသော အချက်အလက်သည် အဖြေကို “မည်သို့” (how to) တွက်ချက်ရမည်ဆိုသည့် လမ်းစကို မပေးနိုင်ပါ။ ဥပမာအားဖြင့် y ကို တွက်ချက်သည့် ပြဿနာကို စဉ်းစားကြည့်ပါ၊ ၎င်းမှာ y 2 = x ဖြစ်သည်။

265 ဂျပန်အစိုးရသည် logic programming language များကို မောင်းနှင်ရန် အကောင်းဆုံးဖြစ်အောင် ပြုလုပ်ထားသော အလွန်မြန်သည့် ကွန်ပျူတာများ တည်ဆောက်ရန် ရည်မှန်းချက်ကြီးမားသော ပရောဂျက်တစ်ခုကို စတင်ခဲ့သည့် ၈၀ ပြည့်လွန်နှစ်များ အစောပိုင်းတွင် logic programming အပေါ် စိတ်ဝင်စားမှု အထွတ်အထိပ်သို့ ရောက်ရှိခဲ့သည်။ ထိုသို့သော ကွန်ပျူတာများ၏ အမြန်နှုန်းကို ပုံမှန် FLOPS (FLoating-point Operations Per Second) ထက် LIPS (Logical Inferences Per Second) ဖြင့် တိုင်းတာရန် ဖြစ်သည်။ ပရောဂျက်သည် မူလစီစဉ်ထားသည့်အတိုင်း hardware နှင့် software များကို ဖော်ဆောင်ရာတွင် အောင်မြင်ခဲ့သော်လည်း၊ နိုင်ငံတကာ ကွန်ပျူတာလုပ်ငန်းသည် အခြားလမ်းကြောင်းတစ်ခုသို့ ကူးပြောင်းသွားခဲ့သည်။ ဂျပန်ပရောဂျက်၏ ခြုံငုံသုံးသပ်ချက် အကဲဖြတ်မှုအတွက် Feigenbaum and Shrobe 1993 ကို ကြည့်ပါ။ Logic programming အသိုင်းအဝိုင်းသည် ရိုးရှင်းသော pattern matching ထက် အခြားသော နည်းစနစ်များအပေါ် အခြေခံသည့် relational programming ကိုလည်း စတင်စဉ်းစားလာခဲ့ကြသည်။ ဥပမာအားဖြင့် 3.3.5 ၏ constraint-propagation system တွင် ဖော်ပြထားသော numerical constraints များကို ကိုင်တွယ်နိုင်စွမ်းကဲ့သို့သော အချက်များ ဖြစ်သည်။

267 အမှန်စင်စစ်၊ not နှင့် ပတ်သက်သော ဤဖော်ပြချက်သည် ရိုးရှင်းသော ကိစ္စများအတွက်သာ မှန်ကန်ပါသည်။ not ၏ တကယ့် လုပ်ဆောင်ပုံမှာ ပိုမို ရှုပ်ထွေးပါသည်။ ကျွန်ုပ်တို့သည် not ၏ ထူးခြားချက်များကို 4.4.2 နှင့် 4.4.3 တို့တွင် လေ့လာပါမည်။

268 Lisp-value ကို query language တွင် မပါဝင်သော operation တစ်ခုကို လုပ်ဆောင်ရန်အတွက်သာ အသုံးပြုသင့်သည်။ အထူးသဖြင့်၊ တူညီမှုကို စစ်ဆေးရန် (၎င်းမှာ query language ရှိ matching ကို ရည်ရွယ်၍ ပြုလုပ်ထားသောကြောင့်ဖြစ်သည်) သို့မဟုတ် မတူညီမှုကို စစ်ဆေးရန် (၎င်းကို အောက်တွင် ပြထားသော same rule ဖြင့် လုပ်ဆောင်နိုင်သောကြောင့်ဖြစ်သည်) အတွက် အသုံးမပြုသင့်ပါ။

269 အရာဝတ္ထုနှစ်ခုကို တူညီစေရန် same ကို မလိုအပ်ကြောင်း သတိပြုပါ- ကျွန်ုပ်တို့သည် တစ်ခုချင်းစီအတွက် တူညီသော pattern variable ကိုသာ အသုံးပြုသည် - ထိုအခါ အမှန်တကယ်တွင် အစကတည်းက အရာနှစ်ခုအစား တစ်ခုတည်းသာ ရှိနေခြင်း ဖြစ်သည်။ ဥပမာအားဖြင့်၊ အောက်ပါ lives-near rule ထဲရှိ ?town နှင့် wheel rule ထဲရှိ ?middle-manager ကို ကြည့်ပါ။ Same သည် lives-near rule ထဲရှိ ?person-1 နှင့် ?person-2 ကဲ့သို့သော အရာနှစ်ခုကို မတူညီစေရန် ဖိအားပေးလိုသောအခါ အသုံးဝင်သည်။ Query တစ်ခု၏ အစိတ်အပိုင်းနှစ်ခုတွင် တူညီသော pattern variable ကို အသုံးပြုခြင်းသည် နေရာနှစ်ခုလုံးတွင် တူညီသော တန်ဖိုးတစ်ခု ပေါ်လာစေရန် ဖိအားပေးသော်လည်း၊ မတူညီသော pattern variable များကို အသုံးပြုခြင်းသည် မတူညီသော တန်ဖိုးများ ပေါ်လာရန် ဖိအားမပေးပါ။ (မတူညီသော pattern variable များအတွက် သတ်မှတ်ထားသော တန်ဖိုးများသည် တူညီနိုင်သလို မတူညီဘဲလည်း ရှိနိုင်ပါသည်။)

270 same တွင်ကဲ့သို့ body မပါသော rule များကိုလည်း ခွင့်ပြုမည်ဖြစ်ပြီး၊ ထိုသို့သော rule ကို ၎င်း၏ rule conclusion သည် variable များ၏ မည်သည့်တန်ဖိုးဖြင့်မဆို ကိုက်ညီသည်ဟု အဓိပ္ပာယ်ဖွင့်ဆိုပါမည်။

271 ယေဘုယျအားဖြင့် matching သည် အလွန် အရင်းအမြစ်ကုန်ကျသောကြောင့် (expensive)၊ database ၏ အစိတ်အပိုင်းတိုင်းတွင် အပြည့်အဝ matcher ကို အသုံးပြုခြင်းကို ရှောင်ရှားလိုပါသည်။ ၎င်းကို ပုံမှန်အားဖြင့် လုပ်ငန်းစဉ်ကို မြန်ဆန်ပြီး အကြမ်းဖျင်း match လုပ်ခြင်းနှင့် နောက်ဆုံး match လုပ်ခြင်းဟူ၍ ခွဲခြားခြင်းဖြင့် လုပ်ဆောင်သည်။ အကြမ်းဖျင်း match လုပ်ခြင်းသည် နောက်ဆုံး match လုပ်ရန်အတွက် လျာထားချက် (candidates) အနည်းငယ်ကို ထုတ်ပေးရန် database ကို စစ်ထုတ်ပေးသည်။ ဂရုတစိုက် ဆောင်ရွက်ခြင်းဖြင့်၊ အကြမ်းဖျင်း match လုပ်ခြင်း၏ အချို့သော လုပ်ဆောင်ချက်များကို candidates များကို ရွေးချယ်လိုသည့် အချိန်တွင် မဟုတ်ဘဲ database ကို တည်ဆောက်ချိန်တွင်ပင် လုပ်ဆောင်နိုင်ရန် စီစဉ်နိုင်သည်။ ယင်းကို indexing the data base ဟု ခေါ်သည်။ Database-indexing စနစ်များတွင် ကျယ်ပြန့်သော နည်းပညာများစွာ ရှိပါသည်။ 4.4.4 တွင် ဖော်ပြထားသော ကျွန်ုပ်တို့၏ အကောင်အထည်ဖော်မှုတွင် ထိုသို့သော အကောင်းဆုံးဖြစ်အောင် ပြုလုပ်ခြင်း (optimization) ၏ ရိုးရှင်းသော ပုံစံတစ်ခု ပါဝင်သည်။

272 သို့သော် ဤကဲ့သို့သော ထပ်ညွှန်းကိန်းအတိုင်း ပေါက်ကွဲတိုးပွားခြင်း (exponential explosion) သည် and query များတွင် အဖြစ်မများပါ၊ အကြောင်းမှာ ထပ်လောင်းထည့်လိုက်သော အခြေအနေများသည် frame အရေအတွက်ကို တိုးပွားစေမည့်အစား လျှော့ချပေးလေ့ရှိသောကြောင့် ဖြစ်သည်။

273 ရှုပ်ထွေးသော query များကို မည်သို့ ထိရောက်စွာ ကိုင်တွယ်ရမည်ကို လေ့လာထားသော data-base-management system များဆိုင်ရာ စာပေအမြောက်အမြား ရှိပါသည်။

274 ဤ filter ဖြင့် အကောင်အထည်ဖော်ထားသော not နှင့် သင်္ချာဆိုင်ရာ logic ရှိ not ၏ ပုံမှန် အဓိပ္ပာယ်တို့အကြား သိမ်မွေ့သော ကွာခြားချက် ရှိသည်။ 4.4.3 ကို ကြည့်ပါ။

275 One-sided pattern matching တွင်၊ pattern variable များ ပါဝင်သော equation များအားလုံးသည် ရှင်းလင်းပြီး မသိကိန်း (pattern variable) အတွက် တွက်ချက်ပြီးသား ဖြစ်သည်။

276 Unification ကို စဉ်းစားနိုင်သည့် နောက်တစ်နည်းမှာ ၎င်းသည် input pattern နှစ်ခု၏ specialization ဖြစ်သော အထွေထွေအကျဆုံး pattern ကို ထုတ်ပေးခြင်း ဖြစ်သည်။ ဆိုလိုသည်မှာ (?x a) နှင့် ((b ?y) ?z) တို့၏ unification သည် ((b ?y) a) ဖြစ်ပြီး၊ အထက်တွင် ဆွေးနွေးခဲ့သော (?x a ?y) နှင့် (?y ?z a) တို့၏ unification မှာ (a a a) ဖြစ်သည်။ ကျွန်ုပ်တို့၏ အကောင်အထည်ဖော်မှုအတွက် unification ၏ ရလဒ်ကို pattern တစ်ခုအဖြစ် မဟုတ်ဘဲ frame တစ်ခုအဖြစ် စဉ်းစားခြင်းက ပို၍ အဆင်ပြေသည်။

277 Unification သည် matching ကို ယေဘုယျပြုထားခြင်း ဖြစ်သောကြောင့်၊ stream နှစ်ခုလုံးကို ထုတ်ပေးရန် unifier ကို အသုံးပြုခြင်းဖြင့် စနစ်ကို ပိုမိုရိုးရှင်းစေနိုင်သည်။ သို့သော် လွယ်ကူသော ကိစ္စကို simple matcher ဖြင့် ကိုင်တွယ်ခြင်းက (အပြည့်အဝ unification နှင့် မတူဘဲ) matching သည် ၎င်း၏ ကိုယ်ပိုင်နည်းလမ်းဖြင့် မည်သို့ အသုံးဝင်နိုင်ကြောင်းကို ဖော်ပြပေးသည်။

278 ကျွန်ုပ်တို့သည် frame များ၏ (စာရင်းများထက်) stream များကို အသုံးပြုရခြင်း၏ အကြောင်းရင်းမှာ rule များကို recursive နည်းအရ အသုံးပြုခြင်းက query တစ်ခုနှင့် ကိုက်ညီသော တန်ဖိုးများကို အကန့်အသတ်မရှိ ထုတ်ပေးနိုင်သောကြောင့် ဖြစ်သည်။ Stream များတွင် ပါဝင်သော delayed evaluation သည် ဤနေရာတွင် အလွန်အရေးကြီးပါသည်- တုံ့ပြန်မှု အရေအတွက်သည် အကန့်အသတ်ရှိသည်ဖြစ်စေ၊ မရှိသည်ဖြစ်စေ စနစ်သည် ၎င်းတို့ကို ထွက်ပေါ်လာသည်နှင့်အမျှ တစ်ခုပြီးတစ်ခု ရိုက်နှိပ်ပေးမည် ဖြစ်သည်။

279 သီးခြား ကောက်ချက်ချမှု နည်းလမ်း (method of inference) တစ်ခုသည် တရားဝင်သည်ဟု ဆိုခြင်းမှာ သာမန်ပြောဆိုချက် မဟုတ်ပါ။ မှန်ကန်သော အချက်အလက်များမှ စတင်လျှင် မှန်ကန်သော ကောက်ချက်များကိုသာ ရရှိနိုင်ကြောင်း သက်သေပြရမည် ဖြစ်သည်။ Rule များကို အသုံးပြုခြင်းဖြင့် ကိုယ်စားပြုသော ကောက်ချက်ချနည်းမှာ modus ponens ဖြစ်သည်။ ၎င်းမှာ A မှန်ကန်ပြီး A implies B လည်း မှန်ကန်ပါက B သည် မှန်ကန်သည်ဟု ကောက်ချက်ချနိုင်သော ရင်းနှီးပြီးသား ကောက်ချက်ချနည်း ဖြစ်သည်။

280 Logic program တစ်ခုမှ ပြီးမြောက်အောင်မြင်သော “ကောက်ချက်ချခြင်း” (inference) အကြောင်းကို ပြောဆိုရာတွင်၊ တွက်ချက်မှုသည် ပြီးဆုံးသွားသည်ဟု ယူဆခြင်းဖြင့် ဤဖော်ပြချက်ကို ကန့်သတ်ရပါမည်။ ကံမကောင်းစွာဖြင့်၊ not နှင့် lisp-value တို့ကို အသုံးပြုထားသောကြောင့် ကျွန်ုပ်တို့၏ query language အကောင်အထည်ဖော်မှုတွင် (Prolog နှင့် အခြားသော လက်ရှိ logic programming language အများစုတွင်လည်း) ဤကန့်သတ်ထားသော ဖော်ပြချက်သည်ပင် မှားယွင်းနေပါသည်။ အောက်တွင် ဖော်ပြမည့်အတိုင်း query language တွင် အကောင်အထည်ဖော်ထားသော not သည် သင်္ချာ logic ရှိ not နှင့် အမြဲတမ်း တူညီမှုမရှိဘဲ၊ lisp-value ကလည်း ထပ်လောင်း ရှုပ်ထွေးမှုများကို ဖြစ်ပေါ်စေသည်။ ကျွန်ုပ်တို့သည် language မှ not နှင့် lisp-value တို့ကို ဖယ်ရှားပြီး simple queries, and, နှင့် or တို့ကိုသာ အသုံးပြု၍ ပရိုဂရမ်များ ရေးသားရန် သဘောတူခြင်းဖြင့် သင်္ချာ logic နှင့် ကိုက်ညီသော language တစ်ခုကို အကောင်အထည်ဖော်နိုင်သည်။ သို့သော်၊ ဤအရာသည် language ၏ ဖော်ပြနိုင်စွမ်း (expressive power) ကို များစွာ ကန့်သတ်သွားမည်ဖြစ်သည်။ Logic programming သုတေသန၏ အဓိက စိုးရိမ်ပူပန်မှုတစ်ခုမှာ expressive power ကို အလွန်အကျွံ မစွန့်လွှတ်ဘဲ သင်္ချာ logic နှင့် ပိုမိုကိုက်ညီမှုရှိစေမည့် နည်းလမ်းများကို ရှာဖွေရန် ဖြစ်သည်။
