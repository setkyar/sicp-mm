1.1The Elements of Programming

စွမ်းအားထက်မြက်သော ပရိုဂရမ်မင်းဘာသာစကားတစ်ခုသည် ကွန်ပျူတာကို လုပ်ငန်းဆောင်တာများ လုပ်ဆောင်ရန် ညွှန်ကြားရုံမျှသာ မဟုတ်ပါ။ ဤဘာသာစကားသည် လုပ်ငန်းစဉ်များ (processes) နှင့်ပတ်သက်သော ကျွန်ုပ်တို့၏ အတွေးအခေါ်များကို စုစည်းပေးသည့် မူဘောင် (framework) တစ်ခုအဖြစ်လည်း အသုံးတော်ခံပါသည်။ ထို့ကြောင့် ဘာသာစကားတစ်ခုအကြောင်း ဖော်ပြသည့်အခါ၊ ရိုးရှင်းသော အတွေးအခေါ်များကို ပေါင်းစပ်ပြီး ပိုမိုရှုပ်ထွေးသော အတွေးအခေါ်များ ဖြစ်လာစေရန် ထိုဘာသာစကားက ပံ့ပိုးပေးထားသော နည်းလမ်းများကို အထူးဂရုပြုသင့်ပါသည်။ စွမ်းအားထက်မြက်သော ဘာသာစကားတိုင်းတွင် ယင်းကို လုပ်ဆောင်ရန် နည်းလမ်း (mechanisms) သုံးမျိုးရှိသည်:

  • primitive expressions၊ ၎င်းတို့သည် ဘာသာစကားတွင် အခြေခံအကျဆုံး အရာဝတ္ထုများကို ကိုယ်စားပြုသည်၊
  • means of combination၊ ၎င်းတို့ဖြင့် ရိုးရှင်းသော အစိတ်အပိုင်းများမှ ပေါင်းစပ်အစိတ်အပိုင်းများ (compound elements) ကို တည်ဆောက်သည်၊ နှင့်
  • means of abstraction၊ ၎င်းတို့ဖြင့် ပေါင်းစပ်အစိတ်အပိုင်းများကို အမည်ပေးနိုင်ပြီး တစ်ခုချင်းစီအလိုက် ကိုင်တွယ်အသုံးပြုနိုင်ပါသည်။

ပရိုဂရမ်ရေးဆွဲခြင်းတွင် ကျွန်ုပ်တို့သည် အစိတ်အပိုင်း နှစ်မျိုးနှင့် လုပ်ဆောင်ကြရသည်- ၎င်းတို့မှာ လုပ်ထုံးလုပ်နည်းများ (procedures) နှင့် ဒေတာ (data) တို့ ဖြစ်သည်။ (နောက်ပိုင်းတွင် ၎င်းတို့သည် အမှန်တကယ်တွင် ထိုမျှလောက် ကွဲပြားမှုမရှိကြောင်း တွေ့ရှိရပါလိမ့်မည်။) အလွတ်သဘောအရမူ ဒေတာ (data) ဆိုသည်မှာ ကျွန်ုပ်တို့ ကိုင်တွယ်အသုံးပြုလိုသော "အရာဝတ္ထုများ" ဖြစ်ပြီး၊ လုပ်ထုံးလုပ်နည်းများ (procedures) ဆိုသည်မှာ ထိုဒေတာများကို ကိုင်တွယ်အသုံးပြုရန် စည်းမျဉ်းများ၏ ဖော်ပြချက်များ ဖြစ်သည်။ ထို့ကြောင့် မည်သည့် စွမ်းအားထက်မြက်သော ပရိုဂရမ်မင်းဘာသာစကားမဆို အခြေခံဒေတာ (primitive data) နှင့် အခြေခံလုပ်ထုံးလုပ်နည်း (primitive procedures) များကို ဖော်ပြနိုင်ရမည်ဖြစ်ပြီး၊ လုပ်ထုံးလုပ်နည်းများနှင့် ဒေတာများကို ပေါင်းစပ်ရန်နှင့် စိတ္တဇပြုရန် နည်းလမ်းများလည်း ရှိသင့်ပါသည်။

ဤအခန်းတွင် ကျွန်ုပ်တို့သည် လုပ်ထုံးလုပ်နည်းများ တည်ဆောက်ခြင်းဆိုင်ရာ စည်းမျဉ်းများကိုသာ အဓိကအာရုံစိုက်နိုင်ရန် ရိုးရှင်းသော ကိန်းဂဏန်းဒေတာ (numerical data) များနှင့်သာ လုပ်ဆောင်သွားပါမည်။4 နောင်အခန်းများတွင် ဤစည်းမျဉ်းများကပင် ပေါင်းစပ်ဒေတာ (compound data) များကို ကိုင်တွယ်အသုံးပြုရန် လုပ်ထုံးလုပ်နည်းများကို တည်ဆောက်နိုင်စေကြောင်း တွေ့မြင်ရပါလိမ့်မည်။

1.1.1Expressions

ပရိုဂရမ်ရေးဆွဲခြင်းကို စတင်ရန် လွယ်ကူသောနည်းလမ်းတစ်ခုမှာ Lisp ၏ မျိုးကွဲတစ်ခုဖြစ်သော Scheme အတွက် interpreter တစ်ခုနှင့် ပုံမှန် အပြန်အလှန်လုပ်ဆောင်မှုအချို့ကို လေ့လာကြည့်ခြင်းပင် ဖြစ်သည်။ သင်သည် ကွန်ပျူတာ တာမီနယ် (terminal) တစ်ခုရှေ့တွင် ထိုင်နေသည်ဟု မြင်ယောင်ကြည့်ပါ။ သင်သည် expression တစ်ခုကို ရိုက်ထည့်လိုက်လျှင် interpreter က ထို expression ကို evaluating (တွက်ချက်ခြင်း) ပြုလုပ်ပြီး ရရှိလာသော ရလဒ်ကို ပြသပေးမည် ဖြစ်သည်။

သင် ရိုက်ထည့်နိုင်သော အခြေခံ expression တစ်မျိုးမှာ ကိန်းဂဏန်း (number) တစ်ခု ဖြစ်သည်။ (ပိုမို တိကျစွာပြောရလျှင် သင်ရိုက်ထည့်လိုက်သော expression သည် ဆယ်လီစိတ်စနစ် (base 10) ဖြင့် ကိန်းဂဏန်းကို ကိုယ်စားပြုသော ဂဏန်းများ ဖြစ်သည်။) အကယ်၍ သင်သည် Lisp သို့ ကိန်းဂဏန်းတစ်ခု ပေးလိုက်လျှင်

1.1.2Naming and the Environment

ပရိုဂရမ်မင်းဘာသာစကားတစ်ခု၏ အရေးကြီးသော ကဏ္ဍတစ်ခုမှာ တွက်ချက်မှုဆိုင်ရာ အရာဝတ္ထု (computational objects) များကို ရည်ညွှန်းရန် အမည်များကို အသုံးပြုနိုင်အောင် ပံ့ပိုးပေးခြင်းပင် ဖြစ်သည်။ ထိုအမည်သည် ၎င်း၏ တန်ဖိုး (value) မှာ အရာဝတ္ထုဖြစ်သော variable တစ်ခုကို သတ်မှတ်ပေးသည်ဟု ဆိုကြသည်။

Lisp ၏ Scheme မျိုးကွဲတွင် အရာဝတ္ထုများကို define ဖြင့် အမည်ပေးသည်။ အောက်ပါအတိုင်း ရိုက်ထည့်ခြင်းသည် 

(define size 2)

interpreter အား size ဆိုသော အမည်တွင် တန်ဖိုး ၂ ကို တွဲဖက်သတ်မှတ်ရန် ဖြစ်စေသည်။8 size ဆိုသော အမည်ကို ကိန်းဂဏန်း ၂ နှင့် တွဲဖက်ပြီးသည်နှင့်၊ ကျွန်ုပ်တို့သည် တန်ဖိုး ၂ ကို အမည်ဖြင့် ရည်ညွှန်းနိုင်ပြီ ဖြစ်သည်:

Define သည် ကျွန်ုပ်တို့ ဘာသာစကား၏ အခြေခံအကျဆုံး စိတ္တဇပြုခြင်း (abstraction) နည်းလမ်း ဖြစ်သည်၊ အကြောင်းမှာ ၎င်းသည် အထက်တွင် တွက်ချက်ခဲ့သော circumference ကဲ့သို့သော ပေါင်းစပ်လုပ်ဆောင်မှုများ၏ ရလဒ်များကို ရိုးရှင်းသော အမည်များဖြင့် ရည်ညွှန်းနိုင်ရန် ခွင့်ပြုသောကြောင့် ဖြစ်သည်။ ယေဘုယျအားဖြင့် တွက်ချက်မှုဆိုင်ရာ အရာဝတ္ထုများသည် အလွန်ရှုပ်ထွေးသော တည်ဆောက်ပုံများ ရှိနိုင်ပြီး၊ ၎င်းတို့ကို အသုံးပြုလိုသည့် အခါတိုင်းတွင် အသေးစိတ်အချက်အလက်များကို မှတ်သားရန်နှင့် ထပ်ခါတလဲလဲ ရေးသားရန်မှာ အလွန်ပင် အဆင်မပြေနိုင်ပါ။ အမှန်စင်စစ်၊ ရှုပ်ထွေးသော ပရိုဂရမ်များကို ပိုမိုရှုပ်ထွေးလာသော တွက်ချက်မှုဆိုင်ရာ အရာဝတ္ထုများကို တစ်ဆင့်ချင်း တည်ဆောက်ခြင်းဖြင့် ဖန်တီးထားခြင်း ဖြစ်သည်။ interpreter သည် ဤကဲ့သို့ တစ်ဆင့်ချင်း ပရိုဂရမ် တည်ဆောက်ခြင်းကို အထူးပင် အဆင်ပြေစေသည်၊ အကြောင်းမှာ အမည်နှင့် အရာဝတ္ထု တွဲဖက်မှုများကို ဆက်တိုက်လုပ်ဆောင်မှုများတွင် တိုးပွားတည်ဆောက်နိုင်သောကြောင့် ဖြစ်သည်။ ဤအင်္ဂါရပ်သည် ပရိုဂရမ်များကို တဖြည်းဖြည်းချင်း တိုးမြှင့်ဖန်တီးခြင်းနှင့် စမ်းသပ်ခြင်းကို အားပေးပြီး၊ Lisp ပရိုဂရမ်တစ်ခုတွင် များသောအားဖြင့် ရိုးရှင်းသော လုပ်ထုံးလုပ်နည်း အမြောက်အမြား ပါဝင်နေရခြင်း၏ အဓိက အကြောင်းရင်းလည်း ဖြစ်သည်။

သင်္ကေတ (symbols) များကို တန်ဖိုးများနှင့် တွဲဖက်နိုင်ခြင်းနှင့် နောက်ပိုင်းတွင် ၎င်းတို့ကို ပြန်လည် ထုတ်ယူနိုင်ခြင်းတို့သည် interpreter အနေဖြင့် အမည်-အရာဝတ္ထု တွဲများကို မှတ်သားထားရန် မှတ်ဉာဏ် (memory) တစ်မျိုး ထိန်းသိမ်းထားရမည်ကို ရှင်းလင်းစေသည်။ ဤမှတ်ဉာဏ်ကို environment (ပတ်ဝန်းကျင်) ဟု ခေါ်သည် (တွက်ချက်မှုတစ်ခုတွင် မတူညီသော ပတ်ဝန်းကျင် အမြောက်အမြား ပါဝင်နိုင်သည်ကို နောက်ပိုင်းတွင် တွေ့ရမည် ဖြစ်သောကြောင့်၊ ပိုမို တိကျစွာပြောရလျှင် global environment ဖြစ်သည်)။9

1.1.3Evaluating Combinations

ဤအခန်း၏ ကျွန်ုပ်တို့၏ ရည်မှန်းချက်တစ်ခုမှာ လုပ်ထုံးလုပ်နည်းအရ စဥ်းစားတွေးခေါ်မှု (thinking procedurally) ဆိုင်ရာ ကိစ္စရပ်များကို သီးခြားခွဲထုတ်ရန်ဖြစ်သည်။ သာဓကတစ်ခုအနေဖြင့် combination များကို တွက်ချက်ရာတွင် (evaluating combinations) interpreter ကိုယ်တိုင်သည် လုပ်ထုံးလုပ်နည်း (procedure) တစ်ခုကို လိုက်နာနေခြင်းဖြစ်သည်ကို သုံးသပ်ကြည့်ကြပါစို့။

Combination တစ်ခုကို တွက်ချက်ရန်အတွက် အောက်ပါအတိုင်း လုပ်ဆောင်ပါ-

  1. Combination ၏ subexpression များကို တွက်ချက်ပါ။
  2. လက်ဝဲဘက်အကျဆုံး subexpression ၏ တန်ဖိုးဖြစ်သော procedure (operator) ကို အခြား subexpression များ၏ တန်ဖိုးများဖြစ်သော argument များ (operands) အပေါ်တွင် သက်ရောက်စေပါ။ (Apply လုပ်ပါ)။

ဤရိုးရှင်းသော စည်းမျဥ်းသည်ပင် ယေဘုယျအားဖြင့် process များအကြောင်း အရေးကြီးသော အချက်အချို့ကို ဖော်ပြနေသည်။ ပထမဦးစွာ၊ combination တစ်ခုအတွက် evaluation process ပြီးမြောက်စေရန် combination ၏ အစိတ်အပိုင်းတစ်ခုချင်းစီအပေါ် evaluation process ကို အရင်လုပ်ဆောင်ရမည်ဟု ပထမအဆင့်က ပြဌာန်းထားသည်ကို သတိပြုပါ။ ထို့ကြောင့် evaluation rule သည် သဘာဝအရ recursive ဖြစ်သည်။ ဆိုလိုသည်မှာ ၎င်း၏ အဆင့်များထဲမှ တစ်ခုအနေဖြင့် ၎င်းစည်းမျဥ်းကိုယ်တိုင်ကို ပြန်လည်ခေါ်ယူရန် (invoke) လိုအပ်ချက် ပါဝင်နေသည်။10

ဆင့်ကဲဆင့်ကဲ ပါဝင်နေသော (deeply nested) combination များတွင် အလွန်ရှုပ်ထွေးသော process အဖြစ် မြင်တွေ့ရမည့်အရာကို ဖော်ပြရန် recursion စိတ်ကူးကို မည်မျှကျစ်လျစ်စွာ အသုံးပြုနိုင်သည်ကို သတိပြုပါ။ ဥပမာအားဖြင့်၊ 

(* (+ 2 (* 4 6)) (+ 3 5 7))

ကို တွက်ချက်ရန်အတွက် evaluation rule ကို မတူညီသော combination လေးခုတွင် သက်ရောက်စေရန် လိုအပ်သည်။ ဤဖြစ်စဥ်၏ ပုံရိပ်ကို Figure 1.1 တွင် ပြထားသည့်အတိုင်း combination ကို သစ်ပင် (tree) ပုံစံဖြင့် ကိုယ်စားပြုခြင်းအားဖြင့် ရရှိနိုင်သည်။ Combination တစ်ခုစီကို node တစ်ခုဖြင့် ကိုယ်စားပြုပြီး ၎င်းမှ ထွက်ပေါ်လာသော operator နှင့် operands တို့သည် အကိုင်းအခက် (branches) များ ဖြစ်ကြသည်။ Terminal node များ (ဆိုလိုသည်မှာ ၎င်းတို့မှ အကိုင်းအခက်များ ထပ်မထွက်သော node များ) သည် operator များ သို့မဟုတ် ကိန်းဂဏန်းများကို ကိုယ်စားပြုသည်။ Evaluation ကို သစ်ပင်ပုံစံဖြင့် ကြည့်မည်ဆိုလျှင်၊ operands များ၏ တန်ဖိုးများသည် terminal node များမှ စတင်ကာ အထက်သို့ စီးဆင်းသွားပြီး (percolate upward) ပိုမိုမြင့်မားသော အဆင့်များတွင် ပေါင်းစပ်သွားသည်ဟု စိတ်ကူးကြည့်နိုင်သည်။ ယေဘုယျအားဖြင့် recursion သည် အထပ်ထပ် အဆင့်ဆင့်ရှိသော (hierarchical)၊ သစ်ပင်ကဲ့သို့သော အရာဝတ္ထုများကို ကိုင်တွယ်ရာတွင် အလွန်ထိရောက်သော နည်းစနစ်တစ်ခုဖြစ်သည်ကို ကျွန်ုပ်တို့ မြင်တွေ့ရမည်ဖြစ်သည်။ အမှန်စင်စစ်၊ evaluation rule ၏ “တန်ဖိုးများ အထက်သို့ စီးဆင်းခြင်း” ပုံစံသည် tree accumulation ဟု လူသိများသော ယေဘုယျ process အမျိုးအစား၏ ဥပမာတစ်ခု ဖြစ်သည်။

နောက်တစ်ချက်အနေဖြင့်၊ ပထမအဆင့်ကို ထပ်ခါတလဲလဲ အသုံးချခြင်းက ကျွန်ုပ်တို့ကို combination များမဟုတ်ဘဲ numeral များ၊ built-in operator များ သို့မဟုတ် အခြားသော အမည်များကဲ့သို့သော primitive expression များကို တွက်ချက်ရန် လိုအပ်သည့် အချက်သို့ ရောက်ရှိစေသည်။ Primitive အခြေအနေများကို အောက်ပါအတိုင်း သတ်မှတ်ခြင်းဖြင့် ဖြေရှင်းသည်-

  • Numeral များ၏ တန်ဖိုးများသည် ၎င်းတို့ အမည်ပေးထားသော ကိန်းဂဏန်းများ ဖြစ်သည်။
  • Built-in operator များ၏ တန်ဖိုးများသည် သက်ဆိုင်ရာ operation များကို လုပ်ဆောင်ပေးသည့် machine instruction sequence များ ဖြစ်သည်။
  • အခြားသော အမည်များ၏ တန်ဖိုးများသည် environment အတွင်းရှိ ထိုအမည်များနှင့် ဆက်စပ်နေသော object များ ဖြစ်သည်။

ဒုတိယစည်းမျဥ်းကို တတိယစည်းမျဥ်း၏ အထူးအခြေအနေတစ်ခုအဖြစ် ယူဆနိုင်သည်။ + နှင့် * ကဲ့သို့သော သင်္ကေတ (symbol) များသည်လည်း global environment တွင် ပါဝင်ပြီး ၎င်းတို့၏ “တန်ဖိုးများ” ဖြစ်သော machine instruction sequence များနှင့် ဆက်စပ်နေသည်ဟု သတ်မှတ်နိုင်သောကြောင့် ဖြစ်သည်။ သတိပြုရန် အဓိကအချက်မှာ expression များရှိ symbol များ၏ အဓိပ္ပာယ်ကို သတ်မှတ်ရာတွင် environment ၏ အခန်းကဏ္ဍဖြစ်သည်။ Lisp ကဲ့သို့သော interactive language တွင် symbol x (သို့မဟုတ် symbol + ပင်လျှင်) အတွက် အဓိပ္ပာယ်ပေးမည့် environment နှင့် ပတ်သက်သော မည်သည့်အချက်အလက်ကိုမျှ မဖော်ပြဘဲ (+ x 1) ကဲ့သို့သော expression တစ်ခု၏ တန်ဖိုးအကြောင်း ပြောဆိုခြင်းသည် အဓိပ္ပာယ်မရှိပေ။ Chapter 3 တွင် ကျွန်ုပ်တို့ မြင်တွေ့ရမည့်အတိုင်း၊ evaluation ပြုလုပ်မည့် context တစ်ခုကို ပံ့ပိုးပေးသော environment ဟူသည့် ယေဘုယျ အယူအဆသည် program execution ကို နားလည်သဘောပေါက်ရန်အတွက် အရေးကြီးသော အခန်းကဏ္ဍမှ ပါဝင်မည်ဖြစ်သည်။

အထက်တွင် ဖော်ပြထားသော evaluation rule သည် definition များကို မကိုင်တွယ်နိုင်သည်ကို သတိပြုပါ။ ဥပမာအားဖြင့် (define x 3) ကို တွက်ချက်ခြင်းသည် define ကို argument နှစ်ခုအပေါ်တွင် သက်ရောက်ခြင်း မဟုတ်ပါ။ အကြောင်းမှာ ထို argument များထဲမှ တစ်ခုသည် symbol x ၏ တန်ဖိုး ဖြစ်ပြီး အခြားတစ်ခုမှာ 3 ဖြစ်နေသောကြောင့် ဖြစ်သည်။ အမှန်တကယ်တွင် define ၏ ရည်ရွယ်ချက်မှာ x ကို တန်ဖိုးတစ်ခုနှင့် ဆက်စပ်ပေးရန် ဖြစ်သည်။ (ဆိုလိုသည်မှာ (define x 3) သည် combination တစ်ခု မဟုတ်ပါ။)

ယေဘုယျ evaluation rule မှ ထိုကဲ့သို့သော ကင်းလွတ်ခွင့်များကို special forms ဟု ခေါ်သည်။ Define သည် ကျွန်ုပ်တို့ ယခုအချိန်အထိ တွေ့မြင်ခဲ့ရသော special form ၏ တစ်ခုတည်းသော ဥပမာဖြစ်သော်လည်း မကြာမီ အခြားအရာများကို တွေ့ရပါမည်။ Special form တစ်ခုစီတွင် ကိုယ်ပိုင် evaluation rule ရှိသည်။ အမျိုးမျိုးသော expression များ (၎င်းတို့နှင့် သက်ဆိုင်ရာ evaluation rule များ အသီးသီးရှိကြသည်) သည် programming language ၏ syntax ကို ဖွဲ့စည်းသည်။ အခြား programming language အများစုနှင့် နှိုင်းယှဥ်ပါက Lisp တွင် အလွန်ရိုးရှင်းသော syntax ရှိသည်။ ဆိုလိုသည်မှာ expression များအတွက် evaluation rule ကို ရိုးရှင်းသော ယေဘုယျ စည်းမျဥ်းတစ်ခုနှင့် special form အနည်းငယ်အတွက် သီးခြား စည်းမျဥ်းများဖြင့် ဖော်ပြနိုင်သည်။11

1.1.4Compound Procedures

စွမ်းအားထက်မြက်သော မည်သည့် programming language တွင်မဆို ပါဝင်ရမည့် အစိတ်အပိုင်းအချို့ကို Lisp တွင် ကျွန်ုပ်တို့ တွေ့ရှိခဲ့ပြီး ဖြစ်သည်-

  • ကိန်းဂဏန်းများနှင့် ဂဏန်းသင်္ချာ လုပ်ဆောင်ချက်များသည် primitive data နှင့် procedures များ ဖြစ်ကြသည်။
  • Combination များကို တစ်ခုအတွင်းတစ်ခု ထည့်သွင်းခြင်း (Nesting) သည် operation များကို ပေါင်းစပ်ရန် နည်းလမ်းတစ်ခု ဖြစ်စေသည်။
  • အမည်များကို တန်ဖိုးများနှင့် ဆက်စပ်ပေးသော Definition များသည် အကန့်အသတ်ရှိသော abstraction နည်းလမ်းကို ပေးစွမ်းသည်။

ယခုအခါ ကျွန်ုပ်တို့သည် procedure definitions အကြောင်းကို လေ့လာပါမည်။ ၎င်းသည် ပေါင်းစပ်လုပ်ဆောင်ချက် (compound operation) တစ်ခုကို အမည်တစ်ခု ပေးနိုင်ပြီး ၎င်းကို ယူနစ်တစ်ခုအနေဖြင့် ပြန်လည်ညွှန်းဆိုနိုင်သော ပိုမိုစွမ်းအားထက်မြက်သည့် abstraction နည်းစနစ်တစ်ခုဖြစ်သည်။

“Square လုပ်ခြင်း” (ကိန်းထပ်ကိန်းရှာခြင်း) ဟူသော စိတ်ကူးကို မည်သို့ဖော်ပြရမည်ကို လေ့လာခြင်းဖြင့် စတင်ကြပါစို့။ “အရာတစ်ခုကို square လုပ်ရန် ၎င်းကို ၎င်းကိုယ်တိုင်နှင့် မြှောက်ပါ” ဟု ကျွန်ုပ်တို့ ပြောနိုင်သည်။ ၎င်းကို ကျွန်ုပ်တို့၏ language တွင် အောက်ပါအတိုင်း ဖော်ပြသည်- 

(define (square x) (* x x))

၎င်းကို အောက်ပါအတိုင်း နားလည်နိုင်သည်- 

(define (square x)    (*       x       x))
  |      |      |      |       |       |
 To square something, multiply it by itself.

ဤနေရာတွင် ကျွန်ုပ်တို့၌ square ဟု အမည်ပေးထားသော compound procedure တစ်ခု ရှိသည်။ ဤ procedure သည် အရာတစ်ခုကို ၎င်းကိုယ်တိုင်နှင့် မြှောက်ခြင်းလုပ်ဆောင်ချက်ကို ကိုယ်စားပြုသည်။ မြှောက်ရမည့် အရာကို local name ဖြစ်သော x ဟု ပေးထားပြီး ၎င်းသည် သဘာဝဘာသာစကားရှိ နာမ်စား (pronoun) တစ်ခုကဲ့သို့ အခန်းကဏ္ဍမှ ပါဝင်သည်။ ဤ definition ကို တွက်ချက်ခြင်းသည် ဤ compound procedure ကို ဖန်တီးပြီး ၎င်းကို square ဟူသော အမည်နှင့် ဆက်စပ်ပေးသည်။12

Procedure definition ၏ ယေဘုယျပုံစံမှာ- 

(define (⟨name⟩ ⟨formal parameters⟩) ⟨body⟩)

⟨name⟩ သည် environment တွင် procedure definition နှင့် ဆက်စပ်ရမည့် symbol ဖြစ်သည်။13 ⟨formal parameters⟩ များသည် procedure ၏ သက်ဆိုင်ရာ argument များကို ညွှန်းဆိုရန်အတွက် procedure ၏ body အတွင်းတွင် အသုံးပြုသော အမည်များဖြစ်သည်။ ⟨body⟩ သည် formal parameter များကို procedure အသုံးချမည့် actual argument များနှင့် အစားထိုးလိုက်သောအခါ procedure application ၏ တန်ဖိုးကို ထုတ်ပေးမည့် expression ဖြစ်သည်။14 ⟨name⟩ နှင့် ⟨formal parameters⟩ တို့ကို သတ်မှတ်နေသော procedure ကို အမှန်တကယ် ခေါ်ယူရာတွင် အသုံးပြုမည့် ပုံစံအတိုင်းပင် လက်သည်းကွင်းများအတွင်း စုစည်းထားသည်။

square ကို define လုပ်ပြီးပါက ၎င်းကို အသုံးပြုနိုင်ပြီဖြစ်သည်- 

(square 21)
441

(square (+ 2 5))
49

(square (square 3))
81

အခြား procedure များကို define လုပ်ရာတွင်လည်း square ကို အခြေခံအဆောက်အအုံ (building block) တစ်ခုအနေဖြင့် အသုံးပြုနိုင်သည်။ ဥပမာအားဖြင့် x 2 + y 2 ကို အောက်ပါအတိုင်း ဖော်ပြနိုင်သည်- 

(+ (square x) (square y))

မည်သည့် ကိန်းနှစ်ခုကိုမဆို argument အဖြစ်ပေးလျှင် ၎င်းတို့၏ square များ ပေါင်းလဒ်ကို ထုတ်ပေးသည့် sum-of-squares procedure ကို အလွယ်တကူ define လုပ်နိုင်သည်။ 

(define (sum-of-squares x y)
  (+ (square x) (square y)))

(sum-of-squares 3 4)
25

ယခုအခါ နောက်ထပ် procedure များ တည်ဆောက်ရန်အတွက် sum-of-squares ကို building block တစ်ခုအနေဖြင့် အသုံးပြုနိုင်ပြီဖြစ်သည်- 

(define (f a)
  (sum-of-squares (+ a 1) (* a 2)))

(f 5)
136

Compound procedure များကို primitive procedure များအတိုင်း ပုံစံတူ အသုံးပြုသည်။ အမှန်စင်စစ်၊ အထက်တွင် ပေးထားသော sum-of-squares ၏ definition ကို ကြည့်ရုံမျှဖြင့် square သည် + နှင့် * တို့ကဲ့သို့ interpreter တွင် built-in ပါဝင်ခြင်းလော သို့မဟုတ် compound procedure အဖြစ် define လုပ်ထားခြင်းလော ဆိုသည်ကို မသိနိုင်ပေ။

1.1.5Procedure Application အတွက် Substitution Model

Compound procedure တစ်ခုကို အမည်ပေးထားတဲ့ operator ပါဝင်တဲ့ combination တစ်ခုကို evaluate လုပ်ဖို့အတွက်၊ interpreter ဟာ 1.1.3 မှာ ဖော်ပြခဲ့တဲ့ primitive procedures တွေကို အမည်ပေးထားတဲ့ operator တွေပါတဲ့ combinations တွေအတွက် လုပ်ဆောင်တဲ့ လုပ်ငန်းစဉ်အတိုင်းပဲ လုပ်ဆောင်ပါတယ်။ အဲဒါကတော့၊ interpreter ဟာ combination ရဲ့ အစိတ်အပိုင်းတွေကို evaluate လုပ်ပြီး၊ ရရှိလာတဲ့ procedure (combination ရဲ့ operator ရဲ့ တန်ဖိုး) ကို arguments တွေ (combination ရဲ့ operands တွေရဲ့ တန်ဖိုးများ) နဲ့ apply လုပ်တာပဲ ဖြစ်ပါတယ်။

Primitive procedures တွေကို arguments တွေနဲ့ apply လုပ်တဲ့ နည်းလမ်း (mechanism) ကိုတော့ interpreter ထဲမှာတင် တစ်ပါတည်း ထည့်သွင်းတည်ဆောက်ထားတယ်လို့ ယူဆနိုင်ပါတယ်။ Compound procedures တွေအတွက်တော့ application process ဟာ အောက်ပါအတိုင်း ဖြစ်ပါတယ် -

Compound procedure တစ်ခုကို arguments တွေနဲ့ apply လုပ်ဖို့အတွက်၊ procedure ရဲ့ formal parameter တစ်ခုချင်းစီကို သက်ဆိုင်ရာ argument နဲ့ အစားထိုးပြီး procedure ရဲ့ body ကို evaluate လုပ်ပါ။

ဒီလုပ်ငန်းစဉ်ကို ရှင်းလင်းစေဖို့အတွက်၊ အောက်ပါ combination ကို evaluate လုပ်ကြည့်ကြစို့ - 

(f 5)

ဒီနေရာမှာ f ဆိုတာက 1.1.4 မှာ သတ်မှတ်ခဲ့တဲ့ procedure ဖြစ်ပါတယ်။ ကျွန်ုပ်တို့ဟာ f ရဲ့ body ကို အရင်ဆုံး ရယူပါမယ် - 

(sum-of-squares (+ a 1) (* a 2))

ပြီးနောက် formal parameter a ကို argument 5 နဲ့ အစားထိုးပါမယ် - 

(sum-of-squares (+ 5 1) (* 5 2))

Conditional expressions များကို အောက်ပါအတိုင်း တွက်ချက်သည်။ ပထမဦးစွာ predicate ⟨ p 1 ⟩ ကို တွက်ချက်သည်။ အကယ်၍ ၎င်း၏တန်ဖိုးသည် false ဖြစ်ပါက ⟨ p 2 ⟩ ကို ဆက်လက်တွက်ချက်သည်။ အကယ်၍ ⟨ p 2 ⟩ ၏ တန်ဖိုးသည်လည်း false ဖြစ်နေပါက ⟨ p 3 ⟩ ကို ထပ်မံတွက်ချက်သည်။ ဤလုပ်ငန်းစဉ်သည် တန်ဖိုး true ဖြစ်သော predicate တစ်ခုကို တွေ့ရှိသည်အထိ ဆက်လက်လုပ်ဆောင်ပြီး၊ ထိုသို့တွေ့ရှိပါက interpreter သည် ထို clause ၏ သက်ဆိုင်ရာ အကျိုးဆက်ဖော်ပြချက် (consequent expression) ⟨ e ⟩ ၏ တန်ဖိုးကို conditional expression ၏ ရလဒ်အဖြစ် ပြန်ပေးသည်။ အကယ်၍ ⟨ p ⟩ များထဲမှ တစ်ခုမျှ true မဖြစ်ပါက cond ၏ တန်ဖိုးကို သတ်မှတ်ထားခြင်းမရှိပါ။

ပကတိတန်ဖိုးတွက်ချက်သည့် procedure ကို ရေးသားနိုင်သော အခြားနည်းလမ်းတစ်ခုမှာ

  (cond ((< x 0) (- x))
        (else x)))

ပကတိတန်ဖိုးတွက်ချက်သည့် procedure ကို ရေးသားနိုင်သော နောက်ထပ်နည်းလမ်းတစ်ခုမှာ-

  (if (< x 0)
      (- x)

1.1.7ဥပမာ- Newton’s Method ဖြင့် နှစ်ထပ်ကိန်းရင်းများ ရှာဖွေခြင်း

အထက်တွင် မိတ်ဆက်ခဲ့သော လုပ်ထုံးလုပ်နည်းများ (Procedures) သည် သာမန် သင်္ချာဖန်ရှင်များနှင့် များစွာ တူညီပါသည်။ ၎င်းတို့သည် parameter တစ်ခု သို့မဟုတ် တစ်ခုထက်ပိုသော parameter များအပေါ် မူတည်၍ တန်ဖိုးတစ်ခုကို သတ်မှတ်ပေးသည်။ သို့သော် သင်္ချာဖန်ရှင်များနှင့် ကွန်ပျူတာ လုပ်ထုံးလုပ်နည်းများအကြား အရေးကြီးသော ကွဲပြားချက်တစ်ခု ရှိသည်။ လုပ်ထုံးလုပ်နည်းများသည် ထိရောက်မှု (effective) ရှိရမည် ဖြစ်သည်။

ဥပမာတစ်ခုအနေဖြင့် နှစ်ထပ်ကိန်းရင်းများ (square roots) ကို တွက်ချက်သည့် ပြဿနာကို စဉ်းစားကြည့်ပါ။ ကျွန်ုပ်တို့သည် square-root ဖန်ရှင်ကို အောက်ပါအတိုင်း အဓိပ္ပာယ်ဖွင့်ဆိုနိုင်သည် - x = ၎င်း y ဖြစ်သော y ≥ 0 နှင့် y 2 = x . ဤအချက်သည် ခိုင်လုံပြည့်စုံသော သင်္ချာဖန်ရှင်တစ်ခုကို ဖော်ပြခြင်း ဖြစ်သည်။ ၎င်းကို အသုံးပြု၍ ကိန်းတစ်ခုသည် အခြားကိန်းတစ်ခု၏ နှစ်ထပ်ကိန်းရင်း ဟုတ်မဟုတ် စစ်ဆေးရန် သို့မဟုတ် ယေဘုယျအားဖြင့် နှစ်ထပ်ကိန်းရင်းများအကြောင်း အချက်အလက်များကို ထုတ်ယူရန် အသုံးပြုနိုင်သည်။ အခြားတစ်ဖက်တွင်မူ၊ ဤအဓိပ္ပာယ်ဖွင့်ဆိုချက်သည် လုပ်ထုံးလုပ်နည်း (procedure) တစ်ခုကို ဖော်ပြခြင်း မဟုတ်ပါ။ အမှန်စင်စစ်၊ ၎င်းသည် ပေးထားသော ကိန်းတစ်ခု၏ နှစ်ထပ်ကိန်းရင်းကို လက်တွေ့တွင် မည်သို့ရှာဖွေရမည်ကို ကျွန်ုပ်တို့အား တစ်စုံတစ်ရာမျှ ပြောပြထားခြင်း မရှိပေ။ ဤအဓိပ္ပာယ်ဖွင့်ဆိုချက်ကို pseudo-Lisp ဖြင့် ပြန်လည်ရေးသားခြင်းကလည်း အကူအညီဖြစ်လိမ့်မည် မဟုတ်ပါ - 

(define (sqrt x)
  (the y (and (>= y 0) 
              (= (square y) x))))

၎င်းသည် လိုရင်းသို့ မရောက်ဘဲ မူလမေးခွန်းကိုသာ ပြန်လည်မေးမြန်းနေသကဲ့သို့ ဖြစ်သည်။

ဖန်ရှင်နှင့် လုပ်ထုံးလုပ်နည်းအကြား ခြားနားချက်မှာ အရာဝတ္ထုများ၏ ဂုဏ်သတ္တိများကို ဖော်ပြခြင်းနှင့် အရာများကို မည်သို့လုပ်ဆောင်ရမည်ကို ဖော်ပြခြင်းတို့အကြား ယေဘုယျ ကွဲပြားမှုကို ရောင်ပြန်ဟပ်ခြင်း ဖြစ်သည်၊ သို့မဟုတ် တစ်ခါတစ်ရံတွင် declarative knowledge နှင့် imperative knowledge အကြား ကွဲပြားမှုဟုလည်း ခေါ်ဆိုကြသည်။ သင်္ချာတွင် ကျွန်ုပ်တို့သည် များသောအားဖြင့် declarative (မည်သို့ဖြစ်သည် - what is) ဖော်ပြချက်များကိုသာ စိတ်ဝင်စားကြသော်လည်း၊ ကွန်ပျူတာသိပ္ပံတွင်မူ ကျွန်ုပ်တို့သည် များသောအားဖြင့် imperative (မည်သို့လုပ်ဆောင်ရမည် - how to) ဖော်ပြချက်များကိုသာ အလေးထားကြသည်။20

နှစ်ထပ်ကိန်းရင်းကို မည်သို့ တွက်ချက်သနည်း။ အသုံးအများဆုံး နည်းလမ်းမှာ Newton’s method of successive approximations ကို အသုံးပြုခြင်း ဖြစ်သည်။ ၎င်းက ကိန်း x ၏ နှစ်ထပ်ကိန်းရင်း တန်ဖိုးအတွက် ခန့်မှန်းခြေ y တစ်ခုရှိလျှင်၊ y ကို x/y နှင့် ပျမ်းမျှရှာခြင်း (averaging) အားဖြင့် ပိုမိုကောင်းမွန်သော (အစစ်အမှန် နှစ်ထပ်ကိန်းရင်းနှင့် ပိုမိုနီးစပ်သော) ခန့်မှန်းခြေတစ်ခု ရရှိရန် ရိုးရှင်းသော ကိုင်တွယ်တွက်ချက်မှုတစ်ခု ပြုလုပ်နိုင်သည်ဟု ဆိုသည်။21 ဥပမာအားဖြင့်၊ ၂ ၏ နှစ်ထပ်ကိန်းရင်းကို အောက်ပါအတိုင်း တွက်ချက်နိုင်သည်။ ကျွန်ုပ်တို့၏ ကနဦးခန့်မှန်းခြေမှာ ၁ ဖြစ်သည်ဟု ယူဆပါစို့ -

ဤလုပ်ငန်းစဉ်ကို ဆက်လက်လုပ်ဆောင်ခြင်းဖြင့်၊ နှစ်ထပ်ကိန်းရင်းနှင့် တစ်ဆင့်ထက်တစ်ဆင့် ပိုမိုနီးစပ်သော ခန့်မှန်းခြေများကို ရရှိမည်ဖြစ်သည်။

ယခု ဤလုပ်ငန်းစဉ်ကို လုပ်ထုံးလုပ်နည်းများ (procedures) အဖြစ် ပုံစံချကြည့်ရအောင်။ ကျွန်ုပ်တို့သည် radicand (နှစ်ထပ်ကိန်းရင်း ရှာလိုသောကိန်း) တန်ဖိုးတစ်ခုနှင့် ခန့်မှန်းခြေ (guess) တန်ဖိုးတစ်ခုတို့ဖြင့် စတင်မည်။ အကယ်၍ ခန့်မှန်းခြေသည် ကျွန်ုပ်တို့၏ ရည်ရွယ်ချက်အတွက် လုံလောက်စွာ ကောင်းမွန်ပါက ပြီးဆုံးပြီ ဖြစ်သည်၊ မဟုတ်ပါက ပိုမိုကောင်းမွန်အောင် ပြုပြင်ထားသော ခန့်မှန်းခြေဖြင့် ဤလုပ်ငန်းစဉ်ကို ထပ်မံလုပ်ဆောင်ရမည်။ ဤအခြေခံ မဟာဗျူဟာကို လုပ်ထုံးလုပ်နည်းတစ်ခုအဖြစ် အောက်ပါအတိုင်း ရေးသားနိုင်သည် - 

(define (sqrt-iter guess x)
  (if (good-enough? guess x)
      guess
      (sqrt-iter (improve guess x) x)))

ခန့်မှန်းခြေတစ်ခုကို radicand နှင့် ယခင်ခန့်မှန်းခြေတို့၏ စားလဒ် (quotient) ကို ပျမ်းမျှရှာခြင်းဖြင့် ပိုမိုကောင်းမွန်အောင် ပြုပြင်နိုင်သည် - 

(define (improve guess x)
  (average guess (/ x guess)))

ဤနေရာတွင် - 

(define (average x y) 
  (/ (+ x y) 2))

“လုံလောက်စွာ ကောင်းမွန်သည် (good enough)” ဆိုသည်မှာ အဘယ်နည်း။ အောက်ပါဖော်ပြချက်သည် ဥပမာပြရန်အတွက် အဆင်ပြေစေသော်လည်း၊ အမှန်တကယ်တွင်မူ အလွန်ကောင်းမွန်သော စမ်းသပ်ချက်တစ်ခု မဟုတ်ပါ။ (လေ့ကျင့်ခန်း ၁.၇ ကို ကြည့်ပါ။) အခြေခံ စိတ်ကူးမှာ ရလဒ်၏ နှစ်ထပ်ကိန်းနှင့် radicand တို့၏ ခြားနားချက်သည် သတ်မှတ်ထားသော သည်းခံနိုင်စွမ်း (tolerance) တန်ဖိုး (ဤနေရာတွင် ၀.၀၀၁) ထက် ငယ်သွားသည်အထိ အဖြေကို ပိုမိုကောင်းမွန်အောင် ပြုပြင်သွားရန် ဖြစ်သည် -22 

(define (good-enough? guess x)
  (< (abs (- (square guess) x)) 0.001))

နောက်ဆုံးတွင်၊ ကျွန်ုပ်တို့ စတင်ရန် နည်းလမ်းတစ်ခု လိုအပ်သည်။ ဥပမာအားဖြင့်၊ မည်သည့်ကိန်း၏ နှစ်ထပ်ကိန်းရင်းကိုမဆို ၁ ဟု အမြဲတမ်း ခန့်မှန်းနိုင်သည် -23 

(define (sqrt x)
  (sqrt-iter 1.0 x))

အကယ်၍ ဤအဓိပ္ပာယ်ဖွင့်ဆိုချက်များကို interpreter ထဲသို့ ရိုက်ထည့်ပါက၊ ကျွန်ုပ်တို့သည် sqrt ကို အခြားမည်သည့် လုပ်ထုံးလုပ်နည်း (procedure) ကဲ့သို့မဆို အသုံးပြုနိုင်သည် -

sqrt ပရိုဂရမ်သည် ကျွန်ုပ်တို့ ယခုအချိန်အထိ မိတ်ဆက်ခဲ့သော ရိုးရှင်းသော procedural language သည် C သို့မဟုတ် Pascal ကဲ့သို့သော ဘာသာစကားများဖြင့် ရေးသားနိုင်သည့် မည်သည့် ကိန်းဂဏန်းဆိုင်ရာ ပရိုဂရမ်ကိုမဆို ရေးသားရန် လုံလောက်ကြောင်း ဖော်ပြနေသည်။ ကျွန်ုပ်တို့၏ ဘာသာစကားတွင် ကွန်ပျူတာကို တစ်ခုခုကို ထပ်ခါတလဲလဲ လုပ်ဆောင်ရန် ညွှန်ကြားသည့် မည်သည့် iterative (looping) constructs များကိုမျှ ထည့်သွင်းထားခြင်း မရှိသောကြောင့် ဤအချက်မှာ အံ့အားသင့်စရာ ဖြစ်နိုင်ပါသည်။ အခြားတစ်ဖက်တွင်မူ sqrt-iter သည် လုပ်ထုံးလုပ်နည်းတစ်ခုကို ခေါ်ဆိုနိုင်သည့် သာမန်စွမ်းရည်မှလွဲ၍ အခြားအထူးဖွဲ့စည်းပုံများ မလိုအပ်ဘဲ iteration ကို မည်သို့လုပ်ဆောင်နိုင်သည်ကို သက်သေပြနေသည်။24

လေ့ကျင့်ခန်း ၁.၇: နှစ်ထပ်ကိန်းရင်းများ တွက်ချက်ရာတွင် အသုံးပြုသည့် good-enough? စမ်းသပ်ချက်သည် အလွန်ငယ်သော ကိန်းများအတွက် ထိရောက်မှု ရှိမည်မဟုတ်ပါ။ ထို့အပြင်၊ လက်တွေ့ကမ္ဘာမှ ကွန်ပျူတာများတွင် သင်္ချာဆိုင်ရာ လုပ်ဆောင်ချက်များကို များသောအားဖြင့် ကန့်သတ်ထားသော တိကျမှု (limited precision) ဖြင့်သာ လုပ်ဆောင်ကြသည်။ ဤအချက်ကြောင့် ကျွန်ုပ်တို့၏ စမ်းသပ်ချက်သည် အလွန်ကြီးမားသော ကိန်းများအတွက် မလုံလောက်မှုများ ဖြစ်ပေါ်စေသည်။ အဆိုပါ ပြောကြားချက်များကို အလွန်ငယ်သောကိန်းများနှင့် အလွန်ကြီးသောကိန်းများအတွက် စမ်းသပ်ချက် မည်သို့ ပျက်ကွက်သည်ကို ဥပမာများဖြင့် ရှင်းပြပါ။ good-enough? ကို အကောင်အထည်ဖော်ရန် အခြားနည်းလမ်းတစ်ခုမှာ iteration တစ်ခုမှ တစ်ခုသို့ guess မည်သို့ပြောင်းလဲသွားသည်ကို စောင့်ကြည့်ရန်နှင့် ပြောင်းလဲမှုသည် ခန့်မှန်းခြေ၏ အလွန်သေးငယ်သော အစိတ်အပိုင်းတစ်ခု ဖြစ်သွားသောအခါတွင် ရပ်တန့်ရန် ဖြစ်သည်။ ဤကဲ့သို့သော အဆုံးသတ်စမ်းသပ်ချက်ကို အသုံးပြုသည့် square-root လုပ်ထုံးလုပ်နည်းတစ်ခုကို ဒီဇိုင်းဆွဲပါ။ ၎င်းသည် အလွန်ငယ်သောကိန်းများနှင့် အလွန်ကြီးသောကိန်းများအတွက် ပိုမိုကောင်းမွန်ပါသလား။

လေ့ကျင့်ခန်း ၁.၈: သုံးထပ်ကိန်းရင်း (cube roots) များအတွက် Newton’s method သည် အကယ်၍ y သည် x ၏ သုံးထပ်ကိန်းရင်းအတွက် ခန့်မှန်းခြေတစ်ခု ဖြစ်ပါက၊ ပိုမိုကောင်းမွန်သော ခန့်မှန်းခြေကို အောက်ပါတန်ဖိုးဖြင့် ရရှိနိုင်သည်ဟူသော အချက်ပေါ်တွင် အခြေခံသည် - x / y 2 + 2 y 3 . ဤပုံသေနည်းကို အသုံးပြု၍ နှစ်ထပ်ကိန်းရင်း လုပ်ထုံးလုပ်နည်းနှင့် ဆင်တူသော သုံးထပ်ကိန်းရင်း (cube-root) လုပ်ထုံးလုပ်နည်းတစ်ခုကို အကောင်အထည်ဖော်ပါ။ (၁.၃.၄ တွင် Newton’s method ကို ယေဘုယျအားဖြင့် ဤနှစ်ထပ်ကိန်းရင်းနှင့် သုံးထပ်ကိန်းရင်း လုပ်ထုံးလုပ်နည်းများ၏ abstract ပုံစံအဖြစ် မည်သို့ အကောင်အထည်ဖော်ရမည်ကို တွေ့ရမည် ဖြစ်သည်။)

1.1.8Procedures as Black-Box Abstractions

Sqrt သည် အပြန်အလှန် အဓိပ္ပာယ်ဖွင့်ဆိုထားသော procedure များဖြင့် ဖွဲ့စည်းထားသည့် process တစ်ခု၏ ပထမဆုံး ဥပမာဖြစ်သည်။ sqrt-iter ၏ အဓိပ္ပာယ်ဖွင့်ဆိုချက်သည် recursive ဖြစ်သည်ကို သတိပြုပါ၊ ဆိုလိုသည်မှာ ၎င်း procedure ကို ၎င်း၏ ကိုယ်ပိုင်အမည်ဖြင့်ပင် ပြန်လည် အဓိပ္ပာယ်ဖွင့်ဆိုထားခြင်း ဖြစ်သည်။ procedure တစ်ခုကို ၎င်း၏ ကိုယ်ပိုင်အမည်ဖြင့် ပြန်လည် အဓိပ္ပာယ်ဖွင့်ဆိုနိုင်သည်ဟူသော အယူအဆသည် စိတ်ရှုပ်ထွေးဖွယ် ဖြစ်နိုင်သည်፤ ထိုကဲ့သို့သော "circular" ဖြစ်နေသည့် အဓိပ္ပာယ်ဖွင့်ဆိုချက်မျိုးသည် ကွန်ပျူတာဖြင့် လုပ်ဆောင်ရမည့် ကောင်းမွန်စွာ သတ်မှတ်ထားသော process တစ်ခုအဖြစ် မည်သို့ အဓိပ္ပာယ်ရှိနိုင်မည်နည်းဟု မရှင်းမလင်း ဖြစ်နိုင်ပါသည်။ ဤအကြောင်းအရာကို 1.2 တွင် ပိုမိုဂရုတစိုက် ဆွေးနွေးသွားမည် ဖြစ်သည်။ သို့သော် ပထမဦးစွာ sqrt ဥပမာမှတစ်ဆင့် ဖော်ပြထားသော အခြား အရေးကြီးသည့် အချက်အချို့ကို စဉ်းစားကြည့်ကြပါစို့။

square roots ရှာဖွေခြင်း ပြဿနာကို သဘာဝအလျောက် ပြဿနာခွဲများစွာအဖြစ် ခွဲထုတ်နိုင်သည်ကို လေ့လာကြည့်ပါ- guess တစ်ခုသည် လုံလောက်မှု ရှိမရှိ မည်သို့သိနိုင်မည်နည်း၊ guess ကို မည်သို့ ပိုမိုကောင်းမွန်အောင် ပြုလုပ်မည်နည်း စသည်တို့ဖြစ်သည်။ ဤလုပ်ဆောင်ချက် တစ်ခုချင်းစီကို သီးခြား procedure တစ်ခုစီဖြင့် လုပ်ဆောင်ပါသည်။ sqrt ပရိုဂရမ်တစ်ခုလုံးကို ပြဿနာကို ပြဿနာခွဲများအဖြစ် ခွဲထုတ်ထားခြင်းကို ထင်ဟပ်စေသော procedure များ စုစည်းမှုတစ်ခု ( Figure 1.2 တွင် ပြသထားသည့်အတိုင်း) အဖြစ် ရှုမြင်နိုင်ပါသည်။

ဤကဲ့သို့ ခွဲထုတ်သည့် မဟာဗျူဟာ၏ အရေးကြီးပုံသည် ပရိုဂရမ်ကို အစိတ်အပိုင်းများအဖြစ် ခွဲဝေလိုက်ခြင်းသက်သက် မဟုတ်ပါ။ အမှန်စင်စစ် ကျွန်ုပ်တို့သည် မည်သည့် ပရိုဂရမ်ကြီးကိုမဆို အစိတ်အပိုင်းများအဖြစ် ခွဲခြားနိုင်ပါသည်—ပထမဆုံး ဆယ်ကြောင်း၊ နောက်ထပ် ဆယ်ကြောင်း၊ နောက်ထပ် ဆယ်ကြောင်း စသည်ဖြင့်ပေါ့။ ထိုသို့မဟုတ်ဘဲ၊ procedure တစ်ခုချင်းစီသည် အခြား procedure များကို အဓိပ္ပာယ်ဖွင့်ဆိုရာတွင် module တစ်ခုအဖြစ် အသုံးပြုနိုင်သည့် ခွဲခြားသိမြင်နိုင်သော လုပ်ဆောင်ချက်တစ်ခုကို ပြီးမြောက်အောင် ဆောင်ရွက်နိုင်ရန်မှာ အလွန်အရေးကြီးပါသည်။ ဥပမာအားဖြင့်၊ ကျွန်ုပ်တို့သည် good-enough? procedure ကို square ဖြင့် အဓိပ္ပာယ်ဖွင့်ဆိုသောအခါ၊ square procedure ကို "black box" တစ်ခုအဖြစ် သဘောထားနိုင်ပါသည်။ ထိုအချိန်တွင် အဆိုပါ procedure က ရလဒ်ကို မည်သို့ တွက်ချက်သည်ဆိုသည်ကို ကျွန်ုပ်တို့ ဂရုမစိုက်ဘဲ၊ ၎င်းသည် square ကို တွက်ချက်ပေးသည်ဟူသော အချက်ကိုသာ အလေးထားပါသည်။ square ကို မည်သို့တွက်ချက်သည်ဟူသော အသေးစိတ်အချက်အလက်များကို ခေတ္တဖယ်ထားပြီး နောက်မှသာ စဉ်းစားနိုင်ပါသည်။ အမှန်စင်စစ်၊ good-enough? procedure နှင့် ပတ်သက်သမျှ ကာလပတ်လုံး square သည် procedure တစ်ခုသက်သက် မဟုတ်ဘဲ procedure တစ်ခု၏ abstraction တစ်ခု၊ ဆိုလိုသည်မှာ procedural abstraction တစ်ခု ဖြစ်ပါသည်။ ဤ abstraction အဆင့်တွင်၊ square ကို တွက်ချက်ပေးသည့် မည်သည့် procedure မဆို တူညီစွာပင် ကောင်းမွန်ပါသည်။

ထို့ကြောင့် ၎င်းတို့ ပြန်ပေးသည့် တန်ဖိုးများကိုသာ စဉ်းစားမည်ဆိုပါက၊ ကိန်းဂဏန်းတစ်ခုကို square ရှာရန် အောက်ပါ procedure နှစ်ခုသည် ခွဲခြား၍မရနိုင်လောက်အောင် တူညီသင့်ပါသည်။ တစ်ခုချင်းစီသည် ကိန်းဂဏန်း argument တစ်ခုကို ယူပြီး ထိုကိန်းဂဏန်း၏ square ကို တန်ဖိုးအဖြစ် ထုတ်ပေးပါသည်။25 

(define (square x) (* x x))

(define (square x) 
  (exp (double (log x))))

(define (double x) (+ x x))

ထို့ကြောင့် procedure အဓိပ္ပာယ်ဖွင့်ဆိုချက်တစ်ခုသည် အသေးစိတ်အချက်အလက်များကို ဖုံးကွယ်ထားနိုင်သင့်ပါသည်။ procedure ကို အသုံးပြုသူများသည် procedure ကို ၎င်းတို့ကိုယ်တိုင် ရေးသားထားခြင်း မဟုတ်ဘဲ အခြား ပရိုဂရမ်မာတစ်ဦးထံမှ black box တစ်ခုအဖြစ် ရရှိထားခြင်း ဖြစ်နိုင်ပါသည်။ အသုံးပြုသူတစ်ဦးအနေဖြင့် procedure ကို အသုံးပြုရန်အတွက် ၎င်းကို မည်သို့ implement လုပ်ထားသည်ကို သိရန် မလိုသင့်ပါ။

procedure ကို အသုံးပြုသူအတွက် အရေးမကြီးသင့်သော implementation ၏ အသေးစိတ် အချက်အလက်တစ်ခုမှာ implementer က procedure ၏ formal parameters များအတွက် ရွေးချယ်ထားသော အမည်များ ဖြစ်သည်။ ထို့ကြောင့် အောက်ပါ procedure များသည် ခွဲခြား၍မရနိုင်သင့်ပါ- 

(define (square x) (* x x))

procedure တစ်ခု၏ အဓိပ္ပာယ်သည် ၎င်းကို ရေးသားသူ အသုံးပြုထားသည့် parameter အမည်များအပေါ် မမှီခိုသင့်ဘူးဟူသော ဤမူဝါဒသည် အပေါ်ယံအားဖြင့် ထင်ရှားလှသော်လည်း၊ ၎င်း၏ နောက်ဆက်တွဲ ရလဒ်များသည် နက်ရှိုင်းလှပါသည်။ အရှင်းလင်းဆုံး နောက်ဆက်တွဲ ရလဒ်မှာ procedure တစ်ခု၏ parameter အမည်များသည် procedure ၏ body အတွင်း၌သာ local ဖြစ်ရပါမည်။ ဥပမာအားဖြင့်၊ ကျွန်ုပ်တို့၏ square-root procedure တွင် good-enough? ကို အဓိပ္ပာယ်ဖွင့်ဆိုရာ၌ square ကို အသုံးပြုခဲ့သည်- 

(define (good-enough? guess x)
  (< (abs (- (square guess) x)) 0.001))

good-enough? ကို ရေးသားသူ၏ ရည်ရွယ်ချက်မှာ ပထမဆုံး argument ၏ square သည် ဒုတိယ argument နှင့် သတ်မှတ်ထားသော tolerance အတွင်း ရှိမရှိကို ဆုံးဖြတ်ရန် ဖြစ်ပါသည်။ good-enough? ကို ရေးသားသူသည် ပထမဆုံး argument အတွက် guess ဟူသော အမည်ကိုလည်းကောင်း၊ ဒုတိယ argument အတွက် x ဟူသော အမည်ကိုလည်းကောင်း အသုံးပြုထားသည်ကို တွေ့ရပါသည်။ square ၏ argument မှာ guess ဖြစ်ပါသည်။ အကယ်၍ square ကို ရေးသားသူက ထို argument ကို ရည်ညွှန်းရန် (အထက်တွင် ဖော်ပြခဲ့သည့်အတိုင်း) x ကို အသုံးပြုခဲ့လျှင်၊ good-enough? ထဲမှ x သည် square ထဲမှ x နှင့် မတူညီသော x ဖြစ်ရမည်ကို ကျွန်ုပ်တို့ တွေ့နိုင်ပါသည်။ square procedure ကို run ခြင်းသည် good-enough? က အသုံးပြုနေသော x ၏ တန်ဖိုးကို မထိခိုက်စေရပါ၊ အဘယ်ကြောင့်ဆိုသော် ထို x ၏ တန်ဖိုးကို square တွက်ချက်ပြီးနောက် good-enough? က ဆက်လက်အသုံးပြုရန် လိုအပ်နိုင်သောကြောင့် ဖြစ်ပါသည်။

အကယ်၍ parameter များသည် ၎င်းတို့နှင့် သက်ဆိုင်ရာ procedure body များတွင် local မဖြစ်ခဲ့လျှင်၊ square မှ parameter x သည် good-enough? မှ parameter x နှင့် မှားယွင်းသွားနိုင်ပြီး၊ good-enough? ၏ လုပ်ဆောင်ချက်သည် ကျွန်ုပ်တို့ မည်သည့် square ဗားရှင်းကို အသုံးပြုသလဲ ဆိုသည့်အပေါ် မူတည်သွားပါလိမ့်မည်။ ထိုသို့ဆိုလျှင် square သည် ကျွန်ုပ်တို့ အလိုရှိသော black box ဖြစ်လာတော့မည် မဟုတ်ပါ။

procedure တစ်ခု၏ formal parameter သည် procedure အဓိပ္ပာယ်ဖွင့်ဆိုချက်တွင် အလွန်ထူးခြားသော အခန်းကဏ္ဍမှ ပါဝင်ပါသည်။ ၎င်း formal parameter တွင် မည်သည့်အမည် ရှိသည်ဆိုသည်မှာ အရေးမကြီးပါ။ ထိုကဲ့သို့သော အမည်ကို bound variable ဟု ခေါ်ဆိုပြီး procedure အဓိပ္ပာယ်ဖွင့်ဆိုချက်သည် ၎င်း၏ formal parameters များကို binds လုပ်သည်ဟု ကျွန်ုပ်တို့ ပြောဆိုကြပါသည်။ အကယ်၍ bound variable တစ်ခုကို အဓိပ္ပာယ်ဖွင့်ဆိုချက် တစ်ခုလုံးတွင် တစ်သမတ်တည်း အမည်ပြောင်းလဲလိုက်လျှင် procedure အဓိပ္ပာယ်ဖွင့်ဆိုချက်သည် ပြောင်းလဲမှု မရှိပါ။26 အကယ်၍ variable တစ်ခုသည် bound မဖြစ်နေလျှင် ၎င်းကို free ဖြစ်သည်ဟု ကျွန်ုပ်တို့ ပြောဆိုကြပါသည်။ binding တစ်ခုက အမည်တစ်ခုကို သတ်မှတ်ပေးလိုက်သော expression များ အစုအဝေးကို ထိုအမည်၏ scope ဟု ခေါ်ဆိုပါသည်။ procedure အဓိပ္ပာယ်ဖွင့်ဆိုချက်တစ်ခုတွင် procedure ၏ formal parameters များအဖြစ် ကြေညာထားသော bound variables များသည် ၎င်းတို့၏ scope အဖြစ် procedure ၏ body တစ်ခုလုံးကို ပိုင်ဆိုင်ကြပါသည်။

အထက်ပါ good-enough? ၏ အဓိပ္ပာယ်ဖွင့်ဆိုချက်တွင် guess နှင့် x တို့သည် bound variables များ ဖြစ်ကြသော်လည်း <, -, abs, နှင့် square တို့သည် free ဖြစ်ကြသည်။ good-enough? ၏ အဓိပ္ပာယ်သည် guess နှင့် x အတွက် ကျွန်ုပ်တို့ ရွေးချယ်ထားသော အမည်များအပေါ် (၎င်းတို့သည် တစ်ခုနှင့်တစ်ခု ကွဲပြားပြီး <, -, abs, နှင့် square တို့နှင့်လည်း မတူသရွေ့) မမှီခိုသင့်ပါ။ (အကယ်၍ ကျွန်ုပ်တို့သည် guess ကို abs ဟု အမည်ပြောင်းလိုက်လျှင် variable abs ကို capturing လုပ်လိုက်ခြင်းဖြင့် bug တစ်ခုကို ဖန်တီးလိုက်သလို ဖြစ်သွားပါလိမ့်မည်။ ၎င်းသည် free မှ bound သို့ ပြောင်းလဲသွားပါလိမ့်မည်။) သို့သော် good-enough? ၏ အဓိပ္ပာယ်သည် ၎င်း၏ free variables များ၏ အမည်များအပေါ် မူတည်ပါသည်။ ၎င်းသည် abs ဟူသော သင်္ကေတသည် ကိန်းဂဏန်းတစ်ခု၏ absolute value ကို တွက်ချက်ရန် procedure တစ်ခု၏ အမည်ဖြစ်သည်ဟူသော (ဤအဓိပ္ပာယ်ဖွင့်ဆိုချက်၏ ပြင်ပရှိ) အချက်အပေါ်တွင် အမှန်တကယ် မူတည်နေပါသည်။ အကယ်၍ ကျွန်ုပ်တို့သည် good-enough? ၏ အဓိပ္ပာယ်ဖွင့်ဆိုချက်တွင် abs အစား cos ကို အစားထိုးလိုက်လျှင် ၎င်းသည် ကွဲပြားသော function တစ်ခုကို တွက်ချက်ပေးပါလိမ့်မည်။

Internal definitions and block structure

ကျွန်ုပ်တို့တွင် ယခုအချိန်အထိ အမည်များကို သီးခြားခွဲထုတ်နိုင်သည့် နည်းလမ်းတစ်ခု ရှိနေပါပြီ- procedure တစ်ခု၏ formal parameters များသည် procedure ၏ body အတွက် local ဖြစ်ကြပါသည်။ square-root ပရိုဂရမ်သည် အမည်များ အသုံးပြုမှုကို ထိန်းချုပ်လိုသည့် အခြားနည်းလမ်းတစ်ခုကိုလည်း သရုပ်ဖော်ထားပါသည်။ လက်ရှိ ပရိုဂရမ်တွင် သီးခြား procedure များ ပါဝင်နေပါသည်- 

(define (sqrt x) 
  (sqrt-iter 1.0 x))

(define (sqrt-iter guess x)
  (if (good-enough? guess x)
      guess
      (sqrt-iter (improve guess x) x)))

(define (good-enough? guess x)
  (< (abs (- (square guess) x)) 0.001))

(define (improve guess x)
  (average guess (/ x guess)))

ဤပရိုဂရမ်၏ အခက်အခဲမှာ sqrt ကို အသုံးပြုသူများအတွက် အရေးကြီးသော တစ်ခုတည်းသော procedure မှာ sqrt သာ ဖြစ်ခြင်းဖြစ်သည်။ အခြား procedure များ (sqrt-iter, good-enough?, နှင့် improve) တို့သည် အသုံးပြုသူများအတွက် စိတ်ရှုပ်စရာသာ ဖြစ်စေပါသည်။ ၎င်းတို့သည် square-root ပရိုဂရမ်နှင့်အတူ တွဲဖက်လုပ်ဆောင်မည့် အခြားပရိုဂရမ်၏ အစိတ်အပိုင်းအဖြစ် good-enough? ဟု အမည်ပေးထားသော အခြား မည်သည့် procedure ကိုမျှ အဓိပ္ပာယ်ဖွင့်ဆိုနိုင်မည် မဟုတ်ပါ၊ အဘယ်ကြောင့်ဆိုသော် sqrt က ထိုအမည်ကို လိုအပ်နေသောကြောင့် ဖြစ်သည်။ ဤပြဿနာသည် ပရိုဂရမ်မာ အများအပြားက စနစ်ကြီးများ တည်ဆောက်ရာတွင် ပိုမိုဆိုးရွားပါသည်။ ဥပမာအားဖြင့်၊ ကိန်းဂဏန်းဆိုင်ရာ procedure များပါဝင်သည့် library အကြီးစားတစ်ခု တည်ဆောက်ရာတွင်၊ များစွာသော numerical functions များကို successive approximations နည်းလမ်းဖြင့် တွက်ချက်ကြသောကြောင့် good-enough? နှင့် improve ဟု အမည်ပေးထားသော အထောက်အကူပြု procedure များ ပါဝင်လာနိုင်ပါသည်။ ကျွန်ုပ်တို့သည် subprocedures များကို sqrt အတွင်း၌သာ ဝှက်ထားပြီး local ဖြစ်စေချင်ပါသည်၊ သို့မှသာ sqrt သည် ၎င်း၏ ကိုယ်ပိုင် good-enough? procedure ဖြင့် အခြား successive approximations နည်းလမ်းများနှင့်အတူ ယှဉ်တွဲ တည်ရှိနိုင်မည် ဖြစ်သည်။ ဤသို့ ဖြစ်နိုင်စေရန်၊ ကျွန်ုပ်တို့သည် procedure တစ်ခုအတွင်း၌ ထို procedure အတွက် local ဖြစ်စေမည့် internal definitions များ ထည့်သွင်းရန် ခွင့်ပြုပါသည်။ ဥပမာအားဖြင့်၊ square-root ပြဿနာတွင် အောက်ပါအတိုင်း ရေးသားနိုင်ပါသည်- 

(define (sqrt x)
  (define (good-enough? guess x)
    (< (abs (- (square guess) x)) 0.001))
  (define (improve guess x)
    (average guess (/ x guess)))
  (define (sqrt-iter guess x)
    (if (good-enough? guess x)
        guess
        (sqrt-iter (improve guess x) x)))
  (sqrt-iter 1.0 x))

ဤကဲ့သို့ အဓိပ္ပာယ်ဖွင့်ဆိုချက်များကို တစ်ခုအတွင်းတစ်ခု ထည့်သွင်းခြင်းကို block structure ဟု ခေါ်ဆိုပြီး၊ ၎င်းသည် အမည်များကို စုစည်းရာတွင် အရှင်းလင်းဆုံးသော ဖြေရှင်းချက် ဖြစ်ပါသည်။ သို့သော် ဤနေရာတွင် ပိုမိုကောင်းမွန်သော အယူအဆတစ်ခု ရှိပါသေးသည်။ အထောက်အကူပြု procedure များ၏ အဓိပ္ပာယ်ဖွင့်ဆိုချက်များကို internalizing လုပ်သည့်အပြင်၊ ၎င်းတို့ကို ပိုမိုရိုးရှင်းအောင်လည်း ပြုလုပ်နိုင်ပါသည်။ x သည် sqrt ၏ အဓိပ္ပာယ်ဖွင့်ဆိုချက်တွင် bound ဖြစ်နေသောကြောင့်၊ sqrt ၏ အတွင်း၌ အဓိပ္ပာယ်ဖွင့်ဆိုထားသော good-enough?, improve, နှင့် sqrt-iter procedure များသည် x ၏ scope အတွင်း၌ ရှိနေပါသည်။ ထို့ကြောင့် ဤ procedure တစ်ခုချင်းစီသို့ x ကို တိုက်ရိုက်ပေးပို့ရန် (pass လုပ်ရန်) မလိုအပ်တော့ပါ။ ယင်းအစား၊ အောက်တွင် ပြသထားသည့်အတိုင်း internal definitions များတွင် x ကို free variable တစ်ခုအဖြစ် ရှိနေရန် ခွင့်ပြုပါသည်။ ထို့နောက် x သည် အပြင်ဘက်ဆုံး procedure ဖြစ်သော sqrt ကို ခေါ်ဆိုလိုက်သည့် argument မှတစ်ဆင့် ၎င်း၏ တန်ဖိုးကို ရရှိပါလိမ့်မည်။ ဤနည်းစနစ်ကို lexical scoping ဟု ခေါ်ဆိုပါသည်။27 

(define (sqrt x)
  (define (good-enough? guess)
    (< (abs (- (square guess) x)) 0.001))
  (define (improve guess)
    (average guess (/ x guess)))
  (define (sqrt-iter guess)
    (if (good-enough? guess)
        guess
        (sqrt-iter (improve guess))))
  (sqrt-iter 1.0))

10 တွက်ချက်မှုဆိုင်ရာ စည်းမျဉ်း၏ ပထမအဆင့်အနေဖြင့် combination တစ်ခု၏ ဘယ်ဘက်အစွန်ဆုံး အစိတ်အပိုင်းကို အရင်တွက်ချက်ရမည်ဟု ဆိုထားခြင်းမှာ ထူးဆန်းနေပုံရနိုင်သည်၊ အကြောင်းမှာ ယခုအချက်တွင် ၎င်းသည် ပေါင်းခြင်း သို့မဟုတ် မြှောက်ခြင်းကဲ့သို့သော built-in primitive procedure များကို ကိုယ်စားပြုသည့် + သို့မဟုတ် * ကဲ့သို့သော operator တစ်ခုသာ ဖြစ်နိုင်သောကြောင့် ဖြစ်သည်။ operator များ ကိုယ်တိုင်ကပင် compound expression များ ဖြစ်နေသော combination များနှင့် အလုပ်လုပ်နိုင်ခြင်းသည် မည်မျှအသုံးဝင်ကြောင်းကို နောင်တွင် တွေ့ရပါလိမ့်မည်။

11 ပိုမိုညီညွတ်သော ပုံစံများဖြင့် ရေးသားနိုင်သည့် အရာများအတွက် အဆင်ပြေစေရန် အစားထိုး အသုံးပြုထားသော မျက်နှာပြင်ပြ စာသားတည်ဆောက်ပုံများကို Peter Landin က အမည်ပေးခဲ့သည့် syntactic sugar ဟု တစ်ခါတစ်ရံ ခေါ်ဆိုကြသည်။ အခြားဘာသာစကား အသုံးပြုသူများနှင့် နှိုင်းယှဉ်ပါက၊ Lisp ပရိုဂရမ်မာများသည် ယေဘုယျအားဖြင့် syntax ဆိုင်ရာ ကိစ္စရပ်များကို သိပ်ဂရုမစိုက်ကြပါ။ (ဆန့်ကျင်ဘက်အနေဖြင့်၊ မည်သည့် Pascal manual ကိုမဆို ကြည့်လိုက်ပါ၊ ၎င်း၏ အစိတ်အပိုင်း အတော်များများကို syntax ဖော်ပြချက်များအတွက် မည်မျှအသုံးပြုထားသည်ကို သတိပြုမိပါလိမ့်မည်။) ဤသို့ syntax အပေါ် ဂရုမထားခြင်းမှာ တစ်စိတ်တစ်ပိုင်းအားဖြင့် Lisp ၏ ပြောင်းလွယ်ပြင်လွယ်ရှိမှုကြောင့် ဖြစ်ပြီး (၎င်းသည် မျက်နှာပြင်ပြ syntax ကို ပြောင်းလဲရန် လွယ်ကူစေသည်)၊ နောက်တစ်ချက်မှာ ဘာသာစကားကို ညီညွတ်မှု လျော့နည်းစေသော "အဆင်ပြေသည့်" syntactic တည်ဆောက်ပုံ အတော်များများသည် ပရိုဂရမ်များ ကြီးမားရှုပ်ထွေးလာသောအခါ တန်ဖိုးထက် ပြဿနာပိုပေးတတ်ခြင်းကို တွေ့ရှိရသောကြောင့် ဖြစ်သည်။ Alan Perlis ၏ စကားအရမူ "Syntactic sugar သည် semicolon ကင်ဆာကို ဖြစ်စေသည်" ဟု ဆိုသည်။

12 ဤနေရာတွင် မတူညီသော လုပ်ဆောင်ချက်နှစ်ခု ပေါင်းစပ်ထားသည်ကို သတိပြုပါ- ကျွန်ုပ်တို့သည် procedure တစ်ခုကို ဖန်တီးနေခြင်းဖြစ်ပြီး၊ ၎င်းကို square ဟု အမည်ပေးနေခြင်း ဖြစ်သည်။ ဤအယူအဆ နှစ်ခုကို သီးခြားစီ ခွဲထုတ်နိုင်ရန် အမှန်တကယ်ပင် အရေးကြီးသည်—ဆိုလိုသည်မှာ procedure များကို အမည်မပေးဘဲ ဖန်တီးနိုင်ခြင်းနှင့် ဖန်တီးပြီးသား procedure များကို အမည်ပေးနိုင်ခြင်းတို့ ဖြစ်သည်။ ၎င်းကို မည်သို့လုပ်ဆောင်ရမည်ကို 1.3.2 တွင် တွေ့ရမည်ဖြစ်သည်။

13 ဤစာအုပ်တစ်လျှောက်လုံးတွင်၊ expression တစ်ခုကို အမှန်တကယ် အသုံးပြုသည့်အခါ ဖြည့်စွက်ရမည့် "နေရာလွတ်" (slots) များကို ဖော်ပြရန်အတွက် angle brackets များဖြင့် ဝန်းရံထားသော စောင်းထားသည့် သင်္ကေတများ—ဥပမာ၊ ⟨name⟩—ကို အသုံးပြုခြင်းဖြင့် expression များ၏ ယေဘုယျ syntax ကို ဖော်ပြသွားမည်ဖြစ်သည်။

14 ပိုမိုယေဘုယျကျစွာ ဆိုရလျှင်၊ procedure တစ်ခု၏ body သည် expression များ၏ အစီအစဉ်တစ်ခု (sequence) ဖြစ်နိုင်သည်။ ဤအခြေအနေတွင်၊ interpreter သည် sequence ထဲရှိ expression တစ်ခုချင်းစီကို အစဉ်လိုက် တွက်ချက်ပြီး နောက်ဆုံး expression မှ ရရှိသော တန်ဖိုးကို procedure application ၏ တန်ဖိုးအဖြစ် ပြန်ပေးမည်ဖြစ်သည်။

15 substitution (အစားထိုးခြင်း) အယူအဆမှာ ရိုးရှင်းသော်လည်း၊ substitution လုပ်ငန်းစဉ်ကို တိကျသော သင်္ချာနည်းကျ အဓိပ္ပာယ်ဖွင့်ဆိုချက် ပေးရန်မှာ အံ့သြဖွယ်ကောင်းလောက်အောင် ရှုပ်ထွေးကြောင်း တွေ့ရှိရသည်။ procedure ၏ formal parameter များအတွက် အသုံးပြုသော အမည်များနှင့် procedure ကို apply လုပ်မည့် expression များတွင် အသုံးပြုထားသော (တူညီနိုင်သည့်) အမည်များအကြား ရောထွေးသွားနိုင်ခြေကြောင့် ဤပြဿနာ ဖြစ်ပေါ်လာခြင်းဖြစ်သည်။ အမှန်စင်စစ်၊ logic နှင့် programming semantics ဆိုင်ရာ စာပေများတွင် substitution ၏ မှားယွင်းသော အဓိပ္ပာယ်ဖွင့်ဆိုချက်များ ပေးခဲ့သည့် သမိုင်းကြောင်း အရှည်ကြီး ရှိခဲ့သည်။ substitution နှင့် ပတ်သက်၍ အသေးစိတ် ဆွေးနွေးချက်ကို Stoy 1977 တွင် ကြည့်ရှုပါ။

16 Chapter 3 တွင် ကျွန်ုပ်တို့သည် အကန့်အသတ်ရှိသော normal-order evaluation ပုံစံကို ထည့်သွင်းအသုံးပြုခြင်းဖြင့် "အကန့်အသတ်မရှိ" သော data structure များကို ကိုင်တွယ်ဖြေရှင်းသည့် နည်းလမ်းတစ်ခုဖြစ်သည့် stream processing ကို မိတ်ဆက်ပေးမည်ဖြစ်သည်။ 4.2 တွင် Scheme interpreter ကို normal-order အသုံးပြုသော Scheme မျိုးကွဲတစ်ခုအဖြစ် ပြောင်းလဲပြင်ဆင်သွားမည်ဖြစ်သည်။

17 “မှန်သည် သို့မဟုတ် မှားသည်ဟု အဓိပ္ပာယ်ကောက်ယူခြင်း” ဆိုသည်မှာ ဤသို့ဖြစ်သည်- Scheme တွင် #t နှင့် #f ဟူသော ကိန်းသေများဖြင့် ကိုယ်စားပြုသည့် ထူးခြားသော တန်ဖိုးနှစ်ခု ရှိသည်။ interpreter သည် predicate တစ်ခု၏ တန်ဖိုးကို စစ်ဆေးသောအခါ၊ ၎င်းသည် #f ကို false (မှား) အဖြစ် အဓိပ္ပာယ်ကောက်ယူသည်။ အခြား မည်သည့်တန်ဖိုးကိုမဆို true (မှန်) အဖြစ် သတ်မှတ်သည်။ (ထို့ကြောင့် #t ကို ပေးထားခြင်းမှာ logic အရ မလိုအပ်သော်လည်း အဆင်ပြေစေရန် ဖြစ်သည်။) ဤစာအုပ်တွင် ကျွန်ုပ်တို့သည် #t နှင့် #f တန်ဖိုးများနှင့် အသီးသီး ဆက်စပ်နေသော true နှင့် false ဟူသော အမည်များကို အသုံးပြုသွားမည်ဖြစ်သည်။

18 Abs သည် “minus” operator - ကိုလည်း အသုံးပြုထားသည်၊ ၎င်းကို (- x) တွင်ကဲ့သို့ operand တစ်ခုတည်းနှင့် အသုံးပြုသောအခါ အနုတ်လက္ခဏာ (negation) ကို ပြသည်။

19 if နှင့် cond တို့၏ အသေးစား ကွဲပြားချက်တစ်ခုမှာ cond clause တစ်ခုချင်းစီ၏ ⟨e⟩ အပိုင်းသည် expression များ၏ အစီအစဉ်တစ်ခု (sequence) ဖြစ်နိုင်သည်။ အကယ်၍ ၎င်းနှင့် သက်ဆိုင်သော ⟨p⟩ သည် မှန်ကန်ကြောင်း တွေ့ရှိပါက၊ ⟨e⟩ ရှိ expression များကို အစဉ်လိုက် တွက်ချက်ပြီး sequence အတွင်းရှိ နောက်ဆုံး expression ၏ တန်ဖိုးကို cond ၏ တန်ဖိုးအဖြစ် ပြန်ပေးသည်။ သို့သော် if expression တစ်ခုတွင်မူ ⟨consequent⟩ နှင့် ⟨alternative⟩ တို့သည် expression တစ်ခုတည်းသာ ဖြစ်ရမည်။

20 ဖော်ပြချက်ပေးသော (Declarative) နှင့် အမိန့်ပေးသော (imperative) ရှင်းလင်းချက်များသည် သင်္ချာနှင့် ကွန်ပျူတာသိပ္ပံကဲ့သို့ပင် အလွန်နီးကပ်စွာ ဆက်စပ်နေသည်။ ဥပမာအားဖြင့်၊ ပရိုဂရမ်တစ်ခုမှ ထုတ်ပေးသော အဖြေသည် “မှန်ကန်သည်” ဟု ပြောခြင်းသည် ပရိုဂရမ်နှင့် ပတ်သက်၍ declarative statement တစ်ခု ပြုလုပ်ခြင်းဖြစ်သည်။ ပရိုဂရမ်များ မှန်ကန်ကြောင်း သက်သေပြရန် နည်းစနစ်များ တည်ထောင်ရန် ရည်ရွယ်သည့် သုတေသနများစွာ ရှိပြီး၊ ဤဘာသာရပ်၏ နည်းပညာဆိုင်ရာ အခက်အခဲများစွာမှာ (ပရိုဂရမ်များ တည်ဆောက်ထားသည့်) imperative statement များနှင့် (အရာများကို ကောက်ချက်ချရန် အသုံးပြုနိုင်သည့်) declarative statement များအကြား ကူးပြောင်းမှုကို ကိုင်တွယ်ဖြေရှင်းခြင်းနှင့် သက်ဆိုင်သည်။ အလားတူပင်၊ ပရိုဂရမ်မင်းဘာသာစကား ဒီဇိုင်းတွင် အရေးကြီးသော လက်ရှိနယ်ပယ်တစ်ခုမှာ declarative statement များဖြင့်သာ ပရိုဂရမ်ရေးသားရသည့် အလွန်မြင့်မားသော အဆင့်ရှိ ဘာသာစကားများ (very high-level languages) ကို လေ့လာခြင်း ဖြစ်သည်။ ယင်း၏ အယူအဆမှာ interpreter များကို ပိုမိုခေတ်မီလာအောင် ပြုလုပ်ရန်ဖြစ်ပြီး၊ ပရိုဂရမ်မာမှ ပေးလိုက်သော "အဘယ်အရာဖြစ်သည်" (what is) ဟူသော အသိပညာမှတစ်ဆင့် "မည်သို့လုပ်ဆောင်ရမည်" (how to) ဟူသော အသိပညာကို အလိုအလျောက် ထုတ်ပေးနိုင်ရန် ဖြစ်သည်။ ၎င်းကို ယေဘုယျအားဖြင့် အားလုံးအတွက် မလုပ်ဆောင်နိုင်သေးသော်လည်း တိုးတက်မှုရရှိထားသော အရေးကြီးသည့် နယ်ပယ်များ ရှိပါသည်။ ဤအယူအဆကို Chapter 4 တွင် ပြန်လည် လေ့လာပါမည်။

21 ဤ square-root algorithm သည် အမှန်တကယ်တွင် ညီမျှခြင်းများ၏ root များကို ရှာဖွေရန်အတွက် ယေဘုယျ နည်းစနစ်တစ်ခုဖြစ်သော Newton’s method ၏ အထူးအခြေအနေတစ်ခု ဖြစ်သည်။ square-root algorithm ကိုယ်တိုင်ကို ခရစ်သက္ကရာဇ် ပထမရာစုနှစ်တွင် Heron of Alexandria က တီထွင်ခဲ့ခြင်း ဖြစ်သည်။ ယေဘုယျ Newton’s method ကို Lisp procedure တစ်ခုအဖြစ် မည်သို့ ဖော်ပြရမည်ကို 1.3.4 တွင် တွေ့ရမည်ဖြစ်သည်။

22 ကျွန်ုပ်တို့သည် predicate များကို ၎င်းတို့သည် predicate များဖြစ်ကြောင်း မှတ်မိစေရန်အတွက် များသောအားဖြင့် မေးခွန်းသင်္ကေတဖြင့် ဆုံးသော အမည်များ ပေးလေ့ရှိသည်။ ၎င်းသည် အရေးအသားဆိုင်ရာ ထုံးစံတစ်ခုသာ ဖြစ်သည်။ interpreter အတွက်မူ မေးခွန်းသင်္ကေတသည် သာမန် စာလုံးတစ်လုံးသာ ဖြစ်သည်။

23 ကျွန်ုပ်တို့၏ ကနဦး ခန့်မှန်းခြေတန်ဖိုးကို 1 အစား 1.0 ဟု ဖော်ပြထားသည်ကို သတိပြုပါ။ ၎င်းသည် Lisp implementation အတော်များများတွင် ထူးခြားမှု ရှိမည်မဟုတ်ပါ။ သို့သော် MIT Scheme သည် တိကျသော ကိန်းပြည့်များ (exact integers) နှင့် ဒသမတန်ဖိုးများ (decimal values) ကို ခွဲခြားထားပြီး၊ ကိန်းပြည့်နှစ်ခုကို စားခြင်းဖြင့် ဒသမကိန်းထက် ရာရှင်နယ်ကိန်း (rational number) ကိုသာ ထုတ်ပေးသည်။ ဥပမာအားဖြင့်၊ 10 ကို 6 နှင့် စားလျှင် 5/3 ရရှိပြီး၊ 10.0 ကို 6.0 နှင့် စားလျှင် 1.6666666666666667 ရရှိသည်။ (ရာရှင်နယ်ကိန်းများပေါ်တွင် သင်္ချာလုပ်ဆောင်ချက်များကို မည်သို့လုပ်ဆောင်ရမည်ကို 2.1.1 တွင် လေ့လာပါမည်။) အကယ်၍ ကျွန်ုပ်တို့၏ square-root ပရိုဂရမ်တွင် ကနဦး ခန့်မှန်းခြေတန်ဖိုး 1 ဖြင့် စတင်ခဲ့ပြီး x သည် တိကျသော ကိန်းပြည့်တစ်ခု ဖြစ်နေပါက၊ square-root တွက်ချက်မှုမှ ထွက်ပေါ်လာသော နောက်ဆက်တွဲ တန်ဖိုးအားလုံးသည် ဒသမကိန်းများမဟုတ်ဘဲ ရာရှင်နယ်ကိန်းများသာ ဖြစ်နေလိမ့်မည်။ ရာရှင်နယ်ကိန်းများနှင့် ဒသမကိန်းများ ရောနှောထားသော လုပ်ဆောင်ချက်များသည် အမြဲတမ်း ဒသမကိန်းများကိုသာ ထုတ်ပေးသောကြောင့်၊ ကနဦး ခန့်မှန်းခြေတန်ဖိုး 1.0 ဖြင့် စတင်ခြင်းသည် နောက်ဆက်တွဲ တန်ဖိုးအားလုံးကို ဒသမကိန်းများ ဖြစ်လာစေရန် အတင်းအကျပ် လုပ်ဆောင်လိုက်ခြင်း ဖြစ်သည်။

24 iteration (အကြိမ်ကြိမ်ပြုလုပ်ခြင်း) ကို အကောင်အထည်ဖော်ရန် procedure call များကို အသုံးပြုခြင်းနှင့် ပတ်သက်၍ ထိရောက်မှုဆိုင်ရာ ကိစ္စရပ်များကို စိုးရိမ်နေသော စာဖတ်သူများသည် 1.2.1 ရှိ “tail recursion” ဆိုင်ရာ မှတ်ချက်များကို ဖတ်ရှုသင့်သည်။

25 ဤ procedure များထဲမှ မည်သည်က ပိုမိုထိရောက်သော implementation ဖြစ်သည်ဆိုသည်မှာပင် ရှင်းရှင်းလင်းလင်း မရှိလှပါ။ ၎င်းသည် ရရှိနိုင်သော hardware ပေါ်တွင် မူတည်သည်။ "သိသာထင်ရှားသော" implementation သည် ပို၍ ထိရောက်မှုနည်းနေတတ်သည့် စက်ပစ္စည်းများလည်း ရှိသည်။ အလွန်ထိရောက်သော နည်းလမ်းဖြင့် သိမ်းဆည်းထားသည့် လော်ဂရစ်သမ် (logarithms) နှင့် အန်တီလော်ဂရစ်သမ် (antilogarithms) ဇယားများစွာ ရှိသော စက်တစ်ခုကို စဉ်းစားကြည့်ပါ။

26 တသမတ်တည်းရှိသော အမည်ပြောင်းလဲခြင်း (consistent renaming) အယူအဆသည် အမှန်တကယ်တွင် သိမ်မွေ့ပြီး တိကျစွာ အဓိပ္ပာယ်ဖွင့်ဆိုရန် ခက်ခဲသည်။ နာမည်ကျော် logic ပညာရှင်များပင် ဤနေရာတွင် ရှက်ဖွယ်ကောင်းသော အမှားများ ပြုလုပ်ခဲ့ဖူးသည်။

27 Lexical scoping ၏ သတ်မှတ်ချက်အရ procedure တစ်ခုအတွင်းရှိ free variable များသည် ၎င်းတို့ကို ဝန်းရံထားသော procedure definition များမှ ပြုလုပ်ထားသည့် binding များကို ရည်ညွှန်းသည်ဟု သတ်မှတ်သည်။ ဆိုလိုသည်မှာ ၎င်းတို့ကို ထို procedure အဓိပ္ပာယ်ဖွင့်ဆိုခဲ့သည့် environment ထဲတွင် ရှာဖွေခြင်းဖြစ်သည်။ ၎င်းသည် မည်သို့ အလုပ်လုပ်သည်ကို environment များနှင့် interpreter ၏ အသေးစိတ် အပြုအမူများကို လေ့လာမည့် အခန်း ၃ တွင် အသေးစိတ် တွေ့ရမည်ဖြစ်သည်။

28 Embedded definition များသည် procedure body တစ်ခု၏ ရှေ့ဆုံးတွင် လာရမည်ဖြစ်သည်။ definition နှင့် အသုံးပြုမှုကို ရောထွေးထားသော ပရိုဂရမ်များကို run ရာတွင် ဖြစ်ပေါ်လာမည့် နောက်ဆက်တွဲ အကျိုးဆက်များအတွက် စီမံခန့်ခွဲမှုပိုင်းမှ တာဝန်ယူမည်မဟုတ်ပါ။
