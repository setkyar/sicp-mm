5.5Compilation

၅.၄ ရှိ explicit-control evaluator သည် Scheme ပရိုဂရမ်များကို အဓိပ္ပာယ်ပြန်ဆိုပေးသည့် controller ပါဝင်သော register machine တစ်ခုဖြစ်သည်။ ဤကဏ္ဍတွင် Scheme interpreter မဟုတ်သော controller ပါဝင်သည့် register machine တစ်ခုပေါ်တွင် Scheme ပရိုဂရမ်များကို မည်သို့လုပ်ဆောင်ရမည်ကို လေ့လာသွားမည်ဖြစ်သည်။

explicit-control evaluator စက်သည် universal ဖြစ်သည်—၎င်းသည် Scheme ဖြင့် ဖော်ပြနိုင်သော မည်သည့် တွက်ချက်မှု လုပ်ငန်းစဉ်ကိုမဆို လုပ်ဆောင်နိုင်သည်။ Evaluator ၏ controller သည် လိုအပ်သော တွက်ချက်မှုများကို လုပ်ဆောင်ရန်အတွက် ၎င်း၏ data paths များ အသုံးပြုမှုကို စီစဉ်ညွှန်ကြားပေးသည်။ ထို့ကြောင့် evaluator ၏ data paths များသည် universal ဖြစ်သည်- သင့်လျော်သော controller ရှိပါက ကျွန်ုပ်တို့ အလိုရှိသော မည်သည့် တွက်ချက်မှုကိုမဆို လုပ်ဆောင်ရန် လုံလောက်ပါသည်။318

စီးပွားဖြစ် ယေဘုယျသုံး ကွန်ပျူတာများသည် ထိရောက်ပြီး အဆင်ပြေသော universal data paths များအဖြစ် စုစည်းထားသည့် registers နှင့် operations များပေါ်တွင် အခြေခံထားသော register machines များဖြစ်သည်။ ယေဘုယျသုံးစက်တစ်ခုအတွက် controller သည် ကျွန်ုပ်တို့ အသုံးပြုခဲ့သည့် register-machine ဘာသာစကားကဲ့သို့သော ဘာသာစကားအတွက် interpreter တစ်ခုဖြစ်သည်။ ဤဘာသာစကားကို native language သို့မဟုတ် ရိုးရှင်းစွာ machine language ဟု ခေါ်သည်။ Machine language ဖြင့် ရေးသားထားသော ပရိုဂရမ်များသည် စက်၏ data paths များကို အသုံးပြုသည့် ညွှန်ကြားချက်များ (instructions) ၏ အစီအစဉ်များဖြစ်သည်။ ဥပမာအားဖြင့်၊ explicit-control evaluator ၏ instruction sequence ကို အထူးပြု interpreter စက်တစ်ခုအတွက် controller အဖြစ်ထက် ယေဘုယျသုံး ကွန်ပျူတာတစ်လုံးအတွက် machine-language ပရိုဂရမ်တစ်ခုအဖြစ် မှတ်ယူနိုင်သည်။

အဆင့်မြင့် ဘာသာစကားများ (higher-level languages) နှင့် register-machine ဘာသာစကားများအကြား ကွာဟချက်ကို ပေါင်းကူးရန် ဘုံနည်းဗျူဟာ နှစ်ခုရှိသည်။ Explicit-control evaluator သည် interpretation နည်းဗျူဟာကို သရုပ်ဖော်သည်။ စက်၏ native language ဖြင့် ရေးသားထားသော interpreter တစ်ခုသည် တွက်ချက်မှုကို လုပ်ဆောင်နေသည့် စက်၏ native language နှင့် ကွဲပြားနိုင်သော ဘာသာစကား ( source language ဟုခေါ်သည်) ဖြင့် ရေးသားထားသော ပရိုဂရမ်များကို လုပ်ဆောင်နိုင်ရန် စက်ကို ပြင်ဆင်ပေးသည်။ Source language ၏ primitive procedures များကို ပေးထားသောစက်၏ native language ဖြင့် ရေးသားထားသော subroutines library တစ်ခုအဖြစ် အကောင်အထည်ဖော်ထားသည်။ အဓိပ္ပာယ်ပြန်ဆိုရမည့် ပရိုဂရမ် ( source program ဟုခေါ်သည်) ကို ဒေတာ တည်ဆောက်ပုံ (data structure) တစ်ခုအဖြစ် ဖော်ပြသည်။ Interpreter သည် ဤ data structure ကို ဖြတ်သန်းကြည့်ရှုပြီး source program ကို ခွဲခြမ်းစိတ်ဖြာသည်။ ထိုသို့လုပ်ဆောင်စဉ်တွင် ၎င်းသည် library မှ သင့်လျော်သော primitive subroutines များကို ခေါ်ယူခြင်းဖြင့် source program ၏ ရည်ရွယ်ထားသော လုပ်ဆောင်ချက်ကို တုပလုပ်ဆောင်သည်။

ဤကဏ္ဍတွင်၊ ကျွန်ုပ်တို့သည် compilation ဟူသော အခြားနည်းဗျူဟာတစ်ခုကို လေ့လာပါမည်။ ပေးထားသော source language နှင့် စက်တစ်ခုအတွက် compiler တစ်ခုသည် source program ကို စက်၏ native language ဖြင့် ရေးသားထားသော ညီမျှသည့် ပရိုဂရမ် ( object program ဟုခေါ်သည်) အဖြစ် ဘာသာပြန်ပေးသည်။ ဤကဏ္ဍတွင် ကျွန်ုပ်တို့ အကောင်အထည်ဖော်မည့် compiler သည် Scheme ဖြင့် ရေးသားထားသော ပရိုဂရမ်များကို explicit-control evaluator စက်၏ data paths များကို အသုံးပြု၍ လုပ်ဆောင်ရမည့် instructions အစီအစဉ်များအဖြစ် ဘာသာပြန်ပေးသည်။319

Interpretation နှင့် နှိုင်းယှဉ်ပါက၊ compilation သည် ပရိုဂရမ်လုပ်ဆောင်မှု၏ ထိရောက်မှု (efficiency) ကို များစွာတိုးတက်စေနိုင်ကြောင်း compiler ၏ ခြုံငုံသုံးသပ်ချက်တွင် အောက်၌ ရှင်းပြပါမည်။ အခြားတစ်ဖက်တွင်မူ interpreter သည် interactive program development နှင့် debugging အတွက် ပိုမိုအားကောင်းသော ပတ်ဝန်းကျင်ကို ပေးဆောင်သည်၊ အကြောင်းမှာ လုပ်ဆောင်နေသော source program ကို run time တွင် စစ်ဆေးခြင်းနှင့် ပြုပြင်မွမ်းမံခြင်းများ ပြုလုပ်နိုင်သောကြောင့်ဖြစ်သည်။ ထို့အပြင် primitive များ၏ library တစ်ခုလုံးရှိနေသောကြောင့် debugging လုပ်နေစဉ်အတွင်း ပရိုဂရမ်အသစ်များကို တည်ဆောက်ပြီး စနစ်ထဲသို့ ထည့်သွင်းနိုင်သည်။

Compilation နှင့် interpretation တို့၏ အပြန်အလှန် အထောက်အကူပြုသော အားသာချက်များကို ထောက်ရှု၍ ခေတ်မီ ပရိုဂရမ်ဖော်ဆောင်ရေး ပတ်ဝန်းကျင်များသည် ရောနှောထားသော နည်းဗျူဟာကို ကျင့်သုံးကြသည်။ Lisp interpreter များကို ယေဘုယျအားဖြင့် interpreted procedures များနှင့် compiled procedures များ အပြန်အလှန် ခေါ်ယူနိုင်ရန် ဖွဲ့စည်းထားသည်။ ၎င်းသည် ပရိုဂရမ်မာတစ်ဦးအား debug လုပ်ပြီးသားဟု ယူဆရသော ပရိုဂရမ်၏ အစိတ်အပိုင်းများကို compile လုပ်ခွင့်ပေးပြီး compilation ၏ ထိရောက်မှု အားသာချက်ကို ရရှိစေကာ၊ interactive development နှင့် debugging အဆင့်တွင် ရှိနေသေးသော အစိတ်အပိုင်းများအတွက် interpretive mode ကို ဆက်လက်အသုံးပြုနိုင်စေသည်။ 5.5.7 တွင်၊ compiler ကို အကောင်အထည်ဖော်ပြီးနောက် ပေါင်းစပ်ထားသော interpreter-compiler development system တစ်ခု ဖန်တီးရန် ၎င်းကို ကျွန်ုပ်တို့၏ interpreter နှင့် မည်သို့ ချိတ်ဆက်ရမည်ကို ပြသပါမည်။

Compiler ၏ ခြုံငုံသုံးသပ်ချက်

ကျွန်ုပ်တို့၏ compiler သည် ၎င်း၏ တည်ဆောက်ပုံနှင့် လုပ်ဆောင်ချက် နှစ်ခုလုံးတွင် ကျွန်ုပ်တို့၏ interpreter နှင့် များစွာ ဆင်တူသည်။ ထို့ကြောင့် expressions များကို ခွဲခြမ်းစိတ်ဖြာရန် compiler အသုံးပြုသည့် လုပ်ဆောင်ချက်များသည် interpreter အသုံးပြုသည့် အရာများနှင့် ဆင်တူလိမ့်မည်။ ထို့အပြင်၊ compiled code နှင့် interpreted code တို့ကို အလွယ်တကူ ချိတ်ဆက်နိုင်ရန်၊ compiler ကို interpreter ကဲ့သို့သော register အသုံးပြုမှုဆိုင်ရာ သတ်မှတ်ချက်များ (conventions) အတိုင်း လိုက်နာသည့် code များ ထုတ်ပေးရန် ဒီဇိုင်းဆွဲပါမည်- ပတ်ဝန်းကျင်ကို env register တွင် သိမ်းဆည်းမည်၊ argument lists များကို argl တွင် စုစည်းမည်၊ အသုံးပြုမည့် procedure သည် proc တွင် ရှိနေမည်၊ procedure များသည် ၎င်းတို့၏ အဖြေများကို val တွင် ပြန်ပေးမည်ဖြစ်ပြီး procedure ပြန်သွားရမည့် နေရာကို continue တွင် သိမ်းဆည်းထားမည်ဖြစ်သည်။ ယေဘုယျအားဖြင့်၊ compiler သည် source program တစ်ခုကို ၎င်း source program အား interpreter က evaluate လုပ်ရာတွင် လုပ်ဆောင်မည့် register operations များအတိုင်း အဓိကအားဖြင့် လုပ်ဆောင်ပေးသော object program တစ်ခုအဖြစ် ဘာသာပြန်ပေးသည်။

ဤဖော်ပြချက်သည် အခြေခံ compiler တစ်ခုကို အကောင်အထည်ဖော်ရန် နည်းဗျူဟာတစ်ခုကို အကြံပြုပါသည်- ကျွန်ုပ်တို့သည် expression ကို interpreter ကဲ့သို့ပင် ဖြတ်သန်းကြည့်ရှုသည်။ Expression ကို evaluate လုပ်ရာတွင် interpreter လုပ်ဆောင်မည့် register instruction တစ်ခုကို တွေ့သောအခါ၊ ကျွန်ုပ်တို့သည် ထို instruction ကို မလုပ်ဆောင်ဘဲ ၎င်းအစား အစီအစဉ်တစ်ခုအဖြစ် စုစည်းထားလိုက်သည်။ ရလဒ်အဖြစ် ရရှိလာသော instruction များ၏ အစီအစဉ်သည် object code ဖြစ်လိမ့်မည်။ Interpretation ထက် compilation ၏ ထိရောက်မှု အားသာချက်ကို သတိပြုပါ။ Interpreter က expression တစ်ခု—ဥပမာ (f 84 96)—ကို evaluate လုပ်တိုင်း၊ ၎င်းသည် expression ကို အမျိုးအစားခွဲခြားခြင်းနှင့် operand list ၏ အဆုံးကို စစ်ဆေးခြင်းတို့ကို လုပ်ဆောင်ရသည်။ Compiler တွင်မူ၊ compile time ၌ instruction sequence ကို ထုတ်လုပ်ချိန်တွင် expression ကို တစ်ကြိမ်သာ ခွဲခြမ်းစိတ်ဖြာသည်။ Compiler မှ ထုတ်လုပ်သော object code တွင် operator နှင့် operands နှစ်ခုကို evaluate လုပ်ခြင်း၊ argument list ကို စုစည်းခြင်းနှင့် procedure (proc တွင် ရှိသော) ကို arguments (argl တွင် ရှိသော) များသို့ apply လုပ်ခြင်း စသည့် instruction များသာ ပါဝင်သည်။

ဤသည်မှာ ၄.၁.၇ ရှိ analyzing evaluator တွင် ကျွန်ုပ်တို့ အကောင်အထည်ဖော်ခဲ့သော optimization မျိုးပင် ဖြစ်သည်။ သို့သော် compiled code တွင် ထိရောက်မှု ပိုမိုရရှိရန် အခြားအခွင့်အလမ်းများ ရှိပါသေးသည်။ Interpreter သည် အလုပ်လုပ်နေစဉ်တွင် ဘာသာစကားရှိ မည်သည့် expression အတွက်မဆို သက်ဆိုင်စေရမည့် လုပ်ငန်းစဉ်တစ်ခုကို လိုက်နာရသည်။ ဆန့်ကျင်ဘက်အနေဖြင့်၊ ပေးထားသော compiled code အစိတ်အပိုင်းတစ်ခုသည် သတ်မှတ်ထားသော expression တစ်ခုကို လုပ်ဆောင်ရန်သာ ရည်ရွယ်သည်။ ၎င်းသည် register များကို သိမ်းဆည်းရန် stack အသုံးပြုမှုတွင် ကြီးမားသော ကွဲပြားမှုကို ဖြစ်စေနိုင်သည်။ Interpreter က expression တစ်ခုကို evaluate လုပ်သောအခါ၊ ၎င်းသည် မမျှော်လင့်ထားသော အခြေအနေများအတွက် အသင့်ရှိနေရမည်ဖြစ်သည်။ Subexpression တစ်ခုကို evaluate မလုပ်မီ၊ interpreter သည် နောက်ပိုင်းတွင် လိုအပ်မည့် registers အားလုံးကို သိမ်းဆည်းထားရသည်၊ အကြောင်းမှာ subexpression သည် မည်သည့် evaluate လုပ်မှုကိုမဆို လိုအပ်နိုင်သောကြောင့်ဖြစ်သည်။ အခြားတစ်ဖက်တွင်မူ compiler သည် မလိုအပ်သော stack operations များကို ရှောင်ရှားသည့် code များ ထုတ်ပေးရန် ၎င်းလုပ်ဆောင်နေသော သတ်မှတ်ထားသော expression ၏ တည်ဆောက်ပုံကို အသုံးချနိုင်သည်။

သာဓကတစ်ခုအနေဖြင့် combination (f 84 96) ကို ကြည့်ပါ။ Interpreter သည် combination ၏ operator ကို evaluate မလုပ်မီ၊ နောက်ပိုင်းတွင် လိုအပ်မည့် operands နှင့် environment ပါဝင်သော registers များကို သိမ်းဆည်းခြင်းဖြင့် ဤ evaluation အတွက် ပြင်ဆင်သည်။ ထို့နောက် interpreter သည် operator ကို evaluate လုပ်ပြီး ရလဒ်ကို val တွင် ရယူကာ၊ သိမ်းဆည်းထားသော registers များကို ပြန်လည် ထည့်သွင်းပြီး၊ နောက်ဆုံးတွင် ရလဒ်ကို val မှ proc သို့ ရွှေ့ပြောင်းသည်။ သို့သော် ကျွန်ုပ်တို့ ကိုင်တွယ်နေသော သတ်မှတ်ထားသော expression တွင်၊ operator သည် symbol f ဖြစ်ပြီး၊ ၎င်းကို evaluate လုပ်ခြင်းအား မည်သည့် registers ကိုမျှ မပြောင်းလဲစေသော machine operation lookup-variable-value ဖြင့် ပြီးမြောက်စေသည်။ ဤကဏ္ဍတွင် ကျွန်ုပ်တို့ အကောင်အထည်ဖော်မည့် compiler သည် ဤအချက်ကို အခွင့်ကောင်းယူကာ operator ကို အောက်ပါ instruction အသုံးပြု၍ evaluate လုပ်မည့် code ကို ထုတ်ပေးမည်ဖြစ်သည်- 

(assign proc 
        (op lookup-variable-value)
        (const f)
        (reg env))

ဤ code သည် မလိုအပ်သော saves နှင့် restores များကို ရှောင်ရှားရုံသာမက lookup မှရရှိသော တန်ဖိုးကို proc သို့ တိုက်ရိုက် သတ်မှတ်ပေးသည်၊ interpreter မှာမူ ရလဒ်ကို val တွင် အရင်ရယူပြီးမှ proc သို့ ရွှေ့ပြောင်းရမည်ဖြစ်သည်။

Compiler သည် environment သို့ access ပြုလုပ်ခြင်းကိုလည်း optimize လုပ်နိုင်သည်။ Code ကို ခွဲခြမ်းစိတ်ဖြာပြီးနောက်၊ compiler သည် များစွာသော အခြေအနေများတွင် သတ်မှတ်ထားသော variable တစ်ခုသည် မည်သည့် frame တွင် ရှိနေမည်ကို သိရှိနိုင်ပြီး lookup-variable-value ရှာဖွေမှုကို လုပ်ဆောင်မည့်အစား ထို frame ကို တိုက်ရိုက် access လုပ်နိုင်သည်။ ထိုသို့သော variable access ကို မည်သို့ အကောင်အထည်ဖော်ရမည်ကို 5.5.6 တွင် ဆွေးနွေးပါမည်။ သို့သော် ထိုအချိန်အထိ ကျွန်ုပ်တို့သည် အထက်တွင် ဖော်ပြခဲ့သော register နှင့် stack optimizations များကိုသာ အာရုံစိုက်ပါမည်။ Compiler တစ်ခုမှ လုပ်ဆောင်နိုင်သော အခြား optimization များစွာ ရှိပါသေးသည်၊ ဥပမာ- ယေဘုယျ apply mechanism ကို အသုံးပြုမည့်အစား primitive operations များကို “in line” code ရေးခြင်းမျိုး (လေ့ကျင့်ခန်း ၅.၃၈ ကိုကြည့်ပါ)။ သို့သော် ကျွန်ုပ်တို့သည် ၎င်းတို့ကို ဤနေရာတွင် အထူးတလည် ဖော်ပြမည်မဟုတ်ပါ။ ဤကဏ္ဍတွင် ကျွန်ုပ်တို့၏ အဓိက ပန်းတိုင်မှာ ရိုးရှင်းသော (သို့သော် စိတ်ဝင်စားစရာကောင်းသော) အကြောင်းအရာတစ်ခုတွင် compilation လုပ်ငန်းစဉ်ကို သရုပ်ဖော်ရန် ဖြစ်သည်။

4.1.7 တွင် ကျွန်ုပ်တို့သည် မူရင်း metacircular interpreter ကို လေ့လာဆန်းစစ်ခြင်း (analysis) နှင့် အကောင်အထည်ဖော်ခြင်း (execution) တို့ကို သီးခြားစီ ခွဲထုတ်ရန် ပြင်ဆင်ခဲ့ကြသည်။ ကျွန်ုပ်တို့သည် expression တစ်ခုချင်းစီကို environment တစ်ခုကို argument အဖြစ် လက်ခံပြီး လိုအပ်သော လုပ်ဆောင်ချက်များကို လုပ်ဆောင်ပေးမည့် execution procedure တစ်ခု ထုတ်လုပ်ရန် လေ့လာဆန်းစစ်ခဲ့ကြသည်။ ကျွန်ုပ်တို့၏ compiler တွင်လည်း အခြေခံအားဖြင့် ထိုကဲ့သို့သော လေ့လာဆန်းစစ်မှုကိုပင် ပြုလုပ်မည်ဖြစ်သည်။ သို့သော် execution procedures များ ထုတ်လုပ်မည့်အစား ကျွန်ုပ်တို့၏ register machine မှ အလုပ်လုပ်နိုင်မည့် ညွှန်ကြားချက်အစီအစဉ်များ (sequences of instructions) ကို ထုတ်လုပ်ပေးမည်ဖြစ်သည်။

compile procedure သည် compiler ၏ အမြင့်ဆုံးအဆင့် dispatch ဖြစ်သည်။ ၎င်းသည် 4.1.1 ရှိ eval procedure၊ 4.1.7 ရှိ analyze procedure နှင့် 5.4.1 ရှိ explicit-control-evaluator ၏ eval-dispatch entry point တို့နှင့် သက်ဆိုင်ပါသည်။ Compiler သည် interpreter များကဲ့သို့ပင် 4.1.2 တွင် သတ်မှတ်ထားသော expression-syntax procedure များကို အသုံးပြုသည်။320 Compile သည် compile လုပ်ရမည့် expression ၏ syntactic အမျိုးအစားပေါ် မူတည်ပြီး ခွဲခြားစိတ်ဖြာမှုကို ပြုလုပ်ပေးသည်။ Expression အမျိုးအစား တစ်ခုချင်းစီအတွက် ၎င်းသည် သီးသန့် code generator တစ်ခုသို့ dispatch လုပ်ပေးသည်: 

(define (compile exp target linkage)
  (cond ((self-evaluating? exp)
         (compile-self-evaluating 
          exp target linkage))
        ((quoted? exp) 
         (compile-quoted exp target linkage))
        ((variable? exp)
         (compile-variable 
          exp target linkage))
        ((assignment? exp)
         (compile-assignment
          exp target linkage))
        ((definition? exp)
         (compile-definition
          exp target linkage))
        ((if? exp)
         (compile-if exp target linkage))
        ((lambda? exp)
         (compile-lambda exp target linkage))
        ((begin? exp)
         (compile-sequence 
          (begin-actions exp) target linkage))
        ((cond? exp) 
         (compile 
          (cond->if exp) target linkage))
        ((application? exp)
         (compile-application 
          exp target linkage))
        (else
         (error "Unknown expression type: 
                 COMPILE" 
                exp))))

Compile နှင့် ၎င်းခေါ်ဆိုသော code generator များသည် compile လုပ်မည့် expression အပြင် နောက်ထပ် argument နှစ်ခုကိုပါ လက်ခံကြသည်။ ပထမတစ်ခုမှာ target ဖြစ်ပြီး၊ ၎င်းသည် compile လုပ်ထားသော code က expression ၏ တန်ဖိုးကို မည်သည့် register ထဲသို့ ပြန်ပို့ပေးရမည်ကို သတ်မှတ်ပေးသည်။ နောက်တစ်ခုမှာ linkage descriptor ဖြစ်ပြီး၊ ၎င်းသည် expression ကို compile လုပ်ပြီး ထွက်လာသော code က ၎င်း၏ လုပ်ဆောင်ချက် ပြီးဆုံးသောအခါ မည်သို့ဆက်လက် လုပ်ဆောင်ရမည်ကို ဖော်ပြသည်။ Linkage descriptor သည် code အား အောက်ပါ အချက်သုံးချက်အနက်မှ တစ်ခုကို လုပ်ဆောင်ရန် လိုအပ်နိုင်သည်:

  • အစဉ်အတိုင်း နောက်ထပ်ညွှန်ကြားချက်တွင် ဆက်လက်လုပ်ဆောင်ရန် (၎င်းကို linkage descriptor next ဖြင့် သတ်မှတ်သည်)၊
  • compile လုပ်နေသော procedure မှ ပြန်ထွက်ရန် (၎င်းကို linkage descriptor return ဖြင့် သတ်မှတ်သည်)၊ သို့မဟုတ်
  • အမည်ပေးထားသော entry point တစ်ခုသို့ ခုန်ကူးရန် (၎င်းကို သတ်မှတ်ထားသော label အား linkage descriptor အဖြစ် အသုံးပြု၍ သတ်မှတ်သည်)။

Code generator တစ်ခုချင်းစီသည် expression အတွက် ၎င်းထုတ်လုပ်လိုက်သော object code များ ပါဝင်သော instruction sequence တစ်ခုကို ပြန်ပေးသည်။ Compound expression တစ်ခုအတွက် code ထုတ်လုပ်ခြင်းကို ၎င်း၏ အစိတ်အပိုင်း expression များအတွက် ပိုမိုရိုးရှင်းသော code generator များမှ ထွက်လာသော ရလဒ်များကို ပေါင်းစပ်ခြင်းဖြင့် လုပ်ဆောင်သည်။ ၎င်းသည် compound expression တစ်ခုကို ၎င်း၏ အစိတ်အပိုင်းများကို evaluate လုပ်ခြင်းဖြင့် လုပ်ဆောင်သည့် ပုံစံနှင့် အတူတူပင်ဖြစ်သည်။

Instruction sequence များကို ပေါင်းစပ်ရန် အရိုးရှင်းဆုံးနည်းလမ်းမှာ append-instruction-sequences ဟု ခေါ်သော procedure တစ်ခုဖြစ်သည်။ ၎င်းသည် အစဉ်လိုက် လုပ်ဆောင်ရမည့် instruction sequence အရေအတွက် မည်မျှကိုမဆို argument များအဖြစ် လက်ခံပြီး ၎င်းတို့ကို ပေါင်းစပ်ကာ ရလဒ် sequence ကို ပြန်ပေးသည်။ ဆိုလိုသည်မှာ အကယ်၍ ⟨ s e q 1 ⟩ နှင့် ⟨ s e q 2 ⟩ တို့သည် ညွှန်ကြားချက် အစဉ်အတန်းများ ဖြစ်ပါက၊ (append-instruction-sequences ⟨seq₁⟩ ⟨seq₂⟩) ကို evaluate လုပ်ခြင်းဖြင့်

Instruction sequence များကို ပေါင်းစပ်ရန် preserving ကို အသုံးပြုခြင်းဖြင့် compiler သည် မလိုအပ်သော stack လုပ်ဆောင်ချက်များကို ရှောင်ရှားနိုင်သည်။ ၎င်းသည် save နှင့် restore ညွှန်ကြားချက်များကို ထုတ်လုပ်ရန် လိုမလိုဆိုသည့် အသေးစိတ်အချက်များကို preserving procedure အတွင်း၌သာ ကန့်သတ်ထားနိုင်စေပြီး၊ သီးခြား code generator တစ်ခုချင်းစီကို ရေးသားရာတွင် ထည့်သွင်းစဉ်းစားရမည့် အချက်များမှ ခွဲထုတ်ပေးသည်။ အမှန်စင်စစ်တွင် code generator များသည် save သို့မဟုတ် restore ညွှန်ကြားချက်များကို တိုက်ရိုက် ထုတ်လုပ်ခြင်း မရှိပါ။

မူအားဖြင့် ကျွန်ုပ်တို့သည် instruction sequence တစ်ခုကို ညွှန်ကြားချက်များ၏ list တစ်ခုအဖြစ် ရိုးရှင်းစွာ ကိုယ်စားပြုနိုင်သည်။ ထိုအခါ append-instruction-sequences သည် သာမန် list append ကို အသုံးပြုပြီး instruction sequence များကို ပေါင်းစပ်နိုင်မည်ဖြစ်သည်။ သို့သော် ထိုအခါ preserving သည် ရှုပ်ထွေးသော လုပ်ဆောင်ချက်တစ်ခု ဖြစ်လာလိမ့်မည်။ အကြောင်းမှာ ၎င်းသည် instruction sequence တစ်ခုချင်းစီက ၎င်း၏ register များကို မမည်သို့အသုံးပြုသည်ကို သိရှိနိုင်ရန် လေ့လာဆန်းစစ်ရမည် ဖြစ်သောကြောင့်ဖြစ်သည်။ Preserving သည် ရှုပ်ထွေးရုံသာမက ထိရောက်မှုလည်း ရှိမည်မဟုတ်ပါ။ အကြောင်းမှာ ၎င်းသည် instruction sequence argument တစ်ခုချင်းစီကို လေ့လာဆန်းစစ်ရမည် ဖြစ်သောကြောင့်ဖြစ်သည်။ ထို sequence များသည် preserving ကို ခေါ်ဆိုခြင်းဖြင့်ပင် တည်ဆောက်ထားခြင်း ဖြစ်နိုင်ပြီး ၎င်းတို့၏ အစိတ်အပိုင်းများကို လေ့လာဆန်းစစ်ပြီးသား ဖြစ်နေနိုင်သော်လည်း ထပ်မံလုပ်ဆောင်ရမည် ဖြစ်သည်။ ထိုကဲ့သို့ ထပ်တလဲလဲ လေ့လာဆန်းစစ်မှုများကို ရှောင်ရှားရန် ကျွန်ုပ်တို့သည် instruction sequence တစ်ခုချင်းစီနှင့်အတူ ၎င်း၏ register အသုံးပြုမှုဆိုင်ရာ အချက်အလက်အချို့ကို တွဲဖက်ထားမည်ဖြစ်သည်။ အခြေခံ instruction sequence တစ်ခုကို တည်ဆောက်သည့်အခါ ထိုအချက်အလက်များကို အတိအလင်း ထည့်သွင်းပေးမည်ဖြစ်ပြီး၊ instruction sequence များကို ပေါင်းစပ်သည့် procedure များက အစိတ်အပိုင်း sequence များရှိ အချက်အလက်များမှတစ်ဆင့် ပေါင်းစပ်ထားသော sequence အတွက် register အသုံးပြုမှု အချက်အလက်ကို တွက်ချက်ရယူမည်ဖြစ်သည်။

Instruction sequence တစ်ခုတွင် အချက်အလက် သုံးမျိုး ပါဝင်လိမ့်မည်:

  • sequence ထဲရှိ ညွှန်ကြားချက်များကို မလုပ်ဆောင်မီ initialize လုပ်ထားရမည့် register များ အစုအဝေး (ဤ register များကို sequence က needed (လိုအပ်သည်) ဟု ဆိုသည်)၊
  • sequence ထဲရှိ ညွှန်ကြားချက်များကြောင့် တန်ဖိုးများ ပြောင်းလဲသွားမည့် register များ အစုအဝေး၊ နှင့်
  • sequence ထဲရှိ အမှန်တကယ် လုပ်ဆောင်မည့် ညွှန်ကြားချက်များ (၎င်းတို့ကို statements ဟုလည်း ခေါ်သည်)။

ကျွန်ုပ်တို့သည် instruction sequence တစ်ခုကို ၎င်း၏ အစိတ်အပိုင်း သုံးခုပါဝင်သော list တစ်ခုအဖြစ် ကိုယ်စားပြုမည်ဖြစ်သည်။ ထို့ကြောင့် instruction sequence များအတွက် constructor မှာ အောက်ပါအတိုင်း ဖြစ်သည် 

(define (make-instruction-sequence 
         needs modifies statements)
  (list needs modifies statements))

(make-instruction-sequence
 '(env continue)
 '(val)
 '((assign val
           (op lookup-variable-value)
           (const x)
           (reg env))
   (goto (reg continue))))

Instruction sequence များကို ပေါင်းစပ်ရန် procedure များကို 5.5.4 တွင် ပြသထားသည်။

လေ့ကျင့်ခန်း ၅.၃၁: Procedure application တစ်ခုကို evaluate လုပ်ရာတွင် explicit-control evaluator သည် operator ကို evaluate မလုပ်မီနှင့် လုပ်ပြီးချိန်တွင် env register ကို အမြဲတမ်း save နှင့် restore လုပ်လေ့ရှိသည်။ ထို့အတူ operand တစ်ခုချင်းစီ (နောက်ဆုံးတစ်ခုမှလွဲ၍) ကို evaluate လုပ်ရာတွင် env ကို save နှင့် restore လုပ်ခြင်း၊ operand တစ်ခုချင်းစီအတွက် argl ကို save နှင့် restore လုပ်ခြင်း၊ နှင့် operand sequence တစ်ခုလုံးကို evaluate လုပ်ရာတွင် proc ကို save နှင့် restore လုပ်ခြင်းတို့ကို ပြုလုပ်သည်။ အောက်ပါ combination တစ်ခုချင်းစီအတွက် မည်သည့် save နှင့် restore လုပ်ဆောင်ချက်များသည် မလိုအပ် (superfluous) ဖြစ်နေပြီး compiler ၏ preserving ယန္တရားဖြင့် ဖယ်ရှားနိုင်သည်ကို ဖော်ပြပါ: 

(f 'x 'y)
((f) 'x 'y)
(f (g 'x) y)
(f (g 'x) 'y)

လေ့ကျင့်ခန်း ၅.၃၂: Preserving ယန္တရားကို အသုံးပြုခြင်းဖြင့်၊ compiler သည် combination ၏ operator မှာ symbol တစ်ခု ဖြစ်နေပါက ၎င်းကို evaluate လုပ်ရာတွင် env ကို save နှင့် restore လုပ်ခြင်းအား ရှောင်ရှားလိမ့်မည်။ ထိုကဲ့သို့သော optimization မျိုးကို evaluator တွင်လည်း ထည့်သွင်းတည်ဆောက်နိုင်သည်။ အမှန်တကယ်ပင် 5.4 ရှိ explicit-control evaluator သည် operand မပါသော combination များကို အထူးအခြေအနေ (special case) အဖြစ် သတ်မှတ်ခြင်းဖြင့် အလားတူ optimization မျိုးကို လုပ်ဆောင်ထားပြီးဖြစ်သည်။

  1. Operator မှာ symbol ဖြစ်နေသော combination များကို သီးခြား expression အမျိုးအစားအဖြစ် ခွဲခြားသိမြင်နိုင်ရန်နှင့် ထိုကဲ့သို့သော expression များကို evaluate လုပ်ရာတွင် အဆိုပါအချက်ကို အသုံးချနိုင်ရန် explicit-control evaluator ကို တိုးချဲ့ပါ။

5.5.2ဖော်ပြချက်များကို ကွန်ပိုင်းလုပ်ခြင်း (Compiling Expressions)

ယခုအပိုင်းနှင့် နောက်အပိုင်းတွင် compile ပရိုစီဂျာမှ တဆင့်ခေါ်ယူအသုံးပြုမည့် ကုတ်ထုတ်လုပ်ပေးသည့်အပိုင်း (code generators) များကို ကျွန်ုပ်တို့ အကောင်အထည်ဖော်ပါမည်။

ချိတ်ဆက်မှုကုတ် (Linkage code) ကို ကွန်ပိုင်းလုပ်ခြင်း

ယေဘုယျအားဖြင့်၊ ကုတ်ထုတ်လုပ်ပေးသည့်အပိုင်းတစ်ခုချင်းစီ၏ ရလဒ်သည် လိုအပ်သော ချိတ်ဆက်မှုကို အကောင်အထည်ဖော်သည့် ညွှန်ကြားချက်များ—compile-linkage ပရိုစီဂျာက ထုတ်လုပ်ပေးသော—နှင့် အဆုံးသတ်မည်ဖြစ်သည်။ အကယ်၍ ချိတ်ဆက်မှုသည် return ဖြစ်ပါက (goto (reg continue)) ဟူသော ညွှန်ကြားချက်ကို ကျွန်ုပ်တို့ ထုတ်လုပ်ရပါမည်။ ၎င်းသည် continue ရက်ဂျစ်စတာကို လိုအပ်ပြီး မည်သည့် ရက်ဂျစ်စတာကိုမျှ ပြုပြင်ပြောင်းလဲခြင်း မရှိပါ။ အကယ်၍ ချိတ်ဆက်မှုသည် next ဖြစ်ပါက နောက်ထပ် ညွှန်ကြားချက်များ ထည့်သွင်းရန် မလိုပါ။ ထိုသို့မဟုတ်ဘဲ ချိတ်ဆက်မှုသည် လေဘယ် (label) တစ်ခုဖြစ်ပါက ထိုလေဘယ်သို့ goto လုပ်သည့် ညွှန်ကြားချက်ကို ထုတ်လုပ်ပေးပြီး၊ ထိုညွှန်ကြားချက်သည် မည်သည့် ရက်ဂျစ်စတာကိုမျှ လိုအပ်ခြင်း သို့မဟုတ် ပြုပြင်ခြင်း မရှိပါ။321 

(define (compile-linkage linkage)
  (cond ((eq? linkage 'return)
         (make-instruction-sequence 
          '(continue)
          '()
          '((goto (reg continue)))))
        ((eq? linkage 'next)
         (empty-instruction-sequence))
        (else
         (make-instruction-sequence '() '()
          `((goto (label ,linkage)))))))

ချိတ်ဆက်မှုကုတ်ကို ညွှန်ကြားချက် အစီအစဉ် (instruction sequence) တစ်ခုတွင် continue ရက်ဂျစ်စတာကို preserving လုပ်ခြင်းဖြင့် ပေါင်းစပ်ထားပါသည်၊ အကြောင်းမှာ return ချိတ်ဆက်မှုသည် continue ရက်ဂျစ်စတာကို လိုအပ်မည်ဖြစ်သောကြောင့် ဖြစ်သည်- အကယ်၍ ပေးထားသော ညွှန်ကြားချက် အစီအစဉ်သည် continue ကို ပြုပြင်ပြီး ချိတ်ဆက်မှုကုတ်က ၎င်းကို လိုအပ်ပါက၊ continue ကို သိမ်းဆည်းပြီး ပြန်လည် ထည့်သွင်းပေးမည် ဖြစ်သည်။ 

(define (end-with-linkage 
         linkage instruction-sequence)
  (preserving '(continue)
   instruction-sequence
   (compile-linkage linkage)))
ရိုးရှင်းသော ဖော်ပြချက်များကို ကွန်ပိုင်းလုပ်ခြင်း

မိမိကိုယ်တိုင် တန်ဖိုးဖြတ်သည့် ဖော်ပြချက်များ (self-evaluating expressions)၊ ကိုးကားချက်များ (quotations) နှင့် ကိန်းရှင်များ (variables) အတွက် ကုတ်ထုတ်လုပ်ပေးသည့် အပိုင်းများသည် လိုအပ်သော တန်ဖိုးကို ပစ်မှတ်ရက်ဂျစ်စတာ (target register) သို့ သတ်မှတ်ပေးပြီး၊ ထို့နောက် ချိတ်ဆက်မှု ဖော်ပြချက် (linkage descriptor) အတိုင်း ဆက်လက်လုပ်ဆောင်သည့် ညွှန်ကြားချက် အစီအစဉ်များကို တည်ဆောက်ပါသည်။ 

(define (compile-self-evaluating 
         exp target linkage)
  (end-with-linkage
   linkage (make-instruction-sequence 
            '()
            (list target)
            `((assign ,target (const ,exp))))))

(define (compile-quoted exp target linkage)

    (list target)

       ,target

(define (compile-variable

ဤတန်ဖိုးသတ်မှတ်ခြင်း ညွှန်ကြားချက်အားလုံးသည် ပစ်မှတ်ရက်ဂျစ်စတာကို ပြုပြင်ပြောင်းလဲကြပြီး၊ ကိန်းရှင်ကို ရှာဖွေသည့် ညွှန်ကြားချက်သည် env ရက်ဂျစ်စတာကို လိုအပ်ပါသည်။

တန်ဖိုးသတ်မှတ်ခြင်း (Assignments) နှင့် အဓိပ္ပာယ်ဖွင့်ဆိုခြင်း (definitions) များကို အင်တာပရီတာ (interpreter) ထဲတွင် လုပ်ဆောင်သကဲ့သို့ပင် ကိုင်တွယ်ပါသည်။ ကိန်းရှင်တွင် သတ်မှတ်မည့် တန်ဖိုးကို တွက်ချက်သည့် ကုတ်ကို ကျွန်ုပ်တို့ ရစ်ကာဆစ် (recursively) ထုတ်လုပ်ပြီး၊ ၎င်းနောက်တွင် ကိန်းရှင်ကို အမှန်တကယ် သတ်မှတ်ပေးသည့် သို့မဟုတ် အဓိပ္ပာယ်ဖွင့်ဆိုပေးသည့် ညွှန်ကြားချက် နှစ်ခုပါ အစီအစဉ်ကို ပေါင်းစပ်ပါသည်။ ထိုအစီအစဉ်သည် ဖော်ပြချက်တစ်ခုလုံး၏ တန်ဖိုး (ok သင်္ကေတ) ကို ပစ်မှတ်ရက်ဂျစ်စတာသို့ သတ်မှတ်ပေးပါသည်။ ရစ်ကာဆစ် ကွန်ပိုင်းလုပ်ခြင်းတွင် ပစ်မှတ်အဖြစ် val နှင့် ချိတ်ဆက်မှုအဖြစ် next ကို ထားရှိသောကြောင့် ကုတ်သည် ၎င်း၏ ရလဒ်ကို val ထဲသို့ ထည့်သွင်းပြီး ၎င်းနောက်တွင် ပေါင်းစပ်ထားသော ကုတ်ကို ဆက်လက်လုပ်ဆောင်မည် ဖြစ်သည်။ ဤသို့ ပေါင်းစပ်ရာတွင် env ကို ထိန်းသိမ်းခြင်း (preserving) ပြုလုပ်ပါသည်၊ အကြောင်းမှာ ကိန်းရှင်ကို သတ်မှတ်ရန် သို့မဟုတ် အဓိပ္ပာယ်ဖွင့်ဆိုရန် ပတ်ဝန်းကျင် (environment) ကို လိုအပ်ပြီး၊ ကိန်းရှင်တန်ဖိုးအတွက် ကုတ်သည် ရက်ဂျစ်စတာများကို စိတ်ကြိုက် ပြုပြင်နိုင်သည့် ရှုပ်ထွေးသော ဖော်ပြချက်တစ်ခု၏ ကွန်ပိုင်းလုပ်ထားသော ကုတ် ဖြစ်နိုင်သောကြောင့် ဖြစ်သည်။ 

(define (compile-assignment 
         exp target linkage)
  (let ((var (assignment-variable exp))
        (get-value-code
         (compile (assignment-value exp) 
                  'val
                  'next)))
    (end-with-linkage 
     linkage
     (preserving 
      '(env)
      get-value-code

       '(env val)
       (list target)

                  (reg val)
                  (reg env))
         (assign ,target (const ok))))))))

 ⟨ပေးထားသော ပစ်မှတ်နှင့် ချိတ်ဆက်မှုဖြင့် 
  အခြားရွေးချယ်စရာကို ကွန်ပိုင်းလုပ်ခြင်း⟩
after-if

ဤကုတ်ကို ထုတ်လုပ်ရန်အတွက်၊ ကျွန်ုပ်တို့သည် ပရီဒီကိတ် (predicate)၊ အကျိုးဆက် (consequent) နှင့် အခြားရွေးချယ်စရာ (alternative) တို့ကို ကွန်ပိုင်းလုပ်ပြီး၊ ရရှိလာသော ကုတ်များကို ပရီဒီကိတ်ရလဒ်ကို စမ်းသပ်မည့် ညွှန်ကြားချက်များ၊ မှန်ကန်သော လမ်းကြောင်း (true branch)၊ မှားယွင်းသော လမ်းကြောင်း (false branch) နှင့် အခြေအနေပြမှု အဆုံးသတ်တို့ကို မှတ်သားရန် အသစ်ထုတ်လုပ်ထားသော လေဘယ်လ်များနှင့် ပေါင်းစပ်ပါသည်။322 ဤကုတ်စီစဉ်မှုတွင်၊ စမ်းသပ်မှု မှားယွင်းပါက မှန်ကန်သောလမ်းကြောင်းကို ကျော်ဖြတ်ရန် ကျွန်ုပ်တို့ လမ်းခွဲရပါမည်။ တစ်ခုတည်းသော အနည်းငယ် ရှုပ်ထွေးသည့်အချက်မှာ မှန်ကန်သော လမ်းကြောင်းအတွက် ချိတ်ဆက်မှုကို မည်သို့ကိုင်တွယ်ရမည်နည်း ဆိုသည့် အချက်ဖြစ်သည်။ အကယ်၍ အခြေအနေပြမှုအတွက် ချိတ်ဆက်မှုသည် return သို့မဟုတ် လေဘယ်လ်တစ်ခု ဖြစ်ပါက၊ မှန်ကန်သော လမ်းကြောင်းနှင့် မှားယွင်းသော လမ်းကြောင်း နှစ်ခုစလုံးသည် ထိုချိတ်ဆက်မှုကိုပင် အသုံးပြုပါလိမ့်မည်။ အကယ်၍ ချိတ်ဆက်မှုသည် next ဖြစ်ပါက၊ မှန်ကန်သော လမ်းကြောင်းသည် မှားယွင်းသော လမ်းကြောင်းအတွက် ကုတ်ကို ကျော်ဖြတ်ပြီး အခြေအနေပြမှု အဆုံးရှိ လေဘယ်လ်သို့ ခုန်ကူးခြင်း (jump) ဖြင့် အဆုံးသတ်ပါသည်။ 

(define (compile-if exp target linkage)
  (let ((t-branch (make-label 'true-branch))
        (f-branch (make-label 'false-branch))
        (after-if (make-label 'after-if)))
    (let ((consequent-linkage
           (if (eq? linkage 'next) 
               after-if
               linkage)))
      (let ((p-code 
             (compile (if-predicate exp)
                      'val
                      'next))
            (c-code
             (compile (if-consequent exp) 
                      target 
                      consequent-linkage))
            (a-code
             (compile (if-alternative exp)
                      target
                      linkage)))
        (preserving 
         '(env continue)
         p-code
         (append-instruction-sequences
          (make-instruction-sequence 
           '(val) 
           '()
           `((test (op false?) (reg val))
             (branch (label ,f-branch))))
          (parallel-instruction-sequences
           (append-instruction-sequences 
            t-branch c-code)
           (append-instruction-sequences
            f-branch a-code))
          after-if))))))

ပရီဒီကိတ် ကုတ်တဝိုက်တွင် Env ကို ထိန်းသိမ်းထားပါသည်၊ အကြောင်းမှာ ၎င်းကို မှန်ကန်သော နှင့် မှားယွင်းသော လမ်းကြောင်းများက လိုအပ်နိုင်သောကြောင့် ဖြစ်ပြီး၊ continue ကိုလည်း ထိုလမ်းကြောင်းများရှိ ချိတ်ဆက်မှုကုတ်များက လိုအပ်နိုင်သောကြောင့် ထိန်းသိမ်းထားပါသည်။ (အစဉ်လိုက် မဟုတ်ဘဲ လုပ်ဆောင်မည့်) မှန်ကန်သော နှင့် မှားယွင်းသော လမ်းကြောင်းများအတွက် ကုတ်ကို 5.5.4 တွင် ဖော်ပြထားသော အထူးပေါင်းစပ်ကိရိယာ parallel-instruction-sequences ကို အသုံးပြု၍ ပေါင်းစပ်ထားပါသည်။

cond သည် ဆင်းသက်လာသော ဖော်ပြချက် (derived expression) တစ်ခု ဖြစ်သောကြောင့်၊ ၎င်းကို ကိုင်တွယ်ရန် ကွန်ပိုင်လာ လုပ်ဆောင်ရန် လိုအပ်သည်မှာ ( 4.1.2 မှ) cond->if အသွင်ပြောင်းလဲကိရိယာကို အသုံးပြုပြီး ရရှိလာသော if ဖော်ပြချက်ကို ကွန်ပိုင်းလုပ်ရန်သာ ဖြစ်သည်ကို သတိပြုပါ။

အစီအစဉ်များကို ကွန်ပိုင်းလုပ်ခြင်း

အစီအစဉ်များ (ပရိုစီဂျာ၏ ကိုယ်ထည်များ သို့မဟုတ် တိုက်ရိုက် ဖော်ပြထားသော begin ဖော်ပြချက်များမှ) ကို ကွန်ပိုင်းလုပ်ခြင်းသည် ၎င်းတို့ကို တန်ဖိုးဖြတ်ခြင်းနှင့် တူညီပါသည်။ အစီအစဉ်၏ ဖော်ပြချက် တစ်ခုချင်းစီကို ကွန်ပိုင်းလုပ်ပါသည်—နောက်ဆုံး ဖော်ပြချက်ကို အစီအစဉ်အတွက် သတ်မှတ်ထားသော ချိတ်ဆက်မှုဖြင့် ကွန်ပိုင်းလုပ်ပြီး၊ ကျန်ဖော်ပြချက်များကိုမူ (အစီအစဉ်၏ ကျန်အပိုင်းများကို ဆက်လက်လုပ်ဆောင်ရန်) next ချိတ်ဆက်မှုဖြင့် ကွန်ပိုင်းလုပ်ပါသည်။ ဖော်ပြချက် တစ်ခုချင်းစီအတွက် ညွှန်ကြားချက် အစီအစဉ်များကို ညွှန်ကြားချက် အစီအစဉ်တစ်ခုတည်း ဖြစ်လာစေရန် ပေါင်းစပ်ပါသည်၊ ထိုသို့ပြုလုပ်ရာတွင် (အစီအစဉ်၏ ကျန်အပိုင်းများအတွက် လိုအပ်သော) env နှင့် (အစီအစဉ်၏ အဆုံးရှိ ချိတ်ဆက်မှုအတွက် လိုအပ်နိုင်သော) continue တို့ကို ထိန်းသိမ်းထားပါသည်။ 

(define (compile-sequence seq target linkage)
  (if (last-exp? seq)
      (compile (first-exp seq) target linkage)
      (preserving '(env continue)
       (compile (first-exp seq) target 'next)
       (compile-sequence (rest-exps seq)
                         target
                         linkage))))
lambda ဖော်ပြချက်များကို ကွန်ပိုင်းလုပ်ခြင်း

Lambda ဖော်ပြချက်များသည် ပရိုစီဂျာများကို တည်ဆောက်ပါသည်။ lambda ဖော်ပြချက်တစ်ခုအတွက် အော့ဗ်ဂျက်ကုတ် (object code) သည် အောက်ပါပုံစံ ရှိရပါမည်

 ၎င်းကို ပစ်မှတ်ရက်ဂျစ်စတာသို့ သတ်မှတ်ပါ⟩
⟨linkage⟩

ကျွန်ုပ်တို့သည် lambda ဖော်ပြချက်ကို ကွန်ပိုင်းလုပ်သောအခါ၊ ပရိုစီဂျာ၏ ကိုယ်ထည် (body) အတွက် ကုတ်ကိုလည်း ထုတ်လုပ်ပါသည်။ ပရိုစီဂျာ တည်ဆောက်ချိန်တွင် ကိုယ်ထည်ကို လုပ်ဆောင်မည် မဟုတ်သော်လည်း၊ ၎င်းကို lambda အတွက် ကုတ်၏နောက်တွင် တိုက်ရိုက် ထည့်သွင်းထားခြင်းက အဆင်ပြေပါသည်။ အကယ်၍ lambda ဖော်ပြချက်အတွက် ချိတ်ဆက်မှုသည် လေဘယ်လ်တစ်ခု သို့မဟုတ် return ဖြစ်ပါက အဆင်ပြေပါသည်။ သို့သော် ချိတ်ဆက်မှုသည် next ဖြစ်ပါက၊ ကိုယ်ထည်၏ နောက်တွင် ထည့်သွင်းထားသော လေဘယ်လ်သို့ ခုန်ကူးသည့် ချိတ်ဆက်မှုကို အသုံးပြု၍ ပရိုစီဂျာ ကိုယ်ထည်အတွက် ကုတ်ကို ကျော်ဖြတ်ရန် လိုအပ်ပါလိမ့်မည်။ ထို့ကြောင့် အော့ဗ်ဂျက်ကုတ်သည် အောက်ပါပုံစံ ရှိလာပါသည်

 ⟨ပေးထားသော ချိတ်ဆက်မှုအတွက် ကုတ်⟩ သို့မဟုတ်

 ⟨ပရိုစီဂျာ ကိုယ်ထည်ကို ကွန်ပိုင်းလုပ်ခြင်း⟩
after-lambda

Compile-lambda သည် ပရိုစီဂျာ အော့ဗ်ဂျက်ကို တည်ဆောက်ရန် ကုတ်နှင့် ၎င်းနောက်တွင် ပရိုစီဂျာ ကိုယ်ထည်အတွက် ကုတ်တို့ကို ထုတ်လုပ်ပေးပါသည်။ ပရိုစီဂျာ အော့ဗ်ဂျက်ကို ပရိုဂရမ် လုပ်ဆောင်နေချိန် (run time) တွင် လက်ရှိ ပတ်ဝန်းကျင် (အဓိပ္ပာယ်ဖွင့်ဆိုသည့် နေရာရှိ ပတ်ဝန်းကျင်) နှင့် ကွန်ပိုင်းလုပ်ထားသော ပရိုစီဂျာ ကိုယ်ထည်၏ အစမှတ် (entry point - အသစ်ထုတ်လုပ်ထားသော လေဘယ်လ်တစ်ခု) တို့ကို ပေါင်းစပ်ခြင်းဖြင့် တည်ဆောက်မည် ဖြစ်သည်။323 

(define (compile-lambda exp target linkage)
  (let ((proc-entry 
         (make-label 'entry))
        (after-lambda 
         (make-label 'after-lambda)))
    (let ((lambda-linkage
           (if (eq? linkage 'next)
               after-lambda
               linkage)))
      (append-instruction-sequences
       (tack-on-instruction-sequence
        (end-with-linkage 
         lambda-linkage
         (make-instruction-sequence 
          '(env)
          (list target)
          `((assign 
             ,target
             (op make-compiled-procedure)
             (label ,proc-entry)
             (reg env)))))
        (compile-lambda-body exp proc-entry))
       after-lambda))))

Compile-lambda သည် ပရိုစီဂျာ ကိုယ်ထည်ကို lambda ဖော်ပြချက် ကုတ်တွင် ပေါင်းစပ်ရန် append-instruction-sequences (5.5.4) ကို အသုံးမပြုဘဲ အထူးပေါင်းစပ်ကိရိယာ tack-on-instruction-sequence ကို အသုံးပြုပါသည်၊ အကြောင်းမှာ ပေါင်းစပ်ထားသော အစီအစဉ်ထဲသို့ စတင်ဝင်ရောက်ချိန်တွင် ကိုယ်ထည်သည် လုပ်ဆောင်မည့် ညွှန်ကြားချက် အစီအစဉ်၏ အစိတ်အပိုင်း မဟုတ်သောကြောင့် ဖြစ်သည်၊ ၎င်းသည် ထိုနေရာတွင် ထားရှိရန် အဆင်ပြေသောကြောင့်သာ ထည့်သွင်းထားခြင်း ဖြစ်သည်။

Compile-lambda-body သည် ပရိုစီဂျာ၏ ကိုယ်ထည်အတွက် ကုတ်ကို တည်ဆောက်ပါသည်။ ဤကုတ်သည် အစမှတ် (entry point) အတွက် လေဘယ်လ်တစ်ခုဖြင့် စတင်ပါသည်။ ထို့နောက် ပရိုဂရမ် လုပ်ဆောင်နေချိန် တန်ဖိုးဖြတ်သည့် ပတ်ဝန်းကျင်ကို ပရိုစီဂျာ ကိုယ်ထည်အား တန်ဖိုးဖြတ်ရန် မှန်ကန်သော ပတ်ဝန်းကျင်သို့ ပြောင်းလဲပေးမည့် ညွှန်ကြားချက်များ လာပါသည်—ဆိုလိုသည်မှာ ပရိုစီဂျာ၏ အဓိပ္ပာယ်ဖွင့်ဆိုချက် ပတ်ဝန်းကျင်ကို ပရိုစီဂျာကို ခေါ်ဆိုသည့် အာဂူးမန့် (arguments) များနှင့် ဖော်မတ် ပါရာမီတာ (formal parameters) များ၏ ပေါင်းစည်းမှုများ ပါဝင်ရန် တိုးချဲ့ထားခြင်း ဖြစ်သည်။ ၎င်းနောက်တွင် ပရိုစီဂျာ ကိုယ်ထည်ကို ဖွဲ့စည်းထားသော ဖော်ပြချက် အစီအစဉ်များအတွက် ကုတ် လာပါသည်။ ထိုအစီအစဉ်ကို ချိတ်ဆက်မှု return နှင့် ပစ်မှတ် val ဖြင့် ကွန်ပိုင်းလုပ်ထားသောကြောင့်၊ ၎င်းသည် ပရိုစီဂျာမှ ရလဒ်ကို val ထဲတွင် ထည့်သွင်းကာ ပြန်လှည့်ခြင်း (returning) ဖြင့် အဆုံးသတ်မည် ဖြစ်သည်။

ကွန်ပိုင်းလုပ်ခြင်း လုပ်ငန်းစဉ်၏ အနှစ်သာရမှာ ပရိုစီဂျာ အသုံးချမှုများ (procedure applications) ကို ကွန်ပိုင်းလုပ်ခြင်း ဖြစ်သည်။ သတ်မှတ်ထားသော target နှင့် linkage တို့ဖြင့် ကွန်ပိုင်းလုပ်ထားသော ပေါင်းစပ်ဖော်ပြချက် (combination) တစ်ခုအတွက် ကုဒ်သည် အောက်ပါပုံစံရှိသည်-

operator နှင့် operands များကို တွက်ချက်နေစဉ်အတွင်း env၊ proc၊ နှင့် argl register များကို သိမ်းဆည်းခြင်း (save) နှင့် ပြန်လည်ဖော်ထုတ်ခြင်း (restore) များ ပြုလုပ်ရန် လိုအပ်နိုင်သည်။ ဤနေရာသည် ကွန်ပိုင်းလာတွင် val မဟုတ်သော အခြား target တစ်ခုကို သတ်မှတ်သည့် တစ်ခုတည်းသောနေရာ ဖြစ်ကြောင်း သတိပြုပါ။

လိုအပ်သော ကုဒ်များကို compile-application က ထုတ်ပေးသည်။ ၎င်းသည် operator ကို recursive နည်းဖြင့် ကွန်ပိုင်းလုပ်ပြီး အသုံးချမည့် ပရိုစီဂျာကို proc ထဲသို့ ထည့်သွင်းပေးမည့် ကုဒ်ကို ထုတ်ပေးသည်၊ ထို့နောက် application ၏ operand တစ်ခုချင်းစီကို တွက်ချက်မည့် ကုဒ်ကို ထုတ်ပေးရန် operand များကို ကွန်ပိုင်းလုပ်သည်။ operand များအတွက် instruction sequence များကို (construct-arglist အသုံးပြု၍) argl ထဲတွင် argument စာရင်းကို တည်ဆောက်မည့် ကုဒ်နှင့် ပေါင်းစပ်ပြီး၊ ရရှိလာသော argument-list ကုဒ်ကို ပရိုစီဂျာကုဒ်နှင့် ပရိုစီဂျာခေါ်ဆိုမှုကို လုပ်ဆောင်မည့် ကုဒ် (compile-procedure-call မှ ထုတ်ပေးသော ကုဒ်) တို့နှင့် ပေါင်းစပ်သည်။ code sequence များကို ဆက်စပ်ရာတွင် operator ကို တွက်ချက်စဉ်အတွင်း env register ကို ထိန်းသိမ်းထားရမည် (အကြောင်းမှာ operator ကို တွက်ချက်ခြင်းသည် operands များကို တွက်ချက်ရန် လိုအပ်မည့် env ကို ပြုပြင်ပြောင်းလဲနိုင်သောကြောင့်ဖြစ်သည်)၊ ထို့ပြင် argument list ကို တည်ဆောက်နေစဉ်အတွင်း proc register ကို ထိန်းသိမ်းထားရမည် (အကြောင်းမှာ operands များကို တွက်ချက်ခြင်းသည် အမှန်တကယ် ပရိုစီဂျာ အသုံးချမှုအတွက် လိုအပ်မည့် proc ကို ပြုပြင်ပြောင်းလဲနိုင်သောကြောင့်ဖြစ်သည်)။ Continue ကိုလည်း တစ်လျှောက်လုံး ထိန်းသိမ်းထားရမည်၊ အကြောင်းမှာ ၎င်းသည် ပရိုစီဂျာခေါ်ဆိုမှုရှိ linkage အတွက် လိုအပ်သောကြောင့် ဖြစ်သည်။ 

(define (compile-application 
         exp target linkage)
  (let ((proc-code 
         (compile (operator exp) 'proc 'next))
        (operand-codes
         (map (lambda (operand)
                (compile operand 'val 'next))
              (operands exp))))
    (preserving 
     '(env continue)
     proc-code
     (preserving 
      '(proc continue)
      (construct-arglist operand-codes)
      (compile-procedure-call 
       target
       linkage)))))

အမှန်တကယ်တွင် အကယ်၍ target သည် val မဟုတ်ပါက၊ ကျွန်ုပ်တို့၏ ကွန်ပိုင်းလာသည် အထက်ပါ ကုဒ်အတိုင်း အတိအကျ ထုတ်ပေးမည် ဖြစ်သည်။324 သို့သော် များသောအားဖြင့် target မှာ val ဖြစ်တတ်သည် (ကွန်ပိုင်းလာသည် operator တွက်ချက်မှု၏ target အဖြစ် proc ကို သတ်မှတ်ချိန်မှလွဲ၍ ကျန်အချိန်များတွင် ဖြစ်သည်)၊ ထို့ကြောင့် ပရိုစီဂျာရလဒ်ကို target register ထဲသို့ တိုက်ရိုက်ထည့်သွင်းပြီးဖြစ်၍ ၎င်းကို ကူးယူရန် အထူးနေရာတစ်ခုသို့ ပြန်လာရန် မလိုအပ်တော့ပါ။ ယင်းအစား ပရိုစီဂျာသည် ခေါ်ဆိုသူ (caller) ၏ linkage မှ သတ်မှတ်ထားသော နေရာသို့ တိုက်ရိုက် “ပြန်သွား” စေရန် continue ကို သတ်မှတ်ပေးခြင်းဖြင့် ကုဒ်ကို ရိုးရှင်းအောင် ပြုလုပ်ပါသည်-

return linkage ၏ ဤအကောင်အထည်ဖော်မှုဖြင့် ကွန်ပိုင်းလာသည် tail-recursive ဖြစ်သော ကုဒ်ကို ထုတ်ပေးသည်။ ပရိုစီဂျာ body တစ်ခု၏ နောက်ဆုံးအဆင့်အဖြစ် ပရိုစီဂျာတစ်ခုကို ခေါ်ဆိုခြင်းသည် stack ပေါ်တွင် မည်သည့်အချက်အလက်မှ သိမ်းဆည်းရန် မလိုဘဲ တိုက်ရိုက် လွှဲပြောင်းမှုကို ပြုလုပ်သည်။

ယင်းအစား အကယ်၍ ကျွန်ုပ်တို့သည် return linkage နှင့် val target ရှိသော ပရိုစီဂျာခေါ်ဆိုမှု အခြေအနေကို အထက်ပါ val မဟုတ်သော target များအတွက် ပြထားသည့်အတိုင်း ကိုင်တွယ်ခဲ့လျှင်၊ ၎င်းသည် tail recursion ကို ပျက်စီးစေပါလိမ့်မည်။ ကျွန်ုပ်တို့၏ စနစ်သည် မည်သည့်ဖော်ပြချက်အတွက်မဆို တူညီသော တန်ဖိုးကို ပေးဦးမည် ဖြစ်သော်လည်း၊ ပရိုစီဂျာတစ်ခု ခေါ်ဆိုတိုင်း continue ကို သိမ်းဆည်းမည်ဖြစ်ပြီး ခေါ်ဆိုမှုအပြီးတွင် ထို (အသုံးမဝင်သော) သိမ်းဆည်းမှုကို ပြန်ဖြုတ်ရန် ပြန်လာပါလိမ့်မည်။ ဤအပို သိမ်းဆည်းမှုများသည် ပရိုစီဂျာခေါ်ဆိုမှုများ ထပ်ဆင့်ဖြစ်ပေါ်နေစဉ်အတွင်း စုပုံလာပါလိမ့်မည်။325

Compile-proc-appl သည် call အတွက် target မှာ val ဖြစ်မဖြစ်နှင့် linkage မှာ return ဖြစ်မဖြစ် အပေါ် မူတည်၍ အခြေအနေ လေးမျိုးကို ထည့်သွင်းစဉ်းစားကာ အထက်ပါ ပရိုစီဂျာ အသုံးချမှုကုဒ်ကို ထုတ်ပေးသည်။ ပရိုစီဂျာ body ကို အကောင်အထည်ဖော်ခြင်းသည် register များကို နည်းလမ်းအမျိုးမျိုးဖြင့် ပြောင်းလဲနိုင်သောကြောင့် instruction sequence များသည် register အားလုံးကို ပြုပြင်မည်ဖြစ်ကြောင်း ကြေညာထားသည်ကို သတိပြုပါ။326 ထို့ပြင် target val နှင့် linkage return ဖြစ်သော အခြေအနေအတွက် code sequence သည် continue ကို လိုအပ်သည်ဟု ကြေညာထားသည်ကို သတိပြုပါ- instruction နှစ်ခုပါ sequence ထဲတွင် continue ကို အတိအလင်း အသုံးမပြုထားသော်လည်း၊ ကွန်ပိုင်းလုပ်ထားသော ပရိုစီဂျာထဲသို့ ဝင်ရောက်ချိန်တွင် continue ၌ မှန်ကန်သော တန်ဖိုးရှိနေရန် သေချာစေရမည် ဖြစ်သည်။ 

(define (compile-proc-appl target linkage)
  (cond ((and (eq? target 'val)
              (not (eq? linkage 'return)))
         (make-instruction-sequence 
          '(proc)
          all-regs
          `((assign continue (label ,linkage))
            (assign 
             val 
             (op compiled-procedure-entry)
             (reg proc))
            (goto (reg val)))))
        ((and (not (eq? target 'val))
              (not (eq? linkage 'return)))
         (let ((proc-return 
                (make-label 'proc-return)))
           (make-instruction-sequence 
            '(proc)
            all-regs
            `((assign continue 
                      (label ,proc-return))
              (assign 
               val 
               (op compiled-procedure-entry)
               (reg proc))
              (goto (reg val))
              ,proc-return
              (assign ,target (reg val))
              (goto (label ,linkage))))))
        ((and (eq? target 'val)
              (eq? linkage 'return))
         (make-instruction-sequence 
          '(proc continue) 
          all-regs
          '((assign 
             val 
             (op compiled-procedure-entry)
             (reg proc))
            (goto (reg val)))))
        ((and (not (eq? target 'val))
              (eq? linkage 'return))
         (error "return linkage, 
                 target not val: COMPILE"
                target))))

5.5.4Instruction Sequence များကို ပေါင်းစပ်ခြင်း

ဤကဏ္ဍတွင် instruction sequence များကို မည်သို့ ကိုယ်စားပြုသနည်းနှင့် မည်သို့ ပေါင်းစပ်သနည်းဟူသော အသေးစိတ်အချက်အလက်များကို ဖော်ပြထားပါသည်။ ၅.၅.၁ မှ instruction sequence တစ်ခုကို လိုအပ်သော register များ၊ ပြောင်းလဲထားသော register များနှင့် အမှန်တကယ် ညွှန်ကြားချက်များ၏ စာရင်း (list) တစ်ခုအဖြစ် ကိုယ်စားပြုကြောင်း ပြန်လည်အမှတ်ရပါ။ ကျွန်ုပ်တို့သည် label (symbol) တစ်ခုကို မည်သည့် register ကိုမျှ လိုအပ်ခြင်း သို့မဟုတ် ပြောင်းလဲခြင်း မရှိသော instruction sequence ၏ အထူးအခြေအနေ (degenerate case) တစ်ခုအဖြစ်လည်း သတ်မှတ်ပါမည်။ ထို့ကြောင့် instruction sequence များမှ လိုအပ်သော နှင့် ပြောင်းလဲလိုက်သော register များကို ဆုံးဖြတ်ရန် အောက်ပါ selector များကို အသုံးပြုပါသည် - 

(define (registers-needed s)
  (if (symbol? s) '() (car s)))
(define (registers-modified s)
  (if (symbol? s) '() (cadr s)))
(define (statements s)
  (if (symbol? s) (list s) (caddr s)))

ပေးထားသော sequence တစ်ခုသည် ပေးထားသော register တစ်ခုကို လိုအပ်ခြင်း ရှိမရှိ သို့မဟုတ် ပြောင်းလဲခြင်း ရှိမရှိကို ဆုံးဖြတ်ရန် အောက်ပါ predicate များကို အသုံးပြုပါသည် - 

(define (needs-register? seq reg)
  (memq reg (registers-needed seq)))
(define (modifies-register? seq reg)
  (memq reg (registers-modified seq)))

အခြေခံ combiner မှာ append-instruction-sequences ဖြစ်သည်။ ၎င်းသည် အစီအစဉ်လိုက် (sequentially) လုပ်ဆောင်ရမည့် instruction sequence အရေအတွက် အကန့်အသတ်မရှိကို argument များအဖြစ် ယူပြီး၊ sequence အားလုံး၏ statement များကို တစ်ဆက်တည်း ပေါင်းစပ်ထားသော statement များပါဝင်သည့် instruction sequence တစ်ခုကို return ပြန်ပေးသည်။ သတိပြုရမည့်အချက်မှာ ရရှိလာသော sequence မှ လိုအပ်သော နှင့် ပြောင်းလဲလိုက်သော register များကို ဆုံးဖြတ်ရန်ဖြစ်သည်။ ၎င်းသည် sequence များထဲမှ တစ်ခုခုမှ ပြောင်းလဲလိုက်သော register များကို ပြောင်းလဲပါသည်။ ၎င်းသည် ပထမဆုံး sequence ကို မလုပ်ဆောင်မီ initialize လုပ်ထားရမည့် register များ (ပထမ sequence မှ လိုအပ်သော register များ) နှင့်အတူ ရှေ့မှ sequence များမှ initialize (ပြောင်းလဲခြင်း) မလုပ်ထားသော အခြား sequence များမှ လိုအပ်သော register များကို လိုအပ်ပါသည်။

Sequence များကို append-2-sequences ဖြင့် တစ်ကြိမ်လျှင် နှစ်ခုစီ ပေါင်းစပ်ပါသည်။ ၎င်းသည် instruction sequence နှစ်ခုဖြစ်သော seq1 နှင့် seq2 ကို ယူပြီး၊ seq1 ၏ statement များနောက်တွင် seq2 ၏ statement များ လိုက်ပါသော instruction sequence ကို return ပြန်ပေးသည်။ ၎င်း၏ ပြောင်းလဲထားသော register များမှာ seq1 သို့မဟုတ် seq2 တစ်ခုခုမှ ပြောင်းလဲလိုက်သော register များဖြစ်ပြီး၊ လိုအပ်သော register များမှာ seq1 မှ လိုအပ်သော register များနှင့် seq1 မှ ပြောင်းလဲခြင်းမရှိသော seq2 မှ လိုအပ်သော register များ ဖြစ်သည်။ (Set လုပ်ဆောင်ချက်များအရ လိုအပ်သော register set အသစ်သည် seq1 မှ လိုအပ်သော register set နှင့် seq2 မှ လိုအပ်သော register များနှင့် seq1 မှ ပြောင်းလဲလိုက်သော register များ၏ set ခြားနားချက် (set difference) တို့၏ union ဖြစ်သည်။) ထို့ကြောင့် append-instruction-sequences ကို အောက်ပါအတိုင်း အကောင်အထည်ဖော်သည် - 

(define (append-instruction-sequences . seqs)
  (define (append-2-sequences seq1 seq2)
    (make-instruction-sequence
     (list-union 
      (registers-needed seq1)
      (list-difference 
       (registers-needed seq2)
       (registers-modified seq1)))
     (list-union
      (registers-modified seq1)
      (registers-modified seq2))
     (append (statements seq1)
             (statements seq2))))
  (define (append-seq-list seqs)
    (if (null? seqs)
        (empty-instruction-sequence)
        (append-2-sequences 
         (car seqs)
         (append-seq-list (cdr seqs)))))
  (append-seq-list seqs))

(define (preserving regs seq1 seq2)
  (if (null? regs)
      (append-instruction-sequences seq1 seq2)
      (let ((first-reg (car regs)))
        (if (and 
             (needs-register? seq2 first-reg)
             (modifies-register? seq1 
                                 first-reg))
            (preserving 
             (cdr regs)
             (make-instruction-sequence
              (list-union 
               (list first-reg)
               (registers-needed seq1))
              (list-difference
               (registers-modified seq1)
               (list first-reg))
              (append `((save ,first-reg))
                      (statements seq1)
                      `((restore ,first-reg))))
             seq2)
            (preserving 
             (cdr regs)
             seq1
             seq2)))))

  (define (iter product counter)
    (if (> counter n)
        product
        (iter (* counter product)
              (+ counter 1))))
  (iter 1 1))

336 Compiled code မှ interpreted procedures များကို ခေါ်ဆိုနိုင်ရန် compiler ကို တိုးချဲ့ခြင်းဖြင့် ပိုမိုကောင်းမွန်အောင် ပြုလုပ်နိုင်ပါသည်။ လေ့ကျင့်ခန်း 5.47 ကို ကြည့်ပါ။

337 execution strategy မည်သို့ပင်ဖြစ်စေ၊ system ကို ရပ်တန့်သွားစေခြင်း သို့မဟုတ် မှားယွင်းသောအဖြေများ ထွက်ပေါ်လာစေခြင်းတို့ထက် user program ကို execute လုပ်ရာတွင် ကြုံတွေ့ရသော error များကို detect လုပ်ရန်နှင့် signal လုပ်ရန် အတင်းအကျပ် လုပ်ဆောင်ပါက ကြီးမားသော overhead ဖြစ်ပေါ်စေပါသည်။ ဥပမာအားဖြင့်၊ out-of-bounds array reference တစ်ခုကို မလုပ်ဆောင်မီ reference ၏ validity ကို စစ်ဆေးခြင်းဖြင့် detect လုပ်နိုင်ပါသည်။ သို့သော် စစ်ဆေးခြင်း၏ overhead မှာ array reference ၎င်းကိုယ်တိုင်ထက် အဆပေါင်းများစွာ ကုန်ကျနိုင်ပြီး၊ ထိုကဲ့သို့သော စစ်ဆေးမှု လိုအပ်ခြင်း ရှိမရှိ ဆုံးဖြတ်ရာတွင် programmer တစ်ဦးသည် speed နှင့် safety ကို ချိန်ဆရပါမည်။ ကောင်းမွန်သော compiler တစ်ခုသည် ထိုကဲ့သို့သော စစ်ဆေးမှုများပါဝင်သော code ကို ထုတ်လုပ်နိုင်ရမည်၊ redundant checks များကို ရှောင်ရှားရမည်ဖြစ်ပြီး၊ programmer များအား compiled code ရှိ error checking အတိုင်းအတာနှင့် အမျိုးအစားကို ထိန်းချုပ်ခွင့် ပေးရပါမည်။

C နှင့် C++ ကဲ့သို့သော အသုံးများသော languages များအတွက် compiler များသည် တတ်နိုင်သမျှ မြန်မြန် run နိုင်ရန်အတွက် running code ထဲတွင် error-checking operations များ လုံးဝနီးပါး ထည့်သွင်းခြင်း မရှိပါ။ ရလဒ်အနေဖြင့် error checking ကို programmer များက တိုက်ရိုက်ပြုလုပ်ရန် တာဝန်ရှိလာပါသည်။ ကံမကောင်းစွာဖြင့်၊ speed မှာ ကန့်သတ်ချက်မဟုတ်သော အရေးကြီးသော applications များတွင်ပင် လူများသည် ဤသို့ပြုလုပ်ရန် ပျက်ကွက်လေ့ရှိကြသည်။ ၎င်းတို့၏ programs များမှာ မြန်ဆန်သော်လည်း အန္တရာယ်များပါသည်။ ဥပမာအားဖြင့်၊ ၁၉၈၈ ခုနှစ်တွင် အင်တာနက်ကို ရပ်တန့်သွားစေခဲ့သော နာမည်ဆိုးဖြင့် ကျော်ကြားသည့် “Worm” သည် finger daemon ရှိ input buffer overflow ဖြစ်မဖြစ်ကို စစ်ဆေးရန် UNIX(tm) operating system ၏ ပျက်ကွက်မှုကို အခွင့်ကောင်းယူခဲ့ခြင်း ဖြစ်ပါသည်။ (Spafford 1989 ကို ကြည့်ပါ။)

338 အမှန်စင်စစ်၊ interpretation သို့မဟုတ် compilation strategy တစ်ခုခုဖြင့် ဖြစ်စေ၊ စက်အသစ်အတွက် storage allocation၊ input နှင့် output နှင့် evaluator နှင့် compiler ဆိုင်ရာ ဆွေးနွေးမှုများတွင် “primitive” အဖြစ် ကျွန်ုပ်တို့ သတ်မှတ်ခဲ့သော အမျိုးမျိုးသော operations များအားလုံးကိုလည်း အကောင်အထည်ဖော်ရပါမည်။ ဤနေရာတွင် အလုပ်ကို လျှော့ချရန် နည်းဗျူဟာတစ်ခုမှာ ထို operations များကို ဖြစ်နိုင်သမျှ Lisp ဖြင့် ရေးသားပြီး စက်အသစ်အတွက် compile လုပ်ရန် ဖြစ်ပါသည်။ နောက်ဆုံးတွင်၊ အရာအားလုံးသည် စက်အသစ်အတွက် လက်ဖြင့် code ရေးသားထားသော kernel သေးသေးလေး (ဥပမာ garbage collection နှင့် အမှန်တကယ် machine primitives များကို apply လုပ်သည့် mechanism ကဲ့သို့သော) အဖြစ်သို့ လျော့နည်းသွားမည် ဖြစ်ပါသည်။

339 ဤနည်းဗျူဟာသည် compiler ၏ မှန်ကန်မှုကို စမ်းသပ်ရာတွင် စိတ်ဝင်စားစရာကောင်းသော စစ်ဆေးမှုများ ဖြစ်ပေါ်စေပါသည်။ ဥပမာအားဖြင့်၊ စက်အသစ်တွင် compiled compiler ကို အသုံးပြု၍ program တစ်ခုကို compile လုပ်ခြင်းသည် မူလ Lisp system ပေါ်တွင် compile လုပ်ခြင်းနှင့် တူညီမှု ရှိမရှိ စစ်ဆေးခြင်းမျိုး ဖြစ်သည်။ ကွာခြားချက်များ၏ အရင်းအမြစ်ကို ရှာဖွေခြင်းသည် ပျော်စရာကောင်းသော်လည်း ရလဒ်များသည် အလွန်သေးငယ်သော အသေးစိတ်အချက်အလက်များအပေါ်တွင် အလွန်အမင်း ထိခိုက်လွယ်သောကြောင့် စိတ်ပျက်စရာ ကောင်းတတ်ပါသည်။
