4.1The Metacircular Evaluator

Lisp အတွက် ကျွန်ုပ်တို့၏ evaluator ကို Lisp ပရိုဂရမ်တစ်ခုအဖြစ် အကောင်အထည်ဖော်မည်ဖြစ်သည်။ ၎င်းဘာသာစကား (Lisp) ကိုယ်တိုင်ဖြင့် ရေးသားထားသော evaluator တစ်ခုကို အသုံးပြု၍ Lisp ပရိုဂရမ်များကို evaluate လုပ်ရန် စဉ်းစားခြင်းသည် သံသရာလည်နေသည်ဟု ထင်ရနိုင်ပါသည်။ သို့သော်လည်း evaluation သည် လုပ်ငန်းစဉ် (process) တစ်ခု ဖြစ်သောကြောင့်၊ လုပ်ငန်းစဉ်များကို ဖော်ပြရန်အတွက် ကျွန်ုပ်တို့ အသုံးပြုသည့် ကိရိယာဖြစ်သော Lisp ကို အသုံးပြု၍ evaluation လုပ်ငန်းစဉ်ကို ဖော်ပြခြင်းသည် သင့်လျော်ပါသည်။207 ၎င်းအကဲဖြတ်တွက်ချက်မည့် ဘာသာစကားအတိုင်း ရေးသားထားသော evaluator ကို metacircular ဟု ခေါ်ဆိုသည်။

Metacircular evaluator သည် အခြေခံအားဖြင့် 3.2 တွင် ဖော်ပြထားသော evaluation ၏ environment model ကို Scheme ဖြင့် ပုံဖော်ထားခြင်းဖြစ်သည်။ အဆိုပါ မော်ဒယ်တွင် အခြေခံ အစိတ်အပိုင်း နှစ်ခုရှိကြောင်း ပြန်လည်အမှတ်ရပါ-

  1. Combination (special form မဟုတ်သော compound expression တစ်ခု) ကို evaluate လုပ်ရန်အတွက်၊ subexpression များကို evaluate လုပ်ပါ၊ ထို့နောက် operator subexpression ၏ တန်ဖိုးကို operand subexpression များ၏ တန်ဖိုးများပေါ်သို့ apply လုပ်ပါ။
  2. Compound procedure တစ်ခုကို arguments အစုအဝေးတစ်ခုသို့ apply လုပ်ရန်အတွက်၊ procedure ၏ body ကို environment အသစ်တစ်ခုတွင် evaluate လုပ်ပါ။ ဤ environment ကို တည်ဆောက်ရန်အတွက်၊ procedure object ၏ environment အစိတ်အပိုင်းကို frame တစ်ခုဖြင့် တိုးချဲ့ပါ၊ ထို frame တွင် procedure ၏ formal parameters များကို procedure apply လုပ်မည့် arguments များနှင့် bind လုပ်ထားသည်။

ဤစည်းမျဉ်းနှစ်ခုသည် evaluation လုပ်ငန်းစဉ်၏ အနှစ်သာရကို ဖော်ပြသည်။ ၎င်းသည် environment များတွင် evaluate လုပ်ရမည့် expression များကို arguments များသို့ apply လုပ်ရမည့် procedure များအဖြစ် လျှော့ချ (reduce) ပြီး၊ ၎င်းမှတစ်ဆင့် environment အသစ်များတွင် evaluate လုပ်ရမည့် expression အသစ်များအဖြစ် ထပ်မံလျှော့ချသွားသည့် အခြေခံသံသရာ (cycle) တစ်ခုဖြစ်သည်။ ဤသို့ဖြင့် environment တွင် တန်ဖိုးရှာဖွေရမည့် symbol များ နှင့် တိုက်ရိုက် apply လုပ်ရမည့် primitive procedure များအထိ ရောက်ရှိသွားမည်ဖြစ်သည် (ပုံ 4.1 ကိုကြည့်ပါ)။208 ဤ evaluation cycle ကို evaluator ထဲရှိ အရေးကြီးသော procedure နှစ်ခုဖြစ်သည့် eval နှင့် apply တို့၏ အပြန်အလှန်လုပ်ဆောင်မှုဖြင့် ဖော်ပြမည်ဖြစ်ပြီး၊ ၎င်းတို့ကို 4.1.1 တွင် ဖော်ပြထားသည် (ပုံ 4.1 ကိုကြည့်ပါ)။

Evaluator ၏ အကောင်အထည်ဖော်မှုသည် evaluate လုပ်ရမည့် expression များ၏ syntax ကို သတ်မှတ်သည့် procedure များအပေါ်တွင် မူတည်လိမ့်မည်။ ကျွန်ုပ်တို့သည် ဘာသာစကား၏ ကိုယ်စားပြုပုံ (representation) နှင့် evaluator အား သီးခြားစီဖြစ်စေရန် data abstraction ကို အသုံးပြုပါမည်။ ဥပမာအားဖြင့်၊ assignment တစ်ခုကို set! symbol ဖြင့် စတင်သော list တစ်ခုအဖြစ် ကိုယ်စားပြုမည်ဟု ပုံသေမသတ်မှတ်ဘဲ၊ assignment ဟုတ်မဟုတ် စစ်ဆေးရန် abstract predicate ဖြစ်သော assignment? ကို အသုံးပြုပြီး၊ assignment တစ်ခု၏ အစိတ်အပိုင်းများကို ရယူရန် abstract selectors ဖြစ်သော assignment-variable နှင့် assignment-value တို့ကို အသုံးပြုပါမည်။ Expression များ၏ အကောင်အထည်ဖော်မှုကို 4.1.2 တွင် အသေးစိတ်ဖော်ပြပါမည်။ 4.1.3 တွင် ဖော်ပြထားသော procedure များနှင့် environment များ၏ ကိုယ်စားပြုမှုကို သတ်မှတ်သည့် operation များလည်း ရှိပါသည်။ ဥပမာအားဖြင့်၊ make-procedure သည် compound procedure များကို တည်ဆောက်ပေးပြီး၊ lookup-variable-value သည် variable များ၏ တန်ဖိုးများကို ရယူပေးကာ၊ apply-primitive-procedure သည် primitive procedure တစ်ခုကို ပေးထားသော arguments list တစ်ခုသို့ apply လုပ်ပေးသည်။

4.1.1Evaluator ၏ အဓိက အစိတ်အပိုင်း

အကဲဖြတ်မှု လုပ်ငန်းစဉ်ကို eval နှင့် apply ဟူသော လုပ်ထုံးလုပ်နည်းနှစ်ခု၏ အပြန်အလှန် အကျိုးပြုမှုအဖြစ် ဖော်ပြနိုင်ပါသည်။

Eval

Eval သည် argument များအဖြစ် expression တစ်ခုနှင့် environment တစ်ခုကို လက်ခံပါသည်။ ၎င်းသည် expression ကို အမျိုးအစားခွဲခြားပြီး အကဲဖြတ်မှုကို လမ်းညွှန်ပေးသည်။ Eval ကို အကဲဖြတ်ရမည့် expression ၏ syntactic type အပေါ် မူတည်၍ ခွဲခြမ်းစိတ်ဖြာသော ပုံစံဖြင့် တည်ဆောက်ထားပါသည်။ လုပ်ထုံးလုပ်နည်းကို ယေဘုယျကျစေရန်အတွက်၊ expression တစ်ခု၏ အမျိုးအစားကို ဆုံးဖြတ်ခြင်းအား စိတ္တဇဆန်ဆန် (abstractly) ဖော်ပြထားပြီး expression အမျိုးအစား အမျိုးမျိုးအတွက် သီးခြားကိုယ်စားပြုမှုတစ်ခုခုအပေါ်တွင်သာ ကန့်သတ်မထားပါ။ expression အမျိုးအစား တစ်ခုစီတွင် ၎င်းကို စစ်ဆေးပေးသော predicate တစ်ခုနှင့် ၎င်း၏ အစိတ်အပိုင်းများကို ရွေးချယ်ရန်အတွက် စိတ္တဇ နည်းလမ်းတစ်ခု ရှိပါသည်။ ဤ စိတ္တဇ ဝါကျတည်ဆောက်ပုံ (abstract syntax) သည် တူညီသော evaluator ကို အသုံးပြုသော်လည်း ဝါကျတည်ဆောက်ပုံဆိုင်ရာ လုပ်ထုံးလုပ်နည်း အစုအဝေး အသစ်တစ်ခုကို အသုံးပြုခြင်းဖြင့် ဘာသာစကား၏ ဝါကျတည်ဆောက်ပုံကို မည်သို့ ပြောင်းလဲနိုင်သည်ကို မြင်သာစေသည်။

အခြေခံ ဖော်ပြချက်များ (Primitive expressions)

  • ကိန်းဂဏန်းများကဲ့သို့ ကိုယ်တိုင် အကဲဖြတ်နိုင်သော expression များအတွက် eval သည် ထို expression ကိုပင် ပြန်ပေးပါသည်။
  • Eval သည် variable များ၏ တန်ဖိုးများကို ရှာဖွေရန် environment အတွင်း၌ ရှာဖွေရမည် ဖြစ်သည်။

အထူးပုံစံများ (Special forms)

  • quote လုပ်ထားသော expression များအတွက် eval သည် quote လုပ်ထားသည့် expression ကို ပြန်ပေးသည်။
  • variable တစ်ခုသို့ တန်ဖိုးသတ်မှတ်ခြင်း (သို့မဟုတ် အဓိပ္ပာယ်သတ်မှတ်ခြင်း) သည် ထို variable နှင့် တွဲဖက်ရမည့် တန်ဖိုးအသစ်ကို တွက်ချက်ရန် eval ကို recursive စနစ်ဖြင့် ပြန်လည်ခေါ်ဆိုရပါမည်။ variable ၏ ချိတ်ဆက်မှု (binding) ကို ပြောင်းလဲရန် (သို့မဟုတ် ဖန်တီးရန်) environment ကို ပြင်ဆင်ရမည် ဖြစ်သည်။
  • if expression တစ်ခုသည် ၎င်း၏ အစိတ်အပိုင်းများအတွက် အထူးစီမံဆောင်ရွက်မှု လိုအပ်ပါသည်။ သို့မှသာ predicate သည် မှန်ကန်ပါက consequent ကို အကဲဖြတ်ပြီး၊ မဟုတ်ပါက alternative ကို အကဲဖြတ်နိုင်မည် ဖြစ်သည်။
  • lambda expression တစ်ခုကို ၎င်းတွင် သတ်မှတ်ထားသော parameter များ၊ body နှင့် အကဲဖြတ်မှုပြုလုပ်နေသည့် environment တို့ကို ပေါင်းစည်းခြင်းဖြင့် အသုံးပြုနိုင်သော လုပ်ထုံးလုပ်နည်း (procedure) တစ်ခုအဖြစ် ပြောင်းလဲပေးရမည်။
  • begin expression တစ်ခုသည် ၎င်းတွင် ပါဝင်သော expression များကို ပေါ်လာသည့် အစီအစဉ်အတိုင်း အကဲဖြတ်ရန် လိုအပ်ပါသည်။
  • ခွဲခြမ်းစိတ်ဖြာမှု ပုံစံ (cond) ကို ထပ်ဆင့် if expression များအဖြစ် ပြောင်းလဲပြီးမှ အကဲဖြတ်သည်။

ပေါင်းစပ်မှုများ (Combinations)

  • လုပ်ထုံးလုပ်နည်း အသုံးချမှု (procedure application) တစ်ခုအတွက် eval သည် ပေါင်းစပ်မှု (combination) ၏ operator အပိုင်းနှင့် operand များကို recursive စနစ်ဖြင့် အကဲဖြတ်ရမည်။ ရရှိလာသော လုပ်ထုံးလုပ်နည်းနှင့် argument များကို apply ထံ ပေးပို့ပြီး၊ ၎င်းက အမှန်တကယ် လုပ်ထုံးလုပ်နည်း အသုံးချခြင်းကို ကိုင်တွယ်ဆောင်ရွက်ပေးသည်။

ဤသည်မှာ eval ၏ အဓိပ္ပာယ်သတ်မှတ်ချက် ဖြစ်သည် - 

(define (eval exp env)
  (cond ((self-evaluating? exp) 
         exp)
        ((variable? exp) 
         (lookup-variable-value exp env))
        ((quoted? exp) 
         (text-of-quotation exp))
        ((assignment? exp) 
         (eval-assignment exp env))
        ((definition? exp) 
         (eval-definition exp env))
        ((if? exp) 
         (eval-if exp env))
        ((lambda? exp)
         (make-procedure 
          (lambda-parameters exp)
          (lambda-body exp)
          env))
        ((begin? exp)
         (eval-sequence 
          (begin-actions exp) 
          env))
        ((cond? exp) 
         (eval (cond->if exp) env))
        ((application? exp)
         (apply (eval (operator exp) env)
                (list-of-values 
                 (operands exp) 
                 env)))
        (else
         (error "Unknown expression 
                 type: EVAL" exp))))

ရှင်းလင်းစေရန်အတွက် eval ကို cond ကို အသုံးပြု၍ ခွဲခြမ်းစိတ်ဖြာသည့် ပုံစံဖြင့် အကောင်အထည်ဖော်ထားသည်။ ယင်း၏ အားနည်းချက်မှာ ကျွန်ုပ်တို့၏ လုပ်ထုံးလုပ်နည်းသည် ခွဲခြားနိုင်သော expression အမျိုးအစား အနည်းငယ်ကိုသာ ကိုင်တွယ်နိုင်ပြီး၊ eval ၏ အဓိပ္ပာယ်သတ်မှတ်ချက်ကို မပြင်ဆင်ဘဲ အမျိုးအစားအသစ်များကို သတ်မှတ်၍မရခြင်း ဖြစ်သည်။ Lisp အကောင်အထည်ဖော်မှု အများစုတွင် expression ၏ အမျိုးအစားအပေါ် မူတည်၍ ခွဲဝေဆောင်ရွက်ခြင်းကို data-directed ပုံစံဖြင့် ပြုလုပ်ကြသည်။ ၎င်းသည် eval ကိုယ်တိုင်ကို ပြင်ဆင်စရာမလိုဘဲ eval ခွဲခြားနိုင်သည့် expression အမျိုးအစားအသစ်များကို အသုံးပြုသူက ထပ်မံထည့်သွင်းနိုင်စေပါသည်။ ( Exercise 4.3 ကို ကြည့်ပါ။)

Apply သည် argument နှစ်ခုကို လက်ခံသည် - တစ်ခုမှာ လုပ်ထုံးလုပ်နည်း (procedure) ဖြစ်ပြီး နောက်တစ်ခုမှာ ထိုလုပ်ထုံးလုပ်နည်းကို အသုံးချမည့် argument များ၏ စာရင်း (list of arguments) ဖြစ်သည်။ Apply သည် လုပ်ထုံးလုပ်နည်းများကို အမျိုးအစား နှစ်မျိုးခွဲခြားသည် - အခြေခံလုပ်ဆောင်ချက်များ (primitives) အတွက် apply-primitive-procedure ကို ခေါ်ဆိုပြီး၊ စုပေါင်းလုပ်ထုံးလုပ်နည်း (compound procedures) များအတွက်မူ ၎င်း၏ body တွင် ပါဝင်သော expression များကို အစဉ်လိုက် အကဲဖြတ်ခြင်းဖြင့် အသုံးချသည်။ စုပေါင်းလုပ်ထုံးလုပ်နည်း၏ body ကို အကဲဖြတ်ရန် ပတ်ဝန်းကျင် (environment) ကို တည်ဆောက်ရာတွင် လုပ်ထုံးလုပ်နည်းတွင် ပါရှိသော အခြေခံပတ်ဝန်းကျင်ကို တိုးချဲ့ပြီး၊ လုပ်ထုံးလုပ်နည်း၏ parameter များကို အသုံးချမည့် argument များနှင့် ချိတ်ဆက်ပေးသော frame တစ်ခုကို ထည့်သွင်းပေးခြင်းဖြင့် ပြုလုပ်သည်။ ဤသည်မှာ apply ၏ အဓိပ္ပာယ်သတ်မှတ်ချက် ဖြစ်သည် - 

(define (apply procedure arguments)
  (cond ((primitive-procedure? procedure)
         (apply-primitive-procedure 
          procedure 
          arguments))
        ((compound-procedure? procedure)
         (eval-sequence
           (procedure-body procedure)
           (extend-environment
             (procedure-parameters 
              procedure)
             arguments
             (procedure-environment 
              procedure))))
        (else
         (error "Unknown procedure 
                 type: APPLY" 
                procedure))))
လုပ်ထုံးလုပ်နည်း Argument များ

eval သည် လုပ်ထုံးလုပ်နည်း အသုံးချမှုတစ်ခုကို လုပ်ဆောင်သောအခါ၊ ၎င်းသည် အသုံးချမည့် argument များ၏ စာရင်းကို ထုတ်လုပ်ရန် list-of-values ကို အသုံးပြုသည်။ List-of-values သည် ပေါင်းစပ်မှု၏ operand များကို argument အဖြစ် လက်ခံသည်။ ၎င်းသည် operand တစ်ခုစီကို အကဲဖြတ်ပြီး သက်ဆိုင်ရာ တန်ဖိုးများ၏ စာရင်းကို ပြန်ပေးသည် - 209 

(define (list-of-values exps env)
  (if (no-operands? exps)
      '()
      (cons (eval (first-operand exps) env)
            (list-of-values 
             (rest-operands exps) 
             env))))
အခြေအနေပြ ဖော်ပြချက်များ (Conditionals)

Eval-if သည် ပေးထားသော environment အတွင်းရှိ if expression ၏ predicate အပိုင်းကို အကဲဖြတ်သည်။ အကယ်၍ ရလဒ်သည် မှန်ကန်ပါက eval-if သည် consequent ကို အကဲဖြတ်ပြီး၊ မဟုတ်ပါက alternative ကို အကဲဖြတ်သည် - 

(define (eval-if exp env)
  (if (true? (eval (if-predicate exp) env))
      (eval (if-consequent exp) env)
      (eval (if-alternative exp) env)))

eval-if တွင် true? ကို အသုံးပြုထားခြင်းသည် အကောင်အထည်ဖော်ခံရသော ဘာသာစကား (implemented language) နှင့် အကောင်အထည်ဖော်သည့် ဘာသာစကား (implementation language) အကြား ဆက်နွှယ်မှု ပြဿနာကို မီးမောင်းထိုးပြနေသည်။ if-predicate ကို အကောင်အထည်ဖော်ခံရသော ဘာသာစကားဖြင့် အကဲဖြတ်သဖြင့် ထိုဘာသာစကားရှိ တန်ဖိုးတစ်ခုကို ရရှိစေသည်။ interpreter predicate true? သည် ထိုတန်ဖိုးကို အကောင်အထည်ဖော်သည့် ဘာသာစကားရှိ if ဖြင့် စစ်ဆေးနိုင်သော တန်ဖိုးတစ်ခုအဖြစ်သို့ ပြောင်းလဲပေးသည် - အမှန်တရား (truth) ကို metacircular နည်းဖြင့် ကိုယ်စားပြုမှုသည် အရင်းခံ Scheme ၏ ကိုယ်စားပြုမှုနှင့် တူညီချင်မှ တူညီလိမ့်မည်။ 210

အစဉ်လိုက် ဖော်ပြချက်များ (Sequences)

Eval-sequence ကို apply က လုပ်ထုံးလုပ်နည်း body အတွင်းရှိ expression များ၏ အစီအစဉ်ကို အကဲဖြတ်ရန်နှင့် eval က begin expression အတွင်းရှိ expression များ၏ အစီအစဉ်ကို အကဲဖြတ်ရန် အသုံးပြုသည်။ ၎င်းသည် argument များအဖြစ် expression အစီအစဉ်တစ်ခုနှင့် environment တစ်ခုကို လက်ခံပြီး၊ expression များကို ၎င်းတို့ ပေါ်လာသည့် အစီအစဉ်အတိုင်း အကဲဖြတ်သည်။ ပြန်ပေးသည့် တန်ဖိုးမှာ နောက်ဆုံး expression ၏ တန်ဖိုးဖြစ်သည်။ 

(define (eval-sequence exps env)
  (cond ((last-exp? exps) 
         (eval (first-exp exps) env))
        (else 
         (eval (first-exp exps) env)
         (eval-sequence (rest-exps exps) 
                        env))))
တန်ဖိုးသတ်မှတ်ခြင်းနှင့် အဓိပ္ပာယ်သတ်မှတ်ခြင်းများ

အောက်ပါ လုပ်ထုံးလုပ်နည်းသည် variable များသို့ တန်ဖိုးသတ်မှတ်ခြင်းကို ကိုင်တွယ်သည်။ ၎င်းသည် သတ်မှတ်မည့် တန်ဖိုးကို ရှာဖွေရန် eval ကို ခေါ်ဆိုပြီး variable နှင့် ရရှိလာသော တန်ဖိုးကို သတ်မှတ်ထားသော environment အတွင်း ထည့်သွင်းရန် set-variable-value! ထံသို့ ပေးပို့သည်။ 

(define (eval-assignment exp env)
  (set-variable-value! 
   (assignment-variable exp)
   (eval (assignment-value exp) env)
   env)
  'ok)

variable များ၏ အဓိပ္ပာယ်သတ်မှတ်ချက်များကိုလည်း အလားတူပင် ကိုင်တွယ်သည်။ 211 

(define (eval-definition exp env)
  (define-variable! 
    (definition-variable exp)
    (eval (definition-value exp) env)
    env)
  'ok)

ဤနေရာတွင် တန်ဖိုးသတ်မှတ်ခြင်း သို့မဟုတ် အဓိပ္ပာယ်သတ်မှတ်ခြင်း၏ တန်ဖိုးအဖြစ် ok ဟူသော သင်္ကေတကို ပြန်ပေးရန် ရွေးချယ်ထားသည်။ 212

Exercise 4.1: metacircular evaluator သည် operand များကို ဘယ်မှညာသို့ အကဲဖြတ်သလား သို့မဟုတ် ညာမှဘယ်သို့ အကဲဖြတ်သလားဆိုသည်ကို ကျွန်ုပ်တို့ မသိနိုင်သည်ကို သတိပြုပါ။ ၎င်း၏ အကဲဖြတ်မှု အစီအစဉ်သည် အရင်းခံ Lisp ထံမှ ဆင်းသက်လာခြင်း ဖြစ်သည် - အကယ်၍ list-of-values ရှိ cons ၏ argument များကို ဘယ်မှညာသို့ အကဲဖြတ်ပါက၊ list-of-values သည် operand များကို ဘယ်မှညာသို့ အကဲဖြတ်မည် ဖြစ်သည်။ အကယ်၍ cons ၏ argument များကို ညာမှဘယ်သို့ အကဲဖြတ်ပါက list-of-values သည် operand များကို ညာမှဘယ်သို့ အကဲဖြတ်မည် ဖြစ်သည်။

အရင်းခံ Lisp ၏ အကဲဖြတ်မှု အစီအစဉ် မည်သို့ပင်ရှိစေကာမူ operand များကို ဘယ်မှညာသို့ အကဲဖြတ်သော list-of-values ပုံစံတစ်ခုကို ရေးသားပါ။ ထို့အပြင် operand များကို ညာမှဘယ်သို့ အကဲဖြတ်သော list-of-values ပုံစံတစ်ခုကိုလည်း ရေးသားပါ။

4.1.2Representing Expressions

ဤ evaluator သည် 2.3.2 တွင် ဆွေးနွေးခဲ့သော symbolic differentiation ပရိုဂရမ်နှင့် ဆင်တူသည်။ ပရိုဂရမ်နှစ်ခုလုံးသည် symbolic expression များအပေါ်တွင် အလုပ်လုပ်ကြသည်။ ပရိုဂရမ်နှစ်ခုလုံးတွင် compound expression တစ်ခုအပေါ် လုပ်ဆောင်ချက်၏ ရလဒ်ကို expression ၏ အစိတ်အပိုင်းများအပေါ်တွင် ထပ်တလဲလဲ (recursively) လုပ်ဆောင်ခြင်းနှင့် expression အမျိုးအစားအပေါ် မူတည်၍ ရလဒ်များကို ပေါင်းစပ်ခြင်းတို့ဖြင့် ဆုံးဖြတ်သည်။ ပရိုဂရမ်နှစ်ခုလုံးတွင် expression များကို မည်သို့ကိုယ်စားပြုသည်ဆိုသည့် အသေးစိတ်အချက်အလက်များမှ အထွေထွေလုပ်ဆောင်မှုစည်းမျဉ်းများကို ခွဲထုတ်ရန် data abstraction ကို ကျွန်ုပ်တို့ အသုံးပြုခဲ့သည်။ differentiation ပရိုဂရမ်တွင် ၎င်းမှာ တူညီသော differentiation လုပ်ထုံးလုပ်နည်း (procedure) သည် ရှေ့ဆက်ပုံစံ (prefix form)၊ အလယ်ဆက်ပုံစံ (infix form) သို့မဟုတ် အခြားပုံစံများဖြင့် ရှိနေသော အက္ခရာသင်္ချာဖော်ပြချက် (algebraic expressions) များကို ကိုင်တွယ်နိုင်ခြင်းကို ဆိုလိုသည်။ evaluator အတွက်မူ ၎င်းမှာ အကဲဖြတ်ခြင်းခံရသော ဘာသာစကား၏ syntax ကို expression များ၏ အစိတ်အပိုင်းများကို အမျိုးအစားခွဲခြားခြင်းနှင့် ထုတ်ယူခြင်းပြုလုပ်သော procedure များကသာ သီးသန့်ဆုံးဖြတ်ပေးသည်ဟု ဆိုလိုခြင်း ဖြစ်သည်။

ဤသည်မှာ ကျွန်ုပ်တို့ဘာသာစကား၏ syntax သတ်မှတ်ချက်များ ဖြစ်သည်-

  • နံပါတ်များနှင့် စာသားများ (strings) သည်သာ ကိုယ်တိုင်အကဲဖြတ်နိုင်သော (self-evaluating) အရာများ ဖြစ်ကြသည်-

    (define (define-variable! var val env)
  (let ((frame (first-frame env)))
    (define (scan vars vals)
      (cond ((null? vars)
             (add-binding-to-frame! 
              var val frame))
            ((eq? var (car vars))
             (set-car! vals val))
            (else (scan (cdr vars) 
                        (cdr vals)))))
    (scan (frame-variables frame)
          (frame-values frame))))

    4.1.4Evaluator ကို ပရိုဂရမ်တစ်ခုအနေဖြင့် လည်ပတ်ခြင်း

    Evaluator ကို ရရှိပြီးနောက် ကျွန်ုပ်တို့၏လက်ထဲတွင် Lisp ဖော်ပြချက် (expression) များကို အကဲဖြတ်သည့် လုပ်ငန်းစဉ်၏ ဖော်ပြချက် (Lisp ဖြင့် ရေးသားထားသော) တစ်ခု ရှိနေပြီဖြစ်သည်။ Evaluator ကို ပရိုဂရမ်တစ်ခုအနေဖြင့် ဖော်ပြခြင်း၏ အားသာချက်တစ်ခုမှာ ကျွန်ုပ်တို့သည် ထိုပရိုဂရမ်ကို လည်ပတ်နိုင်ခြင်း ဖြစ်သည်။ ဤသည်မှာ Lisp ကိုယ်တိုင်က ဖော်ပြချက်များကို မည်သို့အကဲဖြတ်သည်ဆိုသော အလုပ်လုပ်ပုံစံ (working model) တစ်ခုကို Lisp အတွင်း၌ပင် ရရှိစေသည်။ ဤအခန်း၏ နောက်ပိုင်းတွင် ကျွန်ုပ်တို့ ပြုလုပ်ကြမည့်အတိုင်း ဤအရာသည် အကဲဖြတ်ခြင်းဆိုင်ရာ စည်းမျဉ်း (evaluation rules) များကို စမ်းသပ်ရန်အတွက် အခြေခံဘောင် (framework) တစ်ခုအဖြစ် အသုံးဝင်နိုင်သည်။

    4.1.5ဒေတာကို ပရိုဂရမ်အဖြစ် အသုံးပြုခြင်း

    Lisp အိတ်စပရက်ရှင်း (expressions) တွေကို အကဲဖြတ်ပေးတဲ့ Lisp ပရိုဂရမ်တစ်ခုအကြောင်း စဉ်းစားတဲ့အခါမှာ နှိုင်းယှဉ်ချက်တစ်ခုက အထောက်အကူ ဖြစ်စေနိုင်ပါတယ်။ ပရိုဂရမ်တစ်ခုရဲ့ အဓိပ္ပာယ်ကို လုပ်ဆောင်ချက်ဆိုင်ရာ ရှုထောင့်ကနေ ကြည့်မယ်ဆိုရင် ပရိုဂရမ်ဆိုတာ အက်ဘ်စထရက် စက် (abstract machine) တစ်ခု (ဖြစ်နိုင်ရင် အကန့်အသတ်မရှိ ကြီးမားတဲ့စက်) ရဲ့ ဖော်ပြချက်တစ်ခု ဖြစ်ပါတယ်။ ဥပမာအားဖြင့်၊ ဖက်တိုးရီးယယ်လ် (factorials) တွေကို တွက်ချက်ဖို့ ကျွမ်းကျင်ပြီးသား ပရိုဂရမ်ကို ကြည့်ရအောင်။ 

    (define (factorial n)
  (if (= n 1)
      1
      (* (factorial (- n 1)) n)))

    ဒီပရိုဂရမ်ကို တန်ဖိုးလျှော့ချတဲ့အပိုင်း၊ မြှောက်တဲ့အပိုင်းနဲ့ ညီမညီစစ်ဆေးတဲ့အပိုင်းတွေအပြင်၊ အနေအထားနှစ်ခုပါတဲ့ ခလုတ်တစ်ခုနဲ့ နောက်ထပ် ဖက်တိုးရီးယယ်လ်စက် (factorial machine) တစ်ခုတို့ ပါဝင်တဲ့ စက်တစ်ခုရဲ့ ဖော်ပြချက်အဖြစ် ကျနော်တို့ ယူဆနိုင်ပါတယ်။ (ဖက်တိုးရီးယယ်လ်စက်ဟာ သူ့ထဲမှာ နောက်ထပ် ဖက်တိုးရီးယယ်လ်စက်တစ်ခု ထပ်ပါနေတဲ့အတွက် အကန့်အသတ်မရှိ ဖြစ်နေပါတယ်။) Figure 4.2 ဟာ အစိတ်အပိုင်းတွေကို ဘယ်လို ချိတ်ဆက်ထားသလဲဆိုတာကို ပြသထားတဲ့ ဖက်တိုးရီးယယ်လ်စက်ရဲ့ စီးဆင်းမှုပြပုံ (flow diagram) ဖြစ်ပါတယ်။

    အလားတူပဲ၊ အကဲဖြတ်စက် (evaluator) ကို စက်တစ်ခုရဲ့ ဖော်ပြချက်ကို input အနေနဲ့ လက်ခံယူတဲ့ အထူးစက်တစ်ခုအဖြစ် ကျနော်တို့ ယူဆနိုင်ပါတယ်။ ဒီ input ကို ရတဲ့အခါမှာ evaluator ဟာ ဖော်ပြထားတဲ့စက်ကို အတုယူလုပ်ဆောင် (emulate) ဖို့အတွက် သူ့ကိုယ်သူ ပြင်ဆင်ဖွဲ့စည်းပါတယ်။ ဥပမာအားဖြင့်၊ Figure 4.3 မှာ ပြထားတဲ့အတိုင်း ကျနော်တို့ရဲ့ evaluator ထဲကို factorial ရဲ့ အဓိပ္ပာယ်သတ်မှတ်ချက်ကို ထည့်ပေးလိုက်ရင်၊ evaluator ဟာ ဖက်တိုးရီးယယ်လ်တွေကို တွက်ချက်နိုင်မှာ ဖြစ်ပါတယ်။

    ဒီရှုထောင့်ကနေ ကြည့်မယ်ဆိုရင် ကျနော်တို့ရဲ့ evaluator ဟာ ယူနီဗာဆယ် စက် (universal machine) တစ်ခု ဖြစ်နေတာကို တွေ့ရပါမယ်။ တခြားစက်တွေကို Lisp ပရိုဂရမ်တွေအဖြစ် ဖော်ပြထားတဲ့အခါ ဒီ evaluator က အဲဒီစက်တွေရဲ့ လုပ်ဆောင်ချက်ကို လိုက်တုပါတယ်။223 ဒါဟာ အလွန်စိတ်ဝင်စားဖို့ကောင်းပါတယ်။ လျှပ်စစ်ပတ်လမ်းတွေအတွက် အလားတူ evaluator တစ်ခုကို စိတ်ကူးကြည့်ပါ။ ဒါဟာ ဖစ်လ်တာ (filter) တစ်ခုလိုမျိုး တခြားပတ်လမ်းတစ်ခုခုအတွက် ပုံစံထုတ်ချက်တွေကို ကုဒ်သွင်းထားတဲ့ signal ကို input အဖြစ် ယူတဲ့ ပတ်လမ်းတစ်ခု ဖြစ်ပါလိမ့်မယ်။ ဒီ input ကို ရပြီဆိုတာနဲ့ ပတ်လမ်း evaluator ဟာ ဖော်ပြထားတဲ့ ဖစ်လ်တာတစ်ခုလို ပြုမူဆောင်ရွက်မှာ ဖြစ်ပါတယ်။ အဲဒီလို ယူနီဗာဆယ် လျှပ်စစ်ပတ်လမ်းမျိုးဟာ စိတ်ကူးကြည့်လို့တောင် မရနိုင်လောက်အောင် ရှုပ်ထွေးပါလိမ့်မယ်။ ပရိုဂရမ် evaluator ဟာ တော်တော်လေး ရိုးရှင်းတဲ့ ပရိုဂရမ်တစ်ခု ဖြစ်နေတာကတော့ မှတ်သားစရာပါပဲ။224

    evaluator ရဲ့ နောက်ထပ် စိတ်ဝင်စားစရာကောင်းတဲ့ အချက်တစ်ခုကတော့ ကျနော်တို့ရဲ့ programming language က ကိုင်တွယ်အသုံးပြုတဲ့ data object တွေနဲ့ programming language ကိုယ်တိုင်ကြားမှာ တံတားတစ်ခုလို ဆောင်ရွက်ပေးတာပါပဲ။ (Lisp နဲ့ ရေးသားထားတဲ့) evaluator ပရိုဂရမ်ဟာ run နေပြီး၊ အသုံးပြုသူက evaluator ထံ အိတ်စပရက်ရှင်း (expressions) တွေကို ရိုက်ထည့်ပြီး ရလဒ်တွေကို စောင့်ကြည့်နေတယ်လို့ စိတ်ကူးကြည့်ပါ။ အသုံးပြုသူရဲ့ ရှုထောင့်ကနေကြည့်ရင် (* x x) လိုမျိုး input အိတ်စပရက်ရှင်းဟာ evaluator က execute လုပ်ပေးရမယ့် programming language ထဲက အိတ်စပရက်ရှင်းတစ်ခု ဖြစ်ပါတယ်။ ဒါပေမဲ့ evaluator ရဲ့ ရှုထောင့်ကနေ ကြည့်ရင်တော့ ဒီအိတ်စပရက်ရှင်းဟာ သတ်မှတ်ထားတဲ့ စည်းမျဉ်းတွေအတိုင်း ကိုင်တွယ်ရမယ့် list တစ်ခု (ဒီနေရာမှာဆိုရင် *၊ x နဲ့ x ဆိုတဲ့ သင်္ကေတ သုံးခုပါတဲ့ list တစ်ခု) သာ ဖြစ်ပါတယ်။

    အသုံးပြုသူရဲ့ ပရိုဂရမ်တွေဟာ evaluator အတွက် data ဖြစ်နေတယ်ဆိုတာ ဇဝေဇဝါ ဖြစ်စရာ မလိုပါဘူး။ တကယ်တော့ တစ်ခါတလေမှာ ဒီကွာခြားချက်ကို လျစ်လျူရှုပြီး eval ကို ပရိုဂရမ်တွေမှာ အသုံးပြုခွင့်ပေးခြင်းအားဖြင့် data object တစ်ခုကို Lisp အိတ်စပရက်ရှင်းအနေနဲ့ တိုက်ရိုက် evaluate လုပ်နိုင်တဲ့ စွမ်းဆောင်ရည်ကို အသုံးပြုသူကို ပေးထားတာက ပိုအဆင်ပြေပါတယ်။ Lisp အမျိုးအစား (dialects) အတော်များများမှာ အိတ်စပရက်ရှင်းတစ်ခုနဲ့ ပတ်ဝန်းကျင် (environment) တစ်ခုကို argument တွေအနေနဲ့ ယူပြီး၊ အဲဒီ environment နဲ့ သက်ဆိုင်တဲ့ အိတ်စပရက်ရှင်းကို evaluate လုပ်ပေးတဲ့ အခြေခံ (primitive) eval ပရိုစီဂျာ ပါဝင်ပါတယ်။225 ဒါကြောင့်၊ 

    (eval '(* 5 5) user-initial-environment)

    (define (run-forever)
  (run-forever))

(define (try p)
  (if (halts? p p)
      (run-forever)
      'halted))

    4.1.6Internal Definitions

    ကျွန်ုပ်တို့၏ evaluation ဆိုင်ရာ environment model နှင့် metacircular evaluator တို့သည် definition များကို တစ်ခုပြီးတစ်ခု အစဉ်လိုက်လုပ်ဆောင်ပြီး၊ environment frame ကို definition တစ်ခုချင်းစီဖြင့် တိုးချဲ့သွားပါသည်။ ၎င်းသည် interactive program development ပြုလုပ်ရန်အတွက် အထူးပင် အဆင်ပြေစေပြီး၊ programmer အနေဖြင့် procedure အသစ်များကို သတ်မှတ်ခြင်းနှင့် procedure များကို အသုံးချခြင်းတို့ကို လွတ်လပ်စွာ ရောနှောလုပ်ဆောင်နိုင်ပါသည်။ သို့သော်လည်း၊ block structure ကို အကောင်အထည်ဖော်ရန် အသုံးပြုသော internal definition များ (အပိုင်း 1.1.8 တွင် မိတ်ဆက်ခဲ့သည်) အကြောင်းကို သေချာစွာ စဉ်းစားကြည့်ပါက၊ နာမည်တစ်ခုချင်းစီအလိုက် environment ကို တိုးချဲ့ခြင်းသည် local variable များကို သတ်မှတ်ရန်အတွက် အကောင်းဆုံးနည်းလမ်း မဟုတ်နိုင်ကြောင်း တွေ့ရှိရပါလိမ့်မည်။

    (define (analyze-application exp)
  (let ((fproc (analyze (operator exp)))
        (aprocs (map analyze (operands exp))))
    (lambda (env)
      (execute-application 
       (fproc env)
       (map (lambda (aproc) (aproc env))
            aprocs)))))

(define (execute-application proc args)
  (cond ((primitive-procedure? proc)
         (apply-primitive-procedure proc args))
        ((compound-procedure? proc)
         ((procedure-body proc)
          (extend-environment 
           (procedure-parameters proc)
           args
           (procedure-environment proc))))
        (else (error "Unknown procedure type: 
                      EXECUTE-APPLICATION"
                     proc))))

Turing (1912-1954) က ပထမဦးဆုံး ရှင်းလင်းစွာ သက်ေပြခဲ့သည်။ ထိုစာတမ်းတွင် Turing သည် ယခုအခါ Turing machine ဟု သိကြသော ရိုးရှင်းသည့် computational model တစ်ခုကို တင်ပြခဲ့ပြီး မည်သည့် “ထိရောက်သော လုပ်ငန်းစဉ်” (effective process) ကိုမဆို ထိုကဲ့သို့သော စက်အတွက် program တစ်ခုအဖြစ် ဖော်မြူလာထုတ်နိုင်ကြောင်း ငြင်းဆိုခဲ့သည်။ (ဤအဆိုကို Church-Turing thesis ဟု သိကြသည်။) ထို့နောက် Turing သည် Turing-machine program များအတွက် evaluator အဖြစ် လုပ်ဆောင်သော universal machine တစ်ခုကို အကောင်အထည်ဖော်ခဲ့သည်။ သူသည် ဤ framework ကို အသုံးပြု၍ Turing machines များဖြင့် မတွက်ချက်နိုင်သော ပြဿနာများ ရှိနေကြောင်း သက်သေပြခဲ့သည် (Exercise 4.15 ကို ကြည့်ပါ)၊ ထို့ကြောင့် ထိုပြဿနာများကို “ထိရောက်သော လုပ်ငန်းစဉ်များ” အဖြစ် ဖော်မြူလာ မထုတ်နိုင်ကြောင်း သွယ်ဝိုက်၍ သိနိုင်သည်။ Turing သည် လက်တွေ့ကွန်ပျူတာသိပ္ပံတွင်လည်း အခြေခံကျသော ပံ့ပိုးမှုများကို ဆက်လက်လုပ်ဆောင်ခဲ့သည်။ ဥပမာအားဖြင့်၊ သူသည် ယေဘုယျသုံး subroutines များကို အသုံးပြု၍ program များကို တည်ဆောက်ခြင်း (structuring) အယူအဆကို တီထွင်ခဲ့သည်။ Turing ၏ အတ္ထုပ္ပတ္တိအတွက် Hodges 1983 ကို ကြည့်ပါ။

    224 နှိုင်းယှဉ်ချက်အရ ရိုးရှင်းသော procedure ဖြင့် အကောင်အထည်ဖော်ထားသော evaluator တစ်ခုသည် evaluator ထက် ပိုမိုရှုပ်ထွေးသော program များကို တုပနိုင်သည်ဟူသောအချက်ကို အချို့သူများက နားလည်ရခက်သည် (counterintuitive) ဟု ထင်ကြသည်။ universal evaluator machine တစ်ခု တည်ရှိခြင်းသည် တွက်ချက်မှုဆိုင်ရာ နက်ရှိုင်းပြီး အံ့သြဖွယ်ကောင်းသော ဂုဏ်သတ္တိတစ်ခုဖြစ်သည်။ Recursion theory, သင်္ချာဆိုင်ရာ logic ၏ အကိုင်းအခက်တစ်ခုဖြစ်သော ၎င်းသည် တွက်ချက်မှု၏ logical အကန့်အသတ်များနှင့် သက်ဆိုင်သည်။ Douglas Hofstadter ၏ လှပသောစာအုပ်ဖြစ်သည့် Gödel, Escher, Bach တွင် ဤအယူအဆအချို့ကို လေ့လာထားသည် (Hofstadter 1979)။

    225 သတိပေးချက်- ဤ eval primitive သည် ကျွန်ုပ်တို့ 4.1.1 တွင် အကောင်အထည်ဖော်ခဲ့သော eval procedure နှင့် တူညီမှုမရှိပါ၊ အကြောင်းမှာ ၎င်းသည် 4.1.3 တွင် ကျွန်ုပ်တို့ တည်ဆောက်ခဲ့သော နမူနာ environment structure များအစား အမှန်တကယ် (actual) Scheme environment များကို အသုံးပြုသောကြောင့်ဖြစ်သည်။ ဤအမှန်တကယ် environment များကို အသုံးပြုသူမှ သာမန် list များကဲ့သို့ ကိုင်တွယ်၍မရပါ၊ ၎င်းတို့ကို eval သို့မဟုတ် အခြားအထူးလုပ်ဆောင်ချက်များမှတစ်ဆင့်သာ ရယူအသုံးပြုရမည်ဖြစ်သည်။ ထို့အတူပင် ကျွန်ုပ်တို့ ယခင်က တွေ့ခဲ့သော apply primitive သည် metacircular apply နှင့် တူညီမှုမရှိပါ၊ အကြောင်းမှာ ၎င်းသည် 4.1.3 နှင့် 4.1.4 တို့တွင် ကျွန်ုပ်တို့ တည်ဆောက်ခဲ့သော procedure object များအစား အမှန်တကယ် Scheme procedures များကို အသုံးပြုသောကြောင့်ဖြစ်သည်။

    227 ကျွန်ုပ်တို့သည် halts? အား procedure object တစ်ခု ပေးထားသည်ဟု သတ်မှတ်ခဲ့သော်လည်း၊ halts? သည် procedure ၏ text နှင့် ၎င်း၏ environment ကို ရယူနိုင်လျှင်ပင် ဤအကျိုးအကြောင်းပြချက်မှာ သက်ရောက်မှုရှိနေဆဲဖြစ်သည်ကို သတိပြုပါ။ ၎င်းသည် Turing ၏ ကျော်ကြားသော Halting Theorem ဖြစ်ပြီး၊ တွက်ချက်၍မရသော ( non-computable) ပြဿနာ၏ ပထမဦးဆုံး ရှင်းလင်းသော ဥပမာကို ပေးခဲ့သည်၊ ဆိုလိုသည်မှာ တွက်ချက်မှုဆိုင်ရာ procedure တစ်ခုအဖြစ် မလုပ်ဆောင်နိုင်သော ကောင်းမွန်စွာ သတ်မှတ်ထားသည့် အလုပ်တစ်ခုဖြစ်သည်။

    230 Scheme ၏ MIT အကောင်အထည်ဖော်သူများသည် အောက်ပါအချက်များပေါ်မူတည်၍ Alyssa ကို ထောက်ခံကြသည်- Eva သည် မူအားဖြင့် မှန်ကန်သည်—သတ်မှတ်ချက် (definitions) များကို တစ်ပြိုင်နက်တည်း (simultaneous) ဟု ယူဆသင့်သည်။ သို့သော် Eva လိုအပ်သည့်အရာကို လုပ်ဆောင်ပေးနိုင်သော ယေဘုယျကျပြီး စွမ်းဆောင်ရည်ကောင်းမွန်သည့် mechanism တစ်ခုကို အကောင်အထည်ဖော်ရန် ခက်ခဲပုံရသည်။ ထိုသို့သော mechanism မရှိသည့်အခါတွင်၊ simultaneous definitions များ၏ ခက်ခဲသော အခြေအနေများတွင် (Alyssa ၏ အယူအဆ) မှားယွင်းသော အဖြေကို ထုတ်ပေးမည့်အစား (Ben ဖြစ်စေချင်သကဲ့သို့) error တစ်ခု ထုတ်ပေးခြင်းက ပို၍ကောင်းမွန်ပါသည်။

    231 ဤဥပမာသည် define ကို အသုံးမပြုဘဲ recursive procedure များကို ဖော်မြူလာထုတ်ရန် programming trick တစ်ခုကို သရုပ်ဖော်ထားသည်။ ဤကဲ့သို့သော အထွေထွေအကျဆုံး trick မှာ Y operator ဖြစ်ပြီး၊ ၎င်းကို recursion ၏ “pure λ-calculus” အကောင်အထည်ဖော်မှုတစ်ခု ပြုလုပ်ရန် အသုံးပြုနိုင်သည်။ (λ-calculus ဆိုင်ရာ အသေးစိတ်အတွက် Stoy 1977 ကိုလည်းကောင်း၊ Scheme ရှိ Y operator အကြောင်း ရှင်းလင်းချက်အတွက် Gabriel 1988 ကိုလည်းကောင်း ကြည့်ပါ။)

    232 ဤနည်းပညာသည် Chapter 5 တွင် ကျွန်ုပ်တို့ ဆွေးနွေးမည့် compilation process ၏ အရေးကြီးသော အစိတ်အပိုင်းတစ်ခုဖြစ်သည်။ Jonathan Rees သည် ၁၉၈၂ ခုနှစ်ဝန်းကျင်က T project အတွက် ဤကဲ့သို့သော Scheme interpreter တစ်ခုကို ရေးသားခဲ့သည် (Rees and Adams 1982)။ Marc Feeley (1986) ( Feeley and Lapalme 1987 ကိုလည်း ကြည့်ပါ) သည် သူ၏ master ဘွဲ့စာတမ်းတွင် ဤနည်းပညာကို သီးခြားလွတ်လပ်စွာ တီထွင်ခဲ့သည်။

    233 သို့သော်ငြားလည်း၊ syntactic analysis ၏ အစိတ်အပိုင်းတစ်ခုအဖြစ် လုပ်ဆောင်နိုင်သော variable ရှာဖွေခြင်း၏ အရေးကြီးသော အစိတ်အပိုင်းတစ်ခု ရှိပါသည်။ 5.5.6 တွင် ကျွန်ုပ်တို့ ပြသမည်ဖြစ်သည့်အတိုင်း၊ variable ၏ တန်ဖိုးကို တွေ့ရှိနိုင်မည့် environment structure အတွင်းရှိ တည်နေရာကို ဆုံးဖြတ်နိုင်ပြီး၊ ထို့ကြောင့် variable နှင့် ကိုက်ညီသော entry အတွက် environment ကို scan ဖတ်ရန် လိုအပ်မှုကို ဖယ်ရှားပေးနိုင်သည်။
