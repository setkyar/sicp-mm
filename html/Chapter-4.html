4Metalinguistic Abstraction

… မန္တန်တွေဟာ စကားလုံးတွေထဲမှာ ရှိတာပါ—အာဗြကဒါဗြ (Abracadabra)၊ နှမ်းစေ့လေး ပွင့်စေ (Open Sesame) နဲ့ အခြားအရာတွေပေါ့—ဒါပေမဲ့ ပုံပြင်တစ်ပုဒ်ထဲက မန္တန်စကားလုံးတွေဟာ နောက်ပုံပြင်တစ်ပုဒ်မှာတော့ အစွမ်းမရှိတော့ဘူး။ တကယ့်မှော်အတတ်ဆိုတာက ဘယ်စကားလုံးတွေက ဘယ်အချိန်မှာ၊ ဘာအတွက် အလုပ်ဖြစ်လဲဆိုတာကို နားလည်ဖို့ပါပဲ။ လှည့်ကွက်ကို သိအောင်လုပ်ဖို့က အဓိက လှည့်ကွက်ပါပဲ။

… ပြီးတော့ အဲဒီစကားလုံးတွေကို ကျွန်ုပ်တို့ အက္ခရာတွေရဲ့ စာလုံးတွေနဲ့ တည်ဆောက်ထားတာပါ—ကလောင်တံနဲ့ ရေးဆွဲနိုင်တဲ့ တွန့်လိမ်နေတဲ့ စာလုံးအနည်းငယ်ပေါ့။ ဒါဟာ သော့ချက်ပါပဲ! ကျွန်ုပ်တို့သာ ရနိုင်မယ်ဆိုရင် ဒါဟာ ရတနာလည်း ဖြစ်ပါတယ်။ ဒါဟာ ရတနာဆီကို သွားဖို့ သော့ချက်ကပဲ ရတနာဖြစ်နေသလိုပါပဲ!

—John Barth, Chimera

ပရိုဂရမ် ဒီဇိုင်း လေ့လာမှုမှာ ကျွမ်းကျင်တဲ့ ပရိုဂရမ်မာတွေဟာ ရှုပ်ထွေးတဲ့ စနစ်အားလုံးရဲ့ ဒီဇိုင်းပညာရှင်တွေ အသုံးပြုတဲ့ ယေဘုယျ နည်းလမ်းတွေနဲ့ပဲ သူတို့ရဲ့ ဒီဇိုင်းတွေမှာရှိတဲ့ ရှုပ်ထွေးမှုတွေကို ထိန်းချုပ်ကြတာကို တွေ့ခဲ့ရပါတယ်။ သူတို့ဟာ အခြေခံ ဒြပ်စင် (primitive elements) တွေကို ပေါင်းစပ်ပြီး ပေါင်းစပ်အရာဝတ္ထု (compound objects) တွေ ပြုလုပ်ကြတယ်၊ ပေါင်းစပ်အရာဝတ္ထုတွေကို အဆင့်မြင့် တည်ဆောက်ပုံ အစိတ်အပိုင်းတွေအဖြစ် စိတ္တဇဖော်ထုတ် (abstract) ကြတယ်၊ ပြီးတော့ စနစ် တည်ဆောက်ပုံရဲ့ သင့်တော်တဲ့ အကြီးစား မြင်ကွင်းတွေကို အသုံးပြုပြီး အစိတ်အပိုင်းများအဖြစ် ဖွဲ့စည်းနိုင်မှု (modularity) ကို ထိန်းသိမ်းကြပါတယ်။ ဒီနည်းလမ်းတွေကို သရုပ်ဖော်ဖို့အတွက် ကျွန်ုပ်တို့ဟာ လုပ်ငန်းစဉ် (processes) တွေကို ဖော်ပြဖို့နဲ့၊ လက်တွေ့ကမ္ဘာက ရှုပ်ထွေးတဲ့ ဖြစ်စဉ်တွေကို ပုံစံတူ ဖန်တီးဖို့အတွက် တွက်ချက်မှုဆိုင်ရာ ဒေတာ အရာဝတ္ထုတွေနဲ့ လုပ်ငန်းစဉ်တွေကို တည်ဆောက်ရာမှာ Lisp ကို ဘာသာစကားတစ်ခုအနေနဲ့ အသုံးပြုခဲ့ပါတယ်။ ဒါပေမဲ့ ကျွန်ုပ်တို့ဟာ ပိုမိုရှုပ်ထွေးတဲ့ ပြဿနာတွေကို ရင်ဆိုင်လာရတဲ့အခါ Lisp ဒါမှမဟုတ် တခြား သတ်မှတ်ထားပြီးသား ပရိုဂရမ်မင်း ဘာသာစကား တစ်ခုခုဟာ ကျွန်ုပ်တို့ရဲ့ လိုအပ်ချက်တွေအတွက် မလုံလောက်တာကို တွေ့ရပါလိမ့်မယ်။ ကျွန်ုပ်တို့ရဲ့ စိတ်ကူးတွေကို ပိုမိုထိရောက်စွာ ဖော်ပြနိုင်ဖို့အတွက် ဘာသာစကားသစ်တွေကို အမြဲတမ်း ပြောင်းလဲအသုံးပြုရပါမယ်။ ဘာသာစကားအသစ်တွေကို တည်ထောင်ခြင်းဟာ အင်ဂျင်နီယာ ဒီဇိုင်းမှာ ရှုပ်ထွေးမှုတွေကို ထိန်းချုပ်ဖို့ အစွမ်းထက်တဲ့ ဗျူဟာတစ်ခု ဖြစ်ပါတယ်၊ ပြဿနာကို နည်းလမ်းအသစ်နဲ့ ဖော်ပြနိုင်တဲ့ (ဒါကြောင့်ပဲ စဉ်းစားနိုင်တဲ့) ဘာသာစကားအသစ်ကို အသုံးပြုခြင်းအားဖြင့် ရှုပ်ထွေးတဲ့ ပြဿနာတွေကို ဖြေရှင်းနိုင်စွမ်းကို မြှင့်တင်နိုင်လေ့ရှိပါတယ်။ ဒီဘာသာစကားသစ်မှာ လက်ရှိပြဿနာနဲ့ အထူးကိုက်ညီတဲ့ primitives၊ ပေါင်းစပ်မှု နည်းလမ်းတွေနဲ့ abstraction နည်းလမ်းတွေ ပါဝင်မှာ ဖြစ်ပါတယ်။205

ပရိုဂရမ်မင်းမှာ ဘာသာစကားတွေ အများကြီး ရှိပါတယ်။ သတ်မှတ်ထားတဲ့ ကွန်ပျူတာတွေအတွက် machine languages လိုမျိုး ရုပ်ပိုင်းဆိုင်ရာ ဘာသာစကားတွေ ရှိပါတယ်။ ဒီဘာသာစကားတွေဟာ ဒေတာနဲ့ ထိန်းချုပ်မှုတွေကို သိုလှောင်မှု bit တစ်ခုချင်းစီနဲ့ အခြေခံ စက်ညွှန်ကြားချက်တွေ အနေနဲ့ ကိုယ်စားပြုဖို့ အဓိကထားပါတယ်။ Machine-language ပရိုဂရမ်မာဟာ ပေးထားတဲ့ hardware ကို အသုံးပြုပြီး အရင်းအမြစ် ကန့်သတ်ထားတဲ့ တွက်ချက်မှုတွေကို ထိရောက်စွာ လုပ်ဆောင်နိုင်ဖို့ စနစ်တွေနဲ့ အထောက်အကူပြု ပရိုဂရမ်တွေကို တည်ဆောက်ဖို့ အဓိကထားပါတယ်။ High-level languages တွေဟာ machine-language အခြေခံပေါ်မှာ တည်ဆောက်ထားပြီး၊ ဒေတာတွေကို bit အစုအဝေးတွေအနေနဲ့ ကိုယ်စားပြုခြင်းနဲ့ ပရိုဂရမ်တွေကို အခြေခံ ညွှန်ကြားချက် အစီအစဉ်တွေအနေနဲ့ ကိုယ်စားပြုခြင်းဆိုင်ရာ ကိစ္စရပ်တွေကို ဖုံးကွယ်ထားပေးပါတယ်။ ဒီဘာသာစကားတွေမှာ procedure definition လိုမျိုး စနစ်တွေကို အကြီးစား ဖွဲ့စည်းတည်ဆောက်မှုအတွက် သင့်တော်တဲ့ ပေါင်းစပ်မှုနဲ့ abstraction နည်းလမ်းတွေ ပါဝင်ပါတယ်။

Metalinguistic abstraction—ဘာသာစကားအသစ်တွေကို တည်ထောင်ခြင်း—ဟာ အင်ဂျင်နီယာ ဒီဇိုင်းရဲ့ ကဏ္ဍအားလုံးမှာ အရေးကြီးတဲ့ အခန်းကဏ္ဍက ပါဝင်ပါတယ်။ ဒါဟာ ကွန်ပျူတာ ပရိုဂရမ်မင်းမှာ အထူးသဖြင့် ပိုအရေးကြီးပါတယ်၊ ဘာကြောင့်လဲဆိုတော့ ပရိုဂရမ်မင်းမှာ ကျွန်ုပ်တို့ဟာ ဘာသာစကားအသစ်တွေကို ဖန်တီးရုံသာမက evaluator တွေကို တည်ဆောက်ခြင်းအားဖြင့် အဲဒီဘာသာစကားတွေကို အကောင်အထည်ဖော်နိုင်လို့ပါပဲ။ အကောင်အထည်ဖော်ပေးမယ့် evaluator (ဒါမှမဟုတ် interpreter) တစ်ခုဆိုတာကတော့ ပရိုဂရမ်မင်း ဘာသာစကားတစ်ခုရဲ့ expression တစ်ခုကို အသုံးပြုတဲ့အခါ၊ အဲဒီ expression ကို အကဲဖြတ်ဖို့ လိုအပ်တဲ့ လုပ်ဆောင်ချက်တွေကို လုပ်ဆောင်ပေးတဲ့ procedure တစ်ခု ဖြစ်ပါတယ်။

ဒါကို ပရိုဂရမ်မင်းမှာ အခြေခံအကျဆုံး စိတ်ကူးလို့ သတ်မှတ်တာဟာ ပုံကြီးချဲ့တာ မဟုတ်ပါဘူး။

ပရိုဂရမ်မင်း ဘာသာစကားတစ်ခုထဲက expression တွေရဲ့ အဓိပ္ပာယ်ကို ဆုံးဖြတ်ပေးတဲ့ evaluator ဟာ နောက်ထပ် ပရိုဂရမ်တစ်ခုသာ ဖြစ်ပါတယ်။

ဒီအချက်ကို နားလည်သဘောပေါက်ခြင်းဟာ ပရိုဂရမ်မာတွေအဖြစ် ကျွန်ုပ်တို့ရဲ့ ကိုယ့်ကိုယ်ကိုယ် မြင်တဲ့ ပုံရိပ်ကို ပြောင်းလဲစေပါတယ်။ ကျွန်ုပ်တို့ဟာ တခြားသူတွေ ဒီဇိုင်းဆွဲထားတဲ့ ဘာသာစကားတွေကို အသုံးပြုသူတွေ သက်သက်မဟုတ်ဘဲ ဘာသာစကားတွေကို ဒီဇိုင်းဆွဲသူတွေအဖြစ် မြင်လာပါတယ်။

တကယ်တော့၊ ပရိုဂရမ် တော်တော်များများကို ဘာသာစကား တစ်ခုခုအတွက် evaluator အဖြစ် သတ်မှတ်နိုင်ပါတယ်။ ဥပမာအားဖြင့်၊ 2.5.3 မှာပါတဲ့ polynomial manipulation system ဟာ polynomial arithmetic ရဲ့ စည်းမျဉ်းတွေကို အကောင်အထည်ဖော်ထားပြီး list-structured data တွေပေါ်မှာ လုပ်ဆောင်ချက်တွေအဖြစ် လုပ်ဆောင်ပါတယ်။ အကယ်၍ ကျွန်ုပ်တို့ဟာ ဒီစနစ်မှာ polynomial expression တွေကို ဖတ်ဖို့နဲ့ ရိုက်နှိပ်ဖို့ procedure တွေကို ထပ်ဖြည့်လိုက်ရင်၊ သင်္ကေတ သင်္ချာ (symbolic mathematics) ဆိုင်ရာ ပြဿနာတွေကို ဖြေရှင်းဖို့ အထူးပြု ဘာသာစကားတစ်ခုရဲ့ အနှစ်သာရကို ရရှိမှာ ဖြစ်ပါတယ်။ 3.3.4 က digital-logic simulator နဲ့ 3.3.5 က constraint propagator တွေဟာ သူတို့ဘာသာသူတို့ တရားဝင် ဘာသာစကားတွေ ဖြစ်ကြပြီး၊ တစ်ခုစီမှာ ကိုယ်ပိုင် primitives၊ ပေါင်းစပ်မှု နည်းလမ်းတွေနဲ့ abstraction နည်းလမ်းတွေ ရှိကြပါတယ်။ ဒီရှုထောင့်ကကြည့်ရင်၊ အကြီးစား ကွန်ပျူတာစနစ်တွေကို ကိုင်တွယ်ဖြေရှင်းတဲ့ နည်းပညာဟာ ကွန်ပျူတာ ဘာသာစကားအသစ်တွေ တည်ဆောက်တဲ့ နည်းပညာနဲ့ ပေါင်းစပ်သွားပြီး၊ ကွန်ပျူတာ သိပ္ပံ (computer science) ကိုယ်တိုင်ဟာလည်း သင့်တော်တဲ့ descriptive languages တွေကို တည်ဆောက်တဲ့ ပညာရပ်တစ်ခု ဖြစ်လာပါတယ်။

အခုဆိုရင် ဘာသာစကားတွေကို အခြားဘာသာစကားတွေနဲ့ ဘယ်လို တည်ထောင်သလဲဆိုတဲ့ နည်းပညာပိုင်းကို စတင်လေ့လာတော့မှာ ဖြစ်ပါတယ်။ ဒီအခန်းမှာ ကျွန်ုပ်တို့ဟာ Lisp ကို အခြေခံအဖြစ် အသုံးပြုပြီး evaluator တွေကို Lisp procedures အဖြစ် အကောင်အထည်ဖော်မှာ ဖြစ်ပါတယ်။ Lisp ဟာ သင်္ကေတ ဖော်ပြချက် (symbolic expressions) တွေကို ကိုယ်စားပြုနိုင်ပြီး ကိုင်တွယ်နိုင်စွမ်း ရှိတာကြောင့် ဒီအလုပ်အတွက် အထူးသင့်တော်ပါတယ်။ ဘာသာစကားတွေကို ဘယ်လို အကောင်အထည်ဖော်သလဲဆိုတာကို နားလည်ဖို့အတွက် ပထမဆုံးအဆင့်အနေနဲ့ Lisp ကိုယ်တိုင်အတွက် evaluator တစ်ခုကို တည်ဆောက်မှာ ဖြစ်ပါတယ်။ ကျွန်ုပ်တို့ရဲ့ evaluator က အကောင်အထည်ဖော်မယ့် ဘာသာစကားဟာ ဒီစာအုပ်မှာ အသုံးပြုထားတဲ့ Lisp ရဲ့ Scheme dialect ရဲ့ အစိတ်အပိုင်း (subset) တစ်ခု ဖြစ်ပါလိမ့်မယ်။ ဒီအခန်းမှာ ဖော်ပြထားတဲ့ evaluator ဟာ Lisp ရဲ့ သတ်မှတ်ထားတဲ့ dialect တစ်ခုအတွက် ရေးသားထားတာ ဖြစ်ပေမဲ့၊ sequential machine တွေအတွက် ပရိုဂရမ်တွေ ရေးသားဖို့ ဒီဇိုင်းထုတ်ထားတဲ့ ဘယ် expression-oriented language အတွက်မဆို လိုအပ်တဲ့ အခြေခံ တည်ဆောက်ပုံတွေ ပါဝင်ပါတယ်။ (တကယ်တော့၊ ဘာသာစကား ပရိုဆက်ဆာ (language processors) အတော်များများရဲ့ အတွင်းပိုင်းထဲမှာ “Lisp” evaluator သေးသေးလေး တစ်ခု ပါဝင်နေလေ့ ရှိပါတယ်။) သရုပ်ဖော်ဖို့နဲ့ ဆွေးနွေးဖို့အတွက် evaluator ကို ရိုးရှင်းအောင် ပြုလုပ်ထားပြီး၊ လက်တွေ့အသုံးပြုမယ့် production-quality Lisp system တွေမှာ ပါဝင်သင့်တဲ့ အရေးကြီးတဲ့ အင်္ဂါရပ်အချို့ကိုတော့ ချန်လှပ်ထားပါတယ်။ ဒါပေမဲ့ ဒီရိုးရှင်းတဲ့ evaluator ဟာ ဒီစာအုပ်မှာပါတဲ့ ပရိုဂရမ် အများစုကို လုပ်ဆောင်ဖို့အတွက် လုံလောက်ပါတယ်။206

Evaluator ကို Lisp ပရိုဂရမ်တစ်ခုအနေနဲ့ ရရှိနိုင်အောင် ပြုလုပ်ခြင်းရဲ့ အရေးကြီးတဲ့ အားသာချက်တစ်ခုကတော့ evaluator ပရိုဂရမ်မှာ ပြင်ဆင်မှုတွေ လုပ်ဆောင်ခြင်းအားဖြင့် alternative evaluation rules တွေကို အကောင်အထည်ဖော်နိုင်ခြင်းပဲ ဖြစ်ပါတယ်။ ဒီစွမ်းအားကို ကောင်းမွန်စွာ အသုံးပြုနိုင်တဲ့ နေရာတစ်ခုကတော့ Chapter 3 ရဲ့ ဆွေးနွေးမှုမှာ အဓိကကျခဲ့တဲ့ အချိန် (time) ဆိုတဲ့ သဘောတရားကို တွက်ချက်မှုဆိုင်ရာ မော်ဒယ်တွေမှာ ဘယ်လို ထည့်သွင်းမလဲဆိုတာကို ပိုမိုထိန်းချုပ်နိုင်ဖို့ပဲ ဖြစ်ပါတယ်။ အဲဒီမှာ ကျွန်ုပ်တို့ဟာ ကမ္ဘာပေါ်က အချိန်နဲ့ ကွန်ပျူတာထဲက အချိန်ကို ခွဲထုတ်ဖို့ streams တွေကို အသုံးပြုပြီး state နဲ့ assignment ဆိုင်ရာ ရှုပ်ထွေးမှုအချို့ကို လျှော့ချခဲ့ပါတယ်။ ဒါပေမဲ့ ကျွန်ုပ်တို့ရဲ့ stream ပရိုဂရမ်တွေဟာ Scheme ရဲ့ applicative-order evaluation ကြောင့် တစ်ခါတစ်ရံမှာ ရှုပ်ထွေးလေးလံနေတတ်ပါတယ်။ 4.2 မှာတော့ evaluator ကို ပြင်ဆင်ပြီး normal-order evaluation ကို အသုံးပြုနိုင်အောင် ပြုလုပ်ခြင်းအားဖြင့် ပိုမိုကောင်းမွန်တဲ့ နည်းလမ်းတစ်ခုကို ရရှိဖို့အတွက် အခြေခံဘာသာစကားကို ပြောင်းလဲမှာ ဖြစ်ပါတယ်။

Section 4.3 မှာတော့ expression တွေမှာ တန်ဖိုးတစ်ခုတည်း မဟုတ်ဘဲ တန်ဖိုးအများကြီး ရှိနိုင်တဲ့ ပိုမို ရည်မှန်းချက်ကြီးတဲ့ ဘာသာစကားဆိုင်ရာ ပြောင်းလဲမှုကို အကောင်အထည်ဖော်မှာ ဖြစ်ပါတယ်။ ဒီ nondeterministic computing ဘာသာစကားမှာ expression တွေအတွက် ဖြစ်နိုင်ခြေရှိတဲ့ တန်ဖိုးအားလုံးကို ထုတ်ပေးပြီး၊ သတ်မှတ်ထားတဲ့ ကန့်သတ်ချက်တွေနဲ့ ကိုက်ညီတဲ့ တန်ဖိုးတွေကို ရှာဖွေတဲ့ လုပ်ငန်းစဉ်တွေကို ဖော်ပြဖို့ဟာ သဘာဝကျပါတယ်။ တွက်ချက်မှုနဲ့ အချိန် မော်ဒယ်တွေအရဆိုရင် ဒါဟာ အချိန်ကို “ဖြစ်နိုင်ခြေရှိတဲ့ အနာဂတ်များ” အဖြစ် ခွဲထွက်သွားစေပြီး သင့်တော်တဲ့ အချိန်လိုင်းတွေကို ရှာဖွေတာနဲ့ တူပါတယ်။ ကျွန်ုပ်တို့ရဲ့ nondeterministic evaluator နဲ့ဆိုရင် တန်ဖိုးအများကြီးကို မှတ်သားထားခြင်းနဲ့ ရှာဖွေမှုတွေ ပြုလုပ်ခြင်းကို ဘာသာစကားရဲ့ အခြေခံ ယန္တရားကနေ အလိုအလျောက် ကိုင်တွယ်ပေးမှာ ဖြစ်ပါတယ်။

In 4.4 မှာတော့ အသိပညာတွေကို input နဲ့ output တွက်ချက်မှုတွေအနေနဲ့ မဟုတ်ဘဲ ဆက်နွယ်မှုတွေအနေနဲ့ ဖော်ပြတဲ့ logic-programming ဘာသာစကားတစ်ခုကို အကောင်အထည်ဖော်မှာ ဖြစ်ပါတယ်။ ဒါဟာ ဘာသာစကားကို Lisp ဒါမှမဟုတ် တခြား သာမန်ဘာသာစကားတွေနဲ့ သိသိသာသာ ကွဲပြားစေပေမဲ့၊ logic-programming evaluator ဟာ Lisp evaluator ရဲ့ အခြေခံ တည်ဆောက်ပုံကိုပဲ မျှဝေအသုံးပြုထားတာကို တွေ့ရပါလိမ့်မယ်။

205 ဒီစိတ်ကူးဟာ အင်ဂျင်နီယာပညာရပ် အားလုံးမှာ ပျံ့နှံ့နေပါတယ်။ ဥပမာအားဖြင့် လျှပ်စစ်အင်ဂျင်နီယာတွေဟာ ပတ်လမ်းတွေကို ဖော်ပြဖို့ မတူညီတဲ့ ဘာသာစကားများစွာကို အသုံးပြုကြပါတယ်။ အဲဒီထဲက နှစ်ခုကတော့ electrical networks ဘာသာစကားနဲ့ electrical systems ဘာသာစကားတို့ ဖြစ်ပါတယ်။ Network ဘာသာစကားဟာ စက်ပစ္စည်းတွေကို သီးခြားစီဖြစ်တဲ့ လျှပ်စစ် အစိတ်အပိုင်းတွေအနေနဲ့ ရုပ်ပိုင်းဆိုင်ရာ ပုံစံတူ ဖန်တီးခြင်းကို အလေးပေးပါတယ်။ Network ဘာသာစကားရဲ့ primitive objects တွေကတော့ voltage နဲ့ current လို့ ခေါ်တဲ့ ရုပ်ပိုင်းဆိုင်ရာ ကိန်းရှင်တွေနဲ့ သတ်မှတ်ထားတဲ့ resistors၊ capacitors၊ inductors နဲ့ transistors တို့လို အခြေခံ လျှပ်စစ်အစိတ်အပိုင်းတွေ ဖြစ်ပါတယ်။ Network ဘာသာစကားနဲ့ ပတ်လမ်းတွေကို ဖော်ပြတဲ့အခါ အင်ဂျင်နီယာဟာ ဒီဇိုင်းရဲ့ ရုပ်ပိုင်းဆိုင်ရာ လက္ခဏာတွေကို အဓိကထားပါတယ်။ ဆန့်ကျင်ဘက်အနေနဲ့ system ဘာသာစကားရဲ့ primitive objects တွေကတော့ filters နဲ့ amplifiers တွေလိုမျိုး signal-processing modules တွေ ဖြစ်ပါတယ်။ Module တွေရဲ့ လုပ်ဆောင်ချက်ဆိုင်ရာ အမူအကျင့်တွေကသာ သက်ဆိုင်ပြီး၊ signals တွေကို voltage နဲ့ current တွေအဖြစ် ရုပ်ပိုင်းဆိုင်ရာ အကောင်အထည်ဖော်မှုတွေကို ဂရုစိုက်စရာမလိုဘဲ ကိုင်တွယ်နိုင်ပါတယ်။ System ဘာသာစကားကို network ဘာသာစကားပေါ်မှာ တည်ဆောက်ထားတာ ဖြစ်ပြီး၊ signal-processing systems တွေရဲ့ အစိတ်အပိုင်းတွေကို လျှပ်စစ် networks တွေကနေ တည်ဆောက်ထားတယ်ဆိုတဲ့ အဓိပ္ပာယ်နဲ့ ဖြစ်ပါတယ်။ ဒါပေမဲ့ ဒီမှာတော့ ပေးထားတဲ့ ပြဿနာကို ဖြေရှင်းဖို့ လျှပ်စစ်ပစ္စည်းတွေကို အကြီးစား ဖွဲ့စည်းတည်ဆောက်မှုအပေါ်မှာပဲ အာရုံစိုက်ပြီး၊ အစိတ်အပိုင်းတွေရဲ့ ရုပ်ပိုင်းဆိုင်ရာ ဖြစ်နိုင်ခြေကိုတော့ ရှိပြီးသားလို့ ယူဆလိုက်ပါတယ်။ ဒီလို အဆင့်ဆင့် စုစည်းထားတဲ့ ဘာသာစကားတွေဟာ 2.2.4 မှာ သရုပ်ဖော်ထားတဲ့ picture language ရဲ့ stratified design နည်းလမ်းရဲ့ နောက်ထပ် ဥပမာတစ်ခု ဖြစ်ပါတယ်။

206 ကျွန်ုပ်တို့ရဲ့ evaluator မှာ ချန်လှပ်ထားခဲ့တဲ့ အရေးကြီးဆုံး အင်္ဂါရပ်တွေကတော့ error တွေကို ကိုင်တွယ်တဲ့ ယန္တရားတွေနဲ့ debugging ကို ထောက်ပံ့ပေးတဲ့ အရာတွေ ဖြစ်ပါတယ်။ Evaluator တွေအကြောင်း ပိုမိုပြည့်စုံတဲ့ ဆွေးနွေးချက်အတွက် Friedman et al. 1992 ကို ကြည့်ရှုပါ၊ အဲဒီမှာ Scheme နဲ့ ရေးသားထားတဲ့ evaluator တွေကတစ်ဆင့် ပရိုဂရမ်မင်း ဘာသာစကားတွေအကြောင်း ရှင်းလင်းဖော်ပြထားပါတယ်။
