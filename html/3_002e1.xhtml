3.1တန်ဖိုးသတ်မှတ်ခြင်းနှင့် ဒေသတွင်းအခြေအနေ (Assignment and Local State)

ကျွန်ုပ်တို့သည် ကမ္ဘာလောကကြီးကို တစ်ခုနှင့်တစ်ခု အမှီအခိုကင်းသော အရာဝတ္ထု (objects) များဖြင့် ဖွဲ့စည်းထားသည်ဟု ပုံမှန်အားဖြင့် ရှုမြင်ကြပြီး၊ ထိုအရာဝတ္ုတစ်ခုချင်းစီတွင် အချိန်နှင့်အမျှ ပြောင်းလဲနေသော အခြေအနေ (state) တစ်ခုစီ ရှိကြသည်။ အရာဝတ္ထုတစ်ခု၏ အပြုအမူသည် ၎င်း၏ အတိတ်ဖြစ်ရပ်များအပေါ် မူတည်နေပါက ထိုအရာဝတ္ထုတွင် “အခြေအနေ ရှိသည်” ဟု ဆိုနိုင်သည်။ ဥပမာအားဖြင့် ဘဏ်အကောင့်တစ်ခုတွင် အခြေအနေရှိသည်၊ အဘယ်ကြောင့်ဆိုသော် “ကျွန်ုပ် ဒေါ်လာ ၁၀၀ ထုတ်ယူနိုင်ပါသလား?” ဟူသော မေးခွန်း၏ အဖြေသည် ငွေသွင်းငွေထုတ် ပြုလုပ်ခဲ့သည့် အတိတ်မှတ်တမ်းများအပေါ် မူတည်သောကြောင့် ဖြစ်သည်။ ကျွန်ုပ်တို့သည် အရာဝတ္ထုတစ်ခု၏ အခြေအနေကို တစ်ခု သို့မဟုတ် တစ်ခုထက်ပိုသော အခြေအနေပြကိန်းရှင် (state variables) များဖြင့် သတ်မှတ်ဖော်ပြနိုင်သည်။ ထိုကိန်းရှင်များသည် အရာဝတ္ထု၏ လက်ရှိအပြုအမူကို ဆုံးဖြတ်ရန်အတွက် လိုအပ်သော အတိတ်ဖြစ်ရပ် အချက်အလက်များကို ထိန်းသိမ်းပေးထားသည်။ ရိုးရှင်းသော ဘဏ်စနစ်တစ်ခုတွင် ကျွန်ုပ်တို့သည် အကောင့်တစ်ခု၏ အခြေအနေကို ငွေသွင်းငွေထုတ် မှတ်တမ်းအားလုံးကို မှတ်သားထားမည့်အစား လက်ကျန်ငွေ (current balance) ဖြင့် သတ်မှတ်ဖော်ပြနိုင်သည်။

အရာဝတ္ထုများစွာဖြင့် ဖွဲ့စည်းထားသော စနစ်တစ်ခုတွင် အရာဝတ္ထုများသည် လုံးဝလွတ်လပ်နေခြင်းမျိုး ခဲယဉ်းသည်။ အချင်းချင်း ဆက်သွယ်ဆောင်ရွက်မှုများမှတစ်ဆင့် တစ်ခုသည် အခြားအရာဝတ္ထုများ၏ အခြေအနေများအပေါ် သက်ရောက်မှု ရှိနိုင်သည်။ ထိုသို့ ဆက်သွယ်ဆောင်ရွက်ခြင်းက အရာဝတ္ထုတစ်ခု၏ အခြေအနေပြကိန်းရှင်များကို အခြားအရာဝတ္ထုများ၏ အခြေအနေပြကိန်းရှင်များနှင့် ချိတ်ဆက်ပေးသည်။ အမှန်စင်စစ်တွင် စနစ်တစ်ခုကို သီးခြားအရာဝတ္ထုများဖြင့် ဖွဲ့စည်းထားသည်ဟူသော အမြင်သည် စနစ်၏ အခြေအနေပြကိန်းရှင်များကို တစ်ခုနှင့်တစ်ခု အားကောင်းစွာ ချိတ်ဆက်ထားသော စနစ်ငယ် (subsystems) များအဖြစ် စုစည်းနိုင်ပြီး၊ ထိုစနစ်ငယ်များသည် အခြားစနစ်ငယ်များနှင့် အားနည်းစွာသာ ချိတ်ဆက်နေသည့် အခါမျိုးတွင် အသုံးအဝင်ဆုံး ဖြစ်သည်။

စနစ်တစ်ခုအပေါ် ဤသို့ရှုမြင်ခြင်းသည် ထိုစနစ်၏ တွက်ချက်မှုဆိုင်ရာ မော်ဒယ် (computational models) များကို ဖွဲ့စည်းတည်ဆောက်ရန်အတွက် အားကောင်းသော မူဘောင်တစ်ခု ဖြစ်နိုင်သည်။ ထိုကဲ့သို့သော မော်ဒယ်တစ်ခုသည် မော်ဂျူလာ (modular) ဖြစ်စေရန်အတွက် ၎င်းကို စနစ်အတွင်းရှိ အစစ်အမှန်အရာဝတ္ထုများကို ပုံတူပြုထားသော တွက်ချက်မှုဆိုင်ရာ အရာဝတ္ထု (computational objects) များအဖြစ် ခွဲခြမ်းစိတ်ဖြာထားသင့်သည်။ တွက်ချက်မှုဆိုင်ရာ အရာဝတ္ထုတစ်ခုစီတွင် အစစ်အမှန်အရာဝတ္ထု၏ အခြေအနေကို ဖော်ပြသော ၎င်း၏ ကိုယ်ပိုင် ဒေသတွင်းအခြေအနေပြကိန်းရှင် (local state variables) များ ရှိရမည်။ ပုံတူပြုထားသော စနစ်အတွင်းရှိ အရာဝတ္ထုများ၏ အခြေအနေများသည် အချိန်နှင့်အမျှ ပြောင်းလဲနေသောကြောင့်၊ ၎င်းတို့နှင့် သက်ဆိုင်သော တွက်ချက်မှုဆိုင်ရာ အရာဝတ္ထုများ၏ အခြေအနေပြကိန်းရှင်များသည်လည်း ပြောင်းလဲရမည် ဖြစ်သည်။ အကယ်၍ ကျွန်ုပ်တို့သည် စနစ်အတွင်းရှိ အချိန်စီးဆင်းမှုကို ကွန်ပျူတာအတွင်း ကုန်ဆုံးသွားသော အချိန်ဖြင့် ပုံတူပြုရန် ရွေးချယ်ပါက၊ ပရိုဂရမ်များ လုပ်ဆောင်နေစဉ်အတွင်း အပြုအမူများ ပြောင်းလဲနိုင်သော တွက်ချက်မှုဆိုင်ရာ အရာဝတ္ထုများကို တည်ဆောက်ရန် နည်းလမ်းတစ်ခု ရှိရမည်ဖြစ်သည်။ အထူးသဖြင့် ကျွန်ုပ်တို့သည် အခြေအနေပြကိန်းရှင်များကို ပရိုဂရမ်မင်းဘာသာစကားရှိ သာမန် သင်္ကေတအမည် (symbolic names) များဖြင့် ပုံတူပြုလိုပါက၊ ထိုဘာသာစကားတွင် အမည်တစ်ခုနှင့် သက်ဆိုင်နေသော တန်ဖိုးကို ပြောင်းလဲနိုင်ရန်အတွက် တန်ဖိုးသတ်မှတ်ခြင်း အော်ပရေတာ (assignment operator) တစ်ခု ပါရှိရမည်ဖြစ်သည်။

3.1.1Local State Variables

အချိန်နှင့်အမျှ ပြောင်းလဲနေသော အခြေအနေ (time-varying state) ရှိသည့် computational object တစ်ခုအကြောင်းကို ရှင်းပြရန်အတွက် ဘဏ်အကောင့်တစ်ခုမှ ငွေထုတ်ယူခြင်း အခြေအနေကို ပုံစံတူ (model) ပြုလုပ်ကြည့်ကြပါစို့။ ထုတ်ယူမည့် ပမာဏ amount ကို argument အဖြစ် လက်ခံသည့် withdraw ဆိုသော procedure ကို အသုံးပြု၍ ကျွန်ုပ်တို့ ဤသို့ လုပ်ဆောင်ပါမည်။ အကယ်၍ အကောင့်ထဲတွင် ငွေထုတ်ယူရန် လုံလောက်သော ငွေပမာဏရှိပါက၊ withdraw သည် ငွေထုတ်ပြီးနောက် ကျန်ရှိသော လက်ကျန်ငွေ (balance) ကို return ပြန်ပေးရပါမည်။ သို့မဟုတ်ပါက withdraw သည် Insufficient funds ဟူသော မက်ဆေ့ချ်ကို ပြန်ပေးရပါမည်။ ဥပမာအားဖြင့်၊ အကယ်၍ ကျွန်ုပ်တို့သည် အကောင့်ထဲတွင် $100 ဖြင့် စတင်ခဲ့ပါက၊ withdraw ကို အသုံးပြု၍ အောက်ပါအတိုင်း တုံ့ပြန်မှုများကို ရရှိမည်ဖြစ်သည်။

(withdraw 25) ဟူသော expression ကို နှစ်ကြိမ် evaluate လုပ်ရာတွင် မတူညီသော တန်ဖိုးများ ရရှိသည်ကို သတိပြုပါ။ ဤသည်မှာ procedure တစ်ခုအတွက် အပြုအမူ အသစ်တစ်မျိုး ဖြစ်သည်။ ယခုအချိန်အထိ ကျွန်ုပ်တို့၏ procedure များအားလုံးကို သင်္ချာဆိုင်ရာ function များကို တွက်ချက်ရန်အတွက် သတ်မှတ်ချက်များ (specifications) အဖြစ်သာ ရှုမြင်နိုင်ခဲ့သည်။ Procedure တစ်ခုကို ခေါ်ယူခြင်းသည် ပေးထားသော argument များအပေါ်တွင် သက်ရောက်သည့် function ၏ တန်ဖိုးကို တွက်ချက်ခြင်းဖြစ်ပြီး၊ တူညီသော argument များဖြင့် တူညီသော procedure ကို နှစ်ကြိမ်ခေါ်ယူခြင်းသည် အမြဲတမ်း တူညီသော ရလဒ်ကိုသာ ထွက်ပေါ်စေသည်။129

withdraw ကို အကောင်အထည်ဖော်ရန်အတွက် အကောင့်ထဲရှိ လက်ကျန်ငွေကို ဖော်ပြရန် balance ဆိုသော variable ကို အသုံးပြုနိုင်ပြီး၊ balance ကို access လုပ်နိုင်သည့် procedure တစ်ခုအဖြစ် withdraw ကို အဓိပ္ပာယ်ဖွင့်ဆို (define) နိုင်ပါသည်။ withdraw procedure သည် balance သည် တောင်းဆိုထားသော amount ထက် အနည်းဆုံး ကြီးမားခြင်း သို့မဟုတ် ညီမျှခြင်း ရှိမရှိကို စစ်ဆေးသည်။ အကယ်၍ ရှိပါက၊ withdraw သည် balance မှ amount ကို နှုတ်လိုက်ပြီး (decrement)၊ balance ၏ တန်ဖိုးအသစ်ကို return ပြန်ပေးသည်။ သို့မဟုတ်ပါက withdraw သည် Insufficient funds မက်ဆေ့ချ်ကို ပြန်ပေးသည်။ balance နှင့် withdraw တို့၏ အဓိပ္ပာယ်ဖွင့်ဆိုချက်များမှာ အောက်ပါအတိုင်း ဖြစ်သည်။ 

(define balance 100)

(define (withdraw amount)
  (if (>= balance amount)
      (begin (set! balance (- balance amount))
             balance)
      "Insufficient funds"))

balance ကို လျှော့ချခြင်း (decrementing) ကို အောက်ပါ expression ဖြင့် လုပ်ဆောင်သည်။ 

(set! balance (- balance amount))

Withdraw သည် if စမ်းသပ်ချက် မှန်ကန်သောအခါတွင် expression နှစ်ခုကို evaluate လုပ်ရန်အတွက် begin special form ကိုလည်း အသုံးပြုထားသည်။ ပထမဦးစွာ balance ကို လျှော့ချပြီး နောက်ပိုင်းတွင် balance ၏ တန်ဖိုးကို return ပြန်ပေးခြင်း ဖြစ်သည်။ ယေဘုယျအားဖြင့် expression

ကို evaluate လုပ်ခြင်းသည် expression ⟨ e x p 1 ⟩ မှ ⟨ e x p k ⟩ အထိကို အစဉ်လိုက် evaluate လုပ်စေပြီး၊ နောက်ဆုံး expression ⟨ e x p k ⟩ ၏ တန်ဖိုးကို begin form တစ်ခုလုံး၏ တန်ဖိုးအဖြစ် ပြန်ပေးစေသည်။131

withdraw သည် လိုချင်သလို အလုပ်လုပ်သော်လည်း၊ balance variable သည် ပြဿနာတစ်ခု ရှိနေသည်။ အထက်တွင် ဖော်ပြထားသည့်အတိုင်း၊ balance သည် global environment တွင် အဓိပ္ပာယ်ဖွင့်ဆိုထားသော အမည်တစ်ခုဖြစ်ပြီး မည်သည့် procedure ကမဆို လွတ်လပ်စွာ ကြည့်ရှုခြင်း သို့မဟုတ် ပြုပြင်ခြင်း ပြုလုပ်နိုင်သည်။ အကယ်၍ ကျွန်ုပ်တို့သည် balance ကို withdraw ၏ အတွင်းပိုင်း (internal) အဖြစ် တစ်နည်းနည်းဖြင့် ပြုလုပ်နိုင်ပါက ပိုမိုကောင်းမွန်မည် ဖြစ်သည်။ ထိုသို့ပြုလုပ်ခြင်းဖြင့် withdraw သည်သာလျှင် balance ကို တိုက်ရိုက် access လုပ်နိုင်မည်ဖြစ်ပြီး၊ အခြား procedure များသည် withdraw ကို ခေါ်ယူခြင်းမှတစ်ဆင့် သွယ်ဝိုက်သောနည်းဖြင့်သာ balance ကို access လုပ်နိုင်မည် ဖြစ်သည်။ ဤသည်မှာ balance သည် အကောင့်၏ အခြေအနေကို ခြေရာခံရန် withdraw က အသုံးပြုသော local state variable တစ်ခုဖြစ်သည်ဟူသော အယူအဆကို ပိုမိုတိကျစွာ ပုံဖော်နိုင်မည် ဖြစ်သည်။

Definition ကို အောက်ပါအတိုင်း ပြန်လည်ရေးသားခြင်းဖြင့် balance ကို withdraw ၏ အတွင်းပိုင်း (internal) အဖြစ် ပြုလုပ်နိုင်ပါသည်။ 

(define new-withdraw
  (let ((balance 100))
    (lambda (amount)
      (if (>= balance amount)
          (begin (set! balance 
                       (- balance amount))
                 balance)
          "Insufficient funds"))))

ဤနေရာတွင် ကျွန်ုပ်တို့ လုပ်ဆောင်ခဲ့သည်မှာ ကနဦးတန်ဖိုး 100 ဖြင့် သတ်မှတ်ထားသော local variable balance ပါဝင်သည့် environment တစ်ခုကို တည်ဆောက်ရန် let ကို အသုံးပြုခဲ့ခြင်း ဖြစ်သည်။ ဤ local environment အတွင်း၌၊ amount ကို argument အဖြစ် လက်ခံပြီး ကျွန်ုပ်တို့၏ ယခင် withdraw procedure ကဲ့သို့ အလုပ်လုပ်သည့် procedure တစ်ခု ဖန်တီးရန် lambda ကို အသုံးပြုသည်။ let expression ကို evaluate လုပ်ခြင်း၏ ရလဒ်အဖြစ် ပြန်လာသော ဤ procedure သည် new-withdraw ဖြစ်သည်။ ၎င်းသည် withdraw နှင့် အတိအကျ တူညီသော နည်းလမ်းဖြင့် အလုပ်လုပ်သော်လည်း ၎င်း၏ variable balance ကို အခြား မည်သည့် procedure ကမှ access မလုပ်နိုင်ပါ။132

set! ကို local variables များဖြင့် ပေါင်းစပ်ခြင်းသည် local state ရှိသော computational objects များကို တည်ဆောက်ရန် ကျွန်ုပ်တို့ အသုံးပြုမည့် ယေဘုယျ programming နည်းစနစ် ဖြစ်သည်။ ကံမကောင်းစွာဖြင့်၊ ဤနည်းစနစ်ကို အသုံးပြုခြင်းသည် ကြီးမားသော ပြဿနာတစ်ခုကို ဖြစ်ပေါ်စေသည်။ ကျွန်ုပ်တို့ procedure များကို စတင်မိတ်ဆက်စဉ်က procedure application ဆိုသည်မှာ ဘာကို ဆိုလိုသည်ကို ရှင်းပြရန် evaluation ၏ substitution model (1.1.5) ကိုလည်း မိတ်ဆက်ပေးခဲ့သည်။ Procedure တစ်ခုကို apply လုပ်ခြင်းကို formal parameter များအား ၎င်းတို့၏ တန်ဖိုးများဖြင့် အစားထိုးပြီး procedure ၏ body ကို evaluate လုပ်ခြင်းအဖြစ် အဓိပ္ပာယ်ဖွင့်ဆိုရမည်ဟု ကျွန်ုပ်တို့ ပြောခဲ့သည်။ ပြဿနာမှာ ကျွန်ုပ်တို့၏ language ထဲသို့ assignment ကို မိတ်ဆက်လိုက်သည်နှင့် တပြိုင်နက်၊ substitution သည် procedure application အတွက် လုံလောက်သော model တစ်ခု မဟုတ်တော့ခြင်းပင် ဖြစ်သည်။ (ဤသို့ဖြစ်ရခြင်း အကြောင်းရင်းကို 3.1.3 တွင် တွေ့မြင်ရမည် ဖြစ်သည်။) ထို့ကြောင့် နည်းပညာအရ ကျွန်ုပ်တို့အနေဖြင့် new-withdraw procedure သည် အထက်ပါအတိုင်း အဘယ်ကြောင့် အလုပ်လုပ်ရသည်ကို နားလည်ရန် ယခုအချိန်တွင် နည်းလမ်းမရှိသေးပါ။ new-withdraw ကဲ့သို့သော procedure တစ်ခုကို အမှန်တကယ် နားလည်နိုင်ရန်အတွက်၊ procedure application ၏ model အသစ်တစ်ခုကို တည်ဆောက်ရန် လိုအပ်မည် ဖြစ်သည်။ 3.2 တွင် set! နှင့် local variables များအကြောင်း ရှင်းလင်းချက်များနှင့်အတူ ထိုကဲ့သို့သော model တစ်ခုကို မိတ်ဆက်ပေးပါမည်။ သို့သော် ပထမဦးစွာ၊ new-withdraw က တည်ဆောက်ပေးလိုက်သော အယူအဆပေါ်တွင် အခြေခံထားသည့် အခြားပုံစံအချို့ကို လေ့လာကြည့်ကြပါစို့။

ကျွန်ုပ်တို့သည် ငွေထုတ်ယူခြင်းသာမက ငွေသွင်းခြင်းကိုပါ ကိုင်တွယ်နိုင်သော object များကိုလည်း ဖန်တီးနိုင်ပြီး၊ ထို့ကြောင့် ရိုးရှင်းသော ဘဏ်အကောင့်များကို ကိုယ်စားပြုနိုင်သည်။ သတ်မှတ်ထားသော ကနဦးလက်ကျန်ငွေပါဝင်သည့် "bank-account object" တစ်ခုကို return ပြန်ပေးသည့် procedure တစ်ခုမှာ အောက်ပါအတိုင်း ဖြစ်သည်။ 

(define (make-account balance)
  (define (withdraw amount)
    (if (>= balance amount)
        (begin (set! balance 
                     (- balance amount))
               balance)
        "Insufficient funds"))
  (define (deposit amount)
    (set! balance (+ balance amount))
    balance)
  (define (dispatch m)
    (cond ((eq? m 'withdraw) withdraw)
          ((eq? m 'deposit) deposit)
          (else (error "Unknown request: 
                 MAKE-ACCOUNT" m))))
  dispatch)

make-account ကို ခေါ်ယူတိုင်းတွင် local state variable balance ပါဝင်သော environment တစ်ခုကို တည်ဆောက်ပေးသည်။ ဤ environment အတွင်း၌၊ make-account သည် balance ကို access လုပ်သည့် deposit နှင့် withdraw procedure များကို အဓိပ္ပာယ်ဖွင့်ဆိုပြီး၊ "message" တစ်ခုကို input အဖြစ် လက်ခံကာ local procedure နှစ်ခုထဲမှ တစ်ခုကို return ပြန်ပေးသည့် နောက်ထပ် procedure တစ်ခုဖြစ်သော dispatch ကိုလည်း အဓိပ္ပာယ်ဖွင့်ဆိုသည်။ dispatch procedure ကိုယ်တိုင်ကို bank-account object ကို ကိုယ်စားပြုသည့် တန်ဖိုးအဖြစ် ပြန်ပေးသည်။ ဤသည်မှာ message-passing ပုံစံ programming အတိအကျပင် ဖြစ်သည်။ သို့သော် ဤနေရာတွင် ၎င်းကို local variables များကို ပြုပြင်နိုင်စွမ်းနှင့် ပေါင်းစပ်အသုံးပြုထားခြင်း ဖြစ်သည်။

3.1.2Assignment ကို စတင်အသုံးပြုခြင်း၏ အကျိုးကျေးဇူးများ

ကျွန်ုပ်တို့ မြင်တွေ့ရမည့်အတိုင်း၊ ကျွန်ုပ်တို့၏ programming language ထဲသို့ assignment ကို ထည့်သွင်းခြင်းသည် ခက်ခဲသော အယူအဆဆိုင်ရာ ပြဿနာများစွာဆီသို့ ဦးတည်သွားစေပါသည်။ သို့သော်ငြားလည်း၊ စနစ်များကို local state ရှိသော object များ၏ စုစည်းမှုအဖြစ် ရှုမြင်ခြင်းသည် modular design ကို ထိန်းသိမ်းရန်အတွက် အစွမ်းထက်သော နည်းလမ်းတစ်ခု ဖြစ်ပါသည်။ ရိုးရှင်းသော ဥပမာတစ်ခုအနေဖြင့်၊ ခေါ်ယူသည့်အခါတိုင်း random ရွေးချယ်ထားသော ကိန်းပြည့်တစ်ခုကို return ပြန်ပေးသည့် rand ဆိုသော procedure ၏ ဒီဇိုင်းကို စဉ်းစားကြည့်ပါ။

“random ရွေးချယ်သည်” ဆိုသည်မှာ ဘာကိုဆိုလိုသည်မှာ လုံးဝ ရှင်းလင်းမှုမရှိပါ။ ကျွန်ုပ်တို့ အလိုရှိသည်မှာ rand ကို ဆက်တိုက်ခေါ်ယူမှုများမှ uniform distribution ဖြစ်သော စာရင်းအင်းဆိုင်ရာ ဂုဏ်သတ္တိများရှိသည့် ကိန်းဂဏန်း အစီအစဉ်တစ်ခု ထွက်ပေါ်လာရန် ဖြစ်သည်ဟု ယူဆရပါမည်။ သင့်တော်သော အစီအစဉ်များ ထုတ်လုပ်ရန် နည်းလမ်းများကို ဤနေရာတွင် ဆွေးနွေးမည်မဟုတ်ပါ။ ယင်းအစား၊ ပေးထားသော ကိန်းဂဏန်း x 1 ဖြင့် စတင်ပြီး 

x₂ = (rand-update x₁)
x₃ = (rand-update x₂)

ကို ဖွဲ့စည်းပါက၊ တန်ဖိုးများ အစီအစဉ် x 1 ၊ x 2 ၊ x 3 ၊ … သည် အလိုရှိသော စာရင်းအင်းဆိုင်ရာ ဂုဏ်သတ္တိများ ရှိမည်ဖြစ်သော rand-update procedure တစ်ခု ရှိသည်ဟု ယူဆကြပါစို့။134

ကျွန်ုပ်တို့သည် rand ကို သတ်မှတ်ထားသော တန်ဖိုး random-init ဖြင့် စတင်ထားသည့် local state variable x ပါဝင်သော procedure တစ်ခုအဖြစ် အကောင်အထည်ဖော်နိုင်ပါသည်။ rand ကို ခေါ်ယူမှုတိုင်းသည် x ၏ လက်ရှိတန်ဖိုးကို rand-update တွက်ချက်ပြီး၊ ယင်းကို random number အဖြစ် return ပြန်ပေးကာ x ၏ တန်ဖိုးအသစ်အဖြစ်လည်း သိမ်းဆည်းထားမည် ဖြစ်ပါသည်။ 

(define rand
  (let ((x random-init))
    (lambda () (set! x (rand-update x)) x)))

မှန်ပါသည်၊ ကျွန်ုပ်တို့သည် assignment ကို အသုံးမပြုဘဲ rand-update ကို တိုက်ရိုက်ခေါ်ယူခြင်းဖြင့်လည်း အဆိုပါ random number အစီအစဉ်ကိုပင် ထုတ်လုပ်နိုင်ပါသည်။ သို့သော်၊ ဤသို့ပြုလုပ်ခြင်းသည် random numbers များကို အသုံးပြုသည့် ကျွန်ုပ်တို့၏ program ၏ မည်သည့်အစိတ်အပိုင်းမဆို rand-update ထံသို့ argument အဖြစ် ပေးပို့ရန် x ၏ လက်ရှိတန်ဖိုးကို အတိအလင်း မှတ်သားထားရမည်ဟု ဆိုလိုပါသည်။ ဤသည်မှာ မည်မျှအနှောင့်အယှက်ဖြစ်စေမည်ကို သိရှိနိုင်ရန်၊ Monte Carlo simulation ဟုခေါ်သော နည်းစနစ်တစ်ခုကို အကောင်အထည်ဖော်ရန် random numbers များကို အသုံးပြုခြင်းအား စဉ်းစားကြည့်ပါ။

Monte Carlo နည်းလမ်းသည် ကြီးမားသော အစုအဝေးတစ်ခုထဲမှ စမ်းသပ်ချက်နမူနာများကို random ရွေးချယ်ပြီးနောက်၊ ထိုစမ်းသပ်ချက်များ၏ ရလဒ်များကို ဇယားပြုစုခြင်းမှ ခန့်မှန်းရရှိသော ဖြစ်တန်စွမ်းများ (probabilities) အပေါ် အခြေခံ၍ ကောက်ချက်ချခြင်း ဖြစ်ပါသည်။ ဥပမာအားဖြင့်၊ random ရွေးချယ်ထားသော ကိန်းပြည့်နှစ်ခုတွင် တူညီသော ဆခွဲကိန်းများ (factors) မရှိခြင်း၊ ဆိုလိုသည်မှာ ၎င်းတို့၏ အကြီးဆုံးဘုံဆခွဲကိန်း (greatest common divisor) သည် 1 ဖြစ်နေခြင်း၏ ဖြစ်တန်စွမ်းမှာ 6 / π 2 ဖြစ်သည်ဟူသော အချက်ကို အသုံးပြု၍ ကျွန်ုပ်တို့သည် π ကို အနီးစပ်ဆုံး တွက်ချက်နိုင်ပါသည်။135 π ၏ အနီးစပ်ဆုံးတန်ဖိုးကို ရရှိရန် ကျွန်ုပ်တို့သည် အမြောက်အမြား စမ်းသပ်မှုများကို လုပ်ဆောင်ကြသည်။ စမ်းသပ်မှုတစ်ခုစီတွင် ကျွန်ုပ်တို့သည် ကိန်းပြည့်နှစ်ခုကို random ရွေးချယ်ပြီး ၎င်းတို့၏ GCD သည် 1 ဖြစ်၊ မဖြစ် စစ်ဆေးသည်။ စမ်းသပ်မှု အောင်မြင်သည့် အကြိမ်အရေအတွက် အချိုးအစားသည် ကျွန်ုပ်တို့အား 6 / π 2 ၏ ခန့်မှန်းချက်ကို ပေးစွမ်းပြီး၊ ယင်းမှတစ်ဆင့် π ၏ အနီးစပ်ဆုံးတန်ဖိုးကို ရရှိသည်။

ကျွန်ုပ်တို့၏ program ၏ အဓိက အစိတ်အပိုင်းမှာ monte-carlo procedure ဖြစ်ပြီး၊ ၎င်းသည် စမ်းသပ်မှု ပြုလုပ်ရမည့် အကြိမ်အရေအတွက်နှင့်အတူ run သည့်အခါတိုင်း true သို့မဟုတ် false ပြန်ပေးမည့် no-argument procedure အဖြစ် ဖော်ပြထားသော စမ်းသပ်ချက်ကို argument များအဖြစ် လက်ခံပါသည်။ Monte-carlo သည် သတ်မှတ်ထားသော စမ်းသပ်မှု အကြိမ်ရေအတိုင်း လုပ်ဆောင်ပြီး စမ်းသပ်ချက် true ဖြစ်ခဲ့သည့် အကြိမ်အရေအတွက် အချိုးအစားကို return ပြန်ပေးပါသည်။ 

(define (estimate-pi trials)
  (sqrt (/ 6 (monte-carlo trials 
                          cesaro-test))))
(define (cesaro-test)
   (= (gcd (rand) (rand)) 1))

(define (monte-carlo trials experiment)
  (define (iter trials-remaining trials-passed)
    (cond ((= trials-remaining 0)
           (/ trials-passed trials))
          ((experiment)
           (iter (- trials-remaining 1) 
                 (+ trials-passed 1)))
          (else
           (iter (- trials-remaining 1) 
                 trials-passed))))
  (iter trials 0))

ယခု၊ local state ကို ပုံဖော်ရန် assignment ကို အသုံးမပြုခဲ့ပါက ကျွန်ုပ်တို့ လုပ်ဆောင်ရမည့် နည်းလမ်းအတိုင်း rand အစား rand-update ကို တိုက်ရိုက်အသုံးပြု၍ အဆိုပါ တွက်ချက်မှုကို ပြုလုပ်ကြည့်ကြပါစို့။ 

(define (estimate-pi trials)
  (sqrt (/ 6 (random-gcd-test trials 
                              random-init))))

(define (random-gcd-test trials initial-x)
  (define (iter trials-remaining 
                trials-passed 
                x)
    (let ((x1 (rand-update x)))
      (let ((x2 (rand-update x1)))
        (cond ((= trials-remaining 0)
               (/ trials-passed trials))
              ((= (gcd x1 x2) 1)
               (iter (- trials-remaining 1)
                     (+ trials-passed 1)
                     x2))
              (else
               (iter (- trials-remaining 1)
                     trials-passed
                     x2))))))
  (iter trials 0 initial-x))

Program သည် ရိုးရှင်းနေသေးသော်လည်း modularity ဆိုင်ရာ ပြင်းထန်သော ချိုးဖောက်မှုအချို့ကို တွေ့ရှိရသည်။ ကျွန်ုပ်တို့၏ ပထမ version program တွင် rand ကို အသုံးပြုခြင်းဖြင့် စိတ်ကြိုက် experiment procedure ကို argument အဖြစ် လက်ခံသည့် ယေဘုယျ monte-carlo procedure တစ်ခုအဖြစ် Monte Carlo နည်းလမ်းကို တိုက်ရိုက် ဖော်ပြနိုင်ပါသည်။ ကျွန်ုပ်တို့၏ ဒုတိယ version program တွင် random-number generator အတွက် local state မရှိသောကြောင့် random-gcd-test သည် random numbers များဖြစ်သော x1 နှင့် x2 တို့ကို အတိအလင်း ကိုင်တွယ်ရပြီး x2 ကို rand-update ၏ input အသစ်အဖြစ် iterative loop မှတစ်ဆင့် ပြန်လည် အသုံးပြုရပါသည်။ ဤကဲ့သို့ random numbers များကို အတိအလင်း ကိုင်တွယ်ရခြင်းသည် စမ်းသပ်ချက် ရလဒ်များကို စုဆောင်းသည့် တည်ဆောက်ပုံနှင့် ကျွန်ုပ်တို့၏ စမ်းသပ်ချက်တွင် random number နှစ်ခု အသုံးပြုထားသည်ဟူသော အချက်ကို ရောထွေးသွားစေသည် (အခြား Monte Carlo စမ်းသပ်ချက်များတွင် random number တစ်ခု သို့မဟုတ် သုံးခု အသုံးပြုကောင်း အသုံးပြုနိုင်သော်လည်း)။ အမြင့်ဆုံးအဆင့်ရှိ estimate-pi procedure ပင်လျှင် အစပျိုး random number တစ်ခု ပေးပို့ရန်အတွက်ပါ ထည့်သွင်းစဉ်းစားနေရပါသည်။ Random-number generator ၏ အတွင်းပိုင်း အစိတ်အပိုင်းများသည် program ၏ အခြား အစိတ်အပိုင်းများသို့ ပျံ့နှံ့ရောက်ရှိနေခြင်းကြောင့် Monte Carlo အယူအဆကို အခြား လုပ်ဆောင်ချက်များတွင် အသုံးပြုနိုင်ရန် သီးခြားခွဲထုတ်ရန် ခက်ခဲစေပါသည်။ ပထမ version program တွင်မူ assignment သည် random-number generator ၏ state ကို rand procedure အတွင်း၌ encapsulate ပြုလုပ်ထားသောကြောင့် random-number ထုတ်လုပ်ခြင်းဆိုင်ရာ အသေးစိတ် အချက်အလက်များသည် program ၏ အခြားအစိတ်အပိုင်းများနှင့် ကင်းလွတ်နေပါသည်။

Monte Carlo ဥပမာမှ သရုပ်ဖော်ထားသော ယေဘုယျဖြစ်ရပ်မှာ ဤသို့ဖြစ်သည်- ရှုပ်ထွေးသော လုပ်ငန်းစဉ်တစ်ခု၏ အစိတ်အပိုင်းတစ်ခု၏ အမြင်မှကြည့်လျှင် အခြားအစိတ်အပိုင်းများသည် အချိန်နှင့်အမျှ ပြောင်းလဲနေပုံပေါ်သည်။ ၎င်းတို့တွင် ဝှက်ထားသော အချိန်နှင့်အမျှ ပြောင်းလဲနေသည့် local state များ ရှိကြသည်။ ဤခွဲခြမ်းစိပ်ဖြာမှုကို ထင်ဟပ်စေသော computer programs များကို ရေးသားလိုပါက၊ ကျွန်ုပ်တို့သည် အချိန်နှင့်အမျှ အပြုအမူ ပြောင်းလဲသည့် computational objects များ (ဥပမာ- bank accounts များနှင့် random-number generators များ) ကို ပြုလုပ်ကြသည်။ ကျွန်ုပ်တို့သည် state ကို local state variables များဖြင့် ပုံဖော်ပြီး၊ state ပြောင်းလဲမှုများကို ထို variables များသို့ assignment ပြုလုပ်ခြင်းဖြင့် ပုံဖော်ကြသည်။

Assignment နှင့် state ကို local variables များတွင် ဝှက်ထားသော နည်းစနစ်ကို အသုံးပြုခြင်းဖြင့်၊ state အားလုံးကို အပို parameter များ ပေးပို့ကာ အတိအလင်း ကိုင်တွယ်ရခြင်းထက် ပိုမိုကောင်းမွန်သော modular ပုံစံဖြင့် စနစ်များကို တည်ဆောက်နိုင်သည်ဟု ပြောဆိုကာ ဤဆွေးနွေးမှုကို အဆုံးသတ်ချင်စရာ ဖြစ်ပါသည်။ ကံမကောင်းစွာဖြင့်၊ ကျွန်ုပ်တို့ မြင်တွေ့ရမည့်အတိုင်း ၎င်းမှာ ထိုမျှမရိုးရှင်းပါ။

Exercise 3.5: Monte Carlo integration သည် Monte Carlo simulation အသုံးပြု၍ definite integrals များကို ခန့်မှန်းတွက်ချက်သည့် နည်းလမ်းတစ်ခု ဖြစ်ပါသည်။ နယ်ပယ်အတွင်းရှိ (x, y) အမှတ်များအတွက် true ဖြစ်ပြီး နယ်ပယ်ပြင်ပရှိ အမှတ်များအတွက် false ဖြစ်သော predicate P ( x , y ) ဖြင့် ဖော်ပြထားသော ဟင်းလင်းပြင် (space) ၏ ဧရိယာကို တွက်ချက်ရန် စဉ်းစားပါ။ ဥပမာအားဖြင့်၊ (5, 7) ကို ဗဟိုပြု၍ အချင်းဝက် (radius) 3 ရှိသော စက်ဝိုင်းအတွင်းရှိ နယ်ပယ်ကို ( x − 5 ) 2 + ( y − 7 ) 2 ≤ 3 2 ဟုတ်မဟုတ် စစ်ဆေးသော predicate ဖြင့် ဖော်ပြနိုင်ပါသည်။ ထိုသို့သော predicate ဖြင့် ဖော်ပြထားသော နယ်ပယ်၏ ဧရိယာကို ခန့်မှန်းရန်၊ ထိုနယ်ပယ် ပါဝင်နေသော စတုဂံပုံ (rectangle) တစ်ခုကို ရွေးချယ်ခြင်းဖြင့် စတင်ပါ။ ဥပမာအားဖြင့်၊ (2, 4) နှင့် (8, 10) တို့တွင် ထောင့်ဖြတ်ဆန့်ကျင်ဘက် ထောင့်များရှိသော စတုဂံသည် အထက်ပါ စက်ဝိုင်းကို ငုံမိပါသည်။ အလိုရှိသော integral မှာ ထိုစတုဂံ၏ နယ်ပယ်အတွင်း ကျရောက်နေသော အစိတ်အပိုင်း၏ ဧရိယာ ဖြစ်ပါသည်။ ကျွန်ုပ်တို့သည် စတုဂံအတွင်းရှိ (x, y) အမှတ်များကို random ရွေးချယ်ပြီး အမှတ်တစ်ခုစီအတွက် နယ်ပယ်အတွင်း ကျရောက်ခြင်း ရှိ၊ မရှိ သိနိုင်ရန် P ( x , y ) ကို စစ်ဆေးခြင်းဖြင့် integral ကို ခန့်မှန်းနိုင်ပါသည်။ အမှတ်များစွာဖြင့် စမ်းသပ်ကြည့်ပါက၊ နယ်ပယ်အတွင်း ကျရောက်သော အမှတ်အရေအတွက် အချိုးအစားသည် စတုဂံ၏ မည်မျှသော အစိတ်အပိုင်းမှာ နယ်ပယ်အတွင်း ရှိနေသည်ဟူသော ခန့်မှန်းချက်ကို ပေးပါလိမ့်မည်။ ထို့ကြောင့်၊ ထိုအချိုးအစားကို စတုဂံတစ်ခုလုံး၏ ဧရိယာနှင့် မြှောက်ခြင်းဖြင့် integral ၏ ခန့်မှန်းခြေတန်ဖိုးကို ရရှိမည်ဖြစ်သည်။

Monte Carlo integration ကို predicate P၊ စတုဂံအတွက် upper နှင့် lower bounds များဖြစ်သော x1၊ x2၊ y1၊ y2 နှင့် ခန့်မှန်းချက်ရရှိရန် စမ်းသပ်ရမည့် အကြိမ်အရေအတွက်တို့ကို argument များအဖြစ် လက်ခံသော estimate-integral procedure အဖြစ် အကောင်အထည်ဖော်ပါ။ သင်၏ procedure သည် π ကို ခန့်မှန်းရန် အထက်တွင် အသုံးပြုခဲ့သော monte-carlo procedure ကိုပင် အသုံးပြုရမည်။ ယူနစ်စက်ဝိုင်း (unit circle) ၏ ဧရိယာကို တိုင်းတာခြင်းဖြင့် π ၏ ခန့်မှန်းခြေတန်ဖိုးကို ထုတ်ယူရန် သင်၏ estimate-integral ကို အသုံးပြုပါ။

ပေးထားသော range တစ်ခုအတွင်းမှ random ရွေးချယ်ထားသော ကိန်းဂဏန်းတစ်ခုကို return ပြန်ပေးသည့် procedure တစ်ခုရှိခြင်းသည် အသုံးဝင်ပါလိမ့်မည်။ အောက်ပါ random-in-range procedure သည် input ထက်ငယ်သော အနုတ်မဟုတ်သည့် ကိန်းဂဏန်းကို return ပြန်ပေးသည့် 1.2.6 တွင် အသုံးပြုခဲ့သော random procedure ကို အသုံးပြု၍ ဤအချက်ကို အကောင်အထည်ဖော်ထားပါသည်။136 

(define (random-in-range low high)
  (let ((range (- high low)))

ကျွန်ုပ်တို့ တွေ့မြင်ခဲ့ရသည့်အတိုင်း set! operation သည် ကျွန်ုပ်တို့အား local state ရှိသော object များကို model လုပ်ရန် အထောက်အကူပြုသည်။ သို့သော်၊ ဤအားသာချက်တွင် ပေးဆပ်ရမည့် တန်ဖိုးတစ်ခုရှိသည်။ ကျွန်ုပ်တို့၏ programming language ကို 1.1.5 တွင် မိတ်ဆက်ခဲ့သည့် procedure application ၏ substitution model အနေဖြင့် ထပ်မံ အဓိပ္ပာယ်ဖွင့်ဆို၍ မရတော့ပေ။ ထို့အပြင်၊ "ကောင်းမွန်သော" mathematical ဂုဏ်သတ္တိများရှိသည့် မည်သည့်ရိုးရှင်းသော model မျှ programming language များရှိ object များနှင့် assignment များကို ကိုင်တွယ်ဖြေရှင်းရန် လုံလောက်သော framework တစ်ခု မဖြစ်နိုင်တော့ပေ။

ကျွန်ုပ်တို့သည် assignment များကို အသုံးမပြုသရွေ့၊ တူညီသော argument များဖြင့် တူညီသော procedure ကို နှစ်ကြိမ် evaluation ပြုလုပ်ပါက တူညီသော ရလဒ်ကိုသာ ထွက်ပေါ်စေမည်ဖြစ်သည်၊ ထို့ကြောင့် procedure များကို mathematical function များကို တွက်ချက်ခြင်းအဖြစ် ရှုမြင်နိုင်သည်။ ဤစာအုပ်၏ ပထမ အခန်းနှစ်ခန်းလုံးတွင် ကျွန်ုပ်တို့ လုပ်ဆောင်ခဲ့သကဲ့သို့ assignment တစ်ခုမျှ အသုံးမပြုဘဲ programming ရေးသားခြင်းကို functional programming ဟု ခေါ်ဆိုပါသည်။

Assignment သည် ကိစ္စရပ်များကို မည်သို့ ရှုပ်ထွေးစေသည်ကို နားလည်စေရန်၊ 3.1.1 ရှိ make-withdraw procedure ၏ လက်ကျန်ငွေ မလုံလောက်မှုကို စစ်ဆေးရန် မလိုအပ်သော ရိုးရှင်းသည့် version တစ်ခုကို စဉ်းစားကြည့်ပါ: 

(define (make-simplified-withdraw balance)
  (lambda (amount)
    (set! balance (- balance amount))
    balance))

ဤ procedure ကို set! အသုံးမပြုသော အောက်ပါ make-decrementer procedure နှင့် နှိုင်းယှဉ်ကြည့်ပါ: 

(define (make-decrementer balance)
  (lambda (amount)
    (- balance amount)))

Make-decrementer သည် သတ်မှတ်ထားသော ပမာဏ balance ထဲမှ ၎င်း၏ input ကို နုတ်ပေးသော procedure တစ်ခုကို return ပြန်ပေးသည်၊ သို့သော် make-simplified-withdraw တွင်ကဲ့သို့ တစ်ကြိမ်ပြီးတစ်ကြိမ် ခေါ်ဆိုမှုများအပေါ် စုဆောင်းမိသော သက်ရောက်မှု (accumulated effect) မရှိပါ:

ကျွန်ုပ်တို့သည် ဦးစွာ make-decrementer ၏ body ထဲရှိ balance နေရာတွင် 25 ဖြင့် အစားထိုးခြင်းအားဖြင့် combination ၏ operator ကို ရိုးရှင်းအောင် ပြုလုပ်သည်။ ၎င်းက expression ကို အောက်ပါအတိုင်း လျှော့ချပေးသည်

((make-simplified-withdraw 25) 20)

ကျွန်ုပ်တို့သည် ဦးစွာ make-simplified-withdraw ၏ body ထဲရှိ balance နေရာတွင် 25 ဖြင့် အစားထိုးခြင်းအားဖြင့် operator ကို ရိုးရှင်းအောင် ပြုလုပ်သည်။ ၎င်းက expression ကို အောက်ပါအတိုင်း လျှော့ချပေးသည်137

အကယ်၍ ကျွန်ုပ်တို့သည် substitution model ကိုသာ လိုက်နာခဲ့ပါက၊ procedure application ၏ အဓိပ္ပာယ်မှာ ဦးစွာ balance ကို 5 အဖြစ် သတ်မှတ်ပြီး၊ ထို့နောက် expression ၏ တန်ဖိုးအဖြစ် 25 ကို return ပြန်ပေးခြင်း ဖြစ်သည်ဟု ဆိုရမည်ဖြစ်သည်။ ၎င်းသည် မှားယွင်းသော အဖြေကို ရရှိစေသည်။ မှန်ကန်သော အဖြေကို ရရှိရန်အတွက်၊ ကျွန်ုပ်တို့သည် ပထမအကြိမ် ပေါ်ပေါက်သော balance (set! ၏ သက်ရောက်မှု မရှိမီ) နှင့် ဒုတိယအကြိမ် ပေါ်ပေါက်သော balance (set! ၏ သက်ရောက်မှု ရှိပြီးနောက်) တို့ကို တစ်နည်းနည်းဖြင့် ခွဲခြားရန် လိုအပ်မည်ဖြစ်သော်လည်း substitution model သည် ယင်းကို မလုပ်ဆောင်နိုင်ပါ။

ဤနေရာတွင် အခက်အခဲမှာ substitution သည် ကျွန်ုပ်တို့၏ language ထဲရှိ symbol များသည် အခြေခံအားဖြင့် တန်ဖိုးများအတွက် အမည်များသာ ဖြစ်သည်ဟူသော အယူအဆအပေါ်တွင် အဆုံးစွန်အထိ အခြေခံထားခြင်း ဖြစ်သည်။ သို့သော် ကျွန်ုပ်တို့သည် set! နှင့် variable တစ်ခု၏ တန်ဖိုး ပြောင်းလဲနိုင်သည်ဟူသော အယူအဆကို မိတ်ဆက်လိုက်သည်နှင့် တပြိုင်နက်၊ variable တစ်ခုသည် ရိုးရှင်းသော အမည်တစ်ခုအဖြစ်သာ မရှိနိုင်တော့ပေ။ ယခုအခါ variable တစ်ခုသည် တန်ဖိုးတစ်ခုကို သိမ်းဆည်းထားနိုင်သည့် "နေရာ" တစ်ခုကို တစ်နည်းနည်းဖြင့် ရည်ညွှန်းပြီး၊ ထိုနေရာတွင် သိမ်းဆည်းထားသော တန်ဖိုးသည် ပြောင်းလဲနိုင်သည်။ 3.2 တွင် ကျွန်ုပ်တို့၏ computational model ၌ environment များသည် ဤ "နေရာ" (place) ၏ အခန်းကဏ္ဍတွင် မည်သို့ ပါဝင်သည်ကို တွေ့မြင်ရမည်ဖြစ်သည်။

တူညီမှုနှင့် ပြောင်းလဲမှု

ဤနေရာတွင် ပေါ်ပေါက်လာသော ကိစ္စရပ်သည် သီးခြား computation model တစ်ခု ပျက်ပြားသွားခြင်းထက် ပို၍ နက်ရှိုင်းသည်။ ကျွန်ုပ်တို့၏ computational model များထဲသို့ အပြောင်းအလဲ (change) ကို မိတ်ဆက်လိုက်သည်နှင့် တပြိုင်နက်၊ ယခင်က ရှင်းလင်းလှသော အယူအဆ အများအပြားသည် ပြဿနာရှိလာသည်။ အရာဝတ္ထု နှစ်ခုသည် "တူညီသည်" ဟူသော အယူအဆကို စဉ်းစားကြည့်ပါ။

တူညီသော argument ဖြင့် make-decrementer ကို နှစ်ကြိမ်ခေါ်ဆိုပြီး procedure နှစ်ခု ဖန်တီးသည်ဟု ယူဆပါစို့: 

(define D1 (make-decrementer 25))
(define D2 (make-decrementer 25))

D1 နှင့် D2 တို့သည် တူညီပါသလား? လက်ခံနိုင်သော အဖြေမှာ "ဟုတ်ကဲ့" ဖြစ်သည်၊ အကြောင်းမှာ D1 နှင့် D2 တို့သည် တူညီသော computational behavior ရှိသောကြောင့်ဖြစ်သည်—၎င်းတို့ တစ်ခုချင်းစီသည် ၎င်း၏ input ကို 25 ထဲမှ နုတ်ပေးသော procedure များ ဖြစ်ကြသည်။ အမှန်စင်စစ်၊ မည်သည့် computation တွင်မဆို ရလဒ်ကို မပြောင်းလဲစေဘဲ D1 ကို D2 နေရာတွင် အစားထိုးနိုင်သည်။

(define W1 (make-simplified-withdraw 25))

W1 နှင့် W2 တို့သည် တူညီသော expression ဖြစ်သည့် (make-simplified-withdraw 25) ကို evaluate လုပ်ခြင်းဖြင့် ဖန်တီးထားခြင်းဖြစ်သောကြောင့် "တူညီသည်" (equal) ဟု ဆိုနိုင်သော်လည်း၊ မည်သည့် expression တွင်မဆို evaluate လုပ်သည့် ရလဒ်ကို မပြောင်းလဲစေဘဲ W1 ကို W2 နေရာတွင် အစားထိုးနိုင်သည်ဆိုခြင်းမှာ မမှန်ကန်ပါ။

Expression တစ်ခု၏ တန်ဖိုးကို မပြောင်းလဲစေဘဲ "တူညီသောအရာများကို တူညီသောအရာများဖြင့် အစားထိုးနိုင်သည်" ဟူသော အယူအဆကို ထောက်ပံ့သော language တစ်ခုကို referentially transparent ဖြစ်သည်ဟု ဆိုသည်။ ကျွန်ုပ်တို့၏ computer language တွင် set! ကို ထည့်သွင်းလိုက်သောအခါ referential transparency သည် ပျက်ပြားသွားသည်။ ၎င်းသည် တူညီသော expression များဖြင့် အစားထိုးခြင်းအားဖြင့် expression များကို မည်သည့်အချိန်တွင် ရိုးရှင်းအောင် ပြုလုပ်နိုင်သည်ကို ဆုံးဖြတ်ရန် ခက်ခဲစေသည်။ ရလဒ်အနေဖြင့်၊ assignment ကို အသုံးပြုသော program များအကြောင်း ဆင်ခြင်တွေးတောခြင်း (reasoning) သည် သိသိသာသာ ပိုမိုခက်ခဲလာသည်။

ကျွန်ုပ်တို့သည် referential transparency ကို စွန့်လွှတ်လိုက်သည်နှင့် တပြိုင်နက်၊ computational object များ "တူညီခြင်း" ဟူသည်မှာ အဘယ်နည်းဆိုသော အယူအဆကို ပုံသေနည်းလမ်းဖြင့် ဖော်ပြရန် ခက်ခဲလာသည်။ အမှန်စင်စစ်၊ ကျွန်ုပ်တို့၏ program များက model ပြုလုပ်သော လက်တွေ့ကမ္ဘာရှိ "တူညီခြင်း" ၏ အဓိပ္ပာယ်မှာပင် ရှင်းလင်းမှု မရှိလှပေ။ ယေဘုယျအားဖြင့်၊ မြင်တွေ့ရသော အရာဝတ္ထုနှစ်ခုသည် အမှန်တကယ်ပင် "တူညီသော တစ်ခုတည်း" ဖြစ်ကြောင်းကို အရာဝတ္ထုတစ်ခုအား ပြုပြင်ပြောင်းလဲကြည့်ပြီး အခြားအရာဝတ္ထုသည်လည်း ထိုနည်းအတိုင်း ပြောင်းလဲသွားခြင်း ရှိမရှိ စောင့်ကြည့်ခြင်းဖြင့်သာ ဆုံးဖြတ်နိုင်သည်။ သို့သော် အရာဝတ္ထုတစ်ခု "ပြောင်းလဲသွားခြင်း" ကို "တူညီသော" အရာဝတ္ထုအား နှစ်ကြိမ် စောင့်ကြည့်ပြီး အရာဝတ္ထု၏ ဂုဏ်သတ္တိအချို့သည် တစ်ကြိမ်နှင့်တစ်ကြိမ် ကွဲပြားမှု ရှိမရှိကို ကြည့်ခြင်းမှလွဲ၍ အခြားမည်သို့ ပြောနိုင်မည်နည်း? ထို့ကြောင့် ကျွန်ုပ်တို့သည် "တူညီမှု" ဟူသော ကြိုတင်အယူအဆ (a priori notion) မရှိဘဲ "ပြောင်းလဲမှု" ကို ဆုံးဖြတ်၍ မရနိုင်သလို၊ ပြောင်းလဲမှု၏ သက်ရောက်မှုများကို စောင့်ကြည့်ခြင်း မရှိဘဲ တူညီမှုကိုလည်း ဆုံးဖြတ်၍ မရနိုင်ပါ။

ဤကိစ္စရပ်သည် programming တွင် မည်သို့ ပေါ်ပေါက်လာသည်ကို ဥပမာအနေဖြင့်၊ Peter နှင့် Paul တို့တွင် $100 ရှိသော ဘဏ်အကောင့်တစ်ခု ရှိသည်ဟူသော အခြေအနေကို စဉ်းစားကြည့်ပါ။ ယင်းကို အောက်ပါအတိုင်း model ပြုလုပ်ခြင်းနှင့် 

(define peter-acc (make-account 100))
(define paul-acc (make-account 100))

အောက်ပါအတိုင်း model ပြုလုပ်ခြင်းတို့အကြား ကြီးမားသော ခြားနားချက် ရှိသည်: 

(define peter-acc (make-account 100))
(define paul-acc peter-acc)

ပထမအခြေအနေတွင်၊ ဘဏ်အကောင့်နှစ်ခုသည် သီးခြားစီဖြစ်သည်။ Peter ပြုလုပ်သော ငွေကြေးအရောင်းအဝယ်များသည် Paul ၏ အကောင့်ကို ထိခိုက်မည်မဟုတ်သလို၊ Paul ပြုလုပ်သည့်အရာသည်လည်း Peter ၏ အကောင့်ကို ထိခိုက်မည်မဟုတ်ပါ။ သို့သော် ဒုတိယအခြေအနေတွင်၊ ကျွန်ုပ်တို့သည် paul-acc ကို peter-acc နှင့် တစ်ခုတည်းသော အရာ အဖြစ် သတ်မှတ်ခဲ့သည်။ ရလဒ်အားဖြင့်၊ Peter နှင့် Paul တို့သည် ယခုအခါ ပူးတွဲဘဏ်အကောင့်တစ်ခု ရှိသွားပြီဖြစ်ပြီး၊ အကယ်၍ Peter က peter-acc မှ ငွေထုတ်ယူပါက Paul သည် paul-acc တွင် ငွေနည်းသွားသည်ကို တွေ့ရမည်ဖြစ်သည်။ ဤသို့ ဆင်တူသော်လည်း ကွဲပြားသော အခြေအနေနှစ်ခုသည် computational model များ တည်ဆောက်ရာတွင် ရှုပ်ထွေးမှုများ ဖြစ်ပေါ်စေနိုင်သည်။ အထူးသဖြင့် မျှဝေအသုံးပြုသော အကောင့် (shared account) တွင်၊ အရာဝတ္ထုတစ်ခုတည်း (ဘဏ်အကောင့်) ကို အမည်နှစ်ခု (peter-acc နှင့် paul-acc) ပေးထားခြင်းက ပို၍ ရှုပ်ထွေးစေနိုင်သည်။ အကယ်၍ ကျွန်ုပ်တို့သည် paul-acc ပြောင်းလဲနိုင်သော နေရာအားလုံးကို program ထဲတွင် ရှာဖွေနေပါက၊ peter-acc ကို ပြောင်းလဲစေသော အရာများကိုလည်း ရှာဖွေရန် သတိရရမည်ဖြစ်သည်။138

အထက်ဖော်ပြပါ "တူညီမှု" နှင့် "ပြောင်းလဲမှု" ဆိုင်ရာ မှတ်ချက်များကို ရည်ညွှန်းရလျှင်၊ အကယ်၍ Peter နှင့် Paul တို့သည် ၎င်းတို့၏ ဘဏ်လက်ကျန်ငွေများကိုသာ စစ်ဆေးနိုင်ပြီး လက်ကျန်ငွေကို ပြောင်းလဲစေသော လုပ်ဆောင်ချက်များကို မလုပ်ဆောင်နိုင်ပါက၊ အကောင့်နှစ်ခုသည် သီးခြားဖြစ်မဖြစ် ဆိုသည့်ကိစ္စမှာ အငြင်းပွားဖွယ် မဟုတ်တော့ပေ။ ယေဘုယျအားဖြင့်၊ ကျွန်ုပ်တို့သည် data object များကို ဘယ်တော့မှ မပြုပြင်မပြောင်းလဲသရွေ့၊ compound data object တစ်ခုကို ၎င်း၏ အစိတ်အပိုင်းအားလုံး ပေါင်းစည်းထားခြင်းဟု တိတိကျကျ မှတ်ယူနိုင်သည်။ ဥပမာအားဖြင့်၊ rational number တစ်ခုကို ၎င်း၏ numerator နှင့် denominator ပေးခြင်းဖြင့် သတ်မှတ်သည်။ သို့သော် အပြောင်းအလဲများ ရှိလာသောအခါ ဤအမြင်သည် မမှန်ကန်တော့ပေ၊ အကြောင်းမှာ compound data object တစ်ခုတွင် ၎င်းကို ဖွဲ့စည်းထားသော အစိတ်အပိုင်းများနှင့် ကွဲပြားသော "identity" (ကိုယ်ပိုင်လက္ခဏာ) ရှိလာသောကြောင့် ဖြစ်သည်။ ငွေထုတ်ယူခြင်းဖြင့် လက်ကျန်ငွေကို ပြောင်းလဲလိုက်သော်လည်း ဘဏ်အကောင့်သည် "တူညီသော" ဘဏ်အကောင့်အဖြစ်ပင် ရှိနေသေးသည်။ ပြောင်းပြန်အားဖြင့်၊ တူညီသော state information ရှိသည့် မတူညီသော ဘဏ်အကောင့်နှစ်ခုလည်း ရှိနိုင်သည်။ ဤရှုပ်ထွေးမှုသည် ကျွန်ုပ်တို့၏ programming language ကြောင့် မဟုတ်ဘဲ၊ ဘဏ်အကောင့်တစ်ခုကို object တစ်ခုအဖြစ် ကျွန်ုပ်တို့၏ ရှုမြင်ပုံကြောင့် ဖြစ်ပေါ်လာခြင်း ဖြစ်သည်။ ဥပမာအားဖြင့်၊ ကျွန်ုပ်တို့သည် ပုံမှန်အားဖြင့် rational number တစ်ခုကို numerator ပြောင်းလဲသွားသော်လည်း "တူညီသော" rational number တစ်ခုအဖြစ် တည်ရှိနေမည့် identity ရှိသော ပြောင်းလဲနိုင်သည့် object တစ်ခုအဖြစ် မသတ်မှတ်ကြပါ။

Imperative programming ၏ အန္တရာယ်ရှိသော ထောင်ချောက်များ

Functional programming နှင့် ဆန့်ကျင်ဘက်အားဖြင့်၊ assignment များကို အလွန်အကျွံ အသုံးပြုသော programming ကို imperative programming ဟု ခေါ်ဆိုသည်။ Computational model များအကြောင်း ရှုပ်ထွေးမှုများကို မြင့်တက်စေသည့်အပြင်၊ imperative style ဖြင့် ရေးသားထားသော program များသည် functional program များတွင် မဖြစ်ပေါ်နိုင်သော bug များ ဖြစ်ပွားရန် ပိုမိုလွယ်ကူသည်။ ဥပမာအားဖြင့်၊ 1.2.1 မှ iterative factorial program ကို ပြန်လည် အမှတ်ရကြည့်ပါ: 

(define (factorial n)
  (define (iter product counter)
    (if (> counter n)
        product
        (iter (* counter product)
              (+ counter 1))))
  (iter 1 1))

Internal iterative loop ထဲတွင် argument များ ပေးပို့မည့်အစား၊ product နှင့် counter variable များ၏ တန်ဖိုးများကို update လုပ်ရန် တိကျသော (explicit) assignment ကို အသုံးပြုခြင်းဖြင့် ပို၍ imperative ဆန်သော style ကို အသုံးပြုနိုင်သည်: 

(define (factorial n)
  (let ((product 1)
        (counter 1))
    (define (iter)
      (if (> counter n)
          product
          (begin (set! product (* counter 
                                  product))
                 (set! counter (+ counter 1))
                 (iter))))
    (iter)))

၎င်းသည် program မှ ထုတ်ပေးသော ရလဒ်များကို မပြောင်းလဲစေသော်လည်း၊ သိမ်မွေ့သော ထောင်ချောက်တစ်ခုကို မိတ်ဆက်ပေးသည်။ Assignment များ၏ အစီအစဉ်ကို ကျွန်ုပ်တို့ မည်သို့ ဆုံးဖြတ်မည်နည်း? လက်ရှိ ရေးသားထားသည့်အတိုင်းဆိုလျှင် program သည် မှန်ကန်ပါသည်။ သို့သော် assignment များကို ဆန့်ကျင်ဘက် အစီအစဉ်ဖြင့် ရေးသားပါက 

(set! counter (+ counter 1))
(set! product (* counter product))

138 Computational object တစ်ခုတည်းကို အမည်တစ်ခုထက်ပို၍ ခေါ်ဆိုအသုံးပြုခြင်းကို aliasing ဟု ခေါ်သည်။ ပူးတွဲဘဏ်အကောင့် အခြေအနေသည် alias ၏ အလွန်ရိုးရှင်းသော ဥပမာတစ်ခုကို သရုပ်ဖော်သည်။ 3.3 တွင် အစိတ်အပိုင်းအချို့ကို မျှဝေအသုံးပြုထားသည့် “distinct” compound data structures များကဲ့သို့ ပိုမိုရှုပ်ထွေးသော ဥပမာများကို မြင်တွေ့ရမည်ဖြစ်သည်။ “ကွဲပြားသော” object နှစ်ခုသည် အမှန်တကယ်တွင် မတူညီသော alias များအောက်ရှိ object တစ်ခုတည်းဖြစ်နေသဖြင့် object တစ်ခုအား ပြောင်းလဲခြင်းသည် “side effect” အနေဖြင့် “အခြား” object တစ်ခုကိုလည်း ပြောင်းလဲစေနိုင်ကြောင်း ကျွန်ုပ်တို့ မေ့လျော့သွားပါက ပရိုဂရမ်များတွင် bug များ ဖြစ်ပေါ်လာနိုင်သည်။ ဤသို့ခေါ်ဆိုသော side-effect bugs များသည် ရှာဖွေရန်နှင့် ခွဲခြမ်းစိတ်ဖြာရန် အလွန်ခက်ခဲသောကြောင့် အချို့သောသူများက programming languages များကို side effects သို့မဟုတ် aliasing များကို ခွင့်မပြုသော ပုံစံဖြင့် ဒီဇိုင်းထုတ်ရန် အကြံပြုခဲ့ကြသည် (Lampson et al. 1981; Morris et al. 1980)။

139 ဤအချက်ကို ကြည့်ခြင်းအားဖြင့်၊ အခြေခံ programming ကို imperative style ဖြင့် အများဆုံး သင်ကြားလေ့ရှိခြင်းမှာ သရော်စာဆန်လှသည်။ ဒါဟာ ၁၉၆၀ နဲ့ ၁၉၇၀ ခုနှစ်တွေတုန်းက အသုံးများခဲ့တဲ့၊ procedure တွေကို ခေါ်ယူတဲ့ ပရိုဂရမ်တွေဟာ assignment တွေကို လုပ်ဆောင်တဲ့ ပရိုဂရမ်တွေထက် ထိရောက်မှု ပိုနည်းရမယ်ဆိုတဲ့ ယုံကြည်ချက်ရဲ့ အကြွင်းအကျန် ဖြစ်နိုင်ပါတယ်။ (Steele 1977 သည် ဤအငြင်းပွားမှုကို ချေဖျက်ခဲ့သည်။) သို့မဟုတ်ပါက တစ်ဆင့်ချင်းစီ assignment လုပ်ခြင်းသည် စတင်လေ့လာသူများအတွက် procedure call ထက် ပိုမိုမြင်သာလွယ်သည်ဟူသော အမြင်ကို ထင်ဟပ်စေခြင်းလည်း ဖြစ်နိုင်သည်။ မည်သည့်အကြောင်းကြောင့်ဖြစ်စေ၊ ၎င်းသည် စတင်လေ့လာသူ ပရိုဂရမ်မာများအား “ဒီ variable ကို ဟိုတစ်ခု မတိုင်ခင် သို့မဟုတ် နောက်မှ set လုပ်သင့်သလား” ဟူသော စိုးရိမ်ပူပန်မှုများကို ပေးလေ့ရှိပြီး ပရိုဂရမ်ရေးသားခြင်းကို ရှုပ်ထွေးစေကာ အရေးကြီးသော အယူအဆများကို ဝေဝါးစေသည်။
