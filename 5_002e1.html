5.1Designing Register Machines

Register machine တစ်ခုကို ပုံစံထုတ်ရန်အတွက်၊ ၎င်း၏ data paths (registers နှင့် operations များ) နှင့် ၎င်း operations များ၏ အစီအစဉ်ကို ထိန်းချုပ်ပေးသော controller တို့ကို ပုံစံထုတ်ရမည်ဖြစ်သည်။ ရိုးရှင်းသော register machine တစ်ခု၏ ပုံစံထုတ်ပုံကို သရုပ်ဖော်ရန်၊ ကိန်းပြည့်နှစ်ခု၏ အကြီးဆုံးဘုံဆားကိန်း (GCD) ကို ရှာဖွေရာတွင် အသုံးပြုသော Euclid’s Algorithm ကို လေ့လာကြည့်ကြပါစို့။ 1.2.5 တွင် ကျွန်ုပ်တို့ မြင်ခဲ့သည့်အတိုင်း၊ Euclid’s Algorithm ကို အောက်ပါ procedure တွင် သတ်မှတ်ထားသည့်အတိုင်း iterative process တစ်ခုဖြင့် လုပ်ဆောင်နိုင်သည်- 

(define (gcd a b)
  (if (= b 0)
      a
      (gcd b (remainder a b))))

ဤ algorithm ကို လုပ်ဆောင်မည့် machine တစ်ခုသည် ကိန်းဂဏန်းနှစ်ခုဖြစ်သော a နှင့် b ကို မှတ်သားထားရမည်ဖြစ်သဖြင့်၊ ထိုကိန်းဂဏန်းများကို ၎င်းအမည်များပေးထားသော register နှစ်ခုတွင် သိမ်းဆည်းထားသည်ဟု ယူဆကြပါစို့။ လိုအပ်သော အခြေခံ operations များမှာ register b ၏ တန်ဖိုးသည် သုည ဟုတ်မဟုတ် စစ်ဆေးခြင်းနှင့် register a ၏ တန်ဖိုးကို register b ၏ တန်ဖိုးနှင့် စား၍ အကြွင်း (remainder) ကို တွက်ချက်ခြင်းတို့ ဖြစ်သည်။ အကြွင်းရှာခြင်း operation သည် ရှုပ်ထွေးသော လုပ်ငန်းစဉ်တစ်ခု ဖြစ်သော်လည်း၊ လောလောဆယ်တွင် အကြွင်းကို တွက်ချက်ပေးနိုင်သော အခြေခံ device တစ်ခု ရှိသည်ဟု ယူဆပါ။ GCD algorithm ၏ အကြိမ်တိုင်းတွင် register a ၏ တန်ဖိုးကို register b ၏ တန်ဖိုးနှင့် အစားထိုးရမည်ဖြစ်ပြီး၊ b ၏ တန်ဖိုးကိုမူ a ၏ တန်ဖိုးဟောင်းကို b ၏ တန်ဖိုးဟောင်းနှင့် စား၍ ရသော အကြွင်းဖြင့် အစားထိုးရမည်ဖြစ်သည်။ ဤအစားထိုးမှုများကို တစ်ပြိုင်နက်တည်း ပြုလုပ်နိုင်လျှင် အဆင်ပြေမည် ဖြစ်သော်လည်း၊ ကျွန်ုပ်တို့၏ register machine model တွင် အဆင့်တစ်ခုချင်းစီ၌ register တစ်ခုကိုသာ တန်ဖိုးအသစ် သတ်မှတ်နိုင်သည်ဟု ယူဆပါမည်။ ဤအစားထိုးမှုများကို ပြီးမြောက်အောင် ဆောင်ရွက်ရန်၊ ကျွန်ုပ်တို့၏ machine သည် t ဟု ခေါ်သော တတိယ "temporary" register တစ်ခုကို အသုံးပြုပါမည်။ (ပထမဦးစွာ အကြွင်းကို t ထဲသို့ ထည့်မည်၊ ထို့နောက် b ၏ တန်ဖိုးကို a ထဲသို့ ထည့်မည်၊ နောက်ဆုံးတွင် t ၌ သိမ်းထားသော အကြွင်းကို b ထဲသို့ ထည့်မည် ဖြစ်သည်။)

ဤ machine အတွက် လိုအပ်သော registers နှင့် operations များကို Figure 5.1 တွင် ပြထားသော data-path diagram ကို အသုံးပြု၍ သရုပ်ဖော်နိုင်သည်။ ဤ diagram တွင် registers (a, b, နှင့် t) များကို စတုဂံပုံများဖြင့် ဖော်ပြထားသည်။ register တစ်ခုသို့ တန်ဖိုးတစ်ခု သတ်မှတ်ပေးသည့် နည်းလမ်းတစ်ခုချင်းစီကို အရင်းအမြစ်မှ register ဆီသို့ ဦးတည်နေပြီး ခေါင်းပိုင်းနောက်တွင် X ပါသော မြှားတစ်ခုဖြင့် ပြထားသည်။ X ကို ခလုတ်တစ်ခုကဲ့သို့ မှတ်ယူနိုင်ပြီး၊ ၎င်းကို နှိပ်လိုက်သောအခါ အရင်းအမြစ်ရှိ တန်ဖိုးကို သတ်မှတ်ထားသော register ထဲသို့ "စီးဝင်" ခွင့်ပြုသည်။ ခလုတ်တစ်ခုချင်းစီ၏ ဘေးတွင် ရှိသော label မှာ ထိုခလုတ်ကို ရည်ညွှန်းရန် ကျွန်ုပ်တို့ အသုံးပြုမည့် အမည်ဖြစ်သည်။ အမည်များကို စိတ်ကြိုက်ပေးနိုင်ပြီး မှတ်မိလွယ်သော တန်ဖိုးများ ဖြစ်စေရန် ရွေးချယ်နိုင်သည် (ဥပမာ၊ a<-b သည် register b ၏ တန်ဖိုးကို register a သို့ သတ်မှတ်ပေးသော ခလုတ်ကို နှိပ်ခြင်းကို ဆိုလိုသည်)။ register တစ်ခုအတွက် data အရင်းအမြစ်သည် အခြား register တစ်ခု (a<-b သတ်မှတ်ချက်ကဲ့သို့)၊ operation တစ်ခု၏ ရလဒ် (t<-r သတ်မှတ်ချက်ကဲ့သို့)၊ သို့မဟုတ် constant (ပြောင်းလဲ၍မရသော ပုံသေတန်ဖိုး၊ data-path diagram တွင် ထိုတန်ဖိုးပါရှိသော တြိဂံပုံဖြင့် ဖော်ပြသည်) တို့ ဖြစ်နိုင်သည်။

Constants နှင့် registers များ၏ တန်ဖိုးများမှ တန်ဖိုးတစ်ခုကို တွက်ချက်ပေးသော operation ကို data-path diagram တွင် operation အမည် ပါဝင်သော တြိဂံစောင်း (trapezoid) ပုံဖြင့် ဖော်ပြသည်။ ဥပမာအားဖြင့်၊ Figure 5.1 ရှိ rem ဟု မှတ်သားထားသော box သည် ၎င်းနှင့် ချိတ်ဆက်ထားသော registers a နှင့် b တို့၏ တန်ဖိုးများမှ အကြွင်းကို တွက်ချက်ပေးသော operation ကို ကိုယ်စားပြုသည်။ input registers နှင့် constants များမှ box ဆီသို့ (ခလုတ်မပါသော) မြှားများ ဦးတည်နေပြီး၊ operation ၏ output တန်ဖိုးကို registers များသို့ မြှားများဖြင့် ချိတ်ဆက်ထားသည်။ Test တစ်ခုကို test အမည် ပါဝင်သော စက်ဝိုင်းတစ်ခုဖြင့် ဖော်ပြသည်။ ဥပမာအားဖြင့်၊ ကျွန်ုပ်တို့၏ GCD machine တွင် register b ၏ တန်ဖိုးသည် သုည ဟုတ်မဟုတ် စစ်ဆေးသော operation တစ်ခု ပါရှိသည်။ Test တစ်ခုတွင်လည်း ၎င်း၏ input registers နှင့် constants များမှ မြှားများ ရှိသော်လည်း၊ output မြှားများ မရှိပါ။ ၎င်း၏ တန်ဖိုးကို data paths များက အသုံးပြုခြင်း မဟုတ်ဘဲ controller က အသုံးပြုခြင်း ဖြစ်သည်။ ခြုံငုံကြည့်လျှင် data-path diagram သည် machine အတွက် လိုအပ်သော registers နှင့် operations များ၊ ၎င်းတို့ကို မည်သို့ ချိတ်ဆက်ထားရမည်ကို ပြသသည်။ မြှားများကို ဝိုင်ယာကြိုးများအဖြစ် လည်းကောင်း၊ X ခလုတ်များကို ခလုတ်များ (switches) အဖြစ် လည်းကောင်း ရှုမြင်ပါက၊ data-path diagram သည် လျှပ်စစ်ပစ္စည်းများဖြင့် တည်ဆောက်နိုင်သော machine တစ်ခု၏ wiring diagram နှင့် အလွန်တူပါသည်။

Data paths များက GCD များကို အမှန်တကယ် တွက်ချက်နိုင်ရန်အတွက်၊ ခလုတ်များကို မှန်ကန်သော အစီအစဉ်အတိုင်း နှိပ်ရပါမည်။ ဤအစီအစဉ်ကို Figure 5.2 တွင် သရုပ်ဖော်ထားသည့်အတိုင်း controller diagram ဖြင့် ဖော်ပြပါမည်။ controller diagram ၏ အစိတ်အပိုင်းများသည် data-path အစိတ်အပိုင်းများကို မည်သို့ အသုံးပြုရမည်ကို ဖော်ပြသည်။ controller diagram ရှိ စတုဂံပုံ box များသည် နှိပ်ရမည့် data-path ခလုတ်များကို ဖော်ပြပြီး၊ မြှားများသည် အဆင့်တစ်ခုမှ တစ်ခုသို့ လုပ်ဆောင်ရမည့် အစီအစဉ်ကို ဖော်ပြသည်။ diagram ရှိ စိန်ပွင့်ပုံ (diamond) သည် ဆုံးဖြတ်ချက်တစ်ခုကို ကိုယ်စားပြုသည်။ စိန်ပွင့်ပုံတွင် ဖော်ပြထားသော data-path test ၏ ရလဒ်အပေါ် မူတည်၍ sequencing arrows နှစ်ခုအနက် တစ်ခုကို လိုက်နာဆောင်ရွက်မည်ဖြစ်သည်။ Controller ကို ရုပ်ပိုင်းဆိုင်ရာ ပုံပမာတစ်ခုဖြင့် အဓိပ္ပာယ်ဖွင့်ဆိုနိုင်သည်- diagram ကို စကျင်ကျောက်လုံးလေးတစ်ခု လိမ့်နေသည့်ဝင်္ကပါတစ်ခုအဖြစ် စဉ်းစားကြည့်ပါ။ စကျင်ကျောက်လုံးသည် box တစ်ခုထဲသို့ လိမ့်ဝင်သွားသောအခါ၊ ထို box တွင် အမည်ပေးထားသော data-path ခလုတ်ကို နှိပ်လိုက်သည်။ စကျင်ကျောက်လုံးသည် ဆုံးဖြတ်ချက် node (ဥပမာ b = 0 ဟုတ်မဟုတ် စစ်ဆေးခြင်း) သို့ လိမ့်ဝင်သောအခါ၊ သတ်မှတ်ထားသော test ၏ ရလဒ်ပေါ် မူတည်၍ လမ်းကြောင်းမှ ထွက်ခွာသည်။ data paths နှင့် controller တို့ကို ပေါင်းလိုက်ပါက GCD များ တွက်ချက်ရန် machine တစ်ခုကို အပြည့်အစုံ ဖော်ပြပြီးသား ဖြစ်သွားသည်။ ကျွန်ုပ်တို့သည် registers a နှင့် b တွင် ကိန်းဂဏန်းများ ထည့်ပြီးနောက်၊ start ဟု မှတ်သားထားသော နေရာမှ controller (လိမ့်နေသော စကျင်ကျောက်လုံး) ကို စတင်စေသည်။ Controller သည် done သို့ ရောက်ရှိသောအခါ၊ register a တွင် GCD တန်ဖိုးကို တွေ့ရှိရမည်ဖြစ်သည်။

Exercise 5.1: အောက်ပါ procedure တွင် သတ်မှတ်ထားသော iterative algorithm ကို အသုံးပြု၍ factorial တွက်ချက်ရန် register machine တစ်ခုကို ပုံစံထုတ်ပါ။ ဤ machine အတွက် data-path နှင့် controller diagram များကို ဆွဲပါ။ 

(define (factorial n)
  (define (iter product counter)
    (if (> counter n)
        product
        (iter (* counter product)
              (+ counter 1))))
  (iter 1 1))

5.1.1A Language for Describing Register Machines

Data-path နှင့် controller ပုံကြမ်းများသည် GCD ကဲ့သို့ ရိုးရှင်းသော စက်များကို ဖော်ပြရန် လုံလောက်သော်လည်း Lisp interpreter ကဲ့သို့သော စက်ကြီးများကို ဖော်ပြရန်အတွက်မူ ကိုင်တွယ်ရခက်ခဲပါသည်။ ရှုပ်ထွေးသော စက်များကို ကိုင်တွယ်ဖြေရှင်းနိုင်စေရန်အတွက် ကျွန်ုပ်တို့သည် data-path နှင့် controller ပုံကြမ်းများမှ ပေးထားသော အချက်အလက်အားလုံးကို စာသားပုံစံဖြင့် တင်ပြသည့် ဘာသာစကားတစ်ခုကို ဖန်တီးပါမည်။ ပုံကြမ်းများကို တိုက်ရိုက်ထဟပ်စေသော သင်္ကေတစနစ် (notation) တစ်ခုဖြင့် ကျွန်ုပ်တို့ စတင်ပါမည်။

ကျွန်ုပ်တို့သည် register များနှင့် operation များကို ဖော်ပြခြင်းဖြင့် စက်တစ်ခု၏ data path များကို သတ်မှတ်ပါသည်။ register တစ်ခုကို ဖော်ပြရန်အတွက် ၎င်းကို အမည်တစ်ခုပေးပြီး ၎င်းထဲသို့ တန်ဖိုးသတ်မှတ်ခြင်း (assignment) ကို ထိန်းချုပ်သည့် ခလုတ်များကို သတ်မှတ်ပေးရပါမည်။ ဤခလုတ်တစ်ခုချင်းစီကို အမည်ပေးရပြီး ခလုတ်၏ထိန်းချုပ်မှုအောက်တွင် register ထဲသို့ဝင်ရောက်မည့် အချက်အလက်များ၏ အရင်းအမြစ် (source) ကို သတ်မှတ်ပေးရပါမည်။ (အရင်းအမြစ်သည် register တစ်ခု၊ ကိန်းသေ (constant) တစ်ခု သို့မဟုတ် operation တစ်ခု ဖြစ်နိုင်ပါသည်။) operation တစ်ခုကို ဖော်ပြရန်အတွက် ၎င်းကို အမည်ပေးပြီး ၎င်း၏ input များ (register များ သို့မဟုတ် ကိန်းသေများ) ကို သတ်မှတ်ပေးရပါမည်။

ကျွန်ုပ်တို့သည် စက်တစ်ခု၏ controller ကို အစီအစဉ်အတွင်းရှိ entry points များကို ခွဲခြားသတ်မှတ်ပေးသော labels များနှင့်အတူ instructions အစီအစဉ်တစ်ခုအဖြစ် သတ်မှတ်ပါသည်။ instruction တစ်ခုသည် အောက်ပါတို့အနက်မှ တစ်ခုဖြစ်သည်-

  • register တစ်ခုသို့ တန်ဖိုးတစ်ခု သတ်မှတ်ရန် နှိပ်ရမည့် data-path ခလုတ်၏ အမည်။ (ဒါဟာ controller ပုံကြမ်းထဲက လေးထောင့်ကွက်နဲ့ ကိုက်ညီပါတယ်။)
  • သတ်မှတ်ထားသော စမ်းသပ်မှုတစ်ခုကို လုပ်ဆောင်သည့် test instruction တစ်ခု။
  • ရှေ့ကပြုလုပ်ခဲ့သော စမ်းသပ်မှု၏ ရလဒ်အပေါ် အခြေခံ၍ controller label တစ်ခုဖြင့် ညွှန်ပြထားသော တည်နေရာတစ်ခုသို့ ကူးပြောင်းခြင်း (branch instruction)။ (စမ်းသပ်မှုနှင့် branch တို့ ပေါင်းလိုက်ပါက controller ပုံကြမ်းရှိ စိန်ပုံစံ (diamond) နှင့် ကိုက်ညီပါသည်။) အကယ်၍ စမ်းသပ်မှုသည် မှားယွင်းနေပါက controller သည် အစီအစဉ်ရှိ နောက်ထပ် instruction ဖြင့် ဆက်လက်လုပ်ဆောင်သင့်သည်။ သို့မဟုတ်ပါက controller သည် label ၏ နောက်တွင်ရှိသော instruction ဖြင့် ဆက်လက်လုပ်ဆောင်သင့်သည်။
  • လုပ်ဆောင်ချက်ကို ဆက်လက်လုပ်ဆောင်ရမည့် controller label ကို အမည်တပ်ထားသော အကန့်အသတ်မရှိ ကူးပြောင်းခြင်း (goto instruction)။

စက်သည် controller instruction အစီအစဉ်၏ အစတွင် စတင်ပြီး လုပ်ဆောင်ချက်သည် အစီအစဉ်၏ အဆုံးသို့ ရောက်သောအခါ ရပ်တန့်သွားသည်။ branch တစ်ခုက ထိန်းချုပ်မှုစီးဆင်းမှုကို ပြောင်းလဲခြင်းမှအပ instruction များကို ၎င်းတို့စာရင်းသွင်းထားသည့် အစီအစဉ်အတိုင်း လုပ်ဆောင်ပါသည်။

Figure 5.3 တွင် ဤနည်းဖြင့် ဖော်ပြထားသော GCD စက်ကို ပြသထားသည်။ ဤဥပမာသည် ဤဖော်ပြချက်များ၏ ယေဘုယျကျမှုကို အနည်းငယ်သာ အရိပ်အမြွက်ပြနိုင်ပါသည်၊ အကြောင်းမှာ GCD စက်သည် အလွန်ရိုးရှင်းသော ကိစ္စဖြစ်သောကြောင့် ဖြစ်သည်- register တစ်ခုစီတွင် ခလုတ်တစ်ခုစီသာရှိပြီး ခလုတ်နှင့် စမ်းသပ်မှုတစ်ခုစီကို controller တွင် တစ်ကြိမ်သာ အသုံးပြုပါသည်။

ကံမကောင်းစွာဖြင့်၊ ထိုကဲ့သို့သော ဖော်ပြချက်ကို ဖတ်ရန် ခက်ခဲပါသည်။ controller instruction များကို နားလည်နိုင်ရန် ကျွန်ုပ်တို့သည် ခလုတ်အမည်များနှင့် operation အမည်များ၏ အဓိပ္ပာယ်ဖွင့်ဆိုချက်များကို အမြဲတစေ ပြန်လည်ကိုးကားရမည်ဖြစ်ပြီး၊ ခလုတ်များက ဘာလုပ်သလဲဆိုသည်ကို နားလည်ရန် operation အမည်များ၏ အဓိပ္ပာယ်ဖွင့်ဆိုချက်များကိုလည်း ပြန်လည်ကိုးကားရနိုင်ပါသည်။ ထို့ကြောင့် ကျွန်ုပ်တို့သည် data-path နှင့် controller ဖော်ပြချက်များမှ အချက်အလက်များကို ပေါင်းစပ်ပြီး အားလုံးကို တစ်နေရာတည်းတွင် မြင်တွေ့နိုင်စေရန် ကျွန်ုပ်တို့၏ သင်္ကေတစနစ်ကို ပြောင်းလဲပါမည်။

ဤကဲ့သို့သော ဖော်ပြချက်ပုံစံကို ရရှိရန်အတွက် ကျွန်ုပ်တို့သည် စိတ်ကြိုက်ပေးထားသော ခလုတ်နှင့် operation အမည်များကို ၎င်းတို့၏ လုပ်ဆောင်ပုံ အဓိပ္ပာယ်ဖွင့်ဆိုချက်များဖြင့် အစားထိုးပါမည်။ ဆိုလိုသည်မှာ (controller တွင်) “Push button t<-r” ဟု ပြောမည့်အစား နှင့် သီးခြားစီ (data path များတွင်) “Button t<-r assigns the value of the rem operation to register t” နှင့် “The rem operation’s inputs are the contents of registers a and b” ဟု ပြောမည့်အစား၊ (controller တွင်) “Push the button that assigns to register t the value of the rem operation on the contents of registers a and b” ဟု ကျွန်ုပ်တို့ ပြောပါမည်။ ထိုနည်းတူစွာ (controller တွင်) “Perform the = test” ဟု ပြောမည့်အစား နှင့် သီးခြားစီ (data path များတွင်) “The = test operates on the contents of register b and the constant 0” ဟု ပြောမည့်အစား “Perform the = test on the contents of register b and the constant 0” ဟု ကျွန်ုပ်တို့ ပြောပါမည်။ ကျွန်ုပ်တို့သည် data-path ဖော်ပြချက်ကို ချန်လှပ်ထားခဲ့ပြီး controller အစီအစဉ်ကိုသာ ထားရှိပါမည်။ ထို့ကြောင့် GCD စက်ကို အောက်ပါအတိုင်း ဖော်ပြပါသည်- 

(controller
 test-b
   (test (op =) (reg b) (const 0))
   (branch (label gcd-done))
   (assign t (op rem) (reg a) (reg b))
   (assign a (reg b))
   (assign b (reg t))
   (goto (label test-b))
 gcd-done)

ဤဖော်ပြချက်ပုံစံသည် Figure 5.3 တွင် ပြထားသောပုံစံထက် ဖတ်ရန် ပိုမိုလွယ်ကူသော်လည်း အားနည်းချက်များလည်း ရှိပါသည်-

  • စက်ကြီးများအတွက် ဖော်ပြချက်မှာ ပိုမိုရှည်လျားပါသည်၊ အကြောင်းမှာ data-path အစိတ်အပိုင်းများ၏ အပြည့်အစုံဖော်ပြချက်များကို controller instruction အစီအစဉ်တွင် ၎င်းတို့ကို ဖော်ပြသည့်အခါတိုင်း ထပ်ခါတလဲလဲ ဖော်ပြရသောကြောင့် ဖြစ်သည်။ (၎င်းသည် GCD ဥပမာတွင် ပြဿနာမဟုတ်ပါ၊ အကြောင်းမှာ operation နှင့် ခလုတ်တစ်ခုစီကို တစ်ကြိမ်သာ အသုံးပြုသောကြောင့် ဖြစ်သည်။) ထို့အပြင်၊ data-path ဖော်ပြချက်များကို ထပ်ခါတလဲလဲ ဖော်ပြခြင်းသည် စက်၏ အမှန်တကယ် data-path တည်ဆောက်ပုံကို ဖုံးကွယ်သွားစေသည်၊ စက်ကြီးတစ်ခုအတွက် register၊ operation နှင့် ခလုတ် မည်မျှရှိသည်နှင့် ၎င်းတို့ကို မည်သို့ချိတ်ဆက်ထားသည်ကို သိသာထင်ရှားမှုမရှိစေပါ။
  • စက်သတ်မှတ်ချက်တစ်ခုရှိ controller instruction များသည် Lisp expression များနှင့် ဆင်တူသောကြောင့် ၎င်းတို့သည် စိတ်ကြိုက် Lisp expression များမဟုတ်ကြောင်း မေ့လျော့ရန် လွယ်ကူပါသည်။ ၎င်းတို့သည် ခွင့်ပြုထားသော စက်လုပ်ဆောင်ချက်များကိုသာ ဖော်ပြနိုင်ပါသည်။ ဥပမာအားဖြင့်၊ operation များသည် ကိန်းသေများနှင့် register များရှိ တန်ဖိုးများကိုသာ တိုက်ရိုက်လုပ်ဆောင်နိုင်ပြီး အခြား operation များ၏ ရလဒ်များကိုမူ တိုက်ရိုက်မလုပ်ဆောင်နိုင်ပါ။

ဤအားနည်းချက်များရှိသော်လည်း၊ ကျွန်ုပ်တို့သည် data path များရှိ အစိတ်အပိုင်းများနှင့် ချိတ်ဆက်မှုများကို နားလည်ခြင်းထက် controller များကို နားလည်ရန် ပိုမိုစိတ်ဝင်စားသောကြောင့် ဤအခန်းတစ်လျှောက်လုံးတွင် ဤ register-machine language ကို အသုံးပြုပါမည်။ သို့သော် အမှန်တကယ် စက်များကို ဒီဇိုင်းထုတ်ရာတွင် data-path ဒီဇိုင်းသည် အရေးကြီးကြောင်းကိုမူ ကျွန်ုပ်တို့ သတိပြုသင့်ပါသည်။

Read သည် ကျွန်ုပ်တို့ အသုံးပြုခဲ့သော operation များနှင့် ဆင်တူပါသည်၊ ၎င်းသည် register တစ်ခုတွင် သိမ်းဆည်းနိုင်သော တန်ဖိုးတစ်ခုကို ထုတ်ပေးပါသည်။ သို့သော် read သည် မည်သည့် register မှမဆို input များကို မယူပါ၊ ၎င်း၏ တန်ဖိုးသည် ကျွန်ုပ်တို့ ဒီဇိုင်းထုတ်နေသော စက်၏ အစိတ်အပိုင်းများ၏ အပြင်ဘက်တွင် ဖြစ်ပျက်သော အရာတစ်ခုအပေါ် မူတည်ပါသည်။ ကျွန်ုပ်တို့၏ စက်၏ operation များကို ထိုကဲ့သို့သော လုပ်ဆောင်ပုံမျိုး ရှိရန် ခွင့်ပြုမည်ဖြစ်ပြီး၊ ထို့ကြောင့် read ကို အသုံးပြုခြင်းကို တန်ဖိုးတစ်ခုတွက်ချက်သော အခြား operation များကဲ့သို့ပင် ရေးဆွဲဖော်ပြပါမည်။

5.1.2စက်ဒီဇိုင်းတွင် အဆင့်ဆင့် ခွဲထုတ်ခြင်း (Abstraction in Machine Design)

ကျွန်ုပ်တို့သည် အမှန်တကယ်တွင် အလွန်ရှုပ်ထွေးသော လုပ်ဆောင်ချက်များကို "အခြေခံ" (primitive) လုပ်ဆောင်ချက်များအဖြစ် သတ်မှတ်၍ စက်တစ်ခုကို တည်ဆောက်လေ့ရှိသည်။ ဥပမာအားဖြင့် 5.4 နှင့် 5.5 တို့တွင် Scheme ၏ environment manipulations များကို အခြေခံလုပ်ဆောင်ချက်များအဖြစ် ကျွန်ုပ်တို့ ယူဆပါမည်။ ဤကဲ့သို့သော abstraction ပြုလုပ်ခြင်းသည် စက်၏ အစိတ်အပိုင်းအချို့၏ အသေးစိတ်အချက်အလက်များကို လျစ်လျူရှုထားနိုင်ပြီး ဒီဇိုင်း၏ အခြားကဏ္ဍများကိုသာ အာရုံစိုက်နိုင်စေသောကြောင့် တန်ဖိုးရှိလှသည်။ သို့သော် ရှုပ်ထွေးမှုအများအပြားကို ဖုံးကွယ်ထားခြင်းကြောင့် စက်ဒီဇိုင်းတစ်ခုသည် လက်တွေ့မကျဟု မဆိုလိုပါ။ ကျွန်ုပ်တို့သည် ရှုပ်ထွေးသော "အခြေခံ" လုပ်ဆောင်ချက်များကို ပိုမိုရိုးရှင်းသော အခြေခံလုပ်ဆောင်ချက်များဖြင့် အမြဲတမ်း အစားထိုးနိုင်သည်။

GCD စက်ကို လေ့လာကြည့်ကြပါစို့။ ထိုစက်တွင် register a နှင့် b တို့၏ ပါဝင်သောအချက်အလက်များကို စား၍ အကြွင်း (remainder) ကို တွက်ချက်ပြီး ရလဒ်ကို register t ထဲသို့ ထည့်သွင်းပေးသော ညွှန်ကြားချက်တစ်ခု ရှိသည်။ အကယ်၍ ကျွန်ုပ်တို့သည် အခြေခံ remainder လုပ်ဆောင်ချက်ကို အသုံးမပြုဘဲ GCD စက်ကို တည်ဆောက်လိုပါက၊ အနှုတ် (subtraction) ကဲ့သို့သော ပိုမိုရိုးရှင်းသော လုပ်ဆောင်ချက်များကို အသုံးပြု၍ remainder ကို မည်သို့တွက်ချက်ရမည်ကို သတ်မှတ်ပေးရမည်ဖြစ်သည်။ အမှန်စင်စစ်၊ ကျွန်ုပ်တို့သည် ဤနည်းလမ်းအတိုင်း remainder ကို ရှာဖွေပေးသော Scheme procedure တစ်ခုကို ရေးသားနိုင်သည်- 

(define (remainder n d)
  (if (< n d) n (remainder (- n d) d)))

ထို့ကြောင့် ကျွန်ုပ်တို့သည် GCD စက်၏ data paths များရှိ remainder လုပ်ဆောင်ချက်ကို အနှုတ်လုပ်ဆောင်ချက်တစ်ခုနှင့် နှိုင်းယှဉ်စစ်ဆေးမှု (comparison test) တစ်ခုတို့ဖြင့် အစားထိုးနိုင်သည်။ ပုံ ၅.၅ တွင် ပိုမိုအသေးစိတ်ဖော်ပြထားသော စက်အတွက် data paths နှင့် controller တို့ကို ပြသထားသည်။ ညွှန်ကြားချက်

လေ့ကျင့်ခန်း ၅.၃- 1.1.7 တွင် ဖော်ပြထားသည့်အတိုင်း Newton’s method ကို အသုံးပြု၍ square roots တွက်ချက်မည့် စက်တစ်ခုကို ဒီဇိုင်းဆွဲပါ- 

(define (sqrt x)
  (define (good-enough? guess)
    (< (abs (- (square guess) x)) 0.001))
  (define (improve guess)
    (average guess (/ x guess)))
  (define (sqrt-iter guess)
    (if (good-enough? guess)
        guess
        (sqrt-iter (improve guess))))
  (sqrt-iter 1.0))

ပထမဦးစွာ good-enough? နှင့် improve လုပ်ဆောင်ချက်များကို အခြေခံ (primitives) များအဖြစ် ရရှိနိုင်သည်ဟု ယူဆပါ။ ထို့နောက် ၎င်းတို့ကို သင်္ချာဆိုင်ရာ လုပ်ဆောင်ချက်များ (arithmetic operations) အဖြစ် မည်သို့ချဲ့ထွင်ရမည်ကို ပြသပါ။ data-path diagram တစ်ခုဆွဲခြင်းနှင့် register-machine language ဖြင့် controller သတ်မှတ်ချက်တစ်ခုကို ရေးသားခြင်းတို့ဖြင့် sqrt စက်ဒီဇိုင်း၏ ဗားရှင်းတစ်ခုစီကို ဖော်ပြပါ။

တွက်ချက်မှုတစ်ခုကို လုပ်ဆောင်ရန် စက်တစ်ခုကို ပုံစံထုတ်ရာတွင်၊ အစိတ်အပိုင်းများကို ထပ်ခါတလဲလဲ အသုံးပြုမည့်အစား တွက်ချက်မှု၏ မတူညီသော အစိတ်အပိုင်းများတွင် အစိတ်အပိုင်းများကို မျှဝေသုံးစွဲနိုင်ရန် စီစဉ်ခြင်းကို ပိုမိုနှစ်သက်လေ့ရှိသည်။ register a နှင့် b တို့ရှိ အကြောင်းအရာများ၏ GCD ကို ရှာဖွေပေးသည့် တွက်ချက်မှုတစ်ခုနှင့် register c နှင့် d တို့ရှိ အကြောင်းအရာများ၏ GCD ကို ရှာဖွေပေးသည့် တွက်ချက်မှုတစ်ခု — စုစုပေါင်း GCD တွက်ချက်မှု နှစ်ခု ပါဝင်သော စက်တစ်ခုကို စဉ်းစားကြည့်ပါ။ ကျွန်ုပ်တို့တွင် အခြေခံကျသော gcd operation တစ်ခု ရှိသည်ဟု ယူဆခြင်းဖြင့် စတင်နိုင်ပြီး၊ ထို့နောက် gcd ဖြစ်ရပ်နှစ်ခုကို ပိုမိုအခြေခံကျသော operation များဖြင့် ချဲ့ထွင်နိုင်သည်။ Figure 5.7 တွင် ရလဒ်အနေဖြင့် ရရှိလာသော စက်၏ data paths များမှ GCD အစိတ်အပိုင်းများကိုသာ ဖော်ပြထားပြီး၊ ၎င်းတို့ကို စက်၏ ကျန်ရှိသော အစိတ်အပိုင်းများနှင့် မည်သို့ ချိတ်ဆက်ထားသည်ကိုမူ ဖော်ပြထားခြင်း မရှိပါ။ ထိုပုံတွင် စက်၏ controller sequence နှင့် သက်ဆိုင်သော အစိတ်အပိုင်းများကိုလည်း ဖော်ပြထားသည်။

ဤစက်တွင် remainder operation box နှစ်ခုနှင့် ညီမျှမှုကို စစ်ဆေးရန် box နှစ်ခု ပါဝင်သည်။ အကယ်၍ ထပ်ခါတလဲလဲ ပါဝင်နေသော အစိတ်အပိုင်းများသည် remainder box ကဲ့သို့ပင် ရှုပ်ထွေးပါက၊ ဤနည်းလမ်းသည် စက်တည်ဆောက်ရန်အတွက် စရိတ်သက်သာသော နည်းလမ်း ဖြစ်လိမ့်မည် မဟုတ်ပါ။ အကယ်၍ ထိုသို့ပြုလုပ်ခြင်းက ပိုမိုကြီးမားသော စက်၏ ကျန်ရှိသော တွက်ချက်မှုများကို မထိခိုက်စေနိုင်ပါက၊ GCD တွက်ချက်မှု နှစ်ခုစလုံးအတွက် တူညီသော အစိတ်အပိုင်းများကို အသုံးပြုခြင်းဖြင့် data-path အစိတ်အပိုင်းများ ထပ်ခါတလဲလဲ ဖြစ်ခြင်းကို ရှောင်ရှားနိုင်သည်။ အကယ်၍ controller သည် gcd-2 သို့ ရောက်ရှိချိန်တွင် register a နှင့် b တို့ရှိ တန်ဖိုးများကို မလိုအပ်တော့ပါက (သို့မဟုတ် ထိုတန်ဖိုးများကို လုံခြုံစေရန် အခြား register များသို့ ရွှေ့ပြောင်းထားနိုင်ပါက)၊ ဒုတိယ GCD ကို တွက်ချက်ရာတွင်လည်း register c နှင့် d အစား register a နှင့် b ကိုသာ အသုံးပြုစေရန် စက်ကို ပြောင်းလဲနိုင်သည်။ ဤသို့ ပြုလုပ်ပါက Figure 5.8 တွင် ပြထားသော controller sequence ကို ရရှိမည်ဖြစ်သည်။

ဤနည်းလမ်းသည် သေးငယ်သော ပြဿနာများကို ကိုင်တွယ်ရန် သင့်လျော်သော ချဉ်းကပ်မှုဖြစ်သော်လည်း၊ controller sequence ထဲတွင် GCD တွက်ချက်မှုပေါင်း များစွာရှိနေပါက အဆင်မပြေ ဖြစ်လာနိုင်သည်။ gcd subroutine ပြီးနောက် မည်သည့်နေရာမှ ဆက်လက် လုပ်ဆောင်ရမည်ကို ဆုံးဖြတ်ရန်အတွက်၊ gcd ကို အသုံးပြုသော နေရာအားလုံးအတွက် data paths များတွင် test များ လုပ်ဆောင်ရန်နှင့် controller တွင် branch instruction များ လိုအပ်မည် ဖြစ်သည်။ subroutine များကို အကောင်အထည်ဖော်ရန် ပိုမိုထိရောက်သော နည်းလမ်းမှာ subroutine ပြီးဆုံးသောအခါ ဆက်လက်လုပ်ဆောင်ရမည့် controller sequence ရှိ entry point ၏ label ကို continue register တွင် သိမ်းဆည်းထားခြင်း ဖြစ်သည်။ ဤမဟာဗျူဟာကို အကောင်အထည်ဖော်ရန် register machine ၏ data paths နှင့် controller ကြားတွင် ချိတ်ဆက်မှု ပုံစံအသစ်တစ်ခု လိုအပ်သည်- register တစ်ခုထဲသို့ controller sequence ရှိ label တစ်ခုကို တန်ဖိုးအဖြစ် သတ်မှတ်ပေးနိုင်ရန်နှင့် ထိုတန်ဖိုးကို register ထဲမှ ပြန်ထုတ်ယူ၍ သတ်မှတ်ထားသော entry point ၌ ဆက်လက် လုပ်ဆောင်နိုင်ရန် နည်းလမ်းရှိရမည်။

ဤစွမ်းဆောင်ရည်ကို ထင်ဟပ်စေရန်၊ register-machine ဘာသာစကား၏ assign instruction ကို controller sequence မှ label တစ်ခုအား (အထူးကိန်းသေ ပုံစံတစ်ခုအဖြစ်) register ထဲသို့ တန်ဖိုးအဖြစ် သတ်မှတ်နိုင်ရန် ချဲ့ထွင်ပါမည်။ ထို့အပြင် goto instruction ကိုလည်း ကိန်းသေ label တစ်ခုဖြင့် ဖော်ပြထားသော entry point သာမက register တစ်ခု၏ အကြောင်းအရာများဖြင့် ဖော်ပြထားသော entry point မှလည်း ဆက်လက်လုပ်ဆောင်နိုင်ရန် ချဲ့ထွင်ပါမည်။ ဤတည်ဆောက်ပုံအသစ်များကို အသုံးပြု၍ gcd subroutine ကို continue register တွင် သိမ်းဆည်းထားသော နေရာသို့ branch လုပ်ခြင်းဖြင့် အဆုံးသတ်နိုင်သည်။ ယင်းသည် Figure 5.10 တွင် ပြထားသော controller sequence သို့ ဦးတည်စေသည်။

subroutine တစ်ခုထက်ပိုသော စက်တစ်ခုတွင် continuation register အများအပြား (ဥပမာ gcd-continue၊ factorial-continue) ကို အသုံးပြုနိုင်သည်၊ သို့မဟုတ် subroutine အားလုံးကို continue register တစ်ခုတည်းကို မျှဝေသုံးစွဲစေနိုင်သည်။ မျှဝေသုံးစွဲခြင်းက ပိုမိုစရိတ်သက်သာသော်လည်း၊ အခြား subroutine (sub2) ကို ခေါ်ဆိုသည့် subroutine (sub1) ရှိနေပါက သတိပြုရမည်။ sub1 သည် sub2 ကို မခေါ်ဆိုမီ continue register ကို မပြင်ဆင်ခင်၊ ၎င်း၏ လက်ရှိ continue အကြောင်းအရာကို အခြား register တစ်ခုတွင် သိမ်းဆည်းမထားပါက၊ sub1 ပြီးဆုံးသောအခါ မည်သည့်နေရာသို့ ပြန်သွားရမည်ကို သိရှိမည်မဟုတ်ပေ။ နောက်အပိုင်းတွင် recursion ကို ကိုင်တွယ်ရန် တီထွင်ထားသော ယန္တရားသည် ဤ nested subroutine call များ၏ ပြဿနာအတွက် ပိုမိုကောင်းမွန်သော ဖြေရှင်းချက်ကို ပေးစွမ်းနိုင်သည်။

ယခုအချိန်အထိ ဖော်ပြခဲ့သော အယူအဆများဖြင့်၊ ပရိုဆက်စ်၏ state variable တစ်ခုစီအတွက် သက်ဆိုင်ရာ register တစ်ခုစီရှိသော register machine တစ်ခုကို သတ်မှတ်ခြင်းဖြင့် မည်သည့် iterative process ကိုမဆို ကျွန်ုပ်တို့ အကောင်အထည်ဖော်နိုင်သည်။ ထိုစက်သည် ရပ်တန့်ရန် အခြေအနေ (termination condition) တစ်ခု ပြည့်စုံသည်အထိ register များ၏ ပါဝင်အကြောင်းအရာများကို ပြောင်းလဲရင်း controller loop တစ်ခုကို ထပ်ခါတလဲလဲ လုပ်ဆောင်သည်။ Controller sequence ၏ အချက်တိုင်းတွင်၊ စက်၏ အခြေအနေ (iterative process ၏ အခြေအနေကို ကိုယ်စားပြုသော) ကို register များ၏ ပါဝင်အကြောင်းအရာများ (state variable များ၏ တန်ဖိုးများ) ဖြင့် အပြည့်အဝ ဆုံးဖြတ်သည်။

သို့သော်လည်း၊ recursive process များကို အကောင်အထည်ဖော်ရန်အတွက် ထပ်ဆောင်း ယန္တရားတစ်ခု လိုအပ်သည်။ ၁.၂.၁ တွင် ကျွန်ုပ်တို့ ပထမဆုံး လေ့လာခဲ့သော factorial များကို တွက်ချက်ရန်အတွက် အောက်ပါ recursive method ကို သုံးသပ်ကြည့်ပါ။ 

(define (factorial n)
  (if (= n 1) 
      1
      (* (factorial (- n 1)) n)))

ပရိုစီဂျာမှ ကျွန်ုပ်တို့ မြင်တွေ့ရသည့်အတိုင်း၊ n ! ကို တွက်ချက်ရန် ( n − 1 ) ! ကို တွက်ချက်ရန် လိုအပ်သည်။ အောက်ပါ ပရိုစီဂျာပေါ်တွင် အခြေခံထားသော ကျွန်ုပ်တို့၏ GCD စက်သည်လည်း 

(define (gcd a b)
  (if (= b 0) 
      a
      (gcd b (remainder a b))))

အလားတူပင် အခြားသော GCD တစ်ခုကို တွက်ချက်ရမည် ဖြစ်သည်။ သို့သော် မူလတွက်ချက်မှုကို GCD တွက်ချက်မှုအသစ်တစ်ခုသို့ လျှော့ချပေးသော gcd ပရိုစီဂျာနှင့်၊ အခြား factorial တစ်ခုကို subproblem တစ်ခုအနေဖြင့် တွက်ချက်ရန် လိုအပ်သော factorial ကြားတွင် အရေးကြီးသော ကွာခြားချက်တစ်ခု ရှိသည်။ GCD တွင်၊ GCD တွက်ချက်မှုအသစ်၏ အဖြေသည် မူလပုစ္ဆာ၏ အဖြေပင် ဖြစ်သည်။ နောက်ထပ် GCD တစ်ခုကို တွက်ချက်ရန်အတွက်၊ argument အသစ်များကို GCD စက်၏ input register များတွင် ထည့်သွင်းပြီး အလားတူ controller sequence ကိုပင် လုပ်ဆောင်ကာ စက်၏ data path များကို ပြန်လည်အသုံးပြုလိုက်ရုံသာ ဖြစ်သည်။ စက်သည် နောက်ဆုံး GCD ပုစ္ဆာကို ဖြေရှင်းပြီးသောအခါ၊ ၎င်းသည် တွက်ချက်မှုတစ်ခုလုံးကို ပြီးမြောက်သွားမည် ဖြစ်သည်။

factorial (သို့မဟုတ် မည်သည့် recursive process တွင်မဆို) ပုစ္ဆာတွင်မူ၊ factorial subproblem အသစ်၏ အဖြေသည် မူလပုစ္ဆာ၏ အဖြေမဟုတ်ပါ။ ( n − 1 ) ! အတွက် ရရှိသော တန်ဖိုးကို နောက်ဆုံးအဖြေ ရရှိရန်အတွက် n နှင့် မြှောက်ရမည် ဖြစ်သည်။ အကယ်၍ ကျွန်ုပ်တို့သည် GCD ဒီဇိုင်းကို အတုယူရန် ကြိုးစားပြီး factorial subproblem ကို n register ကို လျှော့ချကာ factorial စက်ကို ပြန်လည်ပတ်ခြင်းဖြင့် ဖြေရှင်းရန် ကြိုးစားပါက၊ ရလဒ်ကို မြှောက်ရန်အတွက် လိုအပ်သော n ၏ တန်ဖိုးဟောင်းကို ကျွန်ုပ်တို့ ရရှိနိုင်တော့မည် မဟုတ်ပါ။ ထို့ကြောင့် subproblem ကို လုပ်ဆောင်ရန်အတွက် ဒုတိယ factorial စက်တစ်ခု လိုအပ်မည် ဖြစ်သည်။ ဤဒုတိယ factorial တွက်ချက်မှုတွင်လည်း ၎င်းကိုယ်တိုင် factorial subproblem တစ်ခု ရှိနေပြီး ၎င်းအတွက် တတိယ factorial စက်တစ်ခု လိုအပ်သည်၊ ဤသို့ဖြင့် ဆက်သွားနေမည် ဖြစ်သည်။ Factorial စက်တစ်ခုစီတွင် ၎င်း၏အတွင်း၌ အခြား factorial စက်တစ်ခု ပါဝင်နေသောကြောင့်၊ စက်တစ်ခုလုံးတွင် ဆင်တူသော စက်များ၏ အကန့်အသတ်မရှိသော အဆင့်ဆင့်ထပ်ခြင်း (nesting) ပါဝင်နေမည်ဖြစ်ပြီး ထို့ကြောင့် ပုံသေဖြစ်သော အကန့်အသတ်ရှိသော အစိတ်အပိုင်းများ အရေအတွက်ဖြင့် တည်ဆောက်၍ မရနိုင်ပါ။

မည်သို့ပင်ဆိုစေကာမူ၊ စက်၏ nested instance တစ်ခုစီအတွက် တူညီသော အစိတ်အပိုင်းများကို အသုံးပြုရန် စီစဉ်နိုင်ပါက factorial process ကို register machine တစ်ခုအနေဖြင့် ကျွန်ုပ်တို့ အကောင်အထည်ဖော်နိုင်သည်။ အထူးသဖြင့် n ! ကို တွက်ချက်သော စက်သည် ( n − 1 ) ! ကို တွက်ချက်သော subproblem၊ ( n − 2 ) ! ကို တွက်ချက်သော subproblem စသည်တို့တွင် တူညီသော အစိတ်အပိုင်းများကို အသုံးပြုသင့်သည်။ factorial process သည် တွက်ချက်မှုတစ်ခု ပြုလုပ်ရန် တူညီသော စက်များ၏ အကန့်အသတ်မရှိသော မိတ္တူများ လိုအပ်သည်ဟု သတ်မှတ်ထားသော်လည်း၊ ပေးထားသော အချိန်တစ်ခုတွင် ဤမိတ္တူများအနက် တစ်ခုတည်းသာ အသက်ဝင်ရန် လိုအပ်သောကြောင့် ဤအချက်သည် ဖြစ်နိုင်ခြေရှိသည်။ စက်သည် recursive subproblem တစ်ခုကို ကြုံတွေ့ရသောအခါ၊ ၎င်းသည် မူလပုစ္ဆာပေါ်တွင် လုပ်ဆောင်နေမှုကို ခေတ္တရပ်ဆိုင်းထားနိုင်ပြီး၊ subproblem ကို လုပ်ဆောင်ရန် တူညီသော ရုပ်ပိုင်းဆိုင်ရာ အစိတ်အပိုင်းများကို ပြန်လည်အသုံးပြုကာ၊ ထို့နောက် ရပ်ဆိုင်းထားသော တွက်ချက်မှုကို ပြန်လည်လုပ်ဆောင်နိုင်သည်။

Subproblem တွင် register များ၏ ပါဝင်အကြောင်းအရာများသည် မူလပုစ္ဆာတွင် ရှိခဲ့သည်နှင့် ကွာခြားလိမ့်မည်။ (ဤကိစ္စတွင် n register သည် လျော့နည်းသွားသည်။) ရပ်ဆိုင်းထားသော တွက်ချက်မှုကို ဆက်လက်လုပ်ဆောင်နိုင်ရန်အတွက်၊ subproblem ကို ဖြေရှင်းပြီးနောက် လိုအပ်မည့် မည်သည့် register များ၏ ပါဝင်အကြောင်းအရာများကိုမဆို စက်မှ သိမ်းဆည်းထားရမည် ဖြစ်ပြီး၊ ထို့မှသာ ရပ်ဆိုင်းထားသော တွက်ချက်မှုကို ဆက်လက်လုပ်ဆောင်ရန် ၎င်းတို့ကို ပြန်လည်ဖော်ယူ (restore) နိုင်မည် ဖြစ်သည်။ Factorial ကိစ္စတွင်၊ လျှော့ချထားသော n register ၏ factorial ကို တွက်ချက်ပြီးသောအခါ ပြန်လည်ဖော်ယူရန်အတွက် n ၏ တန်ဖိုးဟောင်းကို သိမ်းဆည်းထားမည် ဖြစ်သည်။287

Nested recursive call များ၏ အနက် (depth) ပေါ်တွင် ကြိုတင်သတ်မှတ်ထားသော ကန့်သတ်ချက်မရှိသောကြောင့်၊ ကျွန်ုပ်တို့သည် စိတ်ကြိုက်အရေအတွက်ရှိသော register တန်ဖိုးများကို သိမ်းဆည်းရန် လိုအပ်နိုင်သည်။ Recursion များ၏ nesting တွင် နောက်ဆုံးဝင်သော subproblem သည် ပထမဆုံး ပြီးဆုံးရမည် ဖြစ်သောကြောင့်၊ ဤတန်ဖိုးများကို သိမ်းဆည်းခဲ့သော အစီအစဉ်၏ ပြောင်းပြန်အတိုင်း ပြန်လည်ဖော်ယူရမည် ဖြစ်သည်။ ဤအချက်သည် register တန်ဖိုးများကို သိမ်းဆည်းရန်အတွက် stack သို့မဟုတ် “နောက်ဆုံးဝင်၊ ပထမထွက်” (last in, first out) data structure တစ်ခုကို အသုံးပြုရန် လိုအပ်စေသည်။ instruction အမျိုးအစား နှစ်မျိုးကို ပေါင်းထည့်ခြင်းဖြင့် register-machine language ကို stack တစ်ခု ပါဝင်အောင် တိုးချဲ့နိုင်သည်။ တန်ဖိုးများကို save instruction ကို အသုံးပြု၍ stack ပေါ်သို့ တင်နိုင်ပြီး restore instruction ကို အသုံးပြု၍ stack မှ ပြန်လည်ဖော်ယူနိုင်သည်။ stack ပေါ်တွင် တန်ဖိုးများ အစီအစဉ်လိုက် save လုပ်ပြီးနောက်၊ restore များကို အစီအစဉ်လိုက် လုပ်ဆောင်ခြင်းဖြင့် ထိုတန်ဖိုးများကို ပြောင်းပြန်အစီအစဉ်အတိုင်း ပြန်လည်ရရှိမည် ဖြစ်သည်။288

Stack ၏ အကူအညီဖြင့်၊ factorial subproblem တစ်ခုစီအတွက် factorial စက်၏ data path မိတ္တူတစ်ခုတည်းကိုပင် ပြန်လည်အသုံးပြုနိုင်သည်။ Data path များကို လည်ပတ်စေသော controller sequence ကို ပြန်လည်အသုံးပြုရာတွင်လည်း အလားတူ ဒီဇိုင်းပြဿနာရှိသည်။ Factorial တွက်ချက်မှုကို ပြန်လည်လုပ်ဆောင်ရန်အတွက်၊ iterative process တစ်ခုကဲ့သို့ပင် controller သည် အစသို့ ရိုးရှင်းစွာ loop ပြန်ပတ်၍ မရနိုင်ပါ၊ အကြောင်းမှာ ( n − 1 ) ! subproblem ကို ဖြေရှင်းပြီးနောက် စက်သည် ရလဒ်ကို n နှင့် မြှောက်ရန် လိုအပ်နေသေးသောကြောင့် ဖြစ်သည်။ Controller သည် ၎င်း၏ n ! တွက်ချက်မှုကို ခေတ္တရပ်ဆိုင်းထားရမည်၊ ( n − 1 ) ! subproblem ကို ဖြေရှင်းရမည်၊ ထို့နောက် ၎င်း၏ n ! တွက်ချက်မှုကို ဆက်လက်လုပ်ဆောင်ရမည်။ Factorial တွက်ချက်မှုအပေါ် ဤအမြင်သည် 5.1.3 တွင် ဖော်ပြထားသော subroutine ယန္တရားကို အသုံးပြုရန် အကြံပြုထားပြီး၊ ၎င်းတွင် controller သည် subproblem တစ်ခုကို ဖြေရှင်းသော sequence ၏ အစိတ်အပိုင်းသို့ ကူးပြောင်းရန်နှင့် မူလပုစ္ဆာတွင် ရပ်နားခဲ့သော နေရာမှ ပြန်လည်စတင်ရန်အတွက် continue register ကို အသုံးပြုသည်။ ထို့ကြောင့် continue register တွင် သိမ်းဆည်းထားသော entry point သို့ ပြန်သွားသည့် factorial subroutine တစ်ခုကို ကျွန်ုပ်တို့ ပြုလုပ်နိုင်သည်။ Subroutine call တစ်ခုစီ၏ ပတ်လည်တွင်၊ factorial တွက်ချက်မှု၏ “level” တစ်ခုစီသည် တူညီသော continue register ကို အသုံးပြုမည်ဖြစ်သောကြောင့် n register ကဲ့သို့ပင် continue ကို save နှင့် restore လုပ်ရမည်။ ဆိုလိုသည်မှာ factorial subroutine သည် subproblem တစ်ခုအတွက် ၎င်းကိုယ်တိုင် ပြန်လည်ခေါ်ဆိုသောအခါ continue တွင် တန်ဖိုးအသစ်တစ်ခု ထည့်ရမည် ဖြစ်သော်လည်း၊ subproblem ကို ဖြေရှင်းရန် ၎င်းအား ခေါ်ဆိုခဲ့သည့် နေရာသို့ ပြန်သွားနိုင်ရန်အတွက် တန်ဖိုးဟောင်းကို လိုအပ်မည် ဖြစ်သည်။

ပုံ ၅.၁၁ တွင် recursive factorial ပရိုစီဂျာကို အကောင်အထည်ဖော်သော စက်တစ်ခုအတွက် data path များနှင့် controller ကို ပြသထားသည်။ ထိုစက်တွင် stack တစ်ခုနှင့် n၊ val နှင့် continue ဟုခေါ်သော register သုံးခု ရှိသည်။ Data-path ပုံကြမ်းကို ရိုးရှင်းစေရန်၊ ကျွန်ုပ်တို့သည် register-assignment ခလုတ်များကို အမည်မပေးထားဘဲ၊ stack-operation ခလုတ်များ (register များကို သိမ်းဆည်းရန် sc နှင့် sn၊ register များကို ပြန်လည်ဖော်ယူရန် rc နှင့် rn) ကိုသာ အမည်ပေးထားသည်။ စက်ကို လည်ပတ်စေရန်၊ ကျွန်ုပ်တို့သည် factorial တွက်ချက်လိုသော ကိန်းဂဏန်းကို n register ထဲသို့ ထည့်ပြီး စက်ကို စတင်သည်။ စက်သည် fact-done သို့ ရောက်ရှိသောအခါ၊ တွက်ချက်မှု ပြီးမြောက်သွားပြီး အဖြေကို val register တွင် တွေ့ရှိရမည် ဖြစ်သည်။ Controller sequence တွင်၊ n နှင့် continue တို့ကို recursive call တစ်ခုစီ၏ ရှေ့တွင် သိမ်းဆည်းထားပြီး call မှ ပြန်လာသောအခါ ပြန်လည်ဖော်ယူသည်။ call တစ်ခုမှ ပြန်လာခြင်းကို continue တွင် သိမ်းဆည်းထားသော တည်နေရာသို့ branch လုပ်ခြင်းဖြင့် လုပ်ဆောင်သည်။ နောက်ဆုံးပြန်လာမှုသည် fact-done သို့ သွားစေရန်အတွက် စက်စတင်ချိန်တွင် continue ကို initialize လုပ်ထားသည်။ Factorial တွက်ချက်မှု၏ ရလဒ်ကို သိမ်းဆည်းထားသော val register ကိုမူ recursive call မတိုင်မီ သိမ်းဆည်းမထားပါ၊ အကြောင်းမှာ subroutine မှ ပြန်လာပြီးနောက် val ၏ ပါဝင်အကြောင်းအရာဟောင်းသည် အသုံးမဝင်တော့သောကြောင့် ဖြစ်သည်။ subcomputation မှ ထွက်ပေါ်လာသော တန်ဖိုးအသစ်ကိုသာ လိုအပ်ပါသည်။

မူအားဖြင့် factorial တွက်ချက်မှုသည် အကန့်အသတ်မရှိသော စက်တစ်ခု လိုအပ်သော်လည်း၊ ပုံ ၅.၁၁ ရှိ စက်သည် အကန့်အသတ်မရှိနိုင်သော stack မှလွဲ၍ အမှန်တကယ်တွင် အကန့်အသတ်ရှိသော စက်တစ်ခုဖြစ်သည်။ သို့သော်လည်း stack ၏ မည်သည့် ရုပ်ပိုင်းဆိုင်ရာ အကောင်အထည်ဖော်မှုမဆို အကန့်အသတ်ရှိသော အရွယ်အစားသာ ရှိမည်ဖြစ်ပြီး၊ ၎င်းသည် စက်မှ ကိုင်တွယ်နိုင်သော recursive call များ၏ အနက်ကို ကန့်သတ်ပေးမည် ဖြစ်သည်။ Factorial ၏ ဤအကောင်အထည်ဖော်မှုသည် recursive algorithm များကို stack များပေါင်းထည့်ထားသော သာမန် register machine များအဖြစ် ဖော်ဆောင်ရန်အတွက် ယေဘုယျ မဟာဗျူဟာကို သရုပ်ဖော်သည်။ Recursive subproblem တစ်ခုကို ကြုံတွေ့ရသောအခါ၊ subproblem ကို ဖြေရှင်းပြီးနောက် လက်ရှိတန်ဖိုးများ လိုအပ်မည့် register များကို stack ပေါ်တွင် သိမ်းဆည်းထားပြီး၊ recursive subproblem ကို ဖြေရှင်းကာ၊ ထို့နောက် သိမ်းဆည်းထားသော register များကို ပြန်လည်ဖော်ယူပြီး မူလပုစ္ဆာပေါ်တွင် ဆက်လက်လုပ်ဆောင်သည်။ continue register ကို အမြဲတမ်း သိမ်းဆည်းထားရမည်။ သိမ်းဆည်းရန် လိုအပ်သော အခြား register များ ရှိမရှိမှာ သက်ဆိုင်ရာ စက်ပေါ်တွင် မူတည်သည်၊ အကြောင်းမှာ recursive တွက်ချက်မှု အားလုံးသည် subproblem ကို ဖြေရှင်းနေစဉ်အတွင်း ပြောင်းလဲသွားသော register များ၏ မူလတန်ဖိုးများကို မလိုအပ်သောကြောင့် ဖြစ်သည် (လေ့ကျင့်ခန်း ၅.၄ ကို ကြည့်ပါ)။

      n 
      (+ (fib (- n 1)) (fib (- n 2)))))

Factorial ကဲ့သို့ပင်၊ recursive Fibonacci တွက်ချက်မှုကို n၊ val နှင့် continue register များပါရှိသော register machine တစ်ခုအနေဖြင့် အကောင်အထည်ဖော်နိုင်သည်။ ထိုစက်သည် factorial အတွက် စက်ထက် ပိုမိုရှုပ်ထွေးသည်၊ အကြောင်းမှာ controller sequence တွင် recursive call များ လုပ်ဆောင်ရန် လိုအပ်သော နေရာနှစ်ခု — Fib(n - 1) ကို တွက်ချက်ရန် တစ်ကြိမ်နှင့် Fib(n - 2) ကို တွက်ချက်ရန် တစ်ကြိမ် — ရှိသောကြောင့် ဖြစ်သည်။ ဤ call တစ်ခုစီအတွက် ပြင်ဆင်ရန်၊ နောက်ပိုင်းတွင် လိုအပ်မည့် တန်ဖိုးများရှိသော register များကို ကျွန်ုပ်တို့ သိမ်းဆည်းထားပြီး၊ n register ကို recursively တွက်ချက်ရန် လိုအပ်သော ကိန်းဂဏန်း ( n − 1 သို့မဟုတ် n − 2 ) သို့ သတ်မှတ်ကာ၊ ပြန်လာရမည့် main sequence ရှိ entry point (အသီးသီး afterfib-n-1 သို့မဟုတ် afterfib-n-2) ကို continue သို့ သတ်မှတ်ပေးသည်။ ထို့နောက် fib-loop သို့ သွားသည်။ recursive call မှ ပြန်လာသောအခါ အဖြေသည် val တွင် ရှိနေမည် ဖြစ်သည်။ ပုံ ၅.၁၂ တွင် ဤစက်အတွက် controller sequence ကို ပြသထားသည်။

ပုံ ၅.၁၂- ↓ Fibonacci ကိန်းဂဏန်းများကို တွက်ချက်ရန် စက်တစ်ခုအတွက် Controller။ 

(controller
   (assign continue (label fib-done))
 fib-loop
   (test (op <) (reg n) (const 2))
   (branch (label immediate-answer))
   ;; set up to compute Fib(n − 1)
   (save continue)
   (assign continue (label afterfib-n-1))
   (save n)           ; save old value of n
   (assign n 
           (op -)
           (reg n)
           (const 1)) ; clobber n to n-1
   (goto 
    (label fib-loop)) ; perform recursive call
 afterfib-n-1 ; upon return, val contains Fib(n − 1)
   (restore n)
   (restore continue)
   ;; set up to compute Fib(n − 2)
   (assign n (op -) (reg n) (const 2))
   (save continue)
   (assign continue (label afterfib-n-2))
   (save val)         ; save Fib(n − 1)
   (goto (label fib-loop))
 afterfib-n-2 ; upon return, val contains Fib(n − 2)
   (assign n 
           (reg val)) ; n now contains Fib(n − 2)
   (restore val)      ; val now contains Fib(n − 1)
   (restore continue)
   (assign val        ; Fib(n − 1) + Fib(n − 2)
           (op +) 
           (reg val)
           (reg n))
   (goto              ; return to caller,
    (reg continue))   ; answer is in val
 immediate-answer
   (assign val 
           (reg n))   ; base case: Fib(n) = n
   (goto (reg continue))
 fib-done)

လေ့ကျင့်ခန်း ၅.၄- အောက်ပါ ပရိုစီဂျာ တစ်ခုစီကို အကောင်အထည်ဖော်သော register machine များကို သတ်မှတ်ပါ။ စက်တစ်ခုစီအတွက် controller instruction sequence တစ်ခုကို ရေးသားပြီး data path များကို ပြသသည့် ပုံကြမ်းတစ်ခုကို ဆွဲပါ။

  1. ၁။ Recursive exponentiation: 
    (define (expt b n)
  (if (= n 0)
      1
      (* b (expt b (- n 1)))))
  2. ၂။ Iterative exponentiation: 
    (define (expt b n)
  (define (expt-iter counter product)
    (if (= counter 0)
        product
        (expt-iter (- counter 1)
                   (* b product))))
  (expt-iter n 1))

လေ့ကျင့်ခန်း ၅.၅- factorial နှင့် Fibonacci စက်များကို (အနည်းဆုံး recursive call တစ်ခု လုပ်ဆောင်ရန် လိုအပ်သော) nontrivial input အချို့ကို အသုံးပြု၍ ကိုယ်တိုင် လက်ဖြင့် စမ်းသပ်မောင်းနှင် (hand-simulate) ကြည့်ပါ။ လုပ်ဆောင်မှု၏ အရေးကြီးသော အချက်တိုင်းတွင် stack ၏ ပါဝင်အကြောင်းအရာများကို ပြသပါ။

လေ့ကျင့်ခန်း ၅.၆- Fibonacci စက်၏ controller sequence တွင် ပိုမိုမြန်ဆန်သော စက်တစ်ခု ဖြစ်လာစေရန် ဖယ်ရှားနိုင်သည့် ပိုနေသော save တစ်ခုနှင့် restore တစ်ခု ရှိနေကြောင်း Ben Bitdiddle က သတိပြုမိသည်။ ထို instruction များသည် မမည်သည့်နေရာတွင် ရှိသနည်း။

5.1.5ညွှန်ကြားချက်များ အကျဉ်းချုပ်

ကျွန်ုပ်တို့၏ register-machine language ရှိ controller ညွှန်ကြားချက်တစ်ခုတွင် အောက်ပါပုံစံများထဲမှ တစ်ခုရှိပြီး၊ တစ်ခုစီသော ⟨ i n p u t i ⟩ သည် (reg ⟨register-name⟩) သို့မဟုတ် (const ⟨constant-value⟩) ဖြစ်သည်။ ဤညွှန်ကြားချက်များကို 5.1.1 တွင် မိတ်ဆက်ခဲ့သည်- 

(assign ⟨register-name⟩ (reg ⟨register-name⟩))
(assign ⟨register-name⟩ 
        (const ⟨constant-value⟩))
(assign ⟨register-name⟩ 
        (op ⟨operation-name⟩) 
        ⟨input₁⟩ … ⟨inputₙ⟩)
(perform (op ⟨operation-name⟩) 
         ⟨input₁⟩ 
         … 
         ⟨inputₙ⟩)
(test (op ⟨operation-name⟩) 
      ⟨input₁⟩ 
      … 
      ⟨inputₙ⟩)
(branch (label ⟨label-name⟩))
(goto (label ⟨label-name⟩))

label များကို သိမ်းဆည်းရန် register များကို အသုံးပြုခြင်းကို 5.1.3 တွင် မိတ်ဆက်ခဲ့သည်- 

(assign ⟨register-name⟩ (label ⟨label-name⟩))
(goto (reg ⟨register-name⟩))

stack ကို အသုံးပြုရန် ညွှန်ကြားချက်များကို 5.1.4 တွင် မိတ်ဆက်ခဲ့သည်- 

(save ⟨register-name⟩)
(restore ⟨register-name⟩)

ယခုအချိန်အထိ ကျွန်ုပ်တို့တွေ့ခဲ့ရသော တစ်ခုတည်းသော ⟨constant-value⟩ အမျိုးအစားမှာ ကိန်းဂဏန်း (number) ဖြစ်သော်လည်း၊ နောက်ပိုင်းတွင် string များ၊ symbol များနှင့် list များကို အသုံးပြုမည်ဖြစ်သည်။ ဥပမာအားဖြင့်၊
(const "abc") သည် string "abc" ဖြစ်သည်၊
(const abc) သည် symbol abc ဖြစ်သည်၊
(const (a b c)) သည် list (a b c) ဖြစ်သည်၊
နှင့် (const ()) သည် empty list ဖြစ်သည်။
