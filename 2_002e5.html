2.5Generic Operations များပါဝင်သော စနစ်များ

ယခင်ကဏ္ဍတွင်၊ ဒေတာအရာဝတ္ထု (data objects) များကို နည်းလမ်းတစ်ခုထက်ပို၍ ကိုယ်စားပြုနိုင်သော စနစ်များကို မည်သို့ဒီဇိုင်းထုတ်ရမည်ကို ကျွန်ုပ်တို့တွေ့ခဲ့ရသည်။ အဓိကအချက်မှာ ဒေတာလုပ်ဆောင်ချက်များကို သတ်မှတ်ပေးသည့် ကုဒ်များကို generic interface လုပ်ထုံးလုပ်နည်းများ (procedures) အသုံးပြု၍ ကိုယ်စားပြုမှုအမျိုးမျိုးနှင့် ချိတ်ဆက်ပေးရန် ဖြစ်သည်။ ယခုအခါ ဤအိုင်ဒီယာကိုပင် အသုံးပြု၍ မတူညီသော ကိုယ်စားပြုမှုများအပေါ် generic ဖြစ်သော လုပ်ဆောင်ချက်များကို သတ်မှတ်ရန်သာမက၊ မတူညီသော argument အမျိုးအစားများအပေါ် generic ဖြစ်သော လုပ်ဆောင်ချက်များကို သတ်မှတ်ရန်အတွက်ပါ မည်သို့အသုံးပြုရမည်ကို တွေ့မြင်ရမည်ဖြစ်သည်။ ကျွန်ုပ်တို့သည် မတူညီသော ဂဏန်းသင်္ချာလုပ်ဆောင်ချက် package အတော်များများကို တွေ့ခဲ့ပြီးဖြစ်သည်- ကျွန်ုပ်တို့၏ဘာသာစကားတွင် ပါရှိပြီးသား အခြေခံဂဏန်းသင်္ချာ (+, -, *, /)၊ 2.1.1 ရှိ ရာရှင်နယ်ကိန်းဂဏန်းသင်္ချာ (add-rat, sub-rat, mul-rat, div-rat) နှင့် 2.4.3 တွင် ကျွန်ုပ်တို့ အကောင်အထည်ဖော်ခဲ့သော ကိန်းရှုပ်ဂဏန်းသင်္ချာတို့ ဖြစ်ကြသည်။ ယခု ကျွန်ုပ်တို့သည် ယခင်က တည်ဆောက်ခဲ့သော ဂဏန်းသင်္ချာ package များအားလုံးကို စုစည်းထားသည့် ဂဏန်းသင်္ချာလုပ်ဆောင်ချက် package တစ်ခုကို တည်ဆောက်ရန်အတွက် data-directed နည်းပညာများကို အသုံးပြုပါမည်။

Figure 2.23 တွင် ကျွန်ုပ်တို့ တည်ဆောက်မည့် စနစ်၏ တည်ဆောက်ပုံကို ပြသထားသည်။ abstraction barriers များကို သတိပြုပါ။ “numbers” ကို အသုံးပြုသူတစ်ဦး၏ အမြင်အရ၊ ပေးပို့လိုက်သော မည်သည့်ကိန်းဂဏန်းမျိုးမဆိုတွင် လုပ်ဆောင်နိုင်သည့် add ဟူသော procedure တစ်ခုတည်းသာ ရှိသည်။ Add သည် သီးခြားစီဖြစ်သော သာမန်ဂဏန်းသင်္ချာ (ordinary-arithmetic)၊ ရာရှင်နယ်ဂဏန်းသင်္ချာ (rational-arithmetic) နှင့် ကိန်းရှုပ်ဂဏန်းသင်္ချာ (complex-arithmetic) package များကို ကိန်းဂဏန်းများအသုံးပြုသည့် ပရိုဂရမ်များမှ တညီတညွတ်တည်း ဝင်ရောက်အသုံးပြုနိုင်ရန် ခွင့်ပြုပေးသော generic interface ၏ အစိတ်အပိုင်းတစ်ခု ဖြစ်သည်။ မည်သည့် တစ်သီးပုဂ္ဂလ ဂဏန်းသင်္ချာ package မဆို (ဥပမာ- complex package ကဲ့သို့) မတူညီသော ကိုယ်စားပြုမှုများ (ဥပမာ- rectangular နှင့် polar) အတွက် ဒီဇိုင်းထုတ်ထားသော package များကို ပေါင်းစပ်ပေးသည့် generic procedures များ (ဥပမာ- add-complex) မှတစ်ဆင့် ၎င်းကိုယ်တိုင် ဝင်ရောက်အသုံးပြုနိုင်သည်။ ထို့အပြင်၊ စနစ်၏ တည်ဆောက်ပုံမှာ ပေါင်းစပ်နိုင်သော ဂုဏ်သတ္တိ (additive) ရှိသောကြောင့် တစ်သီးပုဂ္ဂလ ဂဏန်းသင်္ချာ package များကို သီးခြားစီ ဒီဇိုင်းထုတ်နိုင်ပြီး generic ဂဏန်းသင်္ချာစနစ်တစ်ခု ဖြစ်လာစေရန် ၎င်းတို့ကို ပေါင်းစပ်နိုင်သည်။

2.5.1မျိုးစုံသုံး သင်္ချာ လုပ်ဆောင်ချက်များ (Generic Arithmetic Operations)

မျိုးစုံသုံး သင်္ချာ လုပ်ဆောင်ချက်များကို ပုံစံထုတ်ခြင်း လုပ်ငန်းသည် မျိုးစုံသုံး ကိန်းရှုပ် လုပ်ဆောင်ချက်များကို ပုံစံထုတ်ခြင်းနှင့် ဆင်တူပါသည်။ ဥပမာအားဖြင့်၊ သာမန်ကိန်းများပေါ်တွင် သာမန်အခြေခံပေါင်းခြင်း + ကဲ့သို့လည်းကောင်း၊ အပိုင်းကိန်းများပေါ်တွင် add-rat ကဲ့သို့လည်းကောင်း၊ ကိန်းရှုပ်များပေါ်တွင် add-complex ကဲ့သို့လည်းကောင်း အလုပ်လုပ်နိုင်သည့် မျိုးစုံသုံး ပေါင်းခြင်း လုပ်ထုံးလုပ်နည်း add တစ်ခု ရှိစေချင်ပါသည်။ ကျွန်ုပ်တို့သည် add နှင့် အခြားသော မျိုးစုံသုံး သင်္ချာလုပ်ဆောင်ချက်များကို 2.4.3 တွင် ကိန်းရှုပ်များအတွက် မျိုးစုံသုံး selector များ အကောင်အထည်ဖော်ရာတွင် အသုံးပြုခဲ့သည့် နည်းဗျူဟာအတိုင်းပင် အကောင်အထည်ဖော်နိုင်ပါသည်။ ကျွန်ုပ်တို့သည် ကိန်းဂဏန်း အမျိုးအစား တစ်ခုစီတွင် အမျိုးအစားအမှတ်အသား (type tag) တစ်ခု ချိတ်ဆက်ပေးမည်ဖြစ်ပြီး၊ မျိုးစုံသုံး လုပ်ထုံးလုပ်နည်းကို ၎င်း၏ argument များ၏ data type အပေါ် မူတည်၍ သင့်လျော်သော package ဆီသို့ dispatch လုပ်စေမည် ဖြစ်သည်။

မျိုးစုံသုံး သင်္ချာလုပ်ထုံးလုပ်နည်းများကို အောက်ပါအတိုင်း သတ်မှတ်ထားသည် - 

(define (add x y) (apply-generic 'add x y))
(define (sub x y) (apply-generic 'sub x y))
(define (mul x y) (apply-generic 'mul x y))
(define (div x y) (apply-generic 'div x y))

ကျွန်ုပ်တို့သည် သာမန်ကိန်းများ (ordinary numbers)၊ ဆိုလိုသည်မှာ ကျွန်ုပ်တို့ ဘာသာစကား၏ အခြေခံ (primitive) ကိန်းများကို ကိုင်တွယ်ရန် package တစ်ခုကို စတင် ထည့်သွင်း (install) ပါမည်။ ဤကိန်းများကို scheme-number ဆိုသည့် သင်္ကေတဖြင့် tag ပြုလုပ်ပါမည်။ ဤ package အတွင်းရှိ သင်္ချာလုပ်ဆောင်ချက်များသည် အခြေခံ သင်္ချာလုပ်ထုံးလုပ်နည်းများပင် ဖြစ်သည် (ထို့ကြောင့် အမှတ်အသားမပါသော ကိန်းများကို ကိုင်တွယ်ရန် အပိုလုပ်ထုံးလုပ်နည်းများ သတ်မှတ်ရန် မလိုပါ)။ ဤလုပ်ဆောင်ချက် တစ်ခုစီသည် argument နှစ်ခု ယူသောကြောင့်၊ ၎င်းတို့ကို (scheme-number scheme-number) စာရင်းဖြင့် key ပေးထားသော ဇယားတွင် ထည့်သွင်းထားသည် - 

(define (install-scheme-number-package)
  (define (tag x)
    (attach-tag 'scheme-number x))
  (put 'add '(scheme-number scheme-number)
       (lambda (x y) (tag (+ x y))))
  (put 'sub '(scheme-number scheme-number)
       (lambda (x y) (tag (- x y))))
  (put 'mul '(scheme-number scheme-number)
       (lambda (x y) (tag (* x y))))
  (put 'div '(scheme-number scheme-number)
       (lambda (x y) (tag (/ x y))))
  (put 'make 'scheme-number
       (lambda (x) (tag x)))
  'done)

Scheme-number package ကို အသုံးပြုသူများသည် အမှတ်အသားပြုထားသော (tagged) သာမန်ကိန်းများကို အောက်ပါ လုပ်ထုံးလုပ်နည်းဖြင့် ဖန်တီးမည်ဖြစ်သည် - 

(define (make-scheme-number n)
  ((get 'make 'scheme-number) n))

ယခု မျိုးစုံသုံး သင်္ချာစနစ်၏ မူဘောင် (framework) အဆင်သင့်ဖြစ်ပြီဖြစ်၍ ကိန်းဂဏန်း အမျိုးအစားသစ်များကို အလွယ်တကူ ထည့်သွင်းနိုင်ပြီ ဖြစ်သည်။ ဤသည်မှာ အပိုင်းကိန်း (rational arithmetic) တွက်ချက်မှုများကို လုပ်ဆောင်ပေးသော package ဖြစ်သည်။ ပေါင်းစပ်နိုင်စွမ်း (additivity) ၏ အကျိုးကျေးဇူးအနေဖြင့်၊ 2.1.1 မှ အပိုင်းကိန်းဆိုင်ရာ ကုဒ်များကို ပြင်ဆင်မှုမရှိဘဲ package အတွင်းရှိ internal procedure များအဖြစ် အသုံးပြုနိုင်သည်ကို သတိပြုပါ - 

(define (install-rational-package)
  ;; internal procedures
  (define (numer x) (car x))
  (define (denom x) (cdr x))
  (define (make-rat n d)
    (let ((g (gcd n d)))
      (cons (/ n g) (/ d g))))
  (define (add-rat x y)
    (make-rat (+ (* (numer x) (denom y))
                 (* (numer y) (denom x)))
              (* (denom x) (denom y))))
  (define (sub-rat x y)
    (make-rat (- (* (numer x) (denom y))
                 (* (numer y) (denom x)))
              (* (denom x) (denom y))))
  (define (mul-rat x y)
    (make-rat (* (numer x) (numer y))
              (* (denom x) (denom y))))
  (define (div-rat x y)
    (make-rat (* (numer x) (denom y))
              (* (denom x) (numer y))))
  ;; interface to rest of the system
  (define (tag x) (attach-tag 'rational x))
  (put 'add '(rational rational)
       (lambda (x y) (tag (add-rat x y))))
  (put 'sub '(rational rational)
       (lambda (x y) (tag (sub-rat x y))))
  (put 'mul '(rational rational)
       (lambda (x y) (tag (mul-rat x y))))
  (put 'div '(rational rational)
       (lambda (x y) (tag (div-rat x y))))
  (put 'make 'rational
       (lambda (n d) (tag (make-rat n d))))
  'done)

(define (make-rational n d)
  ((get 'make 'rational) n d))

ကျွန်ုပ်တို့သည် complex ဆိုသည့် tag ကို အသုံးပြု၍ ကိန်းရှုပ်များကို ကိုင်တွယ်ရန် အလားတူ package တစ်ခုကို ထည့်သွင်းနိုင်ပါသည်။ package ကို ဖန်တီးရာတွင်၊ rectangular နှင့် polar package များမှ သတ်မှတ်ထားသော make-from-real-imag နှင့် make-from-mag-ang လုပ်ဆောင်ချက်များကို ဇယားမှ ထုတ်ယူ (extract) ပါသည်။ ပေါင်းစပ်နိုင်စွမ်း (Additivity) ကြောင့် 2.4.1 မှ add-complex၊ sub-complex၊ mul-complex နှင့် div-complex လုပ်ထုံးလုပ်နည်းများကိုပင် internal procedure များအဖြစ် အသုံးပြုနိုင်ခွင့် ရရှိစေပါသည်။ 

(define (install-complex-package)
  ;; imported procedures from rectangular 
  ;; and polar packages
  (define (make-from-real-imag x y)
    ((get 'make-from-real-imag 
          'rectangular) 
     x y))
  (define (make-from-mag-ang r a)
    ((get 'make-from-mag-ang 'polar) 
     r a))
  ;; internal procedures
  (define (add-complex z1 z2)
    (make-from-real-imag 
     (+ (real-part z1) (real-part z2))
     (+ (imag-part z1) (imag-part z2))))
  (define (sub-complex z1 z2)
    (make-from-real-imag 
     (- (real-part z1) (real-part z2))
     (- (imag-part z1 (imag-part z2))))
  (define (mul-complex z1 z2
    (make-from-mag-ang 
     (* (magnitude z1 (magnitude z2)
     (+ (angle z1 (angle z2))))
  (define (div-complex z1 z2
    (make-from-mag-ang 
     (/ (magnitude z1 (magnitude z2)
     (- (angle z1 (angle z2))))
  ;; interface to rest of the system
  (define (tag z) (attach-tag 'complex z))
  (put 'add '(complex complex)
       (lambda (z1 z2) 
         (tag (add-complex z1 z2))))
  (put 'sub '(complex complex)
       (lambda (z1 z2) 
         (tag (sub-complex z1 z2))))
  (put 'mul '(complex complex)
       (lambda (z1 z2) 
         (tag (mul-complex z1 z2))))
  (put 'div '(complex complex)
       (lambda (z1 z2) 
         (tag (div-complex z1 z2))))
  (put 'make-from-real-imag 'complex
       (lambda (x y) 
         (tag (make-from-real-imag x y))))
  (put 'make-from-mag-ang 'complex
       (lambda (r a) 
         (tag (make-from-mag-ang r a))))
  'done)

ကိန်းရှုပ် (complex-number) package ပြင်ပရှိ ပရိုဂရမ်များသည် ကိန်းရှုပ်များကို ကိန်းစစ် (real part) နှင့် ကိန်းယောင် (imaginary part) များမှသော်လည်းကောင်း၊ ပမာဏ (magnitude) နှင့် ထောင့် (angle) များမှသော်လည်းကောင်း တည်ဆောက်နိုင်ပါသည်။ မူလက rectangular နှင့် polar package များတွင် သတ်မှတ်ထားသော အခြေခံလုပ်ထုံးလုပ်နည်းများကို complex package သို့ မည်သို့ export လုပ်ပြီး၊ ထိုမှတစ်ဆင့် ပြင်ပကမ္ဘာသို့ မည်သို့ export လုပ်သည်ကို သတိပြုပါ။ 

(define (make-complex-from-real-imag x y)

(define (make-complex-from-mag-ang r a
  ((get 'make-from-mag-ang 'complex r a)

ဤနေရာတွင် ကျွန်ုပ်တို့ရှိသည်မှာ အဆင့်နှစ်ဆင့်ရှိသော tag စနစ် ဖြစ်သည်။ rectangular ပုံစံဖြင့် 3 + 4 i ကဲ့သို့သော ပုံမှန်ကိန်းရှုပ်တစ်ခုကို ပုံ ၂.၂၄ တွင် ပြထားသည့်အတိုင်း ဖော်ပြပါမည်။ အပြင်ဘက် tag (complex) ကို ကိန်းဂဏန်းအား complex package သို့ လမ်းညွှန်ရန် အသုံးပြုသည်။ complex package အတွင်းသို့ ရောက်ရှိသွားပါက၊ နောက်ထပ် tag (rectangular) ကို rectangular package သို့ လမ်းညွှန်ရန် အသုံးပြုသည်။ ကြီးမားပြီး ရှုပ်ထွေးသော စနစ်တစ်ခုတွင် အဆင့်များစွာ ရှိနိုင်ပြီး၊ အဆင့်တစ်ခုစီသည် နောက်တစ်ခုနှင့် မျိုးစုံသုံး လုပ်ဆောင်ချက်များမှတစ်ဆင့် ချိတ်ဆက်ထားနိုင်သည်။ data object တစ်ခုသည် "အောက်ဘက်" သို့ ဖြတ်သန်းသွားသောအခါ၊ ၎င်းကို သင့်လျော်သော package သို့ လမ်းညွှန်ရန် အသုံးပြုသော အပြင်ဘက် tag ကို (contents အား အသုံးပြု၍) ဖယ်ရှားလိုက်ပြီး၊ နောက်ထပ် dispatch လုပ်ရန်အတွက် နောက်တစ်ဆင့်ရှိ tag (ရှိခဲ့လျှင်) ကို မြင်တွေ့ရမည် ဖြစ်သည်။

အထက်ပါ package များတွင်၊ ကျွန်ုပ်တို့သည် add-rat၊ add-complex နှင့် အခြားသော သင်္ချာလုပ်ထုံးလုပ်နည်းများကို မူလက ရေးသားထားသည့်အတိုင်း အသုံးပြုခဲ့သည်။ သို့သော်၊ ဤသတ်မှတ်ချက်များသည် မတူညီသော installation procedure များ၏ internal ဖြစ်သွားပြီးနောက်တွင်၊ ၎င်းတို့သည် တစ်ခုနှင့်တစ်ခု ကွဲပြားသော အမည်များ ရှိရန် မလိုအပ်တော့ပေ။ ကျွန်ုပ်တို့သည် package နှစ်ခုလုံးတွင် ၎င်းတို့ကို add၊ sub၊ mul နှင့် div ဟု ရိုးရိုးရှင်းရှင်းပင် အမည်ပေးနိုင်ပါသည်။

လေ့ကျင့်ခန်း ၂.၇၇ - Louis Reasoner သည် ပုံ ၂.၂၄ တွင် ပြထားသော object z ပေါ်တွင် (magnitude z) expression ကို တွက်ချက်ရန် ကြိုးစားသည်။ သူ အံ့သြသွားသည်မှာ၊ အဖြေ ၅ ရမည့်အစား apply-generic ထံမှ (complex) အမျိုးအစားများပေါ်တွင် magnitude လုပ်ဆောင်ချက်အတွက် method မရှိကြောင်း error message ကို ရရှိခဲ့သည်။ သူသည် ဤအခြေအနေကို Alyssa P. Hacker အား ပြသရာ၊ သူမက “ပြဿနာက complex-number selector တွေကို complex ကိန်းတွေအတွက် ဘယ်တုန်းကမှ မသတ်မှတ်ခဲ့ဘူး၊ polar နဲ့ rectangular ကိန်းတွေအတွက်ပဲ သတ်မှတ်ခဲ့တာ။ ဒါ အလုပ်လုပ်ဖို့ဆိုရင် complex package ထဲမှာ အောက်ပါအတိုင်း ပေါင်းထည့်လိုက်ရုံပဲ” ဟု ဆိုသည်။

ဤအရာသည် အဘယ်ကြောင့် အလုပ်လုပ်သည်ကို အသေးစိတ် ဖော်ပြပါ။ ဥပမာအနေဖြင့်၊ ပုံ ၂.၂၄ တွင် ပြထားသော object z ပေါ်တွင် (magnitude z) expression ကို တွက်ချက်ရာတွင် ခေါ်ယူသွားသော လုပ်ထုံးလုပ်နည်း အားလုံးကို ခြေရာခံပြပါ။ အထူးသဖြင့်၊ apply-generic ကို ဘယ်နှစ်ကြိမ် ခေါ်ယူသနည်း။ အခြေအနေ တစ်ခုစီတွင် မည်သည့် လုပ်ထုံးလုပ်နည်းသို့ dispatch လုပ်သနည်း။

လေ့ကျင့်ခန်း ၂.၇၈ - scheme-number package အတွင်းရှိ internal procedure များသည် အခြေခံအားဖြင့် primitive procedure များဖြစ်သော +၊ - စသည်တို့ကို ခေါ်ယူခြင်းထက် မပိုပါ။ ကျွန်ုပ်တို့၏ type-tag စနစ်သည် data object တစ်ခုစီတွင် အမျိုးအစားတစ်ခု ချိတ်ဆက်ထားရန် လိုအပ်သောကြောင့် ဘာသာစကား၏ primitive များကို တိုက်ရိုက်အသုံးပြုရန် မဖြစ်နိုင်ခဲ့ပါ။ သို့သော် အမှန်စင်စစ်တွင် Lisp အကောင်အထည်ဖော်မှု အားလုံးတွင် ၎င်းတို့အတွင်းပိုင်း၌ အသုံးပြုသော type စနစ်တစ်ခု ရှိကြသည်။ symbol? နှင့် number? ကဲ့သို့သော primitive predicate များသည် data object များတွင် သတ်မှတ်ထားသော အမျိုးအစားများ ရှိမရှိကို ဆုံးဖြတ်ပေးသည်။ ကျွန်ုပ်တို့၏ မျိုးစုံသုံးစနစ်သည် Scheme ၏ အတွင်းပိုင်း type စနစ်ကို အကျိုးရှိရှိ အသုံးချနိုင်ရန် 2.4.2 မှ type-tag၊ contents၊ နှင့် attach-tag သတ်မှတ်ချက်များကို ပြင်ဆင်ပါ။ ဆိုလိုသည်မှာ၊ သာမန်ကိန်းများကို car အနေဖြင့် scheme-number သင်္ကေတပါသော pair များအဖြစ် ဖော်ပြမည့်အစား Scheme ကိန်းများအဖြစ် ရိုးရိုးရှင်းရှင်း ဖော်ပြခြင်းမှအပ စနစ်သည် ယခင်အတိုင်း အလုပ်လုပ်ရမည်။

လေ့ကျင့်ခန်း ၂.၇၉ - ကိန်းနှစ်ခု ညီမျှခြင်း ရှိမရှိ စစ်ဆေးသော မျိုးစုံသုံး equality predicate equ? ကို သတ်မှတ်ပြီး မျိုးစုံသုံး သင်္ချာ package တွင် ထည့်သွင်းပါ။ ဤလုပ်ဆောင်ချက်သည် သာမန်ကိန်းများ၊ အပိုင်းကိန်းများနှင့် ကိန်းရှုပ်များအတွက် အလုပ်လုပ်ရမည်။

လေ့ကျင့်ခန်း ၂.၈၀ - argument သည် သုည ဟုတ်မဟုတ် စစ်ဆေးသော မျိုးစုံသုံး predicate =zero? ကို သတ်မှတ်ပြီး မျိုးစုံသုံး သင်္ချာ package တွင် ထည့်သွင်းပါ။ ဤလုပ်ဆောင်ချက်သည် သာမန်ကိန်းများ၊ အပိုင်းကိန်းများနှင့် ကိန်းရှုပ်များအတွက် အလုပ်လုပ်ရမည်။

2.5.2Combining Data of Different Types

ကျွန်ုပ်တို့သည် သာမန်ကိန်းများ၊ ကိန်းရှုပ်များ၊ ရာရှင်နယ်ကိန်းများနှင့် ကျွန်ုပ်တို့ တီထွင်ရန် ဆုံးဖြတ်ထားသော အခြား မည်သည့် ကိန်းအမျိုးအစားကိုမဆို လွှမ်းခြုံနိုင်သည့် စုစည်းထားသော အရိသ်မက်တစ် (arithmetic) စနစ်တစ်ခုကို မည်သို့ သတ်မှတ်ရမည်ကို တွေ့ရှိခဲ့ပြီး ဖြစ်သော်လည်း အရေးကြီးသော ပြဿနာတစ်ခုကို လျစ်လျူရှုခဲ့ကြသည်။ ယခုအချိန်အထိ ကျွန်ုပ်တို့ သတ်မှတ်ခဲ့သော လုပ်ဆောင်ချက်များသည် မတူညီသော ဒေတာအမျိုးအစားများကို တစ်ခုနှင့်တစ်ခု လုံးဝ သီးခြားဖြစ်နေသကဲ့သို့ သဘောထားကြသည်။ ထို့ကြောင့် ဥပမာအားဖြင့် သာမန်ကိန်းနှစ်ခု သို့မဟုတ် ကိန်းရှုပ်နှစ်ခုကို ပေါင်းရန်အတွက် သီးခြား package များ ရှိနေသည်။ ကျွန်ုပ်တို့ မစဉ်းစားရသေးသည့် အချက်မှာ ကိန်းရှုပ်တစ်ခုနှင့် သာမန်ကိန်းတစ်ခုကို ပေါင်းခြင်းကဲ့သို့သော အမျိုးအစား နယ်နိမိတ်များကို ကျော်ဖြတ်သည့် လုပ်ဆောင်ချက်များကို သတ်မှတ်ရန်မှာ အဓိပ္ပာယ်ရှိသည်ဟူသော အချက်ဖြစ်သည်။ ကျွန်ုပ်တို့သည် ကျွန်ုပ်တို့၏ ပရိုဂရမ် အစိတ်အပိုင်းများကို သီးခြားစီ တည်ဆောက်နိုင်ရန်နှင့် နားလည်နိုင်ရန်အတွက် ၎င်းတို့အကြားတွင် အတားအဆီးများကို အသေအချာ ထည့်သွင်းထားခဲ့သည်။ ကျွန်ုပ်တို့သည် ဤ အမျိုးအစား-ကျော်ဖြတ် လုပ်ဆောင်ချက်များကို ကျွန်ုပ်တို့၏ မော်ဂျူး (module) နယ်နိမိတ်များကို ဆိုးဆိုးရွားရွား ချိုးဖောက်ခြင်း မရှိဘဲ ဂရုတစိုက် ထိန်းချုပ်ထားသော နည်းလမ်းဖြင့် မိတ်ဆက်ပေးလိုပါသည်။

အမျိုးအစား-ကျော်ဖြတ် လုပ်ဆောင်ချက်များကို ကိုင်တွယ်ရန် နည်းလမ်းတစ်ခုမှာ လုပ်ဆောင်ချက်သည် အကျုံးဝင်သည့် ဖြစ်နိုင်ခြေရှိသော အမျိုးအစား ပေါင်းစပ်မှု တစ်ခုချင်းစီအတွက် မတူညီသော လုပ်ထုံးလုပ်နည်း (procedure) တစ်ခုကို ဒီဇိုင်းထုတ်ရန် ဖြစ်သည်။ ဥပမာအားဖြင့်၊ ကျွန်ုပ်တို့သည် ကိန်းရှုပ် (complex number) package ကို ချဲ့ထွင်နိုင်ပြီး ကိန်းရှုပ်များကို သာမန်ကိန်းများ (ordinary numbers) နှင့် ပေါင်းရန် လုပ်ထုံးလုပ်နည်းတစ်ခုကို ပံ့ပိုးပေးကာ ၎င်းကို (complex scheme-number) ဟူသော တက်ဂ် (tag) ကို အသုံးပြု၍ ဇယားတွင် ထည့်သွင်းနိုင်သည်:115 

(define (add-complex-to-schemenum z x)
  (make-from-real-imag (+ (real-part z) x)
                       (imag-part z)))

(put 'add 
     '(complex scheme-number)
     (lambda (z x) 
       (tag (add-complex-to-schemenum z x))))

ဤနည်းစနစ်သည် အလုပ်ဖြစ်သော်လည်း ရှုပ်ထွေးလှသည်။ ထိုကဲ့သို့သော စနစ်တစ်ခုတွင် အမျိုးအစားသစ်တစ်ခုကို မိတ်ဆက်ခြင်းအတွက် ကုန်ကျစရိတ်မှာ ထိုအမျိုးအစားအတွက် လုပ်ထုံးလုပ်နည်းများ package ကို တည်ဆောက်ရုံသာမက အမျိုးအစား-ကျော်ဖြတ် လုပ်ဆောင်ချက်များကို အကောင်အထည်ဖော်သည့် လုပ်ထုံးလုပ်နည်းများကို တည်ဆောက်ခြင်းနှင့် တပ်ဆင်ခြင်းတို့လည်း ပါဝင်သည်။ ၎င်းသည် အမျိုးအစားကိုယ်တိုင်ပေါ်တွင် လုပ်ဆောင်ချက်များကို သတ်မှတ်ရန် လိုအပ်သည်ထက် ပိုမိုများပြားသော ကုဒ်များ ဖြစ်လာနိုင်သည်။ ဤနည်းလမ်းသည် သီးခြား package များကို ပေါင်းစပ်နိုင်စွမ်းကိုလည်း လျော့နည်းစေသည်၊ သို့မဟုတ် အနည်းဆုံးအားဖြင့် package တစ်ခုချင်းစီကို ဖန်တီးသူများသည် အခြား package များကို ထည့်သွင်းစဉ်းစားရန် လိုအပ်သည့် အတိုင်းအတာကို ကန့်သတ်ရန် ခက်ခဲစေသည်။ ဥပမာအားဖြင့်၊ အထက်ပါ ဥပမာတွင် ကိန်းရှုပ်နှင့် သာမန်ကိန်းများအကြား ရောနှောထားသော လုပ်ဆောင်ချက်များကို ကိုင်တွယ်ခြင်းသည် ကိန်းရှုပ် package ၏ တာဝန်ဖြစ်သင့်သည်ဟု ထင်ရသည်။ သို့သော် ရာရှင်နယ်ကိန်းများနှင့် ကိန်းရှုပ်များကို ပေါင်းစပ်ခြင်းကိုမူ ကိန်းရှုပ် package၊ ရာရှင်နယ် package သို့မဟုတ် ဤ package နှစ်ခုမှ ထုတ်ယူထားသော လုပ်ဆောင်ချက်များကို အသုံးပြုသည့် တတိယ package တစ်ခုခုမှ လုပ်ဆောင်ကောင်း လုပ်ဆောင်နိုင်သည်။ package များအကြား တာဝန်ခွဲဝေမှုဆိုင်ရာ ညီညွတ်သော မူဝါဒများကို ချမှတ်ခြင်းသည် package များစွာနှင့် အမျိုးအစား-ကျော်ဖြတ် လုပ်ဆောင်ချက်များစွာ ရှိသော စနစ်များကို ဒီဇိုင်းထုတ်ရာတွင် အလွန်ခက်ခဲသော အလုပ်တစ်ခု ဖြစ်လာနိုင်သည်။

Coercion

လုံးဝမသက်ဆိုင်သော အမျိုးအစားများပေါ်တွင် လုပ်ဆောင်နေသည့် လုံးဝမသက်ဆိုင်သော လုပ်ဆောင်ချက်များ၏ ယေဘုယျ အခြေအနေတွင်၊ အမျိုးအစား-ကျော်ဖြတ် လုပ်ဆောင်ချက်များကို အတိအလင်း အကောင်အထည်ဖော်ခြင်းသည် ရှုပ်ထွေးသော်လည်း အကောင်းဆုံး မျှော်လင့်နိုင်သည့် အရာဖြစ်သည်။ ကံကောင်းထောက်မစွာဖြင့်၊ ကျွန်ုပ်တို့၏ အမျိုးအစားစနစ်တွင် ရှိနေနိုင်သော ထပ်ဆောင်းတည်ဆောက်ပုံများကို အသုံးချခြင်းဖြင့် ပိုမိုကောင်းမွန်အောင် ပြုလုပ်နိုင်သည်။ မကြာခဏဆိုသလို မတူညီသော ဒေတာအမျိုးအစားများသည် လုံးဝ သီးခြားဖြစ်မနေဘဲ အမျိုးအစားတစ်ခုမှ အရာဝတ္ထုများကို အခြားအမျိုးအစားတစ်ခုအဖြစ် ရှုမြင်နိုင်သည့် နည်းလမ်းများ ရှိနိုင်သည်။ ဤလုပ်ငန်းစဉ်ကို coercion ဟု ခေါ်သည်။ ဥပမာအားဖြင့်၊ သာမန်ကိန်းတစ်ခုကို ကိန်းရှုပ်တစ်ခုနှင့် အရိသ်မက်တစ်နည်းအရ ပေါင်းစပ်ရန် တောင်းဆိုခံရပါက၊ သာမန်ကိန်းကို စိတ်ကူးယဉ်အပိုင်း (imaginary part) သုညဖြစ်သော ကိန်းရှုပ်တစ်ခုအဖြစ် ရှုမြင်နိုင်သည်။ ဤအရာက ပြဿနာကို ကိန်းရှုပ်နှစ်ခု ပေါင်းစပ်ခြင်းအဖြစ်သို့ ပြောင်းလဲပေးလိုက်ပြီး ကိန်းရှုပ်-အရိသ်မက်တစ် (complex-arithmetic) package မှ ပုံမှန်အတိုင်း ကိုင်တွယ်နိုင်သည်။

ယေဘုယျအားဖြင့်၊ အမျိုးအစားတစ်ခုမှ အရာဝတ္ထုတစ်ခုကို အခြားအမျိုးအစားတစ်ခု၏ ညီမျှသော အရာဝတ္ထုတစ်ခုအဖြစ်သို့ ပြောင်းလဲပေးသည့် coercion လုပ်ထုံးလုပ်နည်းများကို ဒီဇိုင်းထုတ်ခြင်းဖြင့် ဤစိတ်ကူးကို အကောင်အထည်ဖော်နိုင်သည်။ ဤသည်မှာ ပေးထားသော သာမန်ကိန်းတစ်ခုကို ထိုကိန်းစစ်အပိုင်း (real part) နှင့် သုညဖြစ်သော စိတ်ကူးယဉ်အပိုင်းတို့ ပါဝင်သည့် ကိန်းရှုပ်တစ်ခုအဖြစ်သို့ ပြောင်းလဲပေးသော ပုံစံကျ coercion လုပ်ထုံးလုပ်နည်းတစ်ခု ဖြစ်သည်- 

(define (scheme-number->complex n)
  (make-complex-from-real-imag 
   (contents n) 0))

ဤ coercion လုပ်ထုံးလုပ်နည်းများကို အမျိုးအစားနှစ်ခု၏ အမည်များအောက်တွင် အညွှန်းတပ်ထားသော အထူး coercion ဇယားတွင် ကျွန်ုပ်တို့ ထည့်သွင်းသည်- 

(put-coercion 'scheme-number 'complex 
              scheme-number->complex)

(ဤဇယားကို စီမံခန့်ခွဲရန်အတွက် put-coercion နှင့် get-coercion လုပ်ထုံးလုပ်နည်းများ ရှိသည်ဟု ကျွန်ုပ်တို့ ယူဆပါသည်။) ယေဘုယျအားဖြင့် အမျိုးအစားတစ်ခုစီမှ မည်သည့် ဒေတာအရာဝတ္ထုကိုမဆို အခြား အမျိုးအစားအားလုံးသို့ ပြောင်းလဲရန် (coerce) အမြဲတမ်း မဖြစ်နိုင်သောကြောင့် ဇယားရှိ အချို့သော နေရာလွတ်များသည် အလွတ်ဖြစ်နေလိမ့်မည်။ ဥပမာအားဖြင့်၊ မည်သည့် ကိန်းရှုပ်ကိုမဆို သာမန်ကိန်းတစ်ခုအဖြစ်သို့ ပြောင်းလဲရန် နည်းလမ်းမရှိသောကြောင့် ဇယားတွင် ယေဘုယျ complex->scheme-number လုပ်ထုံးလုပ်နည်း ပါဝင်မည်မဟုတ်ပါ။

coercion ဇယားကို သတ်မှတ်ပြီးသည်နှင့်၊ 2.4.3 မှ apply-generic လုပ်ထုံးလုပ်နည်းကို ပြင်ဆင်ခြင်းဖြင့် coercion ကို တစ်ပြေးညီ ကိုင်တွယ်နိုင်သည်။ လုပ်ဆောင်ချက်တစ်ခုကို အသုံးချရန် တောင်းဆိုသောအခါ၊ ယခင်အတိုင်းပင် အငြင်းအခုံများ၏ အမျိုးအစားများအတွက် လုပ်ဆောင်ချက်ကို သတ်မှတ်ထားခြင်း ရှိ၊ မရှိကို အရင်စစ်ဆေးသည်။ ရှိပါက၊ လုပ်ဆောင်ချက်-နှင့်-အမျိုးအစား (operation-and-type) ဇယားတွင် တွေ့ရသော လုပ်ထုံးလုပ်နည်းသို့ လွှဲပြောင်းပေးသည်။ မရှိပါက၊ coercion ကို စမ်းကြည့်သည်။ ရိုးရှင်းစေရန်အတွက် အငြင်းအခုံ နှစ်ခုသာ ရှိသော အခြေအနေကိုသာ ကျွန်ုပ်တို့ ထည့်သွင်းစဉ်းစားမည်။116 ပထမအမျိုးအစားမှ အရာဝတ္ထုများကို ဒုတိယအမျိုးအစားသို့ ပြောင်းလဲနိုင်ခြင်း ရှိ၊ မရှိ သိနိုင်ရန် coercion ဇယားကို ကျွန်ုပ်တို့ စစ်ဆေးသည်။ ရှိပါက၊ ပထမအငြင်းအခုံကို ပြောင်းလဲပြီး လုပ်ဆောင်ချက်ကို ထပ်မံကြိုးစားသည်။ ပထမအမျိုးအစားမှ အရာဝတ္ထုများကို ဒုတိယအမျိုးအစားသို့ ယေဘုယျအားဖြင့် ပြောင်းလဲ၍မရပါက၊ ဒုတိယအငြင်းအခုံကို ပထမအငြင်းအခုံ၏ အမျိုးအစားသို့ ပြောင်းလဲရန် နည်းလမ်းရှိ၊ မရှိ အပြန်အလှန် စမ်းသပ်ကြည့်သည်။ နောက်ဆုံးတွင်၊ အမျိုးအစားတစ်ခုကို အခြားတစ်ခုသို့ ပြောင်းလဲရန် သိရှိထားသော နည်းလမ်းမရှိပါက၊ ကျွန်ုပ်တို့ လက်လျှော့လိုက်သည်။ ဤသည်မှာ လုပ်ထုံးလုပ်နည်း ဖြစ်သည်- 

(define (apply-generic op . args)
  (let ((type-tags (map type-tag args)))
    (let ((proc (get op type-tags)))
      (if proc
          (apply proc (map contents args))
          (if (= (length args) 2)
              (let ((type1 (car type-tags))
                    (type2 (cadr type-tags))
                    (a1 (car args))
                    (a2 (cadr args)))
                (let ((t1->t2 
                       (get-coercion type1
                                     type2))
                      (t2->t1 
                       (get-coercion type2 
                                     type1)))
                  (cond (t1->t2
                         (apply-generic 
                          op (t1->t2 a1) a2))
                        (t2->t1
                         (apply-generic 
                          op a1 (t2->t1 a2)))
                        (else
                         (error 
                          "No method for 
                           these types"
                          (list 
                           op 
                           type-tags))))))
              (error 
               "No method for these types"
               (list op type-tags)))))))

ဤ coercion စနစ်သည် အထက်တွင် ဖော်ပြခဲ့သော အတိအလင်း အမျိုးအစား-ကျော်ဖြတ် လုပ်ဆောင်ချက်များကို သတ်မှတ်သည့် နည်းလမ်းထက် အားသာချက်များစွာ ရှိပါသည်။ အမျိုးအစားများကို ဆက်စပ်ပေးရန် coercion လုပ်ထုံးလုပ်နည်းများကို ရေးသားရန် လိုအပ်နေသေးသော်လည်း (အမျိုးအစား n ခုရှိသော စနစ်တစ်ခုအတွက် ဖြစ်နိုင်ခြေရှိသော လုပ်ထုံးလုပ်နည်း n 2 ခု)၊ အမျိုးအစား အစုအဝေးတစ်ခုစီနှင့် ဂျန်နရစ် (generic) လုပ်ဆောင်ချက် တစ်ခုစီအတွက် မတူညီသော လုပ်ထုံးလုပ်နည်းတစ်ခုစီ ရေးမည့်အစား အမျိုးအစား တစ်တွဲစီအတွက် လုပ်ထုံးလုပ်နည်း တစ်ခုစီသာ ရေးသားရန် လိုအပ်ပါသည်။117 ကျွန်ုပ်တို့ ဤနေရာတွင် အားကိုးနေသည်မှာ အမျိုးအစားများအကြား သင့်လျော်သော အသွင်ပြောင်းမှုသည် အမျိုးအစားများအပေါ်တွင်သာ မူတည်ပြီး အသုံးချမည့် လုပ်ဆောင်ချက်ပေါ်တွင် မူတည်ခြင်း မရှိဟူသော အချက်ပင် ဖြစ်သည်။

အခြားတစ်ဖက်တွင်မူ၊ ကျွန်ုပ်တို့၏ coercion စနစ်သည် လုံလောက်သော ယေဘုယျ မဆန်သည့် အသုံးချမှုများလည်း ရှိနိုင်ပါသည်။ ပေါင်းစပ်မည့် အရာဝတ္ထုနှစ်ခုလုံးကို အခြားတစ်ခု၏ အမျိုးအစားသို့ ပြောင်းလဲ၍မရသည့်တိုင်အောင်၊ အရာဝတ္ထုနှစ်ခုလုံးကို တတိယ အမျိုးအစားတစ်ခုသို့ ပြောင်းလဲခြင်းဖြင့် လုပ်ဆောင်ချက်ကို လုပ်ဆောင်နိုင်ကောင်း လုပ်ဆောင်နိုင်သည်။ ထိုကဲ့သို့သော ရှုပ်ထွေးမှုများကို ကိုင်တွယ်ရန်နှင့် ကျွန်ုပ်တို့၏ ပရိုဂရမ်များတွင် မော်ဂျူးဖြစ်မှု (modularity) ကို ထိန်းသိမ်းထားနိုင်ရန်အတွက်၊ နောက်တွင် ဆွေးနွေးမည့်အတိုင်း အမျိုးအစားများအကြား ဆက်နွယ်မှုများရှိ နောက်ထပ် တည်ဆောက်ပုံများကို အသုံးချသည့် စနစ်များကို တည်ဆောက်ရန် များသောအားဖြင့် လိုအပ်ပါသည်။

Hierarchies of types

အထက်တွင် တင်ပြခဲ့သော coercion စနစ်သည် အမျိုးအစား အစုံများအကြား သဘာဝကျသော ဆက်နွယ်မှုများ ရှိနေခြင်းအပေါ် အခြေခံထားသည်။ မကြာခဏဆိုသလို မတူညီသော အမျိုးအစားများ အချင်းချင်း မည်သို့ ဆက်နွယ်နေသည်ဟူသော အချက်တွင် ပိုမို "ကမ္ဘာလုံးဆိုင်ရာ" (global) ကျသော တည်ဆောက်ပုံများ ရှိတတ်သည်။ ဥပမာအားဖြင့်၊ အင်တီဂျာ (integers) များ၊ ရာရှင်နယ်ကိန်းများ၊ ကိန်းစစ်များနှင့် ကိန်းရှုပ်များကို ကိုင်တွယ်ရန် ဂျန်နရစ် အရိသ်မက်တစ် စနစ်တစ်ခုကို တည်ဆောက်နေသည်ဟု ဆိုပါစို့။ ထိုသို့သော စနစ်မျိုးတွင် အင်တီဂျာတစ်ခုကို ရာရှင်နယ်ကိန်း အမျိုးအစားတစ်ခုအဖြစ် မှတ်ယူရန်မှာ အလွန် သဘာဝကျပြီး၊ ရာရှင်နယ်ကိန်းကို တစ်ဖန် ကိန်းစစ် အမျိုးအစားတစ်ခုအဖြစ်လည်းကောင်း၊ ကိန်းစစ်ကို တစ်ဖန် ကိန်းရှုပ် အမျိုးအစားတစ်ခုအဖြစ်လည်းကောင်း မှတ်ယူနိုင်သည်။ ကျွန်ုပ်တို့တွင် အမှန်တကယ် ရှိနေသည်မှာ hierarchy of types (အမျိုးအစား အဆင့်ဆင့်) ဖြစ်ပြီး၊ ၎င်းတွင် ဥပမာအားဖြင့် အင်တီဂျာများသည် ရာရှင်နယ်ကိန်းများ၏ subtype (အမျိုးအစားခွဲ) ဖြစ်သည် (ဆိုလိုသည်မှာ ရာရှင်နယ်ကိန်းတစ်ခုတွင် အသုံးပြုနိုင်သော မည်သည့် လုပ်ဆောင်ချက်ကိုမဆို အင်တီဂျာတစ်ခုတွင်လည်း အလိုအလျောက် အသုံးပြုနိုင်သည်)။ ပြောင်းပြန်အားဖြင့်၊ ရာရှင်နယ်ကိန်းများသည် အင်တီဂျာများ၏ supertype (အထက်တန်းစား အမျိုးအစား) ဖြစ်သည်ဟု ဆိုကြသည်။ ကျွန်ုပ်တို့ ဤနေရာတွင် ရှိနေသော သီးခြား hierarchy မှာ အလွန်ရိုးရှင်းသော အမျိုးအစားဖြစ်ပြီး အမျိုးအစားတစ်ခုစီတွင် အများဆုံး supertype တစ်ခုနှင့် အများဆုံး subtype တစ်ခုသာ ရှိသည်။ tower (မျှော်စင်) ဟု ခေါ်သော ထိုကဲ့သို့သော တည်ဆောက်ပုံကို Figure 2.25 တွင် သရုပ်ဖော်ထားသည်။

ကျွန်ုပ်တို့တွင် tower တည်ဆောက်ပုံရှိပါက၊ hierarchy ထဲသို့ အမျိုးအစားသစ်တစ်ခု ထည့်သွင်းခြင်း ပြဿနာကို များစွာ ရိုးရှင်းစေနိုင်သည်၊ အကြောင်းမှာ အမျိုးအစားသစ်သည် ၎င်း၏ အထက်ရှိ နောက်ထပ် supertype တွင် မည်သို့ အံဝင်ခွင်ကျ ဖြစ်သည်နှင့် ၎င်း၏ အောက်ရှိ အမျိုးအစား၏ supertype မည်သို့ ဖြစ်သည်ကိုသာ ဖော်ပြရန် လိုအပ်သောကြောင့် ဖြစ်သည်။ ဥပမာအားဖြင့်၊ ကျွန်ုပ်တို့သည် အင်တီဂျာတစ်ခုကို ကိန်းရှုပ်တစ်ခုသို့ ပေါင်းလိုပါက၊ integer->complex ဟူသော အထူး coercion လုပ်ထုံးလုပ်နည်းကို အတိအလင်း သတ်မှတ်ရန် မလိုပါ။ အစား၊ အင်တီဂျာတစ်ခုကို ရာရှင်နယ်ကိန်းအဖြစ် မည်သို့ ပြောင်းလဲနိုင်သည်၊ ရာရှင်နယ်ကိန်းတစ်ခုကို ကိန်းစစ်အဖြစ် မည်သို့ ပြောင်းလဲသည်၊ နှင့် ကိန်းစစ်တစ်ခုကို ကိန်းရှုပ်အဖြစ် မည်သို့ ပြောင်းလဲသည်တို့ကို ကျွန်ုပ်တို့ သတ်မှတ်သည်။ ထို့နောက် အင်တီဂျာကို ဤအဆင့်များမှတစ်ဆင့် ကိန်းရှုပ်အဖြစ်သို့ ပြောင်းလဲရန် စနစ်အား ခွင့်ပြုပြီးနောက် ကိန်းရှုပ်နှစ်ခုကို ပေါင်းလိုက်သည်။

ကျွန်ုပ်တို့၏ apply-generic လုပ်ထုံးလုပ်နည်းကို အောက်ပါအတိုင်း ပြန်လည် ဒီဇိုင်းထုတ်နိုင်သည်- အမျိုးအစား တစ်ခုစီအတွက်၊ ထိုအမျိုးအစားရှိ အရာဝတ္ထုများကို tower ၏ အထက်တစ်ဆင့်သို့ "မြှင့်တင်ပေးသော" (raises) raise လုပ်ထုံးလုပ်နည်းတစ်ခုကို ပံ့ပိုးပေးရန် လိုအပ်သည်။ ထို့နောက် စနစ်သည် မတူညီသော အမျိုးအစားရှိ အရာဝတ္ထုများပေါ်တွင် အလုပ်လုပ်ရန် လိုအပ်လာသောအခါ၊ အရာဝတ္ထုများ အားလုံးသည် tower ၏ အဆင့်တစ်ခုတည်းသို့ ရောက်ရှိသည်အထိ နိမ့်သော အမျိုးအစားများကို ဆက်တိုက် မြှင့်တင်ပေးနိုင်သည်။ (Exercise 2.83 နှင့် Exercise 2.84 တို့သည် ထိုကဲ့သို့သော ဗျူဟာကို အကောင်အထည်ဖော်ခြင်း၏ အသေးစိတ် အချက်အလက်များနှင့် သက်ဆိုင်သည်။)

tower ၏ နောက်ထပ် အားသာချက်တစ်ခုမှာ အမျိုးအစားတိုင်းသည် supertype ပေါ်တွင် သတ်မှတ်ထားသော လုပ်ဆောင်ချက်အားလုံးကို "အမွေဆက်ခံသည်" (inherits) ဟူသော အယူအဆကို အလွယ်တကူ အကောင်အထည်ဖော်နိုင်ခြင်း ဖြစ်သည်။ ဥပမာအားဖြင့်၊ အင်တီဂျာတစ်ခု၏ ကိန်းစစ်အပိုင်းကို ရှာရန် အထူးလုပ်ထုံးလုပ်နည်းတစ်ခုကို မပံ့ပိုးထားသော်လည်း၊ အင်တီဂျာများသည် ကိန်းရှုပ်များ၏ subtype ဖြစ်နေသည့်အတွက် real-part သည် အင်တီဂျာများအတွက် သတ်မှတ်ပြီးသား ဖြစ်နေလိမ့်မည်ဟု ကျွန်ုပ်တို့ မျှော်လင့်သင့်သည်။ tower တစ်ခုတွင်၊ apply-generic ကို ပြင်ဆင်ခြင်းဖြင့် ၎င်းကို တစ်ပြေးညီ ဖြစ်ပျက်စေရန် စီစဉ်နိုင်သည်။ လိုအပ်သော လုပ်ဆောင်ချက်ကို ပေးထားသော အရာဝတ္ထု၏ အမျိုးအစားအတွက် တိုက်ရိုက် သတ်မှတ်မထားပါက၊ အရာဝတ္ထုကို ၎င်း၏ supertype သို့ မြှင့်တင်ပြီး ထပ်မံ ကြိုးစားသည်။ ဤသို့ဖြင့် လိုချင်သော လုပ်ဆောင်ချက်ကို လုပ်ဆောင်နိုင်သည့် အဆင့်တစ်ခုကို တွေ့သည့်အထိ သို့မဟုတ် ထိပ်ဆုံးသို့ ရောက်သည့်အထိ (ထိုအခြေအနေတွင် ကျွန်ုပ်တို့ လက်လျှော့လိုက်သည်) ကျွန်ုပ်တို့၏ အငြင်းအခုံကို ပြောင်းလဲရင်း tower ပေါ်သို့ တက်သွားနိုင်သည်။

ယေဘုယျကျသော hierarchy ထက် tower ၏ နောက်ထပ် အားသာချက်တစ်ခုမှာ ဒေတာအရာဝတ္ထုတစ်ခုကို အရိုးရှင်းဆုံး ကိုယ်စားပြုမှုသို့ "နှိမ့်ချရန်" (lower) ရိုးရှင်းသော နည်းလမ်းကို ပေးစွမ်းခြင်း ဖြစ်သည်။ ဥပမာအားဖြင့်၊ ကျွန်ုပ်တို့သည် 2 + 3 i ကို 4 − 3 i နှင့် ပေါင်းပါက၊ ရလဒ်ကို ကိန်းရှုပ် 6 + 0 i အဖြစ် ရရှိမည့်အစား အင်တီဂျာ 6 အဖြစ် ရရှိလျှင် ကောင်းလိမ့်မည်။ Exercise 2.85 တွင် ထိုကဲ့သို့သော lowering လုပ်ဆောင်ချက်ကို အကောင်အထည်ဖော်သည့် နည်းလမ်းကို ဆွေးနွေးထားသည်။ (လှည့်ကွက်မှာ 6 + 0 i ကဲ့သို့သော နှိမ့်ချနိုင်သော အရာဝတ္ထုများကို 6 + 2 i ကဲ့သို့သော နှိမ့်ချ၍မရသော အရာဝတ္ထုများမှ ခွဲခြားသိမြင်ရန် ယေဘုယျနည်းလမ်းတစ်ခု လိုအပ်ခြင်းပင် ဖြစ်သည်။)

Inadequacies of hierarchies

ကျွန်ုပ်တို့၏ စနစ်ရှိ ဒေတာအမျိုးအစားများကို tower တစ်ခုတွင် သဘာဝကျကျ စီစဉ်နိုင်ပါက၊ ကျွန်ုပ်တို့ တွေ့ခဲ့ရသည့်အတိုင်း မတူညီသော အမျိုးအစားများပေါ်ရှိ ဂျန်နရစ် လုပ်ဆောင်ချက်များကို ကိုင်တွယ်ရသည့် ပြဿနာများကို များစွာ ရိုးရှင်းစေသည်။ ကံမကောင်းစွာဖြင့်၊ များသောအားဖြင့် ထိုသို့ မဟုတ်ကြပါ။ Figure 2.26 သည် ရောနှောထားသော အမျိုးအစားများ၏ ပိုမိုရှုပ်ထွေးသော စီစဉ်မှုကို သရုပ်ဖော်ထားပြီး၊ ဤအရာသည် ဂျီဩမေတြီ ရုပ်ပုံလွှာ အမျိုးအစားများအကြား ဆက်နွယ်မှုများကို ပြသနေသည်။ ယေဘုယျအားဖြင့် အမျိုးအစားတစ်ခုတွင် subtype တစ်ခုထက်မက ရှိနိုင်သည်ကို ကျွန်ုပ်တို့ တွေ့ရသည်။ ဥပမာအားဖြင့် တြိဂံများနှင့် စတုဂံများသည် နှစ်ခုလုံး ပိုလီဂွန် (polygons) များ၏ subtype များ ဖြစ်ကြသည်။ ထို့အပြင် အမျိုးအစားတစ်ခုတွင် supertype တစ်ခုထက်မကလည်း ရှိနိုင်သည်။ ဥပမာအားဖြင့် နားသန်ထောင့်မှန်တြိဂံ (isosceles right triangle) တစ်ခုကို နားသန်တြိဂံ သို့မဟုတ် ထောင့်မှန်တြိဂံ အဖြစ် ရှုမြင်နိုင်သည်။ ဤ supertype အများအပြား ရှိနေခြင်း ပြဿနာမှာ အထူးသဖြင့် ခက်ခဲလှသည်၊ အကြောင်းမှာ ၎င်းသည် hierarchy တွင် အမျိုးအစားတစ်ခုကို "မြှင့်တင်ရန်" ထူးခြားသော နည်းလမ်းတစ်ခုတည်း မရှိဟု ဆိုလိုသောကြောင့် ဖြစ်သည်။ အရာဝတ္ထုတစ်ခုသို့ လုပ်ဆောင်ချက်တစ်ခု အသုံးချရန်အတွက် "မှန်ကန်သော" supertype ကို ရှာဖွေခြင်းသည် apply-generic ကဲ့သို့သော လုပ်ထုံးလုပ်နည်းအတွက် တစ်ခုလုံးသော အမျိုးအစား ကွန်ရက်တစ်လျှောက် အတော်အတန် ရှာဖွေရန် လိုအပ်နိုင်သည်။ အမျိုးအစားတစ်ခုအတွက် subtype အများအပြား ရှိနေလေ့ရှိသောကြောင့်၊ တန်ဖိုးတစ်ခုကို အမျိုးအစား hierarchy အောက်သို့ "နှိမ့်ချရန်" (coercing down) တွင်လည်း အလားတူ ပြဿနာမျိုး ရှိသည်။ ကြီးမားသော စနစ်များကို ဒီဇိုင်းထုတ်ရာတွင် မော်ဂျူးဖြစ်မှုကို ထိန်းသိမ်းထားရင်း အပြန်အလှန် ဆက်နွယ်နေသော အမျိုးအစား အမြောက်အမြားကို ကိုင်တွယ်ခြင်းသည် အလွန်ခက်ခဲပြီး လက်ရှိ သုတေသန ပြုလုပ်နေသော နယ်ပယ်တစ်ခု ဖြစ်သည်။118

2.5.3ဥပမာ- သင်္ကေတသုံး အက္ခရာသင်္ချာ (Symbolic Algebra)

သင်္ကေတသုံး အက္ခရာသင်္ချာ ဖော်ပြချက် (symbolic algebraic expressions) များကို တွက်ချက်ကိုင်တွယ်ခြင်းသည် အဆင့်မြင့်စနစ်ကြီးများ (large-scale systems) ကို ဒီဇိုင်းထုတ်ရာတွင် ကြုံတွေ့ရလေ့ရှိသည့် အခက်ခဲဆုံး ပြဿနာများစွာကို ဖော်ပြပေးနိုင်သော ရှုပ်ထွေးသည့် လုပ်ငန်းစဉ်တစ်ခု ဖြစ်သည်။ ယေဘုယျအားဖြင့် အက္ခရာသင်္ချာ ဖော်ပြချက်တစ်ခုကို ဆင့်ကဲတည်ဆောက်ပုံ (hierarchical structure) တစ်ခုအဖြစ် ရှုမြင်နိုင်ပြီး၊ ၎င်းသည် operand များအပေါ် သက်ရောက်သည့် operator များ၏ အပင်သဏ္ဌာန်တည်ဆောက်ပုံ (tree) တစ်ခု ဖြစ်သည်။ ကျွန်ုပ်တို့သည် ကိန်းသေ (constants) နှင့် ကိန်းရှင် (variables) များကဲ့သို့သော အခြေခံအရာဝတ္ထု (primitive objects) များမှ စတင်၍၊ ၎င်းတို့ကို ပေါင်းခြင်းနှင့် မြှောက်ခြင်းကဲ့သို့သော အက္ခရာသင်္ချာဆိုင်ရာ operator များဖြင့် ပေါင်းစပ်ကာ အက္ရာသင်္ချာ ဖော်ပြချက်များကို တည်ဆောက်နိုင်သည်။ အခြားသော ဘာသာစကားများကဲ့သို့ပင်၊ ကျွန်ုပ်တို့သည် ပေါင်းစပ်အရာဝတ္ထု (compound objects) များကို ရိုးရှင်းသော အသုံးအနှုန်းများဖြင့် ရည်ညွှန်းနိုင်စေရန် abstraction များကို ဖန်တီးကြသည်။ သင်္ကေတသုံး အက္ခရာသင်္ချာတွင် တွေ့ရလေ့ရှိသော abstraction များမှာ linear combination၊ ပိုလီနိုမီယယ် (polynomial)၊ rational function သို့မဟုတ် ထရိဂိုနိုမေတြီ ဖန်ရှင် (trigonometric function) တို့ ဖြစ်ကြသည်။ ၎င်းတို့ကို ဖော်ပြချက်များကို လုပ်ဆောင်ရာတွင် လမ်းညွှန်ပေးနိုင်သော ပေါင်းစပ် "အမျိုးအစားများ" (compound types) အဖြစ် သတ်မှတ်နိုင်သည်။ ဥပမာအားဖြင့်၊ ကျွန်ုပ်တို့သည် ဖော်ပြချက် x 2 sin ⁡ ( y 2 + 1 ) + x cos ⁡ 2 y + cos ⁡ ( y 3 − 2 y 2 ) ကို ကိန်းပြည့် (integers) မြှောက်ဖော်ကိန်းများပါဝင်သော y ပိုလီနိုမီယယ်များ၏ ထရိဂိုနိုမေတြီ ဖန်ရှင်များကို မြှောက်ဖော်ကိန်း (coefficients) များအဖြစ် အသုံးပြုထားသည့် x ပိုလီနိုမီယယ်တစ်ခုအဖြစ် ဖော်ပြနိုင်သည်။

ကျွန်ုပ်တို့သည် ဤနေရာတွင် ပြီးပြည့်စုံသော အက္ခရာသင်္ချာ တွက်ချက်ကိုင်တွယ်မှုစနစ်တစ်ခုကို တည်ဆောက်ရန် ကြိုးပမ်းမည်မဟုတ်ပါ။ ထိုသို့သော စနစ်များသည် နက်နဲသော အက္ခရာသင်္ချာ ဗဟုသုတများနှင့် ကောင်းမွန်လှပသော algorithm များပါဝင်သည့် အလွန်တရာ ရှုပ်ထွေးသော ပရိုဂရမ်များ ဖြစ်ကြသည်။ ကျွန်ုပ်တို့ လုပ်ဆောင်မည်မှာ အက္ခရာသင်္ချာ တွက်ချက်မှု၏ ရိုးရှင်းသော်လည်း အရေးကြီးသော အစိတ်အပိုင်းတစ်ခုဖြစ်သည့် ပိုလီနိုမီယယ်များ၏ သင်္ချာတွက်ချက်ခြင်း (arithmetic of polynomials) ကို လေ့လာခြင်း ဖြစ်သည်။ ထိုသို့သော စနစ်တစ်ခုကို ဒီဇိုင်းထုတ်သူ ကြုံတွေ့ရမည့် ဆုံးဖြတ်ချက် အမျိုးအစားများနှင့်၊ ဤလုပ်ငန်းစဉ်ကို စနစ်တကျဖြစ်အောင် ကူညီပေးနိုင်ရန် abstract data နှင့် generic operations ဆိုင်ရာ အကြံဉာဏ်များကို မည်သို့အသုံးချရမည်ကို ကျွန်ုပ်တို့ သရုပ်ဖော်ပြသသွားမည် ဖြစ်သည်။

ပိုလီနိုမီယယ်များ အပေါ်တွင် သင်္ချာတွက်ချက်ခြင်း (Arithmetic on polynomials)

ပိုလီနိုမီယယ်များ အပေါ်တွင် သင်္ချာတွက်ချက်မှု ပြုလုပ်နိုင်မည့် စနစ်တစ်ခုကို ဒီဇိုင်းထုတ်ရာတွင် ကျွန်ုပ်တို့၏ ပထမဆုံးအလုပ်မှာ ပိုလီနိုမီယယ်ဆိုသည်မှာ အဘယ်နည်းဆိုသည်ကို ဆုံးဖြတ်ရန် ဖြစ်သည်။ ပိုလီနိုမီယယ်များကို ပုံမှန်အားဖြင့် ကိန်းရှင်အချို့ (ပိုလီနိုမီယယ်၏ indeterminates များ) နှင့် ဆက်စပ်၍ အဓိပ္ပာယ်ဖွင့်ဆိုကြသည်။ ရိုးရှင်းစေရန်အတွက်၊ ကျွန်ုပ်တို့သည် indeterminate တစ်ခုတည်းသာပါဝင်သော ပိုလီနိုမီယယ်များ ( univariate polynomials) ကိုသာ ကန့်သတ်လေ့လာပါမည်။120 ပိုလီနိုမီယယ်တစ်ခုကို ကိန်းစု (terms) များ၏ စုစုပေါင်းအဖြစ် အဓိပ္ပာယ်ဖွင့်ဆိုပါမည်။ ထိုကိန်းစုတစ်ခုချင်းစီသည် မြှောက်ဖော်ကိန်း (coefficient) တစ်ခု၊ indeterminate ၏ ထပ်ညွှန်းတစ်ခု၊ သို့မဟုတ် မြှောက်ဖော်ကိန်းနှင့် indeterminate ၏ ထပ်ညွှန်းတို့၏ မြှောက်လဒ်တစ်ခု ဖြစ်နိုင်သည်။ မြှောက်ဖော်ကိန်း (coefficient) ဆိုသည်မှာ ပိုလီနိုမီယယ်၏ indeterminate အပေါ် မှီခိုခြင်းမရှိသော အက္ခရာသင်္ချာ ဖော်ပြချက်တစ်ခု ဖြစ်သည်။ ဥပမာအားဖြင့်၊ 5 x 2 + 3 x + 7 သည် x ပါဝင်သော ရိုးရှင်းသည့် ပိုလီနိုမီယယ်တစ်ခုဖြစ်ပြီး၊ ( y 2 + 1 ) x 3 + ( 2 y ) x + 1 သည် y ပိုလီနိုမီယယ်များကို မြှောက်ဖော်ကိန်းများအဖြစ် အသုံးပြုထားသော x ပိုလီနိုမီယယ်တစ်ခု ဖြစ်သည်။

ကျွန်ုပ်တို့သည် အစကတည်းကပင် ခက်ခဲနက်နဲသော ပြဿနာအချို့ကို စတင်ကြုံတွေ့နေရပြီ ဖြစ်သည်။ အထက်ပါ ပိုလီနိုမီယယ်များထဲမှ ပထမတစ်ခုသည် ပိုလီနိုမီယယ် 5 y 2 + 3 y + 7 နှင့် အတူတူပင် ဖြစ်ပါသလား၊ သို့မဟုတ် မတူပါသလား။ သင့်လျော်သော အဖြေမှာ "အကယ်၍ ကျွန်ုပ်တို့သည် ပိုလီနိုမီယယ်ကို သင်္ချာဆိုင်ရာ ဖန်ရှင်တစ်ခုအဖြစ် သက်သက်သာ ရှုမြင်ပါက 'ဟုတ်ပါသည်' ဖြစ်ပြီး၊ အကယ်၍ တည်ဆောက်ပုံဆိုင်ရာ ပုံသဏ္ဌာန် (syntactic form) တစ်ခုအဖြစ် ရှုမြင်ပါက 'မဟုတ်ပါ' ဖြစ်လိမ့်မည်။" ဒုတိယမြောက် ပိုလီနိုမီယယ်သည် x ပိုလီနိုမီယယ်များကို မြှောက်ဖော်ကိန်းများအဖြစ် အသုံးပြုထားသော y ပိုလီနိုမီယယ်တစ်ခုနှင့် အက္ခရာသင်္ချာသဘောအရ ညီမျှသည်။ ကျွန်ုပ်တို့၏ စနစ်သည် ဤအချက်ကို သိရှိနားလည်သင့်သလား၊ သို့မဟုတ် မသိရှိသင့်သလား။ ထို့အပြင်၊ ပိုလီနိုမီယယ်တစ်ခုကို ကိုယ်စားပြုရန် အခြားနည်းလမ်းများစွာ ရှိပါသေးသည်—ဥပမာအားဖြင့် ဆခွဲကိန်းများ၏ မြှောက်လဒ်အဖြစ်၊ သို့မဟုတ် (univariate polynomial အတွက်) အဖြေ (roots) များ၏ အစုအဖြစ်၊ သို့မဟုတ် သတ်မှတ်ထားသော အမှတ်များတွင် ပိုလီနိုမီယယ်၏ တန်ဖိုးများကို စာရင်းပြုစုခြင်းအဖြစ် ကိုယ်စားပြုနိုင်သည်။121 ကျွန်ုပ်တို့၏ အက္ခရာသင်္ချာ တွက်ချက်ကိုင်တွယ်မှုစနစ်တွင် "ပိုလီနိုမီယယ်" ဆိုသည်မှာ ၎င်း၏ နောက်ကွယ်မှ သင်္ချာဆိုင်ရာ အဓိပ္ပာယ်ထက် သတ်သတ်မှတ်မှတ် တည်ဆောက်ပုံဆိုင်ရာ ပုံသဏ္ဌာန် (syntactic form) တစ်ခုသာ ဖြစ်သည်ဟု ဆုံးဖြတ်ခြင်းဖြင့် ဤမေးခွန်းများကို ကျော်လွှားနိုင်သည်။

ယခုအခါ ပိုလီနိုမီယယ်များအပေါ် သင်္ချာတွက်ချက်မှု မည်သို့ပြုလုပ်ရမည်ကို စဉ်းစားရပါမည်။ ဤရိုးရှင်းသော စနစ်တွင် ကျွန်ုပ်တို့သည် ပေါင်းခြင်းနှင့် မြှောက်ခြင်းကိုသာ ထည့်သွင်းစဉ်းစားပါမည်။ ထို့အပြင် ပေါင်းစပ်မည့် ပိုလီနိုမီယယ်နှစ်ခုသည် indeterminate တစ်ခုတည်းသာ ရှိရမည်ဟု ကျွန်ုပ်တို့ ကန့်သတ်ထားပါမည်။

ကျွန်ုပ်တို့၏ စနစ်ကို ဒီဇိုင်းထုတ်ရာတွင် ရင်းနှီးပြီးသားဖြစ်သော data abstraction ၏ စည်းမျဉ်းကို လိုက်နာပါမည်။ ပိုလီနိုမီယယ်များကို ကိန်းရှင်တစ်ခုနှင့် ကိန်းစုများ အစုအဝေးတစ်ခု ပါဝင်သော poly ဟုခေါ်သည့် data structure တစ်ခုကို အသုံးပြု၍ ကိုယ်စားပြုပါမည်။ poly တစ်ခုမှ အဆိုပါ အစိတ်အပိုင်းများကို ထုတ်ယူပေးမည့် variable နှင့် term-list ဟူသော selector များနှင့်၊ ပေးထားသော ကိန်းရှင်နှင့် ကိန်းစုစာရင်းမှ poly တစ်ခုကို တည်ဆောက်ပေးမည့် make-poly ဟူသော constructor တစ်ခု ရှိသည်ဟု ကျွန်ုပ်တို့ ယူဆပါမည်။ ကိန်းရှင်သည် သင်္ကေတ (symbol) တစ်ခုသာ ဖြစ်လိမ့်မည်၊ ထို့ကြောင့် ကိန်းရှင်များကို နှိုင်းယှဉ်ရန် 2.3.2 ပါ same-variable? လုပ်ထုံးလုပ်နည်းကို အသုံးပြုနိုင်သည်။ အောက်ပါ လုပ်ထုံးလုပ်နည်းများသည် poly များ၏ ပေါင်းခြင်းနှင့် မြှောက်ခြင်းကို အဓိပ္ပာယ်ဖွင့်ဆိုထားခြင်း ဖြစ်သည်-

ကျွန်ုပ်တို့၏ polynomial စနစ်သည် အမျိုးအစားတစ်ခုမှ အရာဝတ္ထုများ (polynomials) သည် အမှန်တကယ်တွင် မတူညီသော အမျိုးအစားများစွာရှိသော အရာဝတ္ထုများကို အစိတ်အပိုင်းများအဖြစ် ပိုင်ဆိုင်ထားသည့် ရှုပ်ထွေးသော အရာဝတ္ထုများ ဖြစ်နိုင်ပုံကို သရုပ်ဖော်သည်။ ဤအချက်သည် generic operations များကို သတ်မှတ်ရာတွင် အမှန်တကယ် အခက်အခဲ မရှိစေပါ။ ကျွန်ုပ်တို့သည် compound types များ၏ အစိတ်အပိုင်းများကို လိုအပ်သလို ကိုင်တွယ်ဆောင်ရွက်ရန် သင့်လျော်သော generic operations များကိုသာ ထည့်သွင်းရန် လိုအပ်ပါသည်။ အမှန်တကယ်တွင်၊ polynomial ၏ အစိတ်အပိုင်းများသည်လည်း polynomial များ ဖြစ်နိုင်သောကြောင့် polynomials များသည် “recursive data abstraction” တစ်မျိုးဖြစ်ကြောင်း ကျွန်ုပ်တို့ တွေ့ခဲ့ရသည်။ ကျွန်ုပ်တို့၏ generic operations နှင့် data-directed programming ပုံစံသည် ဤရှုပ်ထွေးမှုကို အခက်အခဲမရှိ ကိုင်တွယ်နိုင်သည်။

အခြားတစ်ဖက်တွင်မူ၊ polynomial algebra သည် data types များကို tower ပုံစံဖြင့် သဘာဝကျကျ မစီစဉ်နိုင်သော စနစ်တစ်ခု ဖြစ်သည်။ ဥပမာအားဖြင့်၊ coefficients များသည် y ၏ polynomials ဖြစ်သော x ၏ polynomials များ ရှိနိုင်သည်။ ထို့အတူ coefficients များသည် x ၏ polynomials ဖြစ်သော y ၏ polynomials များလည်း ရှိနိုင်သည်။ ဤအမျိုးအစားနှစ်ခုလုံးသည် မည်သည့်သဘာဝကျသော နည်းလမ်းနှင့်မျှ တစ်ခုက တစ်ခု၏ “အထက်” တွင် မရှိသော်လည်း၊ set တစ်ခုစီမှ element များကို ပေါင်းစပ်ရန် လိုအပ်လေ့ရှိသည်။ ဤသို့ပြုလုပ်ရန် နည်းလမ်းများစွာ ရှိပါသည်။ ဖြစ်နိုင်ခြေတစ်ခုမှာ polynomial နှစ်ခုလုံးတွင် တူညီသော ပင်မ variable ရှိစေရန် term များကို ချဲ့ထွင်ခြင်းနှင့် ပြန်လည်စီစဉ်ခြင်းဖြင့် polynomial တစ်ခုကို အခြားတစ်ခု၏ အမျိုးအစားသို့ ပြောင်းလဲခြင်း ဖြစ်သည်။ variable များကို အစဉ်လိုက်စီခြင်းဖြင့် ဤအရာပေါ်တွင် tower ကဲ့သို့သော ဖွဲ့စည်းပုံကို သတ်မှတ်နိုင်ပြီး၊ ထိုသို့ဖြင့် မည်သည့် polynomial ကိုမဆို ဦးစားပေးအမြင့်ဆုံး variable က လွှမ်းမိုးပြီး ဦးစားပေးနိမ့်သော variable များကို coefficients များထဲတွင် မြှုပ်နှံထားသည့် “canonical form” သို့ အမြဲပြောင်းလဲနိုင်သည်။ ဤဗျူဟာသည် အတော်အတန် ကောင်းမွန်စွာ အလုပ်လုပ်သော်လည်း၊ ပြောင်းလဲခြင်းသည် polynomial ကို မလိုအပ်ဘဲ ချဲ့ထွင်စေနိုင်ပြီး ဖတ်ရခက်ခဲစေကာ အလုပ်လုပ်ရာတွင် ထိရောက်မှု နည်းပါးစေနိုင်သည်။ tower ဗျူဟာသည် ဤနယ်ပယ်အတွက် သို့မဟုတ် user က trigonometric functions၊ power series နှင့် integrals ကဲ့သို့သော အမျိုးမျိုးသော ပေါင်းစပ်ပုံစံများဖြင့် အမျိုးအစားဟောင်းများကို အသုံးပြု၍ အမျိုးအစားသစ်များကို dynamic နည်းဖြင့် တီထွင်နိုင်သော မည်သည့်နယ်ပယ်အတွက်မဆို သဘာဝမကျသည်မှာ သေချာပါသည်။

ကြီးမားသော algebraic-manipulation စနစ်များ၏ ဒီဇိုင်းတွင် coercion ကို ထိန်းချုပ်ခြင်းသည် ကြီးမားသော ပြဿနာတစ်ခု ဖြစ်နေခြင်းမှာ အံ့သြစရာ မဟုတ်ပါ။ ထိုကဲ့သို့သော စနစ်များ၏ ရှုပ်ထွေးမှု အတော်များများသည် ကွဲပြားသော အမျိုးအစားများအကြား ဆက်စပ်မှုများနှင့် သက်ဆိုင်သည်။ အမှန်စင်စစ်၊ ကျွန်ုပ်တို့သည် coercion ကို လုံးဝနားမလည်သေးဟု ပြောလျှင် မှန်ပါလိမ့်မည်။ အမှန်တကယ်တွင်မူ၊ ကျွန်ုပ်တို့သည် data type ဟူသော အယူအဆကိုပင် လုံးဝနားမလည်သေးပါ။ သို့သော်လည်း၊ ကျွန်ုပ်တို့ သိရှိထားသောအရာများသည် ကြီးမားသော စနစ်များ၏ ဒီဇိုင်းကို ပံ့ပိုးရန် အားကောင်းသော ဖွဲ့စည်းပုံနှင့် modularity အခြေခံမူများကို ပေးဆောင်ပါသည်။

လေ့ကျင့်ခန်း ၂.၉၂: variable များပေါ်တွင် အစီအစဉ်တစ်ခု သတ်မှတ်ခြင်းဖြင့်၊ မတူညီသော variable များရှိသော polynomials များအတွက် ပေါင်းခြင်းနှင့် မြှောက်ခြင်း အလုပ်လုပ်စေရန် polynomial package ကို ချဲ့ထွင်ပါ။ (ဤသည်မှာ မလွယ်ကူပါ!)

နောက်ဆက်တွဲ လေ့ကျင့်ခန်း: Rational functions

ကျွန်ုပ်တို့၏ generic arithmetic စနစ်တွင် rational functions များကိုပါဝင်စေရန် ချဲ့ထွင်နိုင်သည်။ ၎င်းတို့သည် အပိုင်းကိန်းများဖြစ်ပြီး numerator နှင့် denominator တို့မှာ x + 1 x 3 − 1 . ကဲ့သို့သော polynomials များ ဖြစ်ကြသည်။ စနစ်သည် rational functions များကို ပေါင်း၊ နှုတ်၊ မြှောက်၊ စား ပြုလုပ်နိုင်ရမည်ဖြစ်ပြီး၊ x + 1 x 3 − 1 + x x 2 − 1 = x 3 + 2 x 2 + 3 x + 1 x 4 + x 3 − x − 1 . ကဲ့သို့သော တွက်ချက်မှုများကို လုပ်ဆောင်နိုင်ရမည်။ (ဤနေရာတွင် ဘုံဆခွဲကိန်းများကို ဖယ်ထုတ်ခြင်းဖြင့် စုစုပေါင်းပေါင်းလဒ်ကို ရိုးရှင်းအောင် ပြုလုပ်ထားသည်။ သာမန် “cross multiplication” သည် ပဉ္စမထပ်ကိန်း polynomial အပေါ်တွင် စတုတ္ထထပ်ကိန်း polynomial တစ်ခုကို ထွက်ပေါ်စေလိမ့်မည်။)

ကျွန်ုပ်တို့၏ rational-arithmetic package ကို generic operations များ အသုံးပြုရန် ပြုပြင်မွမ်းမံပါက၊ အပိုင်းကိန်းများကို အနိမ့်ဆုံးကိန်းသို့ လျှော့ချသည့် ပြဿနာမှလွဲ၍ ကျန်ရှိသည်များကို ကျွန်ုပ်တို့ လိုချင်သည့်အတိုင်း လုပ်ဆောင်ပေးလိမ့်မည်။

လေ့ကျင့်ခန်း ၂.၉၃: generic operations များကို အသုံးပြုရန် rational-arithmetic package ကို ပြုပြင်မွမ်းမံပါ၊ သို့သော် make-rat ကို အပိုင်းကိန်းများကို အနိမ့်ဆုံးကိန်းသို့ လျှော့ချရန် မကြိုးစားစေရန် ပြောင်းလဲပါ။ rational function တစ်ခု ထုတ်ပေးရန် polynomial နှစ်ခုပေါ်တွင် make-rational ကို ခေါ်ဆိုခြင်းဖြင့် သင်၏စနစ်ကို စမ်းသပ်ပါ: 

(define p1 (make-polynomial 'x '((2 1) (0 1))))
(define p2 (make-polynomial 'x '((3 1) (0 1))))
(define rf (make-rational p2 p1))

ယခု add ကို အသုံးပြု၍ rf ကို ၎င်းကိုယ်တိုင်သို့ ပေါင်းပါ။ ဤပေါင်းခြင်းလုပ်ငန်းစဉ်သည် အပိုင်းကိန်းများကို အနိမ့်ဆုံးကိန်းသို့ မလျှော့ချသည်ကို သင်တွေ့ရပါလိမ့်မည်။

ကျွန်ုပ်တို့သည် ကိန်းပြည့်များတွင် အသုံးပြုခဲ့သည့် အယူအဆအတိုင်းပင် — numerator နှင့် denominator နှစ်ခုလုံးကို ၎င်းတို့၏ အကြီးဆုံးဘုံဆခွဲကိန်း (greatest common divisor) နှင့် စားရန် make-rat ကို ပြုပြင်မွမ်းမံခြင်းဖြင့် — polynomial အပိုင်းကိန်းများကို အနိမ့်ဆုံးကိန်းသို့ လျှော့ချနိုင်သည်။ “အကြီးဆုံးဘုံဆခွဲကိန်း” ဟူသော အယူအဆသည် polynomials များအတွက်လည်း အဓိပ္ပာယ်ရှိသည်။ အမှန်စင်စစ်၊ ကျွန်ုပ်တို့သည် ကိန်းပြည့်များအတွက် အလုပ်လုပ်သော Euclid’s Algorithm နှင့် အခြေခံအားဖြင့် တူညီသော algorithm ကို အသုံးပြု၍ polynomials နှစ်ခု၏ GCD ကို တွက်ချက်နိုင်သည်။126 ကိန်းပြည့်ဗားရှင်းမှာ 

(define (gcd a b)
  (if (= b 0)
      a
      (gcd b (remainder a b))))

ဤအရာကို အသုံးပြု၍ term lists များပေါ်တွင် အလုပ်လုပ်သော GCD operation တစ်ခုကို သတ်မှတ်ရန် ထင်ရှားသော ပြုပြင်မွမ်းမံမှုကို ပြုလုပ်နိုင်သည်: 

(define (gcd-terms a b)
  (if (empty-termlist? b)
      a
      (gcd-terms b (remainder-terms a b))))

ဤနေရာတွင် remainder-terms သည် လေ့ကျင့်ခန်း ၂.၉၁ တွင် အကောင်အထည်ဖော်ခဲ့သော term-list စားခြင်းလုပ်ဆောင်ချက် div-terms မှ ပြန်ပေးသော list ၏ အကြွင်းအပိုင်းကို ထုတ်ယူပေးသည်။

လေ့ကျင့်ခန်း ၂.၉၄: div-terms ကို အသုံးပြု၍ remainder-terms procedure ကို အကောင်အထည်ဖော်ပါ၊ ၎င်းကို အသုံးပြု၍ gcd-terms ကို အထက်ပါအတိုင်း သတ်မှတ်ပါ။ ယခု poly နှစ်ခု၏ polynomial GCD ကို တွက်ချက်သော gcd-poly procedure တစ်ခုကို ရေးသားပါ။ (poly နှစ်ခုသည် variable တူညီခြင်း မရှိပါက procedure သည် error တစ်ခု ပြသရမည်။) polynomials များအတွက် gcd-poly သို့လည်းကောင်း၊ သာမန်ကိန်းဂဏန်းများအတွက် သာမန် gcd သို့လည်းကောင်း လျှော့ချပေးမည့် generic operation greatest-common-divisor ကို စနစ်အတွင်း ထည့်သွင်းပါ။ စမ်းသပ်မှုတစ်ခုအနေဖြင့်၊ အောက်ပါအတိုင်း ကြိုးစားပါ 

(define p1 
  (make-polynomial 
   'x '((4 1) (3 -1) (2 -2) (1 2))))

(define p2 
  (make-polynomial 
   'x '((3 1) (1 -1))))

(greatest-common-divisor p1 p2)

ပြီးလျှင် သင်၏ရလဒ်ကို ကိုယ်တိုင်တွက်ချက်၍ စစ်ဆေးပါ။

Exercise 2.95: P 1 , P 2 , နှင့် P 3 တို့ကို အောက်ပါ polynomial များအဖြစ် သတ်မှတ်ပါ- P 1 : x 2 − 2 x + 1 , P 2 : 11 x 2 + 7 , P 3 : 13 x + 5. ယခု Q 1 ကို P 1 နှင့် P 2 တို့၏ မြှောက်လဒ်အဖြစ်လည်းကောင်း၊ Q 2 ကို P 1 နှင့် P 3 တို့၏ မြှောက်လဒ်အဖြစ်လည်းကောင်း သတ်မှတ်ပြီး၊ Q 1 နှင့် Q 2 တို့၏ GCD ကို တွက်ထုတ်ရန် greatest-common-divisor (Exercise 2.94) ကို အသုံးပြုပါ။ ရရှိလာသော အဖြေသည် P 1 နှင့် တူညီမှုမရှိသည်ကို သတိပြုပါ။ ဤဥပမာသည် တွက်ချက်မှုအတွင်းသို့ ကိန်းပြည့်မဟုတ်သော လုပ်ဆောင်ချက်များ (noninteger operations) ကို ရောက်ရှိလာစေပြီး၊ GCD algorithm တွင် အခက်အခဲများ ဖြစ်ပေါ်စေပါသည်။127 ဘာဖြစ်နေသည်ကို နားလည်စေရန် GCD ကို တွက်ချက်နေစဉ်အတွင်း gcd-terms ကို trace လုပ်ကြည့်ပါ သို့မဟုတ် စားခြင်းကို ကိုယ်တိုင် လက်ဖြင့် တွက်ချက်ကြည့်ပါ။

GCD algorithm ၏ အောက်ပါ ပြုပြင်မွမ်းမံမှုကို အသုံးပြုပါက Exercise 2.95 တွင် တွေ့ရသော ပြဿနာကို ဖြေရှင်းနိုင်ပါသည် (ဤနည်းလမ်းသည် ကိန်းပြည့် coefficient များရှိသော polynomial များအတွက်သာ အမှန်တကယ် အလုပ်လုပ်ပါသည်)။ GCD တွက်ချက်မှုတွင် မည်သည့် polynomial စားခြင်းကိုမဆို မလုပ်ဆောင်မီ၊ စားကိန်း (dividend) ကို ကိန်းပြည့်ကိန်းသေ (integer constant factor) တစ်ခုဖြင့် မြှောက်ပေးရမည်ဖြစ်ပြီး၊ ၎င်းကိန်းသေသည် စားနေစဉ်အတွင်း အပိုင်းကိန်းများ မထွက်ပေါ်လာစေရန် အာမခံနိုင်သော ကိန်းဖြစ်ရပါမည်။ ထို့ကြောင့် ကျွန်ုပ်တို့၏ အဖြေသည် အမှန်တကယ် GCD နှင့် ကိန်းပြည့်ကိန်းသေတစ်ခုစာ ကွာခြားနေမည်ဖြစ်သော်လည်း၊ ရာရှင်နယ်ဖန်ရှင် (rational functions) များကို အရှင်းဆုံးပုံစံ (lowest terms) သို့ လျှော့ချရာတွင် ၎င်းသည် အရေးမကြီးပါ။ GCD ကို ပိုင်းဝေနှင့် ပိုင်းခြေ နှစ်ခုလုံးကို စားရန် အသုံးပြုမည်ဖြစ်သောကြောင့် ထိုကိန်းပြည့်ကိန်းသေသည် အချင်းချင်း ကျေသွားမည် ဖြစ်သည်။

ပို၍ တိတိကျကျ ပြောရလျှင် P နှင့် Q တို့သည် polynomial များ ဖြစ်ပါက၊ O 1 ကို P ၏ order (ဆိုလိုသည်မှာ P ၏ အကြီးဆုံး term ၏ order) ဟု သတ်မှတ်ပြီး O 2 ကို Q ၏ order ဟု သတ်မှတ်ပါ။ c ကို Q ၏ leading coefficient ဟု သတ်မှတ်ပါ။ ထိုအခါ အကယ်၍ ကျွန်ုပ်တို့သည် P ကို integerizing factor ဖြစ်သော c 1 + O 1 − O 2 နှင့် မြှောက်ပါက၊ ရရှိလာသော polynomial ကို div-terms algorithm အသုံးပြု၍ အပိုင်းကိန်းများ မထွက်ပေါ်စေဘဲ Q နှင့် စားနိုင်ကြောင်း ပြသနိုင်ပါသည်။ စားကိန်းကို ဤကိန်းသေဖြင့် မြှောက်ပြီးမှ စားခြင်းလုပ်ဆောင်ချက်ကို တစ်ခါတစ်ရံ pseudodivision ( P ကို Q ဖြင့် စားခြင်း) ဟု ခေါ်ဆိုသည်။ စားခြင်းမှ ရရှိလာသော အကြွင်းကို pseudoremainder ဟု ခေါ်သည်။

117 အကယ်၍ ကျွန်ုပ်တို့သည် ပါးနပ်စွာ စဉ်းစားမည်ဆိုလျှင် n 2 ထက် နည်းပါးသော coercion procedures များဖြင့် များသောအားဖြင့် အဆင်ပြေအောင် လုပ်ဆောင်နိုင်ပါသည်။ ဥပမာအားဖြင့်၊ အကယ်၍ ကျွန်ုပ်တို့သည် type 1 မှ type 2 သို့ ပြောင်းလဲခြင်းနှင့် type 2 မှ type 3 သို့ ပြောင်းလဲခြင်းကို သိရှိထားပါက၊ ဤဗဟုသုတကို အသုံးပြု၍ type 1 မှ type 3 သို့ ပြောင်းလဲနိုင်ပါသည်။ ၎င်းသည် စနစ်ထဲသို့ အမျိုးအစားသစ်တစ်ခု ပေါင်းထည့်သောအခါ ကျွန်ုပ်တို့ တိုက်ရိုက်ထောက်ပံ့ပေးရန် လိုအပ်သော coercion procedures အရေအတွက်ကို များစွာ လျှော့ချပေးနိုင်ပါသည်။ အကယ်၍ ကျွန်ုပ်တို့သည် ကျွန်ုပ်တို့၏ စနစ်အတွင်း လိုအပ်သော ခေတ်မီဆန်းသစ်မှုအဆင့်ကို ထည့်သွင်းလိုပါက၊ အမျိုးအစားများကြားရှိ ဆက်နွယ်မှုဆိုင်ရာ “graph” ကို ရှာဖွေစေပြီး၊ တိုက်ရိုက်ထောက်ပံ့ထားသော အရာများမှတစ်ဆင့် ဆင်ခြင်ကောက်ချက်ချနိုင်သော coercion procedures များကို အလိုအလျောက် ထုတ်ပေးနိုင်အောင် ပြုလုပ်နိုင်ပါသည်။

118 ဤစာအုပ်၏ ပထမအကြိမ်ထုတ်ဝေမှုတွင်လည်း ပါဝင်ခဲ့သော ဤဖော်ပြချက်သည် လွန်ခဲ့သော တစ်ဆယ့်နှစ်နှစ်က ကျွန်ုပ်တို့ ရေးသားခဲ့စဉ်ကကဲ့သို့ပင် ယခုအချိန်တွင်လည်း မှန်ကန်နေဆဲဖြစ်သည်။ မတူညီသော entities အမျိုးအစားများကြားရှိ ဆက်နွယ်မှုများကို ဖော်ပြရန်အတွက် အသုံးဝင်ပြီး ယေဘုယျကျသော မူဘောင် (framework) တစ်ခု (ဒဿနိကဗေဒပညာရှင်များက “ontology” ဟု ခေါ်ဆိုသည်) ကို တည်ဆောက်ရန်မှာ ခက်ခဲလွန်းလှသည်ဟု ထင်ရသည်။ လွန်ခဲ့သော ဆယ်နှစ်ခန့်က ရှိခဲ့သော ရှုပ်ထွေးမှုများနှင့် ယခုရှိနေသော ရှုပ်ထွေးမှုများကြားရှိ အဓိကကွာခြားချက်မှာ ယခုအခါတွင် မလုံလောက်သော ontological theories အမျိုးမျိုးကို ၎င်းနှင့်အညီ မလုံလောက်သော programming languages အမြောက်အမြားတွင် ထည့်သွင်းလာကြခြင်းပင် ဖြစ်သည်။ ဥပမာအားဖြင့်၊ object-oriented programming languages များ၏ ရှုပ်ထွေးမှုများစွာ—နှင့် ခေတ်ပြိုင် object-oriented languages များကြားရှိ သိမ်မွေ့ပြီး ရှုပ်ထွေးသော ကွာခြားချက်များ—သည် ဆက်နွယ်နေသော အမျိုးအစားများပေါ်တွင် generic operations များကို ကိုင်တွယ်ပုံအပေါ်တွင် ဗဟိုပြုနေသည်။ Chapter 3 ပါ computational objects များအကြောင်း ကျွန်ုပ်တို့၏ ဆွေးနွေးမှုတွင် ဤကိစ္စရပ်များကို လုံးဝရှောင်ရှားထားပါသည်။ Object-oriented programming နှင့် ရင်းနှီးသော စာဖတ်သူများသည် ကျွန်ုပ်တို့အနေဖြင့် chapter 3 တွင် local state အကြောင်း များစွာပြောဆိုထားသော်လည်း “classes” သို့မဟုတ် “inheritance” အကြောင်းကို လုံးဝမဖော်ပြထားသည်ကို သတိပြုမိပါလိမ့်မည်။ အမှန်စင်စစ်၊ ဤပြဿနာများကို knowledge representation နှင့် automated reasoning ဆိုင်ရာ လုပ်ငန်းများကို အသုံးပြုမှသာ လုံလောက်စွာ ဖြေရှင်းနိုင်မည်ဖြစ်ပြီး computer-language design သက်သက်ဖြင့်သာ ဖြေရှင်းနိုင်မည်မဟုတ်ကြောင်း ကျွန်ုပ်တို့ သံသယရှိပါသည်။

119 ကိန်းစစ် (real number) တစ်ခုကို ၎င်း၏ argument နှင့် အနီးစပ်ဆုံး ကိန်းပြည့်ကို ပြန်ပေးသော round primitive ကို အသုံးပြု၍ ကိန်းပြည့် (integer) အဖြစ်သို့ ပြောင်းလဲနိုင်ပါသည်။

120 အခြားတစ်ဖက်တွင်မူ၊ coefficient များကိုယ်တိုင်က အခြား variables များပါဝင်သော polynomials များဖြစ်နေသည့် polynomials များကို ကျွန်ုပ်တို့ ခွင့်ပြုပါမည်။ ၎င်းသည် အောက်တွင် ဆွေးနွေးထားသည့်အတိုင်း coercion ပြဿနာများကို ဖြစ်ပေါ်စေသော်လည်း၊ အနှစ်သာရအားဖြင့် ကျွန်ုပ်တို့အား အပြည့်အဝ multivariate system တစ်ခုကဲ့သို့ပင် ဖော်ပြနိုင်စွမ်းအား (representational power) ကို ပေးစွမ်းမည်ဖြစ်သည်။

121 Univariate polynomials များအတွက်၊ ပေးထားသော အမှတ်အစုများတွင် polynomial ၏ တန်ဖိုးကို ပေးခြင်းသည် အထူးသဖြင့် ကောင်းမွန်သော ကိုယ်စားပြုမှု (representation) တစ်ခု ဖြစ်နိုင်ပါသည်။ ၎င်းသည် polynomial arithmetic ကို အလွန်ရိုးရှင်းစေသည်။ ဥပမာအားဖြင့်၊ ဤနည်းဖြင့် ကိုယ်စားပြုထားသော polynomials နှစ်ခု၏ ပေါင်းလဒ်ကို ရရှိရန်အတွက် သက်ဆိုင်ရာ အမှတ်များရှိ polynomials များ၏ တန်ဖိုးများကို ပေါင်းထည့်ရန်သာ လိုအပ်သည်။ ပိုမိုရင်းနှီးသော ကိုယ်စားပြုမှုသို့ ပြန်လည်ပြောင်းလဲရန်အတွက်၊ အမှတ် n + 1 ရှိ polynomial ၏ တန်ဖိုးများကို ပေးထားလျှင် degree n ရှိသော polynomial ၏ coefficients များကို မည်သို့ပြန်လည်ရယူရမည်ကို ပြသသည့် Lagrange interpolation formula ကို အသုံးပြုနိုင်ပါသည်။

122 ဤ operation သည် Exercise 2.62 တွင် ကျွန်ုပ်တို့ တည်ဆောက်ခဲ့သော အစီအစဉ်တကျဖြစ်သော union-set operation နှင့် အလွန်ဆင်တူပါသည်။ အမှန်စင်စစ်၊ အကယ်၍ ကျွန်ုပ်တို့သည် polynomial ၏ terms များကို indeterminate ၏ ထပ်ညွှန်း (power) အလိုက် စီထားသော အစု (set) တစ်ခုအဖြစ် စဉ်းစားပါက၊ ပေါင်းလဒ်အတွက် term list ကို ထုတ်ပေးသော ပရိုဂရမ်သည် union-set နှင့် ထပ်တူနီးပါး ဖြစ်ပါသည်။

123 ဤအရာကို လုံးဝချောမွေ့စွာ လုပ်ဆောင်နိုင်ရန်အတွက်၊ ကျွန်ုပ်တို့၏ generic arithmetic system ထဲသို့ “number” တစ်ခုကို coefficient အဖြစ် ထိုကိန်းပါဝင်သော degree zero ရှိသည့် polynomial အဖြစ် သတ်မှတ်ခြင်းဖြင့် polynomial အဖြစ်သို့ coerce လုပ်နိုင်သော စွမ်းရည်ကိုလည်း ထည့်သွင်းသင့်ပါသည်။ အကယ်၍ ကျွန်ုပ်တို့သည် coefficient y + 1 ကို coefficient 2 နှင့် ပေါင်းရန် လိုအပ်သော [ x 2 + ( y + 1 ) x + 5 ] + [ x 2 + 2 x + 1 ] , ကဲ့သို့သော operations များကို လုပ်ဆောင်မည်ဆိုပါက ၎င်းသည် လိုအပ်ပါသည်။

124 ဤ polynomial ဥပမာများတွင်၊ ကျွန်ုပ်တို့သည် Exercise 2.78 တွင် အကြံပြုထားသော type mechanism ကို အသုံးပြု၍ generic arithmetic system ကို အကောင်အထည်ဖော်ထားသည်ဟု ယူဆပါသည်။ ထို့ကြောင့်၊ သာမန်ကိန်းဂဏန်းများဖြစ်သော coefficients များကို ၎င်းတို့ကိုယ်တိုင် ကိန်းဂဏန်းများအဖြစ်သာ ကိုယ်စားပြုမည်ဖြစ်ပြီး car တွင် scheme-number သင်္ကေတပါဝင်သော pairs များအဖြစ် ကိုယ်စားပြုမည်မဟုတ်ပါ။

125 Term lists များသည် အစီအစဉ်တကျဖြစ်သည်ဟု ကျွန်ုပ်တို့ ယူဆထားသော်လည်း၊ adjoin-term ကို လက်ရှိ term list ပေါ်သို့ term အသစ်ကို cons လုပ်ရုံမျှဖြင့် အကောင်အထည်ဖော်ထားပါသည်။ adjoin-term ကို အသုံးပြုသော လုပ်ထုံးလုပ်နည်းများ (ဥပမာ add-terms) သည် ၎င်းကို list တွင် ရှိနေသည်ထက် ပိုမိုမြင့်မားသော order ရှိသည့် term ဖြင့် အမြဲတမ်း ခေါ်ဆိုမည်ဟု အာမခံနိုင်သရွေ့ ဤသို့ လုပ်ဆောင်နိုင်ပါသည်။ အကယ်၍ ကျွန်ုပ်တို့သည် ထိုကဲ့သို့ အာမမခံလိုပါက၊ adjoin-term ကို အစီအစဉ်တကျရှိသော list အဖြစ် ကိုယ်စားပြုထားသည့် sets များအတွက် adjoin-set constructor နှင့် ဆင်တူအောင် အကောင်အထည်ဖော်နိုင်ခဲ့ပါသည် (Exercise 2.61)။

126 Euclid’s Algorithm သည် polynomials များအတွက် အလုပ်လုပ်သည်ဟူသော အချက်ကို အက္ခရာသင်္ချာတွင် polynomials များသည် Euclidean ring ဟု ခေါ်ဆိုသော algebraic domain အမျိုးအစားတစ်ခု ဖြစ်သည်ဟု ဖော်ပြလေ့ရှိပါသည်။ Euclidean ring ဆိုသည်မှာ ပေါင်းခြင်း၊ နှုတ်ခြင်းနှင့် commutative multiplication တို့ကို လက်ခံပြီး၊ ring ၏ element x တစ်ခုစီအတွက် အောက်ပါဂုဏ်သတ္တိများရှိသော အပေါင်းကိန်းပြည့် “measure” m ( x ) တစ်ခုကို သတ်မှတ်ပေးသည့် နည်းလမ်းရှိသော domain တစ်ခု ဖြစ်ပါသည်။ သုညမဟုတ်သော မည်သည့် x နှင့် y အတွက်မဆို m ( x y ) ≥ m ( x ) ဖြစ်ပြီး၊ မည်သည့် x နှင့် y အတွက်မဆို y = q x + r ဖြစ်ပြီး r = 0 သို့မဟုတ် m ( r ) < m ( x ) ဖြစ်စေမည့် q တစ်ခု ရှိရမည်ဖြစ်သည်။ စိတ္တဇရှုထောင့် (abstract point of view) မှကြည့်လျှင် ၎င်းသည် Euclid’s Algorithm အလုပ်လုပ်ကြောင်း သက်သေပြရန်အတွက် လိုအပ်သောအချက် ဖြစ်သည်။ ကိန်းပြည့်များ (integers) ၏ domain အတွက်၊ ကိန်းပြည့်တစ်ခု၏ measure m သည် ထိုကိန်းပြည့်၏ ပကတိတန်ဖိုး (absolute value) ပင် ဖြစ်သည်။ Polynomials များ၏ domain အတွက်၊ polynomial တစ်ခု၏ measure သည် ၎င်း၏ degree ပင် ဖြစ်သည်။

127 MIT Scheme ကဲ့သို့သော အကောင်အထည်ဖော်မှုတွင်၊ ၎င်းသည် Q 1 နှင့် Q 2 တို့၏ divisor အမှန်တကယ်ဖြစ်သော polynomial တစ်ခုကို ထုတ်ပေးသော်လည်း coefficient များမှာ rational coefficients များ ဖြစ်ပါလိမ့်မည်။ ကိန်းပြည့်များကို စားခြင်းကြောင့် တိကျမှုကန့်သတ်ထားသော ဆယ်လီစိတ်ကိန်း (decimal numbers) များ ထွက်ပေါ်နိုင်သည့် အခြား Scheme systems အများအပြားတွင်မူ မှန်ကန်သော divisor ကို ရရှိရန် ပျက်ကွက်နိုင်ပါသည်။

128 Polynomial GCD များကို တွက်ချက်ရန်အတွက် အလွန်ထိရောက်ပြီး သပ္ပာယ်သော နည်းလမ်းတစ်ခုကို Richard Zippel (1979) က ရှာဖွေတွေ့ရှိခဲ့သည်။ ဤနည်းလမ်းသည် Chapter 1 တွင် ကျွန်ုပ်တို့ ဆွေးနွေးခဲ့သော fast test for primality ကဲ့သို့ပင် probabilistic algorithm တစ်ခု ဖြစ်သည်။ Zippel ၏ စာအုပ် (Zippel 1993) တွင် ဤနည်းလမ်းနှင့်အတူ polynomial GCD များကို တွက်ချက်ရန် အခြားနည်းလမ်းများကိုလည်း ဖော်ပြထားပါသည်။
